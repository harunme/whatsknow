## 8.1 分布式系统设计

### Q1：如何设计一个告警分发系统，保证消息不丢失、不重复？At-least-once 和 Exactly-once 的实现代价是什么？你们在 Kafka 场景下选择哪种语义？

**考察点**：消息语义、Kafka 可靠性配置

**答案要点**：
- **保证消息不丢失**：
  1. **生产者端**：配置 `acks=all`（所有 ISR 副本确认）+ `retries=3` + `enable.idempotence=true`（幂等 Producer）
  2. **Broker 端**：设置 `replication.factor=3`（3 副本）+ `min.insync.replicas=2`
  3. **消费者端**：手动提交 Offset（`enable.auto.commit=false`），处理完消息后再提交
- **保证消息不重复**：
  - **消费端幂等**：消费前查幂等键（如 Redis SETNX），处理后再标记
  - **数据库唯一约束**：处理结果写入 MySQL 时用唯一索引（alert_id + 处理状态）防止重复处理
- **消息语义对比**：
  - **At-least-once**：消息可能重复但不会丢失。消费者必须幂等。实现简单（生产端重试 + 消费端幂等）
  - **Exactly-once**：消息既不丢失也不重复。实现代价极高：需要 Kafka 事务（Producer + Consumer 事务）或分布式事务（如 Seata）
- **Kafka 场景选择**：
  - **At-least-once 是主流选择**：成本合理，消费者通过幂等键处理重复
  - Exactly-once 仅用于**财务/计费等强一致性场景**

**可能的追问**：
- Kafka 的幂等 Producer（`enable.idempotence=true`）是如何实现的？
- `min.insync.replicas=2` 和 `acks=all` 的关系是什么？
- 消费者处理成功但提交 Offset 失败，会发生什么？

### Q2：在分布式系统中，如果消费者处理失败，消息应该如何重试？指数退避、Dead Letter Queue（DLQ）分别解决什么问题？

**考察点**：消息重试策略、DLQ、死信处理

**答案要点**：
- **重试策略**：
  - **本地重试**：消费失败后立即重试 1-2 次（适合临时性错误，如网络抖动）
  - **延迟重试**：失败后等待一段时间再重试（`delay=1000ms`），避免立即重试导致热循环
  - **指数退避**：`delay = base * 2^attempt`（1s → 2s → 4s → 8s），配合最大重试次数和最大延迟上限
  - **最大重试次数**：超过后不再重试，进入 DLQ
- **指数退避解决的问题**：避免重试风暴（Retrying Storm），在服务暂时不可用时减少对下游的压力
- **DLQ（Dead Letter Queue）**：
  - 重试耗尽后，消息进入 DLQ（死信队列）保存
  - DLQ 消息不会丢失，供后续人工排查/修复
  - 应用：Kafka Topic `alert-processing-DLQ`，消费者消费 DLQ 进行补偿
- **Kafka 实现 DLQ**：
  ```java
  // 消费逻辑
  try {
      processAlert(alert);
  } catch (Exception e) {
      if (retryCount < MAX_RETRIES) {
          // 发送到重试 Topic（带 delay header）
          kafkaTemplate.send("alert-processing-retry", alert);
      } else {
          // 发送 DLQ
          kafkaTemplate.send("alert-processing-DLQ", alert);
      }
  }
  ```

**可能的追问**：
- Kafka 延迟消息有哪些实现方式？（时间轮 / 延迟 Topic）
- 如何监控 DLQ 的积压量并告警？
- DLQ 消息如何做补偿处理？（定时任务扫 DLQ 或人工触发）

### Q3：设计一个实时告警处理流水线，从 SIEM 接入到 Agent 研判到 Case 创建，如何设计数据流和容错机制？

**考察点**：系统设计、实时处理流水线、容错

**答案要点**：
- **数据流设计**：
  ```
  [SIEM/日志源] → [Kafka ingestion Topic] → [预处理 Consumer]
  → [告警去重 / 标准化 / Enrichment]
  → [Kafka alert-standardized Topic]
  → [AI Agent Consumer]
  → [Severity 定级 / Attack Stage 识别]
  → [决策路由]
      ├── 高危 → [Kafka case-creation Topic] → [Case Service] → [通知]
      ├── 中危 → [工单系统]
      └── 低危 → [归档 / 忽略]
  ```
- **容错机制**：
  1. **消息持久化**：Kafka 多副本（3 副本）+ `acks=all`，消息不丢
  2. **消费者幂等**：处理前查 Redis（alert_id）判断是否已处理
  3. **Consumer 故障恢复**：手动 Offset 提交，消费者重启后从已处理位置继续
  4. **AI Agent 超时**：超时后标记 alert.needs_human_review，不阻塞流水线
  5. **Kafka Consumer Group 故障转移**：Group Leader 重新分配 Partition，最小化中断
  6. **死信队列**：处理失败消息进入 DLQ，不阻塞主流程
- **性能考量**：
  - 分区数 = 消费者数量（扩展并行度）
  - 告警 Enrichment（威胁情报查询）并行化，不阻塞主流程

**可能的追问**：
- 如何保证告警的因果顺序？（相同 alert_id 的事件必须有序处理）
- 如果 AI Agent 不可用，如何降级？（规则引擎兜底）
- 告警流水线如何监控？（Prometheus + Grafana，监控 lag、错误率）

### Q4：分布式锁的实现方式有哪些？Redis 的 `SETNX + EXPIRE` 和 Zookeeper 的临时有序节点分别有什么坑？你们在 SOC 平台中哪些场景用到了分布式锁？

**考察点**：分布式锁原理、Redis vs Zookeeper

**答案要点**：
- **Redis 分布式锁（SETNX + EXPIRE）**：
  ```lua
  -- 原子加锁
  if redis.call('SET', key, uuid, 'NX', 'PX', 30000) then return 1 else return 0 end
  -- 原子解锁（Lua 保证原子）
  if redis.call('GET', key) == uuid then return redis.call('DEL', key) else return 0 end
  ```
  - **坑 1**：`SETNX` + `EXPIRE` 两步不是原子的，中间崩溃会导致死锁 → 用 `SET key value NX PX ms` 解决
  - **坑 2**：`DEL` 可能误删别人的锁（别人已获取锁后超时释放）→ 用 value = UUID，解锁前校验
- **Zookeeper 临时有序节点**：
  - 原理：每个客户端在 `/lock/` 下创建临时有序节点，最小序号节点持有锁
  - 优点：锁自动释放（客户端崩溃后 Session 过期，临时节点自动删除）
  - **坑**：性能比 Redis 低（ZK 是 CP 系统，写入需要过半节点确认），不适合高频锁场景
- **Redis vs Zookeeper**：
  | 维度 | Redis | Zookeeper |
  |------|-------|-----------|
  | 一致性 | AP（可用性优先） | CP（一致性优先） |
  | 性能 | 高（纯内存） | 中等（需 Zab 协议） |
  | 锁释放 | 依赖 TTL | Session 超期自动释放 |
  | 适用场景 | 高性能、短期锁 | 强一致性需求 |
- **SOC 平台场景**：
  - 告警去重锁（同一 alert_id 同时只能一个 Consumer 处理）
  - Case 创建分布式锁（防止重复创建）
  - AI 模型调用限流（同一模型同时只能 N 个请求调用）

**可能的追问**：
- Redlock（红锁）算法是什么？它真的安全吗？
- Redis 分布式锁的 TTL 如何设置？太短会怎样？
- Zookeeper 的 Watch 机制在分布式锁中起什么作用？

### Q5：接口幂等性如何设计？Token 机制、唯一索引、乐观锁分别在什么场景下适用？

**考察点**：幂等性设计、分布式幂等

**答案要点**：
- **Token 机制（服务端生成）**：
  - 流程：服务端生成唯一 token → 客户端请求携带 token → 服务端 Redis SETNX(token, 1, TTL) 校验 → 存在则幂等
  - 适用场景：**HTTP 接口幂等**（POST 创建资源、POST 支付）
  - 优点：实现简单，不依赖业务逻辑
  - 缺点：多步骤操作中 token 管理复杂
- **唯一索引（数据库层）**：
  - 在 DB 表上建唯一索引（如 `alert_id + operation_type`）
  - 重复请求触发 DuplicateKeyException，捕获后返回成功
  - 适用场景：**数据层幂等**（订单创建、告警处理记录）
  - 优点：数据库保证，无需额外服务
  - 缺点：依赖 DB，频繁冲突影响性能
- **乐观锁（版本号/CAS）**：
  - `UPDATE alerts SET severity = ?, version = version + 1 WHERE id = ? AND version = ?`
  - 版本不匹配时更新失败，捕获后重试
  - 适用场景：**并发更新同一行数据**
  - 优点：无锁，性能高
  - 缺点：需要业务字段支持版本号
- **实际选择**：通常组合使用（Token 在 API 层 + 唯一索引在 DB 层）

**可能的追问**：
- POST 请求的幂等性如何保证？（POST 创建资源，通常不幂等；如果要幂等，可以用唯一键约束）
- 幂等性和并发控制有什么关系？
- 如何设计幂等键的生命周期？（TTL 设置、清理策略）

## 8.2 消息队列深入

### Q6：Kafka 的分区策略（Key Hash、轮询、自定义）是什么？如何根据业务负载设计分区数？分区数过少或过多分别有什么问题？

**考察点**：Kafka 分区机制、分区规划

**答案要点**：
- **分区策略**：
  - **Key Hash**（默认）：`hash(key) % num_partitions`，相同 key 的消息一定发到同一分区（保证有序）
  - **轮询（Round Robin）**：忽略 key，依次发到各分区（负载均衡但不保序）
  - **自定义分区**：实现 `Partitioner` 接口，按业务规则分区（如按告警级别分区）
- **分区数设计**：
  - 经验公式：`分区数 = max(峰值 QPS, 副本数 × 消费者数)`
  - 考虑因素：消费者并行度（通常分区数 = 消费者数）、单分区吞吐（~1MB/s）
  - 示例：告警分发 QPS=1000，消费者 6 个 → 分区数 = max(1000, 6) = 1000（保守按 QPS）+ 适当 buffer
- **过多/过少的问题**：
  - **分区数过少**：并发度不足，吞吐受限；消费者数量受限制；Partition leader 过多增加 controller 负担
  - **分区数过多**：每个分区有 Leader 副本，内存占用增加；Controller 管理分区多，选举慢；单 Consumer 处理多个分区，Rebalance 成本高

**可能的追问**：
- 分区数可以动态增加吗？增加分区数后对已有消息有什么影响？
- Key Hash 策略下，相同 key 的消息一定在同一分区吗？在 rebalance 后呢？
- Kafka 的分区数和 RocketMQ 的队列数概念有什么区别？

### Q7：Kafka 消费者组（Consumer Group）的偏移量（Offset）管理机制是什么？自动提交和手动提交的优缺点？消费者 rebalance 的触发条件和影响？

**考察点**：Consumer Group、Offset 提交、Rebalance

**答案要点**：
- **Offset 管理**：
  - **提交方式**：`enable.auto.commit=true`（自动，每 5s）→ `enable.auto.commit=false`（手动，代码控制）
  - **__consumer_offsets Topic**：Consumer Group 的 Offset 保存在内部 Topic（50 个分区），不依赖外部存储
  - **语义**：`committed offset` = 已处理的最大消息位置，消费者从 `committed offset + 1` 开始消费
- **自动提交 vs 手动提交**：
  - **自动提交**：简单，但可能重复处理（提交了但消息没处理完就崩溃）
  - **手动提交**：精确控制，但需要正确处理提交时机（处理成功后再提交）
  ```python
  consumer = KafkaConsumer('alert-topic', enable_auto_commit=False)
  for msg in consumer:
      process(msg.value)
      consumer.commit()  # 处理成功后提交
  ```
- **Rebalance 触发条件**：
  - Consumer 加入/离开 Group
  - Consumer 崩溃（心跳超时）
  - 分区数变化
- **Rebalance 影响**：
  - 期间所有 Consumer 暂停消费（Stop The World）
  - 新 Consumer 需要重新获取分区分配
  - 影响延迟，多 Consumer 频繁 Rebalance 会导致吞吐下降
- **避免频繁 Rebalance**：设置合理的 `session.timeout.ms`（10s）和 `heartbeat.interval.ms`（3s）

- 如何查看 Consumer Group 的 Lag？（`kafka-consumer-groups --describe`）
- Cooperative Stickiness Rebalance 策略是什么？它比 Eager Rebalance 好在哪里？

### Q8：Kafka 的顺序消息如何保证？为什么只保证单 Partition 内的顺序？你们告警分发中需要全局有序吗？

**考察点**：Kafka 顺序消息保证、分区有序性

**答案要点**：
- **Kafka 顺序保证**：
  - **单 Partition 内有序**：Kafka 的写机制（追加到日志末尾）+ 消费机制（FIFO）天然保证单分区有序
  - **关键配置**：`acks=all`（确保消息被 ISR 全部副本持久化后才返回）+ 幂等 Producer
- **多 Partition 无全局有序**：
  - 不同 Partition 并行消费，无顺序保证
  - 消费者乱序处理消息
- **保证跨 Partition 有序**：
  - **方案 1**：只用一个 Partition（牺牲吞吐，只适合低 QPS 场景）
  - **方案 2**：自定义分区器，确保同一 key（如 alert_id）一定发到同一 Partition
  - **方案 3**：在消费端按 key 排序后再处理（内存排序）
- **告警分发场景**：
  - **不需要全局有序**：同一条告警的处理不需要全局排队
  - **需要单告警有序**：同一告警的 Enrichment → AI 分析 → Case 创建三步必须有序
  - **实现**：同一告警 ID 的所有处理步骤共用同一 Partition Key（alert_id）

**可能的追问**：
- RocketMQ 的顺序消息是如何实现的？和 Kafka 有什么不同？
- 如果消费端需要严格顺序消费，但吞吐量要求高，如何设计？
- 消费端的 Exactly-once 语义如何保证顺序？

### Q9：Kafka 的 Exactly-once 语义（事务 Producer + 消费者幂等）是如何实现的？实际工程中为了实现 Exactly-once 需要付出什么代价？

**考察点**：Kafka Exactly-once 机制、事务配置

**答案要点**：
- **Exactly-once 实现**：
  - **幂等 Producer**：`enable.idempotence=true`，Producer 带 PID（Producer ID）+ Sequence Number，每个 Partition 只处理递增 Sequence，重复消息被 Broker 过滤
  - **事务 Producer**：`transactional.id=unique_id`，开启事务后，所有 send 操作在一个事务内，Broker 通过事务日志保证原子性
  - **消费者幂等**：消费端需要自己保证幂等（外部系统如 DB 做唯一约束），Kafka 事务不保证消费端幂等
- **完整 Exactly-once 配置**：
  ```python
  producer = KafkaProducer(
      enable_idempotence=True,
      transactional_id='soc-alert-producer-1',  # 事务 Producer
      acks='all',
  )
  producer.init_transactions()
  producer.begin_transaction()
  producer.send('alert-topic', value)
  producer.send('case-topic', value)
  producer.commit_transaction()  # 原子提交
  ```
- **实现代价**：
  1. **性能损失**：事务增加 2-3 倍延迟（两阶段提交协议）
  2. **资源消耗**：事务日志占用额外存储
  3. **复杂度增加**：代码需要处理 `TransactionManager` 和异常回滚
  4. **吞吐下降**：事务 Batch 需要等待更多数据才发送（高延迟低吞吐）
- **实际选择**：大多数场景用 **At-least-once + 消费端幂等**，代价小，效果好

**可能的追问**：
- 幂等 Producer 的 Sequence Number 溢出后会发生什么？
- Kafka 事务和数据库两阶段提交（2PC）有什么异同？
- Exactly-once 在流处理（Kafka Streams）中如何实现？

### Q10：RabbitMQ 的 Exchange 类型（Direct/Fanout/Topic/Headers）和 Routing Key 机制是什么？和 Kafka 的 Topic/Partition 模型相比，RabbitMQ 在消息路由灵活性上有什么优势？

**考察点**：RabbitMQ Exchange 路由、灵活路由

**答案要点**：
- **Exchange 类型**：
  - **Direct**：`Routing Key` 必须精确匹配（如 `alert.high` → 绑定 `alert.high`）
  - **Fanout**：忽略 Routing Key，广播到所有绑定队列（如消息通知广播）
  - **Topic**：支持通配符路由，`alert.*` 匹配 `alert.high`，`alert.#` 匹配 `alert.high.critical`（`#` 匹配多层）
  - **Headers**：基于消息头属性匹配（不常用，性能差于 Topic）
- **RabbitMQ vs Kafka 路由模型**：
  | 维度 | RabbitMQ | Kafka |
  |------|---------|-------|
  | 消息路由 | Exchange → Binding → Queue（灵活） | Topic → Partition（简单） |
  | 消费者选择 | 消费者绑定到 Queue | 消费者绑定到 Consumer Group |
  | 路由复杂度 | 高（多种 Exchange） | 低（仅 Topic 分区） |
  | 消息优先 | 无（Queue 是消费者共享） | 有（Partition 内有序） |
- **RabbitMQ 优势**：路由灵活性高，适合复杂路由场景（如告警分级：HIGH → 立即通知队列，LOW → 批量处理队列）；一个 Exchange 可绑定多个 Queue，每个按不同规则消费

**可能的追问**：
- RabbitMQ 的死信队列（DLX）和 Kafka 的 Dead Letter Topic 有什么异同？

## 8.3 微服务与架构

### Q11：微服务拆分粒度如何把握？你们的 SOC 平台按什么维度拆分服务？遇到过哪些"拆分过度"或"拆分不足"的问题？

**考察点**：微服务拆分原则、Bounded Context

**答案要点**：
- **拆分维度（按业务能力 vs 按团队）**：
  - **按业务能力（Domain）**：告警域、Case 域、工单域、Enrichment 域
  - **按团队边界（Team Topologies）**：Two-Pizza Team 对应一个或多个微服务
- **SOC 平台拆分**：
  ```
  [API Gateway / BFF]     # 前端聚合层
  ├── Alert Service       # 告警 CRUD、状态流转
  ├── Enrichment Service  # 告警富化
  ├── AI Agent Service   # AI 分析
  ├── Case Service       # Case 管理
  ├── Ticket Service     # 工单管理
  └── Notification Svc   # 通知服务
  [共享中间件]
  ├── Kafka（告警事件总线）
  ├── Nacos（服务注册中心）
  └── Redis（缓存/锁）
  ```
- **拆分过度的教训**：
  - **跨服务调用链过长**：一次告警处理需要调用 5+ 服务，P99 延迟从 50ms 变成 500ms
  - **事务一致性复杂**：告警创建 + Case 创建跨服务，最终一致性实现成本高
  - **运维成本高**：10 个服务 = 10 个独立部署单元，CI/CD 配置翻倍
- **拆分不足的教训**：
  - Alert 和 Enrichment 早期耦合，导致 Enrichment 逻辑变更影响 Alert 服务稳定性

**可能的追问**：
- 微服务的"泥球"（Big Ball of Mud）反模式是什么？
- 如何判断一个服务是否应该拆分或合并？（高内聚低耦合、团队边界、独立部署频率）

### Q12：微服务之间如何做分布式事务？TCC、Saga、可靠消息最终一致性各自适合什么场景？你们平台中跨服务的告警-案例关联用什么方案？

**考察点**：分布式事务模式、Saga 编排

**答案要点**：
- **三种分布式事务方案**：
  1. **TCC（Try-Confirm-Cancel）**：
     - Try：预留资源（冻结库存/预扣余额）
     - Confirm：确认执行（真正扣库存）
     - Cancel：回滚（释放冻结资源）
     - 适用：强一致性、高价值事务（转账、库存扣减）
     - 缺点：业务代码侵入性高，每个操作都要实现 Try/Confirm/Cancel
  2. **Saga 模式**：
     - 编排：各服务按顺序执行本地事务，任一步失败则执行逆向补偿事务
     - 适用：长流程、业务逻辑复杂的服务链（订单 → 支付 → 物流）
     - 优点：无锁，性能好；补偿事务天然幂等
  3. **可靠消息最终一致性**：
     - 本地消息表 + 定时轮询，或 RocketMQ 事务消息
     - 适用：允许延迟，对实时性要求不高的跨服务通知
- **告警-案例关联方案（SOC 平台）**：
  - **事件驱动 + 本地消息表**：
    1. Alert Service 创建告警 → 写入本地消息表（status=PENDING）→ 提交本地事务
    2. 定时任务轮询消息表 → 发送 Kafka 事件
    3. Case Service 收到事件 → 创建 Case → 发送 ACK
    4. Alert Service 收到 ACK → 更新消息表 status=COMPLETED
    5. 失败则重试，最终失败则告警
  - **理由**：不需要强一致性，最终一致即可；比 TCC 实现简单，不侵入业务代码

**可能的追问**：
- Saga 模式和 TCC 模式的根本区别是什么？
- Saga 的补偿事务如何保证幂等性？
- 可靠消息方案中，本地消息表如何避免重复发送？

### Q13：服务网格（Service Mesh）是什么？Istio 的 Sidecar 注入和 Envoy 代理解决了什么问题？在 TCE 迁移过程中有没有涉及 K8s NetworkPolicy 或 Istio 配置？

**考察点**：Service Mesh 架构、Istio

**答案要点**：
- **Service Mesh 定义**：
  - 将**跨服务通信的网络能力**从应用代码下沉到基础设施层（Sidecar Proxy）
  - 应用层不需要感知服务发现/负载均衡/熔断/重试等网络逻辑
- **Istio 架构**：
  - **Sidecar 注入**：每个 Pod 注入一个 Envoy Sidecar 容器，拦截所有进出流量
  - **Envoy 代理**：处理服务发现、负载均衡、熔断、重试、超时、mTLS 加密
  - **Control Plane（Control Plane）**：Pilot（配置分发）、Citadel（证书管理）、Galley（配置验证）
- **解决的问题**：
  1. **mTLS 双向认证**：所有服务间通信自动加密，无需应用层处理证书
  2. **流量管理**：A/B 测试、金丝雀发布、流量镜像（不影响主流量的情况下测试新版本）
  3. **可观测性**：自动收集所有请求的 metrics、logs、traces（Prometheus + Jaeger）
  4. **安全策略**：NetworkPolicy 定义服务间访问权限
- **TCE 迁移中的配置**：
  - K8s `NetworkPolicy`：限定告警服务只能访问 Case 服务和 Kafka，不可访问外网
  - Istio `DestinationRule`：配置告警服务的熔断规则（如连续 5 次失败则熔断 30s）
  - Istio `VirtualService`：配置告警服务的流量权重（v1=90%, v2=10%）

**可能的追问**：
- Sidecar 注入带来的额外延迟是多少？（Envoy Sidecar 通常增加 1-3ms）
- Istio 和 Linkerd 哪个更轻量？各有何优劣？
- 什么是 Ambient Mesh？它和 Sidecar 模式有什么区别？

### Q14：API 网关（Koa/Nginx）与微服务网关（Spring Cloud Gateway/Kong）的职责有什么不同？你们 SOC 平台的网关层做了哪些跨切面的事情？

**考察点**：API 网关职责、微服务网关对比

**答案要点**：
- **API 网关 vs 微服务网关**：
  | 维度 | API 网关（Koa/Nginx） | 微服务网关（Spring Cloud Gateway/Kong） |
  |------|---------------------|--------------------------------------|
  | 职责 | 协议转换、聚合、认证 | 服务路由、限流、熔断 |
  | 位置 | 最外层（用户侧） | 服务集群入口 |
  | 关注点 | 前端友好、BFF | 后端服务治理 |
  | 典型功能 | 请求聚合、超时配置、响应缓存 | 动态路由、服务发现、灰度发布 |
- **SOC 平台 Koa 网关的跨切面职责**：
  1. **认证鉴权**：JWT 解析 + Token 校验 + 权限路由（RBAC）
  2. **限流**：令牌桶限流（Redis + Lua），按用户/接口级别限流
  3. **日志审计**：记录所有请求的 path/user/response_time/code
  4. **请求转换**：后端微服务接口参数适配（入参转换/出参转换）
  5. **熔断降级**：下游服务超时/错误率超阈值时返回降级数据
  6. **CORS 跨域**：统一处理 OPTIONS 预检请求
  7. **链路追踪**：生成 trace_id，透传到后端服务
- **与 Kong 的关系**：Kong 做**全局入口网关**（外部流量），Koa 网关做**BFF 聚合网关**（内部流量聚合），形成两层网关架构

- Kong 的插件机制（Plugin）是什么？常用插件有哪些？
- API 网关和 Ingress Controller 有什么区别？
