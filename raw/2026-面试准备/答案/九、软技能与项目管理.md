## Q1：产品思维场景题：如果产品经理提出的需求会严重影响系统性能，但业务方催得急，你会怎么做？技术方案和业务优先级冲突时，如何推动多方达成一致？

**考察点**：技术判断力、跨团队沟通、向上管理

**答案要点**：
- **技术判断优先**：
  1. **量化影响**：不要泛泛说"性能差"，要用数据说话：给出具体数字（如"预计 QPS 从 5000 降到 500，延迟从 20ms 升到 2000ms"）
  2. **找出瓶颈**：分析需求对性能影响的具体路径（数据库/缓存/算法/网络）
  3. **提供可选方案**：给出 A/B 方案——满足性能要求的替代方案，或分阶段实施方案
- **推动多方达成一致**：
  1. **沟通框架**：
     - 明确立场：不是"拒绝需求"，而是"如何安全地实现需求"
     - 量化风险：让业务方理解性能问题带来的实际损失（如 SLA 违约、用户流失）
     - 寻求共识：是否存在性能可接受的简化版本？
  2. **TL/总监对齐**：在产品经理层面无法达成一致时，升级到技术 TL 和产品负责人共同决策
  3. **分阶段交付**：第一阶段做性能友好版，第二阶段优化后再加花哨功能
  4. **写 ADR（Architecture Decision Record）**：记录技术决策和原因，形成团队知识
- **实战经验**：在 SOC 平台告警批量处理需求中，通过分批处理 + 消息队列削峰，将批量导入对在线服务的影响从 80% 降到 5%，同时满足了业务需求

**可能的追问**：
- 如何向上管理（manage up）技术决策？
- 当技术负责人和业务方的判断不一致时，如何处理？
- 如何用数据说服非技术背景的产品经理接受技术约束？

## Q2：工作评估：假设产品给了一个"在 SOC 平台中添加告警关联分析功能"的需求，你会如何拆解工作量？Planning Poker 中你是怎么给用户故事评点的？

**考察点**：工作量评估、敏捷估算方法

**答案要点**：
- **需求拆解**：
  ```
  [告警关联分析功能]
  ├── 1. 数据准备
  │   ├── 数据模型设计（Alert → Artifact → Relation）
  │   ├── MySQL 表新增/修改
  │   └── 告警历史数据迁移脚本
  ├── 2. 后端 API
  │   ├── 关联分析接口设计
  │   ├── 关联算法实现（基于时间/IP/行为相似度）
  │   └── API 接入 BFF 层
  ├── 3. 前端交互
  │   ├── 关联关系图可视化（关系图谱）
  │   ├── 关联列表展示
  │   └── 交互筛选功能
  ├── 4. 测试
  │   ├── 单元测试（覆盖率 > 80%）
  │   ├── 集成测试（API 测试）
  │   └── E2E 测试（用户操作流程）
  └── 5. 运维
      ├── 性能测试（关联分析 P99 < 2s）
      └── 上线和监控
  ```
- **Planning Poker 评点方法**：
  - 使用斐波那契数列：`1 / 2 / 3 / 5 / 8 / 13 / 21`（故事点，反映相对复杂度而非绝对时间）
  - 评点原则：
    1. **相对估算**：拿已知复杂度的功能做参照（如"添加告警标签功能 = 5 点"，关联分析 = 13 点因为更复杂）
    2. **去除不确定性**：估不准的先做 Spikes（调研任务）降低不确定性后再估
    3. **团队共识**：不同估值的讨论（分歧=知识点）是最有价值的部分
  - SOC 平台案例：关联分析功能估为 **13 个故事点**（约 2 周工作量）

**可能的追问**：
- 故事点（Story Point）和人天（Man-day）如何换算？
- 如何处理估算和实际偏差较大的情况？
- 什么是"计划扑克"中的"大数字"（Big）策略？

## Q3：线上故障排查思路：假设 SOC 平台的告警突然全部延迟 5 分钟才推出来，你的排查路径是什么？

**考察点**：故障排查方法论、系统性思考

**答案要点**：
- **分层排查路径**：
  1. **告警层（Alert）→ 告警延迟监控系统告警**
     - 查看监控大盘：告警产生时间戳 vs 推送时间戳，是否所有渠道都延迟？
  2. **消息队列层（Kafka）**
     - `kafka-consumer-groups --describe`：查看 Consumer Group 的 lag 是否积压
     - 如果 lag 很大 → 消费者处理能力不足（消费者崩溃/处理过慢）
     - 如果 lag 正常 → 消息生产端问题
  3. **消费者层（Consumer Service）**
     - 查看 Consumer 日志：是否有 ERROR/Exception？
     - 查看 Consumer 延迟监控：`consumer_lag_seconds` 指标
     - 查看 Consumer 资源使用率：CPU/内存/GPU（AI 推理）是否被打满
  4. **AI 推理层（LLM）**
     - 如果告警需要 AI 研判：查看 LLM 响应时间（正常 < 3s）
     - 查看 LLM API 的 P99 延迟：是否超过 SLA
     - 如果是本地模型：查看 GPU 显存是否不足
  5. **网络层**
     - 查看 VPC 流量监控：是否有网络抖动
     - 查看 Kafka broker 和 Consumer 的网络延迟
  6. **数据库层**
     - Enrichment 查询（MySQL）是否慢查询阻塞
     - 查看 MySQL 慢查询日志
- **快速止血步骤**：
  1. 先重启消费者（如果进程崩溃）
  2. 降级 AI 研判（切到规则引擎）
  3. 扩容消费者实例
- **复盘方法**：故障恢复后记录 Timeline（时间线）、Root Cause（根因）、Impact（影响范围）、Action Items（后续行动）

**可能的追问**：
- 如何在故障发生时快速定位根因？（监控 + 日志 + Trace）
- 告警平台自身故障时如何监控？（元监控）
- 如何设计 SLO/SLA 并在故障时自动告警？

## Q4：技术选型：你在选择 Kafka 而非 RabbitMQ/Redis Stream 时，主要考量哪些因素？未来如果出现更合适的中间件，你如何设计技术债务的偿还计划？

**考察点**：技术选型方法论、技术债务管理

**答案要点**：
- **Kafka 选型考量**：
  1. **吞吐量**：SOC 平台告警量高峰可达 5000 QPS，Kafka 的百万级吞吐是必须
  2. **消息持久化**：告警是重要数据，不能丢失，需要多副本 + 持久化
  3. **消息重消费**：告警处理失败后需要重新处理，Kafka 的消息保留机制天然支持
  4. **生态工具**：Kafka Connect 对接 ES/MySQL、Kafka Streams 做实时统计
- **技术债务偿还计划**：
  1. **识别技术债**：定期（每季度）做技术债务评审，记录到技术债 Backlog
  2. **评估影响**：评估每项技术债的"利息"（维护成本/风险/性能损失）
  3. **偿还策略**：
     - **立即偿还**：影响生产稳定的高息债务（如单点故障）
     - **计划偿还**：放入 Sprint Backlog，按优先级排入迭代
     - **接受债务**：ROI 不足偿还成本的债务，选择接受并监控
  4. **偿还路径**：
     - 如果 Kafka 出现更好的替代品：先在非核心功能上试点，再评估是否全量迁移
     - 抽象消息队列接口层（Producer/Consumer 接口），降低迁移成本
     - 用 Feature Flag 控制新旧系统并行，降低切换风险
  5. **持续监控**：技术债偿还后需要持续监控，防止债务复发

**可能的追问**：
- 什么是技术债的"利息"？如何量化？
- 如何说服团队投入时间偿还技术债？（用数据说话：某技术债每月消耗 X 人天）
- 技术债偿还如何在业务迭代中平衡？（每 Sprint 留 20% 容量做技术改进）

## Q5：版本管理与团队协作：Scrum 和 Kanban 的核心区别是什么？你们腾讯云团队用的是哪种方式？你如何平衡独立客户定制分支与主线版本同步的矛盾？

**考察点**：敏捷方法论、Git 分支策略

**答案要点**：
- **Scrum vs Kanban**：
  | 维度 | Scrum | Kanban |
  |------|-------|--------|
  | 迭代周期 | 固定 Sprint（1-4周） | 连续流动（无固定周期） |
  | 团队角色 | Scrum Master + PO | 无强制角色 |
  | 计划方式 | Sprint Planning 批量估算 | WIP Limit 控制流动 |
  | 变更处理 | Sprint 期间不变更 | 可随时插入紧急变更 |
  | 适用场景 | 稳定需求、预测性项目 | 持续交付、运维场景 |
- **腾讯云团队实践**：采用**Scrum**，2 周 Sprint，配合 Git Flow 分支策略
- **客户定制分支与主线同步**：
  1. **问题**：客户定制分支越来越多，主线合并困难，定制代码无法复用
  2. **解决策略**：
     - **抽象定制点**：用 Feature Flag / 配置化将定制项提取出来，避免代码分叉
     - **特性分支同步**：定制分支定期从主线 `rebase`（或 `merge`），保持同步
     - **主干开发（Trunk-Based）**：所有开发合入主线，定制通过配置控制（推荐）
     - **版本标签隔离**：客户版本用 Git Tag + CHANGELOG 管理，不维护长期分支
  3. **实践**：
     - 定制项 < 5% 的代码，用配置文件/Feature Flag 替代分支
     - 超过 5% 的定制，评估是否做成平台可配置功能
     - 实在无法合并的定制，维护独立分支，但要求**每季度同步一次主线**

**可能的追问**：
- Git Flow 的 main/develop/feature 分支各自作用是什么？
- 如何防止长期分支演变成"泥球"（Big Ball of Mud）？
- 如何度量团队的开发效率？（velocity / lead time / cycle time）
