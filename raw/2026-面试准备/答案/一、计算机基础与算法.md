## 1.1 网络基础

### Q1：HTTP 和 HTTPS 的区别是什么？HTTPS 的握手过程是怎样的？

**考察点**：HTTPS 加密原理、TLS 握手流程、对称与非对称加密区别

**答案要点**：
- HTTP 明文传输，HTTPS = HTTP + TLS（SSL），端口 443 vs 80
- TLS 1.2/1.3 握手核心流程：
  1. **Client Hello**：客户端支持的 TLS 版本、加密套件列表、随机数 Client Random
  2. **Server Hello**：服务端选定的 TLS 版本和加密套件、发送证书（含公钥）、随机数 Server Random
  3. **证书校验**：客户端用 CA 根证书验证服务端证书链（信任链/有效期/域名）
  4. **Pre-Master Secret**：客户端生成随机数，用服务端公钥加密后发给服务端（ECDHE 则走 key exchange 协议）
  5. **双方生成会话密钥**：Master Secret = PRF(Client Random + Server Random + Pre-Master Secret)，随后用对称加密通信
- 对称加密（通信速度快）vs 非对称加密（密钥交换安全），HTTPS 两者结合

**可能的追问**：
- TLS 1.3 相比 1.2 握手减少到 1-RTT 或 0-RTT，删除了哪些不安全的加密套件？
- 什么是 CA 证书链？中间人攻击如何被证书机制防止？
- ECDHE 密钥交换的原理是什么？为什么前向安全（Forward Secrecy）很重要？

### Q2：TCP 三次握手和四次挥手的流程是什么？为什么三次握手不能改成两次？TIME_WAIT 状态的作用是什么？

**考察点**：TCP 可靠连接建立/断开原理、协议状态机、半连接队列与全连接队列

**答案要点**：
- **三次握手**：
  1. 客户端 → 服务端：SYN=1, seq=x → **SYN_SENT**
  2. 服务端 → 客户端：SYN=1, ACK=1, seq=y, ack=x+1 → **SYN_RCVD**
  3. 客户端 → 服务端：ACK=1, seq=x+1, ack=y+1 → **ESTABLISHED**（服务端收到后也变 ESTABLISHED）
- **四次挥手**：
  1. 主动方 → 被动方：FIN=1, seq=u → **FIN_WAIT_1**
  2. 被动方 → 主动方：ACK=1, ack=u+1 → **FIN_WAIT_2**（此时被动方还能发送数据）
  3. 被动方 → 主动方：FIN=1, seq=w → **LAST_ACK**
  4. 主动方 → 被动方：ACK=1, ack=w+1 → **TIME_WAIT**（等 2MSL 后 → CLOSED），被动方收到后 → CLOSED
- **为什么三次不能改成两次**：两次握手无法防止历史 SYN 攻击（旧的 SYN 先到达导致服务端建立冗余连接），也无法让双方确认对方的接收能力和序列号同步
- **TIME_WAIT 作用**：
  1. 确保被动方的 FIN 到达（最后一次 ACK 不丢）
  2. 让旧连接的重复数据包在网络中自然消失，避免影响新连接
  3. 2MSL（通常 60s）保证了网络中最长报文的生命周期

**可能的追问**：
- TCP 半连接队列（SYN Queue）和全连接队列（Accept Queue）是什么？accept() 调用不及时会怎样？
- 四次挥手为什么不能合并？什么情况下可以合并为三次？
- 服务器主动关闭连接会产生大量 TIME_WAIT，如何优化？（socket 选项 SO_REUSEADDR、调整 MSL）

### Q3：说说你对 DNS 解析流程的理解，从浏览器输入 URL 到页面渲染完成经历了哪些步骤？

**考察点**：DNS 分层解析机制、浏览器缓存、HTTP 请求全链路

**答案要点**：
- **DNS 解析流程**：
  1. 浏览器 DNS 缓存 → 2. 系统 hosts 文件 → 3. 系统 DNS 缓存 → 4. 本地 DNS 服务器（递归查询）
  5. 本地 DNS → 根域名服务器 → .com TLD → 权威 DNS 服务器，拿到 IP 后逐级缓存
- **全链路**：
  1. URL 解析（协议、域名、端口、路径）
  2. DNS 解析（获取 IP）
  3. TCP 三次握手（HTTP/HTTPS 端口建立）
  4. TLS 握手（HTTPS）
  5. 浏览器发送 HTTP 请求
  6. CDN 边缘节点回源（如命中缓存则返回）
  7. 后端服务处理请求，执行业务逻辑
  8. HTTP 响应返回（可能 chunked 分块传输）
  9. 浏览器解析 HTML → 构建 DOM 树
  10. 遇到 JS 阻塞渲染 → JS 下载/执行 → CSSOM 构建
  11. DOM + CSSOM → Render Tree → Layout → Paint

**可能的追问**：
- DNS 记录类型 A、AAAA、CNAME、MX、TXT 各自用途？
- DNS 污染和 DNS 劫持有什么区别？DoH/DoT 解决什么问题？

### Q4：WebSocket 和 HTTP 长轮询有什么区别？在什么场景下你会选择 WebSocket？

**考察点**：实时通信机制、全双工 vs 半双工、HTTP/2 Server Push 对比

**答案要点**：
- **HTTP 长轮询**：客户端发请求，服务端有数据才返回，否则超时后客户端再发请求。开销大（每次请求有 HTTP 头）、延迟高（最多一个轮询周期）
- **WebSocket**：基于 HTTP Upgrade 握手后升级为 TCP 长连接，全双工通信，服务端可主动推送，协议开销极小（帧头 2-14 字节）
- **选择 WebSocket 的场景**：聊天/消息推送、协同编辑、实时监控大屏、游戏后端、SOC 告警实时推送
- **不适合的场景**：静态内容/CDN、简单 REST API、CDN 缓存频繁的内容

**可能的追问**：
- WebSocket 如何实现断线重连和心跳保活？
- WebSocket 和 SSE（Server-Sent Events）的区别？
- HTTP/2 的 Server Push 能替代 WebSocket 吗？各有什么局限？

### Q5：你在 SOC 平台中用到了 Nacos 做服务注册与发现，简述 Nacos 的工作原理，它与 Eureka、Consul 有什么区别？

**考察点**：服务注册中心原理、AP vs CP、选型对比

**答案要点**：
- **Nacos 工作原理**：
  - **注册**：服务启动时向 Nacos Server 发送注册请求（IP、Port、ServiceName、Metadata），心跳保活（默认 5s）
  - **发现**：服务消费者从 Nacos 拉取服务实例列表，支持配置订阅，实例变更时推送
  - **CP + AP 双模式**：通过 `CP` 或 `AP` 模式切换，默认 AP（临时节点）
- **与 Eureka 对比**：
  - Eureka（AP，只支持临时节点）、Nacos 支持 AP+CP、更丰富的配置管理功能
  - Eureka 2.x 已停止开源，Nacos 活跃度高
- **与 Consul 对比**：
  - Consul 默认 CP，支持 HTTP/DNS 两种服务发现方式，成熟度更高
  - Nacos 配置管理 + 服务发现一体化，对国内用户更友好

**可能的追问**：
- Nacos 的临时节点和持久节点在故障恢复时表现有什么不同？
- Nacos 的 namespace、group、clusterName 三层隔离分别用于什么场景？
- 服务注册中心挂了，新服务还能互相调用吗？（走本地缓存/本地 LB）

### Q6：Nacos 的配置管理（Config）是如何工作的？`@RefreshScope` 的原理是什么？为什么配置变更后不需要重启服务？Nacos 的长轮询（Long Polling）机制是如何实现的？

**考察点**：配置中心原理、Spring Cloud 动态刷新机制、配置一致性

**答案要点**：
- **配置管理流程**：应用启动 → 从 Nacos 获取配置（/nacos/v1/cs/configs?dataId=xxx&group=xxx）→ 存入本地内存 + 本地缓存文件
- **`@RefreshScope` 原理**：Bean 被 CachingConfigurationImportSelector 包装，配置变更时 Spring Cloud 发起 Context Refresh 事件，销毁旧 Bean 并重新创建。底层依赖 `RefreshScope` 类的 `refreshAll()` 和 `get(name)` 方法
- **不需要重启原因**：Bean 是重新创建而非修改旧实例，重新创建时从 Nacos 读取最新配置值
- **长轮询机制**：
  1. 客户端发送 HTTP 请求（带超时时间，如 30s）
  2. 服务端收到请求后，先查本地配置是否有变更（有则立即返回 200 + 变更内容）
  3. 无变更则 hold 住请求，直到有变更或超时
  4. 超时后返回 304，客户端立即发起下一轮长轮询
  5. 实际实现中 Nacos 用 `async-refresh` 模式：有变更时提前响应，无变更时等待超时

**可能的追问**：
- 如果同时有多台 Nacos Server 节点，配置一致性怎么保证？
- `@RefreshScope` 的副作用是什么？（频繁刷新导致内存抖动，Bean 状态丢失）
- Nacos 配置的优先级规则是什么？（dataId > group > namespace > cluster）

### Q7：Nacos 的 Naming 服务中，注册中心的 `ephemeral`（临时/持久）节点有什么区别？服务下线时 Nacos 是如何快速感知到心跳丢失的？临时节点和永久节点在 AP/CP 模式下的表现是什么？

**考察点**：服务注册中心一致性模型、心跳机制、故障检测算法

**答案要点**：
- **临时节点（ephemeral=true）**：
  - 依赖客户端心跳保活，心跳丢失后 Nacos 直接删除节点
  - 无持久化，重启 Nacos 后数据丢失
  - 遵循 AP 模式，节点数据一致性弱但可用性高
- **持久节点（ephemeral=false）**：
  - Nacos 主动持久化，无心跳超时概念
  - 下线需要主动注销（DELETE /nacos/v1/ns/instance）
  - 遵循 CP 模式，强一致性但注册速度较慢
- **心跳丢失感知**：
  1. Nacos 服务端维护心跳超时检查（默认 15s 未收到心跳则标记不健康）
  2. 连续 3 次心跳失败（约 45s）则剔除临时节点
  3. 实际使用 `心跳检测定时任务`（每次扫描所有实例的 lastBeat）
  4. 临时节点被剔除后，Nacos 推送变更事件给订阅的消费者

**可能的追问**：
- Nacos CP 模式下使用 Raft 协议选主，Raft 的日志复制过程是怎样的？
- 临时节点被大量剔除时会产生"惊群效应"，如何优化？
- 服务实例数为 0 时，Nacos 如何处理注册请求？（CP 模式下可以创建保护实例）

### Q8：UDP 和 TCP 的区别？TCP 的可靠传输是如何实现的？

**考察点**：传输层协议区别、可靠传输机制、TCP 头部字段

**答案要点**：
- **核心区别**：
  - UDP：无连接（不需要握手）、不可靠（无确认无重传）、面向报文（不拆分不合并）、首部开销小（8 字节 vs 20+ 字节）
  - TCP：面向连接、可靠传输、字节流、拥塞控制、流量控制
- **可靠传输五大机制**：
  1. **序号与确认应答**：每个字节都有序号（Ack 确认期望收到的下一个序号）
  2. **超时重传**：RTT 动态计算（加权平均），超时未收到 Ack 则重发
  3. **流量控制**：滑动窗口机制，接收方通过 Win 字段告知剩余缓冲区大小，防止发送方淹没接收方
  4. **拥塞控制**：慢启动（指数增长）→ 拥塞避免（加法增长）→ 快速重传/快速恢复（三次重复 Ack 触发），cwnd 动态调整
  5. **校验和**：TCP 头+数据校验，发现错误直接丢弃（不修复）

**可能的追问**：
- 什么是快重传（Fast Retransmit）？为什么要等三次重复 Ack？
- TCP 的 RTO（Retransmission Timeout）如何动态计算？（Jacobson 算法）
- UDP 在QUIC协议中如何实现可靠传输？（QUIC = UDP + 重传 + 流量控制 + 拥塞控制 + 加密）

### Q9：HTTP/1.0、HTTP/1.1、HTTP/2、HTTP/3 各自有什么重大改进？HTTP/2 和 HTTP/3 如何解决队头阻塞问题？

**考察点**：HTTP 协议演进历史、多路复用、队头阻塞

**答案要点**：
| 版本 | 核心改进 |
|------|---------|
| HTTP/1.0 | 短连接（每个请求单独 TCP 连接） |
| HTTP/1.1 | 持久连接（Keep-Alive）、管道化（Pipelining）、Host 头（虚拟主机） |
| HTTP/2 | **多路复用**（Stream 复用同一连接）、Header 压缩（HPACK）、Server Push、流量控制 |
| HTTP/3 | **QUIC**（基于 UDP）、0-RTT/1-RTT 握手、解决 TCP 队头阻塞、连接迁移（Connection Migration） |
- **HTTP/2 队头阻塞**：虽然解决了 HTTP 层面的队头阻塞，但 TCP 层的队头阻塞依然存在——一个 TCP 报文丢包，后续所有 Stream 数据都要等重传
- **HTTP/3 队头阻塞解决**：QUIC 基于 UDP，每个 Stream 独立独立流量控制，单个 Stream 丢包只阻塞该 Stream，不影响其他 Stream

**可能的追问**：
- HTTP/2 的 HPACK 头压缩原理是什么？（静态表、动态表、Huffman 编码）
- QUIC 的 Connection ID 机制如何实现连接迁移？
- ALPN 在 TLS 握手中如何协商 HTTP 版本？

### Q10：CDN 的工作原理？回源率和缓存命中率分别指什么？什么样的内容适合放 CDN？

**考察点**：CDN 缓存机制、性能指标、资源特性

**答案要点**：
- **CDN 工作原理**：用户请求 → DNS 调度到最近边缘节点（Anycast + 负载均衡）→ 命中缓存则返回，miss 则回源拉取 → 缓存到边缘节点 → 返回给用户
- **关键指标**：
  - **缓存命中率（Hit Rate）** = 缓存命中请求 / 总请求。越高越好，直接影响响应速度和源站压力
  - **回源率** = 回源请求 / 总请求。与命中率互补，越低越好
- **适合 CDN 的内容**：静态资源（图片/CSS/JS/字体/视频）、不经常变化的内容、公共 API 响应（可接受一定延迟）
- **不适合的场景**：个性化/用户相关数据、强一致性要求（如库存、金额）、频繁变化的接口

**可能的追问**：
- CDN 缓存失效策略（TTL 配置）和强制刷新如何配合？
- CDN 如何防止缓存被恶意刷空？（鉴权 URL、IP 黑名单）
- 边缘计算（Edge Computing）和 CDN 的关系是什么？

### Q11：什么是 ARP 协议？ARP 欺骗攻击的原理是什么？有什么防范手段？

**考察点**：局域网通信原理、ARP 缓存投毒、中间人攻击

**答案要点**：
- **ARP 作用**：在同一个局域网内，通过 IP 地址解析出对应的 MAC 地址（IP → MAC 的广播请求 + 单播响应）
- **ARP 欺骗原理**：
  1. 攻击者发送伪造的 ARP Reply（自称是网关 IP，对应自己的 MAC）
  2. 受害者的 ARP 缓存表被更新为攻击者的 MAC
  3. 受害者流量先到攻击者（中间人），攻击者可窃听、篡改
- **防范手段**：
  - 静态 ARP 绑定（`arp -s`）
  - 交换机端口安全（DHCP Snooping + ARP Inspection）
  - IDS/IPS 检测 ARP 异常
  - 加密通信（VPN / TLS）使中间人无法解密

**可能的追问**：
- ARP 广播风暴是如何产生的？对交换机有什么影响？
- 为什么 ARP 缓存需要超时机制？TTL 设多长合适？

### Q13：进程、线程、协程的区别是什么？Node.js 的事件循环属于哪一种？

**考察点**：进程/线程/协程模型、并发 vs 并行、Node.js 运行时架构

**答案要点**：
- **进程**：资源分配基本单位，有独立地址空间（代码/数据/堆/栈/文件描述符），隔离性强，切换开销大（TCB 切换、页表切换、缓存失效）
- **线程**：CPU 调度基本单位，同一进程内共享地址空间（堆、全局变量），切换开销小，存在竞态条件
- **协程**（用户态线程）：由程序自己控制切换（非操作系统），无需进入内核，开销极小（如 goroutine、Python asyncio），协作式调度（yield/await），一个线程可挂起多个协程
- **Node.js 事件循环**：属于**单线程 + 事件驱动 + 回调队列**模型，不属于传统进程/线程/协程。其本质是一个基于 libuv 的单线程循环，不断从队列中取回调执行，适合 IO 密集型。CPU 密集型任务会阻塞事件循环。
- **Node.js 多线程补充**：Worker Threads（独立 V8 实例）、libuv 线程池（处理 fs/DNS/crypto 等异步操作）

### Q14：什么是死锁？产生死锁的四个必要条件是什么？如何预防？

**考察点**：死锁概念、竞态条件与同步、操作系统资源管理

**答案要点**：
- **死锁定义**：两个或多个进程/线程相互持有对方需要的资源，且不主动释放，形成循环等待
- **四个必要条件（COIPA）**：
  1. **互斥条件**：资源同一时刻只能被一个线程持有
  2. **持有并等待**：线程持有资源同时请求其他资源
  3. **不可抢占**：已分配资源不能被强制剥夺
  4. **循环等待**：形成线程间的循环等待链
- **预防措施**：
  - 破坏持有并等待：一次性申请所有所需资源
  - 破坏循环等待：按固定顺序申请资源（如哲学家就餐问题统一编号）
  - 破坏不可抢占：允许抢占
  - 破坏互斥：尽量少用互斥锁，或用读写锁等替代
- **避免（银行家算法）**：动态检测安全状态，安全则分配
- **检测与恢复**：定期检测有向图环路，强制回滚或 kill 进程

### Q17：程序运行时的内存布局（代码区/全局区/堆/栈）是什么样的？malloc/free 和 new/delete 在堆上的分配原理是什么？

**考察点**：进程内存布局、动态内存管理、堆内存分配算法

**答案要点**：
- **内存布局（从低地址到高地址）**：
  1. **代码区（.text）**：只读，存放机器指令
  2. **只读常量区（.rodata）**：常量（const、字符串字面量）
  3. **全局/静态区（.data/.bss）**：全局变量、static 变量（bss=未初始化=0）
  4. **堆（Heap）**：动态分配，向上增长（低地址→高地址），手动管理
  5. **内存映射区（mmap）**：动态库、匿名映射
  6. **栈（Stack）**：函数调用、局部变量，向下增长（高地址→低地址），自动管理
  7. **内核空间**：操作系统内核
- **malloc/free 原理**：
  - 小块内存（<128KB）：brk() 调整堆顶，系统调用次数多则用 mmap() 分配
  - 内部维护空闲链表（free list）、位图（bitmap）或边界标签（buddy system）
  - 分配时先查空闲链表，找合适大小块，找不到则向 OS 申请更多页
  - free 并非真正归还 OS，而是加入空闲链表供下次分配使用
- **new/delete**：底层调用 malloc/free，但额外调用构造函数/析函数

### Q18：什么是虚拟内存？为什么需要虚拟内存？Linux 的 swap 机制是什么？

**考察点**：虚拟内存原理、分页/分段、内存置换策略

**答案要点**：
- **虚拟内存**：每个进程有独立的虚拟地址空间（32位=4GB，64位更广），虚拟地址通过 MMU（内存管理单元）+ 页表映射到物理地址
- **为什么需要虚拟内存**：
  1. **进程隔离**：防止进程间越界访问
  2. **内存抽象**：程序可以使用连续虚拟地址，无需关心物理碎片
  3. **内存超售**：让程序使用大于物理内存的空间（物理内存不够时换出到磁盘）
  4. **按需分配**：Lazy loading，只在真正访问时分配物理页（Page Fault 触发分配）
- **swap 机制**：物理内存不足时，把不活跃的页换出到 swap 分区（磁盘），需要时再换入。swap 使用率过高（>60%）会严重影响性能，称为 swap thrashing（内存抖动）
- **Linux 相关配置**：`vm.swappiness`（倾向于 swap 的程度）、`vm.overcommit_memory`（内存超额分配策略）

### Q21：CPU 上下文切换是什么？频繁的上下文切换会带来什么性能问题？在容器化环境中如何控制？

**考察点**：上下文切换机制、性能损耗、容器资源隔离

**答案要点**：
- **上下文切换**：CPU 从一个进程/线程切换到另一个时，保存当前任务状态（寄存器、程序计数器、栈指针）到 PCB/TCB，加载新任务状态。包含：**保存现场 → 切换内核数据结构 → 恢复现场**
- **性能问题**：
  1. 每次切换有固定开销（~1-10μs），大量切换时 CPU 时间浪费在内核代码而非业务代码
  2. 切换导致 CPU 缓存（Cache）失效（Cache Miss 开销远大于切换本身）
  3. 频繁切换可能触发 Thrashing（抖动），吞吐量急剧下降
  4. 指标：`cs`（context switch）数可从 `vmstat 1` 查看
- **容器化控制**：
  - CPU shares / cpu.cfs_quota_us 限制 CPU 使用
  - `--cpus` 指定 CPU 核数（Kubernetes `resources.limits.cpu`）
  - `cpuset.cpus` 绑核避免跨核切换（减少 Cache Miss）
  - 容器级别设置 `schedtune`（Android/Cgroup）或 cgroup v2 资源控制

**可能的追问**：
- 为什么说协程切换比线程切换开销小很多？（用户态切换，无需进入内核）
- 什么工具可以分析上下文切换的具体原因？（perf sched）

### Q23：HashMap 的底层实现原理？哈希冲突有哪些解决方式？

**考察点**：哈希表原理、冲突解决、性能分析

**答案要点**：
- **底层结构（JDK 1.8+）**：数组 + 链表 + 红黑树。Node[] table（散列表），每个桶在链表长度 > 8 且数组长度 >= 64 时转为红黑树
- **插入流程**：
  1. 计算 key 的 hash（扰动函数：`hash = (key == null) ? 0 : (h = key.hashCode()) ^ (h >>> 16)`）
  2. `(n - 1) & hash` 计算数组下标
  3. 如果桶为空，直接插入；否则遍历链表/红黑树
  4. 发现相同 key 则覆盖 value
  5. 插入后 size++，若超过 threshold（capacity * loadFactor），则 resize 扩容（2 倍）
- **哈希冲突解决方式**：
  1. **开放寻址法**：线性探测（hash+1）、二次探测（hash+i²）、双重哈希
  2. **链地址法（拉链法）**：每个桶存链表/红黑树（Java HashMap / Python dict）
  3. **再哈希法**：用第二个哈希函数计算下一个位置
  4. **建立公共溢出区**：冲突的元素放入溢出表
- **为什么用红黑树而非 AVL？**：红黑树是弱平衡，插入/删除开销（旋转次数少）比 AVL 小，适合频繁修改的场景

**可能的追问**：
- HashMap 的容量为什么必须是 2 的幂次？（`(n-1) & hash` 等价于 `hash % n`，位运算更快）
- HashMap 是线程安全的吗？有哪些线程安全方案？（ConcurrentHashMap、Collections.synchronizedMap）
- ConcurrentHashMap 在 JDK 8 中如何实现分段锁？（CAS + synchronized 锁桶头，不再是分段 Segment）

### Q24：给你一个包含 1 亿条安全日志的数据集，如何快速找出出现频率 Top 10 的攻击 IP？

**考察点**：海量数据处理、哈希思想、堆的应用、MapReduce 思维

**答案要点**：
- **方法一：哈希 + 小顶堆（单机）**
  1. 用哈希分流（哈希 IP 到不同桶，减少单节点内存压力）
  2. 每个桶内用 HashMap 计数：`Map<String, Integer> counter`
  3. 维护大小为 10 的小顶堆（MinHeap），堆顶是当前 Top 10 中频次最低的
  4. 每个 IP 出现时，若 count > 堆顶，则替换堆顶并调整
  5. 时间复杂度 O(n log k)，空间复杂度 O(n)（但实际 IP 数量远小于日志数）
- **方法二：MapReduce（分布式）**
  - Map：每条日志提取 IP，输出 `<IP, 1>`
  - Shuffle：相同 IP 的记录发到同一 Reduce
  - Reduce：求和统计每个 IP 总次数
  - 再对结果排序取 Top 10
- **方法三：近似算法（HyperLogLog / Count-Min Sketch）**：精度换空间，适合无法全存内存的超大数据集

**可能的追问**：

### Q25：B+ 树和 B 树的区别？为什么 MySQL InnoDB 选择 B+ 树做索引？

**考察点**：B+ 树 vs B 树、索引结构、磁盘 IO 优化

**答案要点**：
| 特性 | B 树 | B+ 树 |
|------|------|-------|
| 数据存储 | 所有节点都存数据 | 只有叶子节点存数据，非叶子节点只存索引 |
| 高度 | 更高（每个节点都存数据） | 更矮（可容纳更多索引项） |
| 范围查询 | 需要中序遍历所有路径 | 叶子节点链表，支持顺序访问 |
| 查询稳定性 | 非叶子节点也存数据，查询路径长度不稳定 | 所有查询必须到叶子，查询路径长度一致 |
| 空间利用率 | 较低 | 更高（所有非叶子节点可充满） |
- **InnoDB 选 B+ 树原因**：
  1. **磁盘 IO 优化**：每个节点=一页（16KB），B+ 树更矮，磁盘 IO 次数更少（通常 2-3 次磁盘读取即可查到数据）
  2. **范围查询友好**：叶子节点链表顺序访问，避免回溯
  3. **查询稳定**：必须查到叶子，查询时间可预测
  4. **空间利用率高**：非叶子节点不存数据，可容纳更多分叉
  5. **聚簇索引**：主键索引的叶子节点直接存储行数据，减少回表 IO

### Q26：写一个 LRU 缓存，要求 get 和 put 操作都是 O(1) 时间复杂度。

**考察点**：哈希表 + 双向链表、LRU 原理、LinkedHashMap 应用

**答案要点**：
```python
# 方法一：手写哈希表 + 双向链表
class Node:
    def __init__(self, key=0, val=0):
        self.key, self.val = key, val
        self.prev = self.next = None

class LRUCache:
    def __init__(self, capacity: int):
        self.capacity = capacity
        self.cache = {}  # key -> Node
        # 伪头/伪尾，简化边界处理
        self.head = Node()
        self.tail = Node()
        self.head.next = self.tail
        self.tail.prev = self.head

    def _remove(self, node: Node):
        """从链表当前位置摘下节点"""
        node.prev.next = node.next
        node.next.prev = node.prev

    def _add_to_head(self, node: Node):
        """插入到链表头部（最近使用）"""
        node.next = self.head.next
        node.prev = self.head
        self.head.next.prev = node
        self.head.next = node

    def get(self, key: int) -> int:
        if key not in self.cache:
            return -1
        node = self.cache[key]
        self._remove(node)       # 先摘下
        self._add_to_head(node)  # 再移到头部
        return node.val

    def put(self, key: int, val: int) -> None:
        if key in self.cache:
            self._remove(self.cache[key])
            del self.cache[key]
        node = Node(key, val)
        self.cache[key] = node
        self._add_to_head(node)
        if len(self.cache) > self.capacity:
            # 淘汰尾部（最久未使用）
            lru = self.tail.prev
            self._remove(lru)
            del self.cache[lru.key]
```
- **O(1) 原因**：哈希表 O(1) 查找 + 双向链表 O(1) 插入/删除
- **Python 替代**：`collections.OrderedDict` 底层也是双向链表 + 哈希表，可直接用 `move_to_end()` 实现 LRU

**可能的追问**：
- Redis 的 LRU 近似算法是怎么实现的？（不是精确 LRU，用采样 + 淘汰池）
- LFU（Least Frequently Used）和 LRU 的区别？适合什么场景？

### Q27：链表和数组的区别？单向链表、双向链表、循环链表各自的特点和使用场景？链表逆置怎么做？

**考察点**：线性数据结构、指针操作、算法实现

**答案要点**：
- **数组 vs 链表**：
  - 数组：O(1) 随机访问，O(n) 插入/删除（需移动元素），内存连续（Cache 友好）
  - 链表：O(n) 随机访问，O(1) 插入/删除（改指针），内存不连续（动态大小）
- **单向链表**：每个节点有 next 指针。适合做栈（只操作头部）
- **双向链表**：每个节点有 prev + next 指针。适合做 LRU Cache（需前向后向操作）、浏览器的历史记录
- **循环链表**：尾节点的 next 指向头节点。约瑟夫问题、循环队列
- **链表逆置**（迭代）：
```python
def reverse(head):
    prev = None
    curr = head
    while curr:
        nxt = curr.next
        curr.next = prev
        prev = curr
        curr = nxt
    return prev  # 新头节点
```
- **递归逆置**：从尾部开始逆向建立连接，适合学习但有栈溢出风险（n 层递归深度）

### Q29：什么是二叉树的遍历（前序/中序/后序/层序）？递归和迭代实现分别怎么写？

**考察点**：树遍历算法、递归与迭代转换、栈/队列的应用

**答案要点**：
| 遍历 | 顺序 | 递归实现 |
|------|------|---------|
| 前序 | 根 → 左 → 右 | `visit(node) + preorder(node.left) + preorder(node.right)` |
| 中序 | 左 → 根 → 右 | `inorder(node.left) + visit(node) + inorder(node.right)` |
| 后序 | 左 → 右 → 根 | `postorder(node.left) + postorder(node.right) + visit(node)` |
| 层序 | 按层从左到右 | BFS，用队列 |

- **递归**：简洁，但栈深度受限于树高（可能栈溢出）
- **迭代**（前序）：手动维护栈：
```python
def preorder_iter(root):
    stack, res = [root], []
    while stack:
        node = stack.pop()
        if node:
            res.append(node.val)
            stack.append(node.right)  # 先右后左，弹出时顺序为左→右
            stack.append(node.left)
```
- **层序**：队列（BFS），可用 `collections.deque`
- **Morris 遍历**：O(1) 空间（不借助栈），通过线索二叉树实现

**可能的追问**：
- 前序/中序/后序遍历的时间复杂度和空间复杂度各是多少？（时间 O(n)，递归空间 O(h)，迭代用栈空间 O(h)）

### Q31：动态规划的核心思想是什么？「爬楼梯」和「背包问题」的 DP 状态定义和转移方程是什么？

**考察点**：DP 核心思维、状态定义、空间优化

**答案要点**：
- **DP 核心思想**：
  1. **最优子结构**：子问题的最优解能导出全局最优解
  2. **无后效性**：当前状态的选择不受后续选择影响
  3. **重叠子问题**：子问题被重复计算（这是 DP 适用的前提）
  4. **自底向上**：先计算小问题，再递推到大问题（vs 记忆化递归是自顶向下）
- **爬楼梯**：`n` 阶楼梯，每次走 1 或 2 步
  - `dp[i]` = 到达第 `i` 阶的方法数
  - 转移方程：`dp[i] = dp[i-1] + dp[i-2]`（从 `i-1` 走一步 或从 `i-2` 走两步）
  - 初始化：`dp[0]=1, dp[1]=1`
  - 空间优化：`a, b = 1, 1; for i in 2..n: a, b = b, a+b`
- **背包问题**：
  - **0-1 背包**：`dp[i][w]` = 前 i 件物品，容量为 w 时的最大价值
    - 转移：`dp[i][w] = max(dp[i-1][w], dp[i-1][w-weight[i]] + value[i])`（选或不选第 i 件）
    - 空间优化：倒序遍历容量（`for w in range(capacity, weight[i]-1, -1)`）
  - **完全背包**：每件物品无限件，正序遍历容量

**可能的追问**：
- 为什么 0-1 背包空间优化要倒序遍历，完全背包要正序？（倒序保证每个物品只用一次，正序允许复用）
- 什么是状态压缩 DP？什么时候可以用一维数组代替二维？
- 「打家劫舍」「股票买卖最佳时机」和「爬楼梯」有什么共同思维模式？

### Q32：字符串匹配算法：BF（暴力匹配）和 KMP 的区别？KMP 的 next 数组是怎么生成的？

**考察点**：字符串匹配、KMP 优化思想、前缀函数

**答案要点**：
- **BF 算法**：逐个字符比较，最坏 O(n*m)，n=文本长度，m=模式串长度。实现简单，小规模数据足够
- **KMP**：
  - 核心思想：当匹配失败时，利用已匹配信息（最长相等前后缀），避免回退文本串指针
  - 时间复杂度 O(n+m)，空间 O(m)
  - 文本串指针不回退，只回退模式串指针
- **next 数组（失效函数）**：`next[i]` = 模式串前 i+1 个字符的最长相等前后缀长度
  - 生成过程（核心）：`next[0]=0`，从 `i=1` 开始：
    ```python
    j = next[i-1]  # 前一个位置的最长相等前后缀
    while j > 0 and pattern[i] != pattern[j]:
        j = next[j-1]  # 回退
    if pattern[i] == pattern[j]:
        j += 1
    next[i] = j
    ```
  - 本质：动态规划 + 回退，用已计算的信息加速当前计算
- **KMP 应用**：字符串替换（String.replaceAll）、DNA 序列匹配、日志关键词搜索

**可能的追问**：
- KMP 的 next 数组能否进一步优化？（nextval，避免相同前缀的重复比较）

### Q33：回溯算法适合解决什么问题？「子集」「组合」「全排列」如何用回溯模板解决？

**考察点**：回溯算法框架、树形思维、剪枝优化

**答案要点**：
- **回溯适用问题**：所有**组合**问题（从集合中选元素）、**穷举**问题（指数级复杂度，但需要遍历所有解空间）
  - 子集问题（所有可能子集）
  - 组合问题（选 k 个数凑目标和）
  - 排列问题（全排列）
  - N 皇后、数独求解
- **通用模板**：
```python
def backtrack(path, choices, start):
    if 满足终止条件:
        res.append(path[:])
        return
    for choice in choices[start:]:
        # 做选择
        path.append(choice)
        # 递归（关键：排列用 choices 全集，组合用 start 索引避免重复）
        backtrack(path, choices, start + 1)  # 组合
        # backtrack(path, choices, 0)        # 排列（不限制 start）
        # 撤销选择
        path.pop()
```
- **子集**：遍历 start，每次把 `choices[i]` 加入 path，不控制树宽，收集所有节点
- **组合**：控制 `len(path) == k` 时收集答案（剪枝：剩余元素不足则跳过）
- **全排列**：遍历 `choices` 全集，`used` 数组标记已用元素，不限制 start
- **剪枝优化**：排序 + 剪枝（跳过不符合条件的分支）可大幅降低搜索树规模

**可能的追问**：
- 回溯算法的时间复杂度如何分析？（子集问题 O(2^n)，排列 O(n!)，剪枝可改善常数）
- 「括号生成」问题（LeetCode 22）如何用回溯+剪枝解决？
- 回溯和 DFS（深度优先搜索）有什么关系？（本质相同，回溯特指有"撤销选择"操作的 DFS）

### Q35：观察者模式和发布订阅模式的区别？在 SOC 平台的告警分发流程中更接近哪种模式？

**考察点**：设计模式区分、事件驱动架构、实际业务映射

**答案要点**：
- **观察者模式（Observer）**：
  - 角色：Subject（被观察者）+ Observer（观察者）
  - 特点：观察者直接注册到 Subject，Subject 直接调用观察者的 `update()` 方法

  - 耦合：通过 Topic/Channel 解耦，通过 Broker 路由
- **SOC 告警分发流程**：
  - 如果是**告警处理器直接调用所有订阅方**，则为**观察者模式**
  - 如果通过 **Kafka / 消息队列** 解耦，则是**发布订阅模式**（Kafka = Broker，告警服务 = Publisher，Enrichment/推送/存储等服务 = Subscriber）
  - 实际项目中：告警从 SIEM 接入 → 发 Kafka Topic → 各 Consumer 订阅 → 分别处理（Enrichment/AI 研判/通知），**更接近发布订阅模式**

**可能的追问**：
- 如果不用消息队列，直接在代码中写 Observer 模式，能满足 SOC 平台的告警分发需求吗？瓶颈在哪里？
- Kafka 的 Consumer Group 和传统 Pub/Sub 的订阅者概念有什么区别？（Consumer Group 内竞争消费，同一组只有一个实例收到消息）
- 什么是响应式编程（Reactive Programming）？它和 Observer 模式有什么关系？

### Q36：策略模式和工厂模式分别适合解决什么问题？在多模型调度框架中如何应用？

**考察点**：设计模式对比、策略模式实现、框架设计

**答案要点**：
- **策略模式（Strategy）**：
  - 目的：将一组可互换的算法/行为封装为独立类，使它们可以相互替换
  - 适用：多种实现方式需根据条件切换，且运行时可能变化
  - 结构：Context（持有 Strategy 引用）+ Strategy 接口 + 具体策略实现
- **工厂模式（Factory）**：
  - 目的：将对象的创建与使用解耦
  - 简单工厂：用一个工厂根据参数创建不同产品
  - 工厂方法：抽象工厂接口，子类决定创建哪个产品
  - 抽象工厂：创建一族相关产品（跨产品族）
- **在多模型调度框架中的应用**：
  - **策略模式**：cheap/fast/powerful 三种调度策略，实现统一的 `LLMStrategy` 接口：
    ```python
    class LLMStrategy(Protocol):
        def select_model(self, task: Task) -> LLMModel

    class CheapStrategy: ...
    class FastStrategy: ...
    class PowerfulStrategy: ...
    ```
  - **工厂模式**：`LLMFactory.create_model(tag)` 根据 tag 字符串返回对应的模型实例（`openai` → `OpenAIModel`，`qwen` → `QwenModel`）
  - **两者结合**：策略决定"用什么"，工厂决定"怎么创建"，工厂为策略提供实例

**可能的追问**：
- 策略模式和状态模式有什么区别？（策略强调算法可替换，状态强调状态转换）
- 装饰器模式和策略模式都在改变行为，它们有什么区别？（装饰器增加功能但不改变原有行为；策略替换行为）
- 在 LangChain 中，哪些地方体现了策略模式和工厂模式？
