## 1.1 网络基础

### Q1：HTTP 和 HTTPS 的区别是什么？HTTPS 的握手过程是怎样的？

- HTTP 明文传输，HTTPS = HTTP + TLS（SSL），端口 443 vs 80
- TLS 1.2/1.3 握手核心流程：
  1. 客户端发Client Hello，带支持的加密算法列表和随机数Client Random
  2. 服务器回Server Hello，选好双方都支持的加密算法，发自己的随机数Server Random，再把数字证书发给客户端
  3. 客户端验证证书有效后，生成Pre-master Secret，用证书里的服务器公钥加密，发给服务器
  4. 客户端和服务器都用Client Random、Server Random、Pre-master Secret，通过相同算法生成会话密钥

### Q2：TCP 三次握手和四次挥手的流程是什么？为什么三次握手不能改成两次？

- **三次握手**：
  第一步，客户端发SYN包，告诉服务器想建立连接，客户端序列号是x
  第二步，服务器收到SYN包，发SYN+ACK包，自己的序列号是y，确认号是x+1，意思是收到了客户端的请求
  第三步，客户端收到SYN+ACK包，发ACK包，确认号是y+1，告诉服务器收到了它的响应
  这样三次交互后，连接就建立起来了
- **四次挥手**：
  第一步，客户端发FIN包，序列号是u，请求断开连接
  第二步，服务器收到FIN包，发ACK包，确认号是u+1，告诉客户端已收到断开请求，此时服务器可能还有数据要处理
  第三步，等服务器数据发完，发FIN包，序列号是v，确认号还是u+1，正式请求断开
  第四步，客户端收到FIN包，发ACK包，确认号是v+1，服务器收到后，连接就彻底断开了
- **为什么三次不能改成两次**：避免因为网络延迟导致“已失效的连接请求报文段”又传到服务器，浪费服务器资源。三次握手客户端会判断这个连接是不是自己想要的，如果不是，就不会发第三次ACK包，服务器收不到ACK，就不会建立连接，这样就避免了这种问题。

### Q3：说说你对 DNS 解析流程的理解，从浏览器输入 URL 到页面渲染完成经历了哪些步骤？浏览器强缓存（Cache-Control: max-age）和协商缓存（Last-Modified / ETag）如何配合工作？

- **DNS 解析流程**：
  1. 浏览器 DNS 缓存 → 2. 系统 hosts 文件 → 3. 系统 DNS 缓存 → 4. 本地 DNS 服务器（递归查询）
  2. 本地 DNS → 根域名服务器 → .com TLD → 权威 DNS 服务器，拿到 IP 后逐级缓存
- **全链路**：
  1. URL 解析（协议、域名、端口、路径）
  2. DNS 解析（获取 IP）
  3. TCP 三次握手（HTTP/HTTPS 端口建立）
  4. TLS 握手（HTTPS）
  5. 浏览器发送 HTTP 请求
  6. CDN 边缘节点回源（如命中缓存则返回）
  7. 后端服务处理请求，执行业务逻辑
  8. HTTP 响应返回（可能 chunked 分块传输）
  9. 浏览器解析 HTML → 构建 DOM 树
  10. 遇到 JS 阻塞渲染 → JS 下载/执行 → CSSOM 构建
  11. DOM + CSSOM → Render Tree → Layout → Paint

- **缓存**：浏览器请求资源时，先判断强缓存是否有效，有效则直接使用本地缓存、不发请求；若强缓存失效，再发起协商缓存，服务端验证资源未修改则返回304，修改则返回200新资源，二者结合兼顾性能与资源时效性。

### Q4：WebSocket、Server-Sent Events（SSE）和 HTTP 长轮询有什么区别？在什么场景下你会选择它们？

**考察点**：实时通信协议对比、协议层选择、SSE 工程落地细节

---

#### 一、三者核心对比

| 特性 | HTTP 长轮询 | Server-Sent Events（SSE） | WebSocket |
|------|------------|--------------------------|-----------|
| **通信方向** | 半双工（始终由客户端发起请求） | 单向（服务端 → 客户端） | 全双工（双向） |
| **连接类型** | 短连接（每次请求独立） | **一个长 HTTP 连接**（Keep-Alive） | TCP 长连接（非 HTTP） |
| **协议开销** | 每次请求都有 HTTP 头（~500B-2KB） | 仅首次握手，后续帧头极小（几字节） | 帧头 2-14 字节，开销最小 |
| **服务端推送** | ❌ 无法主动推，只能等待超时返回 | ✅ 原生支持 | ✅ 原生支持 |
| **复杂度** | 简单，兼容所有浏览器 | 简单，标准 API | 较复杂，需心跳/重连 |
| **自动重连** | 手动实现 | 浏览器原生自动重连（基于 last-event-id） | 需手动实现 |
| **二进制数据** | Base64 编码 | ❌ 仅文本（可传 JSON） | ✅ 支持二进制帧 |
| **同源限制** | 无（CORS 可跨域） | 受同源策略限制 | 受同源策略限制 |
| **HTTP/2 兼容** | ✅ | ✅（共用 TCP 连接） | ✅（但需 HTTP Upgrade） |

---

#### 二、HTTP 长轮询原理

- 客户端发请求，服务端无数据则**挂起**，有数据或超时（15-30s）才返回
- 客户端收到响应后**立即**发起下一轮请求
- 每次请求都带完整 HTTP 头，有不必要开销
- 延迟上限 = 轮询周期，不适合高频推送场景
- 典型应用：早期 WebQQ、邮件推送、ChatGPT 轮询（接口轮询，非流式）

#### 三、SSE（Server-Sent Events）深度解析

**1. 工作原理**

- 客户端通过 `EventSource` API 订阅：`new EventSource('/api/stream')`
- 服务端返回 `Content-Type: text/event-stream` 的 HTTP 响应，保持连接
- 服务端通过写文本数据推送，格式为多行字段：

```
: 注释行（可选，用于心跳/保活）
id: 1
event: alert
retry: 3000
data: {"level":"high","msg":"可疑登录"}

id: 2
event: alert
data: {"level":"medium","msg":"暴力破解"}
```

**2. 四个核心字段的作用**

| 字段 | 作用 | 典型场景 |
|------|------|---------|
| `data:` | 消息内容，多行 data 自动用 `\n` 合并 | 推送 JSON 结构化数据 |
| `id:` | 事件 ID，浏览器自动记录。下次重连时自动发 `Last-Event-ID` 头 | 断点续传，保证不漏消息 |
| `event:` | 事件类型，客户端按类型监听 `source.addEventListener('alert', handler)` | 多路复用，不同事件类型 |
| `retry:` | 告知浏览器断连后重试间隔（毫秒） | 控制重试频率，默认 3s |

**3. 为什么选择 SSE 而非 WebSocket？**

适合 SSE 的场景（单向推送为主）：
- **SOC 告警实时推送**：告警从服务端推向前端，前端只读展示，不需要前端反向发消息
- **仪表盘数据刷新**：大屏监控、股票行情、服务器指标
- **邮件/通知推送**：只需服务端主动通知
- **AI 流式响应**：LLM token 流式返回（OpenAI Chat API 同时支持 SSE 和 WebSocket）
- **运维日志流**：CI/CD 流水线日志实时展示

SSE 的原生优势：
1. **自动重连 + 断点续传**：`Last-Event-ID` 机制让浏览器自动从上次断开位置续接，无需手动实现
2. **基于 HTTP/2**：可复用同一 TCP 连接推送多个独立的 SSE 流，开销更低
3. **纯文本 + JSON**：对结构化文本数据天然友好
4. **实现简单**：无需 WebSocket 的升级握手、心跳保活、背压处理

**4. SSE 在 Koa/Node.js 中的实现**

```javascript
// Koa 路由（服务端）
router.get('/api/alerts/stream', async (ctx) => {
  ctx.type = 'text/event-stream';
  ctx.set('Cache-Control', 'no-cache');
  ctx.set('Connection', 'keep-alive');
  ctx.set('X-Accel-Buffering', 'no'); // 禁用 Nginx 缓冲

  const stream = new Readable({
    read() {} // 不消费 buffer，由服务端 push
  });

  // 模拟告警推送
  const alertTimer = setInterval(() => {
    const data = JSON.stringify({ level: 'high', ts: Date.now() });
    stream.push(`id: ${Date.now()}\nevent: alert\ndata: ${data}\n\n`);
  }, 2000);

  // 优雅关闭：客户端断开时清理定时器
  ctx.req.on('close', () => {
    clearInterval(alertTimer);
    stream.destroy();
  });

  ctx.body = stream;
});
```

```javascript
// 客户端
const source = new EventSource('/api/alerts/stream');

// 监听特定类型事件
source.addEventListener('alert', (e) => {
  const alert = JSON.parse(e.data);
  console.log('收到告警:', alert);
});

// 错误处理和自动重连
source.onerror = (err) => {
  console.error('SSE 连接错误:', err);
  // EventSource 在 onerror 后会自动重连，无需手动处理
};

// 手动关闭
source.close();
```

**5. SSE 的常见坑**

| 坑点 | 说明 | 解决方案 |
|------|------|---------|
| **Nginx 缓冲** | Nginx 默认缓冲 SSE 响应直到满，才会推送给客户端 | `proxy_buffering off;` + `X-Accel-Buffering: no` |
| **Node.js 单进程阻塞** | SSE 长连接占用事件循环，导致其他请求阻塞 | 用 Worker Threads 或独立 SSE 服务进程 |
| **进程退出未清理** | 服务重启时 SSE 连接未关闭，造成僵尸连接 | `ctx.req.on('close', ...)` 清理资源 |
| **同源限制** | `EventSource` 不支持跨域（不像 fetch） | 服务端设置 `Access-Control-Allow-Origin` |
| **代理超时** | 企业代理/负载均衡器可能超时断连 | Nginx `proxy_read_timeout 86400;` + 定期 `:` 空注释保活 |

#### 四、WebSocket 深度解析

- **握手**：基于 HTTP Upgrade，客户端发 `Upgrade: websocket`，服务端回 `101 Switching Protocols`
- **全双工**：握手中不再走 HTTP 协议帧，后续直接走 WebSocket 帧（opcode = 文本/二进制/关闭帧）
- **心跳保活**：`ping/pong` 帧（不是 HTTP），约 30s 无活动时双方发心跳
- **断线重连**：需手动实现，通常配合指数退避（1s → 2s → 4s → 8s...上限 30s）
- **二进制支持**：适合传图片、文件、Protocol Buffers 等二进制数据

---

#### 五、场景选择决策树

```
是否需要服务端主动推送？
├── 否 → HTTP REST API（轮询）
├── 是，需要全双工（双向数据交换）
│   ├── 低延迟 + 二进制数据 → WebSocket
│   └── 需要最大兼容性 + 跨语言服务端 → WebSocket
└── 是，只需单向推送（服务端→客户端）
    ├── 高频推送（>1 msg/s），或需二进制 → WebSocket
    ├── 文本数据，兼容性/实现简单优先 → SSE ✅
    └── 旧浏览器（IE10以下），无现代需求 → HTTP 长轮询
```

#### 六、与 gRPC Streaming 的对比

- **gRPC bidirectional streaming**：全双工 + 强类型（Protocol Buffers）+ 跨语言，适合微服务间内部通信
- **SSE vs gRPC Streaming**：SSE 更适合浏览器作为客户端的场景（原生 EventSource，无需 SDK）；gRPC Streaming 需要 gRPC-Web 代理，对浏览器兼容性较差

### Q5：你在 SOC 平台中用到了 Nacos 做服务注册与发现，简述 Nacos 的工作原理

Nacos的服务注册与发现核心是“服务端存储+客户端心跳+动态感知”。服务提供者启动时，会向Nacos Server发送注册请求，带上服务名、IP、端口等元数据，Server将这些信息存储在内存和持久化介质中。注册后，提供者会定期发送心跳包，Server若长时间收不到心跳，会将服务标记为不可用并剔除。服务消费者启动时，会向Server订阅目标服务，Server返回可用服务实例列表，同时消费者会通过长轮询或推送机制，实时监听Server上服务实例的变化，当实例新增或下线时，消费者能及时更新本地缓存的实例列表，从而实现动态的服务发现。

### Q6：UDP 和 TCP 的区别？TCP 的可靠传输是如何实现的？

UDP是无连接、不可靠的传输层协议，发送数据前不握手，数据报可能丢失、乱序，没有拥塞控制，速度快，适合实时场景（如视频通话、直播）。TCP是面向连接、可靠的协议，发送前需三次握手建立连接，通过序列号、确认应答、超时重传等机制保证数据不丢不重、按序到达，有拥塞控制，速度较慢但可靠，适合文件传输、HTTP通信等。

TCP可靠传输的实现

1. 三次握手建立连接：确保双方收发能力正常，为可靠传输奠定基础。

2. 序列号与确认应答：每个TCP段带序列号，接收方收到后返回“确认应答号”（期望下一段的序列号），发送方通过确认应答判断数据是否送达。

3. 超时重传：发送方若在超时时间内未收到确认应答，会重传该段数据。

4. 滑动窗口机制：通过窗口大小控制发送速率，接收方根据自身缓存能力动态调整窗口，避免发送方发送过快导致数据丢失，同时实现批量传输提升效率。

5. 拥塞控制：通过慢开始、拥塞避免、快重传、快恢复等算法，根据网络拥塞程度调整发送速率，减少因网络拥堵导致的丢包。

6. 数据校验与重排：接收方对收到的TCP段进行校验，丢弃损坏段；对乱序段通过序列号重新排序，确保数据按序交付给应用层。

### Q7：CDN 的工作原理？回源率和缓存命中率分别指什么？什么样的内容适合放 CDN？

- **CDN 工作原理**：CDN 就是把静态资源缓存到全国各地的节点。用户访问时，就近访问节点，不用回源站，速度更快、减轻源站压力
- **关键指标**：
  - **缓存命中率（Hit Rate）** = 缓存命中请求 / 总请求。越高越好，直接影响响应速度和源站压力
  - **回源率** = 回源请求 / 总请求。与命中率互补，越低越好
- **适合 CDN 的内容**：静态资源（图片/CSS/JS/字体/视频）、不经常变化的内容、公共 API 响应（可接受一定延迟）
- **不适合的场景**：个性化/用户相关数据、强一致性要求（如库存、金额）、频繁变化的接口

### Q8：常见的 Web 安全漏洞（XSS、CSRF、SQL 注入）原理是什么？防范手段有哪些？你在后端代码层面做了哪些防护？

- **XSS（Cross-Site Scripting）**：
  - 原理：未过滤的用户输入被当作 HTML/JS 执行，窃取 Cookie/Session、篡改页面
  - 类型：存储型（存入 DB）、反射型（URL 参数）、DOM 型（纯前端）
  - 防范：输入过滤/白名单、输出编码（HTMLEncode）、HttpOnly Cookie、CSP（Content-Security-Policy）
- **CSRF（Cross-Site Request Forgery）**：
  - 原理：用户登录后访问恶意页面，页面自动发送已登录 Cookie 的跨域请求
  - 防范：CSRF Token（表单/请求体）、SameSite Cookie（Strict/Lax）、Referer/Origin 头校验
- **SQL 注入**：
  - 原理：用户输入拼接到 SQL 语句，破坏原语句结构，执行任意 SQL
  - 防范：参数化查询（PreparedStatement）、ORM 框架、严格类型校验、DB 最小权限原则
- **实际防护实践**：
  - Koa 网关层统一注入 CSRF Token 校验中间件
  - 所有 DB 查询走 ORM 或 PreparedStatement，禁用字符串拼接
  - 输入校验用 Joi/Zod schema validation
  - 安全响应头（X-Frame-Options、X-Content-Type-Options）中间件统一设置

### Q13：进程、线程、协程的区别是什么？Node.js 的事件循环属于哪一种？

**考察点**：进程/线程/协程模型、并发 vs 并行、Node.js 运行时架构

**答案要点**：

- **进程**：资源分配基本单位，有独立地址空间（代码/数据/堆/栈/文件描述符），隔离性强，切换开销大（TCB 切换、页表切换、缓存失效）
- **线程**：CPU 调度基本单位，同一进程内共享地址空间（堆、全局变量），切换开销小，存在竞态条件
- **协程**（用户态线程）：由程序自己控制切换（非操作系统），无需进入内核，开销极小（如 goroutine、Python asyncio），协作式调度（yield/await），一个线程可挂起多个协程
- **Node.js 事件循环**：属于**单线程 + 事件驱动 + 回调队列**模型，不属于传统进程/线程/协程。其本质是一个基于 libuv 的单线程循环，不断从队列中取回调执行，适合 IO 密集型。CPU 密集型任务会阻塞事件循环。
- **Node.js 多线程补充**：Worker Threads（独立 V8 实例）、libuv 线程池（处理 fs/DNS/crypto 等异步操作）

### Q14：什么是死锁？产生死锁的四个必要条件是什么？如何预防？

**考察点**：死锁概念、竞态条件与同步、操作系统资源管理

**答案要点**：

- **死锁定义**：两个或多个进程/线程相互持有对方需要的资源，且不主动释放，形成循环等待
- **四个必要条件（COIPA）**：
  1. **互斥条件**：资源同一时刻只能被一个线程持有
  2. **持有并等待**：线程持有资源同时请求其他资源
  3. **不可抢占**：已分配资源不能被强制剥夺
  4. **循环等待**：形成线程间的循环等待链
- **预防措施**：
  - 破坏持有并等待：一次性申请所有所需资源
  - 破坏循环等待：按固定顺序申请资源（如哲学家就餐问题统一编号）
  - 破坏不可抢占：允许抢占
  - 破坏互斥：尽量少用互斥锁，或用读写锁等替代
- **避免（银行家算法）**：动态检测安全状态，安全则分配
- **检测与恢复**：定期检测有向图环路，强制回滚或 kill 进程

### Q17：程序运行时的内存布局（代码区/全局区/堆/栈）是什么样的？malloc/free 和 new/delete 在堆上的分配原理是什么？

**考察点**：进程内存布局、动态内存管理、堆内存分配算法

**答案要点**：

- **内存布局（从低地址到高地址）**：
  1. **代码区（.text）**：只读，存放机器指令
  2. **只读常量区（.rodata）**：常量（const、字符串字面量）
  3. **全局/静态区（.data/.bss）**：全局变量、static 变量（bss=未初始化=0）
  4. **堆（Heap）**：动态分配，向上增长（低地址→高地址），手动管理
  5. **内存映射区（mmap）**：动态库、匿名映射
  6. **栈（Stack）**：函数调用、局部变量，向下增长（高地址→低地址），自动管理
  7. **内核空间**：操作系统内核
- **malloc/free 原理**：
  - 小块内存（<128KB）：brk() 调整堆顶，系统调用次数多则用 mmap() 分配
  - 内部维护空闲链表（free list）、位图（bitmap）或边界标签（buddy system）
  - 分配时先查空闲链表，找合适大小块，找不到则向 OS 申请更多页
  - free 并非真正归还 OS，而是加入空闲链表供下次分配使用
- **new/delete**：底层调用 malloc/free，但额外调用构造函数/析函数

### Q18：什么是虚拟内存？为什么需要虚拟内存？Linux 的 swap 机制是什么？

**考察点**：虚拟内存原理、分页/分段、内存置换策略

**答案要点**：

- **虚拟内存**：每个进程有独立的虚拟地址空间（32位=4GB，64位更广），虚拟地址通过 MMU（内存管理单元）+ 页表映射到物理地址
- **为什么需要虚拟内存**：
  1. **进程隔离**：防止进程间越界访问
  2. **内存抽象**：程序可以使用连续虚拟地址，无需关心物理碎片
  3. **内存超售**：让程序使用大于物理内存的空间（物理内存不够时换出到磁盘）
  4. **按需分配**：Lazy loading，只在真正访问时分配物理页（Page Fault 触发分配）
- **swap 机制**：物理内存不足时，把不活跃的页换出到 swap 分区（磁盘），需要时再换入。swap 使用率过高（>60%）会严重影响性能，称为 swap thrashing（内存抖动）
- **Linux 相关配置**：`vm.swappiness`（倾向于 swap 的程度）、`vm.overcommit_memory`（内存超额分配策略）

**可能的追问**：

- Page Fault 有哪几种？（Minor/Major/Invalid）Minor Fault 为什么不需要磁盘 IO？
- TLB（Translation Lookaside Buffer）的作用是什么？TLB miss 的代价有多大？
- 大页（HugePages）如何提升性能？

### Q21：CPU 上下文切换是什么？频繁的上下文切换会带来什么性能问题？在容器化环境中如何控制？

**考察点**：上下文切换机制、性能损耗、容器资源隔离

**答案要点**：

- **上下文切换**：CPU 从一个进程/线程切换到另一个时，保存当前任务状态（寄存器、程序计数器、栈指针）到 PCB/TCB，加载新任务状态。包含：**保存现场 → 切换内核数据结构 → 恢复现场**
- **性能问题**：
  1. 每次切换有固定开销（~1-10μs），大量切换时 CPU 时间浪费在内核代码而非业务代码
  2. 切换导致 CPU 缓存（Cache）失效（Cache Miss 开销远大于切换本身）
  3. 频繁切换可能触发 Thrashing（抖动），吞吐量急剧下降
  4. 指标：`cs`（context switch）数可从 `vmstat 1` 查看
- **容器化控制**：
  - CPU shares / cpu.cfs_quota_us 限制 CPU 使用
  - `--cpus` 指定 CPU 核数（Kubernetes `resources.limits.cpu`）
  - `cpuset.cpus` 绑核避免跨核切换（减少 Cache Miss）
  - 容器级别设置 `schedtune`（Android/Cgroup）或 cgroup v2 资源控制

**可能的追问**：

- 为什么说协程切换比线程切换开销小很多？（用户态切换，无需进入内核）
- 什么工具可以分析上下文切换的具体原因？（perf sched）

### Q23：HashMap 的底层实现原理？哈希冲突有哪些解决方式？

**考察点**：哈希表原理、冲突解决、性能分析

**答案要点**：

- **底层结构（JDK 1.8+）**：数组 + 链表 + 红黑树。Node[] table（散列表），每个桶在链表长度 > 8 且数组长度 >= 64 时转为红黑树
- **插入流程**：
  1. 计算 key 的 hash（扰动函数：`hash = (key == null) ? 0 : (h = key.hashCode()) ^ (h >>> 16)`）
  2. `(n - 1) & hash` 计算数组下标
  3. 如果桶为空，直接插入；否则遍历链表/红黑树
  4. 发现相同 key 则覆盖 value
  5. 插入后 size++，若超过 threshold（capacity \* loadFactor），则 resize 扩容（2 倍）
- **哈希冲突解决方式**：
  1. **开放寻址法**：线性探测（hash+1）、二次探测（hash+i²）、双重哈希
  2. **链地址法（拉链法）**：每个桶存链表/红黑树（Java HashMap / Python dict）
  3. **再哈希法**：用第二个哈希函数计算下一个位置
  4. **建立公共溢出区**：冲突的元素放入溢出表
- **为什么用红黑树而非 AVL？**：红黑树是弱平衡，插入/删除开销（旋转次数少）比 AVL 小，适合频繁修改的场景

**可能的追问**：

- HashMap 的容量为什么必须是 2 的幂次？（`(n-1) & hash` 等价于 `hash % n`，位运算更快）
- HashMap 是线程安全的吗？有哪些线程安全方案？（ConcurrentHashMap、Collections.synchronizedMap）
- ConcurrentHashMap 在 JDK 8 中如何实现分段锁？（CAS + synchronized 锁桶头，不再是分段 Segment）

### Q24：给你一个包含 1 亿条安全日志的数据集，如何快速找出出现频率 Top 10 的攻击 IP？

**考察点**：海量数据处理、哈希思想、堆的应用、MapReduce 思维

**答案要点**：

- **方法一：哈希 + 小顶堆（单机）**
  1. 用哈希分流（哈希 IP 到不同桶，减少单节点内存压力）
  2. 每个桶内用 HashMap 计数：`Map<String, Integer> counter`
  3. 维护大小为 10 的小顶堆（MinHeap），堆顶是当前 Top 10 中频次最低的
  4. 每个 IP 出现时，若 count > 堆顶，则替换堆顶并调整
  5. 时间复杂度 O(n log k)，空间复杂度 O(n)（但实际 IP 数量远小于日志数）
- **方法二：MapReduce（分布式）**
  - Map：每条日志提取 IP，输出 `<IP, 1>`
  - Shuffle：相同 IP 的记录发到同一 Reduce
  - Reduce：求和统计每个 IP 总次数
  - 再对结果排序取 Top 10
- **方法三：近似算法（HyperLogLog / Count-Min Sketch）**：精度换空间，适合无法全存内存的超大数据集

### Q25：B+ 树和 B 树的区别？为什么 MySQL InnoDB 选择 B+ 树做索引？

**考察点**：B+ 树 vs B 树、索引结构、磁盘 IO 优化

**答案要点**：
| 特性 | B 树 | B+ 树 |
|------|------|-------|
| 数据存储 | 所有节点都存数据 | 只有叶子节点存数据，非叶子节点只存索引 |
| 高度 | 更高（每个节点都存数据） | 更矮（可容纳更多索引项） |
| 范围查询 | 需要中序遍历所有路径 | 叶子节点链表，支持顺序访问 |
| 查询稳定性 | 非叶子节点也存数据，查询路径长度不稳定 | 所有查询必须到叶子，查询路径长度一致 |
| 空间利用率 | 较低 | 更高（所有非叶子节点可充满） |

- **InnoDB 选 B+ 树原因**：
  1. **磁盘 IO 优化**：每个节点=一页（16KB），B+ 树更矮，磁盘 IO 次数更少（通常 2-3 次磁盘读取即可查到数据）
  2. **范围查询友好**：叶子节点链表顺序访问，避免回溯
  3. **查询稳定**：必须查到叶子，查询时间可预测
  4. **空间利用率高**：非叶子节点不存数据，可容纳更多分叉
  5. **聚簇索引**：主键索引的叶子节点直接存储行数据，减少回表 IO

### Q26：写一个 LRU 缓存，要求 get 和 put 操作都是 O(1) 时间复杂度。

**考察点**：哈希表 + 双向链表、LRU 原理、LinkedHashMap 应用

**答案要点**：

```python
# 方法一：手写哈希表 + 双向链表
class Node:
    def __init__(self, key=0, val=0):
        self.key, self.val = key, val
        self.prev = self.next = None

class LRUCache:
    def __init__(self, capacity: int):
        self.capacity = capacity
        self.cache = {}  # key -> Node
        # 伪头/伪尾，简化边界处理
        self.head = Node()
        self.tail = Node()
        self.head.next = self.tail
        self.tail.prev = self.head

    def _remove(self, node: Node):
        """从链表当前位置摘下节点"""
        node.prev.next = node.next
        node.next.prev = node.prev

    def _add_to_head(self, node: Node):
        """插入到链表头部（最近使用）"""
        node.next = self.head.next
        node.prev = self.head
        self.head.next.prev = node
        self.head.next = node

    def get(self, key: int) -> int:
        if key not in self.cache:
            return -1
        node = self.cache[key]
        self._remove(node)       # 先摘下
        self._add_to_head(node)  # 再移到头部
        return node.val

    def put(self, key: int, val: int) -> None:
        if key in self.cache:
            self._remove(self.cache[key])
            del self.cache[key]
        node = Node(key, val)
        self.cache[key] = node
        self._add_to_head(node)
        if len(self.cache) > self.capacity:
            # 淘汰尾部（最久未使用）
            lru = self.tail.prev
            self._remove(lru)
            del self.cache[lru.key]
```

- **O(1) 原因**：哈希表 O(1) 查找 + 双向链表 O(1) 插入/删除
- **Python 替代**：`collections.OrderedDict` 底层也是双向链表 + 哈希表，可直接用 `move_to_end()` 实现 LRU

**可能的追问**：

### Q27：链表和数组的区别？单向链表、双向链表、循环链表各自的特点和使用场景？链表逆置怎么做？

**考察点**：线性数据结构、指针操作、算法实现

**答案要点**：

- **数组 vs 链表**：
  - 数组：O(1) 随机访问，O(n) 插入/删除（需移动元素），内存连续（Cache 友好）
  - 链表：O(n) 随机访问，O(1) 插入/删除（改指针），内存不连续（动态大小）
- **单向链表**：每个节点有 next 指针。适合做栈（只操作头部）
- **双向链表**：每个节点有 prev + next 指针。适合做 LRU Cache（需前向后向操作）、浏览器的历史记录
- **循环链表**：尾节点的 next 指向头节点。约瑟夫问题、循环队列
- **链表逆置**（迭代）：

```python
def reverse(head):
    prev = None
    curr = head
    while curr:
        nxt = curr.next
        curr.next = prev
        prev = curr
        curr = nxt
    return prev  # 新头节点
```

- **递归逆置**：从尾部开始逆向建立连接，适合学习但有栈溢出风险（n 层递归深度）

### Q29：什么是二叉树的遍历（前序/中序/后序/层序）？递归和迭代实现分别怎么写？

**考察点**：树遍历算法、递归与迭代转换、栈/队列的应用

**答案要点**：
| 遍历 | 顺序 | 递归实现 |
|------|------|---------|
| 前序 | 根 → 左 → 右 | `visit(node) + preorder(node.left) + preorder(node.right)` |
| 中序 | 左 → 根 → 右 | `inorder(node.left) + visit(node) + inorder(node.right)` |
| 后序 | 左 → 右 → 根 | `postorder(node.left) + postorder(node.right) + visit(node)` |
| 层序 | 按层从左到右 | BFS，用队列 |

- **递归**：简洁，但栈深度受限于树高（可能栈溢出）
- **迭代**（前序）：手动维护栈：

```python
def preorder_iter(root):
    stack, res = [root], []
    while stack:
        node = stack.pop()
        if node:
            res.append(node.val)
            stack.append(node.right)  # 先右后左，弹出时顺序为左→右
            stack.append(node.left)
```

- **层序**：队列（BFS），可用 `collections.deque`
- **Morris 遍历**：O(1) 空间（不借助栈），通过线索二叉树实现

### Q30：什么是 Trie（字典树）？在安全场景中（关键词匹配、白名单匹配）有什么应用？

### Q31：动态规划的核心思想是什么？「爬楼梯」和「背包问题」的 DP 状态定义和转移方程是什么？

**考察点**：DP 核心思维、状态定义、空间优化

**答案要点**：

- **DP 核心思想**：
  1. **最优子结构**：子问题的最优解能导出全局最优解
  2. **无后效性**：当前状态的选择不受后续选择影响
  3. **重叠子问题**：子问题被重复计算（这是 DP 适用的前提）
  4. **自底向上**：先计算小问题，再递推到大问题（vs 记忆化递归是自顶向下）
- **爬楼梯**：`n` 阶楼梯，每次走 1 或 2 步
  - `dp[i]` = 到达第 `i` 阶的方法数
  - 转移方程：`dp[i] = dp[i-1] + dp[i-2]`（从 `i-1` 走一步 或从 `i-2` 走两步）
  - 初始化：`dp[0]=1, dp[1]=1`
  - 空间优化：`a, b = 1, 1; for i in 2..n: a, b = b, a+b`
- **背包问题**：
  - **0-1 背包**：`dp[i][w]` = 前 i 件物品，容量为 w 时的最大价值
    - 转移：`dp[i][w] = max(dp[i-1][w], dp[i-1][w-weight[i]] + value[i])`（选或不选第 i 件）
    - 空间优化：倒序遍历容量（`for w in range(capacity, weight[i]-1, -1)`）
  - **完全背包**：每件物品无限件，正序遍历容量

**可能的追问**：

- 为什么 0-1 背包空间优化要倒序遍历，完全背包要正序？（倒序保证每个物品只用一次，正序允许复用）
- 什么是状态压缩 DP？什么时候可以用一维数组代替二维？
- 「打家劫舍」「股票买卖最佳时机」和「爬楼梯」有什么共同思维模式？

### Q32：字符串匹配算法：BF（暴力匹配）和 KMP 的区别？KMP 的 next 数组是怎么生成的？

**考察点**：字符串匹配、KMP 优化思想、前缀函数

**答案要点**：

- **BF 算法**：逐个字符比较，最坏 O(n\*m)，n=文本长度，m=模式串长度。实现简单，小规模数据足够
- **KMP**：
  - 核心思想：当匹配失败时，利用已匹配信息（最长相等前后缀），避免回退文本串指针
  - 时间复杂度 O(n+m)，空间 O(m)
  - 文本串指针不回退，只回退模式串指针
- **next 数组（失效函数）**：`next[i]` = 模式串前 i+1 个字符的最长相等前后缀长度
  - 生成过程（核心）：`next[0]=0`，从 `i=1` 开始：
    ```python
    j = next[i-1]  # 前一个位置的最长相等前后缀
    while j > 0 and pattern[i] != pattern[j]:
        j = next[j-1]  # 回退
    if pattern[i] == pattern[j]:
        j += 1
    next[i] = j
    ```
  - 本质：动态规划 + 回退，用已计算的信息加速当前计算
- **KMP 应用**：字符串替换（String.replaceAll）、DNA 序列匹配、日志关键词搜索

**可能的追问**：

- KMP 的 next 数组能否进一步优化？（nextval，避免相同前缀的重复比较）

### Q33：回溯算法适合解决什么问题？「子集」「组合」「全排列」如何用回溯模板解决？

**考察点**：回溯算法框架、树形思维、剪枝优化

**答案要点**：

- **回溯适用问题**：所有**组合**问题（从集合中选元素）、**穷举**问题（指数级复杂度，但需要遍历所有解空间）
  - 子集问题（所有可能子集）
  - 组合问题（选 k 个数凑目标和）
  - 排列问题（全排列）
  - N 皇后、数独求解
- **通用模板**：

```python
def backtrack(path, choices, start):
    if 满足终止条件:
        res.append(path[:])
        return
    for choice in choices[start:]:
        # 做选择
        path.append(choice)
        # 递归（关键：排列用 choices 全集，组合用 start 索引避免重复）
        backtrack(path, choices, start + 1)  # 组合
        # backtrack(path, choices, 0)        # 排列（不限制 start）
        # 撤销选择
        path.pop()
```

- **子集**：遍历 start，每次把 `choices[i]` 加入 path，不控制树宽，收集所有节点
- **组合**：控制 `len(path) == k` 时收集答案（剪枝：剩余元素不足则跳过）
- **全排列**：遍历 `choices` 全集，`used` 数组标记已用元素，不限制 start
- **剪枝优化**：排序 + 剪枝（跳过不符合条件的分支）可大幅降低搜索树规模

**可能的追问**：

- 回溯算法的时间复杂度如何分析？（子集问题 O(2^n)，排列 O(n!)，剪枝可改善常数）
- 「括号生成」问题（LeetCode 22）如何用回溯+剪枝解决？
- 回溯和 DFS（深度优先搜索）有什么关系？（本质相同，回溯特指有"撤销选择"操作的 DFS）

### Q35：观察者模式和发布订阅模式的区别？在 SOC 平台的告警分发流程中更接近哪种模式？

**考察点**：设计模式区分、事件驱动架构、实际业务映射

**答案要点**：

- **观察者模式（Observer）**：
  - 角色：Subject（被观察者）+ Observer（观察者）
  - 特点：观察者直接注册到 Subject，Subject 直接调用观察者的 `update()` 方法
  - 耦合：Subject 知道所有观察者是谁（直接依赖）
- **发布订阅模式（Pub/Sub）**：
  - 角色：Publisher（发布者）+ Subscriber（订阅者）+ Broker（消息代理/Event Bus）
  - 特点：发布者和订阅者通过 Broker 解耦，双方不需要知道对方存在
  - 耦合：通过 Topic/Channel 解耦，通过 Broker 路由
- **SOC 告警分发流程**：
  - 如果是**告警处理器直接调用所有订阅方**，则为**观察者模式**
  - 如果通过 **Kafka / 消息队列** 解耦，则是**发布订阅模式**（Kafka = Broker，告警服务 = Publisher，Enrichment/推送/存储等服务 = Subscriber）
  - 实际项目中：告警从 SIEM 接入 → 发 Kafka Topic → 各 Consumer 订阅 → 分别处理（Enrichment/AI 研判/通知），**更接近发布订阅模式**

### Q36：策略模式和工厂模式分别适合解决什么问题？在多模型调度框架中如何应用？

**考察点**：设计模式对比、策略模式实现、框架设计

**答案要点**：

- **策略模式（Strategy）**：
  - 目的：将一组可互换的算法/行为封装为独立类，使它们可以相互替换
  - 适用：多种实现方式需根据条件切换，且运行时可能变化
  - 结构：Context（持有 Strategy 引用）+ Strategy 接口 + 具体策略实现
- **工厂模式（Factory）**：
  - 目的：将对象的创建与使用解耦
  - 简单工厂：用一个工厂根据参数创建不同产品
  - 工厂方法：抽象工厂接口，子类决定创建哪个产品
  - 抽象工厂：创建一族相关产品（跨产品族）
- **在多模型调度框架中的应用**：
  - **策略模式**：cheap/fast/powerful 三种调度策略，实现统一的 `LLMStrategy` 接口：

    ```python
    class LLMStrategy(Protocol):
        def select_model(self, task: Task) -> LLMModel

    class CheapStrategy: ...
    class FastStrategy: ...
    class PowerfulStrategy: ...
    ```

  - **工厂模式**：`LLMFactory.create_model(tag)` 根据 tag 字符串返回对应的模型实例（`openai` → `OpenAIModel`，`qwen` → `QwenModel`）
  - **两者结合**：策略决定"用什么"，工厂决定"怎么创建"，工厂为策略提供实例

- 装饰器模式和策略模式都在改变行为，它们有什么区别？（装饰器增加功能但不改变原有行为；策略替换行为）
