## 10.1 SOC 安全运营

### Q1：SOC 是什么？SOC 安全运营的核心流程（识别→保护→检测→响应→恢复）你是怎么理解的？

**考察点**：SOC 概念、NIST CSF 框架

**答案要点**：
- **SOC（Security Operations Center）**：企业内集中监控、分析、响应安全事件的团队和技术平台
- **NIST CSF 五大功能**：
  1. **识别（Identify）**：资产发现、风险评估、供应链管理
     - 理解 SOC 平台中所有监控资产的清单
  2. **保护（Protect）**：安全策略、访问控制、员工培训
     - 告警规则、访问控制列表（ACL）、敏感数据保护
  3. **检测（Detect）**：持续监控、异常检测、威胁狩猎
     - 告警产生、关联分析、SIEM 规则匹配
     - 这是 SOC 平台的核心价值所在
  4. **响应（Respond）**：事件调查、遏制、根因分析、沟通协调
     - Alert → Case → Ticket → 通知处置
  5. **恢复（Recover）**：业务恢复、事后分析、改进措施
     - 复盘报告、规则优化、防御增强
- **在 SOC 平台中的落地**：平台的每一功能都对应 CSF 的某个环节

**可能的追问**：
- 什么是安全成熟度模型（Capability Maturity Model）？
- 什么是 MTTD（Mean Time To Detect）和 MTTR（Mean Time To Respond）？
- SOC 平台和 NOC（Network Operations Center）有什么区别？

### Q2：SIEM 和 SOAR 的区别是什么？你们平台的定位更偏向哪个？为什么需要两者结合？

**考察点**：SIEM vs SOAR、平台定位

**答案要点**：
- **SIEM vs SOAR**：
  | 维度 | SIEM | SOAR |
  |------|------|------|
  | 核心能力 | 日志采集 + 关联分析 + 告警 | 自动化剧本 + 工作流 + 响应 |
  | 关注点 | 检测（发现威胁） | 响应（处置威胁） |
  | 数据来源 | 各类日志（网络/主机/应用） | SIEM 告警、威胁情报、CMDB |
  | 典型工具 | Splunk/ELK/弧面 | Palo Alto XSOAR/Demisto |
  | SOC 角色 | "眼睛"（发现） | "手"（处置） |
- **SOC 平台定位**：**SIEM + SOAR 融合**，即"检测+响应一体化"
  - 检测层（SIEM）：ES 日志关联、告警产生
  - 响应层（SOAR）：AI Agent 自动研判、Playbook 处置
  - 两者通过告警事件总线（Kafka）打通
- **为什么结合**：
  1. 单独 SIEM 只管发现，不管处置，分析师手动处置效率低
  2. 单独 SOAR 缺少检测能力，需要 SIEM 提供告警输入
  3. 告警疲劳问题：SOAR 自动处置低危告警，减少分析师工作量

**可能的追问**：
- SIEM 的日志采集 Agent（如 Filebeat/Fluentd）和 SIEM 本身的部署模式是什么？
- 什么是 UEBA（User and Entity Behavior Analytics）？和 SIEM 有什么关系？
- 如何衡量 SOAR 平台的 ROI？（处置自动化率节省的人天）

### Q3：OCSF 是什么？为什么你们平台的 Alert 模型选择遵循 OCSF 风格？这对后续对接有什么好处？

**考察点**：OCSF 标准、安全数据互操作

**答案要点**：
- **OCSF（Open Cybersecurity Schema Framework）**：
  - OCSF 是一个开源的安全事件标准化 Schema，旨在统一不同安全工具的数据格式
  - 核心概念：将安全事件抽象为**类别（Class）→ 事件（Event）→ 属性（Attributes** 的层级结构
  - 示例：`intrusion_detection` 类下的 `malware` 事件，包含 `activity_id`, `device`, `network`, `threat` 等属性
- **SOC 平台遵循 OCSF 的原因**：
  1. **互操作性**：对接多种 SIEM/EDR/XDR 时，只需要做格式映射（ETL），不需改核心模型
  2. **降低对接成本**：新接入一个数据源，按 OCSF Schema 映射即可，平台侧改动最小
  3. **社区生态**：OCSF 有丰富的预定义 Schema，可以直接使用
- **OCSF 适配实践**：
  - 平台 Alert 数据模型映射到 OCSF `discovery`/`intrusion_detection` 等类
  - 通过 OCSF Schema Hub 定义各数据源的 Connector（如 WAF/防火墙/终端）

**可能的追问**：
- OCSF 和 CEF（Common Event Format）有什么关系？
- 什么是 OCSF 的"Schema Evolution"（模式演进）？
- 如果某个数据源无法映射到 OCSF 怎么办？

### Q4：MITRE ATT&CK 框架是什么？你们在 Agent 分析告警时如何将攻击阶段映射到 ATT&CK 战术？举个具体例子。

**考察点**：MITRE ATT&CK、威胁建模

**答案要点**：
- **MITRE ATT&CK 框架**：
  - ATT&CK = Adversarial Tactics, Techniques & Common Knowledge
  - 战术（Tactics）：攻击者目标（Initial Access、Execution、Persistence、Privilege Escalation...）
  - 技术（Techniques）：实现战术的具体方法（Phishing、T1123 伪装任务计划...）
  - 共 14 个战术，约 200+ 技术
- **映射到 ATT&CK 战术**：
  1. **规则匹配**：告警规则中关联 ATT&CK Technique ID
  2. **AI 推断**：SOC Analyst Agent 分析告警内容，推断对应的战术
- **具体例子**（告警 → ATT&CK 映射）：
  ```
  告警：内网主机 192.168.1.101 向外网 IP 185.234.xx.xx 发起 HTTP 请求，
        User-Agent = "Mozilla/4.0 (compatible; MSIE 6.0)"
        请求频率 = 1000次/分钟
        目标域名为随机字符串

  推断过程：
  1. User-Agent 伪装 → T1036（伪装）
  2. 随机域名 + 高频请求 → T1568（动态域名解析）
  3. C2 通信特征 → T1071.001（应用层协议 C2）
  4. 横向移动前兆 → T1048（外泄）

  ATT&CK 战术链：
  Initial Access → Execution → Command and Control → Exfiltration
  ```
- **在 Agent 中的实现**：
  - System Prompt 注入 ATT&CK 战术知识
  - AI 分析告警后输出 `attack_stage` 字段（对应 ATT&CK 战术）

**可能的追问**：
- ATT&CK Navigator 是什么？如何在威胁分析中可视化攻击路径？
- 什么是 PRE-ATT&CK？它和 ATT&CK 有什么关系？
- ATT&CK 矩阵如何用于威胁情报评估？

### Q5：什么是告警疲劳？你们平台通过什么机制来缓解告警疲劳？

**考察点**：告警疲劳、根因分析

**答案要点**：
- **告警疲劳**：安全分析师被海量告警淹没，无法区分真正威胁，导致忽略告警甚至职业倦怠
- **缓解机制**：
  1. **关联汇总（Correlation UID）**：
     - 同一攻击事件产生的多个告警，合并为同一个 Case
     - 1000 条独立告警 → 10 个 Case，分析师只处理 10 个
  2. **Enrichment 富化**：
     - 告警接入威胁情报（IP/域名信誉）
     - 已知的内部资产白名单过滤（员工 IP、已知服务）
     - 自动降低低风险告警优先级
  3. **AI 自动研判**：
     - AI Agent 自动分析告警，自动分级（severity）
     - 高危告警直接推送，低危告警批量处理
     - 实测：自动研判后，需人工处理的告警减少 70%
  4. **动态阈值**：
     - 正常业务流量产生的告警（如批量登录），动态调整阈值
     - 节假日/业务高峰期调整规则，避免误报激增
  5. **告警分类**：
     - 按攻击类型（Phishing/Malware/Brute Force）分类，分析师按专长分工

**可能的追问**：
- 如何量化告警疲劳？（告警处理时间、忽略率、误报率）
- 如何识别"打嗝告警"（Chatter）？
- 告警优先级（Priority）如何计算？

## 10.2 平台核心对象模型

### Q6：请描述你们平台 Case、Alert、Artifact 三者的关系。为什么 Artifact 被设计为最小原子对象？

**考察点**：数据模型设计、对象关系

**答案要点**：
- **三者关系**：
  ```
  Case（案件/工单）
  ├── 聚合多个 Alert（告警）
  └── Alert（告警）
      ├── 包含多个 Artifact（痕迹/证据）
      └── 一个 Alert 至少一个 Artifact

  Artifact（痕迹/证据）← 最小原子对象
  └── 不从属于 Alert，作为独立对象存在
  ```
- **为什么 Artifact 是最小原子对象**：
  1. **复用性**：同一 IP/域名可能出现在多个 Alert 中，Artifact 独立存储避免重复
  2. **关联分析**：可以按 Artifact 维度查询（如"查询这个 IP 关联的所有 Alert"）
  3. **可追溯性**：每个 Alert 处理记录（Artifact）独立保存，便于审计
  4. **灵活性**：Alert 和 Artifact 解耦，新 Alert 类型可以复用已有 Artifact
- **模型设计原则**：
  - **Case**：业务对象，代表一个安全事件（相当于"工单"）
  - **Alert**：检测对象，由 SIEM/规则产生（相当于"报案记录"）
  - **Artifact**：证据对象，代表具体的威胁指标（IP/域名/哈希）

**可能的追问**：
- Alert 的状态机是什么样的？（New → Triaged → Investigating → Resolved）
- Alert 和 Case 之间的映射关系是一对一还是一对多？
- Artifact 的类型有哪些？（Network/Host/File/Process/Registry）

### Q7：Enrichment 的设计定位是什么？它和直接在 Alert 上加字段有什么本质区别？什么信息适合挂 Enrichment 而不适合直接写入原始字段？

**考察点**：数据模型设计、Enrichment 定位

**答案要点**：
- **Enrichment 定义**：在原始 Alert 基础上，附加的外部信息（如威胁情报、资产信息、地理位置）
- **vs 直接加字段**：
  | 维度 | 直接加字段 | Enrichment |
  |------|-----------|-----------|
  | 数据来源 | 检测系统自身 | 外部查询（API/DB） |
  | 时效性 | 实时（原生） | 异步（后补） |
  | 可变性 | 相对稳定 | 随时变化（情报更新） |
  | 关联性 | Alert 绑定 | Alert + Artifact 绑定 |
  | 存储成本 | 低 | 可选（Enrichment 可过期清理） |
- **适合 Enrichment 的信息**：
  1. **动态情报**：IP 信誉、域名 Whois、ASN 信息（随时间变化）
  2. **上下文信息**：资产归属（CMDB）、责任人、漏洞信息
  3. **AI 推理结果**：LLM 对告警的判断和置信度（可能随模型更新而变化）
- **不适合直接加字段的场景**：
  1. 数据来源不稳定（可能查询失败）
  2. 占用大量存储但使用频率低
  3. 需要定期刷新（如 IP 归属可能变化）

**可能的追问**：
- Enrichment 的生命周期如何管理？（TTL、过期清理）
- Enrichment 的数据一致性如何保证？（告警创建时 Enrichment 未返回怎么办？）
- 什么信息适合作为 Case 级别而非 Alert 级别的 Enrichment？

### Q8：Correlation UID 的汇总逻辑是怎么设计的？相同 Correlation UID 的 Alert 归并到同一个 Case 的好处是什么？

**考察点**：告警归并、关联分析

**答案要点**：
- **Correlation UID 设计**：
  - 规则维度：相同攻击源 IP + 相同攻击目标 IP + 时间窗口（30min）→ 相同 Correlation UID
  - 或：相同恶意域名 + 时间窗口 → 相同 Correlation UID
  - 公式：`correlation_uid = hash(ip_src, ip_dst, time_bucket)`
- **归并规则示例**：
  ```
  Alert A: src=1.1.1.1, dst=10.0.0.1, time=10:00 → uid="c1"
  Alert B: src=1.1.1.1, dst=10.0.0.1, time=10:15 → uid="c1"  ← 归并
  Alert C: src=1.1.1.1, dst=10.0.0.2, time=10:05 → uid="c2"  ← 目标 IP 不同，单独归并
  ```
- **归并到同一 Case 的好处**：
  1. **减少重复工作**：分析师只需处理一个 Case，无需重复处理同一个攻击事件的多条告警
  2. **全局视角**：一个 Case 包含所有相关告警，可以完整看到攻击路径
  3. **优先级提升**：同一攻击多次告警，severity 应动态提升（如 5 个相同 Alert → HIGH）
  4. **统计准确**：归并后统计真实攻击事件数，而非告警条数

**可能的追问**：
- Correlation UID 的时间窗口如何设置？太长太短各有什么问题？
- 归并后的 Case 的 Severity 如何计算？（max / 加权平均 / 计数加权）
- 如果归并规则太宽泛，会产生什么问题？（过度归并，把不同攻击合在一起）

### Q9：Knowledge 层在平台中的作用是什么？它和传统的静态知识库有什么区别？它如何同时服务人工分析员和 AI Agent？

**考察点**：知识管理、RAG 应用

**答案要点**：
- **Knowledge 层定位**：
  - Knowledge = 结构化的安全运营知识库（SOP/分析手册/威胁情报报告）
  - 格式：Markdown/JSON，存储在文档数据库或本地文件
- **vs 传统静态知识库**：
  | 维度 | 传统静态知识库 | Knowledge 层 |
  |------|--------------|-----------|
  | 访问方式 | 人工搜索/浏览 | AI 自动检索 + 人工查阅 |
  | 实时性 | 低（人工更新） | 可自动化更新 |
  | 检索方式 | 全文搜索 | 向量检索（RAG）+ 全文搜索 |
  | 与 Alert 关联 | 弱关联 | 强关联（通过 prompt 注入） |
- **同时服务人工和 AI**：
  1. **人工分析员**：通过搜索界面查阅相关 SOP/案例，作为分析参考
  2. **AI Agent**：分析告警时，通过 RAG 检索相关知识，注入到 Prompt（Retrieval Augmented Prompting）
  3. **统一内容来源**：同一份 Markdown 文档，人工和 AI 共享，无需维护两套
- **与 RAG 系统的关系**：
  - Knowledge = **结构化/静态**知识（适合直接拼接）
  - RAG = **非结构化/海量**知识（需要向量检索）

**可能的追问**：
- Knowledge 的更新频率如何？（SOP 人工更新，威胁情报 API 定时同步）
- Knowledge 如何做版本管理？（Git 管理 Markdown 文件）
- 如何评估 Knowledge 的覆盖率？（覆盖率 = 告警类型有对应知识条目的比例）

### Q10：Ticket 和外部工单系统的关系是什么？为什么不直接在平台内实现完整工单系统？

**考察点**：平台边界、系统集成

**答案要点**：
- **Ticket 的定位**：
  - Ticket = 平台内的"工单"概念，对接外部工单系统的入口
  - 不是完整的工单系统，而是**状态同步层**
- **与外部工单系统的关系**：
  ```
  Alert/Case → Ticket（平台内） → 外部工单系统（如钉钉/飞书/ServiceNow）
  ```
  - 平台内 Ticket 创建 → 同步到外部工单 → 外部处置完成后 → 同步状态回平台
  - Ticket 作为中间层，隔离平台和外部工单系统的耦合
- **不在平台内实现完整工单的原因**：
  1. **外部已有成熟工单系统**：企业已有钉钉/飞书/ServiceNow，内部打通了审批流程和人员体系
  2. **开发成本**：工单系统复杂度高（状态机/权限/审批流/报表），自研 ROI 低
  3. **平台定位**：SOC 平台专注于"安全运营"，工单系统是横向支撑工具
- **集成方式**：
  - Webhook 回调：外部工单系统通过 Webhook 通知平台 Ticket 状态变更
  - API 轮询：平台定时拉取外部工单系统状态

**可能的追问**：
- Ticket 的状态机是什么样的？（Open → In Progress → Resolved → Closed）
- 如何处理外部工单系统不可用的情况？（本地缓存 + 恢复后同步）
- Ticket 和 Case 的关系是什么？一个 Case 可以对应多个 Ticket 吗？

## 10.3 园区运营业务

### Q11：智慧园区管理平台的核心业务模块有哪些？物业管理、电商、智能硬件这几条业务线的数据流是如何打通的？

**考察点**：智慧园区业务架构、数据集成

**答案要点**：
- **核心业务模块**：
  1. **物业管理**：门禁管理、访客预约、停车管理、设施报修、工单派发
  2. **电商平台**：商品管理、订单处理、支付结算、会员管理、仓储物流
  3. **智能硬件**：IoT 设备管理（摄像头/门锁/电梯传感器）、设备状态监控、告警上报
  4. **数据看板**：运营大屏、营收报表、设备使用率报表
- **数据流打通方案**：
  ```
  [物业管理] ←── Kafka/消息队列 ──→ [数据中台]
  [电商平台] ←── Kafka/消息队列 ──→ [数据中台] ──→ [数据仓库 ODS/DWD/DWS]
  [IoT 硬件] ←── MQTT ──→ [IoT 网关] ──→ Kafka ──→ [数据中台]
                  ↓
             [实时流处理 Flink]
                  ↓
             [数据应用层]（大屏/BI/告警）
  ```
- **打通策略**：
  1. **统一消息总线**：Kafka 作为各业务线的中央事件总线，统一事件格式（如 CloudEvents 规范）
  2. **共享数据模型**：统一用户 ID（SSO）、统一设备 ID（IoT 平台）
  3. **API 网关**：BFF 层聚合多业务数据，统一对外暴露接口

**可能的追问**：

- 电商和物业的会员体系如何打通？（统一会员 ID + OpenID）
- 跨业务线的实时告警如何处理？（IoT 故障 + 物业工单 + 租户通知联动）

### Q12：你负责的数据报表业务，ODS → DWD → DWS 三层建模各自的作用是什么？为什么不能直接从原始数据出报表？

**考察点**：数据仓库建模、星型模型

**答案要点**：
- **三层建模作用**：
  - **ODS（Operational Data Store）**：原始数据层，和源表一一对应，保留原始数据，加 ETL 批次标识
  - **DWD（Data Warehouse Detail）**：明细宽表层，统一业务过程（如"物业费用缴纳"）、清洗脏数据、统一字段口径
  - **DWS（Data Warehouse Summary）**：汇总层，按主题域聚合（如"每物业项目月收入"），供 BI 直接查询
- **为什么不能直接从原始数据出报表**：
  1. **数据质量不一致**：各业务系统同一字段口径不同（如"订单金额"在物业系统和电商系统定义不同）
  2. **跨系统关联困难**：原始表结构各异，直接 JOIN 复杂且低效
  3. **性能问题**：原始表没有针对分析场景优化（无分区/索引），大报表查询极慢
  4. **口径统一**：DWD 层统一定义"什么是有效订单"、"什么是活跃用户"
- **实际报表开发流程**：
  ```
  原始数据 → ODS（保留原貌）→ DWD（清洗+打宽）→ DWS（聚合）→ BI 报表
  ```

- 什么是维度表和事实表？园区运营中有哪些典型的事实表？
- 数据仓库的 T+1 更新和实时更新如何实现？

### Q13：多业务线并行交付时，你如何协调 20+ 人的前端团队解决人力瓶颈？核心用例人工回归测试 + 缺陷晾晒的具体做法是什么？

**考察点**：团队协调、测试策略、质量把控

**答案要点**：
- **人力瓶颈问题**：
  - 20+ 前端并行开发多个业务线（物业/电商/IoT），测试资源严重不足
  - 每个业务线改动后，需要全量回归测试，但测试人员只有 3-4 人
- **核心用例人工回归**：
  1. **核心用例定义**：
     - 影响线上核心流程的用例（登录/支付/门禁放行）
     - 每个业务线定义 Top 20 核心用例
     - 总计 60 个核心用例，每次迭代全量回归
  2. **自动化优先**：核心用例优先自动化（Playwright E2E），自动化覆盖率 > 80%
  3. **人工测试聚焦**：人工测试聚焦在探索性测试和新功能测试
- **缺陷晾晒机制**：
  1. **每日站会 + 缺陷板**：各业务线在钉钉群/飞书群更新缺陷数/缺陷状态
  2. **缺陷数上墙**：用看板工具（如 Jira/PingCode）公开各前端团队的缺陷数
  3. **缺陷率考核**：引入"缺陷逃逸率"指标（测试阶段发现缺陷 vs 上线后客户发现缺陷）
  4. **复盘机制**：每迭代结束后复盘逃逸缺陷，找出根因（测试盲点/需求变更未通知）
- **效果**：
  - 核心用例自动化率 > 80% 后，回归测试时间从 2 天缩短到 4 小时
  - 缺陷逃逸率从 15% 降到 5%

**可能的追问**：
- 如何协调前后端联调的资源冲突？（联调优先级排序 + Mock 服务隔离）
- 前端代码质量如何保证？（ESLint + Prettier + CI 卡点）
- 多业务线共用的组件如何管理？（Monorepo / 私有 npm 包）
