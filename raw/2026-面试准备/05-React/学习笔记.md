## 一、React 基础概念

### 1.1 什么是 React

React 是 Facebook（现 Meta）于 2013 年开源的 **UI 库**，专注于构建用户界面。它的核心思想是：**用声明式的方式描述 UI，React 负责计算出如何高效地将 UI 更新到真实 DOM 上**。

与直接操作 DOM 的命令式编程相比，声明式编程让我们只关心"UI 应该是什么样子"，而不必关心"如何一步步修改 DOM"。

### 1.2 组件化：React 的核心思想

组件（Component）是 React 的基本构建单元。每个组件都是一段独立的 UI 逻辑，它可以接收数据（props），内部维护状态（state），并渲染出 UI。

**为什么组件化重要？**
- **复用性**：相同的 UI 逻辑可以复用，避免重复代码
- **可维护性**：每个组件职责单一，出了问题容易定位
- **可测试性**：组件可以独立测试

```jsx
// 函数组件示例：最简单的 React 组件
function Welcome({ name }) {
  return <h1>Hello, {name}</h1>;
}

// 使用组件
const element = <Welcome name="SOC Analyst" />;
// 渲染结果：<h1>Hello, SOC Analyst</h1>
```

### 1.3 声明式 vs 命令式

| 对比维度 | 命令式（Imperative） | 声明式（Declarative） |
|----------|---------------------|---------------------|
| 思路 | "如何做"：一步步告诉计算机怎么做 | "做什么"：描述期望的结果 |
| DOM 操作 | 手动获取元素、修改属性、添加事件 | 描述 UI 状态，React 自动更新 |
| 代码示例 | `document.getElementById('x').textContent = 'hi'` | `<div>{text}</div>` |
| 调试难度 | 高（状态散落在各处） | 低（状态即 UI，UI 即状态） |

**实战场景：** 在 SOC 系统告警列表中，告警的"已读/未读"状态如果用命令式写，需要手动追踪哪些 DOM 节点需要更新；用声明式只需修改状态，React 自动处理差异。

### 1.4 Virtual DOM（虚拟 DOM）的工作原理

这是 React 最核心的面试知识点，必须深入理解。

**Virtual DOM 是什么？**
虚拟 DOM 是真实 DOM 的 JavaScript 对象表示。React 在内存中维护两棵虚拟 DOM 树：
- `current`（当前树）
- `workInProgress`（正在构建的新树）

**为什么需要 Virtual DOM？**
直接操作真实 DOM 是最昂贵的 Web 操作之一。真实 DOM 涉及：
- 浏览器布局计算（Layout）
- 重新绘制（Paint）
- 触发重排（Reflow）

虚拟 DOM 的批处理和 diff 算法可以把多次 DOM 修改合并成一次最小化更新，大幅提升性能。

**Virtual DOM 的工作流程：**

```
用户交互 → setState() → 创建新虚拟 DOM 树
                            ↓
                    与旧树进行 Diff 对比
                            ↓
                    计算出最小更新集（Patch）
                            ↓
                    批量 commit 到真实 DOM
```

```jsx
// 实际代码层面：React 内部帮你做了这些
// 你写的：
function Counter() {
  const [count, setCount] = useState(0);
  return <button onClick={() => setCount(c => c + 1)}>{count}</button>;
}

// React 内部做的事情（伪代码）：
// 1. 触发 setState，创建新的虚拟 DOM
// newVirtualDOM = { type: 'button', props: { onClick: ..., children: 1 } }
// 2. Diff: 新虚拟DOM vs 旧虚拟DOM
// 3. 发现只有文本节点从 "0" 变成 "1"，只更新文本节点
// 4. 真实 DOM 只做一次最小化更新
```

**Diff 算法的核心策略（让复杂度降为 O(n)）：**

1. **同层对比**：只比较同一层级的节点，不同则直接替换子树。不会跨层级比较（想像一个人从第5楼换到第3楼，整棵树都得换）
2. **类型对比**：标签名不同（如 `<div>` 变成 `<span>`）直接替换整棵树
3. **Key 匹配**：列表中用 `key` 识别元素，允许跨位置移动而不是重建（详见 JSX 列表渲染部分）

## 二、JSX 语法详解

### 2.1 JSX 的本质

JSX 并不是 HTML，而是一种 JavaScript 语法扩展。React 源码中定义了 `createElement` 函数来构建虚拟 DOM 对象，JSX 只是它的语法糖。

```jsx
// JSX 写法（你写的）
const element = <h1 className="title">Hello, SOC</h1>;

// Babel 编译后（实际上运行的）
const element = React.createElement(
  'h1',
  { className: 'title' },
  'Hello, SOC'
);

// React.createElement 返回的对象结构（虚拟 DOM 节点）
// {
//   type: 'h1',
//   props: { className: 'title', children: 'Hello, SOC' },
//   key: null,
//   ref: null
// }
```

这就是为什么在 JSX 文件顶部通常需要 `import React from 'react'`——在老版本 React 中，`JSX` 会被编译成对 `React.createElement` 的调用。

### 2.2 嵌入表达式

JSX 中用 `{}` 包裹 JavaScript 表达式：

```jsx
const user = { name: 'Alice', role: 'Security Engineer' };
const isAdmin = true;

// 支持：变量、表达式、三元运算符、函数调用
const greeting = (
  <div>
    <h1>{user.name}</h1>                        {/* 变量 */}
    <p>状态: {isAdmin ? '管理员' : '普通用户'}</p>  {/* 三元 */}
    <p>角色长度: {user.role.length}</p>          {/* 表达式 */}
    <p>今日时间: {new Date().toLocaleDateString()}</p> {/* 函数调用 */}
  </div>
);
```

**注意：** `{}` 内只能放表达式，不能放语句（如 `if/for/while`）。

### 2.3 条件渲染

React 中没有专门的 `if/else` 语法，用 JavaScript 表达式实现：

```jsx
function AlertPanel({ alerts, severity }) {
  // 方式1：三元运算符（适合简单条件）
  const status = severity === 'critical' ? '严重' : '普通';

  // 方式2：短路求值（&&）（适合条件显示/隐藏）
  // 场景：没有告警时显示空状态
  return (
    <div>
      <h2>告警面板 {status}</h2>
      {alerts.length === 0 && <p>暂无告警</p>}
      {/* ⚠️ 注意：左边是数字0时会被渲染！0 && <Component /> 渲染为 0 */}
      {/* 正确做法：alerts.length > 0 && ... 或 Boolean(alerts.length) && ... */}

      {/* 方式3：三元嵌套（适合多条件） */}
      {severity === 'critical' ? (
        <div className="alert-critical">紧急告警！</div>
      ) : severity === 'warning' ? (
        <div className="alert-warning">警告</div>
      ) : (
        <div className="alert-info">正常</div>
      )}

      {/* 方式4：抽取为组件（适合复杂逻辑） */}
      <AlertContent severity={severity} alerts={alerts} />
    </div>
  );
}
```

### 2.4 列表渲染与 Key 的重要性

渲染列表时，React 需要唯一 `key` 来识别每个元素：

```jsx
const securityAlerts = [
  { id: 'A001', type: '入侵检测', level: '高' },
  { id: 'A002', type: '异常登录', level: '中' },
  { id: 'A003', type: '恶意软件', level: '高' },
];

function AlertList({ alerts }) {
  return (
    <ul>
      {/* ⚠️ 错误做法：使用 index 作为 key */}
      {alerts.map((alert, index) => (
        <li key={index}>{alert.type} - {alert.level}</li>
      ))}
      {/* 正确做法：使用唯一 ID */}
      {alerts.map(alert => (
        <li key={alert.id}>{alert.type} - {alert.level}</li>
      ))}
    </ul>
  );
}
```

**Key 为什么这么重要？**

没有 key 或 key 不稳定时，React 的 Diff 算法会把列表中所有元素当成新创建的，导致：
- 组件状态丢失（如输入框内容、表单数据）
- 动画重置
- 性能恶化（无法复用已有 DOM 节点）

```jsx
// 举例：没有 key 导致的问题
// 初始: [AlertA, AlertB, AlertC]
// 更新: [AlertB, AlertA, AlertC]  (顺序变了，但内容没变)

// 无 key 时：React 认为全部重建 → 3个组件全部重新挂载
// 有 key 时：React 识别出 AlertA/AlertB 只是换了位置 → 复用 DOM，只做移动
```

## 三、Props 与 State

### 3.1 Props：组件的输入参数

Props 是组件对外的接口，父组件通过 props 向子组件传递数据。**Props 是只读的**——React 组件必须像纯函数一样使用 props，不能修改自己的 props。

```jsx
// 父组件
function App() {
  const alertData = {
    title: '检测到异常行为',
    count: 5,
    timestamp: new Date().toISOString()
  };

  return <AlertCard {...alertData} type="security" />;
  // 等价于：
  // return <AlertCard title={alertData.title} count={alertData.count} ... />
}

// 子组件：接收 props 作为函数参数
function AlertCard({ title, count, timestamp, type }) {
  return (
    <div className={`alert-card alert-${type}`}>
      <h3>{title}</h3>
      <span>告警数: {count}</span>
      <time>{timestamp}</time>
    </div>
  );
}
```

**Props 传递函数（子组件通知父组件）：**

```jsx
function Parent() {
  const [selectedId, setSelectedId] = useState(null);

  return (
    <AlertList
      onSelect={(id) => setSelectedId(id)}  // 父组件提供回调
    />
  );
}

function AlertList({ onSelect }) {
  const alerts = useAlerts(); // 假设这是一个自定义 Hook

  return (
    <ul>
      {alerts.map(alert => (
        <li key={alert.id}>
          <button onClick={() => onSelect(alert.id)}>
            {alert.title}
          </button>
        </li>
      ))}
    </ul>
  );
}
```

### 3.2 State：组件的内部状态

State 是组件私有、可变的数据。当 state 变化时，React 会触发组件重新渲染（Re-render）。

```jsx
function AlertBadge() {
  // useState 返回 [当前值, 更新函数]
  const [count, setCount] = useState(0); // 初始值为 0

  return (
    <button onClick={() => setCount(count + 1)}>
      未处理告警: {count}
    </button>
  );
}
```

**State 的核心原则：**

1. **不可变性（Immutability）**：永远不要直接修改 state 对象
2. **同步 vs 异步**：React 的 `setState` 在大多数情况下是**异步**的（批量更新）

```jsx
// ⚠️ 错误：直接修改 state
const [user, setUser] = useState({ name: 'Alice', loggedIn: false });
user.loggedIn = true; // React 不会感知到变化，不会重新渲染！

// ✅ 正确：创建新对象（展开运算符）
setUser({ ...user, loggedIn: true });

// ⚠️ 错误：对数组直接 push
const [items, setItems] = useState([]);
items.push(newItem); // 直接修改原数组，React 检测不到

// ✅ 正确：创建新数组
setItems([...items, newItem]);
```

### 3.3 this.setState 的异步性

这是最容易踩坑的知识点。在 React 18 之前，`setState` 是异步的（批量更新），在事件处理器和生命周期函数中表现尤为明显。

```jsx
// 问题场景：连续调用 setState
this.setState({ count: this.state.count + 1 });
this.setState({ count: this.state.count + 1 });
// 预期: count += 2，实际: count += 1（两个 setState 共享同一个 state）

// ✅ 解决方案1：函数式更新
this.setState(prevState => ({ count: prevState.count + 1 }));
this.setState(prevState => ({ count: prevState.count + 1 }));
// 两次调用后: count += 2

// ✅ 解决方案2（React 18+）：批量更新更智能，无需担心
setCount(c => c + 1);
setCount(c => c + 1);
// 结果同样是 count += 2，且性能更好
```

```jsx
// ⚠️ 注意：setTimeout 内的 setState
// React 17 之前不批量处理，React 18 全部批量
setTimeout(() => {
  setCount(c => c + 1); // 18后：仍批量，性能好
  setName('Alice');      // 18后：仍批量，性能好
}, 0);

// ⚠️ 注意：原生事件中的 setState（addEventListener）
// React 17 之前：原生事件中 setState 不批量，每次都触发重渲染
// React 18：全部批量
```

### 3.3 State 与 Props 的区别

| 对比维度 | Props | State |
|----------|-------|-------|
| 所有者 | 父组件传入 | 组件自己拥有 |
| 可修改性 | 只读（不可修改） | 可修改（通过 setState） |
| 作用范围 | 父子间传递 | 组件内部私有 |
| 变化时 | 触发重新渲染 | 触发重新渲染 |
| 初始化 | 父组件决定 | 组件内部设置默认值 |
| 用途 | 配置项、外部数据 | UI 状态、内部数据 |

## 四、生命周期（旧版类组件）

### 4.1 为什么还要学生命周期

虽然新代码推荐用函数组件 + Hooks，但面试中**类组件的生命周期**仍然是高频考点，理解它有助于理解 React 的渲染机制。而且很多老项目仍在使用类组件。

### 4.2 完整生命周期图解

```
挂载阶段（Mount）
─────────────────────────────────────────────────────────
constructor() → render() → React DOM 更新 → componentDidMount()
                                                         ↓
更新阶段（Update）←────────────┐
    │                         │
shouldComponentUpdate() ──(true)──→ render() → getSnapshotBeforeUpdate()
    │                                    ↓
    ↓(false)                     React DOM 更新
                               componentDidUpdate()
                                  ↑___________│

卸载阶段（Unmount）
─────────────────────────────────────────────────────────
componentWillUnmount()
```

### 4.3 各阶段详解

**挂载阶段（Mounting）：**

```jsx
class SecurityDashboard extends React.Component {
  constructor(props) {
    super(props); // ⚠️ 必须调用 super(props)，否则 this.props 为 undefined
    this.state = { alerts: [] };
    // 适合：初始化 state、绑定方法
  }

  componentDidMount() {
    // ✅ 最佳时机：发起网络请求、设置订阅、启动定时器
    this.fetchAlerts();
    this.ws = new WebSocket('wss://soc-system/alerts');
    this.ws.onmessage = (event) => {
      this.setState({ alerts: [...this.state.alerts, JSON.parse(event.data)] });
    };
  }

  render() {
    // 必须实现：返回 JSX 或 null/false/数字/数组/Portal
    return <div>告警数量: {this.state.alerts.length}</div>;
  }
}
```

**更新阶段（Updating）：**

```jsx
class AlertItem extends React.Component {
  // 决定是否重新渲染（性能优化的最后防线）
  shouldComponentUpdate(nextProps, nextState) {
    // 比较新旧 props/stste，返回 false 则跳过 render
    return nextProps.alert.id !== this.props.alert.id ||
           nextProps.alert.status !== this.props.alert.status;
    // ⚠️ 不要做深度比较，代价太高，除非你确定有这个必要
  }

  getSnapshotBeforeUpdate(prevProps, prevState) {
    // DOM 更新前获取信息（如滚动位置）
    // 必须和 componentDidUpdate 配合使用
    return this.listRef ? this.listRef.scrollHeight : null;
  }

  componentDidUpdate(prevProps, prevState, snapshot) {
    // ✅ 适合：DOM 操作（聚焦、滚动）、网络请求（有条件判断）
    if (prevProps.alert.status !== this.props.alert.status) {
      this.playSound(); // 告警状态变化时播放提示音
    }
    // ⚠️ 注意：如果要 setState，必须放在条件判断内，否则会死循环
  }
}
```

**卸载阶段（Unmounting）：**

```jsx
class AlertItem extends React.Component {
  componentWillUnmount() {
    // ✅ 必须清理：取消订阅、清除定时器、取消请求
    if (this.ws) {
      this.ws.close();
    }
    clearInterval(this.timer);
    this.abortController?.abort(); // 取消未完成的请求
  }
}
```

### 4.4 React 16.3+ 重大变化

React 16.3 废弃了三个"will"前缀的生命周期：
- `componentWillMount` — 异步渲染时会在渲染中途被调用，造成数据不一致
- `componentWillReceiveProps` — 逻辑混乱，容易产生 bugs
- `componentWillUpdate` — 和 `componentWillReceiveProps` 有类似问题

新增了两个安全替代：
- `static getDerivedStateFromProps` — 每次 render 前调用，适合从 props 派生 state
- `getSnapshotBeforeUpdate` — DOM 变更前获取快照，适合保留滚动位置等

```jsx
// getDerivedStateFromProps 示例
static getDerivedStateFromProps(nextProps, prevState) {
  // 当父组件传入的 alertId 变化时，更新当前告警数据
  if (nextProps.alertId !== prevState.lastAlertId) {
    return {
      currentAlert: nextProps.alerts.find(a => a.id === nextProps.alertId),
      lastAlertId: nextProps.alertId,
    };
  }
  return null; // 不需要更新
}
```

## 五、Hooks 核心

### 5.1 Hooks 简介

Hooks 是 React 16.8 引入的新特性，让函数组件能够使用 state 和其他 React 特性。Hooks 的本质是**让你在函数组件里"钩入"React 状态和生命周期特性**。

**Hooks 的规则（必须遵守）：**
1. 只在组件顶层调用，不要在循环、条件、嵌套函数中调用
2. 只在 React 函数组件或自定义 Hook 中调用

```jsx
// ✅ 正确：顶层调用
function AlertPanel() {
  const [status, setStatus] = useState('active'); // 顶层
  if (status === 'loading') return <Loading />;     // 条件返回 OK
  const msg = useNotification();                    // 顶层，OK
  return <div>...</div>;
}

// ⚠️ 错误：在条件中调用 Hook
function AlertPanel() {
  if (condition) {
    const [a, setA] = useState(0); // 错误！可能在某些渲染中跳过
  }
}
```

### 5.2 useState：最基础的 Hook

```jsx
function AlertCounter() {
  const [count, setCount] = useState(0);   // 泛型推断为 number
  // const [count, setCount] = useState<number>(0); // 显式类型

  return (
    <button onClick={() => setCount(c => c + 1)}>
      告警数: {count}
    </button>
  );
}
```

**函数式更新（重要）：**

```jsx
// 场景：处理数组中的复杂状态
function AlertList() {
  const [alerts, setAlerts] = useState([]);

  const addAlert = (newAlert) => {
    // ⚠️ 不要这样做：依赖外部 state 的 setState
    // setAlerts([...alerts, newAlert]);

    // ✅ 正确：使用函数式更新
    setAlerts(prev => [...prev, newAlert]);
  };

  const markAsRead = (id) => {
    setAlerts(prev =>
      prev.map(alert =>
        alert.id === id ? { ...alert, read: true } : alert
      )
    );
  };

  return (
    <ul>
      {alerts.map(a => (
        <li key={a.id} onClick={() => markAsRead(a.id)}>
          {a.message} {a.read ? '✓' : ''}
        </li>
      ))}
    </ul>
  );
}
```

**初始化函数（延迟初始化）：**

```jsx
// ⚠️ 昂贵计算不要写在 useState 参数里（每次渲染都执行）
const [data, setData] = useState(expensiveComputation(props.filter));
// expensiveComputation 在每次渲染时都运行！

// ✅ 正确：传入函数，React 只在初始化时调用一次
const [data, setData] = useState(() => expensiveComputation(props.filter));
```

### 5.3 useEffect：处理副作用

`useEffect` 是 React 中处理"副作用"（Side Effects）的钩子。副作用包括：网络请求、DOM 操作、订阅、定时器、打印日志等。

**依赖数组的三种模式：**

```jsx
function AlertFeed({ filter }) {
  useEffect(() => {
    // 模式1：无依赖 —— 每次渲染后都执行
    console.log('组件渲染了，当前筛选条件:', filter);

    // ⚠️ 如果需要访问最新值，可以使用 ref（见 useRef 部分）
  });

  useEffect(() => {
    // 模式2：空数组 [] —— 只在挂载时执行一次（类似 componentDidMount）
    const ws = new WebSocket('wss://soc/alerts');
    ws.onmessage = handleMessage;

    // ✅ cleanup 函数：组件卸载时清理（类似 componentWillUnmount）
    return () => {
      ws.close();
    };
  }, []); // 空数组 = 没有依赖 = 只执行一次

  useEffect(() => {
    // 模式3：有依赖 [dep] —— dep 变化时执行
    // 适合：发起基于筛选条件的 API 请求
    const controller = new AbortController();

    fetch(`/api/alerts?filter=${filter}`, { signal: controller.signal })
      .then(res => res.json())
      .then(data => setAlerts(data));

    // cleanup：下次 effect 执行前 + 卸载时调用
    return () => controller.abort();
  }, [filter]); // filter 变化 → 重新请求

  return <AlertList alerts={alerts} />;
}
```

### 5.4 useEffect 的常见问题

**问题1：无限循环**

```jsx
// ⚠️ 错误代码示例（造成无限循环）
function AlertDashboard({ onAlert }) {
  const [alerts, setAlerts] = useState([]);

  useEffect(() => {
    // ❌ 在 effect 内更新了 alerts，而 alerts 在依赖数组中
    // → setAlerts → alerts 变化 → 再次触发 effect → 无限循环！
    setAlerts(prev => [...prev, onAlert]);
  }, [alerts, onAlert]); // onAlert 是新函数引用，每次渲染都变！
}
```

```jsx
// ✅ 修复方案
useEffect(() => {
  if (onAlert) {
    setAlerts(prev => [...prev, onAlert]);
  }
  // onAlert 依赖问题：使用 useCallback 包裹，或从依赖中移除
}, [onAlert.id]); // 只关心 id，不关心函数引用变化

// 或者更好的方案：事件驱动
useEffect(() => {
  const handleAlert = (alert) => setAlerts(prev => [...prev, alert]);
  eventBus.on('NEW_ALERT', handleAlert);
  return () => eventBus.off('NEW_ALERT', handleAlert);
}, []); // 不依赖具体数据，组件卸载时清理
```

**问题2：缺少依赖导致 stale closure（闭包陷阱）**

```jsx
function AlertTimer() {
  const [count, setCount] = useState(0);

  useEffect(() => {
    const timer = setInterval(() => {
      console.log('Count 现在是:', count);
      // ⚠️ 问题：这里的 count 永远是初始值 0！
      // 因为闭包捕获了创建时的 count 值
    }, 1000);

    return () => clearInterval(timer);
  }, []); // 缺少 count 依赖 → 只创建一次 timer，count 永远是 0

  return <button onClick={() => setCount(c => c + 1)}>{count}</button>;
}
```

```jsx
// ✅ 修复方案：使用函数式更新
useEffect(() => {
  const timer = setInterval(() => {
    setCount(c => c + 1); // ✅ 不依赖外部 count 值
  }, 1000);
  return () => clearInterval(timer);
}, []); // ✅ 空依赖 OK，因为 setCount 函数式更新不依赖外部值
```

### 5.5 useRef

`useRef` 返回一个可变的 ref 对象，其 `.current` 属性可以存储任意值，且**修改 ref 不会触发重新渲染**。

```jsx
function AlertInput() {
  const inputRef = useRef(null);
  // useRef<HTMLInputElement>(null) —— 泛型指定 DOM 元素类型
  const countRef = useRef(0); // 也可以存普通值，不触发重渲染

  const focusInput = () => {
    inputRef.current?.focus(); // 安全调用，因为 current 初始为 null
  };

  useEffect(() => {
    // 组件挂载后自动聚焦
    inputRef.current?.focus();
  }, []);

  return (
    <div>
      {/* inputRef 连接到真实 DOM 元素 */}
      <input ref={inputRef} placeholder="输入告警关键词" />
      <button onClick={focusInput}>聚焦</button>
    </div>
  );
}
```

**useRef 的两个核心用途：**

| 用途 | 说明 | 示例 |
|------|------|------|
| DOM 引用 | 获取/操作真实 DOM | `inputRef.current.focus()` |
| 存储可变值 | 跨渲染保存值，不触发重渲染 | 计时器计数、previous props |

### 5.6 useCallback vs useMemo

这是面试中最容易混淆的 Hooks 对比。

```jsx
// useCallback：缓存函数
// useMemo：缓存计算结果

function AlertList({ alerts, filterId }) {
  // ⚠️ 每次渲染都创建新函数 → 子组件 props 变化 → 子组件重渲染
  const handleClick = (id) => {
    console.log('Clicked:', id);
  };

  // ✅ 缓存函数，只有 filterId 变化时才创建新函数
  const handleClick = useCallback((id) => {
    console.log('Clicked:', id);
  }, [filterId]);

  // ✅ 缓存 expensiveComputation 的结果
  // 只有 alerts 或 filterId 变化时才重新计算
  const filteredAlerts = useMemo(
    () => alerts.filter(a => a.filterId === filterId),
    [alerts, filterId]
  );

  return (
    <ul>
      {filteredAlerts.map(alert => (
        // MemoizedListItem 用 React.memo 包裹
        // 传入的 handleClick 不变 → 不会重渲染
        <MemoizedListItem
          key={alert.id}
          alert={alert}
          onClick={handleClick}
        />
      ))}
    </ul>
  );
}
```

**对比表格：**

| 维度 | useCallback | useMemo |
|------|-------------|---------|
| 缓存内容 | 函数 | 计算值（任意类型） |
| 本质 | `useMemo(fn, deps)` 的语法糖 | `useMemo(() => value, deps)` |
| 适用场景 | 传递给子组件的回调函数 | 昂贵的计算结果 |
| 滥用后果 | 过度优化，增加内存 | 过度优化，增加内存 |
| 判断标准 | "这个函数要不要作为 props 传给 React.memo 组件？" | "这个计算是否昂贵且可能被重复执行？" |

**什么时候不用 useCallback/useMemo？**

```jsx
// ⚠️ 过度使用 —— 反而浪费性能
const value = useMemo(() => x + y, [x, y]); // 简单加法不需要 memo
const fn = useCallback(() => x + y, [x, y]); // 简单函数不需要 memo

// ✅ 真正需要的场景：
// 1. 函数传给用 React.memo 包裹的子组件
// 2. 函数作为 useEffect 的依赖（稳定引用很重要）
// 3. 昂贵的计算（大数据排序、过滤、复杂计算）
```

### 5.7 useContext：跨层级数据传递

Context 解决了"prop drilling"（层层传递 props）的问题：

```jsx
// 创建 Context（通常单独一个文件）
import { createContext, useContext } from 'react';

export const ThemeContext = createContext({
  theme: 'dark',
  toggleTheme: () => {},
```

**Context 的注意事项：**

```jsx
// ⚠️ Context 陷阱：value 每次都是新对象！
// 每次 App 重渲染，Provider value 是新对象 → 所有 Consumer 重渲染
function App() {
  return (
    <ThemeContext.Provider value={{ theme: 'dark' }}>
      {/* ⚠️ 错误：value 对象字面量每次渲染都是新的！
           解决：用 useMemo 缓存 value */}
      <Dashboard />
    </ThemeContext.Provider>
  );
}

function AppFixed() {
  const [theme, setTheme] = useState('dark');
  const value = useMemo(
    () => ({ theme, toggleTheme: () => setTheme(t => t === 'dark' ? 'light' : 'dark') }),
    [theme]

  return (
    <ThemeContext.Provider value={value}>
      <Dashboard />
    </ThemeContext.Provider>
  );
}
```

### 5.8 useReducer：复杂状态逻辑

当 state 逻辑复杂（有多个子值、涉及状态转换）时，`useReducer` 比 `useState` 更适合：

```jsx
const initialState = { status: 'idle', alerts: [], error: null };

function alertsReducer(state, action) {
  switch (action.type) {
    case 'FETCH_START':
      return { ...state, status: 'loading', error: null };
    case 'FETCH_SUCCESS':
      return { ...state, status: 'success', alerts: action.payload };
    case 'FETCH_ERROR':
      return { ...state, status: 'error', error: action.error };
    case 'ADD_ALERT':
      return { ...state, alerts: [...state.alerts, action.payload] };
    case 'MARK_READ':
      return {
        ...state,
        alerts: state.alerts.map(a =>
          a.id === action.id ? { ...a, read: true } : a
        ),
      };
    default:
      throw new Error(`Unknown action: ${action.type}`);
  }
}

function AlertManager() {
  const [state, dispatch] = useReducer(alertsReducer, initialState);

  useEffect(() => {
    dispatch({ type: 'FETCH_START' });
    fetch('/api/alerts')
      .then(res => res.json())
      .then(data => dispatch({ type: 'FETCH_SUCCESS', payload: data }))
      .catch(err => dispatch({ type: 'FETCH_ERROR', error: err.message }));
  }, []);

  return (
    <div>
      {state.status === 'loading' && <Spinner />}
      {state.status === 'error' && <ErrorBanner message={state.error} />}
      {state.alerts.map(alert => (
        <AlertItem
          key={alert.id}
          alert={alert}
          onMarkRead={(id) => dispatch({ type: 'MARK_READ', id })}
        />
      ))}
    </div>
  );
}
```

## 六、虚拟 DOM 与 Fiber 架构

### 6.1 Fiber 架构的诞生背景

React 15 及之前版本的渲染是**同步且不可中断**的。这意味着：
- 一次大规模渲染（如渲染10000条告警）会阻塞主线程
- 用户交互（点击、输入）必须等待渲染完成才能响应
- 在低性能设备上可能造成明显的卡顿

React 16 引入了 **Fiber** 架构来解决这些问题。Fiber 的核心目标：**把渲染工作拆分成小单元，可以暂停、恢复、优先处理紧急任务**。

### 6.2 Fiber 的数据结构

每个虚拟 DOM 节点在 Fiber 架构中对应一个 Fiber 对象：

```javascript
// Fiber 节点的关键字段
{
  type: 'div',                    // 元素类型
  key: null,                      // 列表中的唯一标识
  stateNode: <actual DOM node>,   // 指向真实 DOM 节点的引用

  // 树结构
  child: fiberNode,               // 第一个子节点
  sibling: fiberNode,             // 下一个兄弟节点
  return: fiberNode,              // 父节点

  // 双缓存树
  alternate: fiberNode,           // 指向另一棵树中对应的节点

  // 更新相关
  pendingProps: {},               // 新的 props
  memoizedProps: {},              // 已提交的 props
  memoizedState: {},              // 已提交的 state

  // 副作用标记
  flags: Update | Placement | Deletion,  // 需要执行的 DOM 操作
  effectTag: ...,                          // 具体是什么操作

  // 渲染阶段信息
  lanes: 0b0010,               // 优先级车道（React 18）
  childLanes: 0b0010,
}
```

### 6.3 双缓存树（Double Buffering）

React 在内存中同时维护两棵 Fiber 树：
- `current`（当前显示的树，对应屏幕上的真实 DOM）
- `workInProgress`（正在构建的新树）

```
当前渲染帧：
    current Fiber 树（屏幕可见）
         ↕ (通过 alternate 引用)
  workInProgress Fiber 树（内存中构建）

构建完成后：
    workInProgress → 变成新的 current（一次性提交到屏幕）
    旧的 current → 变成新的 workInProgress（供下次更新用）
```

**为什么这样做？**
- 用户始终看到一致的 UI（current 树保证）
- 新树的构建可以随时暂停，不影响当前显示
- 构建完成后通过切换指针（Root fiber 的 `current` 指向）完成更新

### 6.4 Reconciliation（协调）与 Diff 算法

Reconciliation 是 React 决定如何高效更新真实 DOM 的过程。Diff 算法是其中的核心策略。

**Diff 算法的三条核心规则：**

```jsx
// 规则1：不同类型元素 → 卸载旧树，重建新树
<div>                        <span>     // div → span，不同类型
  <Alert />                    Hello!
</div>                      </span>
// React 不会尝试比较 div 和 span，会直接卸载整个子树

// 规则2：同类型元素 → 保留 DOM 节点，只更新属性
<div className="old">        <div className="new">  // 同为 div
  <Alert />                    <Alert />
</div>                       </div>
// React 保留 div，只更新 className，保留子节点

// 规则3：列表元素用 key 识别 → 尽量复用
// key="1" 和 key="2" 告诉 React 这是"同一个元素"，只是位置变了
```

**为什么 key 很重要？**

```jsx
// 场景：告警列表中插入一条新告警
// 原来: [Alert1, Alert2, Alert3]  (key: "1","2","3")
// 现在: [NewAlert, Alert1, Alert2, Alert3]  (key: "new","1","2","3")

// ❌ 用 index：无 key 时 React 只能假设位置对应
// index 0 → NewAlert (原来 Alert1 的内容被替换！)
// index 1 → Alert1 (原来 Alert2 的内容...)
// ⚠️ 结果：所有告警的内容都变了，状态丢失

// ✅ 用唯一 ID：
// key="new" → 这是新元素，插入
// key="1" → Alert1 识别出自己，移到位置1
// key="2" → Alert2 识别出自己，移到位置2
// 只有新增的元素重建，DOM 复用最大化
```

### 6.5 React 16 到 React 18 的 Scheduler 演进

| 版本 | Scheduler | 说明 |
|------|-----------|------|
| React 16 | 基于 `setTimeout` 的退避方案 | 不稳定，有兼容性问题 |
| React 17 | 继续改进，稳定的异步模式 | Concurrent Mode 雏形 |
| React 18 | **全新 Scheduler（基于 MessageChannel）** | 精确控制优先级，与浏览器帧协调 |

## 七、状态管理方案对比

### 7.1 什么时候需要状态管理库

不是所有项目都需要 Redux。以下是判断标准：

### 7.3 Context vs Redux 使用场景

```jsx
// 场景：SOC 系统的主题切换（适合 Context）

// 场景：告警列表的多维度筛选（适合 Redux/Zustand）

// 更新频率：高（每次筛选条件变化影响多个组件）
// ✅ 适合集中管理，配合 Selector 实现精准订阅
```

### 7.4 Zustand 示例

```jsx
import { create } from 'zustand';
import { devtools, subscribeWithSelector } from 'zustand/middleware';

const useAlertStore = create(
  devtools(
    subscribeWithSelector((set, get) => ({
      alerts: [],
      filter: { severity: 'all', search: '' },
      selectedId: null,

      // Action
      setAlerts: (alerts) => set({ alerts }),

      // Selector + Action 合一
      addAlert: (alert) =>
        set(state => ({ alerts: [...state.alerts, alert] })),

      // 带逻辑的 action
      markAsRead: (id) =>
        set(state => ({
          alerts: state.alerts.map(a =>
            a.id === id ? { ...a, read: true } : a
          ),
        })),

      // 计算属性（Zustand 推荐在组件中用 useMemo）
      getFilteredAlerts: () => {
        const { alerts, filter } = get();
        return alerts.filter(a =>
          (filter.severity === 'all' || a.severity === filter.severity) &&
          a.message.includes(filter.search)
        );
      },
    }))
  )
);

// 在组件中使用
function AlertList() {
  const alerts = useAlertStore(state => state.alerts);
  const filter = useAlertStore(state => state.filter);
  const getFilteredAlerts = useAlertStore(state => state.getFilteredAlerts);

  const filtered = getFilteredAlerts();
  return <ul>{filtered.map(a => <li key={a.id}>{a.message}</li>)}</ul>;
}
```

## 八、性能优化

### 8.1 重新渲染的原因

理解"什么时候会重新渲染"是性能优化的基础：

```jsx
// 触发重新渲染的几种情况：
// 1. 组件自身调用了 setState
// 2. 父组件重新渲染（即使 props 没变）
// 3. Context Provider 的 value 变化
// 4. 组件使用了 useState / useReducer（所属的组件重渲染）

// 不触发重新渲染的：
// - 父组件的 state 变化，但该组件用 React.memo 且 props 不变
// - 父组件渲染，但子组件 props 的引用未变（React.memo 保护）
```

### 8.2 React.memo 与 PureComponent

```jsx
// 函数组件：React.memo
const AlertItem = React.memo(function AlertItem({ alert, onMarkRead }) {
  // ✅ 只有当 alert 或 onMarkRead 引用变化时才重新渲染
  return (
    <div className={`alert-${alert.severity}`}>
      <span>{alert.message}</span>
      <button onClick={() => onMarkRead(alert.id)}>标为已读</button>
    </div>
  );
});

// 自定义比较函数（慎用，默认浅比较足够）
const AlertItem = React.memo(
  function AlertItem({ alert, onMarkRead }) { ... },
  (prevProps, nextProps) => {
    // 返回 true = 不重新渲染（认为 props 相等）
    return prevProps.alert.id === nextProps.alert.id &&
           prevProps.alert.read === nextProps.alert.read;
    // ⚠️ 深度比较开销大，一般不用
  }
);
```

**为什么 props 引用不变很重要？**

```jsx
// ⚠️ 每次渲染，父组件都创建新函数 —— React.memo 白用
function AlertList({ alerts }) {
  return (
    <div>
      {alerts.map(alert => (
        <AlertItem
          key={alert.id}
          alert={alert}
          onMarkRead={(id) => console.log(id)} // ⚠️ 每次渲染都是新函数
        />
      ))}
    </div>
  );
}

// ✅ 用 useCallback 稳定函数引用
function AlertList({ alerts }) {
  const handleMarkRead = useCallback((id) => {
    console.log(id);
  }, []); // ✅ 空依赖，函数引用稳定

  return (
    <div>
      {alerts.map(alert => (
        <AlertItem
          key={alert.id}
          alert={alert}
          onMarkRead={handleMarkRead} // ✅ 引用稳定
        />
      ))}
    </div>
  );
}
```

### 8.3 常见的性能问题与解决方案

| 问题 | 表现 | 解决方案 |
|------|------|---------|
| 父组件重渲染导致所有子组件重渲染 | 列表中一个项变化，全部闪烁 | React.memo + useCallback |
| 每次渲染都创建新对象/数组/函数 | 子组件认为 props 变了 | useMemo / useCallback |
| 巨大列表渲染 | 页面卡顿、白屏 | 虚拟滚动（react-window） |
| Context value 是新对象 | 所有 Consumer 重渲染 | useMemo 包裹 value |
| 动画/滚动时渲染掉帧 | 不流畅 | useDeferredValue、CSS will-change |

### 8.4 虚拟滚动：大列表的终极方案

```jsx
import { FixedSizeList } from 'react-window';

// 渲染 10000 条告警，只在 DOM 中保留可见的 ~20 条
function VirtualAlertList({ alerts }) {
  const Row = ({ index, style }) => (
    <div style={style}>
      <AlertItem alert={alerts[index]} />
    </div>
  );

  return (
    <FixedSizeList
      height={600}           // 列表可见高度
      width="100%"
      itemCount={alerts.length}  // 总数据量
      itemSize={80}              // 每行高度
    >
      {Row}
    </FixedSizeList>
  );
}
// DOM 节点数：~20 个（无论 alerts 有多少条）
// 性能提升：从 O(n) DOM 节点 → O(1) 常数个 DOM 节点
```

## 九、React 18 新特性

### 9.1 Concurrent Mode（并发模式）

Concurrent Mode 是 React 18 最重要的架构升级。核心思想：**渲染工作可以被打断、优先级调度**。

```
没有 Concurrent Mode：
render() ────────────────────> [完整渲染完成前无法响应用户]

有 Concurrent Mode：
高优先级更新（用户输入）:
render() ───> [中断] ───> render() ───> 完成 → 立即响应

低优先级更新（数据刷新）:
render() ───> [被打断] ───────────────> 继续 ───> 完成
                              ↑
                        高优先级可以插队
```

### 9.2 useTransition 与 useDeferredValue

```jsx
import { useTransition, useDeferredValue } from 'react';

// useTransition：标记哪些更新是"非紧急的"
function AlertSearch({ alerts }) {
  const [isPending, startTransition] = useTransition();
  const [query, setQuery] = useState('');
  const [displayAlerts, setDisplayAlerts] = useState(alerts);

  const handleSearch = (value) => {
    setQuery(value);

    // startTransition 包裹的更新被视为非紧急
    // 即使搜索结果计算很慢，输入框仍然流畅响应
    startTransition(() => {
      setDisplayAlerts(alerts.filter(a => a.message.includes(value)));
    });
  };

  return (
    <div>
      {/* isPending 可以用来显示加载状态 */}
      <input value={query} onChange={e => handleSearch(e.target.value)} />
      {isPending && <Spinner />}
      <AlertList alerts={displayAlerts} />
    </div>
  );
}

// useDeferredValue：延迟更新某个值（值的"防抖"）
function AlertPanel({ alerts }) {
  // 原始值立即更新，deferred 值延迟更新
  const deferredAlerts = useDeferredValue(alerts);

  const isStale = deferredAlerts !== alerts;

  return (
    <div style={{ opacity: isStale ? 0.6 : 1 }}>
      {/* deferredAlerts 先显示旧数据，然后快速更新 */}
      <AlertList alerts={deferredAlerts} />
    </div>
  );
}
```

### 9.3 Automatic Batching（自动批量更新）

React 18 之前，批量更新只在事件处理器中生效。React 18 中，**所有场景**都会批量更新。

```jsx
// React 17 之前：
function handleClick() {
  fetch('/api').then(() => {
    setCount(c => c + 1);  // 触发一次重渲染
    setName('Alice');       // 再触发一次重渲染（共2次）
  });
}

// React 17（事件处理内批量）：
function handleClick() {
  setCount(c => c + 1);   // 不立即渲染
  setName('Alice');        // 不立即渲染
  // 事件结束后，一次性重渲染（共1次）
}

// React 18（所有场景批量）：
function handleClick() {
  fetch('/api').then(() => {
    setCount(c => c + 1);  // 不触发
    setName('Alice');       // 不触发
  }); // Promise 解决后，一次性重渲染（共1次）
}

// setTimeout / 原生事件 / async 函数中的批量更新全部生效
```

## 十一、面试高频问题汇总

### Q1: React 的单向数据流是什么意思？
数据从父组件通过 props 向下传递，子组件不能直接修改 props，只能通过调用父组件传来的回调函数通知父组件，由父组件决定如何更新数据。这种设计让数据流向清晰可追踪，便于调试和理解。

### Q2: setState 为什么是异步的？
React 为了提升性能，会将多个 setState 调用合并（批量更新）成一次重渲染。如果每次 setState 都同步重渲染，100次状态更新就会触发100次重渲染，性能极差。React 18 进一步扩大了批量更新的范围，连 Promise/setTimeout 中的 setState 也会批量处理。

### Q3: Virtual DOM 的优缺点？
优点：减少直接 DOM 操作次数、提升大规模更新性能、跨平台能力（React Native 等）；缺点：首次渲染稍慢（多了一层转换）、小规模更新时可能比原生 DOM 慢、内存占用增加。

### Q4: useEffect 的依赖数组为空和完全不传有什么区别？
`[]`（空数组）= 只在挂载时执行一次，类似 `componentDidMount`，包括卸载时的 cleanup；完全不传依赖数组 = 每次渲染后都执行，类似 `componentDidMount + componentDidUpdate`。

### Q5: useCallback 和 useMemo 的区别？
`useCallback(fn, deps)` 等价于 `useMemo(() => fn, deps)`。useCallback 缓存函数（避免子组件因为函数引用变化而重渲染），useMemo 缓存计算结果（避免昂贵的重复计算。两者都是优化工具，过度使用反而有性能开销）。

### Q6: React.memo 和 useMemo 有什么区别？
`React.memo` 是高阶组件，包装整个组件（判断是否需要重新渲染）；`useMemo` 是 Hook，缓存组件内部的计算值（避免重复计算）。两者作用于不同层级：`React.memo` 防止不必要的渲染，`useMemo` 防止不必要的计算。

### Q7: useEffect 和 useLayoutEffect 的区别？
`useEffect` 在渲染后异步执行，不会阻塞浏览器绘制，适合大多数场景；`useLayoutEffect` 在渲染后同步执行，会阻塞浏览器绘制，适合需要同步读取/修改 DOM 的场景（如测量元素尺寸、主动聚焦）。**能用 useEffect 就用 useEffect**，useLayoutEffect 有性能风险。

### Q8: React 18 Concurrent Mode 解决了什么问题？
解决了渲染阻塞问题。大量数据渲染时（比如 SOC 系统加载数千条历史告警），用户输入、点击等交互不会被阻塞，React 可以智能调度优先级，确保 UI 始终流畅响应。
