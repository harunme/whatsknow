## 答案

### A1: SIEM 核心功能
**[10秒结论]** SIEM（Security Information and Event Management）是安全日志聚合与分析平台，核心功能：采集、存储、搜索、关联分析、告警生成；Splunk 功能强但贵（按数据量），ELK 开源免费但运维复杂。

**[90秒拆解]**
- **核心功能**：① 日志采集（Agent/Syslog） ② 海量存储（ES/Splunk Indexer） ③ 快速搜索（SPL/KQL） ④ 关联分析（跨源关联发现威胁） ⑤ 告警生成（规则匹配触发）
- **Splunk vs ELK**：
  - Splunk：功能完整，TSIDX 关键字搜索快，生态成熟，但按数据量收费（TB 级成本高）
  - ELK（Elasticsearch）：开源免费，Lucence 倒排索引导出范围查询快，但全链路运维复杂（Elasticsearch + Logstash + Kibana）
- **日志保留**：金融行业（等保三级）通常 5-7 年；普通企业热数据 30-90 天，温数据 1 年，冷数据归档

**[20秒收尾]** SIEM 是 SOC 平台的数据中枢，日志质量和查询性能直接决定威胁检测能力。

---

### A2: 日志采集
**[10秒结论]** FileBeat 轻量 Agent 支持断点续传和 ACK 确认，适合应用日志；Syslog 无状态推送，适合网络设备；At-Least-Once 通过生产端 ACK + 幂等写入 + 消费者提交 OFFSET 三步保证。

**[90秒拆解]**
- **FileBeat vs Syslog**：
  - FileBeat：精确读取文件偏移量、本地缓存+ACK 确认、不丢日志，CPU 消耗 ~1-3%
  - Syslog：UDP 推送（可能丢包）、无法精确控制、无法感知 logrotate，但几乎无资源消耗
- **采集延迟控制**：MTTD 直接受采集延迟影响，目标：< 5 秒（实时告警场景）
- **防丢策略（At-Least-Once）**：
  - 生产端：发送 ACK 后再删除本地缓存
  - 存储端：幂等写入（Elasticsearch `_id` = 日志内容 hash）
  - 消费者：提交 OFFSET 后才认为消息处理完成
- **logrotate 丢日志**：FileBeat 用 Inotify 监听 inode 变化，`force_close_files: true`

**[20秒收尾]** 采集是 SIEM 数据质量的第一环，采集丢日志后续分析再强也无用，At-Least-Once 是生产级标准。

### A3: MITRE ATT&CK 框架
**[10秒结论]** ATT&CK 矩阵将攻击者行为归纳为 14 个战术阶段，典型链：初始访问→执行→持久化→权限提升→防御规避→凭证访问→发现→横向移动→收集→命令与控制→泄露→影响。

**[90秒拆解]**
- **14 阶段速记**：初（初始访问）→执（执行）→持（持久化）→提（权限提升）→躲（防御规避）→凭（凭证访问）→发（发现）→横（横向移动）→集（收集）→控（命令与控制）→泄（泄露）→影（影响）
- **Initial Access 典型技术**：钓鱼邮件（Spearphishing Attachment/Link）、利用面向互联网资产漏洞（Exploit Public-Facing Application）、供应链攻击
- **Execution 典型技术**：PowerShell/VBScript 恶意脚本、`rundll32.exe`/`mshta.exe` 等 LOLBAS 工具、命令行接口
- **ATT&CK Navigator**：JSON layer 文件导入，以矩阵热力图展示检测覆盖率（绿色=已覆盖，红色=盲区）
- **SOC 平台应用**：检测规则关联 ATT&CK Technique ID → 评估检测覆盖率 → Navigator 可视化 → 优先补充红色盲区规则

**[20秒收尾]** ATT&CK 是安全行业的"统一语言"，用好 Navigator 是评估 SOC 平台检测能力的关键工具。

### A4: NL to SPL/DSL
**[10秒结论]** Few-shot Prompt 路线（零训练成本，快速上线）+ 意图分类路由 + Schema 注入防越权；Fine-tune 路线（5-10 万对训练数据）用于生产级高准确率需求。

**[90秒拆解]**
- **Few-shot 路线**：精心设计的 system prompt + (自然语言, SPL) 对示例，LLM 上下文学习零训练成本；复杂嵌套查询（join/transaction）意图还原不稳定
- **Fine-tune 路线**：用开源模型（Llama-3/Qwen-2）+ LoRA 微调，训练数据 5-10 万对；精度更高但需要标注数据积累
- **Schema 注入**：每次推理前注入当前用户有权访问的 index/field 白名单，防止跨租户查询
- **AST 校验**：生成的 SPL 用 ANTLR 解析 AST，拒绝 `|script`/`|delete` 等危险操作
- **覆盖范围**：Few-shot 路线覆盖约 80% 常见查询，剩余 20% 复杂查询需要持续补充示例

**[20秒收尾]** NL to SPL 是 AI+安全最落地的应用之一，Few-shot 快速验证，Fine-tune 持续优化，是工程迭代的正确路径。

### A5: 查询安全
**[10秒结论]** 三道防线：Schema 注入（index 白名单）、AST 校验（拒绝危险命令）、查询范围限制（时间窗口 + 条数上限）。

**[90秒拆解]**
- **Prompt Injection 防御**：用户输入永远作为变量绑定（`{user_query}`），不和 system prompt 混合，防止注入指令覆盖系统行为
- **AST 校验**：对生成的 SPL/ES|QL 做语法解析（ANTLR），拒绝不合语法和危险操作的查询
- **范围限制**：最大时间窗口 30 天、最大结果条数 10000 条、最大 wildcard 深度限制
- **白名单**：index 名称白名单（用户只能查询自己有权访问的 index）、字段白名单、命令白名单

**[20秒收尾]** 查询安全是多租户 SIEM 的生命线，Schema 注入 + AST 校验是生产级 NL 查询的必备保障。

### A6: 告警分类
**[10秒结论]** P0（紧急，活跃攻击，立即响应）/P1（高危，确认入侵，1小时）/P2（中危，可疑异常，4小时）/P3（低危，低风险，24小时）；规则引擎做确定性初判，LLM 做 P0/P1 二次研判降误报。

**[90秒拆解]**
- **定级标准**：
  - P0：勒索软件行为、高管钓鱼、活跃内网攻击
  - P1：已确认外部攻击、横向移动行为
  - P2：可疑但未确认的异常行为
  - P3：政策违规、扫描行为、误报高发规则
- **规则引擎**：确定性、可解释性强，毫秒级响应，适合 P3/P2 批量处理
- **LLM 判断**：上下文感知，能发现跨告警关联，但有幻觉风险，适合 P0/P1 精判
- **误报控制**：ML Enrichment 计算置信度分数，低于阈值不推送；建立反馈回路将分析师处置结果用于模型重训

**[20秒收尾]** 告警定级是 SOC 运营效率的调节阀，LLM + 规则引擎的混合架构是目前最优解。

### A7: 告警聚合
**[10秒结论]** 时间窗口聚合（同源 IP 5 分钟内告警合并）、攻击链聚合（同主机按 ATT&CK 阶段构建关联图）、实体维度聚合（同资产告警合并）；ML 聚类用于发现规则无法覆盖的新型攻击。

**[90秒拆解]**
- **Alert Fatigue**：告警过多导致分析师麻木，真实威胁被淹没。聚合是减少告警疲劳的核心手段
- **时间窗口聚合**：NIDS 扫描告警（同一 IP 短时间内对多端口探测）→ 5 分钟窗口内合并为 1 个 Incident
- **攻击链聚合**：同主机告警按 ATT&CK 战术阶段建模，构成攻击链者合并为高优先级 Incident
- **ML 聚类**：Sentence-BERT 嵌入计算告警文本相似度，余弦相似度 > 0.85 自动归并；优势是发现未知攻击模式，劣势是推理开销大

**[20秒收尾]** 分层聚合架构：实时层（秒级）→ 窗口层（5分钟）→ 异步层（后台 ML），原始告警始终保留，聚合视图是降噪展示而非数据丢失。

### A8: MTTD 与 MTTR
**[10秒结论]** MTTD（平均检测时间）行业均值约 200 天，MTTR（平均响应时间）约 70 天；AI/ML 引入后领先企业 MTTD 可缩至 24-48 小时，但误报率同步上升需配套 triage 体系。

**[90秒拆解]**
- **MTTD**：攻击发生到 SIEM 产生告警的平均时间；受采集延迟、管道处理延迟、检测规则时间窗影响
- **MTTR**：告警产生到完成响应（隔离、止血、溯源）的平均时间；受告警定级准确性、分析师响应速度、SOAR 自动化程度影响
- **行业基准**（Verizon DBIR + IBM Cost of Data Breach 2024）：
  - 领先企业（<25th percentile）：MTTD < 24h，MTTR < 30 天
  - 业界均值：MTTD ~200 天，MTTR ~70 天
- **AI 对 MTTD 的影响**：UEBA 异常检测使领先企业 MTTD 从 200+ 天缩至 24-48 小时，但误报率上升 30-40%，需配套告警 triage 质量评估体系

**[20秒收尾]** MTTD 和 MTTR 是衡量 SOC 平台效果的核心 KPI，MTTD 越短损失越小，MTTR 越短威胁影响越可控。
