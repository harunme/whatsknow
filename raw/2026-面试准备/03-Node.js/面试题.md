### 事件循环
```
Node.js 事件循环的几个阶段是什么？setTimeout、setImmediate、process.nextTick 各自的执行顺序？宏任务和微任务有什么区别？
追问：
- 为什么 Node.js 能用单线程处理高并发？
- 什么是Libuv？
```
**[10秒结论]** Node.js 事件循环分 timers → pending callbacks → idle/prepare → poll → check → close callbacks 六个阶段，process.nextTick 优先级高于微任务，微任务（Promise）优先于宏任务（setTimeout/setImmediate）。

**[90秒拆解]**
- **六个阶段**：timers（执行 setTimeout/setInterval 回调）→ pending callbacks → idle/prepare（内部）→ poll（获取新 I/O 事件）→ check（执行 setImmediate）→ close callbacks
- **执行顺序**：`process.nextTick`（最高）→ 微任务（Promise.then）→ 宏任务（setTimeout/setImmediate）→ I/O 回调
- **宏任务 vs 微任务**：微任务在当前阶段结束后、下一个阶段开始前全部执行完（Promise 回调）；宏任务在事件循环的对应阶段执行
- **单线程高并发**：基于事件循环 + I/O 多路复用（Linux epoll/macOS kqueue），单线程不阻塞等待 I/O，异步 I/O 完成后通过回调通知

**[20秒收尾]** Libuv 是 Node.js 跨平台 I/O 抽象层，封装了各操作系统的 I/O 多路复用机制，是事件循环的底层引擎。

### 异步编程与 Promise
```
Promise 的三种状态是什么？`.then/.catch` 链式调用怎么工作的？async/await 怎么捕获异常？和同步代码有什么区别？
追问：
- Promise.resolve() 和 new Promise() 的区别？
- async/await 会阻塞事件循环吗？
```
**[10秒结论]** Promise 有 pending/fulfilled/rejected 三种状态，链式调用通过返回新 Promise 实现；async/await 是 Promise 的语法糖，用 try/catch 捕获异常。

**[90秒拆解]**
- **Promise 三种状态**：pending（进行中）、fulfilled（已成功）、rejected（已失败），状态一旦改变不可逆
- **链式调用原理**：`promise.then(onFulfilled, onRejected)` 返回新 Promise，新 Promise 的状态由 onFulfilled/onRejected 的返回值决定（返回值→fulfilled，抛异常→rejected）
- **async/await 捕获异常**：
  ```javascript
  try {
    const result = await fetchData();
  } catch (error) {
    console.error('捕获到错误:', error);
  }
  ```
- **vs 同步代码**：async 函数本身不阻塞事件循环，内部 await 时让出控制权；只是写法上像同步代码

**[20秒收尾]** 面试中常问：`await promise` 会阻塞吗？答案是不阻塞当前事件循环，只在当前协程内等待。

### 模块机制
```
CommonJS 和 ES Module 的区别？require/exports 和 import/export 各自的加载时机？循环引用怎么处理？
追问：
- `module.exports` 和 `exports.xxx` 有什么区别？
- ESM 可以动态 import 吗？
```
**[10秒结论]** CJS 是运行时同步加载，`require/exports`；ESM 是编译时静态分析，`import/export`。循环引用时，被引用的模块会返回已执行部分（不会 undefined）。

**[90秒拆解]**
- **CJS vs ESM 核心区别**：
  - CJS：`require()` 运行时同步执行，可动态加载（条件加载）；值拷贝
  - ESM：`import` 编译时解析，必须在模块顶层；值引用，动态只读绑定
- **`exports.xxx` vs `module.exports`**：exports 是 module.exports 的引用别名，`exports = {}` 会切断引用导致 export 失效；`exports.xxx = val` 正确
- **循环引用处理**：A 引用 B，B 又引用 A，B 在 A 未加载完时返回已执行部分（undefined），JavaScript 以深度优先保证加载顺序

**[20秒收尾]** Node.js 16+ 支持同时使用 CJS 和 ESM，通过 `.mjs`/`.cjs` 扩展名或 `package.json` 的 `type` 字段切换。

### 内存管理与 V8 垃圾回收
```
V8 的垃圾回收机制是什么？新生代和老年代的算法分别是什么？怎么排查 Node.js 内存泄漏？
追问：
- 什么是堆内存和栈内存？各自存什么？
- `node --inspect` 怎么用？
```
**[10秒结论]** V8 用分代回收：新生代用 Scavenge（复制），老年代用 Mark-Sweep + Mark-Compact（标记清除/整理）；内存泄漏通过 `--inspect` + Chrome DevTools 或 heapdump 排查。

**[90秒拆解]**
- **新生代**：对象存活时间短，Scavenge 算法将新生代分成 From/To 两半，活跃对象复制到 To 空间，效率高但空间利用率 50%
- **老年代**：对象存活时间长，Mark-Sweep（标记清除）后可能有内存碎片，用 Mark-Compact（标记整理）压缩
- **内存泄漏排查**：`node --inspect` 启动调试端口，Chrome DevTools 截图 Heap Snapshot 对比；或用 `heapdump` 模块定期生成堆快照
- **常见泄漏**：全局变量累积、闭包引用、EventEmitter 监听未移除、定时器未清理

**[20秒收尾]** 用 `process.memoryUsage()` 监控堆内存使用，HEAP USED > HEAP TOTAL * 0.8 时应关注。

### Express vs Koa vs Fastify
```
Express、Koa、Fastify 各自的特点是什么？中间件机制是怎么工作的？Koa 的洋葱模型是什么？
追问：
- 中间件的 next() 是什么原理？
- Koa 为什么比 Express 性能更好？
```
**[10秒结论]** Express 中间件链式传递 context，Koa 用洋葱模型所有中间件先入后出全部执行一遍，Fastify 以插件化和 Schema 验证著称，性能最高。

**[90秒拆解]**
- **Express**：中间件按顺序执行，`next()` 跳到下一个，无洋葱模型；同步友好，异步需注意错误处理
- **Koa 洋葱模型**：
  ```javascript
  app.use(async (ctx, next) => {
    console.log('1');      // 前置逻辑
    await next();
    console.log('2');     // 后置逻辑（所有下游中间件完成后执行）
  });
  ```
  每个中间件都经历「进入 → 执行下游 → 回来」的过程
- **Fastify**：基于 Schema（JSON Schema）做请求/响应验证，内置 Fastify-plugin 避免命名空间污染，性能比 Express 快 2-3 倍
- **Koa 比 Express 更好**：更好的错误处理（try/catch 全局捕获）、更轻量（无内置路由/视图）、async/await 原生支持

**[20秒收尾]** SOC 平台后端选 Fastify + TypeBox（类型化 Schema），性能好且类型安全。

### Buffer 与 Stream
```
Buffer 和 Stream 是什么？什么时候用流比用 Buffer 更好？背压（backpressure）是怎么回事？
追问：
- `fs.createReadStream` 和 `fs.readFile` 的区别是什么？
- pipeline 和 pipe 有什么区别？
```
**[10秒结论]** Buffer 是固定大小的二进制数据块，Stream 是处理流式数据的抽象；流式处理避免一次性加载大文件到内存；背压是下游处理速度跟不上上游时的流控制机制。

**[90秒拆解]**
- **Buffer**：`Buffer.from('hello')`、`Buffer.alloc(1024)`，用于处理 TCP 流、文件 I/O、加密等二进制数据
- **Stream 四种类型**：Readable（可读）、Writable（可写）、Duplex（双向）、Transform（转换）
- **背压处理**：当 Writable 的写入速度跟不上 Readable 的读取速度时，`write()` 返回 false，Readable 应暂停发送数据；`pipe()` 自动处理背压
- **`readFile` vs `createReadStream`**：`readFile` 一次性读入内存，大文件 OOM；`createReadStream` 分块读取，内存占用稳定

**[20秒收尾]** 日志处理、文件上传、音视频转码等大文件场景必须用 Stream，ES 的 bulk API 也支持流式写入避免内存爆掉。

### 性能优化
```
单线程怎么利用多核 CPU？cluster 模块和 child_process 怎么用？pm2 怎么实现负载均衡和零宕机部署？
追问：
- pm2 的 cluster 模式和 fork 模式有什么区别？
- `process.on('uncaughtException')` 能捕获所有异常吗？
```
**[10秒结论]** Node.js 单线程用 cluster 模块 fork 多进程利用多核；pm2 通过 cluster 模式实现负载均衡和零宕机重启。

**[90秒拆解]**
- **cluster 模块**：主进程 fork 多个 Worker 进程，每个 Worker 独立运行事件循环，通过 IPC 与主进程通信；主进程负责监听端口并分发请求
- **child_process**：适合运行独立子进程（如调用 Python 脚本、FFmpeg），不共享内存，通信通过 stdin/stdout 或消息队列
- **pm2 零宕机**：收到 SIGTERM 时先不再接收新请求，等现有请求处理完再退出；重载时先启动新进程再关闭旧进程
- **pm2 命令**：`pm2 start app.js -i 4`（4 个实例负载均衡）、`pm2 reload all`（零宕机重载）

**[20秒收尾]** 生产环境 Node.js 服务至少部署 2 个实例，pm2 cluster 模式 + Nginx 负载均衡，避免单点故障。

### 常见安全问题
```
怎么防止原型链污染？eval 为什么危险？依赖漏洞怎么管理？
追问：
- 什么是 `__proto__` 污染？
- npm audit 原理是什么？
```
**[10秒结论]** 原型链污染通过 Object.freeze() 或禁止用户控制 `__proto__` 防止；eval 执行任意代码危险，可用 Function/new Function 代替；依赖漏洞用 npm audit + Snyk/Dependabot 管理。

**[90秒拆解]**
- **原型链污染**：`merge`/`clone` 等函数如果用户输入控制 `__proto__` 或 `constructor.prototype`，可修改 Object 原型：
  ```javascript
  // 污染前
  ({}.a) // undefined
  // 污染后（攻击代码）
  payload = {"__proto__": {"a": "polluted"}
  ```
  **防御**：使用 `Object.create(null)` 创建无原型对象；禁用用户输入控制原型属性
- **eval 危险**：eval 执行字符串为代码，攻击者可注入恶意脚本。**替代**：`new Function('param', 'return code')` 或模板引擎
- **依赖漏洞管理**：`npm audit` 扫描已知 CVE；`npm audit fix` 自动修复；CI 集成 Snyk/Dependabot 持续监控；`npm ci` 确保锁文件版本一致

**[20秒收尾]** SOC 平台处理外部输入（告警日志、用户查询）必须做输入校验和清理，防止原型链污染和注入攻击。
