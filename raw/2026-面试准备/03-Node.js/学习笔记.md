## 一、Node.js 基础概念

### 1.1 是什么？为什么存在？

Node.js 是基于 Chrome V8 引擎的 JavaScript 运行时，允许 JavaScript 运行在服务器端。它诞生于 2009 年，由 Ryan Dahl 创建，初衷是解决 web 服务器的 I/O 瓶颈问题。

**为什么需要 Node.js？**

传统服务器（如 Apache）每个连接都会创建一个线程，线程是稀缺资源。假设一台服务器有 4GB 内存，每个线程占用 2MB，那么理论最多只能处理 2000 个并发连接。而 Node.js 单线程 + 事件驱动的设计，可以用极少的资源处理海量连接。

```javascript
// 这段代码展示 Node.js 处理并发的简单方式
// 不需要为每个请求创建线程，事件循环自动调度

const http = require('http');

const server = http.createServer((req, res) => {
  // 模拟 I/O 操作（如查询数据库），这里用 setTimeout 模拟
  setTimeout(() => {
    res.end('Hello, SOC Alert!');
  }, 100); // 100ms 的 I/O 操作期间，CPU 可以处理其他请求
});

server.listen(3000, () => {
  console.log('Server running on port 3000');
  // Node.js 可以同时接收数千个请求，而不是阻塞等待每个 I/O
});
```

**核心问题：Node.js 擅长什么，不擅长什么？**

| 场景 | 适合度 | 原因 |
|------|--------|------|
| I/O 密集型（API、实时通信） | 非常适合 | 事件驱动 + 非阻塞 I/O |
| CPU 密集型（图像处理、加密） | 不适合 | 单线程，无法利用多核 |
| 简单脚本工具 | 非常适合 | 生态丰富，npm 上百万包 |
| 大数据流处理 | 适合 | Stream API 高效 |
| 复杂业务逻辑 | 一般 | 回调地狱（已被 Promise/async 改善）|

### 1.2 单线程与事件驱动

"单线程"指的是 JavaScript 执行层面只有一个主线程，所有 JS 代码都在这个线程上运行。但这不意味着 Node.js 只能做一件事——**它通过事件驱动模型，在等待 I/O 时去处理其他任务**。

```javascript
// 演示事件驱动的核心概念

console.log('1 - 同步代码开始');

// setTimeout 是异步的，0 毫秒后执行
setTimeout(() => {
  console.log('4 - setTimeout 回调（属于 check 阶段）');
}, 0);

// process.nextTick 优先级极高
process.nextTick(() => {
  console.log('3 - nextTick（优先级高于 Promise 微任务）');
});

// Promise 微任务
Promise.resolve().then(() => {
  console.log('3.5 - Promise.then（微任务）');
});

console.log('2 - 同步代码结束');
// 输出顺序：1 → 2 → 3 → 3.5 → 4
```

**事件驱动的工作原理**：Node.js 内部有一个"事件队列"（实际是 libuv 的事件循环），当异步 I/O 完成时，回调函数被放入队列，主线程从队列中取出执行。这就实现了"单线程但非阻塞"。

### 1.3 非阻塞 I/O

阻塞 I/O：线程发起读文件请求后，必须等待磁盘读取完成，才能继续执行后续代码。这段时间线程是空闲的，浪费 CPU。

非阻塞 I/O：线程发起读文件请求后，立即返回（不等待），继续执行后续代码。当磁盘读取完成后，系统通知线程，线程再处理数据。

```javascript
// 阻塞 I/O 示例（伪代码）
const data = fs.readFileSync('alert.log'); // 程序在这里卡住
console.log('文件读取完成');
processData(data);

// 非阻塞 I/O（Node.js 标准做法）
fs.readFile('alert.log', (err, data) => { // 立即返回，不阻塞
  console.log('文件读取完成');
  processData(data);
});
console.log('这行代码在文件读取完成前就会执行！');
```

**为什么非阻塞 I/O 对 SOC 系统特别重要？**

SOC（安全运营中心）系统需要处理大量实时告警、日志流、威胁情报查询。这些场景的核心特征是：请求多、等待时间长（数据库查询、外部 API 调用）。Node.js 的非阻塞 I/O 非常适合这种 I/O 密集型场景，能用少量线程支撑高并发。

## 二、事件循环

### 2.1 六个阶段详解

Node.js 事件循环分为六个阶段，每个阶段都有特定任务。这是面试超高频考点，必须烂熟于心。

```
┌─────────────────────────────────────────────────────────────┐
│                      事件循环（Event Loop）                   │
├─────────────────────────────────────────────────────────────┤
│  阶段 1: timers          │ setTimeout / setInterval 回调     │
│  阶段 2: pending callbacks│ 系统内部回调                      │
│  阶段 3: idle, prepare   │ libuv 内部使用                    │
│  阶段 4: poll            │ I/O 回调、等待新 I/O 事件          │
│  阶段 5: check           │ setImmediate 回调                 │
│  阶段 6: close callbacks │ socket close 等关闭事件           │
└─────────────────────────────────────────────────────────────┘
       ↑                                                    │
       │            微任务（每个阶段结束后全部执行完）              │
       │  process.nextTick → Promise.then → queueMicrotask    │
       └────────────────────────────────────────────────────┘
```

**每个阶段的详细说明：**

**阶段 1 - timers**：执行所有到期的 `setTimeout` 和 `setInterval` 回调。例如设置了 100ms 后执行的定时器，100ms 到了就进入这个阶段执行。

**阶段 2 - pending callbacks**：执行上一轮事件循环中延迟的 I/O 回调（比如 TCP 错误回调）。

**阶段 3 - idle, prepare**：libuv 内部调度用，开发者无需关心。

**阶段 4 - poll（轮询阶段）**：这是最复杂的阶段，做两件事：
- 如果有可执行的 I/O 回调，执行它们
- 如果没有，回调队列为空，则在这里等待——**这就是 Node.js 不消耗 CPU 的秘密**

**阶段 5 - check**：专门执行 `setImmediate` 的回调。为什么单独设置这个阶段？因为 poll 阶段可能会阻塞等待 I/O，而 `setImmediate` 的回调需要**立即**在 poll 阶段之后执行。

**阶段 6 - close callbacks**：执行 `socket.on('close')` 等关闭事件回调。

```javascript
// 对比 setTimeout(fn, 0) vs setImmediate
setTimeout(() => console.log('1 - setTimeout'), 0);
setImmediate(() => console.log('2 - setImmediate'));

/*
运行结果（取决于运行环境）：
- 直接运行 node test.js：两者都可能先执行（取决于系统计时器精度）
- 在 I/O 回调内运行：setImmediate 几乎总是先执行

原因：在 I/O 回调中，poll 阶段已经完成，紧接着就是 check 阶段
*/

// I/O 回调中的执行顺序
const fs = require('fs');
fs.readFile('package.json', () => {
  console.log('I/O 回调执行');
  setTimeout(() => console.log('3 - setTimeout（来自 I/O 回调内）'), 0);
  setImmediate(() => console.log('4 - setImmediate（来自 I/O 回调内）'));
  Promise.resolve().then(() => console.log('5 - Promise.then（微任务）'));
});
// 输出：I/O 回调执行 → 5（微任务）→ 4（check 阶段）→ 3（timers 阶段）
```

### 2.2 宏任务 vs 微任务

**核心规则**：每个阶段（宏任务）执行完毕后，会**清空所有微任务队列**，然后才进入下一个阶段。

| 类别 | 示例 | 调度方式 |
|------|------|----------|
| 宏任务（Macrotask） | setTimeout, setInterval, setImmediate, I/O 回调 | 在对应阶段执行 |
| 微任务（Microtask） | Promise.then/.catch/.finally, process.nextTick, queueMicrotask | 当前阶段结束后全部执行 |

**为什么这样设计？**

微任务的优先级更高——Promise 的 resolve/reject 回调应该尽快执行，不能等到下一个事件循环。设想一个 API 响应的数据处理，如果还要等下一个循环才能更新 UI，体验会很差。

```javascript
// 演示微任务优先于宏任务
console.log('1 - 同步代码');

setTimeout(() => console.log('4 - setTimeout（宏任务）'), 0);

Promise.resolve()
  .then(() => console.log('2.5 - Promise.then（微任务）'))
  .then(() => console.log('3 - Promise.then 链（微任务）'));

// 即使 Promise 在 setTimeout 之后才创建，微任务仍优先执行
// 输出：1 → 2.5 → 3 → 4
```

### 2.3 process.nextTick 的特殊性

`process.nextTick` 虽然也是微任务，但它比 Promise 微任务**优先级更高**。它不是在微任务队列中，而是在一个单独的 `nextTickQueue` 里，每次微任务阶段会先清空 `nextTickQueue`。

**为什么 Node.js 要单独设置 `nextTickQueue`？**

设计给库作者用的。当你在编写一个模块，需要确保某个回调**在本轮事件循环中、在其他微任务之前**执行时，`process.nextTick` 是唯一选择。

```javascript
// nextTick vs Promise.then
process.nextTick(() => console.log('1 - nextTick'));
Promise.resolve().then(() => console.log('2 - Promise.then'));
queueMicrotask(() => console.log('3 - queueMicrotask'));

// 输出：1 → 3 → 2
// nextTick 的优先级高于所有其他微任务！
```

⚠️ **注意**：不要在 `process.nextTick` 中写递归调用或大量代码，这会阻塞事件循环，导致其他 I/O 无法处理。

### 2.4 经典面试题解析

**面试题**：以下代码输出顺序是什么？

```javascript
async function main() {
  console.log('1 - 同步开始');
  const data = await fetchData();
  console.log('3 - await 之后');
}

function fetchData() {
  return Promise.resolve();
}

console.log('0 - 同步代码');
main();
console.log('2 - 同步代码结束');

// 输出：0 → 1 → 2 → 3
// 原因：async 函数在遇到 await 时会立即返回，
//       之后的代码（本轮同步代码）继续执行，
//       await 之后的代码进入微任务队列
```

**面试题**：Node.js 为什么适合高并发？

答案要点：
1. 基于非阻塞 I/O，线程在等待 I/O 时不阻塞，可以处理其他请求
2. 事件循环驱动，异步 I/O 完成后通过回调通知
3. 单线程避免上下文切换开销
4. 适合 I/O 密集型场景（API、实时通信），不适合 CPU 密集型

## 三、Libuv 引擎

### 3.1 跨平台 I/O 抽象层

Libuv 是 Node.js 的底层核心库，封装了操作系统提供的异步 I/O 能力。它的设计目标是：**让 Node.js 用同一套 API，在不同操作系统上都能高效工作**。

```
┌─────────────────────────────────────┐
│          JavaScript 层              │  Node.js API (fs, http, etc.)
├─────────────────────────────────────┤
│          Node.js 核心层              │  V8 引擎 + Node.js C++ 绑定
├─────────────────────────────────────┤
│          Libuv（异步 I/O 抽象层）     │
│  ┌──────────┬──────────┬──────────┐ │
│  │  epoll   │  kqueue  │  IOCP   │ │  Linux  │ macOS  │ Windows
│  └──────────┴──────────┴──────────┘ │
├─────────────────────────────────────┤
│          操作系统内核                 │
└─────────────────────────────────────┘
```

**Libuv 的职责**：
- 管理事件循环（timers、poll、check 等阶段）
- 封装不同操作系统的 I/O 多路复用机制
- 处理线程池（用于无法异步化的操作，如 DNS 查询、文件读取）
- 提供跨平台的线程和同步原语

### 3.2 epoll 与 kqueue

这是操作系统层面的高性能 I/O 通知机制，Libuv 封装了它们，让 Node.js 可以在不同平台使用。

**epoll（Linux）**：传统 select/poll 需要遍历所有文件描述符检查是否有事件，复杂度 O(n)。epoll 只关注"有事件发生的文件描述符"，复杂度 O(1)。可以同时监控数十万个文件描述符。

```javascript
// Linux 系统中查看 epoll 实例数（一个 Node.js 进程通常 1 个 epoll 实例）
// $ cat /proc/<pid>/fdinfo/1
// 该文件描述符类型为 eventpoll 即 epoll
```

**kqueue（macOS/FreeBSD）**：类似 epoll 的高效 I/O 事件通知机制，macOS 专用。Libuv 在 macOS 上自动使用 kqueue。

**IOCP（Windows）**：Windows 的异步 I/O 模型，和 epoll/kqueue 理念不同但效果类似。Libuv 在 Windows 上使用 IOCP。

**为什么 SOC 系统要了解这个？** SOC 告警处理可能需要同时监控数千个安全设备的连接，epoll 是 Linux 高性能服务器的标配。

## 四、Promise 与异步编程

### 4.1 三种状态与状态机

Promise 是一个状态机，有且仅有三种状态：

```
        ┌──────────────────┐
        │     pending      │  (进行中，初始状态)
        │   （可转换状态）   │
        └────────┬─────────┘
                 │ 调用 resolve() 或 reject()
                 ▼
    ┌────────────┴────────────┐
    │                         │
    ▼                         ▼
┌─────────────┐        ┌─────────────┐
│  fulfilled  │        │  rejected   │
│  (已成功)    │        │  (已失败)    │
│ 不可转换其他  │        │ 不可转换其他  │
└─────────────┘        └─────────────┘
```

**关键规则**：状态一旦改变，不可逆。这和 Promise 名字的语义一致——"承诺"要么兑现（fulfilled），要么拒绝（rejected），不能既兑现又拒绝。

```javascript
const promise = new Promise((resolve, reject) => {
  resolve('success'); // 状态从 pending → fulfilled
  reject('error');    // 这行无效，因为状态已改变
  resolve('success2'); // 这行也无效
});

promise.then(console.log); // 输出：success
```

### 4.2 链式调用原理

Promise 的 `.then()` 方法**总是返回一个新的 Promise**。这个新 Promise 的状态由回调函数的返回值决定：

- 回调返回正常值 → 新 Promise 变为 fulfilled
- 回调抛出异常 → 新 Promise 变为 rejected
- 回调返回 Promise → 新 Promise "跟随"返回的 Promise 的状态

```javascript
// 链式调用示例
function delay(ms) {
  return new Promise(resolve => setTimeout(() => resolve(ms), ms));
}

delay(100)
  .then(ms => {
    console.log(`等待了 ${ms}ms`);
    return ms * 2; // 返回值决定下一个 Promise 的状态
  })
  .then(double => {
    console.log(`翻倍后: ${double}ms`);
    return Promise.reject('故意失败'); // 返回 rejected Promise
  })
  .catch(err => {
    console.log(`捕获错误: ${err}`);
    return 'recovery'; // catch 也返回 Promise，保持链式
  })
  .then(recovery => console.log(`恢复值: ${recovery}`))
  .finally(() => console.log('无论成功失败都会执行'));

// 输出：
// 等待了 100ms
// 翻倍后: 200ms
// 捕获错误: 故意失败
// 恢复值: recovery
// 无论成功失败都会执行
```

### 4.3 async/await 语法糖

`async/await` 是 Promise 的上层封装，让异步代码看起来像同步代码。本质上编译器会把 `await` 转换成 `.then()` 链。

```javascript
// async/await 本质是 Promise 的语法糖
// 以下两段代码完全等价：

// 写法 1: Promise 链
function fetchAlert(id) {
  return fetch(`/api/alerts/${id}`)
    .then(res => res.json())
    .then(alert => processAlert(alert));
}

// 写法 2: async/await
async function fetchAlert(id) {
  const res = await fetch(`/api/alerts/${id}`);
  const alert = await res.json();
  return processAlert(alert);
}
```

**async 函数总是返回 Promise**。即使你写 `return 42`，调用者收到的是 `Promise.resolve(42)`。

```javascript
async function getNumber() {
  return 42;
}

getNumber().then(console.log); // 42
// async 函数隐式包装了返回值
```

### 4.4 异常处理

async/await 的异常处理用 `try/catch`，这比 Promise 的 `.catch()` 更直观，也更接近同步代码的写法。

```javascript
// 标准异常处理模式
async function queryThreatIntel(ioc) {
  try {
    const res = await fetch(`https://threatintel.example.com/${ioc}`);
    if (!res.ok) throw new Error(`HTTP ${res.status}`);
    return await res.json();
  } catch (err) {
    // 网络错误、JSON 解析错误、HTTP 错误都能被捕获
    console.error('威胁情报查询失败:', err.message);
    return null; // 返回安全默认值
  }
}
```

**⚠️ 注意：Promise.reject() 不会被 try/catch 捕获**

```javascript
async function wrong() {
  try {
    return await Promise.reject('error'); // 会被 catch 捕获 ✓
  } catch (e) {
    console.log('捕获');
  }
}

async function alsoWrong() {
  // 注意：throw 语句必须在 async 函数内
  // return Promise.reject('error') 本身不会抛异常
  // 只有 await 之后才会
  return Promise.reject('error');
}
// 调用 alsoWrong() 不会触发 catch
// 应该在 alsoWrong 外部 .catch() 或用 try/await
```

### 4.5 常见陷阱

**陷阱 1：忘记 return await，导致意外行为**

```javascript
// 错误示例
async function bad() {
  try {
    return fetchData(); // 错误：返回的是 Promise，不是 await 后的值
  } catch (e) {
    return fallback();
  }
}

// 正确写法
async function good() {
  try {
    return await fetchData(); // await 确保异常被 try/catch 捕获
  } catch (e) {
    return await fallback();
  }
}
```

**陷阱 2：并行请求写成串行**

```javascript
// 串行：3 个请求总共需要 300ms
async function bad() {
  const a = await fetchA(); // 100ms
  const b = await fetchB(); // 100ms（等待 a 完成才发）
  const c = await fetchC(); // 100ms
  return [a, b, c];
}

// 并行：3 个请求总共只需要 100ms
async function good() {
  const [a, b, c] = await Promise.all([fetchA(), fetchB(), fetchC()]);
  return [a, b, c];
}
```

**陷阱 3：Promise.all 未处理部分失败**

```javascript
// 如果任意一个 Promise reject，Promise.all 整体 reject
// SOC 场景：如果只想收集结果，忽略失败的

const results = await Promise.allSettled([
  fetchAlert(1),
  fetchAlert(2),
  fetchAlert(3),
]);

results.forEach((result, i) => {
  if (result.status === 'fulfilled') {
    console.log(`告警 ${i + 1}:`, result.value);
  } else {
    console.log(`告警 ${i + 1} 失败:`, result.reason.message);
  }
});
```

## 五、模块机制

### 5.1 CommonJS 详解

CommonJS（CJS）是 Node.js 最初的模块系统，通过 `require()` 同步加载模块，`module.exports` 导出接口。

**模块加载流程**（运行时）：

```javascript
// 加载流程：resolve → load → compile → run
// 1. resolve: 找到模块的绝对路径（node_modules/、./、/ 等）
// 2. load: 读取文件内容
// 3. compile: V8 编译成函数（每个模块被包装成函数）
// 4. run: 执行模块代码，填充 module.exports
```

**模块缓存机制**：第一次 `require` 后，结果被缓存，后续 `require` 直接返回缓存，**不重新执行模块代码**。

```javascript
// a.js
console.log('a 模块被执行');
module.exports = { value: Math.random() };

// main.js
const a1 = require('./a'); // 输出：a 模块被执行
const a2 = require('./a'); // 不输出，命中缓存
console.log(a1 === a2);   // true，同一个对象
```

### 5.2 ES Module 详解

ES Module（ESM）是 ES6 引入的标准模块系统，Node.js 14+ 支持。核心特征是**编译时静态分析**——`import` 必须写在模块顶层，不能在条件语句中。

```javascript
// 静态 import（必须顶层）
import { readFile } from 'fs/promises';
import config from './config.json' assert { type: 'json' };

// 动态 import() 返回 Promise（运行时加载）
async function loadModule(name) {
  const module = await import(`./modules/${name}.js`);
  return module.default;
}
```

**值引用 vs 值拷贝**：ESM 导入的是值的只读绑定，导出模块修改值时，导入方能看到变化。

```javascript
// counter.mjs
export let count = 0;
export function inc() { count++; }

// main.mjs
import { count, inc } from './counter.mjs';
console.log(count); // 0
inc();
console.log(count); // 1，值被修改了（通过绑定引用）
```

### 5.3 CJS vs ESM 对比

| 特性 | CommonJS | ES Module |
|------|----------|-----------|
| 加载时机 | 运行时同步 | 编译时静态 |
| 语法 | `require()` / `module.exports` | `import` / `export` |
| 导入位置 | 可在条件语句中 | 必须模块顶层 |
| 值传递 | 值拷贝（原始类型） | 值引用（动态绑定）|
| 循环引用 | 返回已执行部分（可能 undefined） | 更优雅的延迟求值 |
| 能否动态 | 可以（`require(variable)`） | 需要 `import()`（返回 Promise）|
| this | `module.exports` 对象 | `undefined` |

```javascript
// CJS 可以动态 require
const driver = process.env.DATABASE === 'mongo' ? 'mongodb' : 'postgres';
const db = require(driver);

// ESM 不允许动态 import 路径拼接（字符串拼接的 import 是语法错误）
// import fs from 'fs'; // OK
// const name = 'fs'; import { readFile } from name; // 语法错误！

// 正确做法：动态 import
const name = 'fs';
import(name).then(fs => fs.readFileSync('test'));
```

### 5.4 循环引用

循环引用（circular dependency）是指 A 模块依赖 B 模块，B 模块又依赖 A 模块。Node.js 的处理方式是：**被引用的模块返回已执行部分（尚未完成的模块体以 undefined 替代）**。

```javascript
// a.js
const { bMethod } = require('./b');
console.log('a 模块执行');
function aMethod() {
  console.log('a 的方法');
  bMethod(); // 调用 b 的方法
}
module.exports = { aMethod };

// b.js
const { aMethod } = require('./a');
console.log('b 模块执行');
function bMethod() {
  console.log('b 的方法');
  aMethod(); // 调用 a 的方法
}
module.exports = { bMethod };

// main.js
const { aMethod } = require('./a');

// 执行顺序：
// 1. require('./a') 开始加载 a
// 2. a 中 require('./b') 开始加载 b
// 3. b 中 require('./a') 发现 a 尚未完成，返回已导出部分（{}）
// 4. b 继续执行，完成，exports { bMethod }
// 5. a 收到 bMethod，执行完成，exports { aMethod }
// 6. main 收到 aMethod

// 注意：b 调用 aMethod 时，aMethod 尚未赋值！
// 如果 bMethod 中调用 aMethod() 会报错
```

### 5.5 exports vs module.exports

这是 Node.js 中最容易踩的坑之一。

```javascript
// module.exports 初始指向一个空对象
// exports 是 module.exports 的引用（别名）

// 正确方式 1：exports.xxx
exports.name = 'Alice';
exports.getName = () => 'Alice';

// 正确方式 2：module.exports
module.exports = { name: 'Alice', getName: () => 'Alice' };

// 错误方式：切断引用
exports = { name: 'Bob' }; // 错误！exports 现在指向新对象，和 module.exports 无关了
// 这样导出的仍然是 module.exports 的原对象（空对象 {}）

// 面试常考：exports 和 module.exports 是什么关系？
// 答案：exports = module.exports（初始时），但导出的是 module.exports
```

## 六、Buffer 与 Stream

### 6.1 Buffer 基础

Buffer 是 Node.js 专门用来处理**二进制数据**的对象。JavaScript 字符串是 Unicode 编码，但网络协议和文件系统的底层数据都是二进制（TCP 流、文件 I/O、加密、压缩）。

```javascript
// 创建 Buffer 的几种方式
const buf1 = Buffer.from('Hello, SOC!');         // 从字符串创建
const buf2 = Buffer.alloc(1024);                  // 分配 1KB 零填充
const buf3 = Buffer.allocUnsafe(1024);             // 分配但不清空（更快）
const buf4 = Buffer.from([0x48, 0x65, 0x6c, 0x6c]); // 从字节数组创建

// 常用操作
console.log(buf1.toString());        // Hello, SOC!
console.log(buf1.length);            // 10（字节数）
console.log(buf1[0]);               // 72（H 的 ASCII 码）

// Buffer 拼接（注意编码）
const chunk1 = Buffer.from('告警1: ');
const chunk2 = Buffer.from('DDoS 攻击');
const combined = Buffer.concat([chunk1, chunk2]);
console.log(combined.toString('utf8')); // 告警1: DDoS 攻击
```

**Buffer 内存分配策略**：Node.js 使用一个 8KB 的内存池（slab pool），小于 8KB 的 Buffer 共享这个池，避免频繁系统调用。小于 4KB 的 Buffer 效率最高。

### 6.2 Stream 四种类型

Stream 是"流式处理"的抽象，让数据像水流一样分块传输，而不是一次性全部加载到内存。

| 类型 | 说明 | 典型用途 |
|------|------|----------|
| Readable | 可读流，只能读取 | `fs.createReadStream()`、`http.IncomingMessage` |
| Writable | 可写流，只能写入 | `fs.createWriteStream()`、`http.ServerResponse` |
| Duplex | 双向，既可读又可写 | `net.Socket`（TCP 套接字）|
| Transform | 转换流，读取后转换输出 | `zlib.createGzip()`（压缩）、`crypto` |

```javascript
// Readable 流示例
const fs = require('fs');
const readStream = fs.createReadStream('/var/log/soc-alerts.log', {
  encoding: 'utf8',
  highWaterMark: 64 * 1024, // 每次读取 64KB（默认）
});

readStream.on('data', (chunk) => {
  console.log(`收到 ${chunk.length} 字节数据`);
  // chunk 是字符串（因为指定了 encoding）
});

readStream.on('end', () => console.log('文件读取完成'));
readStream.on('error', (err) => console.error('读取错误:', err));

// Writable 流示例
const writeStream = fs.createWriteStream('/tmp/processed-alerts.txt');
writeStream.write('告警记录\n');
writeStream.write('========\n');
writeStream.end('结束写入'); // 调用 end 后不能再 write

// pipe：最常用的连接方式，自动处理背压
const gzip = require('zlib').createGzip();
const input = fs.createReadStream('large-file.log');
const output = fs.createWriteStream('large-file.log.gz');

input.pipe(gzip).pipe(output); // 自动处理流速不匹配
```

### 6.3 背压机制

背压（Backpressure）是流式处理中最重要的概念之一：当**Writable 消费数据的速度 < Readable 生产数据的速度**时，Writable 会通知 Readable 暂停。

```javascript
// 没有 pipe 的手写背压处理（pipe 会自动处理）
const readable = fs.createReadStream('big-data.csv');
const writable = fs.createWriteStream('output.csv');

readable.on('data', (chunk) => {
  const canContinue = writable.write(chunk); // 返回 false = 缓冲区满了
  if (!canContinue) {
    readable.pause(); // 暂停读取
    writable.once('drain', () => readable.resume()); // 缓冲区清空后恢复
  }
});

// 或者用 pipeline（Node 10+ 推荐，自动处理错误和背压）
const { pipeline } = require('stream/promises');
const zlib = require('zlib');

await pipeline(
  fs.createReadStream('big-data.csv'),
  zlib.createGzip(),
  fs.createWriteStream('big-data.csv.gz')
);
// pipeline 会自动暂停/恢复，不用手动处理背压
```

⚠️ **注意**：如果不用 `pipe` 或 `pipeline`，必须手动处理背压，否则：
- Readable 继续产生数据
- Writable 的缓冲区不断积累
- 内存持续增长，最终 OOM

### 6.4 createReadStream vs readFile

这是面试高频对比题：

| 维度 | createReadStream | readFile |
|------|-----------------|----------|
| 内存占用 | 恒定（highWaterMark 大小）| 全文件大小 |
| 适用场景 | 大文件、流式处理 | 小文件、配置文件 |
| 能否实时处理 | 是（分块回调）| 否（一次性）|
| 能否流式 pipe | 是 | 否 |

```javascript
// readFile：一次性读入内存（适合小文件）
const config = fs.readFileSync('./config.json'); // 同步
const config = await fs.promises.readFile('./config.json'); // 异步
// 500MB 的告警日志文件会直接 OOM

// createReadStream：分块读取（适合任意大小）
const stream = fs.createReadStream('./huge-alert-log.txt', {
  encoding: 'utf8',
  start: 0,      // 从哪里开始读
  end: 1000,     // 读到哪（读取 1KB）
  highWaterMark: 16 * 1024, // 每次读 16KB
});
// 无论文件多大，内存占用稳定在 ~16KB
```

### 6.5 实际应用场景

**场景 1：处理 SOC 日志流**

```javascript
// 实时处理传入的安全日志流（假设来自某个 agent）
const { Transform } = require('stream');

// 自定义 Transform：解析日志行并提取关键信息
class SOCLogParser extends Transform {
  constructor() {
    super({ objectMode: true }); // objectMode 输出对象而非 Buffer
  }

  _transform(line, encoding, callback) {
    try {
      const parsed = JSON.parse(line);
      if (parsed.severity === 'CRITICAL') {
        this.push({ alert: parsed.message, time: parsed.timestamp });
      }
      callback(); // 处理成功
    } catch (e) {
      callback(null); // 跳过无效行
    }
  }
}

process.stdin
  .pipe(new SOCLogParser())
  .on('data', alert => console.log('告警:', alert));
```

**场景 2：HTTP 文件上传**

```javascript
// 用 Stream 处理文件上传，不将文件全部存内存
const http = require('http');
const fs = require('fs');

const server = http.createServer((req, res) => {
  if (req.url === '/upload' && req.method === 'POST') {
    const writeStream = fs.createWriteStream('/tmp/uploaded.file');
    req.pipe(writeStream); // 直接从网络流写入文件，内存占用极低

    writeStream.on('finish', () => {
      res.end('上传完成');
    });
  }
});
```

## 七、V8 垃圾回收

### 7.1 堆内存与栈内存

Node.js 中内存分为栈（Stack）和堆（Heap）两部分：

```javascript
// 栈内存：存储原始类型（number, string, boolean, null, undefined, symbol, bigint）
//        + 存储引用类型的指针（变量指向堆的地址）
let num = 42;              // 栈：变量名 num，值 42
let str = 'hello';         // 栈：变量名 str，值 'hello'
let obj = { a: 1 };        // 栈：变量名 obj，值是堆的地址（指针）
let arr = [1, 2, 3];       // 栈：变量名 arr，值是堆的地址

// 堆内存：存储引用类型的实际数据（对象、数组、函数等）
function foo() {
  const local = { x: 1 }; // 函数调用时在栈上创建，函数结束立即回收
}
foo(); // local 在函数返回后立即从栈上移除
```

**为什么要区分栈和堆？**
- 栈：访问速度快（CPU 直接操作），大小固定，适合存小型固定数据
- 堆：大小动态，适合存运行时创建的对象

### 7.2 分代回收策略

V8 使用分代回收（Generational GC），核心思想是：**大多数对象存活时间很短，回收频繁；少数对象存活时间长，回收代价高**。

```
┌──────────────────────────────────────────────┐
│                   V8 堆内存                    │
├─────────────────┬────────────────────────────┤
│    新生代空间     │         老年代空间           │
│  (1~8MB, Scavenge)│     (可达数百MB, Mark-Sweep) │
│                 │                             │
│  From  │  To    │  存活久的对象（大对象、常量）   │
│  (半满) │ (空闲) │                             │
└─────────────────┴────────────────────────────┘
```

**新生代（Young Generation）**：对象存活时间短，使用 Scavenge 算法
- 空间：1~8MB（可配置 `--max-new-space-size`）
- 算法：From/To 两半空间复制，将存活对象从 From 复制到 To
- 特点：速度快，但空间利用率只有 50%（一半空间永远空着）
- 晋升条件：对象在新生代经历 2 次 GC 仍存活，或 To 空间使用超过 25%

**老年代（Old Generation）**：对象存活时间长，使用 Mark-Sweep + Mark-Compact
- 空间：可达数百 MB
- 标记阶段（Mark）：从根对象（全局变量、调用栈）出发，标记所有可达对象
- 清除阶段（Sweep）：回收未被标记的内存
- 整理阶段（Compact）：移动存活对象，消除内存碎片（代价高，必要时才做）

```javascript
// 监控 Node.js 内存使用
setInterval(() => {
  const mem = process.memoryUsage();
  console.log({
    heapUsed:  Math.round(mem.heapUsed / 1024 / 1024) + 'MB',
    heapTotal: Math.round(mem.heapTotal / 1024 / 1024) + 'MB',
    rss:       Math.round(mem.rss / 1024 / 1024) + 'MB', // 实际物理内存
    external: Math.round(mem.external / 1024 / 1024) + 'MB', // Buffer 等
  });
}, 5000);
```

### 7.3 常见内存泄漏场景

内存泄漏（Memory Leak）是指程序分配的内存不再使用但没有被回收，导致内存持续增长，最终 OOM。

**⚠️ 场景 1：全局变量无限增长**

```javascript
// 错误：在全局对象上无限累积
setInterval(() => {
  global.alertCache = global.alertCache || [];
  global.alertCache.push({ /* 每次告警数据 */ }); // 永远不清理
}, 100);

// 正确：设置上限或定期清理
const alertCache = [];
setInterval(() => {
  alertCache.push({ id: Date.now() });
  if (alertCache.length > 1000) alertCache.shift(); // FIFO 清理
}, 100);
```

**⚠️ 场景 2：闭包引用**

```javascript
// 闭包捕获了外部变量，即使对象不再使用，闭包仍在引用
function createHandler() {
  const hugeData = new Array(1000000); // 占用大量内存

  // handler 函数捕获了 hugeData
  const handler = () => {
    console.log('use:', hugeData.length);
  };

  return handler;
}

// 正确做法：及时释放
function createHandler() {
  const hugeData = new Array(1000000);
  const handler = () => {
    console.log('use:', hugeData.length);
  };
  hugeData = null; // 主动释放引用
  return handler;
}
```

**⚠️ 场景 3：EventEmitter 监听器未移除**

```javascript
// 错误：每次请求都添加监听器，但从不移除
server.on('request', (req) => {
  // req 对象持有大量数据
  req.on('data', chunk => /* 处理 */);
});

// 正确：使用 once 或显式 removeListener
server.on('request', (req) => {
  req.once('data', chunk => /* 处理 */); // 数据到达后自动移除
});

// 或在完成后显式清理
emitter.removeListener('event', handler);
emitter.removeAllListeners('event');
```

**⚠️ 场景 4：定时器未清理**

```javascript
// 错误：组件销毁后定时器仍在运行
function startMonitor() {
  this.intervalId = setInterval(() => {
    checkSystem(); // 持有 this 引用，整个对象无法被回收
  }, 1000);
}

// 正确：提供销毁方法
function stopMonitor() {
  clearInterval(this.intervalId);
  this.intervalId = null;
}
```

### 7.4 内存泄漏排查方法

**方法 1：`node --inspect` + Chrome DevTools**

```bash

node --inspect server.js

# 然后在 Chrome 中打开 chrome://inspect
# 点击 "Open DevTools for Node.js"
# 在 Memory 标签中选择 "Allocation Timeline" 或 "Heap Snapshot"
```

**方法 2：heapdump 模块（生产环境推荐）**

## 八、性能优化

### 8.1 cluster 模块

Node.js 单进程只能用到一个 CPU 核心（V8 是单线程的）。`cluster` 模块通过 fork 多个 Worker 进程，利用多核 CPU。

```javascript
// cluster-example.js
const cluster = require('cluster');
const http = require('http');
const os = require('os');

const numCPUs = os.cpus().length;

if (cluster.isMaster) {
  // 主进程：负责 fork workers 和分发请求
  console.log(`主进程 PID: ${process.pid}, 启动 ${numCPUs} 个 workers`);

  for (let i = 0; i < numCPUs; i++) {
    cluster.fork(); // 启动一个新 worker
  }

  // 监听 worker 退出事件
  cluster.on('exit', (worker, code, signal) => {
    console.log(`Worker ${worker.process.pid} 退出，重新启动...`);
    cluster.fork(); // 自动重启
  });

} else {
  // Worker 进程：运行实际的服务器
  http.createServer((req, res) => {
    res.end(`Handled by worker ${process.pid}\n`);
  }).listen(3000);

  console.log(`Worker ${process.pid} 启动`);
}

// 请求分发：主进程监听端口，轮询分发给 workers
// 多个 worker 共享同一端口，操作系统负责负载均衡
```

⚠️ **注意**：
- 各 Worker 进程内存独立，不共享（不能直接用全局变量共享数据）
- Worker 间通信通过 IPC（`process.send()` / `process.on('message')`）
- 主进程挂了所有 Worker 都会退出，所以主进程要处理异常

### 8.2 pm2 零宕机部署

pm2 是 Node.js 生产环境进程管理器，核心功能是**零宕机重载**（Graceful Reload）。

**零宕机原理**：

```
重载前（假设 2 个 worker）：
  [主进程] ─┬─ Worker A（处理请求中）
            └─ Worker B（处理请求中）

重载时：
  Step 1: 启动新的 Worker C、Worker D
  Step 2: 向 Worker A 发送 SIGTERM，等当前请求处理完
  Step 3: Worker A 请求处理完毕后自动退出
  Step 4: Worker C 接管 Worker A 的流量

结果：始终有 2 个 Worker 在运行，请求不中断
```

```bash
# 基本命令
pm2 start app.js -i 4         # 启动 4 个实例（cluster 模式）
pm2 list                       # 查看进程列表
pm2 logs                       # 查看日志
pm2 reload all                # 零宕机重载所有实例
pm2 restart all               # 重启（有短暂停机）
pm2 stop all                  # 停止

# 生产环境推荐配置
pm2 start app.js \
  -i 0 \                        # 0 = 自动检测 CPU 核数
  --name "soc-api" \           # 命名，方便管理
  --max-memory-restart 512M \  # 内存超限自动重启
  -e err.log -o out.log \      # 日志文件
  --exp-backoff-restart-delay 100  # 崩溃后指数退避重启
```

### 8.3 child_process

`child_process` 用于运行**独立子进程**，适合调用 Python 脚本、FFmpeg、系统命令等 CPU 密集型任务（Node.js 单线程不擅长这类任务）。

```javascript
const { spawn, exec, fork } = require('child_process');

// spawn：流式处理，适合长时间运行的进程
const child = spawn('python3', ['analyze.py', '--input', 'alerts.json']);
child.stdout.on('data', (data) => console.log('stdout:', data.toString()));
child.stderr.on('data', (data) => console.error('stderr:', data.toString()));
child.on('close', (code) => console.log('退出码:', code));

// exec：一次性执行，返回完整输出（适合短命令）
exec('cat package.json | grep version', (err, stdout, stderr) => {
  if (err) { console.error('执行失败:', err); return; }
  console.log('版本:', stdout.trim());
});

// fork：运行另一个 Node.js 脚本，建立 IPC 通道
// 主进程
const worker = fork('./worker.js');
worker.send({ type: 'task', data: 'alert-123' });
worker.on('message', (msg) => console.log('来自 worker:', msg));
worker.on('exit', (code) => console.log('worker 退出:', code));

// worker.js
process.on('message', (msg) => {
  console.log('收到任务:', msg);
  const result = heavyComputation(msg.data);
  process.send({ type: 'result', value: result });
});
```

## 九、安全

### 9.1 原型链污染

原型链污染（Prototype Pollution）是当用户输入能够修改 JavaScript 内置对象（如 `Object.prototype`）的行为，从而影响应用中所有对象的原型链。

```javascript
// 污染前：所有对象都能访问 toString
({}.a) === undefined; // true

// 污染后：所有对象都有了 .a 属性
const payload = JSON.parse('{"__proto__": {"a": "polluted"}}');
Object.assign({}, payload);
// 现在 ({}.a) === "polluted"，原型链被污染了！

// 更隐蔽的攻击：利用 constructor.prototype
const evilPayload = JSON.parse('{"constructor": {"prototype": {"admin": true}}}');
```

**为什么 SOC 系统要特别注意？** SOC 系统处理大量外部输入（告警数据、日志、用户查询），如果合并对象时没有校验输入，极易被原型链污染攻击。

**防御措施：**

```javascript
// 方法 1：使用 Object.create(null) 创建无原型对象
const safeObj = Object.create(null); // 原型链为空，无法污染
safeObj.__proto__; // undefined，无法通过 __proto__ 攻击

// 方法 2：禁止用户输入控制原型属性
function safeMerge(target, source) {
  for (const key of Object.keys(source)) {
    // 排除危险属性
    if (key === '__proto__' || key === 'constructor' || key === 'prototype') {
      continue; // 或 throw Error
    }
    target[key] = source[key];
  }
  return target;
}

// 方法 3：使用 Map 而非 Object 存储用户数据
const userData = new Map(); // Map 不受原型链污染影响

// 方法 4：深层克隆而非浅层合并
function deepClone(obj) {
  return JSON.parse(JSON.stringify(obj)); // 简单但有效（性能差）
  // 或使用 lodash 的 _.cloneDeep()（内部也做了防护）
}
```

### 9.2 eval 危险

`eval()` 将字符串作为 JavaScript 代码执行。如果字符串来自用户输入，攻击者可以注入恶意脚本。

```javascript
// 危险：eval 执行用户输入
const userInput = 'console.log("hacked")';
eval(userInput); // 攻击者直接获得代码执行权！

// 更隐蔽的攻击
const userQuery = '"; require("child_process").exec("rm -rf /"); "';
const code = `processQuery("${userQuery}")`;
// 看似只是调用函数，实际上执行了系统命令

// eval 还会绕过 CSP（Content Security Policy）
// eval 的代码运行在当前作用域，可以访问闭包变量
```

**安全替代方案：**

```javascript
// 场景 1：动态计算（用 Function 代替 eval）
// eval('(a + b) * 2')
const result = new Function('a', 'b', 'return (a + b) * 2')(5, 3);

// 场景 2：表达式求值（用第三方库）
const math = require('mathjs');
const result = math.evaluate('(2 + 3) * 4'); // 安全解析

// 场景 3：模板引擎（代替字符串拼接）
const template = require('art-template');
const html = template.render('{{name}}', { name: '<script>' });
// 自动转义 HTML，<script> 变成 &lt;script&gt;
```

⚠️ **注意**：虽然 `new Function` 比 `eval` 稍微安全一点（不访问当前作用域），但本质上都是执行动态代码，**永远不要对用户输入执行**。

### 9.3 依赖漏洞管理

Node.js 项目的依赖数量庞大（一个典型项目可能有数千个间接依赖），每个依赖都可能存在安全漏洞。

```bash
# 查看依赖树（间接依赖清晰可见）
npm ls

# 审计已知 CVE 漏洞
npm audit

# 自动修复（修改 package-lock.json）
npm audit fix

# 仅修复安全更新，不升级大版本
npm audit fix --force

# 在 CI/CD 中集成（失败则阻止发布）
# package.json 中配置
{
  "scripts": {
    "prepublishOnly": "npm audit --audit-level=high"
  }
}
```

**SOC 平台依赖安全最佳实践：**

| 实践 | 说明 |
|------|------|
| `npm ci` 替代 `npm install` | 确保锁文件版本完全一致 |
| 定期更新依赖 | `npm outdated` 查看可更新的包 |
| 最小化依赖 | 避免引入不必要的包 |
| 私有注册表 | 使用公司内部镜像，过滤高危包 |
| Snyk/Dependabot | 自动监控并修复漏洞 |
| `.npmrc` 设置 | `engine-strict=true` 强制版本校验 |

```bash
# 配置 .npmrc 防止安装不安全的包
# .npmrc
engine-strict=true
audit=true
fund=false
```
