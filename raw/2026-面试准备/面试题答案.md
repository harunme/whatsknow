## 一、计算机基础与算法

### 1.1 网络基础

**A1**：

| 对比项   | HTTP                    | HTTPS                        |
| -------- | ----------------------- | ---------------------------- |
| 传输协议 | 明文传输                | SSL/TLS 加密传输             |
| 默认端口 | 80                      | 443                          |
| 安全性   | 数据可被中间人窃听/篡改 | 加密 + 身份验证 + 完整性校验 |
| 证书     | 不需要                  | 需要 CA 证书                 |

HTTPS 握手过程（TLS 1.2）：

1. **Client Hello**：客户端发送支持的 TLS 版本、加密套件列表、随机数 R1
2. **Server Hello**：服务端选定 TLS 版本和加密套件，返回随机数 R2 + 服务器证书
3. **证书验证**：客户端通过 CA 证书链验证服务器身份
4. **密钥交换**：客户端生成预主密钥（Pre-Master Secret），用服务器公钥加密发送；双方通过 R1 + R2 + 预主密钥计算出会话密钥（Master Secret）
5. **Finished**：双方用会话密钥加密发送验证消息，握手完成，后续通信使用对称加密

TLS 1.3 简化为 1-RTT：Client Hello 时直接发送密钥交换参数，省去一次往返。

**A2**：

**三次握手**：

1. Client → SYN, seq=x → Server
2. Server → SYN+ACK, seq=y, ack=x+1 → Client
3. Client → ACK, seq=x+1, ack=y+1 → Server

为什么不能两次：防止已失效的连接请求到达服务器。如果 Client 发的旧 SYN 因网络延迟才到 Server，Server 回 SYN+ACK 就建立连接浪费资源；三次握手让 Client 确认这是当前请求才真正建立连接。

**四次挥手**：

1. Client → FIN → Server（Client 进入 FIN_WAIT_1）
2. Server → ACK → Client（Server 进入 CLOSE_WAIT，Client 进入 FIN_WAIT_2）
3. Server → FIN → Client（Server 进入 LAST_ACK）
4. Client → ACK → Server（Client 进入 TIME_WAIT，2MSL 后关闭）

为什么是四次不是三次：FIN 和 ACK 分开发送。Server 收到 FIN 后可能还有数据没发完，先回 ACK，等数据发完再发 FIN。

**TIME_WAIT 的作用**：

- 保证最后一个 ACK 能到达对方（若丢失，对方重发 FIN，TIME_WAIT 状态可重发 ACK）
- 等待网络中残留的延迟报文消失，避免新连接收到旧连接的报文
- 持续时间 2MSL（Maximum Segment Lifetime）

**A3**：

**DNS 解析流程**：

1. 浏览器 DNS 缓存 → OS DNS 缓存 → hosts 文件
2. 本地 DNS 服务器（递归查询）
3. 根域名服务器 → 返回顶级域名服务器地址
4. 顶级域名服务器（.com）→ 返回权威域名服务器地址
5. 权威域名服务器 → 返回目标 IP
6. 本地 DNS 服务器缓存结果，返回给浏览器

**从输入 URL 到页面渲染**：

1. **DNS 解析**：域名 → IP
2. **TCP 三次握手**：建立连接（HTTPS 还需 TLS 握手）
3. **发送 HTTP 请求**：请求行 + 请求头 + 请求体
4. **服务器处理请求**：返回 HTTP 响应
5. **浏览器解析渲染**：
   - 解析 HTML → 构建 DOM 树
   - 解析 CSS → 构建 CSSOM 树
   - DOM + CSSOM → 渲染树（Render Tree）
   - 布局（Layout/Reflow）：计算节点位置大小
   - 绘制（Paint）：像素绘制
   - 合成（Composite）：分层合成最终画面
6. **执行 JS**：遇到 `<script>` 阻塞 DOM 构建（async/defer 可优化）

**A4**：

| 对比项     | HTTP 长轮询                          | WebSocket                    |
| ---------- | ------------------------------------ | ---------------------------- |
| 通信模式   | 客户端发起，服务端挂起响应直到有数据 | 全双工，双方可随时发消息     |
| 连接开销   | 每次请求/响应都有 HTTP 头开销        | 握手后只传数据帧，无额外头   |
| 实时性     | 有延迟（取决于轮询间隔）             | 真正实时                     |
| 服务器资源 | 大量挂起连接占用线程/资源            | 一个 TCP 连接即可            |
| 复杂度     | 实现简单                             | 需要维护连接状态、心跳、重连 |

**选择 WebSocket 的场景**：

- SOC 平台的实时告警推送：新告警产生后需要立刻推送到前端
- 大屏互动、在线直播弹幕（简历中鲸会务项目）
- 协同编辑、实时监控仪表盘

**选择长轮询的场景**：

- 消息频率低、实时性要求不高
- 只需兼容老浏览器、网络环境限制 WebSocket

**A5**：

**Nacos 工作原理**：

- **服务注册**：服务实例启动时向 Nacos Server 注册自己的 IP、端口、服务名等信息
- **心跳保活**：临时实例通过心跳（默认 5s）维持注册状态，超过 15s 标记不健康，超过 30s 剔除；持久实例由 Nacos 主动探测
- **服务发现**：客户端订阅服务列表，Nacos 通过 UDP 推送变更 + 客户端定时拉取
- **配置中心**：支持配置的动态下发，长轮询监听配置变更

**三者对比**：

| 对比项     | Nacos                                | Eureka                 | Consul               |
| ---------- | ------------------------------------ | ---------------------- | -------------------- |
| CAP 模型   | AP/CP 可切换                         | AP                     | CP                   |
| 健康检查   | 心跳 + TCP/HTTP 探测                 | 客户端心跳             | Agent 探测           |
| 配置中心   | 内置支持                             | 不支持                 | 支持（KV 存储）      |
| 一致性协议 | Raft（持久实例）/ Distro（临时实例） | 无强一致               | Raft                 |
| 雪崩保护   | 支持                                 | 支持                   | 不支持               |
| 适用场景   | Spring Cloud 生态、国内主流          | Netflix 生态（已停更） | 多语言、Service Mesh |

**项目中的使用**：SOC 平台 Node 层（Koa BFF）通过 Nacos 发现后端微服务地址，实现动态路由。当后端服务扩缩容时，Node 层自动感知新实例，无需手动修改配置。
