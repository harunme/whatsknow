## 1. SOAR 定义与核心能力

### 1.1 什么是 SOAR

**SOAR = Security Orchestration, Automation and Response**

拆解三个单词的真实含义：

- **Orchestration（编排）**：协调多个安全工具按特定顺序工作。比如收到告警后，先查 IP 威胁情报，再调 EDR 隔离主机，最后发 Slack 通知——这些动作之间存在顺序依赖和数据传递，就是"编排"。
- **Automation（自动化）**：把人工重复操作写成剧本自动执行。典型场景：每分钟都有钓鱼告警，分析师不需要次次手动封 IP，剧本自动完成。
- **Response（响应）**：实际执行动作——封禁账号、隔离主机、关闭端口。响应必须可靠，不能执行了却没有确认结果。

**一句话总结**：SOAR 是安全工具的"导演"，告诉谁在什么时间做什么事，并确保每一步都成功或被妥善处理。

### 1.2 SOAR 在安全运营体系中的位置

```
[SIEM 告警流]              [EDR 事件流]
     │                         │
     ▼                         ▼
 ┌───────────────────────────────┐
 │         SOAR 引擎            │
 │   （消费告警，执行动作）        │
 └───────────────────────────────┘
     │
     ▼
[防火墙 API] [邮件网关 API] [工单系统 API] [通知渠道]
```

**三个工具的分工对比**：

| 工具 | 职责 | 执行动作 | 代表产品 |
|------|------|---------|---------|
| SIEM | 收集日志、关联分析、生成告警 | **不执行** | Splunk, Elastic |
| EDR | 终端检测、杀进程、隔离主机 | **终端侧**执行 | CrowdStrike, SentinelOne |
| SOAR | 编排响应流程、协调多工具 | **跨系统**执行 | Splunk SOAR, Palo Alto XSIAM |

⚠️ **注意**：很多公司误以为买了 SIEM 就有了自动化响应能力。实际上 SIEM 只"看"，SOAR 才"动"。两者是互补关系，不是替代关系。

### 1.3 告警生命周期与 P0-P3 定级

告警从产生到关闭经历完整链路：

```
产生(Generate) → 分诊(Triage) → 分析(Analysis)
→ 调查(Investigation) → 响应(Response) → 关闭(Closure)
```

**分诊**阶段决定剧本走哪条路径（P0 立即执行，P3 先通知），**响应**阶段是 SOAR 的主战场。

告警级别定义（决定 SLA 和走哪个剧本）：

| 级别 | 典型场景 | SLA 响应时间 | 剧本类型 |
|------|---------|------------|---------|
| **P0** | 勒索软件、高管钓鱼、已执行命令 | 15 分钟 | 全自动 + 紧急人工审批 |
| **P1** | 横向移动、凭据泄露、已确认入侵 | 1 小时 | 半自动 + 标准审批 |
| **P2** | 可疑异常行为、政策违规 | 4 小时 | 半自动 |
| **P3** | 扫描行为、误报清理 | 24 小时 | 纯通知类 |

### 1.4 MTTD / MTTR — 安全运营的核心指标

```
MTTD（Mean Time to Detect，平均检测时间）
  = 从攻击发生到系统发现的时间
  → 越小越好，说明检测能力强

MTTR（Mean Time to Respond，平均响应时间）
  = 从发现告警到响应完成的时间
  → 越小越好，说明自动化能力强

SOAR 的核心价值 = 降低 MTTR
（通过自动化减少人工介入时间）
```

## 2. 安全剧本 Playbook 是什么

### 2.1 剧本的本质：有向状态图

Playbook（剧本）在计算机科学中的抽象是**有向状态图（Directed State Graph）**：

- **节点（Node）** = 一个个操作步骤（查情报、执行封禁、等待审批）
- **边（Edge）** = 节点之间的条件转移（情报分数 > 80 → 执行封禁）
- **初始状态** = 告警进入剧本
- **终止状态** = 响应完成或升级人工

```
[Trigger: 告警进入]
        │
        ▼
[Action: 提取 IOC] ──(context["ip"])──► [Condition: 威胁分数?]
                                         │         │
                              [>80: 封禁] │    [<80: 记录并放行]
                                         ▼         ▼
                               [Approval: 人工审批]  [End: 关闭]
                                         │
                                         ▼
                               [Action: 执行封禁]
                                         │
                                         ▼
                               [Notification: 通知分析师]
                                         │
                                         ▼
                                      [End]
```

### 2.2 为什么用 YAML 描述剧本

很多人会问：直接写 Python 代码不行吗？为什么要用 YAML？

**核心原因：让非工程师（安全分析师、SOC 运维）能读能改。**

安全剧本需要频繁调整规则（比如"这次钓鱼活动要临时加一个审批步骤"），不能让每一次调整都走工程师代码评审流程。YAML 的优势：

1. **人类可读**：安全分析师直接看懂"第3步是审批"而非看 Python 代码
2. **结构化**：引擎可以解析、校验、版本化
3. **与前端集成**：可视化拖拽的底层配置就是 YAML/JSON

⚠️ **注意**：YAML 不是银弹。复杂业务逻辑（循环、聚合、动态分支）在 YAML 里表达能力很弱，这时候就需要 Python Class 补位——这就是双轨设计的来源。

### 2.3 YAML 剧本基础结构

```yaml
# playbook_phishing_ip_block.yaml
name: phishing_ip_block                    # 全局唯一标识
version: "2.1.0"                           # 版本号（升级时重要）
description: "钓鱼 IP 自动封禁剧本"

# 触发条件：SIEM/EDR 告警进入
trigger:
  type: alert
  severity: [P0, P1]                       # 只接收 P0/P1 告警

# 全局配置
config:
  approval_timeout: 3600                  # 审批超时 1 小时
  rollback_on_failure: true               # 节点失败自动回滚

# 节点定义
nodes:
  - id: extract_ip
    type: action
    action: threat_intel.lookup_ip
    input:
      ip: "{{ alert.source_ip }}"         # 从告警中提取 IP

  - id: check_score
    type: condition
    condition: "context.score >= 80"
    branches:
      high:
        - id: auto_block
          type: action
          action: firewall.block_ip
      low:
        - id: notify_only
          type: notification
          channel: slack
```

## 3. DSL 双轨：YAML 与 Python 并行设计

### 3.1 什么是 DSL，为什么需要双轨

**DSL（Domain Specific Language，领域特定语言）** 是专为某个专业领域设计的语言。SOAR 的 DSL 需要同时满足两类人：

| 用户 | 需求 | DSL 方案 |
|------|------|---------|
| 安全分析师 | 声明意图、改规则、不写代码 | YAML |
| 工程师 | 复杂逻辑、循环、异常处理 | Python Class |

**双轨架构的工作流程**：

```
[安全分析师编辑 YAML]
          │
          ▼（编译阶段）
[Python Class 自动生成]
          │
          ▼（执行阶段）
[剧本引擎运行]
```

### 3.2 YAML 的编译：将声明转为代码

YAML 配置需要被"翻译"成可执行对象。核心思路是**配置即代码**：

```python
from dataclasses import dataclass, field
from typing import Any, Callable
import yaml

@dataclass
class PlaybookNode:
    """剧本节点基类"""
    id: str
    type: str                              # action/condition/approval/...
    next: list[str] = field(default_factory=list)

@dataclass
class ActionNode(PlaybookNode):
    """动作节点"""
    action_name: str = ""                  # e.g. "firewall.block_ip"
    input_mapping: dict[str, str] = field(default_factory=dict)

@dataclass
class ConditionNode(PlaybookNode):
    """条件节点"""
    condition: str = ""                   # e.g. "context.score >= 80"
    branches: dict[str, list[str]] = field(default_factory=dict)

# YAML → Python Class 的编译函数
class PlaybookCompiler:
    def compile(self, yaml_path: str) -> "Playbook":
        with open(yaml_path) as f:
            raw = yaml.safe_load(f)

        nodes: dict[str, PlaybookNode] = {}
        for nd in raw["nodes"]:
            node_type = nd["type"]
            if node_type == "action":
                nodes[nd["id"]] = ActionNode(
                    id=nd["id"],
                    type="action",
                    action_name=nd.get("action", ""),
                    input_mapping=nd.get("input", {}),
                    next=nd.get("next", [])
                )
            elif node_type == "condition":
                nodes[nd["id"]] = ConditionNode(
                    id=nd["id"],
                    type="condition",
                    condition=nd.get("condition", ""),
                    branches=nd.get("branches", {})
                )
            # ... 其他节点类型同理

        return Playbook(name=raw["name"], nodes=nodes)

print("✅ YAML 编译为 Python 对象成功")
```

**为什么要这样设计？** YAML 解决"说什么"的问题，Python Class 解决"怎么做"的问题。两者各司其职，互不耦合。

### 3.3 Python DSL：复杂逻辑的归宿

当 YAML 表达力不足时，直接写 Python 类。典型场景：聚合多个情报源结果、做循环重试。

```python
class PhishingIPBlockPlaybook:
    """
    钓鱼 IP 封禁剧本（Python DSL 版本）
    适合复杂业务逻辑，不需要 YAML 中转
    """
    def __init__(self, dependencies: dict):
        self.intel = dependencies["threat_intel"]
        self.firewall = dependencies["firewall"]
        self.notification = dependencies["notification"]

    async def execute(self, alert: dict) -> ExecutionResult:
        # Step 1: 提取 IOC
        ip = alert["source_ip"]
        domain = alert.get("domain", "")

        # Step 2: 查多个情报源（并发请求）
        intel_results = await asyncio.gather(
            self.intel.lookup_ip(ip),           # VirusTotal
            self.intel.lookup_ip_abuse(ip),      # AbuseIPDB
            self.intel.lookup_domain(domain)     # WHOIS
        )

        # Step 3: 聚合评分（加权平均）
        aggregated_score = self._compute_score(intel_results)

        if aggregated_score >= 80:
            # Step 4a: 执行封禁
            await self.firewall.block_ip(ip, reason=f"威胁分数={aggregated_score}")
            await self.notification.send(
                channel="soc-alerts",
                message=f"🔴 已自动封禁恶意 IP: {ip}（分数={aggregated_score}）"
            )
            return ExecutionResult(success=True, action="blocked")

        else:
            # Step 4b: 低于阈值，记录但不阻断
            return ExecutionResult(success=True, action="logged_only")

    def _compute_score(self, results: list) -> float:
        """内部评分算法"""
        weights = {"virustotal": 0.5, "abuseipdb": 0.3, "whois": 0.2}
        return sum(
            r["score"] * weights.get(r["source"], 0.2)
            for r in results
        )

# 典型输出示例
async def demo():
    dependencies = {
        "threat_intel": MockThreatIntel(),
        "firewall": MockFirewall(),
        "notification": MockNotification(),
    }
    playbook = PhishingIPBlockPlaybook(dependencies)

    result = await playbook.execute({
        "id": "alert-001",
        "source_ip": "198.51.100.42",
        "domain": "evil-phish.example.com",
        "severity": "P1"
    })

    print(f"执行结果: {result.action}")       # 输出: blocked
    print(f"成功标志: {result.success}")     # 输出: True

import asyncio
asyncio.run(demo())
```

## 4. 节点类型详解

### 4.1 节点类型总览

| 节点类型 | 作用 | 状态变化 | 典型场景 |
|---------|------|---------|---------|
| `trigger` | 剧本入口，接收外部事件 | — | 告警进入、手动触发、定时触发 |
| `action` | 执行具体操作（API 调用） | pending → running → success/failed | 查情报、封禁 IP、创工单 |
| `condition` | 条件路由 | — | 分数判断、类型判断 |
| `approval` | 暂停等待人工 | running → pending_approval | 封禁前确认、高危操作复核 |
| `notification` | 发送通知 | — | Slack、邮件、短信 |
| `loop` | 循环处理集合 | — | 批量封禁多个 IP |
| `transform` | 数据转换 | — | 提取字段、格式转换 |

### 4.2 Action 节点

Action 节点是最基础的执行单元。它需要：
- 唯一 ID
- action_name（调用哪个依赖服务）
- input_mapping（告警字段 → API 参数的映射）
- timeout（超时时间，防止挂死）

```python
@dataclass
class ActionConfig:
    id: str = "block_ip"
    action_name: str = "firewall.block_ip"
    timeout_seconds: int = 30
    retry_policy: RetryPolicy = RetryPolicy(max_attempts=3, backoff=2.0)
    input_mapping: dict[str, str] = field(default_factory=lambda: {
        "ip": "{{ context.target_ip }}",    # 从上下文取 IP
        "duration": "{{ config.block_hours }}",  # 从配置取时长
    })
    output_key: str = "block_result"        # 执行结果存入上下文的 key
```

**Action 的执行语义**（重要）：

```python
async def execute_action(config: ActionConfig, context: PlaybookContext) -> NodeResult:
    """
    动作节点的标准执行流程
    """
    # 1. 参数渲染：从模板字符串替换真实值
    rendered_input = render_template(config.input_mapping, context)

    # 2. 执行动作（带超时和重试）
    try:
        result = await with_timeout(
            config.timeout_seconds,
            call_action(config.action_name, rendered_input)
        )
    except TimeoutError:
        return NodeResult(status="timeout", error="Action exceeded timeout")
    except Exception as e:
        return NodeResult(status="failed", error=str(e))

    # 3. 存储结果到上下文，供后续节点使用
    context.set(config.output_key, result)

    # 4. 决定下一步走向（默认 success 分支）
    return NodeResult(status="success", next_node=config.next)
```

⚠️ **注意**：Action 节点的 output 必须存入 context，否则后续节点无法使用执行结果。比如查情报得到分数，后续条件节点需要用这个分数做判断。

### 4.3 Condition 节点

Condition 节点不做实际操作，只做路由决策。它的输入是当前 context，输出是下一步节点 ID。

```python
class ConditionNode:
    def evaluate(self, context: PlaybookContext) -> str:
        """
        根据条件表达式返回下一个节点 ID
        """
        expression = self.config.condition  # e.g. "context.score >= 80"

        # 安全执行表达式（不允许任意代码执行！）
        allowed_names = {"context": context}
        result = eval(expression, {"__builtins__": {}}, allowed_names)

        # 根据结果选分支
        if result:
            return self.config.branches.get("true", self.config.next)
        else:
            return self.config.branches.get("false", self.config.next)

# 使用示例
def test_condition_routing():
    context = PlaybookContext()
    context.set("score", 85)

    node = ConditionNode(
        condition="context.score >= 80",
        branches={"true": "auto_block", "false": "notify_only"}
    )

    next_id = node.evaluate(context)
    print(f"条件 score>=80 为 True，下一步: {next_id}")  # 输出: auto_block
```

⚠️ **注意**：Condition 节点的 condition 字段**绝不能直接 eval 用户输入**。必须使用安全的表达式解析器（如 `ast.literal_eval` 或专门的条件解析器），防止注入攻击。

### 4.4 Approval 节点（详见第 6 节）

Approval 节点会将整个剧本的执行暂停，等待人工审批结果。引擎需要维护一个"审批等待表"。

### 4.5 Notification 节点

Notification 是纯副作用节点（不修改 context），用于通知干系人。

```python
@dataclass
class NotificationConfig:
    channel: str                   # slack / email / sms
    template: str                  # "IP {ip} 已被封禁，原因: {reason}"
    recipients: list[str]          # 用户 ID 或邮件地址
    priority: str = "normal"       # normal / high / urgent

async def send_notification(config: NotificationConfig, context: PlaybookContext):
    message = render_template(config.template, context)
    channels = {
        "slack": slack_client.send,
        "email": smtp_client.send,
        "sms": sms_client.send,
    }
    sender = channels[config.channel]
    await sender(config.recipients, message, priority=config.priority)
```

### 4.6 Loop 节点

处理批量操作（如一次告警包含 10 个恶意 IP），避免重复写 for 循环。

```yaml
nodes:
  - id: process_batch
    type: loop
    items: "{{ alert.malicious_ips }}"     # 迭代的集合
    item_var: "current_ip"                 # 循环变量名
    body:
      - id: block_each
        type: action
        action: firewall.block_ip
        input:
          ip: "{{ current_ip }}"
```

## 5. 状态机设计与上下文传递

### 5.1 为什么 SOAR 引擎是状态机

剧本是一个有向图，引擎执行剧本就是在图上做**状态转移**。每个时刻，剧本实例（Playbook Instance）处于某个节点，上下文包含历史执行数据。

**关键概念区分**：

```
Playbook（剧本模板）  →  类（Class），定义流程结构，不包含数据
Playbook Instance   →  实例（Instance），一次具体的告警处理执行
Context（上下文）    →  实例的运行时数据，节点之间共享
```

### 5.2 状态机实现

```python
from enum import Enum
from dataclasses import dataclass, field
from typing import Any
import asyncio

class NodeState(Enum):
    """节点状态枚举"""
    IDLE = "idle"                 # 未执行
    PENDING = "pending"           # 等待中（如等待审批）
    RUNNING = "running"           # 执行中
    SUCCESS = "success"           # 执行成功
    FAILED = "failed"             # 执行失败
    TIMEOUT = "timeout"           # 超时
    SKIPPED = "skipped"           # 跳过（如条件不满足）

@dataclass
class PlaybookContext:
    """
    剧本运行时上下文——所有节点共享的数据存储
    相当于一次剧本执行过程中的"全局变量"
    """
    alert: dict                   # 原始告警数据
    variables: dict[str, Any] = field(default_factory=dict)
    node_results: dict[str, NodeResult] = field(default_factory=dict)
    created_at: str = ""
    playbook_id: str = ""

    def get(self, key: str, default: Any = None) -> Any:
        """从上下文获取值，支持嵌套 key 如 'alert.source_ip'"""
        keys = key.split(".")
        val = self.variables if key.startswith("context") else self.alert
        for k in keys:
            if isinstance(val, dict):
                val = val.get(k)
            else:
                return default
        return val if val is not None else default

    def set(self, key: str, value: Any) -> None:
        """存入上下文，供后续节点读取"""
        self.variables[key] = value

@dataclass
class NodeResult:
    """节点执行结果"""
    status: NodeState
    output: Any = None
    error: str = ""
    duration_ms: int = 0

class PlaybookStateMachine:
    """
    剧本状态机引擎
    负责按拓扑顺序执行节点，处理状态转移
    """

    def __init__(self, playbook: "Playbook", dependencies: dict):
        self.playbook = playbook
        self.dependencies = dependencies
        # 节点执行器注册表
        self.executors: dict[str, Callable] = {
            "action": self._exec_action,
            "condition": self._exec_condition,
            "approval": self._exec_approval,
            "notification": self._exec_notification,
            "loop": self._exec_loop,
        }

    async def run(self, alert: dict) -> PlaybookContext:
        """运行剧本，返回最终上下文"""
        context = PlaybookContext(alert=alert, playbook_id=self.playbook.name)
        current_node_id = self.playbook.entry_node_id  # 从 trigger 开始

        # 图遍历：循环直到遇到终止节点
        while current_node_id and current_node_id != "END":
            node = self.playbook.nodes[current_node_id]
            print(f"[执行节点] {node.id} (type={node.type})")

            # 调用对应类型的执行器
            executor = self.executors.get(node.type)
            if not executor:
                raise ValueError(f"未知节点类型: {node.type}")

            result = await executor(node, context)
            context.node_results[node.id] = result

            # 决定下一步
            if result.status == NodeState.SUCCESS:
                current_node_id = result.next_node or self._get_default_next(node)
            elif result.status == NodeState.PENDING_APPROVAL:
                # 审批节点：保存状态，等待外部触发恢复
                await self._persist_and_wait(node.id, context)
                current_node_id = "END"  # 审批完成后从外部 resume
            else:
                # 失败/超时：停止执行
                print(f"[状态机] 节点 {node.id} 状态异常: {result.status}")
                break

        return context

    async def _exec_action(self, node, context) -> NodeResult:
        """执行动作节点"""
        import time
        start = time.time()

        try:
            # 调用依赖服务（mock 示例）
            service_name, method_name = node.config.action_name.split(".")
            service = self.dependencies[service_name]
            method = getattr(service, method_name)

            output = await asyncio.wait_for(
                method(**node.config.input_mapping),
                timeout=node.config.timeout_seconds
            )

            # 存入上下文
            context.set(node.config.output_key, output)

            return NodeResult(
                status=NodeState.SUCCESS,
                output=output,
                duration_ms=int((time.time() - start) * 1000),
                next_node=node.next[0] if node.next else None
            )
        except asyncio.TimeoutError:
            return NodeResult(status=NodeState.TIMEOUT, error="执行超时")
        except Exception as e:
            return NodeResult(status=NodeState.FAILED, error=str(e))

    async def _exec_condition(self, node, context) -> NodeResult:
        """执行条件节点"""
        expr = node.config.condition
        # 安全评估条件表达式
        result = self._safe_eval(expr, {"context": context})
        next_node = node.config.branches.get(
            "true" if result else "false",
            node.next[0] if node.next else None
        )
        return NodeResult(status=NodeState.SUCCESS, next_node=next_node)

    def _safe_eval(self, expr: str, scope: dict) -> bool:
        """
        ⚠️ 安全注意：绝不允许在 eval 中执行任意代码
        这里仅做简单数值/逻辑比较，不支持函数调用
        """
        import ast
        # 实际生产中应使用 ast.parse 验证只包含合法的比较操作
        allowed = {"True": True, "False": False, "None": None}
        allowed.update(scope)
        return eval(expr, {"__builtins__": {}}, allowed)

    async def _exec_approval(self, node, context) -> NodeResult:
        """执行审批节点：挂起等待"""
        return NodeResult(status=NodeState.PENDING_APPROVAL, next_node=None)

    async def _exec_notification(self, node, context) -> NodeResult:
        """执行通知节点"""
        ch = self.dependencies.get(node.config.channel)
        if ch:
            await ch.send(
                recipients=node.config.recipients,
                message=render_template(node.config.template, context)
            )
        return NodeResult(status=NodeState.SUCCESS, next_node=node.next[0] if node.next else None)

    async def _persist_and_wait(self, node_id: str, context: PlaybookContext):
        """审批节点特殊处理：序列化状态，等待外部恢复"""
        print(f"[状态机] 节点 {node_id} 等待审批，状态已持久化...")
        # 实际实现：写入数据库，外部触发 resume()
        # await db.save_playbook_instance(context)

    def _get_default_next(self, node) -> str | None:
        return node.next[0] if node.next else None

print("✅ 状态机基础框架定义完成")
```

## 7. 资源锁 ResourceLock 与并发控制

### 7.1 什么是竞态条件

SOAR 引擎的并发问题场景：

```
剧本实例 A（告警-001）：检测到 IP 198.51.100.42 为恶意，准备封禁
剧本实例 B（告警-002）：同一 IP 也被检测为恶意，准备封禁
     │
     ├─► A 执行：firewall.block_ip("198.51.100.42")  ✓ 成功
     │
     └─► B 执行：firewall.block_ip("198.51.100.42")  ✓ 成功
          （防火墙幂等，无问题，但浪费了 1 次 API 调用）

更严重的情况：
剧本实例 A：提取 IP 198.51.100.42
剧本实例 B：在 A 还没执行封禁时，误将同一 IP 解封 ← 竞态！
```

### 7.2 ResourceLock 的实现

**核心思路：用 `asyncio.Lock` 对同一个资源 key 加锁，保证同一时间只有一个剧本实例能操作该资源。**

```python
import asyncio
from dataclasses import dataclass, field
from typing import Any

@dataclass
class ResourceLock:
    """
    资源锁：防止多个剧本实例同时操作同一资源

    典型使用场景：
    - 同一个 IP 的并发封禁/解封操作
    - 同一个用户账号的并发锁定操作
    - 共享外部资源（如工单系统的同一工单）
    """
    key: str                        # 资源唯一标识，如 "ip:198.51.100.42"
    owner_id: str                   # 持锁者（剧本实例 ID）
    acquired_at: float = 0.0        # 持锁时间戳

class ResourceLockManager:
    """
    全局资源锁管理器
    维护所有活跃锁的状态
    """

    def __init__(self):
        # _locks: key → asyncio.Lock
        # 每个资源 key 对应一个锁，保证对同一资源的串行访问
        self._locks: dict[str, asyncio.Lock] = {}
        # _owners: key → owner_id（持锁者记录，用于调试和死锁检测）
        self._owners: dict[str, str] = {}
        # _timers: key → 定时器任务（用于自动释放超时锁）
        self._timers: dict[str, asyncio.Task] = {}

    def _get_lock(self, key: str) -> asyncio.Lock:
        """懒加载：为每个新 key 创建独立的锁对象"""
        if key not in self._locks:
            self._locks[key] = asyncio.Lock()
        return self._locks[key]

    async def acquire(
        self,
        key: str,
        owner_id: str,
        timeout: float = 30.0
    ) -> bool:
        """
        尝试获取资源的独占锁

        参数:
            key: 资源标识（e.g. "ip:198.51.100.42"）
            owner_id: 持锁者 ID（用于记录和调试）
            timeout: 获取锁的最大等待时间

        返回:
            True = 成功获取锁
            False = 超时未获取到（可能存在死锁）
        """
        lock = self._get_lock(key)

        try:
            # await lock.acquire() 会阻塞直到获取锁
            # 用 wait_for 添加超时，防止永久阻塞
            await asyncio.wait_for(lock.acquire(), timeout=timeout)
            self._owners[key] = owner_id
            self._locks[key].locked_time = asyncio.get_event_loop().time()

            print(f"[ResourceLock] ✅ 成功获取锁: {key} (owner={owner_id})")
            return True

        except asyncio.TimeoutError:
            # 获取锁超时：可能是死锁（持锁者异常未释放）
            print(f"[ResourceLock] ❌ 获取锁超时: {key} (owner={owner_id}, "
                  f"当前持锁者={self._owners.get(key, 'unknown')})")
            return False

    def release(self, key: str, owner_id: str) -> bool:
        """
        释放锁（只能由持锁者释放，防止误释放他人锁）
        """
        if self._owners.get(key) != owner_id:
            print(f"[ResourceLock] ⚠️ 非法释放: {key} 由 {owner_id} 尝试释放，"
                  f"但实际持锁者是 {self._owners.get(key)}")
            return False

        lock = self._locks.get(key)
        if lock and lock.locked():
            lock.release()
            del self._owners[key]
            print(f"[ResourceLock] 🔓 释放锁: {key}")
            return True
        return False

    def get_owner(self, key: str) -> str | None:
        """查询某个资源的当前持锁者"""
        return self._owners.get(key)

    def get_locked_resources(self) -> list[str]:
        """获取所有当前被锁定的资源（用于监控）"""
        return [k for k, lock in self._locks.items() if lock.locked()]

    async def set_auto_release(self, key: str, delay_seconds: int):
        """
        设置自动释放定时器：防止持锁者异常退出导致死锁
        ⚠️ 建议每个锁都设置自动释放超时
        """
        async def _auto_release():
            await asyncio.sleep(delay_seconds)
            owner = self._owners.get(key)
            if owner:
                self.release(key, owner)
                print(f"[ResourceLock] ⏰ 自动释放超时锁: {key} (owner={owner})")

        self._timers[key] = asyncio.create_task(_auto_release())


# 在剧本引擎中集成资源锁
class SecurePlaybookEngine(ResourceLockManager):
    """带资源锁保护的剧本引擎"""

    async def execute_with_lock(
        self,
        playbook_id: str,
        resource_key: str,
        action_fn: callable
    ):
        """
        带锁保护的执行：确保同一资源不会并发操作
        """
        # Step 1: 获取锁
        acquired = await self.acquire(resource_key, playbook_id, timeout=30)
        if not acquired:
            return {"error": "资源忙，请重试", "code": "LOCK_TIMEOUT"}

        # Step 2: 设置自动释放（防止死锁）
        asyncio.create_task(self.set_auto_release(resource_key, delay_seconds=120))

        try:
            # Step 3: 执行动作
            return await action_fn()
        finally:
            # Step 4: 无论如何都要释放锁（放在 finally 确保执行）
            self.release(resource_key, playbook_id)

async def demo_lock_scenario():
    """
    演示并发锁场景：
    两个剧本实例同时要封禁同一 IP
    """
    engine = SecurePlaybookEngine()
    resource = "ip:198.51.100.42"

    async def block_ip(instance_id: str) -> dict:
        """模拟封禁 IP 的动作"""
        acquired = await engine.acquire(resource, instance_id, timeout=5)
        if not acquired:
            return {"error": f"{instance_id} 获取锁失败"}

        try:
            # 模拟 API 调用
            await asyncio.sleep(1)
            print(f"[{instance_id}] 执行封禁: {resource}")
            return {"success": True, "instance": instance_id}
        finally:
            engine.release(resource, instance_id)

    # 并发执行两个剧本实例
    results = await asyncio.gather(
        block_ip("alert-001"),
        block_ip("alert-002"),  # 这个会等 alert-001 释放锁
    )

    print(f"\n结果汇总:")
    for r in results:
        print(f"  {r}")

asyncio.run(demo_lock_scenario())
```

## 8. 回滚机制：幂等设计与补偿事务

### 8.1 为什么需要回滚机制

安全剧本执行的是真实的生产操作（封禁 IP、隔离主机），一旦执行失败：

1. **已执行的操作留下副作用**（IP 被封了，但后续的情报查询失败）
2. **状态不一致**：告警关闭了但封禁没执行
3. **无法追溯**：不知道做到哪一步停下来的

**回滚机制 = 事后补救，保证"要么全做，要么全不做"的原子性。**

### 8.2 幂等设计：根本性解法

**幂等（Idempotent）** = 同一个操作执行多次，结果相同。幂等是最优雅的回滚方式——不需要回滚，因为重复执行没有副作用。

```python
class FirewallService:
    """带幂等保证的防火墙服务"""

    async def block_ip(self, ip: str, reason: str = "") -> dict:
        """
        幂等封禁 IP：使用请求 ID + 幂等键
        关键：每次封禁操作带上唯一的 operation_id
        防火墙服务根据 operation_id 判断是否已执行
        """
        operation_id = f"block_{ip}_{reason[:20]}"  # 构造幂等键

        # 防火墙服务端存储操作记录：
        # 如果 operation_id 已存在 → 返回"已执行过，不重复操作"
        # 如果 operation_id 不存在 → 执行封禁

        response = await self._call_api(
            "POST", "/firewall/block",
            json={"ip": ip, "operation_id": operation_id, "reason": reason}
        )
        return response

    async def unblock_ip(self, ip: str, reason: str = "") -> dict:
        """
        幂等解封 IP：解封是封禁的逆操作，也需要幂等
        """
        operation_id = f"unblock_{ip}"  # 构造幂等键

        response = await self._call_api(
            "POST", "/firewall/unblock",
            json={"ip": ip, "operation_id": operation_id, "reason": reason}
        )
        return response

# ⚠️ 幂等的判断在服务端，不在客户端
# 客户端发请求时带幂等键，服务端去重
```

### 8.3 补偿事务：没有幂等时的补救

当操作不支持幂等时，需要**补偿事务（Compensating Transaction）**——执行逆操作来撤销。

```python
from dataclasses import dataclass
from enum import Enum
from typing import Callable

class RollbackStrategy(Enum):
    """回滚策略"""
    IDEMPOTENT = "idempotent"        # 本身幂等，直接重试
    COMPENSATE = "compensate"        # 执行逆操作补偿
    ESCALATE = "escalate"            # 无法处理，升级人工
    SKIP = "skip"                    # 可忽略，不影响最终结果

@dataclass
class NodeWithRollback:
    """带回滚能力的节点定义"""
    node_id: str
    execute_fn: Callable             # 正向执行函数
    compensate_fn: Callable | None   # 补偿函数（可选）
    rollback_strategy: RollbackStrategy = RollbackStrategy.IDEMPOTENT

class RollbackManager:
    """
    回滚管理器：记录执行历史，必要时执行补偿操作
    采用"后进先出"（LIFO）顺序回滚
    """

    def __init__(self):
        # 执行历史栈：每步操作的结果
        self.execution_stack: list[dict] = []

    def record(self, node_id: str, result: dict) -> None:
        """记录每一步的执行结果"""
        self.execution_stack.append({
            "node_id": node_id,
            "result": result,
            "timestamp": asyncio.get_event_loop().time()
        })

    async def rollback(self, context: PlaybookContext) -> RollbackReport:
        """
        执行回滚：从最近一步开始，逐个向前补偿

        回滚顺序（LIFO）:
        Step 3 (最后执行) → Step 2 → Step 1 (最先执行)
        """
        print(f"\n[回滚] 开始回滚，共 {len(self.execution_stack)} 步")

        report = RollbackReport()
        # 逆序遍历（后进先出）
        for step in reversed(self.execution_stack):
            node_id = step["node_id"]
            print(f"[回滚] 处理节点: {node_id}")

            # 查找该节点是否有补偿函数
            rollback_info = self._get_rollback_info(node_id)
            if not rollback_info:
                print(f"[回滚] ⚠️ 节点 {node_id} 无补偿函数，跳过")
                report.skipped.append(node_id)
                continue

            strategy = rollback_info.rollback_strategy

            if strategy == RollbackStrategy.IDEMPOTENT:
                # 幂等操作：重新执行一次即可（通常会返回"已完成"）
                try:
                    await rollback_info.execute_fn(context)
                    print(f"[回滚] ✅ {node_id} 幂等重试成功")
                    report.success.append(node_id)
                except Exception as e:
                    report.failed.append((node_id, str(e)))

            elif strategy == RollbackStrategy.COMPENSATE:
                # 执行补偿操作
                if rollback_info.compensate_fn:
                    try:
                        await rollback_info.compensate_fn(context)
                        print(f"[回滚] ✅ {node_id} 补偿操作完成")
                        report.success.append(node_id)
                    except Exception as e:
                        # 补偿也失败了 → 最坏情况，升级人工
                        print(f"[回滚] ❌ {node_id} 补偿失败: {e}")
                        report.failed.append((node_id, str(e)))
                        report.needs_human_review.append(node_id)

            elif strategy == RollbackStrategy.ESCALATE:
                print(f"[回滚] ⚠️ {node_id} 无法自动回滚，升级人工处理")
                report.escalated.append(node_id)

        return report

    def _get_rollback_info(self, node_id: str) -> NodeWithRollback | None:
        """查询节点对应的回滚策略（实际从注册表查）"""
        # 实际实现：从节点注册表查询
        rollback_registry = {
            "block_ip": NodeWithRollback(
                node_id="block_ip",
                execute_fn=lambda ctx: None,
                compensate_fn=lambda ctx: ctx.set("rollback_unblock", True),
                rollback_strategy=RollbackStrategy.COMPENSATE
            ),
        }
        return rollback_registry.get(node_id)

@dataclass
class RollbackReport:
    success: list = None
    failed: list = None
    skipped: list = None
    escalated: list = None

    def __post_init__(self):
        self.success = []
        self.failed = []
        self.skipped = []
        self.escalated = []
```

### 8.4 三层回滚保障

| 层级 | 策略 | 适用场景 | 示例 |
|------|------|---------|------|
| **L1：幂等自动** | 重复执行 | 支持幂等的 API | 封禁已封禁的 IP |
| **L2：补偿事务** | 逆操作 | 有明确逆操作 | 封禁后解封、创建后删除 |
| **L3：人工升级** | 无法自动处理 | 不可逆操作 | 杀进程（无补偿）、删数据 |

### 10.2 超时与熔断

```python
import asyncio
from dataclasses import dataclass
import time

@dataclass
class CircuitBreaker:
    """
    熔断器：防止外部依赖故障拖垮整个系统
    三态：CLOSED（正常）→ OPEN（熔断）→ HALF_OPEN（试探恢复）
    """
    name: str
    failure_threshold: int = 5     # 连续失败多少次后熔断
    recovery_timeout: int = 60    # 熔断后多久尝试恢复（秒）
    half_open_max_calls: int = 3  # 半开状态下允许多少请求

    def __post_init__(self):
        self.failures = 0
        self.state = "CLOSED"
        self.last_failure_time: float | None = None

    async def call(self, fn: callable) -> any:
        """带熔断保护的调用"""
        if self.state == "OPEN":
            # 检查是否到了恢复时间
            if time.time() - self.last_failure_time > self.recovery_timeout:
                self.state = "HALF_OPEN"
                print(f"[熔断器] {self.name}: OPEN → HALF_OPEN")
            else:
                raise CircuitOpenError(f"熔断器 {self.name} 处于 OPEN 状态")

        try:
            result = await fn()
            self._on_success()
            return result
        except Exception as e:
            self._on_failure()
            raise e

    def _on_success(self):
        self.failures = 0
        if self.state == "HALF_OPEN":
            self.state = "CLOSED"
            print(f"[熔断器] {self.name}: HALF_OPEN → CLOSED（恢复成功）")

    def _on_failure(self):
        self.failures += 1
        self.last_failure_time = time.time()
        if self.failures >= self.failure_threshold:
            self.state = "OPEN"
            print(f"[熔断器] {self.name}: CLOSED → OPEN（触发熔断）")

class CircuitOpenError(Exception):
    """熔断器打开时的异常"""
    pass
```

## 12. 面试高频问题速记

### Q1: SOAR 和 SIEM 的核心区别是什么？

> **SIEM 是眼睛（看日志、生成告警），SOAR 是手脚（执行动作、协调工具）。** SIEM 负责发现，SOAR 负责响应。两者是互补关系，不是替代关系。

### Q2: 剧本的状态机如何保证节点按正确顺序执行？

> **有向无环图（DAG）的拓扑排序。** 引擎从入口节点（trigger）开始，按拓扑顺序执行每个节点。条件节点通过评估表达式决定下一跳，审批节点暂停状态机等待外部信号恢复。

### Q3: 如果审批节点超时了怎么处理？

> **三层策略：升级（escalate）、自动拒绝（auto_reject）、自动批准（auto_approve）。** 生产环境推荐 `escalate`，即通知 SOC Lead 并延后 1 小时。`auto_approve` 风险最高，需要 CISO 书面授权。

### Q4: ResourceLock 防的是什么并发问题？

> **防的是多个剧本实例同时操作同一个资源（IP、账号等）产生的竞态条件。** asyncio.Lock 保证同进程内对同一资源的串行访问。跨进程/跨节点需要 Redis 分布式锁。

### Q5: 回滚的三层保障是什么？

> **L1 幂等（重复执行无副作用）→ L2 补偿事务（执行逆操作）→ L3 人工升级（无法自动处理）。** 好的设计尽量让操作幂等化，从源头减少回滚需求。

### Q6: YAML 和 Python Class 双轨的优势？

> **YAML 供安全分析师声明意图（说什么），Python Class 提供复杂逻辑执行（怎么做）。** YAML 解决可读性和协作问题，Python Class 解决表达力和可靠性问题。两者通过编译阶段桥接。

### Q7: SOAR 引擎的核心质量指标有哪些？

> **执行成功率 > 98%，P0 告警 MTTR < 10 分钟，回滚触发率 < 2%，人工审批响应 < 30 分钟。** 这些指标直接反映 SOC 的自动化成熟度。

### Q8: Condition 节点的表达式为什么要安全评估？

> **防止注入攻击。** 如果直接将用户输入拼接到 eval 表达式，攻击者可以通过构造特殊输入执行任意代码。必须使用 ast 解析器限制只允许比较和逻辑操作。
