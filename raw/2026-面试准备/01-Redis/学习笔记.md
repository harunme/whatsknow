## 1. Redis 基础概述

### 1.1 什么是 Redis

Redis（Remote Dictionary Server）是一个开源的、基于内存的 Key-Value 存储系统，常被用作**数据库、缓存和消息队列**。

核心特点：
- **纯内存操作**：数据存在内存中，读写速度极快（QPS 可达 10万+，具体视数据结构和操作复杂度而定）
- **单线程模型**：Redis 使用单线程（6.0 之前纯单线程，6.0+ 引入了 IO 线程），避免了锁竞争和上下文切换的开销。注意：单线程是指命令执行线程，持久化、集群通信等操作由后台子进程处理，不会阻塞主线程
- **支持多种数据结构**：String、Hash、List、Set、ZSet，还扩展了 BitMap、HyperLogLog、Geospatial、Stream 等
- **支持数据持久化**：可把内存数据落盘，防止断电丢失

**为什么 Redis 这么快？**
1. 纯内存操作，没有磁盘 I/O 瓶颈
2. 单线程避免锁竞争（非阻塞 I/O 模型 + 事件循环）
3. 使用 I/O 多路复用（epoll/select/kqueue）处理并发连接
4. C 语言实现，贴近系统层，性能好

### 1.2 为什么用 Redis

Redis 和 MySQL/PostgreSQL 等关系型数据库的核心区别：

| 对比维度 | 关系型数据库 | Redis |
|---------|------------|-------|
| 数据模型 | 行/表，有 schema | Key-Value，schema-less |
| 存储介质 | 磁盘 | 内存（可配置持久化） |
| 适用场景 | 强事务、结构化存储 | 缓存、高性能读写 |
| 查询能力 | SQL 强大 | Key-Value 为主，有限范围查询 |
| 数据量 | TB ~ PB | 通常 GB ~ TB（受内存限制）|
| 事务支持 | ACID 完整 | 只保证单命令原子性，事务用 MULTI/EXEC 批量 |

在 SOC AI 系统中，Redis 的典型应用场景：
- **Agent 状态缓存**：存储 AI Agent 的当前状态、中间结果
- **RAG 结果缓存**：缓存向量检索结果，加速对话生成
- **限流/计数**：滑动窗口限流、防爬
- **分布式锁**：任务调度时的并发控制
- **消息队列**：轻量级异步任务分发（Redis Stream）

### 1.3 NoSQL 分类

NoSQL（Not Only SQL）是对非关系型数据库的统称，按数据模型可分为四类：

| 类型 | 代表产品 | 数据模型 | 适用场景 |
|------|---------|---------|---------|
| Key-Value | Redis, Memcached | 键值对 | 缓存、会话、分布式锁 |
| 文档型 | MongoDB, Elasticsearch | JSON/XML 文档 | 内容管理、日志、搜索 |
| 列族存储 | HBase, Cassandra | 列族，宽列家族 | 大数据写入，时序数据 |
| 图数据库 | Neo4j | 图（节点+边） | 社交关系、知识图谱、推荐 |

Redis 属于 Key-Value 类型，但它的 Value 支持多种复杂数据结构，因此有时也被称为"数据结构服务器"。

## 2. 五大数据类型与代码示例

> 以下所有示例使用 `redis-cli` 命令行演示。实际项目中推荐使用 Jedis、Lettuce（Java）、redis-py（Python）等客户端。

### 2.1 String（字符串）

String 是 Redis 最基本的数据类型，Value 可以是字符串、整数或浮点数。

**核心命令**

```redis
# 设置值
SET name "Alice"
SET count 1

# 设置值 + 过期时间（原子操作，等价于 SETNX + EXPIRE，避免了非原子问题）
SET user:token:"abc123" "session_id_xxx" EX 3600 NX

# 批量设置
MSET name "Alice" age "28" city "Beijing"

# 获取值
GET name        # -> "Alice"

# 批量获取
MGET name age   # -> ["Alice", "28"]

# 自增/自减（原子操作，线程安全，适合计数器）
INCR count      # -> 2
INCRBY count 5  # -> 7
DECR count      # -> 6
INCRBYFLOAT score 1.5  # -> 8.5

# 获取并设置（原子操作，用于分布式锁实现）
GETSET name "Bob"  # -> "Alice"（返回旧值）

# 字符串追加
APPEND name "-Engineer"  # -> "Bob-Engineer"

# 获取字符串长度
STRLEN name  # -> 12
```

**String 在业务中的典型用法**

```python
# Python 代码示例（使用 redis-py）
import redis

r = redis.Redis(host='localhost', port=6379, db=0)

# 1. 分布式 session
r.setex("session:user123", 1800, "encrypted_token_xxx")
token = r.get("session:user123")

# 2. 计数器（文章阅读量、API 调用次数）
r.incr("article:1001:views")     # 阅读量 +1
views = r.get("article:1001:views")
r.expire("article:1001:views", 86400)  # 24小时后过期

# 3. 分布式 ID 生成器（简单版，高并发用 Snowflake）
r.set("id:generator", 10000)
next_id = r.incr("id:generator")  # 原子递增，线程安全

# 4. 限流（滑动窗口计数法）
def is_allowed(key, limit, window):
    """滑动窗口限流：限制 window 秒内最多 limit 次请求"""
    now = int(time.time() * 1000)  # 毫秒时间戳
    pipe = r.pipeline()
    pipe.zremrangebyscore(key, 0, now - window * 1000)  # 清理过期时间戳
    pipe.zcard(key)          # 当前窗口内请求数
    pipe.zadd(key, {str(now): now})  # 记录本次请求
    pipe.expire(key, window)  # 过期时间
    _, count, _, _ = pipe.execute()
    return count < limit

# 输出示例
# is_allowed("rate:user123", limit=10, window=60)
# 第一次调用：True，当前窗口1个请求
# 第11次调用（60秒内）：False，触发限流
```

**面试追问：String 的最大值是多少？**
Redis String 的 Value 最大为 **512MB**。实际生产中如果需要存储大文件（如图片 base64），建议存对象存储（OSS/S3），Redis 只存 URL 引用。

### 2.2 Hash（哈希）

Hash 是一个键值对集合，类似于 Python 的 dict 或 MySQL 的一行记录。适合存储对象。

```redis
# 设置 hash 中的字段
HSET user:1001 name "Alice" age "28" email "alice@example.com"

# 批量设置
HSET user:1002 name "Bob" age "30"

# 获取单个字段
HGET user:1001 name         # -> "Alice"

# 获取所有字段和值
HGETALL user:1001
# -> ["name", "Alice", "age", "28", "email", "alice@example.com"]
# ⚠️ 注意：HGETALL 在 bigkey 场景下会阻塞，慎用！

# 批量获取字段
HMGET user:1001 name email age
# -> ["Alice", "alice@example.com", "28"]

# 判断字段是否存在
HEXISTS user:1001 name   # -> 1（存在）

# 删除字段
HDEL user:1001 email     # 返回删除的字段数量

# 获取所有字段名
HKEYS user:1001
# -> ["name", "age"]

# 获取所有值
HVALS user:1001
# -> ["Alice", "28"]

# 获取字段数量
HLEN user:1001
# -> 2

# 字段值自增（必须是数值型）
HSET user:1001 login_count "0"
HINCRBY user:1001 login_count 1   # -> 1
HINCRBY user:1001 login_count 5   # -> 6
```

**Hash 在业务中的典型用法**

```python
# Python 示例：存储用户信息（替代序列化字符串）
r.hset("user:1001", mapping={
    "name": "Alice",
    "age": "28",
    "role": "admin",
    "last_login": "2026-04-10 10:00:00"
})

# 更新单个字段（不影响其他字段，这是 Hash 相比 String 的优势）
r.hset("user:1001", "last_login", "2026-04-10 12:00:00")

# 获取部分字段（减少网络传输）
info = r.hmget("user:1001", ["name", "role", "last_login"])

# HINCRBY 实现分项统计（如用户画像计数）
r.hincrby("stats:user:1001", "page_views", 1)
r.hincrby("stats:user:1001", "searches", 1)
```

**String vs Hash：怎么选？**

| 维度 | String (JSON) | Hash |
|------|-------------|------|
| 更新粒度 | 整体更新 | 单字段独立更新 |
| 内存效率 | 序列化后存储 | 字段级管理，稍多开销 |
| 适用场景 | 整个对象不常更新 | 对象字段频繁单独更新 |
| 序列化开销 | 需要 JSON 编解码 | 无需序列化，直接操作字段 |

**举例说明差异**：
```python
# 方案1：String 存 JSON
user_json = json.dumps({"name": "Alice", "age": 28, "role": "admin"})
r.set("user:1001", user_json)
# 缺点：每次更新 role 要：get -> json.loads -> 修改 -> json.dumps -> set

# 方案2：Hash 存字段
r.hset("user:1001", mapping={"name": "Alice", "age": "28", "role": "admin"})
r.hset("user:1001", "role", "super_admin")  # 直接更新一个字段
```

### 2.3 List（列表）

List 是一个有序的字符串列表，按插入顺序排序，支持两端操作（LPUSH/RPUSH、LPOP/RPOP）。

```redis
# 从左/右插入
LPUSH tasks "task_A"           # -> 1，当前列表长度
LPUSH tasks "task_B" "task_C"  # -> 3

RPUSH logs "log_20260410"      # -> 4

# 查看列表（LRANGE 按索引范围，闭区间）
LRANGE tasks 0 -1               # -> ["task_C", "task_B", "task_A"]
# 索引说明：0 是第一个，-1 是最后一个，-2 是倒数第二个

# 按索引获取
LINDEX tasks 0                 # -> "task_C"
LINDEX tasks -1                # -> "task_A"

# 获取列表长度
LLEN tasks                     # -> 3

# 弹出元素（从左/右弹，自动从列表中删除）
LPOP tasks                     # -> "task_C"，列表变为 ["task_B", "task_A"]
RPOP tasks                     # -> "task_A"，列表变为 ["task_B"]

# 阻塞弹出（BRPOP/BLPOP，当列表为空时阻塞等待，超时返回 nil）
# BRPOP tasks 10   # 阻塞最多10秒，timeout=0 表示无限等待

# 范围裁剪（保留指定区间）
LPUSH numbers 1
LPUSH numbers 2
LPUSH numbers 3
LTRIM numbers 0 1   # 只保留前2个：[3, 2]

# 列表间移动（RPOPLPUSH source dest，BRPOPLPUSH 是阻塞版）
LPUSH queue:pending "job1"
LPUSH queue:pending "job2"
RPOPLPUSH queue:pending queue:processed  # -> "job1"，job1 从 pending 移到 processed

# 在某个元素前后插入
LINSERT tasks BEFORE "task_A" "task_X"   # 在 task_A 前插入
LINSERT tasks AFTER "task_A" "task_Y"   # 在 task_A 后插入

# 修改指定索引的值
LSET tasks 0 "new_first_task"
```

**List 在业务中的典型用法**

```python
# 1. 异步任务队列（生产端 LPUSH，消费端 BRPOP）
# 生产者
r.lpush("queue:ai:tasks", json.dumps({
    "task_id": "task_001",
    "agent_id": "agent_abc",
    "prompt": "分析SOC告警",
    "priority": 1
}))

# 消费者（阻塞消费，保证消息不丢）
while True:
    _, data = r.brpop("queue:ai:tasks", timeout=30)
    if data:
        task = json.loads(data)
        result = process_task(task)
        r.lpush("queue:ai:results", json.dumps(result))
        print(f"Processed: {task['task_id']}")

# 2. 最新N条消息（如朋友圈最新10条动态）
def post_message(user_id, content):
    msg_id = r.incr("msg:id:counter")
    msg_key = f"feed:{user_id}"
    r.lpush(msg_key, json.dumps({"id": msg_id, "content": content, "ts": time.time()}))
    r.ltrim(msg_key, 0, 9)  # 只保留最新10条
    return msg_id

def get_feed(user_id, limit=10):
    messages = r.lrange(f"feed:{user_id}", 0, limit - 1)
    return [json.loads(m) for m in messages]

# 3. 访问历史（最新100条，LIFO）
r.lpush(f"history:{user_id}", page_url)
r.ltrim(f"history:{user_id}", 0, 99)
```

### 2.4 Set（集合）

Set 是无序且不重复的字符串集合，支持交、并、差集运算。

```redis
# 添加/删除
SADD tags:article:1001 "AI" "NLP" "RAG"    # -> 3（新增3个）
SADD tags:article:1001 "AI" "Security"     # -> 1（AI已存在，返回0，新加Security）

SREM tags:article:1001 "Security"          # -> 1（删除成功）

# 获取所有成员
SMEMBERS tags:article:1001   # -> ["AI", "NLP", "RAG"]
# ⚠️ 注意：SMEMBERS 在成员数量大时会阻塞，返回完整集合用 SSCAN

# 判断成员是否存在
SISMEMBER tags:article:1001 "AI"  # -> 1（存在）

# 获取集合大小
SCARD tags:article:1001    # -> 3

# 随机操作
SRANDMEMBER tags:article:1001 2   # 随机取2个，不删除：["AI", "RAG"]
SPOP tags:article:1001           # 随机弹出1个，会从集合中删除

# 集合运算
SADD set:a 1 2 3 4 5
SADD set:b 4 5 6 7 8

SINTER set:a set:b       # 交集：["4", "5"]
SUNION set:a set:b       # 并集：[1,2,3,4,5,6,7,8]
SDIFF set:a set:b        # 差集（a有b没有）：["1", "2", "3"]

# 将运算结果存到新集合
SINTERSTORE set:intersection set:a set:b  # 将交集存到 set:intersection

# 遍历（SCAN 族，不用 SMEMBERS 避免阻塞）
SSCAN set:a 0 MATCH 2 COUNT 10
```

**Set 在业务中的典型用法**

```python
# 1. 标签系统（去重 + 快速判断）
r.sadd("tags:user:1001", "AI", "Security", "SOC", "ML")
r.sadd("tags:user:1001", "AI")  # 自动去重

# 查询用户兴趣（判断是否包含某标签）
if r.sismember("tags:user:1001", "AI"):
    recommend_ai_content()

# 2. 关注/粉丝列表（用 Set 存储 mutual followers）
r.sadd("followers:Alice", "Bob", "Carol", "David")
r.sadd("followers:Bob", "Alice", "Eve")

# 共同关注（交集）
mutual = r.sinter("followers:Alice", "followers:Bob")  # {"Alice"}

# 我关注但对方没关注我（差集）
only_i_follow = r.sdiff("following:Alice", "followers:Alice")

# 3. UV 统计（独立访客去重）
r.sadd(f"uv:daily:2026-04-10", "user_abc123")
r.sadd(f"uv:daily:2026-04-10", "user_def456")
r.sadd(f"uv:daily:2026-04-10", "user_abc123")  # 重复访问，自动去重
uv_count = r.scard("uv:daily:2026-04-10")  # -> 2
```

### 2.5 ZSet（有序集合）

ZSet 是有序的且不重复的字符串集合，每个元素关联一个浮点数 score，按 score 从小到大排序。

```redis
# 添加元素（带分数）
ZADD leaderboard 100 "Alice" 90 "Bob" 85 "Carol" 100 "David"

# 获取分数
ZSCORE leaderboard "Alice"     # -> "100"

# 获取排名（0是最小/最差，ZREVRANK 是从大到小排名）
ZRANK leaderboard "Alice"      # -> 3（Alice 排第4）
ZREVRANK leaderboard "Alice"  # -> 0（从高到低排第1）

# 获取指定排名范围
ZREVRANGE leaderboard 0 9 WITHSCORES
# -> ["David", "100", "Alice", "100", "Bob", "90", "Carol", "85"]
# ⚠️ WITHSCORES 会同时返回分数

# 获取分数范围
ZRANGEBYSCORE leaderboard 80 100
ZRANGEBYSCORE leaderboard -inf +inf LIMIT 0 10  # 分页

# 增/减分数
ZINCRBY leaderboard 5 "Bob"   # Bob 的分数从 90 变成 95

# 集合大小
ZCARD leaderboard             # -> 4

# 删除成员
ZREM leaderboard "Carol"      # -> 1

# 按排名删除
ZREMRANGEBYRANK leaderboard 0 2  # 删除排名前3（分数最低的3个）
```

**ZSet 在业务中的典型用法**

```python
import time

# 1. 排行榜系统（如游戏积分、SOC 告警优先级排序）
def update_score(player_id, score_delta):
    """更新玩家分数"""
    r.zincrby("game:leaderboard", score_delta, player_id)

def get_top_n(n=10):
    """获取 Top N"""
    result = r.zrevrange("game:leaderboard", 0, n - 1, withscores=True)
    return [{"rank": i+1, "player": p, "score": s} for i, (p, s) in enumerate(result)]

def get_my_rank(player_id):
    """获取我的排名"""
    rank = r.zrevrank("game:leaderboard", player_id)
    score = r.zscore("game:leaderboard", player_id)
    return {"rank": rank + 1 if rank is not None else None, "score": score}

# 2. 时间线排序（用时间戳做分数，实现时间有序）
def add_event(entity_id, event_data):
    """给实体添加事件（按时间排序）"""
    score = time.time()  # 当前时间戳作为分数
    r.zadd(f"events:{entity_id}", {json.dumps(event_data): score})

def get_recent_events(entity_id, limit=20):
    """获取最近的事件"""
    items = r.zrevrange(f"events:{entity_id}", 0, limit - 1, withscores=True)
    return [{"data": json.loads(data), "timestamp": ts} for data, ts in items]

# 3. 延时任务队列（score = 执行时间戳，轮询 ZRANGEBYSCORE）
def add_delay_task(task_id, delay_seconds, task_data):
    """添加延时任务（delay_seconds 秒后执行）"""
    execute_at = time.time() + delay_seconds
    r.zadd("delay:tasks", {json.dumps({"id": task_id, "data": task_data}): execute_at})

def poll_ready_tasks(batch_size=10):
    """取出已到期的任务"""
    now = time.time()
    items = r.zrangebyscore("delay:tasks", 0, now, start=0, num=batch_size)
    if not items:
        return []
    # 取出后立即删除（实际生产中用 Lua 脚本原子删除）
    for item in items:
        r.zrem("delay:tasks", item)
    return [json.loads(i) for i in items]
```

## 3. SDS 动态字符串

### 3.1 C 字符串的问题

Redis 早期版本使用 C 原生字符串 (`char*`)，存在三个根本性问题：

**问题一：获取长度需要 O(n) 时间**
```c
// C 字符串要获取长度，必须遍历到 \0 为止
strlen("hello world");  // O(n)，每次都要从头扫一遍
```

**问题二：缓冲区溢出风险**
```c
// strcat 没有检查目标缓冲区大小，可能溢出
char buf[10] = "hello";
strcat(buf, " world");  // 崩溃！buf 只分配了10字节
```

**问题三：二进制不安全**
C 字符串以 `\0` 作为结束标志，如果 Value 中包含 `\0`（如图片/音频二进制数据），字符串会被截断。

### 3.2 SDS 的设计

SDS（Simple Dynamic String）是 Redis 自定义的字符串结构：

```c
// SDS 源码结构（Redis 3.0 版本的实现）
struct sdshdr {
    int len;      // 已使用的字节数（不含 \0）
    int free;     // 剩余可用字节数
    char buf[];   // 柔性数组，存储实际字符串
};
```

**核心优势详解：**

```
举例：SDS 存储 "hello"
┌─────────────────────────────────────────────┐
│  struct sdshdr                              │
│  len  = 5                                   │
│  free = 3          （预分配了额外空间）        │
│  buf  = ['h','e','l','l','o','\0',?, ?, ?]   │
│                     ↑                       │
│              free=3 这么大                   │
└─────────────────────────────────────────────┘

vs C 字符串: char buf[] = "hello\0";  // len=6, 无多余空间
```

### 3.3 SDS 的三大优化策略

**策略一：空间预分配（Pre-allocation）**

当 SDS 长度小于 1MB 时，每次扩展多分配 1 倍的 free 空间。当长度大于 1MB 时，每次固定多分配 1MB 的 free 空间。

```python
# 用图示说明：
# "hello" (5字节) -> 扩展到 len=5, free=5（< 1MB，翻倍）
# "hello...very_long_string_1MB+" -> 扩展时 free 固定 +1MB
# 这样减少了频繁 realloc() 带来的性能开销
```

**策略二：惰性空间释放（Lazy Free）**

当 SDS 缩短时（如 `SDELRIM`），不立即回收 free 空间，而是保留下来供将来使用：

```redis
# SDS 惰性释放演示
SET greeting "hello world"
STRLEN greeting     # -> 11
APPEND greeting ""  # append 空字符串，实际触发 SDS 的惰性释放逻辑演示
# 缩短字符串时，Redis 不立即缩小 buf，而是 free 增大
# 真正释放由后台线程完成，不阻塞主线程
```

**这样做的好处**：避免了每次缩短都调用 realloc()，减少了系统调用次数。真正释放由 Redis 后台的 `lazyfree` 线程处理。

**策略三：二进制安全**

SDS 的 `len` 字段记录了实际长度，不依赖 `\0` 判断结束，所以可以存储任意二进制数据：

```python
# SDS 二进制安全示例
binary_data = b'\x00\x01\x02\x03\x00\x04\x05'  # 包含 \x00 字节
r.set("binary:key", binary_data)
retrieved = r.get("binary:key")
assert retrieved == binary_data  # 完整保留，无截断

# 如果用 C 字符串存储，\x00 会导致截断，数据丢失
```

### 3.4 embstr vs raw

Redis 在存储短字符串时，使用一种更节省内存的编码方式：**embstr**（embedded string）。

```
普通 raw 编码（至少两次内存分配）：
┌────────────────┐     ┌─────────────────────┐
│ redisObject    │ --> │ sdshdr + "hello"     │
│ (type/encoding │     │ 单独一块内存          │
│  /ptr/lrefcount│     └─────────────────────┘
│  /lclock)      │
└────────────────┘

embstr 编码（一次内存分配，连续内存）：
┌──────────────────────────────────────────┐
│  redisObject | sdshdr | "hello"           │
│  所有数据在一块连续内存中，减少内存碎片     │
│  ≤ 44 字节时使用 embstr（Redis 3.0+）    │
└──────────────────────────────────────────┘
```

**44 字节是怎么来的？**
redisObject 固定 16 字节 + sdshdr（3.2+ 版本约 16 字节）+ 实际内容 ≤ 44 字节。连续内存访问更快（CPU Cache 友好），也减少了内存碎片。

**embstr 变成 raw 的时机**：当字符串长度超过 44 字节时，Redis 会将其迁移为 raw 编码。这个过程在 Redis 内部自动完成，用户无感知。

## 4. 底层编码详解

Redis 的每种数据类型，在底层会根据数据量大小自动选择最优的编码实现。这种设计叫做 **数据编码自适应**，是 Redis "简单但实用" 设计哲学的体现。

### 4.1 完整编码对照表

```
┌─────────────┬────────────────────────────┬──────────────────────────────────────────┐
│  数据类型   │         小数据量            │             大数据量                    │
├─────────────┼────────────────────────────┼──────────────────────────────────────────┤
│  String     │  int (整数 ≤ 2^32-1)       │  raw (动态字符串)                        │
│             │  embstr (≤ 44 字节短串)    │                                          │
├─────────────┼────────────────────────────┼──────────────────────────────────────────┤
│  Hash       │  ziplist (压缩列表)        │  hashtable (哈希表)                      │
├─────────────┼────────────────────────────┼──────────────────────────────────────────┤
│  List       │  ziplist (早期版本)        │  quicklist (3.2+ 标准实现)               │
│             │  quicklist (3.2+)           │                                          │
├─────────────┼────────────────────────────┼──────────────────────────────────────────┤
│  Set        │  intset (整数集合)         │  hashtable (哈希表，value 为 NULL)        │
├─────────────┼────────────────────────────┼──────────────────────────────────────────┤
│  ZSet       │  ziplist (压缩列表)        │  skiplist + dict（跳表+字典）             │
└─────────────┴────────────────────────────┴──────────────────────────────────────────┘
```

### 4.2 各编码原理详解

**ziplist（压缩列表）**

一种紧凑的线性数据结构，将所有元素连续存储，无指针开销，适合数据量小的场景：

```
ziplist 结构示例（存储 [1, "two", 3]）：
┌───┬─────┬─────┬──────┬─────┬──────┬─────┐
│len│entry│entry│entry │ ... │end  │total│
│(4)│  1  │  2  │  3   │     │(1B) │     │
└───┴─────┴─────┴──────┴─────┴──────┴─────┘
  ↑                                      ↑
zlbytes(ZLEND)                      zltail
（总字节数）                        （尾节点偏移）
```

ziplist 优点：内存紧凑，比 hashtable 省内存，无指针引用（CPU Cache 友好）
ziplist 缺点：插入/删除元素最坏 O(n)，因为要移动后续所有元素

**触发 ziplist → hashtable 的阈值：**
```redis
# Hash：最多512个field，且每个field/value 都小于64字节
hash-max-ziplist-entries 512
hash-max-ziplist-value 64

# ZSet：最多128个元素，且每个元素都小于64字节
zset-max-ziplist-entries 128
zset-max-ziplist-value 64

# List：3.2+ 版本直接用 quicklist，配置项是
list-max-ziplist-size -2   # 单个 ziplist 最大 8KB（-2 表示8KB）
list-compress-depth 0       # 两端不压缩的节点数
```

**intset（整数集合）**

Set 在只存整数时使用 intset，内存更紧凑：

```c
// intset 内部是有序数组（从小到大）
// 支持 int16_t / int32_t / int64_t 自动升级
// 一旦加入非整数元素，立刻升级为 hashtable

# 示例：
SADD set:ids 1 2 3 4 5    # -> intset（都是整数）
SADD set:ids "hello"     # -> 触发升级为 hashtable
```

**quicklist（快速列表）**

Redis 3.2+ 的 List 实现，是 ziplist 的双向链表——每个节点是一个 ziplist：

```
quicklist 结构：
┌──────┐    ┌──────┐    ┌──────┐    ┌──────┐
│ zl1  │<-->│ zl2  │<-->│ zl3  │<-->│ zl4  │
│ (2个)│    │ (8个)│    │ (5个)│    │ (1个)│
└──────┘    └──────┘    └──────┘    └──────┘
  ↑                                              ↑
head                                         tail
```

为什么用 quicklist 而不用纯 ziplist 或纯 linkedlist？
- 纯 ziplist：插入/删除 O(n)，不适用于长列表
- 纯 linkedlist：每个节点独立内存，指针开销大
- quicklist：两者兼顾——ziplist 紧凑存储，linkedlist 快速两端操作，综合 O(1) 插入/删除

**skiplist + dict（ZSet 大数据量编码）**

ZSet 需要同时支持：
- 按分数排序（ZADD、ZRANGEBYSCORE）→ skiplist 提供 O(log n) 有序访问
- 按 member 查分数（ZSCORE）→ dict 提供 O(1) 查找

```c
// Redis ZSet 底层结构（6.0- 版本）
typedef struct zset {
    dict *dict;         // {member: score} 的哈希表，O(1) 查分数
    zskiplist *zsl;    // 跳表，按分数排序，O(log n) 范围查询
} zset;
```

**跳表原理简述**：
```
Level 3: ──────────►────────────────────────
Level 2: ─────►──────►────────────────────────
Level 1: ─►──►──►──►──►──►──►──►──►──►──►──►
Level 0: [1]─►[3]─►[5]─►[8]─►[9]─►[12]─►[15]─►
          ↑
      跳表从 Level 0 逐层索引，
      Level 越高步长越大，搜索时从高层快速缩小范围
      插入时随机决定新节点层数（Redis 用 zsl->level 控制）
```

### 4.3 编码切换的时机

```python
# 查看 key 的编码类型
r.object("ENCODING", "hash:user:1001")
r.debug("object", "encoding", "hash:user:1001")

# 实际触发编码升级的示例
# Hash: 添加超过 512 个 field，或单个 field/value 超过 64 字节
for i in range(600):
    r.hset("hash:big", f"field{i}", "x" * 100)  # 触发升级为 hashtable

# Set: 添加超过 512 个整数，或加入非整数元素
r.sadd("set:nums", *range(1000))  # 超过512，升级为 hashtable
r.sadd("set:nums", "string")       # 非整数，直接升级为 hashtable

# ZSet: 添加超过 128 个元素，或单个元素超过 64 字节
for i in range(200):
    r.zadd("zset:big", {f"member{i}": i})  # 触发升级为 skiplist+dict
```

## 5. 持久化机制

Redis 的持久化解决了"内存数据随进程消失"的问题，有三种模式：RDB、AOF、混合持久化。

### 5.1 RDB（快照持久化）

RDB 是定时将**整个 Redis 内存数据**生成一个快照文件（`.rdb`），保存到磁盘。

**工作原理：**

```
主进程 (Redis Server)
    │
    │  执行 bgsave 命令（或达到自动触发条件）
    │  fork() ──────────────────────────────────────► 子进程（Copy-on-Write）
    │  （fork 瞬间子进程获得父进程内存的完整副本）           │
    │                                                   │ 遍历内存数据
    │  继续处理客户端请求                                │ 写入 rdb 文件
    │                                                   │
    │                                                   │ 写完后通知主进程
    ▼                                                   ▼
                                          .rdb 文件（dump.rdb）写入磁盘
```

**关键点：fork() 和 Copy-on-Write（COW）**

fork() 创建子进程时，Linux 使用 Copy-on-Write 机制：
- fork() 瞬间，子进程和父进程**共享同一份物理内存页**，不立即复制
- 只有当父进程或子进程**修改某个页**时，才真正复制那一页
- 这使得 fork() 即使在 Redis 占用几十 GB 内存时也能快速完成

**RDB 触发条件：**

```redis
# 手动触发
BGSAVE    # 后台异步保存（不阻塞主进程，推荐）
SAVE      # 同步保存（阻塞主进程，生产环境禁用！）

# 自动触发（redis.conf）
save 900 1      # 900秒内 ≥1 个 key 变化就触发 RDB
save 300 10     # 300秒内 ≥10 个 key 变化
save 60 10000   # 60秒内 ≥10000 个 key 变化

# 关闭自动 RDB
save ""          # 或注释掉所有 save 行
```

**RDB 的优点和缺点：**

| 优点 | 缺点 |
|------|------|
| 文件紧凑（压缩后几 MB~百 MB），适合备份/灾难恢复 | 两次快照之间数据丢失（最多丢一个周期） |
| 恢复速度快（直接加载 .rdb 文件） | fork() 在数据量大时耗时（几十GB可能几十秒）|
| 适合做冷备（定期上传到对象存储） | 频繁 fork() 对延迟敏感业务有影响 |

### 5.2 AOF（追加文件持久化）

AOF（Append Only File）将每个**写命令**追加到 `.aof` 文件末尾，恢复时重放所有命令。

```redis
# 开启 AOF
appendonly yes
appendfilename "appendonly.aof"

# AOF 重写（压缩 .aof 文件，去掉过期/被覆盖的命令）
BGREWRITEAOF   # 后台重写，不阻塞

# 查看 AOF 文件内容（文本格式，可读性好）
# $6
# SELECT
# $1
# 0
# $3
# SET
# $5
# hello
# $5
# world
```

**appendfsync 三种策略：**

| 策略 | 含义 | 性能 | 数据安全性 |
|------|------|------|-----------|
| `always` | 每个写命令都 `fsync()` 刷盘 | 最慢（万级 QPS）| 最高（几乎不丢数据）|
| `everysec`（推荐）| 每秒一次 `fsync()` | 较快（10万级 QPS）| 最多丢1秒数据 |
| `no` | 交给 OS 控制何时刷盘 | 最快 | 丢数据最多（OS 宕机可能丢大量）|

**为什么推荐 everysec？**
everysec 是性能和安全的最佳平衡——每秒刷盘，最多丢 1 秒数据；而 always 每个命令都刷盘，QPS 可能从 10万 降到几千，严重影响吞吐。

**AOF 重写机制：**

AOF 文件会随写命令不断增大，AOF 重写通过后台子进程扫描现有数据，重新生成一份最小命令集：

```
BGREWRITEAOF 触发后：
1. fork() 子进程
2. 子进程遍历内存数据，生成新 AOF 文件（只保留最终状态的最优命令）
3. 父进程继续接收新命令，同时将新命令也写入一个 "AOF rewrite buffer"
4. 子进程写完后，父进程把 rewrite buffer 追加到新 AOF 文件
5. 用新 AOF 文件替换旧文件，原子完成

这个过程中，fork() 瞬间不阻塞，新命令不丢失。
```

### 5.3 混合持久化（Hybrid Persistence）

Redis 4.0 引入了混合持久化，将 RDB 和 AOF 的优点结合：

```redis
# 开启混合持久化
aof-use-rdb-preamble yes
appendonly yes
```

**工作流程：**
- AOF 重写时，先写入 RDB 格式的**全量数据**
- 后续增量命令以 AOF 格式追加
- 恢复时：先加载 RDB 部分（快速全量恢复），再重放 AOF 增量命令（精确恢复）

```
混合 AOF 文件结构：
┌─────────────────────────┬──────────────────────────┐
│     RDB 格式（全量）     │    AOF 格式（增量追加）    │
│   （二进制，体积小）      │   （文本命令，更精准）      │
└─────────────────────────┴──────────────────────────┘
```

**生产环境推荐配置：**

```redis
# redis.conf 推荐配置
appendonly yes
appendfilename "appendonly.aof"
appendfsync everysec
aof-use-rdb-preamble yes      # 开启混合持久化

# AOF 文件大小控制
auto-aof-rewrite-percentage 100  # AOF 文件比上次重写后大 100% 时再触发重写
auto-aof-rewrite-min-size 64mb   # AOF 至少 64MB 才触发重写（避免频繁重写）
```

**企业实际使用情况：**
- **小规模/开发环境**：RDB 定期快照，关闭 AOF（恢复快，丢了最多几分钟数据可接受）
- **生产环境**：同时开 AOF（everysec）+ RDB（定期），配合混合持久化
- **金融/支付类**：AOF always + 同步双写 MySQL，双重保障

## 6. 缓存穿透、击穿与雪崩

这三个问题被称为"缓存三剑客"，是面试高频考点，也是生产环境中导致数据库被打垮的主要原因。

### 6.1 缓存穿透（Cache Penetration）

**问题描述：**

大量请求查询**数据库中根本不存在**的数据，每次都绕过缓存直接打到数据库。

```
正常请求流程：         穿透请求流程：
  客户端                   客户端
    │                         │
    ▼                         ▼
 Redis缓存                 Redis缓存
 (命中则返回)              (key不存在)
    │                         │
    ▼                         ▼
  返回数据               查询 MySQL
                         (key不存在)
                            │
                            ▼
                         返回空/null
                            │
                            ▼
                      大量空请求压垮DB
```

**常见场景：**
- 恶意攻击：攻击者构造大量不存在 ID 的请求（如 `user:-1`、`user:999999`）
- 业务漏洞：查询条件组合不当，返回了不存在的结果

**解决方案：**

**方案一：布隆过滤器（Bloom Filter）**

将所有合法 key 预先存入布隆过滤器，查询前先检查：

```python
# 使用 RedisBloom 或 Python 实现的布隆过滤器
from bloom_filter import BloomFilter

# 初始化（假设从数据库加载所有合法用户ID）
bf = BloomFilter(max_elements=10000000, error_rate=0.01)
for user_id in db.query("SELECT id FROM users"):
    bf.add(f"user:{user_id}")

def get_user(user_id):
    key = f"user:{user_id}"
    # 第一步：布隆过滤器检查
    if key not in bf:
        return None  # 一定不存在，直接返回（不查 DB）
    # 第二步：查缓存
    data = r.get(key)
    if data:
        return json.loads(data)
    # 第三步：缓存未命中，查 DB
    user = db.query("SELECT * FROM users WHERE id = ?", user_id)
    if user:
        r.setex(key, 3600, json.dumps(user))
    return user
```

布隆过滤器特点：
- **误报率（False Positive）**：可能把不存在的 key 判定为存在，但不会漏报（存在的一定会被判定为存在）
- 内存占用极小（1亿个 key 只需 ~1.2GB，误差率 1%）
- **不支持删除**（需要支持删除用 Count-Min Sketch 或 RedisBloom 的 CBF 模块）

**方案二：空值缓存（Null Value Cache）**

DB 查不到时，在 Redis 中缓存一个空值标记，TTL 短一些（如 3~5 分钟）：

```python
def get_user(user_id):
    key = f"user:{user_id}"
    # 1. 查缓存
    data = r.get(key)
    if data == "__NULL__":
        return None  # 已知不存在的 key
    if data:
        return json.loads(data)
    # 2. 查 DB
    user = db.query("SELECT * FROM users WHERE id = ?", user_id)
    if user:
        r.setex(key, 3600, json.dumps(user))  # 正常数据，TTL 正常
    else:
        r.setex(key, 300, "__NULL__")  # 空值缓存，TTL 短（3分钟）
    return user
```

**⚠️ 注意：TTL 不能太长**，否则当用户注册后，这个空值缓存还没过期，合法用户反而查不到。

**方案三：接口限流**

用 Redis 实现滑动窗口限流，防止大量穿透请求涌入：

```python
import time, redis

r = redis.Redis()

def sliding_window_rate_limit(key, limit=100, window=60):
    """滑动窗口计数器：60秒内最多100次请求"""
    now_ms = int(time.time() * 1000)
    window_ms = window * 1000

    pipe = r.pipeline()
    # 1. 清理窗口外的旧记录
    pipe.zremrangebyscore(key, 0, now_ms - window_ms)
    # 2. 计数
    pipe.zcard(key)
    # 3. 加入本次请求
    pipe.zadd(key, {str(now_ms): now_ms})
    # 4. 设置过期
    pipe.expire(key, window + 1)
    _, count, _, _ = pipe.execute()

    return count < limit

# 生产使用示例
def api_handler(request):
    if not sliding_window_rate_limit(f"limit:ip:{request.ip}", limit=100, window=60):
        return 429, "Too Many Requests"
    return get_user(request.user_id)
```

### 6.2 缓存击穿（Cache Breakdown）

**问题描述：**

某个**热点 key 过期**的瞬间，大量并发请求同时穿透到数据库，导致数据库瞬时压力过大。

```
正常情况：              击穿情况：
 T1: key存在 → 直接返回    T1: key过期
 T2: key存在 → 直接返回    T2: 100个并发请求同时发现key不存在
                         T3: 100个请求全部去查DB（DB被打垮）
```

**解决方案：**

**方案一：分布式锁（Mutual Exclusion）**

只让一个请求去加载数据，其他请求等待或返回降级数据：

```python
import redis, time, json

def get_with_lock(key, db_loader_fn, ttl=3600, lock_ttl=5):
    """分布式锁 + 缓存：解决击穿问题"""
    lock_key = f"lock:{key}"
    result = r.get(key)  # 先查缓存
    if result:
        if result == "__NULL__":
            return None
        return json.loads(result)

    # 缓存未命中，尝试获取分布式锁
    lock_val = str(time.time())
    is_locked = r.set(lock_key, lock_val, nx=True, ex=lock_ttl)

    if is_locked:
        # 获取到锁，去 DB 加载数据
        try:
            data = db_loader_fn()
            if data:
                r.setex(key, ttl, json.dumps(data))
            else:
                r.setex(key, 300, "__NULL__")  # 空值缓存
            return data
        finally:
            # ⚠️ 释放锁（用 Lua 脚本保证原子性）
            r.eval(
                "if redis.call('get', KEYS[1]) == ARGV[1] then "
                "return redis.call('del', KEYS[1]) else return 0 end",
                1, lock_key, lock_val
            )
    else:
        # 没获取到锁，短暂等待后重试查缓存（最多等 lock_ttl 秒）
        for _ in range(lock_ttl * 10):
            time.sleep(0.1)
            result = r.get(key)
            if result:
                return json.loads(result) if result != "__NULL__" else None
        # 等待超时，返回 None 或降级数据
        return None

# 使用示例
def get_user(user_id):
    key = f"user:{user_id}"
    return get_with_lock(key, lambda: db.query_user(user_id))
```

**方案二：逻辑过期（永不过期 + 后台重建）**

不设置真正的过期时间，而是用额外字段存储"逻辑过期时间"，过期后用后台线程重建缓存：

```python
def get_with_logic_expiry(key, db_loader_fn, logic_ttl=3600):
    """逻辑过期策略：key 永不过期，后台线程重建"""
    data = r.get(key)
    if not data:
        # 缓存为空，直接加载（首次预热时用）
        d = db_loader_fn()
        r.set(key, json.dumps({"value": d, "expire_at": time.time() + logic_ttl}))
        return d

    entry = json.loads(data)
    # 检查逻辑过期时间
    if time.time() > entry["expire_at"]:
        # 逻辑已过期，尝试异步重建
        lock_key = f"lock:{key}"
        if r.set(lock_key, "1", nx=True, ex=5):
            # 发起异步重建（生产中用后台线程或异步任务）
            rebuild_async(key, db_loader_fn, logic_ttl)
        # 返回旧数据（不过期，保证可用性）
        return entry["value"]
    return entry["value"]

def rebuild_async(key, loader_fn, ttl):
    """异步重建缓存（生产用 Celery/后台线程）"""
    try:
        data = loader_fn()
        r.set(key, json.dumps({"value": data, "expire_at": time.time() + ttl}))
    except Exception as e:
        print(f"Rebuild failed: {e}")
```

**击穿 vs 雪崩的本质区别：**
- **击穿**：一个热点 key 过期 → 问题集中在单个 key
- **雪崩**：大量 key 同时过期 → 问题是群体性的

### 6.3 缓存雪崩（Cache Avalanche）

**问题描述：**

大量普通 key 在**同一时间点集中过期**，导致大量请求同时穿透到数据库。

```
时间 ──────────────────────────────────────────────────►
  │
  │ ★★★ 一大批 key 在同一时刻过期 ★★★
  │        │        │        │        │
  │        ▼        ▼        ▼        ▼
  │      DB      DB      DB      DB    （数据库被打垮）
  │  ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━►  雪崩！
```

**三大解决方案：**

**方案一：过期时间随机化（最简单有效）**

```python
# 给 TTL 加一个随机偏移量，避免同时过期
BASE_TTL = 3600
RANDOM_OFFSET = random.randint(0, 300)  # 0~300秒随机偏移
actual_ttl = BASE_TTL + RANDOM_OFFSET
r.setex(f"user:{user_id}", actual_ttl, data)
```

**方案二：Redis 高可用部署**

Sentinel 或 Cluster 模式，确保部分节点故障时服务仍可用，不至于全部缓存失效。

**方案三：多级缓存**

```
请求 → 本地缓存(Caffine/Guava) → Redis → DB
              │                      │
         极快(~μs)              较快(~ms)
         容量小                 容量大
```

```python
import hashlib

local_cache = {}  # 生产用 Caffine Cache

def get_multi_level(key):
    """两级缓存：本地内存 + Redis"""
    # 1. 本地缓存（优先，极快）
    if key in local_cache:
        return local_cache[key]

    # 2. Redis 缓存
    data = r.get(key)
    if data:
        local_cache[key] = data  # 回填本地缓存
        return data

    # 3. DB
    data = db_query(key)
    r.setex(key, 3600, data)
    local_cache[key] = data
    return data
```

## 7. Redis Stream 消息队列

Redis Stream 是 Redis 5.0 引入的**持久化消息队列**数据结构，弥补了 Pub/Sub 的不足。

### 7.1 为什么需要 Stream

Pub/Sub 的局限性：
- **无状态**：发布后无记录，订阅者掉线消息就丢失
- **不持久化**：消息在推送后就丢弃了
- **无消费者组**：无法将消息分发给不同消费者而不重复

Stream 的设计目标：提供一个有 ID、可持久化、支持消费者组的消息队列。

### 7.2 Stream 核心命令

**添加消息 XADD：**

```redis
# XADD stream_name ID field value [field value ...]
# ID 为 * 表示让 Redis 自动生成（推荐）
XADD mystream * event "user_login" user_id 1001
# -> "1700000000000-0"（自动生成的 ID：时间戳-序列号）
XADD mystream * event "user_logout" user_id 1001
# -> "1700000000001-0"

# ID 可以手动指定（必须是递增的）
XADD mystream "1700000000002-0" event "user_action" action "click"
```

**读取消息 XREAD：**

```redis
# 从头读所有消息（不阻塞）
XREAD STREAMS mystream 0
# -> 1) 1) "mystream"
#       2) 1) 1) "1700000000000-0"
#             2) 1) "event"
#                2) "user_login"
#                3) "user_id"
#                4) "1001"

# 从最新消息开始读（$ = 截止到写这条命令时的最新消息）
XREAD STREAMS mystream $
# -> (nil) （当前无新消息）

# 阻塞读（等待新消息，最多30秒）
XREAD BLOCK 30000 STREAMS mystream $
# -> 超时返回 nil，有新消息则立即返回
```

**XREADGROUP（消费者组模式）：**

```redis
# 1. 创建消费者组（MKSTREAM 自动创建 stream）
XGROUP CREATE mystream mygroup $ MKSTREAM

# 2. 消费者 c1 读取新消息（> = 只取新消息）
XREADGROUP GROUP mygroup c1 STREAMS mystream ">"
# -> 1) 1) "mystream"
#       2) 1) 1) "1700000000000-0"
#             2) 1) "event"
#                2) "user_login"
#                3) "user_id"
#                4) "1001"

# 3. 消费者 c2 同时读（同一消息只会分给一个消费者）
XREADGROUP GROUP mygroup c2 STREAMS mystream ">"
# -> (nil) （消息已被 c1 消费，c2 拿不到这条了）
```

**确认处理 XACK：**

```redis
# 消费者处理完成后确认
XACK mystream mygroup "1700000000000-0"
# -> 1（确认成功）

# 查看待确认消息（消费者掉线后可重新分发）
XPENDING mystream mygroup
# -> 1) (integer) 0   （0个待确认消息）
#    2) 1) 1) "1700000000001-0"
#          2) "c1"
#          3) (integer) 0
#          4) 1) "delivery_id"
#             2) "1700000000001-0"
```

**消息转移 XCLAIM：**

当某个消费者掉线、消息未处理时，其他消费者可以用 XCLAIM 接管：

```redis
# 将超时未ACK的消息转移给其他消费者
# XCLAIM stream group consumer min-idle-time msg-id [msg-id ...]
XCLAIM mystream mygroup c3 60000 "1700000000001-0"
# -> 把超过60秒未被ACK的消息 "1700000000001-0" 转移给 c3
# min-idle-time=60000 防止消息刚被分配就被抢走
```

### 7.3 Stream vs Pub/Sub vs Kafka

| 特性 | Pub/Sub | Stream | Kafka |
|------|---------|--------|-------|
| 消息持久化 | 无 | 有（RDB/AOF）| 有 |
| 消息 ACK | 无 | 支持 | 支持 |
| 消费者组 | 无 | 支持 | 支持 |
| 消息持久化 | 即发即弃 | 持久化到 RDB/AOF | 持久化到磁盘 |
| 消息回溯 | 不支持 | 支持（读历史消息）| 支持（从 offset 读）|
| 广播能力 | 强（一对多）| 一般（组内竞争）| 强（分区+消费者组）|
| 运维复杂度 | 低 | 低（嵌入 Redis）| 高（独立集群）|
| 适用规模 | 广播/实时 | 中小规模 | 大规模数据流 |

### 7.4 Python 生产级示例

```python
import redis, json, time, threading
from redis import Redis

r = Redis(decode_responses=True)

# ===== 生产者 =====
def submit_task(task_type, payload):
    """提交任务到 Stream"""
    msg_id = r.xadd("stream:tasks", {
        "type": task_type,
        "payload": json.dumps(payload),
        "submitted_at": time.time()
    })
    return msg_id

# ===== 消费者（后台线程）=====
def start_consumer(consumer_name):
    """后台消费者线程，持续消费任务"""
    group = "ai-task-consumers"
    stream = "stream:tasks"

    # 确保消费者组存在
    try:
        r.xgroup_create(stream, group, id="0", mkstream=True)
    except redis.exceptions.ResponseError:
        pass  # 组已存在

    while True:
        try:
            # 阻塞读取，60秒超时
            messages = r.xreadgroup(group, consumer_name, {stream: ">"}, count=10, block=60000)
            if not messages:
                continue

            for stream_name, entries in messages:
                for msg_id, fields in entries:
                    try:
                        task_type = fields["type"]
                        payload = json.loads(fields["payload"])
                        print(f"[{consumer_name}] Processing {task_type}: {payload}")

                        # 执行业务逻辑
                        result = process_task(task_type, payload)

                        # 确认处理成功
                        r.xack(stream, group, msg_id)
                        print(f"[{consumer_name}] Done: {msg_id}")
                    except Exception as e:
                        print(f"[{consumer_name}] Error: {e}, XCLAIM to retry...")
                        # 转移给其他消费者重试
                        r.xclaim(stream, group, consumer_name, 30000, msg_id)
    except Exception as e:
        print(f"Consumer {consumer_name} died: {e}")
        time.sleep(5)  # 失败后等5秒重启

def process_task(task_type, payload):
    """实际业务处理"""
    if task_type == "analyze_alert":
        return {"status": "analyzed", "alert_id": payload["alert_id"]}
    elif task_type == "enrich_context":
        return {"status": "enriched", "context_id": payload["context_id"]}
    return {"status": "unknown"}

# 启动多个消费者
for i in range(3):
    t = threading.Thread(target=start_consumer, args=(f"worker-{i}",), daemon=True)
    t.start()

# 主线程提交任务
for i in range(100):
    submit_task("analyze_alert", {"alert_id": f"alert_{i}", "severity": "high"})
    time.sleep(0.1)
```

## 8. 集群方案

### 8.1 三种方案对比

| 维度 | 主从复制 | Sentinel | Cluster |
|------|---------|---------|---------|
| 解决的问题 | 读写分离、读性能 | 自动故障转移 | 数据分片 + 高可用 |
| 数据复制 | 异步 | 异步 | 每个分片内异步 |
| 写能力 | 单点（主） | 单点（主） | 分片（多主并发写）|
| 数据量 | 受单机内存限制 | 受单机内存限制 | 横向扩展（多节点）|
| 自动故障恢复 | 无 | 有 | 有（每分片独立）|
| 客户端路由 | 无需修改 | 无需修改 | 需要支持 MOVED 重定向 |
| 适用场景 | 读多写少，数据量小 | 高可用，数据量小 | 数据量大，需水平扩容 |

### 8.2 主从复制（Master-Replica）

**原理：**

```
写请求 ──► 主节点（Master）
            │  SYNC/PSYNC
            ▼
    ┌───────────────┐
    │  从节点(Replica)│  (只读)
    │  从节点(Replica)│  (只读)
    │  从节点(Replica)│  (只读)
    └───────────────┘
```

从节点启动时，向主节点发送 `SYNC`（Redis 2.8+ 用 `PSYNC`），主节点 fork 子进程将内存数据快照发给从节点（**全量复制**），之后通过 **命令传播**（主节点每个写命令发给从节点）保持同步。

**Redis 2.8+ PSYNC 断点续传：**

之前用 `SYNC`，从节点断线重连要重新全量复制（RDB 文件可能几个 GB），代价极大。`PSYNC` 支持**增量复制**：

```
PSYNC 流程：
1. 从节点断线重连
2. 发送 PSYNC <runid> <offset>（主节点的 replication ID + 已接收偏移量）
3. 主节点判断 runid 是否匹配，offset 是否还在环形缓冲区（backlog）中
4. 匹配成功 → 主节点只发送增量命令（几十字节）
5. 匹配失败 → 回到全量复制

# 主节点 replication backlog 配置（环形缓冲区大小）
repl-backlog-size 10mb
repl-backlog-ttl 3600  # 从节点断线超过1小时，backlog 释放
```

### 8.3 Sentinel（哨兵）

Sentinel 是 Redis 的**高可用方案**，由一组（建议 3 或 5 个）Sentinel 进程组成，自动监控主从健康状态，完成故障转移：

```
                    ┌─────────────┐
                    │  Sentinel 1  │
                    │  (Leader)    │
                    └──────┬──────┘
                           │
          ┌────────────────┼────────────────┐
          │                │                │
    ┌─────▼─────┐   ┌──────▼──────┐  ┌──────▼──────┐
    │ Sentinel 2│   │ Sentinel 3  │  │ Sentinel N   │
    └─────┬─────┘   └──────┬──────┘  └──────┬──────┘
          │                │                │
    ┌─────▼────────────────▼────────────────▼─────┐
    │           Redis 主从集群                      │
    │   [Master] ← Replica ← Replica              │
    └─────────────────────────────────────────────┘
```

**故障转移流程：**
1. Sentinel 监控主节点心跳，超过 `down-after-milliseconds`（默认 30s）判定为 **主观下线（SDown）**
2. 超过 `quorum`（法定人数）个 Sentinel 都认为主节点下线 → **客观下线（ODown）**
3. 所有 Sentinel 投票选出一个 **Leader**（基于 `raft` 风格的投票）
4. Leader 从从节点中选出一个新主节点（优先级：`replica-priority`，或数据最新者）
5. 其他从节点向新主节点发起复制
6. 客户端配置更新（新主节点地址由 Sentinel 提供）

**客户端感知新主节点：**

```python
# Jedis 连接 Sentinel（自动获取新主节点地址）
from redis.sentinel import Sentinel

sentinel = Sentinel([
    ('sentinel-1', 26379),
    ('sentinel-2', 26379),
    ('sentinel-3', 26379)
], socket_timeout=0.1)

# 获取主节点连接（自动故障转移后自动拿到新主节点）
master = sentinel.master_for('mymaster', decode_responses=True)
slave = sentinel.slave_for('mymaster', decode_responses=True)

# 之后所有操作对客户端透明，故障转移时 Jedis 内部重连
master.set("key", "value")
```

### 8.4 Cluster（集群）

Redis Cluster 通过**哈希槽（Hash Slot）** 将数据分布在 16384 个槽上，每个节点负责其中一部分槽。

**数据如何路由到具体节点：**

```
客户端（任何节点）
     │
     │ 计算 key 的槽号：crc16(key) % 16384
     │
     ▼
  槽 0  ───────────► 节点 A（master:7000）
  槽 1  ───────────► 节点 A（master:7000）
  ...
  槽 5460 ─────────► 节点 A（master:7000）
  槽 5461 ─────────► 节点 B（master:7001）
  ...
  槽 10922 ────────► 节点 B（master:7001）
  槽 10923 ────────► 节点 C（master:7002）
  ...
  槽 16383 ────────► 节点 C（master:7002）
```

```python
# Python 客户端连接 Cluster（Redis-py cluster）
from redis.cluster import RedisCluster

rc = RedisCluster(
    host="127.0.0.1",
    port=7000,
    decode_responses=True
)
# Redis-py cluster 自动计算槽号并路由到正确节点
rc.set("name", "Alice")
rc.get("name")

# 如果请求的 key 在另一个节点：
# Redis Cluster 返回 MOVED 错误，客户端自动重定向
# MOVED 5461 127.0.0.1:7001
# 客户端自动更新槽映射，下次直接发到正确节点
```

**为什么是 16384 个槽？**

这个问题几乎是 Redis Cluster 面试必问，核心在于**心跳包带宽优化**：

```
Gossip 协议心跳包中，每个节点需要广播自己负责哪些槽：

16384 个槽用 bit 表示：16384 / 8 = 2048 字节 = 2KB
65,536 个槽用 bit 表示：65536 / 8 = 8192 字节 = 8KB

100 个节点的集群中：
- 16384 槽：每个心跳包 2KB × 100 = 200KB/节点
- 65536 槽：每个心跳包 8KB × 100 = 800KB/节点（4倍带宽）

同时：
- 16384 足够精细（10,000 台机器 × 1.6 槽 = 精细分配）
- 槽迁移以槽为单位，16384 的粒度足够细
- 集群规模通常远小于 1000 节点，16384 足够用
```

**槽迁移（Resharding）：**

当集群需要扩缩容时，槽会在线迁移：
```
迁移中状态：
  槽 5000 从节点A → 节点B

节点A（迁移源）：
  客户端请求 key 在槽5000内
    ├── key 还在 A → 正常处理
    └── key 已迁移到 B → 返回 ASK 错误（客户端向 B 重发请求，但不更新本地槽映射）
```

⚠️ **ASK vs MOVED 的区别**：
- `MOVED`：槽永久迁移，客户端**更新本地槽映射表**
- `ASK`：槽临时迁移（迁移进行中），客户端**临时重定向，但不更新映射**

## 9. 分布式锁

### 9.1 为什么需要分布式锁

在单机多线程环境下，用语言自带的 Mutex（Java synchronized、Python threading.Lock）即可。但在分布式多进程环境下，不同机器上的进程看不到彼此的锁，需要一个**跨机器的协调机制**，这就是分布式锁。

典型应用场景：
- 任务调度：保证同一任务同一时刻只有一个节点执行
- 库存扣减：防止超卖
- 幂等控制：保证同一请求只处理一次

### 9.2 Redis 分布式锁实现

**第一步：原子加锁**

```redis
# 错误做法：SETNX + EXPIRE 是两步，不是原子操作
SETNX lock:order:12345 "worker_A"   # 加锁
EXPIRE lock:order:12345 30            # 设置过期
# 如果进程在 SETNX 成功后、EXPIRE 前崩溃，锁永远不会过期！

# 正确做法：SET key val NX EX ttl（Redis 2.6.12+）
SET lock:order:12345 "worker_A" NX EX 30
# -> OK（加锁成功）
# -> nil（锁已被他人持有）
```

**第二步：原子释放锁**

释放锁时必须验证"这把锁是我的才能删"，否则可能误删他人持有的锁：

```lua
-- Lua 脚本：只删除自己持有的锁
-- KEYS[1] = 锁的 key
-- ARGV[1] = 锁的 value（持有者的唯一标识）

if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("del", KEYS[1])
else
    return 0
end
```

**为什么不用 DEL key 直接删？**

```
危险场景：
1. 客户端A获取锁，TTL=10s
2. 客户端A处理时间超过10s，锁自动过期
3. 客户端B获取同一把锁（因为A的锁已过期）
4. 客户端A处理完，执行 DEL lock（删掉了B的锁！）
```

**完整 Python 实现：**

```python
import redis, time, uuid, threading

class RedisLock:
    """Redis 分布式锁（可重入版本）"""

    def __init__(self, redis_client, key, ttl=30, retry_times=3, retry_delay=0.2):
        self.r = redis_client
        self.key = f"lock:{key}"
        self.ttl = ttl
        self.retry_times = retry_times
        self.retry_delay = retry_delay
        self.token = str(uuid.uuid4())  # 每个客户端唯一标识

    def acquire(self):
        """尝试获取锁"""
        for _ in range(self.retry_times):
            if self.r.set(self.key, self.token, nx=True, ex=self.ttl):
                return True
            time.sleep(self.retry_delay)
        return False

    def release(self):
        """释放锁（Lua 脚本原子操作）"""
        script = """
        if redis.call("get", KEYS[1]) == ARGV[1] then
            return redis.call("del", KEYS[1])
        else
            return 0
        end
        """
        self.r.eval(script, 1, self.key, self.token)

    def __enter__(self):
        if not self.acquire():
            raise RuntimeError(f"Failed to acquire lock: {self.key}")
        return self

    def __exit__(self, exc_type, exc_val, exc_tb):
        self.release()
        return False  # 不吞掉异常

# 使用示例
lock = RedisLock(r, "order:process:12345", ttl=30)
try:
    with lock:
        # 业务逻辑：检查库存、扣减、下单
        process_order(12345)
except RuntimeError as e:
    print(f"获取锁失败，跳过或重试: {e}")
```

### 9.3 RedLock 争议

RedLock 由 Redis 作者 Antirez 提出，思路是向 **5 个独立 Redis 节点** 同时加锁，N/2+1 成功才算成功：

```
RedLock 步骤：
1. 获取当前时间 T1（毫秒）
2. 依次向 5 个节点发送 SET lock_key token NX EX 30
3. 记录每个节点响应时间
4. 获取当前时间 T2
5. 计算成功节点数和总耗时 (T2 - T1)
6. 如果成功节点 ≥ 3，且总耗时 < TTL/2，则加锁成功
7. 否则，向所有节点发送 DEL（释放锁）

为什么 TTL/2？
- 防止时钟漂移：如果获取锁耗时过长，等锁真正生效时已经快过期了
- 如果某个节点响应慢，即使拿到了也忽略它
```

**RedLock 的争议（Martin Kleppmann 提出）：**

1. **时钟漂移问题**：RedLock 依赖多个节点的时钟一致，但在云环境中不同机器时钟可能有偏差，安全性难以证明
2. **单节点故障不等于锁丢失**：RedLock 认为单节点故障会导致锁丢失，但实际上如果主节点有从节点（Sentinel/Cluster），故障转移后锁仍有效
3. **性能 vs 安全权衡**：RedLock 需要 5 个独立节点，运维复杂度高，在多数场景下单节点锁 + 合理 TTL 已足够

**⚠️ 面试结论**：生产中 RedLock 用得不多。单节点 Redis 锁 + 合理 TTL + 可重试机制，对大多数业务足够。如果对一致性要求极高（如金融交易），考虑 etcd/Consul 的分布式锁。

## 10. 缓存与数据库一致性

### 10.1 四种缓存更新模式

**Cache Aside（旁路缓存，最推荐）：**

```
读操作：
  缓存命中 ──► 直接返回
  缓存未命中 ──► 查数据库 ──► 写缓存 ──► 返回

写操作：
  更新数据库 ──► 删除缓存（不是更新缓存）
```

```python
def read_user(user_id):
    key = f"user:{user_id}"
    data = r.get(key)
    if data:
        return json.loads(data)
    user = db.query("SELECT * FROM users WHERE id = ?", user_id)
    if user:
        r.setex(key, 3600, json.dumps(user))
    return user

def update_user(user_id, name):
    # 写：先更新 DB，再删缓存
    db.execute("UPDATE users SET name = ? WHERE id = ?", name, user_id)
    r.delete(f"user:{user_id}")  # 删除缓存，让下次读触发加载新数据
```

**为什么"删缓存"而不是"更新缓存"？**
并发场景下，如果先删缓存再更新 DB（或反过来），可能导致：
```
线程A（写）：删缓存
线程B（读）：缓存未命中，查 DB（旧值），写回缓存
线程A（写）：更新 DB（新值）
→ 缓存里是旧值，数据库是新值，不一致！
```
用"删缓存"而非"更新缓存"，是因为缓存不一致后，删掉它会自然在下一次读取时从 DB 恢复正确数据，而更新缓存则需要精确处理并发逻辑，更复杂。

**Write Behind（写穿透）：**
先写缓存，异步批量写 DB。性能最高，但风险也最大（缓存故障会丢数据）。

**Read/Write Through：** 缓存层代理所有读写操作，对应用层透明。实现复杂，一般不选用。

### 10.2 延迟双删（Delayed Double Delete）

在并发场景下，先删缓存再更新 DB 可能导致短暂不一致，延迟双删来解决：

```python
def write_with_delay_double_delete(user_id, name):
    """
    延迟双删：解决"先删缓存再更新DB"时的并发竞态
    """
    # 1. 先删缓存
    r.delete(f"user:{user_id}")

    # 2. 更新数据库（此期间读线程可能已将旧值写回缓存）
    db.execute("UPDATE users SET name = ? WHERE id = ?", name, user_id)

    # 3. 延迟一段时间后再删一次（让读线程的写缓存操作完成）
    time.sleep(0.5)  # 延迟 500ms
    r.delete(f"user:{user_id}")
```

**延迟时间怎么确定？**
没有标准答案。通常设为业务中读线程从"发现缓存未命中"到"写回缓存"的最大耗时。可以预估为 **100~500ms**，或者通过压测确定 P99 时间。

### 10.3 一致性方案对比

| 模式 | 实时一致性 | 性能 | 复杂度 | 数据安全性 |
|------|-----------|------|--------|-----------|
| Cache Aside | 最终一致 | 高 | 低 | 可能短暂不一致 |
| 延迟双删 | 最终一致 | 高 | 中 | 比 Cache Aside 更安全 |
| Write Through | 强一致 | 低 | 高 | 最高 |
| Write Behind | 弱一致 | 最高 | 高 | 可能丢数据 |

**SOC AI 场景的推荐策略：**
RAG 缓存、SOC 告警详情等读多写少的场景，用 **Cache Aside + 合理 TTL** 即可。AI 任务状态等写多的场景，用 **延迟双删** 保证数据一致性。

## 11. 生产问题排查

### 11.1 bigkey 问题

**什么是 bigkey？**

单个 Key 的 Value 过大，称为 bigkey。常见于：
- `HGETALL` 一个有几万字段的 Hash
- `SMEMBERS` 一个有几百万成员的 Set
- `LRANGE` 一个有几万元素的 List

**bigkey 的危害：**
1. **内存不均**：大数据导致节点内存碎片化
2. **操作阻塞**：`HGETALL` 可能耗时几秒甚至几十秒，阻塞主线程
3. **主从同步阻塞**：`SYNC` 传输大 RDB 文件耗时极长
4. **网络带宽**：一次读取 bigkey 可能占满网络带宽
5. **RDB/AOF 重写**：AOF rewrite 和 RDB bgsave 时内存压力大

**如何发现 bigkey：**

```redis
# 方法1：redis-cli --bigkeys（扫描全库，按类型找最大的key，快速但不准）
redis-cli --bigkeys

# 方法2：SCAN + MEMORY USAGE（精确找到bigkey，不阻塞）
redis-cli --scan | head -1000 | while read key; do
    size=$(redis-cli memory usage "$key")
    if [ "$size" -gt 10485760 ]; then  # > 10MB
        echo "$key: $size bytes"
    fi
done

# 方法3：redis-cli --bigkeys -i 0.1（加扫描间隔，减少CPU影响）
# 方法4：INFO memory 查看内存使用详情
redis-cli info memory | grep -E "used_memory_human|mem_fragmentation_ratio"
```

**如何安全删除 bigkey：**

```python
# ⚠️ DEL 是同步删除，会阻塞主线程！严禁在大数据量场景直接用 DEL

# 正确做法1：用 UNLINK（非阻塞删除）
r.unlink("big:hash:key")  # Redis 4.0+，后台线程异步释放内存

# 正确做法2：渐进式删除（手动分批删）
def unlink渐进式删除(key, batch_size=1000):
    """分批删除 bigkey 的所有字段，避免一次性删除阻塞"""
    while True:
        # 分批删字段（每次删1000个）
        result = r.execute_command("HDEL", key, *r.hkeys(key)[:batch_size])
        if result == 0:
            break
        time.sleep(0.05)  # 每批之间暂停50ms，减少对主线程影响

# 正确做法3：在从节点删除（主从同步后再删主）
# 先在从节点 DEL，主从同步完成后在主节点执行 UNLINK
```

### 11.2 内存碎片

**为什么会产生内存碎片？**

1. **物理内存 vs 已用内存**：Redis 用 `jemalloc` 等内存分配器分配内存，分配/释放的内存块大小不一
2. **频繁更新**：频繁 `APPEND`、`SET` 会导致 Redis 重新分配内存，产生碎片
3. **大 Key 删除**：删掉大对象后，释放的内存不一定能完全利用

**查看碎片率：**

### 11.4 连接数问题

```python
# 连接泄漏排查示例
r = redis.Redis(host='localhost', port=6379)

# 查看当前连接数
info = r.info("clients")
print(f"当前连接数: {info['connected_clients']}")
print(f"阻塞连接数: {info['blocked_clients']}")

# 如果连接数接近 maxclients，问题通常是：
# 1. 未使用连接池，短连接频繁创建
# 2. 连接未关闭（try/finally 确保归还连接）
# 3. 客户端 maxclients 设置过小

# ✅ 正确做法：使用连接池（所有客户端库都支持）
pool = redis.ConnectionPool(host='localhost', port=6379, max_connections=50)
r1 = redis.Redis(connection_pool=pool)  # 从池中取连接
r2 = redis.Redis(connection_pool=pool)  # 复用连接，不创建新连接

# ⚠️ 错误做法：每次操作都新建连接
def bad_get(key):
    r = redis.Redis()  # 每次新建连接！连接数会爆炸
    return r.get(key)
```
