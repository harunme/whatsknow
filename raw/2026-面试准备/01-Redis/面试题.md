### 常用数据结构与底层实现
```
Redis 的 String、Hash、List、Set、ZSet 底层分别用什么实现？SDS 动态字符串相比 C 字符串有什么优势？
追问：
- SDS 的空间预分配策略是什么？
- 什么是 embstr 编码？什么时候会从 embstr 变成 raw？
```
**[10秒结论]** Redis 各数据类型根据数据量和大小选择不同底层编码，SDS 动态字符串通过空间预分配和惰性释放解决 C 字符串的溢出手痛和重复分配问题。

**[90秒拆解]**
- **数据类型 → 底层编码**：String 使用 int（整数）/ embstr（≤44字节）/ raw；Hash 使用 ziplist（元素少且小）→ hashtable；List 使用 ziplist → quicklist（3.2+）；ZSet 使用 ziplist → skiplist + dict；Set 使用 intset → hashtable
- **SDS vs C 字符串**：C 字符串获取长度 O(n)，SDS 存 len 字段 O(1)；C 追加可能缓冲区溢出，SDS 空间预分配减少分配次数；C 遇 `\0` 截断，SDS 二进制安全；SDS 用 `embstr` 编码让 redisObject + SDS 头 + 内容在一块连续内存，减少碎片

**[20秒收尾]** 生产中 Hash 的 `ziplist` 阈值（默认 `hash-max-ziplist-entries=512`, `hash-max-ziplist-value=64`）如果配置不当，容易触发 bigkey 问题。

### 持久化机制
```
RDB 和 AOF 各自的优缺点？混合持久化是怎么做的？为什么企业里通常同时开？
追问：
- `appendfsync` 的三个选项分别是什么？生产推荐哪个？
- RDB 的 `save` 和 `bgsave` 有什么区别？
```
**[10秒结论]** RDB 是定时快照文件恢复快但可能丢数据，AOF 是追加日志可靠性高但文件大速度慢，企业同时开两者让 RDB 做冷备、AOF 做实时保障。

**[90秒拆解]**
- **RDB 优点**：文件紧凑（压缩后 MB 级）、恢复快（`bgrewriteaof` 后台 fork 子进程不阻塞）；**缺点**：两次快照之间数据丢失，fork 在大数据量时阻塞
- **AOF 优点**：最多丢 1 秒数据（`everysec` 模式），rewrite 机制压缩文件；**缺点**：文件体积大，写入性能差
- **混合持久化**：`aof-use-rdb-preamble yes`，AOF 重写时先写 RDB 格式再追加增量，避免 AOF 重写期间 fork 阻塞
- **`appendfsync`**：`always`（每次写都刷盘，最安全但慢）、`everysec`（每秒一次，推荐）、`no`（交给 OS，最快但丢数据最多）

**[20秒收尾]** 腾讯云 SOC 项目中 Redis 主要存 Agent 状态和 RAG 缓存，用的是 AOF everysec + RDB 定时快照，保障数据不丢。

### 缓存穿透
```
什么是缓存穿透？怎么解决？（布隆过滤器、空值缓存、接口限流）
追问：
- 布隆过滤器的误报率和漏报率各是什么？
- 空值缓存的 TTL 应该设多长？
```
**[10秒结论]** 查询不存在的数据每次都穿透到数据库，通过布隆过滤器拦截不存在 key + 空值缓存 + 接口限流三招解决。

**[90秒拆解]**
- **什么是穿透**：数据在 DB 也不在缓存，攻击者可构造大量不存在 ID 的请求打垮数据库
- **布隆过滤器**：将所有合法 key 预先存入 Bloom Filter，查询前先检查，存在则继续查缓存/DB，不存在直接返回 null。误报率（存在但实际不存在）可通过增加位数降低
- **空值缓存**：DB 也查不到时，缓存一个 `"__NULL__"` 标记，TTL 短一些（如 5 分钟），防止缓存真实存在的数据
- **接口限流**：用 Redis 滑动窗口限流，防止大量穿透请求涌入

**[20秒收尾]** 生产中布隆过滤器 + 空值缓存组合最常见，成本低效果明显。注意布隆过滤器不支持删除（可用 Count-Min Sketch 或 RedisBloom 的 CBF）。

### 缓存击穿与雪崩
```
什么是缓存击穿和雪崩？各自有哪些解决方案？分布式锁在击穿场景怎么用？
追问：
- 热点 key 永不过期的逻辑过期方案怎么做？
- 雪崩和击穿的本质区别是什么？
```
**[10秒结论]** 击穿是热点 key 过期瞬间大量并发打到数据库，雪崩是大批 key 同时过期或 Redis 宕机，击穿用分布式锁，雪崩用过期时间随机化 + 高可用部署。

**[90秒拆解]**
- **击穿**：热点 key 过期时，大量请求全部穿透到 DB。**解决方案**：分布式锁（只让一个请求去加载数据，其他等待或返回旧数据）；逻辑过期（不设真正过期，用额外字段存逻辑过期时间，过期后用后台线程重建）
- **雪崩**：大量普通 key 集中过期。**解决方案**：过期时间加随机偏移 `TTL + rand(0, 300)`；Redis 高可用（Sentinel/Cluster）避免全量不可用；多级缓存（本地 Caffine + Redis + DB）
- **本质区别**：击穿是"一个 key 的问题"，雪崩是"一批 key 的问题"

**[20秒收尾]** 在 SOC 告警系统中，告警详情的 key 如果是热点 key，用分布式锁 + 逻辑过期方案最稳，保证高并发下 DB 不被打垮。

### 集群方案
```
主从复制、Sentinel、Cluster 三种方案各自解决什么问题？Cluster 的哈希槽是什么？为什么是 16384 个？
追问：
- 主从复制的 PSYNC 断点续传原理是什么？
- 槽迁移过程中请求会失败吗？
```
**[10秒结论]** 主从解决读写分离，Sentinel 在主从上实现自动故障切换，Cluster 通过 16384 个哈希槽实现数据分片和水平扩容。

**[90秒拆解]**
- **主从复制**：1 主 N 从，从只读。解决读性能瓶颈，不解决写性能和数据量瓶颈
- **Sentinel**：在主从基础上加监控进程，自动做故障检测和主从切换（故又称自动故障转移）
- **Cluster**：数据分片（每节点持有一部分槽）+ 每个分片主从复制。数据量 > 10GB、需水平扩容时用
- **16384 个槽的原因**：心跳包携带槽位信息（bit），16384 = 2KB；65,536 = 8KB，在百节点集群中心跳带宽会成为瓶颈；16384 足够精细，每个槽可对应数十万 key；Gossip 协议通信量低

**[20秒收尾]** 简历里写的是 Redis Sentinel 保障 AI 服务高可用，RAG 结果缓存用 Redis Cluster 做分片存储。

### 分布式锁
```
怎么用 Redis 实现分布式锁？SETNX + EXPIRE 有什么原子性问题？RedLock 解决了什么？
追问：
- 为什么释放锁需要 Lua 脚本？
- RedLock 有什么争议？
```
**[10秒结论]** 用 `SET key val NX EX ttl` 原子加锁，释放锁用 Lua 脚本保证只释放自己的锁；RedLock 通过多节点多数派解决单点故障但引入了时钟漂移问题。

**[90秒拆解]**
- **SETNX + EXPIRE 非原子问题**：SETNX 成功后进程崩溃，锁不过期成为死锁。**解决**：Redis 2.6.12+ 的 `SET key val NX EX ttl` 一步到位原子操作
- **释放锁的原子性**：用 Lua 脚本：`if redis.call("get", KEYS[1]) == ARGV[1] then return redis.call("del", KEYS[1]) else return 0 end`，防止误删他人持有的锁
- **RedLock**：向 5 个独立 Redis 节点同时加锁，N/2+1 成功才算加锁成功。**解决了单点故障**导致的锁丢失问题，但争议在于时钟漂移和多机房延迟问题

**[20秒收尾]** 生产中单节点 Redis 锁 + 合理 TTL + 重试已足够，RedLock 的运维复杂度通常大于收益。

### 缓存与数据库一致性
```
Cache Aside、Read/Write Through、Write Behind 三种模式各自的优缺点？延迟双删怎么做？
追问：
- 先更新库再删缓存和先删缓存再更新库哪个更好？为什么？
- 延迟双删的延迟时间怎么确定？
```
**[10秒结论]** Cache Aside（旁路缓存）最常用——读时懒加载、写时先更新 DB 再删缓存；延迟双删解决并发场景下先删缓存再更新库的竞态问题。

**[90秒拆解]**
- **Cache Aside**（最推荐）：读：缓存 → 未命中 → 读 DB → 写缓存；写：先更新 DB → 再删缓存（不是更新缓存，因为并发时可能让缓存变成脏数据）
- **Read/Write Through**：缓存层代理读写，逻辑复杂，一般不选
- **Write Behind**：先写缓存，异步批量写 DB，性能高但数据安全性低
- **延迟双删**：并发场景下，写线程先删缓存再更新 DB，更新 DB 期间读线程可能已将旧数据写回缓存。延迟 500ms 左右再删一次，让读线程的写操作完成

**[20秒收尾]** AI 场景中 RAG 缓存与向量数据库的更新也可以用类似思路——更新向量数据库后延迟清空相关 Redis 缓存条目。

### 生产问题排查
```
Redis CPU 飙高、内存突增、连接数爆了，怎么排查？bigkey 有什么危害？怎么发现？
追问：
- `redis-cli --bigkeys` 的原理是什么？
- bigkey 删除会阻塞吗？怎么安全删除？
```
**[10秒结论]** CPU 飙高查大命令或 O(n) 操作（禁用 KEYS），内存突增查 bigkey 和内存碎片，连接数爆了查连接池配置和短连接问题。

**[90秒拆解]**
- **CPU 飙高**：`redis-cli info stats` 查 `total_commands_processed`，看是否有异常 QPS 尖刺；`redis-cli --slowlog get 10` 查慢查询；生产禁用 `KEYS *`（阻塞主线程），用 `SCAN` 代替
- **内存突增**：查 `INFO memory` 的 `used_memory_human`；`redis-cli --bigkeys` 快速定位最大 key；`MEMORY USAGE key` 精确查单个 key 内存占用
- **bigkey 危害**：RDB 加载慢、AOF rewrite 阻塞、HGETALL 操作可能卡几十秒；**安全删除**：`UNLINK`（非阻塞删除）代替 `DEL`；或在从节点删除后主从同步
- **连接数爆**：`redis-cli info clients` 查连接数；原因通常是未使用连接池、短连接频繁创建、`maxclients` 配置过低

**[20秒收尾]** 生产中一定要配监控（Prometheus + Grafana），Redis Exporter 采集 `INFO` 指标，bigkey 和连接数设告警阈值。

### Redis Stream
```
Redis Stream 和 Redis Pub/Sub 有什么区别？消费者组（Consumer Group）解决了什么问题？消息 ID 是怎么生成的？
追问：
- 消费者挂了，没 ACK 的消息怎么处理？
- XREAD 和 XREADGROUP 各自的适用场景？
- 相比 RabbitMQ/Kafka，Redis Stream 的优缺点是什么？
```
**[10秒结论]** Redis Stream 是持久化消息队列，Pub/Sub 无状态不持久，Stream 用消费者组实现消息 ACK、断线重连不丢；消息 ID 由时间戳和序列号组成保证唯一有序。

**[90秒拆解]**
- **Pub/Sub vs Stream**：Pub/Sub 发布即推送无状态，订阅者掉线消息就丢；Stream 有持久化，消息写入 RDB/AOF，消费者组支持 ACK 确认
- **消费者组核心价值**：
  - **消息分发不重复**：同一消息只发给一个消费者
  - **断线不丢消息**：`XREADGROUP` 不带 `>` 取 pending 消息（已分发但未 ACK），重连后继续处理
  - **pending 转移**：`XCLAIM` 将超时未处理的消息转移给其他消费者
- **消息 ID**：`timestamp-msSeqNum`，如 `1700000000000-0`，毫秒级精度，同一毫秒内自增
- **XADD**：追加消息，自动生成 ID；`XREAD`：实时消费；`XREADGROUP`：消费者组模式，`>` 只取新消息
- **RabbitMQ vs Kafka**：RabbitMQ 功能完整但运维复杂；Kafka 适合大数据量有序分区；Redis Stream 轻量、内嵌 Redis 无独立部署、适合中小型异步任务

**[20秒收尾]** 腾讯云 SOC 中 AI 任务队列用 Redis Stream 做异步编排——任务创建 XADD，执行器 XREADGROUP 消费，XACK 确认，失败 XCLAIM 转移，结合重试次数限制实现可靠任务调度。
