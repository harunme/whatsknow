## 细节

### [初] 数据结构

**SDS vs C 字符串**
- C 字符串：O(n) 获取长度，可能缓冲区溢出
- SDS：存 len 字段 O(1) 获取长度，空间预分配减少分配次数，二进制安全

**数据类型→底层编码**

| 数据类型 | 小数据量 | 大数据量 |
|---------|---------|---------|
| String | int / embstr | raw |
| Hash | ziplist | hashtable |
| List | ziplist → quicklist | quicklist |
| ZSet | ziplist | skiplist + dict |
| Set | intset | hashtable |

**embstr**：redisObject + SDS 头 + 内容在一块连续内存，减少碎片；≤44 字节时用 embstr

### [初] 持久化

**RDB**
- 定时快照，fork 子进程执行，不阻塞主进程
- 优点：文件紧凑，恢复快
- 缺点：可能丢失最后一次快照后的数据

**AOF**
- 追加写命令到日志，rewrite 压缩文件
- `appendfsync`：`always`（每次）、`everysec`（每秒，推荐）、`no`（OS 控制）

**混合持久化**：`aof-use-rdb-preamble yes`，AOF 重写时先写 RDB 格式再追加增量

---

### [中] 缓存穿透/击穿/雪崩

**穿透**：数据在 DB 也不在缓存，攻击构造不存在 ID 大量请求
- 解决：布隆过滤器拦截 + 空值缓存（TTL 短，如 5 分钟）

**击穿**：热点 key 过期瞬间大量并发打到 DB
- 解决：分布式锁（只让一个请求加载数据）+ 逻辑过期（不设真正过期，后台线程重建）

**雪崩**：大批 key 同时过期或 Redis 宕机
- 解决：过期时间加随机偏移 `TTL + rand(0,300)` + Redis 高可用（Sentinel/Cluster）

---

### [中] 集群方案

**主从复制**：1 主 N 从，从只读，解决读性能瓶颈

**Sentinel**：自动故障检测和主从切换

**Cluster**：数据分片（16384 哈希槽）+ 每分片主从复制

**16384 槽原因**：心跳包携带槽位信息（bit），16384=2KB；65,536=8KB，百节点集群中心跳带宽成为瓶颈

---

### [高] 分布式锁

**正确加锁**：`SET key val NX EX ttl`（原子操作，解决 SETNX+EXPIRE 非原子问题）

**安全释放**：Lua 脚本
```lua
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("del", KEYS[1])
else return 0 end
```

**RedLock**：向 5 个独立节点同时加锁，N/2+1 成功才算加锁成功；解决了单点故障，但有时钟漂移争议

---

### [高] 一致性

**Cache Aside**（最推荐）：
- 读：缓存→未命中→读DB→写缓存
- 写：先更新DB→再删缓存（不是更新缓存）

**延迟双删**：先删缓存→更新DB→延迟500ms再删一次（解决并发竞态）

---

### [高] 生产问题

**bigkey**：HGETALL 卡顿、RDB 加载慢
- 发现：`redis-cli --bigkeys` 或 `SCAN` + `MEMORY USAGE key`
- 危害：主从同步阻塞、内存碎片
- 安全删除：`UNLINK`（非阻塞）代替 `DEL`

**slowlog**：`redis-cli --slowlog get 10`，阈值默认 10ms

**连接数爆**：`redis-cli info clients`，原因：短连接频繁创建、`maxclients` 过低
