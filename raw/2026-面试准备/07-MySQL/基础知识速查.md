## 细节

### [初] 存储引擎

**InnoDB vs MyISAM**

| 特性 | InnoDB | MyISAM |
|------|--------|--------|
| 事务 | 支持 | 不支持 |
| 锁粒度 | 行锁 | 表锁 |
| 外键 | 支持 | 不支持 |
| 崩溃恢复 | 自动 | 需手动 |
| 适用场景 | OLTP/高并发 | OLAP/读多写少 |

**聚簇索引 vs 非聚簇索引**
- 聚簇索引：叶子节点存储完整行数据（主键索引 = 聚簇索引）
- 非聚簇索引：叶子节点存储主键值，需要回表查询完整行数据
- 每张表只有一个聚簇索引

**主键选择**
- 优先使用 BIGINT 自增主键（避免页分裂、索引体积小）
- 避免 UUID/字符串主键（无序、索引膨胀）

### [初] 索引类型

**索引分类**
- 主键索引：聚簇索引，唯一且非空
- 唯一索引：唯一约束，可有空值
- 普通索引：无约束，加速查询
- 联合索引：多列索引，遵循最左前缀原则

**最左前缀原则**
- 联合索引 (a, b, c) 能命中：a、(a,b)、(a,b,c)
- 不能命中：b、c、(b,c)

**覆盖索引**
- 查询字段全部在索引中，无需回表
- `SELECT name, age FROM users WHERE name = 'Tom'` 若有联合索引 (name, age) 则覆盖

**索引失效**
- `%abc`（左模糊）、OR 连接非索引列、函数/表达式操作索引列

### [中] 事务与隔离级别

**ACID**
- Atomicity（原子性）：事务要么全成功，要么全失败
- Consistency（一致性）：事务前后数据状态一致
- Isolation（隔离性）：并发事务互不干扰
- Durability（持久性）：提交后数据持久保存

**四种隔离级别**
- READ UNCOMMITTED：最低，允许脏读
- READ COMMITTED：每次读取生成新 readView，解决脏读
- REPEATABLE READ（InnoDB 默认）：事务开始时生成 readView，事务内一致，解决不可重复读
- SERIALIZABLE：强制串行

**MVCC 原理**
- 每行数据有隐式主键和两个版本字段（DATA_TRX_ID、DATA_ROLL_PTR）
- readView 包含活跃事务 IDs 列表和最小/最大事务 ID
- 可见性判断：遍历版本链，找第一个 trx_id < readView.min_id 的版本

**Next-Key Lock**
- Record Lock（行锁）+ Gap Lock（间隙锁）
- 锁定索引记录及其前面的间隙，防止幻读

### [中] 锁机制

**行锁 vs 表锁**
- InnoDB 行锁加在索引记录上，非索引字段查询退化为表锁
- MyISAM 只有表锁

**间隙锁（Gap Lock）**
- 锁定索引间隙，防止幻读
- 索引有 1, 5, 10，更新 WHERE id > 3 AND id < 9，锁定 (5,10) 间隙

**死锁产生**
- 事务 A 锁 X1 等 X2，事务 B 锁 X2 等 X1
- 解决：InnoDB 默认超时回滚代价最小的事务

**两阶段锁协议**
- 加锁阶段只能加锁，解锁阶段只能解锁
- InnoDB 遵循此协议，锁在事务提交/回滚时才释放

### [中] SQL 优化

**EXPLAIN 关键字段**
- type：const > eq_ref > ref > range > index > ALL（ref 以上才算好）
- key：实际使用的索引
- rows：扫描行数，越少越好
- Extra：Using index（覆盖索引）、Using filesort（需排序）、Using temporary（需临时表）

**LIMIT 分页优化**
```sql
-- 反例：需扫描前100020行
SELECT * FROM orders LIMIT 100000, 20

-- 正例：延迟关联
SELECT t.* FROM orders t
INNER JOIN (SELECT id FROM orders WHERE user_id=1 ORDER BY id LIMIT 100000,20) r
ON t.id = r.id
```

**JOIN 优化**
- 小表驱动大表（LEFT JOIN 时右表尽量小）
- 确保 ON 条件有索引
- 避免 SELECT *

### [高] 主从复制

**binlog 格式**
- STATEMENT：记录 SQL 语句（可能主从不一致，如 NOW()）
- ROW：记录行变化（最安全，占用空间大）
- MIXED：自动切换

**异步 vs 半同步**
- 异步：从库不确认收到即提交
- 半同步：主库等待至少一个从库确认收到 binlog 才提交

**GTID**
- Global Transaction Identifier：UUID:SN，每个事务唯一标识
- 故障切换时新主库可直接从断点继续，无需 binlog 位置对齐

**主从延迟解决**
- 多线程并行复制（5.7+ LOGICAL_CLOCK）
- 延迟从库读（读流量分离）

---

### [高] 分库分表

**水平 vs 垂直分片**
- 水平：按行拆分（按 ID mod N 或按时间）
- 垂直：按列拆分（不同业务列分不同库）

**ShardingSphere**
- ShardingSphere-JDBC：客户端分片，Java 应用引入
- ShardingSphere-Proxy：代理分片，MySQL 协议接入

**分片键选择**：选择查询最频繁字段，避免跨分片 JOIN

**分布式 ID 生成**：雪花算法（时间戳+机器ID+序列号）

### [高] 备份恢复

**mysqldump**
```bash
mysqldump -A -F --single-transaction > backup.sql
# --single-transaction 保证一致性（InnoDB）
```

**xtrabackup**：物理备份，支持增量备份，适合 TB 级数据库

**Point-in-time Recovery**
```bash
mysql < full_backup.sql
mysqlbinlog --stop-datetime="2024-01-01 12:00:00" binlog.000001 | mysql
```
