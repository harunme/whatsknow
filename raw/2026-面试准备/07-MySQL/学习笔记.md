## 一、MySQL 基础概念

### 1.1 关系型数据库核心概念

关系型数据库（Relational Database）是用**关系模型**（即二维表）组织数据的数据库系统。每个数据库由多个**表（Table）**组成，表与表之间通过**外键（Foreign Key）**建立关联关系。

**为什么 MySQL 能成为互联网标配？**

```
关系模型 → 结构化查询语言（SQL）→ 数据一致性强
+ 行式存储 → 高效单行读写
+ 事务支持 → 并发场景下数据不错乱
+ 成熟生态 → 主从复制、高可用方案丰富
```

对于 SOC 平台来说，告警事件、资产数据、用户权限等天然适合用二维表结构存储，并通过 SQL 进行复杂的安全事件关联分析。

### 1.2 SQL 分类（DDL/DML/DQL/DCL）

SQL 不是铁板一块，按功能分为四类，每类掌管数据库的不同层面：

| 分类 | 全称 | 核心命令 | 对数据的影响 |
|------|------|----------|-------------|
| **DDL** | Data Definition Language | CREATE, DROP, ALTER, TRUNCATE | 定义/删除表结构，不涉及数据 |
| **DML** | Data Manipulation Language | INSERT, UPDATE, DELETE | 增删改数据，会记录 binlog |
| **DQL** | Data Query Language | SELECT | 查询，不修改数据 |
| **DCL** | Data Control Language | GRANT, REVOKE, COMMIT, ROLLBACK | 权限控制和事务控制 |

**代码示例 - DDL：**

```sql
-- 创建告警表（DDL）
CREATE TABLE alerts (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,       -- 主键
    tenant_id BIGINT NOT NULL COMMENT '租户ID',
    alert_name VARCHAR(255) NOT NULL COMMENT '告警名称',
    severity TINYINT NOT NULL COMMENT '严重程度：1低 2中 3高 4严重',
    status TINYINT DEFAULT 1 COMMENT '状态：1未处理 2已确认 3已解决',
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
    updated_at DATETIME DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='安全告警表';

-- 修改表结构（DDL）：添加索引
ALTER TABLE alerts ADD INDEX idx_tenant_created (tenant_id, created_at);

-- 删除表（DDL）
DROP TABLE IF EXISTS alerts;

-- 清空表（DDL/DML 边界，TRUNCATE 是 DDL，立即释放空间）
TRUNCATE TABLE alerts;
```

**⚠️ 注意：** TRUNCATE 和 DELETE 有本质区别。TRUNCATE 是 DDL（自动 commit，无法回滚），速度极快但不可恢复；DELETE 是 DML（可回滚），逐行删除并触发触发器。

**代码示例 - DML：**

```sql
-- 插入告警记录
INSERT INTO alerts (tenant_id, alert_name, severity, status)
VALUES (1001, '暴力破解告警', 3, 1);

-- 批量插入（推荐，性能更好）
INSERT INTO alerts (tenant_id, alert_name, severity, status)
VALUES
    (1001, '异常登录', 2, 1),
    (1001, 'webshell上传', 4, 1),
    (1002, '横向移动', 3, 1);

-- 更新告警状态
UPDATE alerts
SET status = 2, updated_at = NOW()
WHERE id = 1 AND status = 1;  -- 只更新未处理的告警

-- 删除已解决的旧数据（按条件删除）
DELETE FROM alerts
WHERE status = 3 AND updated_at < DATE_SUB(NOW(), INTERVAL 180 DAY);
```

**代码示例 - DQL：**

```sql
-- 关联查询：告警 + 租户信息
SELECT
    a.id,
    a.alert_name,
    CASE a.severity
        WHEN 1 THEN '低'
        WHEN 2 THEN '中'
        WHEN 3 THEN '高'
        WHEN 4 THEN '严重'
    END AS severity_text,
    t.tenant_name,
    DATE_FORMAT(a.created_at, '%Y-%m-%d %H:%i') AS created_time
FROM alerts a
INNER JOIN tenants t ON a.tenant_id = t.id
WHERE a.tenant_id = 1001
  AND a.severity >= 3
  AND a.created_at >= DATE_SUB(NOW(), INTERVAL 7 DAY)
ORDER BY a.created_at DESC
LIMIT 100;
```

### 1.3 存储引擎概述

MySQL 是**插件式存储引擎**架构，同一个数据库里不同表可以有不同的引擎。查看当前默认引擎：

```sql
-- 查看默认存储引擎
SHOW VARIABLES LIKE 'default_storage_engine';
-- 结果：InnoDB

-- 查看表使用的引擎
SHOW TABLE STATUS FROM db_name LIKE 'alerts';
```

MySQL 5.5+ 默认引擎从 MyISAM 切换为 InnoDB，这一改变直接影响了几代 MySQL 开发者的认知。原因是互联网高并发写场景下，MyISAM 的表锁机制成为瓶颈，InnoDB 的行锁和事务支持更适合 OLTP 场景。

## 二、存储引擎详解：InnoDB vs MyISAM

### 2.1 核心特性对比

| 特性 | InnoDB | MyISAM |
|------|--------|--------|
| **事务支持** | ACID 完整支持 | 不支持 |
| **锁粒度** | 行锁（row-level lock） | 表锁（table-level lock） |
| **外键约束** | 支持 | 不支持 |
| **并发能力** | 高（行锁 + MVCC） | 低（表锁，写阻塞读） |
| **崩溃恢复** | 自动恢复（redo log） | 需手动 repair table |
| **全文索引** | 5.6+ 支持 | 原生支持 |
| **存储结构** | 共享表空间或独立表空间 | 每表三个文件（.frm/.MYD/.MYI） |
| **索引类型** | 聚簇索引（B+Tree） | 非聚簇索引（B+Tree） |
| **适用场景** | OLTP / 高并发写 | OLAP / 读多写少 |

**为什么 InnoDB 的行锁比 MyISAM 的表锁更适合写多场景？**

MyISAM 的表锁机制意味着：当一个事务在写（UPDATE）时，其他所有事务对该表的读和写都被阻塞。假设 1000 QPS 的告警入库场景，每条写事务持有锁 1ms，则同一时刻只能有一个写操作进行，吞吐量上限是 1000 TPS。

InnoDB 的行锁只在索引记录上加锁：更新 `id=1` 的行时，只锁定 `id=1` 的索引记录，`id=2` 的行完全不受影响。这样 1000 条并发告警入库可以同时进行，吞吐量提升数十倍。

### 2.2 表空间与物理存储

**InnoDB 物理存储结构：**

```
InnoDB 表空间（tablespace）
├── 系统表空间（ibdata1）：默认共享，包含：
│   ├── 数据字典（表结构元数据）
│   ├── 变更缓冲区（change buffer）
│   ├── 双写缓冲区（doublewrite buffer）
│   └── undo log（回滚段）
└── 独立表空间（*.ibd）：每表一个，包含：
    ├── 数据页（16KB/page）
    ├── 索引页
    └── 页内行数据
```

MySQL 5.6+ 默认启用 `innodb_file_per_table=ON`，每个表独立存储为 `.ibd` 文件。这带来几个关键优势： DROP TABLE 可以立即释放磁盘空间（共享表空间需要重建）；单表可以单独备份；大表删除后空间立即归还 OS。

```sql
-- 查看表空间使用情况（MySQL 8.0+）
SELECT
    TABLE_NAME,
    ROUND(DATA_LENGTH / 1024 / 1024, 2) AS '数据大小(MB)',
    ROUND(INDEX_LENGTH / 1024 / 1024, 2) AS '索引大小(MB)',
    ROUND(DATA_FREE / 1024 / 1024, 2) AS '碎片空间(MB)',
    TABLE_ROWS
FROM information_schema.TABLES
WHERE TABLE_SCHEMA = 'security_db'
ORDER BY DATA_LENGTH DESC;
```

**⚠️ 注意：** DATA_FREE 列显示的碎片空间，是因为 DELETE 不会归还空间给 OS，只有 OPTIMIZE TABLE 或 ALTER TABLE ... ENGINE=InnoDB 才能重建表回收空间。

### 2.3 如何选择存储引擎

```
选 InnoDB 的场景：
  ✅ 几乎所有 OLTP 业务表（SOC 平台的告警、资产、事件表）
  ✅ 需要事务支持（告警状态变更、用户权限更新）
  ✅ 并发写入量大（SIEM 日志采集每秒万级写入）
  ✅ 需要崩溃自动恢复（服务器宕机后自动恢复）

选 MyISAM 的场景：
  ✅ 日志表：只追加（APPEND），几乎不更新和删除
  ✅ 全文搜索表：需要 FULLTEXT 索引（5.5 以下版本）
  ✅ 历史归档表：只读，不用事务，不需要崩溃恢复
  ✅ 测试/演示环境：不需要高可靠
```

**SOC 平台场景实战：**

```sql
-- 告警表：必须 InnoDB（事务 + 行锁 + 崩溃恢复）
CREATE TABLE alerts (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    tenant_id BIGINT NOT NULL,
    alert_name VARCHAR(255) NOT NULL,
    severity TINYINT NOT NULL,
    status TINYINT DEFAULT 1,
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
    updated_at DATETIME DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;

-- 告警日志表：只追加，不更新，可考虑 MyISAM（节省内存）
CREATE TABLE alert_audit_log (
    id BIGINT AUTO_INCREMENT PRIMARY KEY,
    alert_id BIGINT NOT NULL,
    operator_id BIGINT NOT NULL,
    action VARCHAR(50) NOT NULL,
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP
) ENGINE=MyISAM DEFAULT CHARSET=utf8mb4;  -- 审计日志只写不读，MyISAM 表锁影响可接受
```

## 三、索引原理与结构

### 3.1 B+Tree 为什么是 MySQL 的选择

理解索引，首先要理解 MySQL 如何存储和查找数据。

**磁盘 I/O 是性能瓶颈：**

CPU 访问内存的速度是纳秒级（~100ns），而访问磁盘是毫秒级（~10ms），差距高达 **100,000 倍**。数据库为了减少磁盘 I/O次数，必须尽量让每次 I/O 读取足够多的数据到内存。MySQL 的数据页（page）大小固定为 **16KB**，每次磁盘 I/O 读取一页数据。

**B-Tree vs B+Tree：**

```
B-Tree（阶数 m=3 的示意图）：

        [30 | 60]
       /    |    \
  [10|20] [40|50] [70|80|90]
  /   \    /   \    /    \
叶子节点存储数据（无序链表）

B+Tree（阶数 m=3 的示意图）：

        [30 | 60]
       /    |    \
  [10|20] [40|50] [70|80|90]
  /   \    /   \    /    \
[10→] [20→] [40→] [50→] [70→] [80→] [90→]  ← 所有数据在叶子节点，链表连接
```

**B+Tree 的三个关键特性使其成为 MySQL 索引的首选：**

1. **矮胖结构，减少 I/O 次数**：非叶子节点只存储索引键（不存完整行数据），相同深度能容纳更多索引。每个非叶子节点可存 16KB / (8B 主键 + 6B 指针) ≈ 1170 个条目。高度 3 的 B+Tree 可索引 1170 × 1170 × 16 ≈ **2000万** 条记录，只需 3 次 I/O。

2. **所有数据在叶子节点，查询稳定**：B-Tree 的数据可能出现在任意层，导致不同查询路径深度不一致；B+Tree 所有查询都必须走到叶子层，时间复杂度恒定为 O(log n)。

3. **叶子节点链表，范围查询极快**：`WHERE id > 50 AND id < 80` 的查询，只需定位到 50，从链表向右遍历即可，无需回溯树。

```sql
-- 在 2000 万行数据的表上，主键查询只需要 3 次 I/O（0.03ms 级别）
SELECT * FROM alerts WHERE id = 15000000;
-- type=const（常量级别查找），key=PRIMARY，rows=1
```

### 3.2 聚簇索引 vs 非聚簇索引

这是 MySQL 索引体系中最核心的概念，理解清楚了，其他索引问题迎刃而解。

**聚簇索引（Clustered Index）：**

- 索引键值确定后，叶子节点**直接存储完整行数据**（整行数据就挂在索引叶子节点上）
- 每张表**只能有一个**聚簇索引（因为数据物理上只能按一种方式排序存储）
- InnoDB 中，主键索引就是聚簇索引；如果没有主键，选择第一个唯一索引；如果都没有，InnoDB 生成隐式 ROW_ID

```
聚簇索引结构（InnoDB）：

主键索引 B+Tree 叶子节点：
  [id=1] → 完整行数据：{id=1, name='告警A', severity=3, status=1, ...}
  [id=2] → 完整行数据：{id=2, name='告警B', severity=4, status=1, ...}
  [id=3] → 完整行数据：{id=3, name='告警C', severity=2, status=2, ...}
  ...
```

**非聚簇索引（Secondary Index / 二级索引）：**

- 叶子节点**不存完整行数据，只存储主键值**
- 查询时先在二级索引查到主键，再回表（查主键索引）获取完整行数据
- 这就是经典的"**回表查询**"

```
非聚簇索引结构（InnoDB）：

二级索引 (tenant_id) 叶子节点：
  [tenant_id=1001, id=1] → 存储主键 id=1
  [tenant_id=1001, id=5] → 存储主键 id=5
  [tenant_id=1002, id=2] → 存储主键 id=2
  ...

回表：先在 tenant_id 索引查到 id=1，再查主键索引获取完整行数据
```

**代码示例 - 验证聚簇索引 vs 非聚簇索引：**

```sql
-- 创建测试表，明确指定主键（聚簇索引）
CREATE TABLE index_demo (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,    -- 聚簇索引，叶子节点存完整行
    tenant_id BIGINT NOT NULL,
    alert_name VARCHAR(100),
    severity TINYINT NOT NULL,
    INDEX idx_tenant (tenant_id),             -- 非聚簇索引，叶子节点存主键
    UNIQUE KEY uk_name (alert_name)           -- 非聚簇索引，唯一约束
) ENGINE=InnoDB;

INSERT INTO index_demo (tenant_id, alert_name, severity)
VALUES (1001, '暴力破解', 3), (1001, '异常登录', 2), (1002, 'webshell', 4);

-- 查询1：通过主键查询（直接命中聚簇索引，无需回表）
EXPLAIN SELECT * FROM index_demo WHERE id = 1;
-- 结果：type=const, key=PRIMARY, Extra=(空，表示直接返回)
-- 解释：聚簇索引叶子节点就是完整行数据，不需要二次查询

-- 查询2：通过普通索引查询（先查二级索引得主键，再回表）
EXPLAIN SELECT * FROM index_demo WHERE tenant_id = 1001;
-- 结果：type=ref, key=idx_tenant, Extra=''
-- 解释：先在 idx_tenant 查到 id=1,5（两个主键），再分别回表查主键索引

-- 查询3：覆盖索引（只查索引字段，不回表）
EXPLAIN SELECT tenant_id, id FROM index_demo WHERE tenant_id = 1001;
-- 结果：type=ref, key=idx_tenant, Extra=Using index
-- 解释：tenant_id 和 id 都在 idx_tenant 索引中，直接返回，无需回表
```

### 3.3 索引分类与创建

MySQL 支持多种索引类型，适用于不同场景：

```sql
-- 主键索引：聚簇索引，唯一且非空
CREATE TABLE t1 (
    id BIGINT PRIMARY KEY  -- 隐式创建主键索引
);

-- 唯一索引：非聚簇索引，唯一约束，可有空值
CREATE TABLE t2 (
    email VARCHAR(100) UNIQUE KEY  -- email 值唯一
);

-- 普通索引：非聚簇索引，加速查询
CREATE TABLE t3 (
    alert_name VARCHAR(100),
    INDEX idx_alert_name (alert_name)  -- 普通索引
);

-- 全文索引：用于文本内容的关键词搜索
ALTER TABLE t3 ADD FULLTEXT INDEX ft_alert (alert_name);

-- 联合索引：多列索引，最左前缀原则
CREATE TABLE alerts (
    tenant_id BIGINT NOT NULL,
    severity TINYINT NOT NULL,
    created_at DATETIME,
    INDEX idx_tenant_severity (tenant_id, severity, created_at)
);
```

**MySQL 8.0+ 新增特性：** 支持函数索引（Expression Index），可以基于表达式创建索引：

```sql
-- 在 created_at 列上按日期索引（绕过函数导致的不走索引问题）
CREATE INDEX idx_created_date ON alerts (DATE(created_at));
SELECT * FROM alerts WHERE DATE(created_at) = '2026-04-10';  -- 现在能命中索引
```

### 3.4 联合索引与最左前缀原则

联合索引（Composite Index）是对多列创建的索引，其底层 B+Tree 结构按照创建顺序排列各列的值。

**最左前缀原则的底层原理：**

```sql
-- 创建联合索引
INDEX idx_tenant_severity_created (tenant_id, severity, created_at)

-- B+Tree 排列顺序（字典序，先比较 tenant_id，相同时比较 severity，再比 created_at）：
--   (1001, 1, '2026-01-01')
--   (1001, 1, '2026-01-15')
--   (1001, 2, '2026-01-01')
--   (1001, 3, '2026-04-10')
--   (1002, 1, '2026-02-01')
--   (1002, 2, '2026-03-01')
```

**为什么这样排列？因为 B+Tree 是有序树，必须从左到右依次比较。** 查询 `(1001, 2, '2026-01-01')` 时，可以直接定位；但查询 `(1001, ?, '2026-01-01')` 时，severity 未知，无法确定顺序；查询 `(?, 2, '2026-01-01')` 时，severity 和 created_at 都已知，但 tenant_id 是第一排序字段，完全无法利用索引。

```sql
-- 测试最左前缀原则
EXPLAIN SELECT * FROM alerts WHERE tenant_id = 1001;
-- ✅ 命中索引 idx_tenant_severity_created（使用了第一列）

EXPLAIN SELECT * FROM alerts WHERE tenant_id = 1001 AND severity = 3;
-- ✅ 命中索引（使用前两列）

EXPLAIN SELECT * FROM alerts
WHERE tenant_id = 1001 AND severity = 3 AND created_at > '2026-04-01';
-- ✅ 命中索引（使用全部三列）

EXPLAIN SELECT * FROM alerts WHERE severity = 3;
-- ❌ 不命中索引（跳过了第一列，MySQL 不知道从哪开始找）

EXPLAIN SELECT * FROM alerts WHERE created_at > '2026-04-01';
-- ❌ 不命中索引（跳过了前两列）

EXPLAIN SELECT * FROM alerts
WHERE tenant_id > 1001 AND severity = 3;
-- ⚠️ 只用到了 tenant_id（> 是范围查询，之后的列断链）
```

**为什么范围查询会导致后面的列断链？** 以 `(tenant_id > 1001, severity = 3)` 为例：tenant_id > 1001 会匹配 `(1002, ...)` 和 `(1003, ...)` 等多个值，每个值内部的 severity 无序，所以 severity 列无法利用索引。

### 3.5 覆盖索引

覆盖索引（Covering Index）是指**查询的所有字段都在索引结构中，无需回表查询聚簇索引**。这是 SQL 优化的核心手段之一。

```sql
-- 场景：查询告警列表，需要展示 severity 等级的中文含义
CREATE TABLE alerts (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    tenant_id BIGINT NOT NULL,
    alert_name VARCHAR(255) NOT NULL,
    severity TINYINT NOT NULL,           -- 1低 2中 3高 4严重
    status TINYINT DEFAULT 1,
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP
);
CREATE INDEX idx_tenant_severity ON alerts (tenant_id, severity);

-- 反例：SELECT * 会回表
EXPLAIN SELECT * FROM alerts WHERE tenant_id = 1001;
-- type=ref, key=idx_tenant_severity, Extra=(空，extra 无覆盖索引提示)

-- 正例：只查索引列，覆盖索引，不回表
EXPLAIN SELECT tenant_id, severity FROM alerts WHERE tenant_id = 1001;
-- type=ref, key=idx_tenant_severity, Extra=Using index
-- Using index 表示使用了覆盖索引（索引覆盖查询）

-- 正例2：告警列表分页，只取需要的字段
EXPLAIN SELECT id, alert_name, severity, created_at
FROM alerts
WHERE tenant_id = 1001
ORDER BY created_at DESC
LIMIT 20;
-- key=idx_tenant_severity_created（联合索引同时满足过滤+排序+覆盖）
```

**性能差异的量化：** 在 1000 万行表中，回表查询可能需要 50-200ms（多次随机 I/O），而覆盖索引查询仅需 1-5ms（索引树遍历 + 顺序读）。对于 SOC 平台告警大屏的实时查询，覆盖索引可将 P99 延迟从 500ms 降至 20ms。

## 四、索引失效的场景

### 4.1 常见失效原因分析

索引虽好，但 MySQL 优化器在以下场景会选择不走索引（走全表扫描），这是面试高频考点。

**原因1：OR 条件中存在非索引列**

```sql
-- tenant_id 有索引，status 无索引
EXPLAIN SELECT * FROM alerts
WHERE tenant_id = 1001 OR status = 2;
-- ❌ 索引失效，走全表扫描（type=ALL）
-- 原因：OR 条件要求两边都有索引才能走 INDEX MERGE；status 无索引，优化器直接放弃

-- 正确做法：拆分为 UNION，或确保 OR 两边都有索引
EXPLAIN SELECT * FROM alerts WHERE tenant_id = 1001
UNION ALL
SELECT * FROM alerts WHERE status = 2 AND tenant_id != 1001;
```

**原因2：对索引列使用函数或表达式**

```sql
-- created_at 有索引
EXPLAIN SELECT * FROM alerts WHERE YEAR(created_at) = 2026;
-- ❌ 索引失效，YEAR() 函数作用于索引列，无法利用 B+Tree 的有序性

EXPLAIN SELECT * FROM alerts WHERE created_at >= '2026-01-01';
-- ✅ 正确：对索引列直接比较，而不是用函数包裹

-- 对主键使用表达式也失效
EXPLAIN SELECT * FROM alerts WHERE id + 1 = 100;
-- ❌ 索引失效：id + 1 不是原值，无法二分查找

EXPLAIN SELECT * FROM alerts WHERE id = 99;
-- ✅ 正确：直接使用索引列
```

**⚠️ 注意：** 很多人知道"不要在索引列上用函数"，但忽略了隐式类型转换。例如：`phone VARCHAR(20)` 列有索引，`WHERE phone = 123456` 会将字符串列转为数字，反而触发函数转换导致索引失效。

**原因3：LIKE 前导通配符**

```sql
-- alert_name 有索引
EXPLAIN SELECT * FROM alerts WHERE alert_name LIKE '暴力%';
-- ✅ 走索引，'暴力%' 是前缀匹配，可按字典序定位起点

EXPLAIN SELECT * FROM alerts WHERE alert_name LIKE '%暴力%';
-- ❌ 索引失效，全表扫描，前缀通配符无法利用 B+Tree 的有序性

EXPLAIN SELECT * FROM alerts WHERE alert_name LIKE '%暴力';
-- ❌ 索引失效，前导通配符

-- 解决方案：全文索引（FULLTEXT）适合后缀/中缀匹配
ALTER TABLE alerts ADD FULLTEXT INDEX ft_alert (alert_name);
SELECT * FROM alerts WHERE MATCH(alert_name) AGAINST('暴力' IN NATURAL LANGUAGE MODE);
```

**原因4：隐式类型转换**

```sql
-- phone 列定义为 VARCHAR，但传入的是数字
EXPLAIN SELECT * FROM users WHERE phone = 13800138000;
-- ⚠️ 可能失效：MySQL 将 VARCHAR 转成 DOUBLE 比较（隐式函数）
-- phone = '13800138000' → 才走索引

-- 判断方法：用 EXPLAIN 查看 type，ALL 表示全表扫描
```

**原因5：使用 IS NULL / IS NOT NULL**

```sql
-- status 列有索引，但大量行 status IS NOT NULL
EXPLAIN SELECT * FROM alerts WHERE status IS NOT NULL;
-- ⚠️ 可能不走索引：如果 NULL 值极少（<5%），优化器认为全表扫描更快
-- 因为索引需要额外一次 I/O 跳过 NULL 值，直接全表顺序读更快

-- 建议：给 status 加默认值 NOT NULL DEFAULT 1，避免 NULL 判断
```

**原因6：优化器选择（统计信息不准确）**

MySQL 依赖表统计信息（`ANALYZE TABLE` 刷新）决定是否走索引。当数据量小、统计信息过期或数据分布极不均匀时，优化器可能误判。

```sql
-- 手动刷新统计信息
ANALYZE TABLE alerts;
SHOW INDEX FROM alerts;  -- 查看索引基数（cardinality）
```

### 4.2 如何验证索引是否生效

```sql
-- 最基本：EXPLAIN 查看执行计划
EXPLAIN SELECT * FROM alerts WHERE tenant_id = 1001;

-- 深度分析：EXPLAIN ANALYZE（MySQL 8.0+）
EXPLAIN ANALYZE
SELECT t.* FROM alerts t
WHERE tenant_id = 1001 AND severity >= 3
ORDER BY created_at DESC LIMIT 20;
-- 输出示例：
-- -> Sort: t.created_at DESC  (cost=1234 rows=89) (actual time=0.015..0.089)
--     -> Index lookup on t using idx_tenant_sev (tenant_id=1001, severity>=3)
--         (cost=890 rows=89) (actual time=0.008..0.067 rows=89)
```

**⚠️ 注意：** EXPLAIN 的 `rows` 列是估算值，`actual time` 是实际执行时间。两者都重要——如果 rows 估算与实际 rows 差距巨大，说明统计信息过期了，需要 `ANALYZE TABLE`。

## 五、事务详解

### 5.1 ACID 特性

事务（Transaction）是数据库操作的**最小工作单元**，要么全部成功，要么全部失败。ACID 是事务的四个基本特性：

| 特性 | 含义 | 实现原理 | 破坏场景 |
|------|------|----------|---------|
| **A - Atomicity 原子性** | 事务内操作不可分割，要么全做，要么全不做 | undo log（回滚日志） | 停电、进程崩溃 |
| **C - Consistency 一致性** | 事务前后数据状态一致（满足所有约束） | 其他三个特性共同保证 | 外键约束违反、唯一键冲突 |
| **I - Isolation 隔离性** | 并发事务互不干扰 | 锁机制 + MVCC | 脏读、不可重复读、幻读 |
| **D - Durability 持久性** | 提交后数据永久保存 | redo log（重做日志）+ 磁盘写入 | 磁盘损坏 |

**为什么 undo log 实现原子性？** 每个事务执行 UPDATE 时，MySQL 先将旧值记录到 undo log。如果事务执行 ROLLBACK，从 undo log 读取旧值并恢复即可。undo log 本身也是记录在 redo log 中的（双重保险）。

**为什么 redo log 实现持久性？** 事务提交时，先将"数据修改"写入 redo log（顺序写，速度快），再写磁盘（随机 I/O，速度慢）。事务提交 = redo log 刷盘成功。崩溃恢复时，MySQL 读取 redo log 重放未刷盘的操作，实现已提交事务不丢失。

```sql
-- 开启事务
START TRANSACTION;

-- 告警状态变更（原子性：两步要么都成功，要么都回滚）
UPDATE alerts SET status = 2, updated_at = NOW() WHERE id = 1;
UPDATE alert_audit_log SET operator_id = 100, action = '确认' WHERE alert_id = 1;

-- 提交事务
COMMIT;
-- 或回滚
-- ROLLBACK;
```

### 5.2 四种隔离级别

隔离级别通过控制"能看到其他事务的哪些修改"来平衡**并发性能**和**数据一致性**。标准 SQL 定义了四种隔离级别，MySQL 全都支持：

| 隔离级别 | 脏读 | 不可重复读 | 幻读 | 并发性能 | 说明 |
|---------|------|-----------|------|---------|------|
| READ UNCOMMITTED | 可能 | 可能 | 可能 | 最高 | 读取未提交的数据（读到别人的中间状态） |
| READ COMMITTED | 不可能 | 可能 | 可能 | 较高 | 只能读取已提交的数据 |
| REPEATABLE READ（MySQL 默认） | 不可能 | 不可能 | 可能* | 中等 | InnoDB 通过 MVCC + Gap Lock 解决了幻读 |
| SERIALIZABLE | 不可能 | 不可能 | 不可能 | 最低 | 强制事务串行执行 |

```
脏读（Dirty Read）：读取到其他事务未提交的数据（别人可能 ROLLBACK，数据是假的）
不可重复读（Non-repeatable Read）：同一事务内两次读取同一行，数据不一样（别人修改并提交了）
幻读（Phantom Read）：同一事务内两次查询结果集行数不一样（别人插入了新行）
```

**为什么 InnoDB 默认是 REPEATABLE READ？**

READ COMMITTED 在 Oracle、PostgreSQL 中是默认级别，适合大多数场景。MySQL 选择 REPEATABLE READ 作为默认值，有两个原因：
1. MySQL 的 binlog 格式在 STATEMENT 模式下，需要 REPEATABLE READ 才能保证主从复制一致性
2. InnoDB 的 Next-Key Lock 机制在 REPEATABLE READ 下防止幻读，兼顾了一致性和并发性

```sql
-- 查看当前会话隔离级别
SELECT @@transaction_isolation;
-- 结果：REPEATABLE-READ

-- 设置隔离级别（仅当前会话）
SET SESSION TRANSACTION ISOLATION LEVEL READ COMMITTED;

-- 设置隔离级别（全局，下次新连接生效）
SET GLOBAL transaction_isolation = 'READ-COMMITTED';
```

### 5.3 事务隔离级别实战

**场景：告警状态并发更新（SOC 平台常见场景）**

```sql
-- 终端A：查询告警状态（此时 status=1）
START TRANSACTION;
SELECT status FROM alerts WHERE id = 1;
-- 结果：status=1（未处理）

-- 终端B：同时将告警确认为已处理
START TRANSACTION;
UPDATE alerts SET status = 2 WHERE id = 1;
COMMIT;  -- 提交，status 变为 2

-- 终端A：再次查询告警状态，准备更新
SELECT status FROM alerts WHERE id = 1;
-- READ COMMITTED 下：结果=2（已提交的最新值）→ 可能覆盖 B 的更新
-- REPEATABLE READ 下：结果=1（事务开始时的快照）→ A 知道数据已变化

-- 终端A：执行状态变更
UPDATE alerts SET status = 3 WHERE id = 1;
COMMIT;
```

**⚠️ 注意：** 即使在 REPEATABLE READ 下，上述并发更新也可能出问题（"先查后写"不是原子操作）。正确做法是用 `SELECT ... FOR UPDATE` 显式加锁，或用乐观锁（版本号）：

```sql
-- 方案1：悲观锁（SELECT FOR UPDATE）
START TRANSACTION;
SELECT * FROM alerts WHERE id = 1 FOR UPDATE;  -- 锁定这行，其他人改不了
-- ... 业务逻辑 ...
UPDATE alerts SET status = 2 WHERE id = 1;
COMMIT;

-- 方案2：乐观锁（版本号）
UPDATE alerts
SET status = 2, version = version + 1
WHERE id = 1 AND version = 5;  -- 版本匹配才更新

-- 影响行数为 0 → 说明别人改过了，事务回滚重试
```

## 六、MVCC 原理

### 6.1 版本链结构

MVCC（Multi-Version Concurrency Control，多版本并发控制）的核心思想是：**每个事务看到的数据是一个"快照"，而不是当前最新值**。这使得读操作不阻塞写操作，写操作不阻塞读操作。

**每行数据的隐藏字段：**

InnoDB 给每一行数据添加了两个隐藏字段：

```sql
-- 逻辑上，每行数据有这些列（MySQL 内部管理，对用户透明）
DATA_TRX_ID    BIGINT UNSIGNED  -- 最近修改这条记录的事务 ID（6 bytes）
DATA_ROLL_PTR  VARBINARY(6)    -- 指向 undo log 记录的指针（7 bytes）
```

**版本链的形成过程：**

```
初始插入（trx_id=10）：
  [id=1, name='告警A', severity=3, trx_id=10, roll_ptr=null]

事务B 更新（trx_id=20）：
  旧版本进入 undo log：{name='告警A', severity=3, trx_id=10, roll_ptr=null}
  当前行变为：      [id=1, name='告警B', severity=4, trx_id=20, roll_ptr=→undo]

事务C 更新（trx_id=30）：
  旧版本进入 undo log：{name='告警B', severity=4, trx_id=20, roll_ptr=→undo}
  当前行变为：      [id=1, name='告警C', severity=2, trx_id=30, roll_ptr=→undo]
                         ↑ roll_ptr 形成链表（版本链）
```

**版本链从头到尾就是一行数据的"修改历史"**，每个版本都包含修改该版本的事务 ID。

### 6.2 ReadView 机制

ReadView（读视图）是 MVCC 的核心判断工具，决定当前事务能看到哪个版本的数据。

**ReadView 包含的信息（生成时机决定可见性）：**

```sql
ReadView {
    m_ids: [10, 20]      -- 当前活跃（未提交）事务的 ID 列表
    min_trx_id: 10        -- 活跃事务中的最小 ID
    max_trx_id: 31        -- 创建 ReadView 时分配的最大事务 ID（已分配但未提交的不算）
    creator_trx_id: 25    -- 当前事务自己的 ID
}
```

**可见性判断算法（重点！）：**

```
遍历版本链，从最新版本到最老版本：
1. 如果版本的 trx_id == 当前事务的 creator_trx_id → 可见（自己的修改）

2. 如果版本的 trx_id < min_trx_id
   → 可见（trx_id=9 的事务在 ReadView 创建前就提交了）

3. 如果版本的 trx_id >= max_trx_id
   → 不可见（trx_id=31 的事务在 ReadView 创建后才开始的）

4. 如果版本的 trx_id 在 m_ids 列表中
   → 不可见（trx_id=20 在活跃列表中，说明它还未提交）

5. 否则 → 继续沿 roll_ptr 遍历更老的版本
```

**REPEATABLE READ vs READ COMMITTED 的核心区别：**

| 隔离级别 | ReadView 生成时机 | 事务内多次读取一致性 |
|---------|------------------|-----------------|
| REPEATABLE READ | 事务**开始时**生成，只生成一次 | 每次读都用同一个 ReadView，结果一致 |
| READ COMMITTED | 每次读取都**重新生成** ReadView | 别人提交后，再次读取能看到新值 |

```sql
-- REPEATABLE READ 演示
-- 事务A（trx_id=25）开始
START TRANSACTION;

-- 事务A 生成了 ReadView：m_ids=[], min=10, max=31
SELECT * FROM alerts WHERE id = 1;
-- 读到的是 ReadView 创建时的最新可见版本

-- 事务B（trx_id=26）提交了新的告警
INSERT INTO alerts (tenant_id, alert_name, severity) VALUES (1001, '新告警', 3);
COMMIT;

-- 事务A 再次查询同一行
SELECT * FROM alerts WHERE id = 1;
-- 用的还是事务开始时生成的 ReadView，新插入的行不可见
-- ✅ 保证可重复读
```

### 6.3 快照读 vs 当前读

MVCC 只对"快照读"生效。MySQL 有两种读取方式：

| 读取方式 | SQL 示例 | 读到的数据 | 是否加锁 |
|---------|---------|-----------|---------|
| **快照读（Snapshot Read）** | SELECT ... | 历史版本（MVCC 快照） | 不加锁 |
| **当前读（Current Read）** | SELECT ... FOR UPDATE / INSERT / UPDATE / DELETE | 最新已提交的数据 | 加锁 |

```sql
-- 快照读：读取历史版本，不加锁（MVCC 的核心优势）
SELECT * FROM alerts WHERE id = 1;  -- 不阻塞并发写入

-- 当前读：读取最新数据，加锁（防止并发修改）
SELECT * FROM alerts WHERE id = 1 FOR UPDATE;  -- 锁定这一行

-- 当前读：UPDATE/DELETE 默认就是当前读
UPDATE alerts SET status = 2 WHERE id = 1;  -- 先加排他锁，再更新
DELETE FROM alerts WHERE id = 1;           -- 先加排他锁，再删除
```

**⚠️ 注意：** 即使在 REPEATABLE READ 下，快照读也可能出现幻读（读到别人新插入的行）。解决幻读需要 Next-Key Lock（下一节详述）。MVCC 解决的是"同一行数据的读写冲突"，Next-Key Lock 解决的是"同一范围内的插入冲突"。

## 七、锁机制

### 7.1 锁的分类

**按锁的粒度：**

| 粒度 | 引擎 | 特点 | 使用场景 |
|------|------|------|---------|
| **行锁（Row Lock）** | InnoDB | 锁一行数据，并发高 | 大多数 OLTP 场景 |
| **表锁（Table Lock）** | MyISAM / InnoDB | 锁整张表，并发低 | DDL 操作、MyISAM 表 |

**InnoDB 行锁的实现依赖于索引：** 如果 SQL 没有用到索引（或索引失效），行锁会"退化"为表锁，因为 InnoDB 必须在索引上加锁，无法确定具体行时只能锁所有行。

```sql
-- tenant_id 有索引，锁具体几行
BEGIN;
SELECT * FROM alerts WHERE tenant_id = 1001 FOR UPDATE;
-- 只锁定 tenant_id=1001 的行，其他 tenant_id 的行不受影响

-- alert_name 没有索引（或索引失效），锁整张表
BEGIN;
SELECT * FROM alerts WHERE alert_name LIKE '%暴力%' FOR UPDATE;
-- 锁住整张表，所有 INSERT/UPDATE/DELETE 都阻塞 → ⚠️ 生产级故障

-- 验证：查看锁信息
SHOW ENGINE INNODB STATUS;
-- LATEST DETECTED DEADLOCK 部分显示死锁信息
-- TRANSACTIONS 部分显示当前锁等待
```

**按锁的模式：**

| 锁类型 | 简称 | 兼容性 | 用途 |
|--------|------|--------|------|
| **共享锁（Shared Lock）** | S 锁 | S 与 S 兼容，S 与 X 互斥 | SELECT ... LOCK IN SHARE MODE |
| **排他锁（Exclusive Lock）** | X 锁 | X 与任何锁都互斥 | SELECT ... FOR UPDATE / UPDATE / DELETE |

```sql
-- 共享锁：允许多个事务同时持有，但不能和排他锁共存
START TRANSACTION;
SELECT * FROM alerts WHERE id = 1 LOCK IN SHARE MODE;  -- 加 S 锁
-- 另一个事务也可以：SELECT * FROM alerts WHERE id = 1 LOCK IN SHARE MODE; ✅
-- 但无法：UPDATE alerts SET status = 2 WHERE id = 1; ❌（需要 X 锁）

-- 排他锁：只能有一个事务持有
START TRANSACTION;
SELECT * FROM alerts WHERE id = 1 FOR UPDATE;  -- 加 X 锁
-- 其他任何事务的读/写都阻塞
```

### 7.2 间隙锁与临键锁

**Gap Lock（间隙锁）：** 锁定索引记录之间的间隙，防止其他事务在间隙中插入新记录，解决幻读问题。

**Next-Key Lock（临键锁）：** Gap Lock + Record Lock 的组合，既锁定索引记录本身，也锁定其前面的间隙。

```sql
-- 假设 alerts 表当前有 id = [1, 5, 10, 15, 20]
-- 执行：SELECT * FROM alerts WHERE id = 8 FOR UPDATE;

-- 锁定范围：
--   Record Lock：锁定 id=5（8 前面最近的记录）
--   Gap Lock：锁定 (5, 10) 的间隙（阻止在 5 和 10 之间插入新 id）
--   Next-Key Lock = [5, 10)（锁定左闭右开的范围）

-- 效果：其他事务无法在 id ∈ (5, 10) 范围内插入新行（防止幻读）
-- 但 id=5 和 id=10 本身可能被其他事务修改（行锁）

-- 按范围查询也会有间隙锁
SELECT * FROM alerts WHERE id > 8 AND id < 12 FOR UPDATE;
-- Gap Lock: (5,10) + (10,15) + (15,20) = (5,20) 锁定这段间隙
```

**为什么间隙锁只锁定"间隙"而不是锁住具体行？** 因为 id=8 这一行根本不存在（只有 id=5 和 id=10），InnoDB 无法锁定一个不存在的记录，退而求其次锁定两者之间的间隙，防止其他事务在这个间隙里插入新行。

### 7.3 死锁：成因与处理

**死锁（Dead Lock）** 是两个或多个事务相互持有对方需要的锁，形成循环等待。

```
死锁形成过程：

事务A：                    事务B：
START TRANSACTION;         START TRANSACTION;
UPDATE alerts SET status=2 WHERE id=1;  -- 锁定 id=1
                           UPDATE alerts SET status=2 WHERE id=2;  -- 锁定 id=2
UPDATE alerts SET status=3 WHERE id=2;  -- 等 id=2 的锁 ❌
                           UPDATE alerts SET status=3 WHERE id=1;  -- 等 id=1 的锁 ❌
                           -- 死锁！事务B 被 InnoDB 检测到
```

**InnoDB 的死锁处理：**

InnoDB 默认启用**死锁检测**（`innodb_deadlock_detect=on`）。检测到死锁后，InnoDB 会自动回滚"代价最小的事务"（通常是最小的事务，即插入数据最少的事务）。

```sql
-- 死锁发生时的错误信息：
-- ERROR 1213 (40001): Deadlock found when trying to get lock;
-- try restarting transaction

-- 处理方法：应用层捕获异常，重试事务
-- 业务代码示例（伪代码）：
-- try {
--     updateAlert(id);
-- } catch (DeadlockException e) {
--     sleep(100);  // 等待对方事务释放锁
--     updateAlert(id);  // 重试
-- }
```

**如何避免死锁？**

| 策略 | 说明 | 示例 |
|------|------|------|
| **固定顺序访问资源** | 所有事务按同一顺序访问表/行 | 先改告警表再改日志表 |
| **减少锁持有时间** | 大事务拆小，减少锁冲突窗口 | 批量更新改为分批 |
| **SELECT FOR UPDATE 先锁定** | 先读后写的场景用 FOR UPDATE | 避免先快照读后写 |
| **降低隔离级别** | READ COMMITTED 下 Gap Lock 范围更小 | 需要权衡一致性 |
| **加锁超时** | `innodb_lock_wait_timeout=5` | 快速失败，避免长时间等待 |

### 7.4 两阶段锁协议

**两阶段锁协议（Two-Phase Locking, 2PL）：** InnoDB 遵循此协议，事务的锁操作分为两个阶段：

```
加锁阶段：只能加锁，不能解锁
解锁阶段：只能解锁，不能加锁
```

```sql
-- 两阶段锁的体现：
START TRANSACTION;

UPDATE alerts SET status = 2 WHERE id = 1;  -- 加锁阶段：加 X 锁
UPDATE alerts SET status = 3 WHERE id = 2;  -- 加锁阶段：再加 X 锁
-- ... 更多 SQL ...

COMMIT;  -- 解锁阶段：提交时一次性释放所有锁
-- 或 ROLLBACK;  -- 解锁阶段：回滚时一次性释放所有锁
```

**为什么两阶段锁协议很重要？** 它保证了调度的可串行化——无论事务如何交叉执行，最终结果等价于某个串行顺序。对于 SOC 平台的高并发告警处理，这意味着数据一致性是可控的。

## 八、SQL 优化

### 8.1 EXPLAIN 深度解析

EXPLAIN 是 MySQL SQL 优化的起点，必须掌握每一个输出字段的含义：

```sql
EXPLAIN SELECT
    a.id, a.alert_name, a.severity, a.status,
    t.tenant_name
FROM alerts a
INNER JOIN tenants t ON a.tenant_id = t.id
WHERE a.tenant_id = 1001
  AND a.severity >= 3
ORDER BY a.created_at DESC
LIMIT 20;
```

**EXPLAIN 输出字段详解：**

| 字段 | 含义 | 好值 | 坏值 |
|------|------|------|------|
| **id** | 查询中 SELECT 的序号，id 大的先执行 | - | - |
| **select_type** | 查询类型 | SIMPLE, DERIVED | SUBQUERY, UNCACHEABLE |
| **type** | 关联类型（最重要） | const > eq_ref > ref > range > index > ALL | ALL（必须优化！） |
| **possible_keys** | 可用的索引列表 | - | - |
| **key** | 实际使用的索引 | 应在 possible_keys 中 | NULL（没走索引！） |
| **key_len** | 索引使用长度（字节） | 越接近索引总长度越好 | - |
| **rows** | 估算扫描行数 | 越少越好 | 巨大数字（需优化） |
| **Extra** | 附加信息 | Using index, Using where | Using filesort, Using temporary |

**type 字段详解（按性能从好到差排序）：**

```
const    ✅ 最优：主键或唯一索引的常量查找（只扫描 1 行）
         SELECT * FROM alerts WHERE id = 1;  -- id 是主键

eq_ref   ✅ 极好：JOIN 时，被驱动表的唯一索引等值查找
         SELECT * FROM alerts a JOIN tenants t ON a.tenant_id = t.id
         -- t.id 是主键，每个 t.id 只匹配一行

ref      ✅ 好：普通索引的等值查找（非唯一索引）
         SELECT * FROM alerts WHERE tenant_id = 1001;
         -- tenant_id 是普通索引，匹配多行

range    ✅ 可接受：索引范围扫描（>, <, >=, <=, BETWEEN, IN）
         SELECT * FROM alerts WHERE id > 1000 AND id < 5000;

index    ⚠️ 一般：全索引扫描（扫描整个索引树，比 ALL 快但不回表）
         SELECT id FROM alerts;  -- 只查主键，直接扫主键索引

ALL      ❌ 最差：全表扫描（必须优化！）
         SELECT * FROM alerts WHERE alert_name LIKE '%暴力%';
```

**Extra 字段最常见的几种情况：**

```
Using index     ✅ 覆盖索引，无需回表
Using where     ⚠️ 需要在存储引擎层过滤数据（常见，正常）
Using filesort  ❌ 需要额外排序操作（ORDER BY 字段无索引）
Using temporary ❌ 需要创建临时表（GROUP BY / DISTINCT 无索引）
Using index condition (ICP)  ✅ 索引下推，减少回表次数（MySQL 5.6+）
Using join buffer (Block Nested Loop) ❌ JOIN 时无索引，用了 BNLL 算法
```

### 8.2 慢查询日志

```sql
-- 开启慢查询日志
SET GLOBAL slow_query_log = 'ON';
SET GLOBAL slow_query_log_file = '/var/log/mysql/slow.log';
SET GLOBAL long_query_time = 1;    -- 超过 1 秒的查询记录到慢日志

-- 查看慢查询日志配置

-- 查看当前慢查询数量
SHOW GLOBAL STATUS LIKE 'Slow_queries';
```

**慢查询日志格式示例：**

```sql
-- 慢查询示例（实际日志文件内容）
# Time: 2026-04-10T03:00:15.123456Z
# User@Host: app_user[app_user] @ 192.168.1.100 [192.168.1.100]
# Query_time: 3.245132  Lock_time: 0.000089  Rows_sent: 20  Rows_examined: 1548923
SET timestamp=1744248015;
SELECT a.id, a.alert_name, a.severity, t.tenant_name
FROM alerts a
INNER JOIN tenants t ON a.tenant_id = t.id
WHERE a.tenant_id = 1001 AND a.severity >= 3
ORDER BY a.created_at DESC LIMIT 20;
```

**关键指标解读：**
- `Query_time: 3.24` — 查询耗时 3.24 秒（差！）
- `Rows_examined: 1548923` — 扫描了 154 万行（差！）
- `Rows_sent: 20` — 只返回 20 行

**优化思路：** `Rows_examined / Rows_sent = 77000`，扫描/返回比极高，说明没有有效利用索引。需要加联合索引 `(tenant_id, severity, created_at)` 覆盖这个查询。

### 8.3 分页优化

**LIMIT offset, n 的深度分页陷阱：**

```sql
-- 反例：查询第 10000 页，每页 20 条
SELECT * FROM alerts
WHERE tenant_id = 1001
ORDER BY created_at DESC
LIMIT 1000000, 20;
-- 问题：MySQL 需要先读取并排序前 1,000,020 条数据，然后丢弃前 1,000,000 条
-- 在 1000 万行表中，这可能需要 3-5 秒

-- ✅ 方案1：延迟关联（利用覆盖索引快速定位 ID）
SELECT t.*
FROM alerts t
INNER JOIN (
    SELECT id FROM alerts
    WHERE tenant_id = 1001
    ORDER BY created_at DESC
    LIMIT 1000000, 20  -- 子查询只扫描主键索引，不需要排序和回表
) AS r ON t.id = r.id;

-- ✅ 方案2：游标分页（基于上一页最大 ID，推荐！）
-- 第一页
SELECT * FROM alerts
WHERE tenant_id = 1001
ORDER BY id DESC LIMIT 20;
-- 最后一行的 id = 50000

-- 第二页（下一页）
SELECT * FROM alerts
WHERE tenant_id = 1001 AND id < 50000  -- 利用主键索引的顺序
ORDER BY id DESC LIMIT 20;
-- 无论翻到多少页，性能恒定 O(1) 级别
```

**⚠️ 注意：** 游标分页的前提是**排序字段单调递增**（ID 或时间戳）。如果用户需要跳页（不按顺序翻页），只能用延迟关联+全排序，无法避免深度 offset 的问题。这是产品设计时需要考虑的技术约束。

### 8.4 JOIN 优化

**小表驱动大表原则：** MySQL 总是从右表取数据去左表查找，所以**右表应该越小越好**（小结果集驱动大表）。

```sql
-- 反例：orders 表 1000 万行，customers 表 10 万行
SELECT * FROM orders o
INNER JOIN customers c ON o.customer_id = c.id
WHERE c.region = '华北';

-- 如果用 LEFT JOIN，优化器可能选错了驱动表
-- 明确指定：让小表 customers 先过滤，再去大表 orders 匹配
SELECT * FROM customers c
INNER JOIN orders o ON c.id = o.customer_id
WHERE c.region = '华北';
-- c.region='华北' 可能过滤到 1 万行（总 10 万的 10%）
-- 再用这 1 万行去 orders 匹配，比直接扫 1000 万行快 1000 倍
```

**JOIN 的索引策略：**

```sql
-- JOIN 优化黄金法则：
-- 1. ON 条件的字段必须有索引
-- 2. 被驱动表（内层循环的表）的 JOIN 字段必须加索引

-- 正确的索引设计：
CREATE TABLE orders (
    id BIGINT PRIMARY KEY,
    customer_id BIGINT NOT NULL,    -- 加索引！
    product_id BIGINT NOT NULL,
    INDEX idx_customer (customer_id)
);

CREATE TABLE customers (
    id BIGINT PRIMARY KEY,
    region VARCHAR(50)
);

-- JOIN 字段 customer_id 有索引 → NLJ（Nested Loop Join）高效
-- 无索引 → BNLL（Block Nested Loop Join）大表做内层循环 → 极慢
```

## 九、主从复制

### 9.1 复制原理

MySQL 主从复制是基于 **binlog（二进制日志）** 实现的异步复制，核心流程分为三步：

```
主库：                                    从库：
┌────────────────────┐                  ┌────────────────────┐
│ 1. DML 操作          │                  │                     │
│    (UPDATE alerts)  │                  │                     │
│         ↓            │                  │                     │
│ 2. 写入 redo log     │                  │                     │
│         ↓            │                  │                     │
│ 3. 提交事务          │                  │                     │
│         ↓            │                  │                     │
│ 4. 写入 binlog      │ ──binlog──→  │ 5. IO 线程接收        │
│    (binlog dump)    │    (网络)     │    写入 relay-log    │
│                    │                  │         ↓            │
│                    │                  │ 6. SQL 线程重放      │
│                    │                  │    (relay-log)       │
│                    │                  │         ↓            │
└────────────────────┘                  │ 7. 数据落地          │
                                         └────────────────────┘
```

```sql
-- 主库配置（my.cnf）
[mysqld]
server-id = 1                      -- 必须唯一
log_bin = /var/log/mysql/binlog   -- 开启 binlog
binlog_format = ROW                -- 推荐 ROW 格式
sync_binlog = 1                    -- 每条事务同步刷盘（性能略降，安全性最高）

-- 从库配置（my.cnf）
[mysqld]
server-id = 2                      -- 必须唯一
relay_log = /var/log/mysql/relay-bin  -- 中继日志
read_only = ON                     -- 从库只读（注意：root 仍可写）
super_read_only = ON               -- MySQL 8.0+，所有用户都只读

-- 从库执行复制命令（传统方式）
CHANGE REPLICATION TO
  MASTER_HOST = '192.168.1.100',
  MASTER_USER = 'repl_user',
  MASTER_PASSWORD = 'password',
  MASTER_LOG_FILE = 'binlog.000001',
  MASTER_LOG_POS = 1234;
START SLAVE;

-- 从库查看复制状态
SHOW SLAVE STATUS\G
-- 关键字段：Slave_IO_Running, Slave_SQL_Running, Seconds_Behind_Master, Relay_Log_Pos
```

### 9.2 binlog 三种格式

binlog 记录的是数据变更的"日志"，有三种格式，各有优劣：

| 格式 | 记录内容 | 优点 | 缺点 |
|------|---------|------|------|
| **STATEMENT** | SQL 语句本身 | 日志量小，可审计执行的 SQL | NOW() 等函数主从不一致 |
| **ROW** | 每行的具体变化（前后值） | 绝对一致，适合数据修复 | 日志量大（可达 10 倍） |
| **MIXED** | 自动切换（默认 STATEMENT，不安全时切 ROW） | 平衡 | 不够透明，难以预测 |

```sql
-- 查看当前 binlog 格式
SHOW VARIABLES LIKE 'binlog_format';
-- 结果：ROW

-- ROW 格式下，查看 binlog 内容（MySQL 8.0+ mysqlbinlog）
-- shell 命令：
-- mysqlbinlog --base64-output=decode-rows -v /var/log/mysql/binlog.000001
```

**为什么推荐 ROW 格式？** 考虑这个场景：

```sql
-- 主库执行（STATEMENT 模式下）：
UPDATE accounts SET balance = balance - 100 WHERE id = 1;
-- 主库：balance 从 1000 减到 900

-- 从库执行同一 SQL（但余额已经不同了）：
UPDATE accounts SET balance = balance - 100 WHERE id = 1;
-- 从库：balance 从 900 减到 800  ← 数据不一致！
```

ROW 格式记录的是行级别的具体值变化，避免了函数依赖和先后顺序问题，是生产环境的推荐选择。

### 9.3 异步复制 vs 半同步复制

| 复制模式 | 主库等待从库确认 | 数据安全性 | 性能影响 | RPO |
|---------|---------------|-----------|---------|-----|
| **异步复制** | 不等待，从库不确认即提交 | 低（主库崩溃可能丢失数据） | 最低 | 可能丢失多个事务 |
| **半同步复制** | 等待至少一个从库确认收到 binlog | 中等 | 略低（至少等一次网络 RTT） | < 1 秒事务数据 |
| **全同步复制** | 等待所有从库都确认 | 最高 | 极低（等最慢的从库） | 0（零丢失） |

```sql
-- 安装半同步复制插件（主库和从库都需要）
INSTALL PLUGIN rpl_semi_sync_master SONAME 'semisync_master.so';
INSTALL PLUGIN rpl_semi_sync_slave SONAME 'semisync_slave.so';

-- 主库启用半同步
SET GLOBAL rpl_semi_sync_master_enabled = ON;
SET GLOBAL rpl_semi_sync_master_timeout = 1000;  -- 等待 1 秒，超时自动降级为异步

-- 从库启用半同步
SET GLOBAL rpl_semi_sync_slave_enabled = ON;
STOP SLAVE;
START SLAVE;

-- 查看半同步状态
SHOW STATUS LIKE 'Rpl_semi_sync%';
-- Rpl_semi_sync_master_status = ON（主库半同步正常）
-- Rpl_semi_sync_master_no_tx = 0（无超时等待的事务数）
```

**⚠️ 注意：** 半同步复制的"等待"时间是主库线程的阻塞时间。如果从库网络故障或延迟过大，主库会等待超时（默认 10 秒），然后降级为异步复制，恢复性能。在这段时间内主库写入会变慢，这是性能和安全性的权衡。

### 9.4 GTID 复制

GTID（Global Transaction Identifier）是一种更现代的复制方式，用"事务 ID"代替传统的"文件+位置"来定位复制进度。

```sql
-- GTID 格式：server_uuid:transaction_sequence_number
-- 示例：3E11FA47-71CA-11E1-9E33-C80AA9429562:23
--        ↑ 服务器唯一ID              ↑ 第23个事务

-- GTID 配置（主库）
[mysqld]
gtid_mode = ON
enforce_gtid_consistency = ON
log_slave_updates = ON  -- 从库也要记录 binlog（GTID 复制必须开启）

-- GTID 配置（从库）
[mysqld]
gtid_mode = ON
enforce_gtid_consistency = ON
log_slave_updates = ON

-- GTID 模式CHANGE REPLICATION TO（比传统方式简单得多）
CHANGE REPLICATION TO
  MASTER_HOST = '192.168.1.100',
  MASTER_USER = 'repl_user',
  MASTER_PASSWORD = 'password',
  MASTER_AUTO_POSITION = 1;  -- 自动找断点，无需指定 binlog 文件和位置！
START SLAVE;

-- 查看 GTID 复制状态
SHOW SLAVE STATUS\G
-- Executed_Gtid_Set: 3E11FA47-71CA-11E1-9E33-C80AA9429562:1-158
-- 自动显示已执行到哪个 GTID，无需计算文件位置
```

**GTID 的优势对比：**

```
传统复制（文件+位置）：
  故障切换后，需要人工计算：
    "主库崩了，最后一个 binlog 是 binlog.000123，位置是 4567"
  新主库必须精确到 binlog.000123:4567，多一个字节或少一个字节都不行

GTID 复制：
  故障切换后，自动找断点：
    "主库崩了，已执行的事务：3E11FA47:1-158，新主库从 159 开始"
  任何从库都可以精确接续，无需人工对齐
```

## 十、分库分表

### 10.1 何时需要分库分表

分库分表是数据库架构的"核武器"，使用前必须确认真的需要。常见的信号：

| 信号 | 说明 | 阈值（参考） |
|------|------|------------|
| 单表行数过多 | B+Tree 层级增加，索引效率下降 | > 2000 万行 |
| 单表数据量过大 | 磁盘占用大，DDL 操作（如加索引）极慢 | > 100 GB |
| QPS 过高 | 单机 MySQL 连接数上限（通常 2000-4000） | > 5000 QPS |
| 磁盘 I/O 瓶颈 | SSD 也扛不住随机读写 | > 5000 IOPS |

**⚠️ 注意：** 分库分表会引入分布式复杂度（跨分片查询、分布式事务、分布式 ID），是最后的手段。在考虑分库分表之前，先尝试这些优化手段：
1. 加索引（解决 80% 的慢查询问题）
2. 读写分离（读流量分摊）
3. 冷热数据分离（历史数据归档到另一张表/库）
4. 表结构优化（去掉 TEXT/BLOB 大字段，用对象存储）
5. 配置优化（buffer pool、连接池）

### 10.2 垂直分 vs 水平分

| 方式 | 策略 | 适用场景 | 复杂度 |
|------|------|---------|--------|
| **垂直分库** | 按业务模块拆到不同数据库 | 微服务架构，各服务独立数据库 | 中等 |
| **垂直分表** | 按列拆到不同表（大字段拆出） | 宽表，频繁查询字段少 | 低 |
| **水平分库** | 按行拆到不同数据库 | 单库 QPS 过高 | 高 |
| **水平分表** | 按行拆到同一库的不同表 | 单表行数过多 | 中等 |

```sql
-- 垂直分表示例：告警表拆分
-- 原始宽表（热点字段和冷字段混在一起）
CREATE TABLE alerts (
    id BIGINT PRIMARY KEY,
    tenant_id BIGINT,
    alert_name VARCHAR(255),      -- 热点：频繁查询
    severity TINYINT,             -- 热点：频繁查询
    status TINYINT,                -- 热点：频繁查询
    raw_log TEXT,                 -- 冷数据：详情页才需要
    attack_signature BLOB,         -- 冷数据：分析时才需要
    full_pcap LONGBLOB,            -- 冷数据：取证才需要
    created_at DATETIME
);

-- 垂直拆分为 alerts（热点字段）和 alerts_detail（冷数据）
CREATE TABLE alerts (
    id BIGINT PRIMARY KEY,
    tenant_id BIGINT,
    alert_name VARCHAR(255),
    severity TINYINT,
    status TINYINT,
    created_at DATETIME,
    INDEX idx_tenant (tenant_id)
);

CREATE TABLE alerts_detail (
    alert_id BIGINT PRIMARY KEY,   -- 主键就是告警 ID
    raw_log TEXT,
    attack_signature BLOB,
    full_pcap LONGBLOB
);
-- alerts 表行数少，热点查询极快；alerts_detail 按需加载
```

### 10.3 ShardingSphere 实战

ShardingSphere 是 Apache 基金会的顶级项目，是分库分表的主流中间件：

```
ShardingSphere-JDBC：
  应用层（Java/Python/Go）          MySQL 实例 1 (sharding_0)
       ↓                              MySQL 实例 2 (sharding_1)
  引入 JAR，SQL 路由/改写/执行       MySQL 实例 3 (sharding_2)
       ↓                              ...
  应用直接连数据库

ShardingSphere-Proxy：
       应用层                          MySQL 实例 1 (sharding_0)
       ↓                               MySQL 实例 2 (sharding_1)
  连接 ShardingSphere-Proxy            MySQL 实例 3 (sharding_2)
       ↓（MySQL 协议）                    ...
  透明分片，对应用无感知
```

**ShardingSphere-JDBC 配置示例（告警表按 tenant_id 分 4 张表）：**

```yaml
# application.yml
spring:
  shardingsphere:
    datasource:
      ds_0:
        driver-class-name: com.mysql.cj.jdbc.Driver
        url: jdbc:mysql://192.168.1.10:3306/security_db_0
        username: root
        password: password
      ds_1:
        url: jdbc:mysql://192.168.1.11:3306/security_db_1
        username: root
        password: password

    sharding:
      tables:
        # 按 tenant_id % 2 路由到 ds_0 或 ds_1
        # 按 (tenant_id / 2) % 2 路由到 alerts_0 到 alerts_3
        alerts:
          actual-data-nodes: ds_${0..1}.alerts_${0..3}
          table-strategy:
            standard:
              sharding-column: tenant_id
              sharding-algorithm-name: alerts_mod
          key-generator:
            column: id
            type: SNOWFLAKE  # 雪花算法分布式 ID

      sharding-algorithms:
        alerts_mod:
          type: INLINE
          props:
            algorithm-expression: |
              ds_${Long.hashCode(tenant_id) % 2}
              .alerts_${(tenant_id / 2) % 4}
```

### 10.4 跨分片查询策略

分库分表后，最棘手的问题是跨分片查询：

```sql
-- 场景1：同分片键的查询（能路由到单一分片，最优）
SELECT * FROM alerts
WHERE tenant_id = 1001        -- 直接路由到 ds_${1001%2}.alerts_${(1001/2)%4}
ORDER BY created_at DESC
LIMIT 20;

-- 场景2：聚合统计（跨所有分片）
-- 反例：直接 COUNT(*)
SELECT COUNT(*) FROM alerts WHERE tenant_id = 1001;
-- ShardingSphere 会广播到 4 个分片，合并结果

-- 正例：使用 ShardingSphere 的 STREAM 模式
SELECT SUM(severity), AVG(severity)
FROM alerts
WHERE tenant_id = 1001;
-- 注意：跨分片的 GROUP BY + ORDER BY 无法在 SQL 层完成，
-- 需要借助 ES/Hive 做离线聚合

-- 场景3：跨分片 JOIN（同分片键才可能本地 JOIN）
SELECT a.*, t.tenant_name
FROM alerts a
INNER JOIN tenants t ON a.tenant_id = t.id
WHERE a.tenant_id = 1001;
-- 如果 tenants 表按 tenant_id 广播（冗余存储），可以本地 JOIN
-- 如果 tenants 表是单表，则需要 Scatter-Gather 模式

-- 场景4：跨分片分页
-- 限制：无法在数据库层做精确的分页计数（COUNT 总数）
-- 方案：使用 ElasticSearch 做搜索和分页，MySQL Sharding 只做存储
```

**分布式 ID 生成方案对比：**

| 方案 | 实现 | 优点 | 缺点 |
|------|------|------|------|
| **雪花算法（Snowflake）** | 时间戳+机器ID+序列号 | 高性能，有序 | 依赖时钟（机器时钟回拨会重复） |
| **UUID** | 128 位随机数 | 简单，不依赖中心 | 无序（页分裂），长度大 |
| **数据库号段** | 批量从数据库拿 ID 区间 | 简单，有序 | 依赖数据库，有可用性风险 |
| **ZooKeeper/Curator** | 中心化序列生成器 | 有序，可控 | 单点，有性能瓶颈 |

```sql
-- 雪花算法生成的 ID 示例：
-- 41位时间戳(毫秒) + 10位机器ID(5机房+5机器) + 12位序列号
-- 2026-04-10 09:00:00 的时间戳 ≈ 1744262400000
-- 生成的 ID：1765083429412946944（64位，bigint）
-- 按时间有序 → 插入 B+Tree 顺序写入 → 性能最优
```

## 附录：MySQL 8.0 新特性速查

| 特性 | 说明 | 面试加分点 |
|------|------|-----------|
| **窗口函数（Window Functions）** | `ROW_NUMBER()`, `RANK()`, `SUM() OVER()` | 替代复杂自 JOIN，简化 SQL |
| **CTE（公用表表达式）** | `WITH t AS (...)` | 递归查询，SQL 可读性大增 |
| **函数索引** | `INDEX idx((YEAR(date)))` | 解决函数索引失效问题 |
| **JSON Table** | JSON 转表结构查询 | 半结构化数据存储 |
| **隐藏列（Invisible Columns）** | 列对应用隐藏 | 内部计算列，不暴露给应用 |
| **角色（Roles）** | `CREATE ROLE r1; GRANT r1 TO user1` | 权限管理更清晰 |
| **Instant ADD COLUMN** | `ALTER TABLE ADD COLUMN` 秒级完成 | 大表加列不再需要 PT-osc |

```sql
-- MySQL 8.0 窗口函数示例：告警排名
SELECT
    tenant_id,
    alert_name,
    severity,
    RANK() OVER (
        PARTITION BY tenant_id
        ORDER BY severity DESC, created_at DESC
    ) AS severity_rank,
    SUM(severity) OVER (
        PARTITION BY tenant_id
        ORDER BY created_at
        ROWS BETWEEN 6 PRECEDING AND CURRENT ROW
    ) AS severity_sum_7d  -- 过去7天的告警严重度累计
FROM alerts;

-- MySQL 8.0 CTE 示例：递归查询组织架构
WITH RECURSIVE org_tree AS (
    -- 根节点
    SELECT id, name, parent_id, 1 AS level
    FROM departments WHERE parent_id IS NULL
    UNION ALL
    -- 递归：子节点
    SELECT d.id, d.name, d.parent_id, t.level + 1
    FROM departments d
    INNER JOIN org_tree t ON d.parent_id = t.id
)
SELECT * FROM org_tree;
```
