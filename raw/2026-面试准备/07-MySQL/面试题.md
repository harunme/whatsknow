### 索引原理与类型
```
MySQL 索引是怎么工作的？B+Tree 和 B-Tree 的区别是什么？聚簇索引和非聚簇索引各自的优缺点？覆盖索引是什么？
追问：
- 联合索引的最左前缀匹配原则是什么？
- 什么情况下索引会失效？
```
**[10秒结论]** MySQL 索引基于 B+Tree（所有数据在叶子节点且链表连接）实现 O(log n) 查找；聚簇索引叶子节点存储完整行数据，非聚簇索引叶子节点存主键值；覆盖索引用索引直接返回查询字段，无需回表。

**[90秒拆解]**
- **B+Tree vs B-Tree**：B+Tree 非叶子节点只存索引键（不含数据），相同深度能存更多索引；叶子节点链表连接，范围查询只需遍历链表；所有数据在叶子节点，查询稳定
- **聚簇索引**：主键索引即为聚簇索引，叶子节点存储完整行数据（数据和索引在一起）；每张表只有一个聚簇索引
- **非聚簇索引**：二级索引叶子节点存主键值（InnoDB），查询时先查二级索引得主键，再回表查聚簇索引（两次树查找）
- **覆盖索引**：查询字段全部在索引中，无需回表：
  ```sql
  SELECT name, age FROM users WHERE name = 'Tom';
  -- 若联合索引 (name, age) 存在，直接返回，无需回表
  ```
- **最左前缀匹配**：联合索引 (a, b, c) 能命中 a、(a,b)、(a,b,c) 的查询，不能命中 (b,c) 或 c 单独查询

**[20秒收尾]** SOC 平台的告警记录表用联合索引 (tenant_id, created_at, status) 覆盖了 90% 的查询场景，避免回表，查询延迟降低 60%。

### 事务与隔离级别
```
MySQL 的四种隔离级别分别是什么？脏读、不可重复读、幻读各自是什么？MVCC 是怎么工作的？readView 是什么？
追问：
- InnoDB 默认隔离级别是什么？
- MVCC 能完全解决幻读吗？
```
**[10秒结论]** MySQL 四种隔离级别：读未提交、读已提交、可重复读（InnoDB 默认）、串行化；MVCC 通过 readView 控制版本链可见性解决脏读和不可重复读，Next-Key Lock 解决幻读。

**[90秒拆解]**
- **四种隔离级别**：
  - READ UNCOMMITTED：最低，允许脏读
  - READ COMMITTED：每次读取生成新的 readView，解决脏读
  - REPEATABLE READ（InnoDB 默认）：事务开始时生成 readView，事务内一致，解决不可重复读
  - SERIALIZABLE：强制串行，读加共享锁，写加排他锁
- **MVCC 工作原理**：
  - 每行数据有隐式主键和两个版本字段（DATA_TRX_ID、DATA_ROLL_PTR）
  - update 时生成新版本，旧版本通过 roll_pointer 形成版本链
  - readView 包含活跃事务 IDs 列表，最小/最大事务 ID
  - 可见性判断：版本链中从头遍历，找到第一个 trx_id < readView.min_id 的版本可见
- **Next-Key Lock**：Record Lock（行锁）+ Gap Lock（间隙锁），锁定索引记录及其前面的间隙，防止幻读（REPEATABLE READ 下保证）

**[20秒收尾]** REPEATABLE READ 下，MVCC + Next-Key Lock 组合解决幻读，InnoDB 的隔离级别保证强一致性，是互联网公司主流选择。

### 锁机制
```
InnoDB 的行锁是怎么实现的？什么情况下会退化为表锁？间隙锁（Gap Lock）和临键锁（Next-Key Lock）是什么？死锁是怎么产生的？怎么避免？
追问：
- 什么是两阶段锁协议？
- SELECT ... FOR UPDATE 会加什么锁？
```
**[10秒结论]** InnoDB 行锁基于索引实现，锁加在索引记录上；非索引字段查询会退化为表锁；间隙锁锁定索引间隙，临键锁是 record lock + gap lock 组合；死锁由互相等待环产生，用超时或死锁检测解决。

**[90秒拆解]**
- **行锁退化表锁**：查询条件无索引或索引失效时，InnoDB 会锁定所有扫描到的行（临键锁），效果等同于表锁
- **间隙锁与临键锁**：
  - 索引有 1, 5, 10 三个值，UPDATE WHERE id > 3 AND id < 9：锁定 (5,10) 的间隙和 (5) 记录本身
  - Next-Key Lock = Record Lock（5）+ Gap Lock（5,10）
- **死锁产生**：事务 A 锁住 X1 等 X2，事务 B 锁住 X2 等 X1。InnoDB 默认超时（innodb_lock_wait_timeout）回滚代价最小的事务
- **避免死锁**：
  - 按固定顺序访问表
  - 减少锁持有时间（大 SQL 拆小）
  - 降低隔离级别
  - SELECT ... FOR UPDATE 加排他锁，避免先读后写
- **两阶段锁协议**：加锁阶段只能加锁，解锁阶段只能解锁。InnoDB 遵循此协议，锁在事务提交/回滚时才释放

**[20秒收尾]** 告警记录写操作用 SELECT FOR UPDATE 先锁定（避免并发写入冲突），分析操作用 MVCC 快照读（不加锁），两者组合实现高并发 + 数据一致性。

### SQL 优化
```
慢查询怎么排查？怎么用 EXPLAIN 分析 SQL 执行计划？Using filesort、Using temporary 意味着什么？怎么优化分页查询（LIMIT offset, n）？
追问：
- COUNT(*) 和 COUNT(1) 哪个更快？
- 为什么要避免 SELECT *？
```
**[10秒结论]** 慢查询用 slow_query_log + explain 分析执行计划；Using filesort（需额外排序）、Using temporary（需临时表）都是优化信号；大 LIMIT 分页用延迟关联或游标分页代替。

**[90秒拆解]**
- **慢查询排查**：
  ```sql
  SET GLOBAL slow_query_log = 'ON';
  SET GLOBAL long_query_time = 1;
  -- 查看慢查询日志或 performance_schema
  ```
- **EXPLAIN 关键字段**：
  - type：访问类型（const > eq_ref > ref > range > index > ALL），ref 以上才算好
  - key：实际使用的索引
  - rows：扫描行数，越少越好
  - Extra：Using index（覆盖索引）、Using filesort（需排序）、Using temporary（需临时表）
- **LIMIT 分页优化**：
  - 反例：`LIMIT 100000, 20` — 需扫描前 100020 行
  - 正例 1（延迟关联）：`SELECT t.* FROM orders t INNER JOIN (SELECT id FROM orders WHERE user_id = 1 ORDER BY id LIMIT 100000, 20) r ON t.id = r.id`
  - 正例 2（游标分页）：`WHERE id > last_id ORDER BY id LIMIT 20`
- **COUNT(*) vs COUNT(1)**：InnoDB 中两者性能相同，MySQL 优化器自动处理；COUNT(column) 需要判断是否为 NULL，比 COUNT(*) 略慢

**[20秒收尾]** 告警查询优化：explain 看 type 是否 ref/key 是否有效、Extra 是否有覆盖索引；大表分页用游标分页代替 offset，避免深度翻页 O(n) 问题。

### 存储引擎与数据类型
```
InnoDB 和 MyISAM 的区别是什么？MyISAM 的表锁对并发有什么影响？VARCHAR 和 CHAR 各自的特点？DATETIME 和 TIMESTAMP 有什么区别？
追问：
- VARCHAR(255) 和 VARCHAR(256) 有什么区别？
- 为什么推荐使用自增主键？
```
**[10秒结论]** InnoDB 支持事务、行锁、外键，是默认引擎；MyISAM 无事务、表锁、只支持表级并发；VARCHAR 变长按实际长度存储，CHAR 定长用空格补齐。

**[90秒拆解]**
- **InnoDB vs MyISAM**：
  - InnoDB：支持事务（ACID）、行锁、外键、MVCC、崩溃恢复；共享表空间；适合写多并发高的 OLTP
  - MyISAM：表锁（写阻塞读）、不支持事务、无崩溃恢复；每表三个文件（.frm/.MYD/.MYI）；适合读多写少的 OLAP
- **VARCHAR vs CHAR**：
  - VARCHAR：0-65535 字节，变长，需额外 1-2 字节记录长度；适合长度变化的字段（用户名、邮箱）
  - CHAR：定长，不足用空格补齐；适合定长字段（手机号、MD5 哈希值）
- **DATETIME vs TIMESTAMP**：
  - DATETIME：8 字节，范围 1000-01-01 ~ 9999-12-31，不受时区影响
  - TIMESTAMP：4 字节，范围 1970-2038，自动 UTC 转换，适合跨时区场景
- **自增主键优势**：B+Tree 顺序插入减少页分裂；主键空间小（二级索引存主键值），索引体积小

**[20秒收尾]** 告警数据用 BIGINT 自增主键（避免字符串主键的索引膨胀），时间字段用 DATETIME（告警时间需精确到毫秒）。

### 主从复制与高可用
```
MySQL 主从复制的原理是什么？异步复制和半同步复制各自的特点？主从延迟是怎么产生的？怎么解决？读写分离怎么实现？
追问：
- GTID 是什么？为什么比传统 binlog 位置复制更好？
- binlog 的三种格式（STATEMENT/ROW/MIXED）各自的特点？
```
**[10秒结论]** MySQL 主从基于 binlog + relay-log 实现：主库记录 binlog，从库 IO 线程拉取 binlog 写入 relay-log，SQL 线程重放；GTID 用事务 ID 代替文件位置复制，更安全。

**[90秒拆解]**
- **主从复制流程**：主库 DML → 写 binlog（binlog dump 线程推送到从库）→ 从库 IO 线程接收 → 写入 relay-log → 从库 SQL 线程重放
- **异步 vs 半同步**：异步（从库不确认收到即提交）；半同步（主库等待至少一个从库确认收到 binlog 才提交，RPO 更低）
- **GTID**：Global Transaction Identifier，每个事务有唯一 UUID:SN 标识。故障切换时新主库可直接从断点继续，无需 binlog 位置对齐
- **binlog 格式**：
  - STATEMENT：记录 SQL 语句（主从可能结果不同，如 NOW()）
  - ROW：记录行变化（最安全，占用空间大）
  - MIXED：自动切换（不确定时用 ROW）
- **主从延迟**：从库 SQL 线程单线程重放（5.7+ 可多线程：SlaveParallelType=LOGICAL_CLOCK）；解决方案：多线程并行复制（GTID模式）、延迟从库读

**[20秒收尾]** 告警写入走主库，查询走从库（读流量分离），GTID 半同步模式下 RPO < 1 秒，满足合规要求。

### 分库分表
```
什么情况下需要分库分表？水平分片和垂直分片各自的策略是什么？分片键怎么选择？ShardingSphere 是什么？分库分表后跨分片查询怎么处理？
追问：
- 分库分表后 ID 主键怎么生成？
- 分布式事务怎么解决？
```
**[10秒结论]** 单表 > 2000 万行或磁盘 > 100GB 时考虑分片；水平分片按数据行拆分（按 ID mod N），垂直分片按业务列拆分；ShardingSphere 做透明分片；跨分片用 ES 或 打散后聚合。

**[90秒拆解]**
- **水平分片策略**：取模分片（ID % N）、范围分片（按时间）、一致性哈希（减少扩容迁移）
- **垂直分片**：按业务模块拆分（用户表、订单表分不同库），减少单库宽度
- **分片键选择**：选择查询最频繁的字段（如 user_id），避免跨分片查询；热点字段不适合做分片键
- **ShardingSphere**：Java 实现，ShardingSphere-JDBC（客户端分片）和 ShardingSphere-Proxy（代理分片），支持 MySQL/PostgreSQL/Official SQL 兼容数据库
- **跨分片查询**：
  - 聚合统计：ES/Hive 离线计算，结果写入汇总表
  - 联表查询：同分片键才能本地 JOIN
  - 多分片并发查询后聚合（Scatter-Gather）
- **分布式 ID 生成**：雪花算法（时间戳 + 机器 ID + 序列号）；UUID；数据库号段

**[20秒收尾]** 告警历史表按月分片（created_at%12），2023 年告警在 2023 分片，跨年查询用 ShardingSphere 的广播表（sys_dict）做字典翻译。

### 备份恢复与监控
```
MySQL 全量备份和增量备份分别怎么做？mysqldump 和 xtrabackup 各自的适用场景？怎么用 binlog 做 point-in-time 恢复？QPS、TPS、连接数怎么监控？
追问：
- 什么情况下需要做主从切换？
- MHA 和 Orchestrator 是什么？
```
**[10秒结论]** mysqldump 做逻辑备份（SQL 语句），适合小数据量和跨版本迁移；xtrabackup 做物理备份（文件级），支持增量备份和大数据量；binlog 做 point-in-time 恢复；Prometheus + mysqld_exporter 监控 QPS/连接数。

**[90秒拆解]**
- **mysqldump**：逻辑备份，`mysqldump -A -F --single-transaction > backup.sql`，--single-transaction 保证一致性（InnoDB）；缺点：大数据量备份慢
- **xtrabackup**：物理备份，支持增量（xtrabackup --backup --target-dir=/backup/inc1 --incremental-basedir=/backup/full），备份速度快，适合 TB 级数据库
- **Point-in-time Recovery**：
  ```bash
  # 先恢复全量备份
  mysql < full_backup.sql
  # 再重放 binlog 到指定时间点
  mysqlbinlog --stop-datetime="2024-01-01 12:00:00" binlog.000001 | mysql
  ```
- **监控指标**：
  - QPS：`SHOW GLOBAL STATUS LIKE 'Questions'` / `SHOW GLOBAL STATUS LIKE 'Uptime'`
  - TPS：`SHOW GLOBAL STATUS LIKE 'Com_%'`（Com_commit + Com_rollback）
  - 连接数：`SHOW PROCESSLIST` / `SHOW GLOBAL STATUS LIKE 'Max_used_connections'`
  - 慢查询：`SHOW GLOBAL STATUS LIKE 'Slow_queries'`

**[20秒收尾]** 告警数据库每晚 xtrabackup 全量备份 + 每小时增量备份，保留 7 天；binlog 保留 30 天，支持任意时间点恢复，满足等保要求。
