### 状态机设计
```
你的 Agent 怎么建模告警研判的状态流转？请描述状态节点、边（edges）和条件判断的具体实现逻辑？
追问：为什么用 `loop_count` 限制而不是 ReAct 的自适应停止？
```
**[10秒结论]** 用 LangGraph StateGraph 定义 AlertState，通过 `conditional_edges` 根据 triage_result 字段动态路由，配合显式 `loop_count` 防止无限循环。

**[90秒拆解]**
- **State 定义**：
  ```python
  class AlertState(TypedDict):
      alert: dict
      triage_result: str      # true_positive / false_positive / need_review
      attack_stage: str       # MITRE 战术阶段
      mitre_tactics: list[str]
      loop_count: int        # 循环计数器
      messages: Annotated[list, operator.add]  # 对话历史
  ```
- **节点设计**：triage（研判）→ enrich（情报丰富）→ escalate（升级处置）或 auto_close（自动关闭）
- **条件边路由**：
  ```python
  def route_after_triage(state: AlertState) -> str:
      if state["triage_result"] == "true_positive":
          return "escalate"
      elif state["loop_count"] >= 3:
          return "auto_close"  # 循环超限强制关闭
      return "enrich"
  ```
- **loop_count vs 自适应停止**：自适应停止依赖 LLM 自己判断"我已经完成了"，容易受 prompt 影响不稳定；loop_count 是确定性约束，生产环境必须用

**[20秒收尾]** 实际生产中 loop_count=3 配合 enrich 不满意则回去重 triage，保证最多 3 次迭代，平衡了质量与成本。

### Attack-Stage-识别
```
Attack Stage（初始访问→执行→持久化…）的识别是基于 LLM 推理还是规则匹配？当 LLM 输出一致性差时怎么兜底？
追问：14 个阶段有没有出现过 LLM 集中在某几个阶段的系统性偏差？
```
**[10秒结论]** 混合方案：LLM 推理作为主路径 + 规则+NLP 预筛选兜底；LLM 一致性差时通过 Prompt Few-Shot 示例校准，长期通过评估数据持续微调。

**[90秒拆解]**
- **识别策略**：LLM 推理为主，输出 MITRE Tactic ID（如 initial_access）。LLM 输出置信度低于阈值时（如 0.6），降级走规则匹配兜底（基于告警类型的静态映射表）
- **兜底规则**：如告警类型=SSH_BRUTEFORCE → initial_access；告警类型=Ransomware_Detected → impact
- **一致性偏差处理**：监控各阶段分布比例，若某阶段远高于均值（> 均值+2σ）则启动 prompt 调优；定期用标注数据做评估，保留 bad case 加入 few-shot 示例
- **14 阶段覆盖**：实践中发现 LLM 对 initial_access、execution、command_and_control 识别率高，对 resource_development、reconnaissance 等前置阶段识别率低，因为这些阶段日志特征不明显

**[20秒收尾]** 每周 review LLM 阶段分布热力图，发现 bias 及时调 prompt，保持各阶段识别均衡，避免系统性漏检。

```
告警的 P0/P1/P2/P3 是怎么输出的？是 prompt 里写死规则，还是让 LLM 自由判断？你怎么做评估和调优？
追问：什么场景会降级？什么场景会升级？有没有具体例子？
```
**[10秒结论]** 混合方案：规则引擎做 P3/P2 初判，LLM 做 P0/P1 二次分析，结合置信度阈值动态定级；评估用 Precision/Recall/F1 指标持续迭代。

**[90秒拆解]**
- **定级流程**：告警进入 → 规则引擎命中（如"勒索软件签名"→ P0）→ 规则未命中但有上下文关联 → LLM 推理 severity
- **降级例子**：告警为 PowerShell 执行，但用户是运维岗且在维护窗口内 → P1→P2（降级）
- **升级例子**：P2 告警（异常登录）在 15 分钟内同用户又触发账户篡改告警 → P2→P1（升级）

**[20秒收尾]** 告警定级不是一成不变的，通过告警聚合分析（关联告警提升级别）和分析师反馈闭环持续调优。

### 混合检索融合策略
```
Qdrant 向量检索 + BM25 的融合策略是什么？Reranker 用的是 BGE 什么版本？有做过多路召回的召回率评估吗？
亮点：bge-reranker-v2-m3 本地部署，适配专有云；参数 k=20、top_n=3、threshold=0.5
```
**[10秒结论]** Qdrant 的 dense 向量检索 + sparse BM25 全文检索双路召回，RRF（Reciprocal Rank Fusion）融合，k=60 取倒数排名求和；Reranker 用 bge-reranker-v2-m3 本地 GPU 推理，top_n=3，threshold=0.5。

**[90秒拆解]**
- **RRF 融合公式**：$RRF_{doc} = \sum_{r \in R} \frac{1}{k + rank_r(doc)}$，k=60 是经验最优值（原始论文验证）
- **两路检索**：向量检索擅长语义相似（如"恶意软件"↔"病毒"），BM25 擅长专有名词精确匹配（如 CVE-2024-XXXX、IP 地址）
- **Reranker 参数**：k=20 为 Reranker 提供充足候选（实际命中 top-3 约 85%），threshold=0.5 过滤低相关结果
- **本地部署优势**：bge-reranker-v2-m3 在专有云部署，P99 检索延迟 45ms（仅检索链路，不含 LLM 生成）

**[20秒收尾]** 混合检索 + 两阶段 Rerank 相比纯向量检索，端到端准确率提升 20-30 个百分点（实测数据）。

### chunking-与多源关联
```
SOP、历史案例等多源知识是怎么切分（chunking）的？块大小怎么定？跨文档的上下文关联怎么做？
追问：chunk 大小怎么定的？过大/过小分别有什么问题？
```
**[10秒结论]** SOP 按层级结构 chunk（保留步骤链完整性），历史案例按事件边界 chunk；chunk_size=512 tokens，overlap=64 tokens；跨文档关联通过 metadata 的 group_key（alert_id）实现。

**[90秒拆解]**
- **SOP chunking**：按 section 级别保留完整步骤链，避免步骤被切断导致上下文丢失；同时生成 doc-level summary chunk 用于高层检索
- **历史案例 chunking**：以事件边界切分，每个案例独立成块，避免多事件混杂
- **chunk_size Trade-off**：<128 太碎，上下文稀缺；>1024 稀释核心信息；512 是 SOC 场景最佳平衡点
- **跨文档关联**：统一用 `alert_id` 作为 `group_key`，检索结果通过 metadata filter 关联同一告警相关的 SOP 和案例

**[20秒收尾]** SOP 用层级 chunk 保留处置步骤链，历史案例用事件级 chunk 避免混淆，metadata 标签体系支撑多源关联查询。

### RAG-效果评估
```
你们怎么衡量 RAG 效果？有做过 Human Evaluation 或 A/B 测试吗？线上准确率大概在什么水平？
追问：端到端 RAG 准确率是多少？
```
**[10秒结论]** 四维评估体系：检索质量（Recall@K、MRR）、生成质量（RAGAS faithfulness/relevancy）、端到端准确率（分析师采纳率）、系统性能（P50/P99 延迟）；端到端准确率 78-83%。

**[90秒拆解]**
- **Retrieval 评估**：Recall@20 > 0.85，Recall@3 > 0.70，MRR > 0.65 为达标
- **Generation 评估**：RAGAS faithfulness > 0.85，answer_relevancy > 0.80，context_precision > 0.75
- **端到端评估**：每周分析师采纳率统计，目标 > 78%。Bad case 分析发现 CVE 编号检索、跨文档关系丢失是主要瓶颈
- **A/B 测试**：用 stable hash（user_id 分桶）保证同用户稳定分组，对比新旧版本采纳率差异，p < 0.05 视为显著

**[20秒收尾]** 端到端准确率 78-83% 意味着约 1/5 的回答需要分析师修正，是持续优化方向（目标是 90%）。

### Tag-路由机制
```
cheap / fast / powerful 三种模式具体指哪些模型？Tag 标签切换的路由层是怎么实现的？
追问：多个 Tag 同时指定时（如 `["fast", "function_calling"]`）怎么匹配？
```
**[10秒结论]** Tag 是集合标签（cheap/fast/powerful），路由层用集合运算匹配注册模型；多 Tag 组合用 `required_tags.issubset(config.tags)` 精确筛选。

**[90秒拆解]**
- **Tag 定义**：cheap（gpt-4o-mini、qwen-turbo，低成本+低温，适合格式化输出）；fast（qwen-plus，适合对话摘要）；powerful（gpt-4o、llama3，适合复杂推理）
- **匹配逻辑**：
  ```python
  async def route(self, required_tags: set[str]) -> BaseLLMClient:
      candidates = [cfg for cfg in self._all_configs()
                   if required_tags.issubset(cfg.tags)]
      return max(candidates, key=lambda x: x.priority).client
  ```
- **多 Tag 匹配**：`{"fast", "function_calling"}` → 筛选同时包含 fast 和 function_calling 的模型；若没有完全匹配，降级到更宽泛的 Tag

**[20秒收尾]** 框架解耦了模型实现和路由策略，新增模型只需注册 Tag，无需改业务代码。

### Token-成本优化
```
简历提到"平衡推理效果与 Token 成本"，具体怎么做的？有没有用过缓存（如 Redis）、结果复用、或 prompt 压缩策略？
陷阱：简历提了成本优化就必问，要能说出具体手段
```
**[10秒结论]** 三板斧：Redis 语义缓存（相同语义问题直接返回）、Prompt 历史压缩（保留系统提示+最近 N 条对话）、Tag 路由强制分层（简单任务限制 cheap 模式）。

**[90秒拆解]**
- **Redis 语义缓存**：相同 prompt + temperature 的 SHA256 hash 作为 key，命中即返回，成本为 0。命中率 30-50%（客服场景）
- **Prompt 压缩**：保留 system prompt + 最近对话，过期对话按 token 预算截断。估算公式：中文按字符数，英文按单词数 × 1.3
- **Tag 路由强制**：业务层在代码层面限制哪些场景用 cheap/fast/powerful，而非依赖开发者自觉
- **效果数据**：综合优化后 Token 消耗降低 40-60%（实测数据）

**[20秒收尾]** 缓存命中率是关键指标，在告警场景下相同类型告警的研判结果可被缓存，避免重复 LLM 调用。

### 多模型输出-schema-一致性
```
OpenAI / 通义千问 / Ollama 混用时，怎么保证输出 schema 一致？有没有做 schema validation？
```
**[10秒结论]** Pydantic BaseModel 定义严格 Schema，`model.with_structured_output()` 强制输出；多模型 JSON 格式差异（引号、尾逗号、Markdown 包裹）通过正则提取 + JSON 容错解析兜底。

**[90秒拆解]**
- **Pydantic 校验**：定义输出 Schema，通过 `field_validator` 做字段级校验，拒绝不合规结果
- **正则提取 JSON**：处理 ` ```json ... ``` ` 包裹、单引号、尾逗号等格式差异
- **降级策略**：JSON 解析失败 → 正则提取可用字段 → 字段级正则提取 → 部分填充 + `is_fallback=True` 标记
- **Validation 失败率监控**：若某模型失败率 > 5%，说明该模型对该任务不稳定，应切换或优化 prompt

**[20秒收尾]** 各模型 Schema Validation 失败率 < 3%，Pydantic 的错误处理保证了系统鲁棒性。

### 可视化编排-代码化扩展
```
剧本的可视化编排和代码化扩展，底层是怎么关联的？DSL 定义是什么样子？新剧本的开发周期大概多久？
```
**[10秒结论]** 双轨架构：YAML 声明式剧本（分析师编辑）编译为 Python Class（执行逻辑），中间通过统一 AST 解耦；ReactFlow 可视化渲染节点图，拖拽实时生成 YAML。

**[90秒拆解]**
- **YAML DSL**：声明节点类型（action/approval/condition/notification）、动作名、输入输出、条件判断、超时配置
- **Python 编译层**：`yaml.safe_load()` → `PlaybookNode` dataclass → `StateGraph` 节点函数 → LangGraph 执行
- **ReactFlow 可视化**：节点类型映射 DSL schema，拖拽操作实时生成 YAML，代码编辑修改 YAML 同步刷新图形
- **新剧本开发周期**：简单剧本（5步以内）1-2 天；复杂剧本（含审批节点+并行分支）1-2 周

**[20秒收尾]** DSL 将"说什么"和"怎么做"解耦，安全分析师写 YAML，业务逻辑工程师写 Python，两者互不干扰。

### 人工审批节点
```
人工节点怎么设计状态挂起和恢复？审批超时、审批人失联这些边界情况怎么处理？
```
**[10秒结论]** 审批节点将状态机挂起（status=paused），写入 `pending_approvals` 表；恢复时通过 `resume_playbook_instance` API 传入决策；超时后按策略升级/自动审批/自动拒绝。

**[90秒拆解]**
- **挂起机制**：`approval_node` 将 `status: paused` + 审批请求写入 DB，前端展示待审批列表
- **恢复 API**：`POST /api/v1/approvals/{id}/decision`，更新 DB 状态后调用 `resume_playbook_instance` 恢复执行
- **超时策略**：超时 → 升级通知 SOC Lead → 延后 1h；再超时 → 判定失联 → 剧本暂停 + 告警
- **失联处理**：超过 2 次升级通知无响应，进入人工介入模式，剧本暂停并告警 SOC Lead

**[20秒收尾]** 高危操作（隔离主机、封锁用户）必须审批，但超时策略保证业务不因审批人缺席而无限等待。

注意：这里的 50+ 是 SOAR 剧本数量，与 MITRE 战术技术数量无关
```
**[10秒结论]** 50+ 是 SOAR 安全剧本数量，包括钓鱼封禁、横向移动阻断、挖矿检测等；剧本以规则+脚本为主，LLM 参与复杂推理和告警摘要；回滚通过幂等操作+补偿机制保证一致性。

**[90秒拆解]**
- **剧本类型**：钓鱼封禁（IOC 提取+邮件网关封锁）、横向移动（EDR API+风险评分）、挖矿检测（CPU+矿池IP+进程名三合一）、勒索软件响应（快照+隔离+通知）
- **LLM 参与程度**：LLM 用于告警上下文摘要、攻击阶段推理；核心动作执行（封禁IP、隔离主机）走确定性脚本
- **回滚机制**：三层——可逆操作（解封 IP）精确回滚；不可逆操作（杀进程）执行补偿（创建快照）；无法处理升级人工

**[20秒收尾]** SOAR 剧本的本质是有状态的状态机，LangGraph 提供开箱即用的条件路由和 checkpoint，比自研状态机效率高 3 倍以上。

### NL-to-SPL-DSL-技术路线
```
自然语言转 Splunk SPL 或 ELK Query DSL，具体用了什么技术路线？few-shot prompt、fine-tune 还是微调模型？覆盖了多少查询模式？
```
**[10秒结论]** Few-shot Prompt 快速上线为主，配合意图分类路由；Schema 注入（index/field 白名单）防止越权；AST 语法校验防止注入攻击。

**[90秒拆解]**
- **Few-shot Prompt**：精心设计的 system prompt + 自然语言↔SPL 对示例，LLM 上下文学习零训练成本
- **意图分类路由**：先分类（统计查询/搜索查询/威胁狩猎），再分发到对应 DSL（ES|QL / SPL / SQL）
- **Schema 注入**：每次推理前注入当前用户有权访问的 index 白名单和字段列表，防止跨租户查询
- **AST 校验**：生成的 SPL 用 ANTLR 解析 AST，拒绝包含 `|script`、`|delete` 等危险操作的语句

**[20秒收尾]** Few-shot 路线适合快速上线，准确率 85-90%；复杂嵌套查询（join/transaction）是当前瓶颈，需要持续收集 bad case 补充示例。

### 查询安全
```
怎么防止 prompt injection 或恶意查询（比如查询范围过大导致集群过载）？
```
**[10秒结论]** 三道防线：Schema 注入（index 白名单）、AST 语法校验（拒绝危险命令）、查询范围限制（时间窗口 + 结果条数上限）；防止 prompt injection 的核心是用户输入与系统指令严格分离。

**[90秒拆解]**
- **Prompt Injection 防御**：用户输入永远作为变量绑定（`{user_query}`），不和 system prompt 混合，防止注入指令覆盖系统行为
- **恶意查询防护**：限制查询时间窗口（如最多查 30 天）、限制结果条数（如最多 10000 条）、限制高消耗操作（如 wildcard 匹配）
- **AST 校验层**：对生成的 SPL/ES|QL 做语法解析，拒绝不合语法的查询和危险操作

**[20秒收尾]** 查询安全是多租户 SIEM 的生命线，Schema 注入 + AST 校验是生产级 NL 查询的必备保障。

### React-状态管理
```
React + TypeScript 项目里，你们的状态管理方案是什么？React Query / Redux / Zustand？有遇到过大型列表性能问题吗？
```
**[10秒结论]** SOC 平台告警面板用 Zustand 管理（轻量），全局认证状态用 Context；React Query 做服务端状态管理（告警列表分页/缓存）；告警详情用 URL state（query params）。

**[90秒拆解]**
- **Zustand**：告警面板状态（选中告警、过滤条件、排序），轻量且无需 Provider
- **React Query**：服务端状态，告警列表的缓存、自动刷新（每 30s polling）、分页，配置 staleTime 减少不必要请求
- **Context**：认证 token、主题设置、国际化语言
- **大列表优化**：react-window 做虚拟滚动（10000 行 DOM 节点 → 20 个可见节点），HEAP 使用稳定在 150MB 以内

**[20秒收尾]** 告警分析页面的性能瓶颈在于 DOM 节点数，虚拟滚动是核心解法，配合 React DevTools Profiler 逐个击破其他 re-render。

### CI-CD-流水线
```
Jenkins 构建触发、分支策略、灰度/回滚机制是怎么设计的？
```
**[10秒结论]** Jenkins 主分支 PR 合入触发构建 → Docker 镜像打包 → Trivy 安全扫描 → 推送镜像到 Registry → K8s 滚动更新（金丝雀 10% → 50% → 100%）；回滚用 `kubectl rollout undo` 一键恢复。

**[90秒拆解]**
- **构建触发**：GitLab CI / Jenkins，当 PR 合入 main 时触发 pipeline
- **分支策略**：Git Flow — main（生产）、develop（开发）、feature/*（功能）、release/*（发布）
- **灰度发布**：K8s 滚动更新策略 `maxUnavailable=0, maxSurge=10%`，先更新 1 个 Pod，验证通过后再更新剩余
- **回滚机制**：`kubectl rollout undo deployment/xxx`，或通过 Jenkins UI 一键回滚上一镜像版本
- **镜像安全扫描**：Trivy 扫描 Dockerfile 和依赖，CVSS ≥ 7.0 的高危漏洞阻断构建并告警

**[20秒收尾]** K8s 滚动更新 + readinessProbe 健康检查，保证新版本有问题时只影响极少用户，且可一键回滚。

### 历史技术债清理
```
你说"清除历史技术债"，具体清的是什么？Vue 1/2 → Vue 3 迁移？还是重构了哪些核心模块？有没有量化成果？
陷阱："2周→3天上手周期"容易被追问具体怎么做的，要有具体例子
```
**[10秒结论]** 主要是 Vue 2 → Vue 3 迁移（Composition API 重构）、老旧组件库升级（Ant Design Vue 1.x → 3.x）、重复封装的工具函数统一；通过 CodeMod 脚本辅助批量替换，量化成果用首屏加载和告警处理速度衡量。

**[90秒拆解]**

**[20秒收尾]** 技术债清理的优先级靠 ROI 说话——影响开发者体验（构建速度、上手时间）和影响用户体验（首屏速度、告警延迟）的债优先清。

```
20+ 人分散多条业务线，你是怎么做资源分配和优先级排序的？有没有出现过资源冲突？怎么解决的？

**[20秒收尾]** 资源协调本质是利益分配，用数据说话（ROI、用户影响面）比拍脑袋决定更容易让各方信服。
