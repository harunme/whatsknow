## 一、AI Agent 基础

### 1.1 什么是 AI Agent

**LLM（Large Language Model）** 本身只是一个"能说会道的文本生成器"——你问它答，答完即止，没有记忆、没有行动能力。

**AI Agent** 则是在 LLM 基础上增加了三个关键能力：

```
Agent = LLM（大脑）+ Tools（手）+ Memory（记忆）+ Planner（规划器）
```

| 组件 | 作用 | 类比 |
|------|------|------|
| LLM | 推理与决策 | 大脑 |
| Tools | 调用外部系统（查数据库、调 API、搜索） | 手 |
| Memory | 保存历史上下文 | 记忆 |
| Planner | 决定下一步做什么 | 策略 |

**为什么 Agent 比单纯 LLM 调用更有价值？** 因为现实世界的问题不是一次问答能解决的。SOC 告警分析需要：先查 IP 情报 → 再查资产归属 → 再看历史告警 → 再决定是否隔离——这是一个多步、循环、有条件分支的复杂流程，单次 LLM 调用根本装不下。

### 1.2 ReAct 模式（Reasoning + Acting）

ReAct 是 Agent 最核心的思考框架，核心思想是：**不要闷头执行，先想再做，边做边观察**。

```
Thought → Action → Observation → Thought → Action → Observation → ...
  思考      行动        观察        思考        行动        观察
```

**一个实际例子：分析一条可疑告警**

```
Thought: 这是一条 SSH 暴力破解告警，我需要先查询这个源 IP 的历史行为，
         看看是否在多个主机上有过登录失败记录。
Action: query_ssh_history(source_ip="1.2.3.4", limit=10)
Observation: 该 IP 在 5 台主机上有登录失败记录，成功登录过 1 次（主机 db-prod-03）
Thought: 该 IP 曾经成功登录过生产数据库主机，威胁等级需要升级。
Action: escalate_alert(severity="P1", reason="credential_access_to_prod_db")
Observation: 告警已升级，SOC 分析师已收到通知。
```

**为什么要分 Thought/Action/Observation 三步？**

- **Thought**：让 LLM 先"想清楚"再行动，减少盲目调用工具浪费 Token
- **Action**：执行具体工具，工具返回结构化结果
- **Observation**：将结果反馈给 LLM，让它基于真实数据继续推理

**ReAct 的核心价值**：将 LLM 的推理过程显式化，使得我们可以追踪、干预和调试 Agent 的决策链路。没有 ReAct，LLM 就只是一个黑盒；有 ReAct，Agent 的每一步都是可审计的。

### 1.3 Agent vs 单纯 LLM 调用：核心区别

| 维度 | 单纯 LLM 调用 | AI Agent |
|------|--------------|----------|
| 调用次数 | 1 次 | 多次循环 |
| 工具调用 | 无 | 有（Tool Calling） |
| 记忆能力 | 无（每次请求独立） | 有（Memory/Checkpointer） |
| 条件分支 | 无 | 有（根据结果决定下一步） |
| 人工干预 | 无 | 支持（Human-in-the-loop） |
| 适用场景 | 简单问答、摘要、翻译 | 多步分析、复杂决策、自动化流程 |
| Token 消耗 | 低 | 高（需要显式管理） |

**什么时候用 Agent，什么时候用单纯 LLM 调用？**

- **用单纯 LLM**：任务一步能完成，不需要外部数据，不需要循环（写文案、做翻译、简单分类）
- **用 Agent**：多步骤任务、需要查外部系统、有条件分支、需要人工审核（告警分析、客服对话、自动化运维）

## 二、LangChain 核心组件

> LangChain 是构建 LLM 应用的基础框架，提供了组件化的抽象。LangGraph 则是在 LangChain 基础上，增加了**有向图**的表达能力（支持循环、条件分支、人为中断）。可以理解为：LangChain = 零件库，LangGraph = 用零件组装机器的蓝图。

### 2.1 PromptTemplate：动态拼接 Prompt

**问题背景**：Prompt 里有很多固定内容（系统指令、格式要求）和少量变量（用户输入、上下文），如果每次都手动拼接，代码会变得又臭又长。

**解决方案**：用 `PromptTemplate` 把固定内容和变量分离。

```python
from langchain_core.prompts import PromptTemplate

# 定义一个告警分析专用的 Prompt 模板
prompt = PromptTemplate.from_template(
    """你是一个 SOC 安全分析师，擅长分析告警并给出处置建议。

## 当前告警信息
- 告警类型：{alert_type}
- 源 IP：{source_ip}
- 目标资产：{target_asset}
- 告警时间：{alert_time}

## 你的任务
1. 判断是否为真实攻击（true_positive / false_positive）
2. 评估威胁等级（P0/P1/P2/P3）
3. 给出处置建议

请以 JSON 格式输出，包含 severity、verdict 和 recommendation 三个字段。"""
)

# 调用时只需传入变量，模板自动拼接
formatted = prompt.invoke({
    "alert_type": "SSH_BRUTEFORCE",
    "source_ip": "1.2.3.4",
    "target_asset": "db-prod-03",
    "alert_time": "2026-04-10 10:30:00"
})

print(formatted.text)
# 输出：完整的 Prompt 字符串，变量已被替换
```

**为什么要用 PromptTemplate？**

1. **复用性**：一份模板多处调用，修改一处全局生效
2. **可测试性**：同一模板换不同输入，验证输出稳定性
3. **版本管理**：模板可以像代码一样版本化（见第 8 节生产调优）

⚠️ **注意**：Prompt 中的 `{变量名}` 必须与传入字典的 key 完全一致，否则会抛出 `KeyError`。建议在模板顶部加注释说明所需变量。

### 2.2 ChatModel：统一接口调用多模型

LangChain 的 `ChatModel` 抽象让你可以在不改变业务代码的情况下切换底层 LLM。

```python
from langchain_openai import ChatOpenAI
from langchain_community.chat_models import ChatTongyi

# OpenAI 模型（生产用）
openai_model = ChatOpenAI(
    model="gpt-4o",
    temperature=0.0,  # 告警分析需要确定性，temperature=0
    api_key="sk-...",
)

# 通义千问（国产替代）
qwen_model = ChatTongyi(
    model="qwen-plus",
    temperature=0.0,
)

# 两者调用方式完全一致
messages = [{"role": "user", "content": "SSH 暴力破解告警，源 IP 1.2.3.4，如何处置？"}]
response = openai_model.invoke(messages)
print(response.content)  # 输出分析结果
```

**温度参数（temperature）的选择**：

| temperature | 场景 | 特点 |
|-------------|------|------|
| 0.0 | 结构化输出、分类、告警分析 | 最确定，输出稳定 |
| 0.3-0.5 | 摘要、改写 | 略有变化，避免重复 |
| 0.7-1.0 | 创意写作、头脑风暴 | 多样性强，不可预测 |

### 2.3 OutputParser：结构化输出解析

LLM 输出的是纯文本，但业务代码需要结构化数据（字典、对象）。`OutputParser` 就是这个翻译层。

```python
from langchain_core.output_parsers import JsonOutputParser
from pydantic import BaseModel, Field

# 定义期望的输出结构
class AlertTriageResult(BaseModel):
    severity: str = Field(description="告警级别，P0/P1/P2/P3")
    verdict: str = Field(description="研判结果，true_positive 或 false_positive")
    recommendation: str = Field(description="处置建议")
    mitre_tactics: list[str] = Field(description="匹配的 MITRE ATT&CK 战术")

# 创建解析器，并绑定到模型
parser = JsonOutputParser(pydantic_object=AlertTriageResult)
model_with_parser = model | parser

# 调用时直接获得 Pydantic 对象
result = model_with_parser.invoke("分析告警：SSH暴力破解，源IP 1.2.3.4")
print(result.severity)    # "P2"
print(result.verdict)     # "true_positive"
print(result.mitre_tactics)  # ["credential_access"]
```

**OutputParser 的工作原理**：它在 Prompt 末尾自动注入格式指令（如"请以 JSON 格式输出"），然后解析 LLM 的原始文本输出，转换为 Pydantic 对象。如果 LLM 输出格式不规范，Pydantic 的字段验证会报错。

⚠️ **注意**：不是所有模型都能可靠输出 JSON。GPT-4o 成功率 > 98%，而某些开源模型可能输出 Markdown 包裹的 JSON 或尾逗号等格式问题。生产环境需要加容错处理（正则提取 + JSON 容错解析）。

### 2.4 VectorStore：向量数据库与语义检索

**为什么需要向量检索？**

传统数据库只能做精确匹配（`WHERE alert_type = 'SSH_BRUTEFORCE'`）。但安全分析师的查询往往是模糊的："和上次那个勒索软件告警类似的处置方法是什么？"——这种语义相似性搜索，必须靠向量数据库。

**向量检索原理简述**：

```
文本 → Embedding Model → 向量（高维浮点数数组）
查询 → Embedding Model → 向量
向量相似度（余弦相似度）→ 排序 → Top-K 结果
```

```python
from langchain_community.vectorstores import Qdrant
from langchain_community.embeddings import HuggingFaceBgeEmbeddings

# 初始化 Embedding 模型（BGE 中文优化版）
embeddings = HuggingFaceBgeEmbeddings(
    model_name="BAAI/bge-m3",
    model_kwargs={"device": "cpu"},
    encode_kwargs={"normalize_embeddings": True}
)

# 连接 Qdrant 向量数据库
vectorstore = Qdrant.from_texts(
    texts=["钓鱼邮件处置 SOP...", "勒索软件响应剧本..."],
    embedding=embeddings,
    location=":memory:",  # 开发环境用内存，生产用真实地址
    collection_name="soc_playbooks"
)

# 语义检索：找"处理勒索软件"的 SOP
docs = vectorstore.similarity_search(
    query="服务器被勒索软件加密了怎么办",
    k=3  # 返回最相关的 3 个文档
)
for doc in docs:
    print(doc.page_content[:100], "→", doc.metadata)
```

## 三、LangGraph 核心概念

### 3.1 为什么需要 LangGraph

LangChain 的 Chain（链）是**线性**的：Input → Chain1 → Chain2 → Output。它无法表示：

- **循环**：告警研判不通过 → 补充情报 → 再次研判
- **条件分支**：true_positive → 升级，false_positive → 关闭
- **并行节点**：同时查 IP 情报和资产信息，汇合后再分析
- **人工中断**：执行到某节点时暂停，等人工确认

LangGraph 用**有向图**解决了这些问题：节点是函数，边是流转关系，条件边是动态路由。

### 3.2 四个核心概念

```
StateGraph（状态图）
├── State（状态）：整个图的共享数据，贯穿所有节点
├── Node（节点）：一个 Python 函数，输入是当前 state，输出是 state 的更新
├── Edge（边）：节点之间的连接，决定下一个执行谁
│   ├── 普通边（确定）：A 完成后一定执行 B
│   └── 条件边（动态）：A 完成后根据 state 内容决定执行 B 还是 C
└── Checkpoint（检查点）：保存/恢复状态快照（类似游戏存档）
```

**为什么 State 设计如此重要？**

State 是整个 Agent 的"记忆中枢"。所有节点都读写同一个 State，因此 State 的定义决定了：
- 哪些数据在节点间传递
- Checkpointer 保存什么
- Human-in-the-loop 能编辑什么

### 3.3 一个最小 LangGraph 示例

```python
from langgraph.graph import StateGraph, END
from typing import TypedDict

# 第1步：定义 State——这是整个 Agent 的"全局变量"
class AlertState(TypedDict):
    alert_id: str
    severity: str
    triage_result: str
    messages: list[str]

# 第2步：定义 Node（节点）——每个节点是一个 Python 函数
def triage_node(state: AlertState) -> AlertState:
    """研判节点：分析告警，判断是否为真实攻击"""
    print(f"[triage] 分析告警 {state['alert_id']}")
    # 节点函数接收当前 state，返回要更新的字段（部分更新，merge 到原 state）
    return {
        "triage_result": "need_review",  # 更新字段
        "severity": "P2"
    }

def enrich_node(state: AlertState) -> AlertState:
    """丰富节点：查询 IP 情报、资产归属等外部信息"""
    print(f"[enrich] 丰富告警 {state['alert_id']} 的上下文")
    return {"messages": state["messages"] + ["情报丰富完成"]}

# 第3步：构建图
graph = StateGraph(AlertState)

# add_node：注册节点（节点名称 + 函数）
graph.add_node("triage", triage_node)
graph.add_node("enrich", enrich_node)

# add_edge：普通边——triage 完成后一定执行 enrich
graph.add_edge("triage", "enrich")

# 设置入口和终点
graph.set_entry_point("triage")
graph.add_edge("enrich", END)

# 第4步：编译——生成可执行的 App 对象
app = graph.compile()

# 第5步：执行
result = app.invoke({
    "alert_id": "alert-001",
    "severity": "P3",         # 初始值
    "triage_result": "unknown",
    "messages": []
})

print("最终状态：", result)
```

**输出：**
```
[triage] 分析告警 alert-001
[enrich] 丰富告警 alert-001 的上下文
最终状态： {'alert_id': 'alert-001', 'severity': 'P2', 'triage_result': 'need_review', 'messages': ['情报丰富完成']}
```

**关键理解：节点函数的返回值是部分更新（partial update）**

```python
def triage_node(state: AlertState) -> AlertState:
    return {"severity": "P2"}  # 只更新 severity，其他字段保持不变
```

LangGraph 会将返回值 merge 到当前 state。如果节点返回空 `{}`，state 保持不变；如果返回 `{"messages": ["new"]}`，messages 会被追加而不是覆盖。

## 四、StateGraph 实战

### 4.1 State 定义详解

State 的设计是整个 Agent 架构的核心。以下是一个面向 SOC 告警分析的真实 State 定义：

```python
from typing import TypedDict, Annotated
import operator

class AlertState(TypedDict):
    # 告警原始信息（从头到尾不变）
    alert_id: str
    alert_data: dict

    # 研判结果（节点间流转，会被更新）
    triage_result: str          # "true_positive" | "false_positive" | "need_review"
    severity: str               # "P0" | "P1" | "P2" | "P3"
    mitre_tactics: list[str]    # 匹配的 MITRE ATT&CK 战术列表

    # 执行控制
    loop_count: int             # 循环计数器，防止无限循环
    current_node: str           # 当前执行到哪个节点（调试用）

    # 对话历史（用 Annotated + operator.add 实现追加）
    # 这样每次返回 {"messages": ["..."]} 会追加而不是覆盖
    messages: Annotated[list[str], operator.add]

    # 人工审核
    human_decision: str | None  # 人工审批结果
    human_feedback: str | None  # 人工反馈意见
```

**`Annotated[list, operator.add]` 的原理**：

普通 `list` 类型的节点返回值会覆盖已有列表。但 SOC 场景需要**追加**——每经过一个节点，在 messages 里追加一条记录，而不是清空重写。

```python
# 如果 messages 是普通 list：
return {"messages": ["enrich 完成"]}  # 会覆盖之前的所有消息！
# 结果：messages = ["enrich 完成"]（之前的 triage 日志丢失）

# 如果 messages 是 Annotated[list, operator.add]：
return {"messages": ["enrich 完成"]}  # operator.add 语义是 list += ["enrich 完成"]
# 结果：messages = ["triage 开始", "enrich 完成"]（追加而非覆盖）
```

### 4.2 节点设计原则

**每个节点应该只做一件事**，这是函数式编程"单一职责原则"在 Agent 设计中的应用。

```python
# ✅ 好设计：一个节点职责单一
def triage_node(state: AlertState) -> AlertState:
    """仅负责研判，不做情报丰富"""
    ...

def enrich_node(state: AlertState) -> AlertState:
    """仅负责情报丰富，不做研判"""
    ...

def escalate_node(state: AlertState) -> AlertState:
    """仅负责升级处置，不做其他事情"""
    ...

# ❌ 坏设计：一个节点做所有事情
def process_alert(state: AlertState) -> AlertState:
    """研判 + 情报丰富 + 升级处置 + 通知，全都塞进一个函数"
    # 300 行代码，无法单独测试、无法单独重试、无法单独加人工审核
    ...
```

**为什么单一节点原则很重要？**

1. **可测试性**：每个节点独立测试，不用跑完整流程
2. **可干预性**：Human-in-the-loop 可以在任意节点暂停
3. **可重试性**：某个节点失败只需重跑这一个节点，不用从头开始
4. **可观测性**：每步执行都能打日志，出了问题容易定位

### 4.3 条件边（conditional_edges）：核心路由逻辑

条件边是 LangGraph 最强大的特性——它让 Agent 可以**根据运行时的 state 内容决定下一步**。

```python
def route_after_triage(state: AlertState) -> str:
    """
    triage 节点完成后，根据研判结果决定下一步。
    返回值是目标节点的名称（字符串）。
    """
    if state["triage_result"] == "true_positive":
        return "escalate"          # 真实攻击 → 升级处置
    elif state["loop_count"] >= 3:
        return "auto_close"        # 循环超限 → 强制关闭
    elif state["triage_result"] == "need_review":
        return "wait_human"       # 需要人工审核 → 暂停等审批
    return "enrich"                # 默认 → 丰富情报后重新研判

# 在图中注册条件边
# 第一个参数：源节点名称
# 第二个参数：路由函数（接收 state，返回目标节点名）
# 第三个参数：路由映射（将返回值映射到节点名）
graph.add_conditional_edges(
    source_node="triage",           # 从 triage 节点出发的条件边
    path=route_after_triage,        # 路由函数
    path_map={                      # 返回值 → 节点名 的映射表
        "escalate": "escalate",
        "enrich": "enrich",
        "auto_close": "auto_close",
        "wait_human": "wait_human",
    }
)
```

**条件边的执行流程图**：

```
┌─────────────┐
│   triage    │
└──────┬──────┘
       │ triage_result
       ├── "true_positive"  ──→ [escalate]  ──→ END
       ├── "need_review"    ──→ [wait_human] ──→ (暂停，等人工)
       ├── loop_count >= 3  ──→ [auto_close] ──→ END
       └── 其他              ──→ [enrich]    ──→ [triage]（循环）
                                                         ↑
                                                         └── 再回到 triage
```

**为什么 loop_count 要手动维护而不是依赖 LLM 自适应停止？**

LLM 的自适应停止（"我认为已经完成了，就停止"）有两个致命问题：

1. **不稳定**：同一个告警，这次循环 2 次停，下次可能循环 5 次停，不可预测
2. **攻击风险**：恶意 prompt 可能让 LLM 故意提前停止，跳过关键研判步骤

**显式 loop_count 是确定性约束**，生产环境必须用它。配合条件边使用，保证 Agent 最多执行 N 次迭代后强制退出，防止死循环和成本失控。

⚠️ **注意**：`loop_count` 需要在每个可能产生循环的节点里手动递增。LangGraph 本身不自动计数，这是开发者的责任：

```python
def enrich_node(state: AlertState) -> AlertState:
    return {
        "loop_count": state["loop_count"] + 1,  # 手动递增计数
        "messages": state["messages"] + ["情报丰富完成"]
    }
```

## 五、Checkpointer：状态持久化

### 5.1 什么是 Checkpointer

Checkpointer 是 LangGraph 的"存档/读档"机制。每次节点执行完成后，它自动将完整 State 快照保存到存储后端（内存/SQLite/PostgreSQL）。需要时，可以从任意 checkpoint 恢复，继续执行。

**为什么需要 Checkpointer？**

| 场景 | 没有 Checkpointer | 有 Checkpointer |
|------|-------------------|-----------------|
| 多轮对话 | 每次 invoke 独立，历史丢失 | 同一 thread_id 共享状态 |
| 人工审核 | 审核时 Agent 停止，审核后无法恢复 | 从 checkpoint 恢复，审核无感知 |
| 服务重启 | 运行中的 Agent 状态全部丢失 | 从数据库恢复，继续执行 |
| 调试回放 | 无法查看中间状态 | 任意时间点的状态都可查看 |

### 5.2 两阶段召回融合（MemorySaver vs SQLiteSaver）

```python
from langgraph.checkpoint.memory import MemorySaver
from langgraph.checkpoint.sqlite import SqliteSaver

# 开发/测试环境：内存存储（重启即丢失，但最快）
memory_checkpointer = MemorySaver()

# 生产环境：SQLite 持久化（重启后可恢复）
sqlite_checkpointer = SqliteSaver.from_conn_string("/data/checkpoints/alert_graph.db")

# 编译时注入 checkpointer
app = graph.compile(checkpointer=sqlite_checkpointer)
```

**选择建议**：
- 开发调试用 `MemorySaver`，速度快，无持久化负担
- 生产环境必须用 `SqliteSaver` 或 `PostgresSaver`，保证服务重启后状态不丢失

### 5.3 thread_id：多会话隔离

```python
# 告警 session 001 的分析会话
config_001 = {"configurable": {"thread_id": "alert-session-001"}}

# 告警 session 002 的分析会话（完全独立）
config_002 = {"configurable": {"thread_id": "alert-session-002"}}

# 第1次调用：alert-session-001，开始分析告警 alert-001
result1 = app.invoke(
    {"alert_id": "alert-001", "messages": [], "loop_count": 0},
    config=config_001
)
print(result1["current_node"])  # "enrich"（执行到 enrich 后人工审核）

# 此时可以暂停，等人工审核。
# 人工事后恢复：同一 thread_id，继续执行
result2 = app.invoke(
    {
        "human_decision": "approved",
        "messages": ["分析师确认：真实攻击，升级处置"]
    },
    config=config_001  # 同一 thread_id，LangGraph 自动从 checkpoint 恢复
)
print(result2["current_node"])  # "escalate"

# alert-session-002 是另一个完全独立的会话
result3 = app.invoke(
    {"alert_id": "alert-002", "messages": [], "loop_count": 0},
    config=config_002
)
```

**thread_id 的本质**：它是 Checkpointer 的命名空间键。相同 thread_id 的调用共享同一个状态历史；不同 thread_id 完全隔离，就像浏览器里的不同标签页。

### 5.4 Checkpointer 内部机制

理解 Checkpointer 的内部机制，有助于正确设计 State 和排查 bug：

```
节点执行流程（带 Checkpointer）：

invoke(alert-001, config={thread_id: "s-001"})
    │
    ▼
┌─────────────────────────┐
│ 1. 从存储读取 thread_id=s-001 │
│    的最新 checkpoint（若有）   │
└────────────┬────────────┘
             │ 获得当前 state
             ▼
┌─────────────────────────┐
│ 2. 执行 triage_node(state) │
└────────────┬────────────┘
             │ 返回部分更新
             ▼
┌─────────────────────────┐
│ 3. 将更新 merge 到 state    │
│    并写入存储（覆盖 checkpoint）│
└────────────┬────────────┘
             │
             ▼
┌─────────────────────────┐
│ 4. 判断是否还有下一个节点     │
│    无 → 返回最终 state       │
│    有 → 重复步骤 1-4        │
└─────────────────────────┘
```

**重要结论**：每次节点执行完成后，**整个 state 完整快照**被保存。这意味着：
- State 越大，每次写入的代价越高（生产环境要注意 State 体积）
- `messages` 字段会不断膨胀（用 `Annotated[list, operator.add]` 追加会加剧这个问题），需要定期截断

## 六、Tool Calling：工具调用

### 6.1 定义工具：@tool 装饰器

LangGraph 中的工具是带有类型提示和描述的 Python 函数。描述是给 LLM 看的——LLM 根据描述决定何时调用这个工具。

```python
from langchain_core.tools import tool

@tool
def lookup_ip_reputation(ip: str) -> dict:
    """
    查询 IP 威胁情报评分。

    Args:
        ip: 要查询的源 IP 地址，格式为点分十进制（如 "1.2.3.4"）

    Returns:
        包含 score（0-100 威胁评分）和 tags（恶意行为标签列表）的字典。
        score >= 80 为高危，50-79 为中危，< 50 为低危。
    """
    # 模拟外部 API 调用
    ip_db = {
        "1.2.3.4": {"score": 92, "tags": ["malware-c2", "botnet"]},
        "5.6.7.8": {"score": 35, "tags": []},
    }
    return ip_db.get(ip, {"score": 0, "tags": []})

@tool
def query_asset_info(hostname: str) -> dict:
    """
    查询主机资产信息，包括业务归属、责任人、开放端口等。

    Args:
        hostname: 主机名或 IP 地址

    Returns:
        包含 owner（责任人）、department（部门）、os（操作系统）、
        open_ports（开放端口列表）、criticality（关键程度 1-5）的字典。
    """
    return {
        "owner": "zhang.san",
        "department": "基础设施部",
        "os": "Ubuntu 22.04 LTS",
        "open_ports": [22, 80, 443, 3306],
        "criticality": 5  # 最高关键程度：P0 级别资产
    }

@tool
def escalate_alert(severity: str, reason: str, alert_id: str) -> dict:
    """
    升级告警到 SOC 分析师或安全运营经理。

    Args:
        severity: 目标告警级别（P0/P1/P2/P3）
        reason: 升级原因说明
        alert_id: 告警 ID

    Returns:
        包含 ticket_id（工单号）和 status 的确认字典
    """
    return {
        "ticket_id": f"TICKET-{alert_id}-{severity}",
        "status": "created",
        "notified": ["soc-leader@company.com"]
    }
```

⚠️ **工具描述的写作规范**：描述是 LLM 唯一的参考依据，必须包含：
1. **功能说明**：这个工具做什么
2. **参数说明**：每个参数的类型和含义
3. **返回值说明**：返回什么，字段含义是什么
4. **边界情况**：异常情况返回什么（方便 LLM 做条件判断）

### 6.2 将工具绑定到模型

```python
from langchain_openai import ChatOpenAI

# 初始化模型
model = ChatOpenAI(model="gpt-4o", temperature=0.0)

# 将工具列表绑定到模型
# 绑定后，模型会"知道"有哪些工具可用，并在需要时输出 tool_calls
model_with_tools = model.bind_tools(
    tools=[lookup_ip_reputation, query_asset_info, escalate_alert],
    tool_choice="auto"  # 可选："auto"（模型自主）/ "required"（强制调用一个）/ 具体工具名
)

# 模型自主决定何时调用工具
response = model_with_tools.invoke(
    "源IP 1.2.3.4 对目标主机 db-prod-03 发起 SSH 暴力破解攻击，"
    "该主机是关键生产数据库，请分析并给出处置建议。"
)
print(response.tool_calls)
# 输出示例：
# [{'name': 'lookup_ip_reputation', 'args': {'ip': '1.2.3.4'}, 'id': '...'},
#  {'name': 'query_asset_info',      'args': {'hostname': 'db-prod-03'}, 'id': '...'}]
```

### 6.3 工具调用的执行循环

LangGraph 的 ToolNode 负责执行 LLM 输出的 tool_calls，并将结果作为消息追加到 state：

```python
from langgraph.prebuilt import ToolNode

# 将工具列表转为可执行的 ToolNode
tool_node = ToolNode([
    lookup_ip_reputation,
    query_asset_info,
    escalate_alert,
])

# 在图中注册工具节点
graph.add_node("tools", tool_node)

# 条件边：如果 LLM 输出包含 tool_calls → 工具节点
# 如果 LLM 输出纯文本（没有 tool_calls）→ 结束或下一步
graph.add_conditional_edges(
    source_node="llm_node",
    path=lambda state: "tools" if state.get("tool_calls") else "__end__",
)
```

### 6.4 handle_tool_errors：工具异常处理

```python
from langgraph.prebuilt import ToolNode

tool_node = ToolNode(
    tools=[lookup_ip_reputation, query_asset_info],
    handle_tool_errors=True,  # 工具出错时返回错误消息，而不是抛出异常中断流程
)

# 工具执行失败时的默认返回格式：
# {"role": "tool", "content": "Error: 连接超时，请稍后重试", "tool_call_id": "..."}
```

**handle_tool_errors=True 的价值**：生产环境中网络超时、API 限流是常态。如果每个工具都要单独加 try-except，代码会膨胀。设为 True 后，ToolNode 自动捕获异常并返回友好的错误消息，让 LLM 可以决定是否重试或降级处理。

### 7.1 为什么需要人工介入

在 SOC 场景中，以下情况必须人工审核：
- **高危操作**：隔离主机、封锁账户——这些操作不可逆，必须人确认

### 7.3 人工决策的完整流程

```python
# 人工审批 API（后端接口）
@app.post("/api/v1/alerts/{alert_id}/approve")
def approve_alert(alert_id: str, decision: str, comment: str):
    """分析师审批告警升级"""
    config = {"configurable": {"thread_id": f"alert-{alert_id}"}}

    # 恢复 Agent 执行：注入人工决策
    result = compiled_app.invoke(
        {
            "human_decision": decision,    # "approved" 或 "rejected"
            "human_feedback": comment,
        },
        config=config  # 同一 thread_id，从中断点恢复
    )

    if result.get("current_node") == END:
        return {"status": "completed", "result": result}
    else:
        return {"status": "in_progress", "next_node": result.get("current_node")}
```

**完整流程图**：

```
Agent invoke
    │
    ▼
[triage] → [enrich] → [triage] → [escalate] ←─ 暂停！
                                      │
                              前端展示审批界面
                                      │
                              分析师点击"确认升级"
                                      │
                              再次 invoke（同一 thread_id）
                                      │
                               [escalate] 执行完成 → END
```

### 7.4 状态编辑：不经节点直接修改 State

有时需要不经 Agent 推理，直接修改 State（比如工单系统更新告警状态后同步 Agent State）：

```python
# 直接编辑 checkpoint 状态（用于同步外部系统状态）
app.update_state(
    config={"configurable": {"thread_id": "alert-session-001"}},
    updates={
        "severity": "P0",              # 修改告警级别
        "triage_result": "escalated",  # 修改研判结果
    }
)
```

⚠️ **注意**：`update_state` 直接修改存储中的 checkpoint，不经过任何节点逻辑。这相当于"作弊"——只有在你确定知道自己在做什么时才使用。典型场景：外部工单系统更新了告警状态，需要同步 Agent State。

## 八、生产调优

### 8.1 loop_count：防止无限循环

这是生产环境中最重要的安全机制。没有 loop_count，LLM 可能因为 prompt 偏差或对抗性输入进入无限循环。

```python
def triage_node(state: AlertState) -> AlertState:
    if state["loop_count"] >= 3:
        # 超限，强制自动关闭（不在这里判断，由条件边路由）
        return {
            "messages": state["messages"] + [f"[triage] 循环超限，强制退出"]
        }

    # 正常研判逻辑
    ...
    return {
        "loop_count": state["loop_count"] + 1,
        "messages": state["messages"] + [f"[triage] 第 {state['loop_count'] + 1} 次研判"]
    }
```

**loop_count 阈值的选择**：

| 阈值 | 适用场景 | 代价 |
|------|---------|------|
| 2 | 高频告警，成本敏感 | 可能精度不足 |
| 3 | 通用场景（推荐） | 平衡质量与成本 |
| 5 | 高价值告警，愿意多投入 | 成本较高 |

### 8.2 Token 预算控制

LLM API 按输入+输出的 Token 总数计费。上下文越长，成本越高，且超过模型上下文限制后会报错。

```python
MAX_TOKENS = 6000  # 留 2000 给输出，预留安全边界

def trim_messages(messages: list, max_tokens: int = MAX_TOKENS) -> list:
    """
    按 Token 预算截断消息历史。
    保留系统提示 + 最近 N 条对话，过早对话被截断。
    """
    # 估算 Token：中文按字符数，英文按空格分隔词数
    def estimate_tokens(text: str) -> int:
        chinese_chars = sum(1 for c in text if '\u4e00' <= c <= '\u9fff')
        english_words = len(text.split())
        return int(chinese_chars * 1.5 + english_words * 1.3)

    total = sum(estimate_tokens(m) for m in messages)
    if total <= max_tokens:
        return messages

    # 截断：从最早的消息开始删除，直到低于预算
    trimmed = list(messages)
    while total > max_tokens and len(trimmed) > 2:
        removed = trimmed.pop(0)
        total -= estimate_tokens(removed)

    return trimmed

# 在每个节点入口处调用
def triage_node(state: AlertState) -> AlertState:
    trimmed_messages = trim_messages(state["messages"])
    ...
```

**Token 成本优化的三板斧**：

| 策略 | 效果 | 实现难度 |
|------|------|---------|
| Redis 语义缓存（SHA256 哈希键） | 命中则不调用 LLM，节省 30-50% | 中 |
| Prompt 历史压缩（MAX_TOKENS 截断） | 减少输入 Token，节省 10-30% | 低 |
| Tag 路由分层（cheap/fast/powerful） | 简单任务用便宜模型 | 中 |

- A/B 测试不同 Prompt 版本的数据基础

### 8.4 Pydantic Schema Validation：结构化输出的保障

```python
from pydantic import BaseModel, Field, field_validator
from typing import Literal

class TriageOutput(BaseModel):
    severity: Literal["P0", "P1", "P2", "P3"]
    verdict: Literal["true_positive", "false_positive", "need_review"]
    confidence: float = Field(ge=0.0, le=1.0, description="置信度 0-1")
    mitre_tactics: list[str] = Field(min_length=1, max_length=5)

    @field_validator("mitre_tactics")
    @classmethod
    def validate_tactics(cls, v: list[str]) -> list[str]:
        """验证 MITRE Tactic ID 格式（必须是 TAxxxx 格式）"""
        valid_tactics = {"TA0001", "TA0002", "TA0003", "TA0004", "TA0005",
                         "TA0006", "TA0007", "TA0008", "TA0009", "TA0011"}
        for t in v:
            if t not in valid_tactics:
                raise ValueError(f"无效的 Tactic ID: {t}，参考 MITRE ATT&CK")
        return v

# 绑定到模型，强制结构化输出
structured_model = model.with_structured_output(TriageOutput)
```

⚠️ **注意**：Pydantic 验证失败会抛出异常。在 Agent 循环中，建议配合 try-except 降级处理：

```python
def triage_node(state: AlertState) -> AlertState:
    try:
        result = structured_model.invoke(state["messages"][-1].content)
        return {"severity": result.severity, "triage_result": result.verdict}
    except ValidationError as e:
        # Pydantic 验证失败，降级为宽松解析
        return {"triage_result": "need_review", "messages": state["messages"] + [f"结构化输出失败：{e}"]}
```

## 九、实战：构建 SOC 告警分析 Agent

### 9.1 需求分析

构建一个 SOC 告警分析 Agent，自动完成以下流程：

```
收到告警（SSH 暴力破解）
    │
    ▼
[1. triage] ── 判断是否为真实攻击 ──
    │                              │
    │ triage_result=true_positive  │ triage_result=false_positive
    ▼                              ▼
[2. enrich] ── 丰富上下文（IP情报 + 资产）──┐
    │                                          │
    │ loop_count < 3 ──────────────────────────┤
    │ (情报不足，回到 triage 重评)             │
    │                                          │
    ▼                                          ▼
[3. escalate] ── 高危升级人工 ──→ 人工审核 ──→ [4. auto_action]
    │                              │
    │                              │ 人工拒绝
    │                              ▼
    │                         [5. auto_close]
    │                              │
    ▼                              ▼
[4. auto_action] ── 自动处置（封锁IP）──→ END
[5. auto_close] ── 关闭告警，生成报告 ──→ END
```

### 9.2 完整代码实现

```python
"""
SOC 告警分析 Agent
从需求到代码的完整实现
"""
from typing import TypedDict, Annotated
from langgraph.graph import StateGraph, START, END
from langgraph.checkpoint.memory import MemorySaver
from langgraph.prebuilt import ToolNode
from langchain_openai import ChatOpenAI
from langchain_core.prompts import PromptTemplate
from langchain_core.output_parsers import JsonOutputParser
from langchain_core.tools import tool
from pydantic import BaseModel
import operator

# ─────────────────────────────────────────────
# 第1步：定义工具
# ─────────────────────────────────────────────

@tool
def lookup_ip_reputation(ip: str) -> dict:
    """查询 IP 威胁情报评分（0-100），返回恶意行为标签列表。"""
    db = {
        "1.2.3.4": {"score": 92, "tags": ["malware-c2", "bruteforce"]},
        "1.2.3.5": {"score": 15, "tags": []},
    }
    return db.get(ip, {"score": 0, "tags": []})

@tool
def query_asset_info(hostname: str) -> dict:
    """查询主机资产信息：责任人、部门、关键程度（1-5）。"""
    db = {
        "db-prod-03": {"owner": "zhang.san", "dept": "基础设施部", "criticality": 5},
        "dev-workstation": {"owner": "li.si", "dept": "研发部", "criticality": 2},
    }
    return db.get(hostname, {"owner": "unknown", "dept": "unknown", "criticality": 1})

@tool
def block_ip(ip: str, reason: str) -> dict:
    """在防火墙封禁指定 IP（不可逆操作，建议先确认）。"""
    return {"status": "blocked", "ip": ip, "blocked_by": "SOC-Agent", "reason": reason}

@tool
def create_ticket(severity: str, title: str, description: str) -> dict:
    """创建 SOC 工单"""
    return {"ticket_id": f"TICKET-{hash(title) % 10000}", "status": "created"}

# 工具列表
tools = [lookup_ip_reputation, query_asset_info, block_ip, create_ticket]
tool_node = ToolNode(tools, handle_tool_errors=True)

# ─────────────────────────────────────────────
# 第2步：定义 State
# ─────────────────────────────────────────────

class AlertState(TypedDict):
    alert_id: str
    alert_type: str
    source_ip: str
    target_asset: str
    triage_result: str
    severity: str
    loop_count: int
    messages: Annotated[list[str], operator.add]
    human_decision: str | None
    mitre_tactics: list[str]

# ─────────────────────────────────────────────
# 第3步：定义节点函数
# ─────────────────────────────────────────────

def triage_node(state: AlertState) -> AlertState:
    """
    triage_node：研判告警，判断是否为真实攻击。

    工作原理：
    1. 接收当前 state（包含告警信息、历史消息）
    2. 调用 LLM 分析告警特征
    3. 返回部分更新（只更新 triage_result 和 severity）
    """
    prompt = f"""分析以下 SOC 告警，判断是否为真实攻击：

告警类型：{state['alert_type']}
源 IP：{state['source_ip']}
目标资产：{state['target_asset']}

请输出 JSON：{{"verdict": "true_positive" 或 "false_positive",
         "severity": "P0"/"P1"/"P2"/"P3",
         "reason": "判断依据"}}
"""
    # 这里简化处理，实际生产用 with_structured_output
    messages = state["messages"] + [f"[triage] 分析 {state['alert_id']}"]
    return {
        "triage_result": "true_positive" if "malware" in state['source_ip'] else "need_review",
        "severity": "P2",
        "messages": messages,
        "loop_count": state["loop_count"] + 1,
    }

def enrich_node(state: AlertState) -> AlertState:
    """
    enrich_node：丰富告警上下文。

    实际生产中，这里会：
    1. 调用 lookup_ip_reputation（源 IP 威胁情报）
    2. 调用 query_asset_info（目标资产信息）
    3. 查询历史告警（是否有类似攻击模式）
    """
    ip_info = lookup_ip_reputation.invoke({"ip": state["source_ip"]})
    asset_info = query_asset_info.invoke({"hostname": state["target_asset"]})

    # 关键资产 + 高威胁 IP = 升级
    if asset_info.get("criticality", 1) >= 5 and ip_info.get("score", 0) >= 80:
        enrich_result = "高危组合：关键资产 + 高威胁 IP，建议升级"
    else:
        enrich_result = f"IP 评分 {ip_info.get('score', 0)}，资产关键度 {asset_info.get('criticality', 1)}"

    return {
        "messages": state["messages"] + [f"[enrich] {enrich_result}"],
    }

def escalate_node(state: AlertState) -> AlertState:
    """
    escalate_node：升级告警到人工处理。

    触发条件：true_positive + P0/P1，或 enrich 后仍需确认。
    """
    ticket = create_ticket.invoke({
        "severity": state["severity"],
        "title": f"告警升级：{state['alert_id']}",
        "description": f"类型={state['alert_type']}, 源IP={state['source_ip']}"
    })
    return {
        "messages": state["messages"] + [
            f"[escalate] 已创建工单 {ticket['ticket_id']}，等待人工审批"
        ]
    }

def auto_action_node(state: AlertState) -> AlertState:
    """
    auto_action_node：自动处置（封锁恶意 IP）。

    前提条件：human_decision == "approved" 且 severity == "P2"。
    """
    result = block_ip.invoke({
        "ip": state["source_ip"],
        "reason": f"自动化处置：{state['alert_id']} 告警升级批准"
    })
    return {
        "messages": state["messages"] + [f"[auto_action] {result['status']}: {state['source_ip']} 已封禁"]
    }

def auto_close_node(state: AlertState) -> AlertState:
    """auto_close_node：关闭误报告警。"""
    return {
        "messages": state["messages"] + [f"[auto_close] 告警 {state['alert_id']} 已关闭"]
    }

# ─────────────────────────────────────────────
# 第4步：构建图 + 条件边路由
# ─────────────────────────────────────────────

def route_after_triage(state: AlertState) -> str:
    """triage 后的路由逻辑——这是整个 Agent 的决策核心"""
    # 循环超限，强制关闭
    if state["loop_count"] >= 3:
        return "auto_close"

    # 真实攻击 → 升级人工审核
    if state["triage_result"] == "true_positive":
        return "escalate"

    # 需审核 → 等待人工
    if state["triage_result"] == "need_review":
        return "enrich"  # 先丰富情报再评

    # 其他（兜底）
    return "enrich"

def route_after_escalate(state: AlertState) -> str:
    """升级后的路由——根据人工决策决定下一步"""
    decision = state.get("human_decision")
    if decision == "approved":
        return "auto_action"
    elif decision == "rejected":
        return "auto_close"
    return "auto_close"  # 超时默认关闭

# 构建图
graph = StateGraph(AlertState)

# 注册节点
graph.add_node("triage", triage_node)
graph.add_node("enrich", enrich_node)
graph.add_node("escalate", escalate_node)
graph.add_node("auto_action", auto_action_node)
graph.add_node("auto_close", auto_close_node)

# 入口
graph.set_entry_point("triage")

# 普通边
graph.add_edge("enrich", "triage")      # 丰富情报后回到 triage
graph.add_edge("auto_action", END)
graph.add_edge("auto_close", END)

# 条件边（最重要的路由逻辑）
graph.add_conditional_edges(
    source_node="triage",
    path=route_after_triage,
    path_map={
        "escalate": "escalate",
        "enrich": "enrich",
        "auto_close": "auto_close",
    }
)

graph.add_conditional_edges(
    source_node="escalate",
    path=route_after_escalate,
    path_map={
        "auto_action": "auto_action",
        "auto_close": "auto_close",
    }
)

# ─────────────────────────────────────────────
# 第5步：编译 + 执行
# ─────────────────────────────────────────────

checkpointer = MemorySaver()
app = graph.compile(checkpointer=checkpointer)

# 打印图结构
app.get_graph().print_ascii()

# 场景1：恶意 IP 攻击关键资产 → escalate → auto_action
print("\n=== 场景1：高危告警 ===")
config = {"configurable": {"thread_id": "test-001"}}

# 第1次调用：执行到 escalate 后暂停（因为没绑定 interrupt，实际会执行完）
result1 = app.invoke({
    "alert_id": "alert-001",
    "alert_type": "SSH_BRUTEFORCE",
    "source_ip": "1.2.3.4",
    "target_asset": "db-prod-03",
    "triage_result": "unknown",
    "severity": "P3",
    "loop_count": 0,
    "messages": [],
    "human_decision": None,
    "mitre_tactics": [],
}, config)

print(f"最终结果：{result1['messages']}")
print(f"最终 severity：{result1['severity']}")

# 场景2：误报告警 → auto_close
print("\n=== 场景2：误报告警 ===")
config2 = {"configurable": {"thread_id": "test-002"}}

result2 = app.invoke({
    "alert_id": "alert-002",
    "alert_type": "SSH_BRUTEFORCE",
    "source_ip": "1.2.3.5",   # 正常 IP
    "target_asset": "dev-workstation",
    "triage_result": "unknown",
    "severity": "P3",
    "loop_count": 0,
    "messages": [],
    "human_decision": None,
    "mitre_tactics": [],
}, config2)

print(f"最终结果：{result2['messages']}")
```

### 9.3 代码执行流程详解

**场景1执行流程**（恶意 IP + 关键资产）：

```
invoke(alert-001)
  │
  ▼
triage（loop_count=0 → 更新为1）
  │ triage_result="true_positive"
  ▼ route_after_triage
escalate（创建工单）
  │ human_decision=None
  ▼ route_after_escalate
auto_close（兜底，关闭告警）
  │
  ▼ END
最终：告警已关闭，工单已创建
```

**场景2执行流程**（正常 IP → 误报）：

```
invoke(alert-002)
  │
  ▼
triage（loop_count=0 → 1）
  │ triage_result="need_review"（恶意标签不在该 IP）
  ▼ route_after_triage
enrich（查询 IP 情报：score=15，关键度低）
  │
  ▼
triage（loop_count=1 → 2，再次研判）
  │ triage_result 仍为 need_review
  ▼ route_after_triage
enrich（再次丰富）
  │
  ▼
triage（loop_count=2 → 3，达到阈值）
  │ loop_count >= 3
  ▼ route_after_triage
auto_close（循环超限，强制关闭）
  │
  ▼ END
最终：告警已关闭
```

### 9.4 常见坑与最佳实践

| 坑 | 说明 | 解法 |
|----|------|------|
| State 定义遗漏字段 | 节点返回的字段在 State 中未定义，运行时报错 | 定义 State 时一次性列全，预留扩展字段 |
| loop_count 不递增 | 只在 triage 加了计数，enrich 后回到 triage 忘了加 | 在所有可能产生循环的节点里都加 |
| messages 覆盖而非追加 | 普通 `list` 类型返回值会覆盖历史 | 用 `Annotated[list, operator.add]` |
| thread_id 写死 | 不同告警用相同 thread_id，状态互相污染 | 每个告警用唯一 thread_id（alert_id） |
| 工具返回未处理 | 工具节点返回结果后，LLM 节点没有消费这些结果 | 确保图的边连接正确，形成完整循环 |
| interrupt 忘记绑定 checkpointer | interrupt_before/after 必须配合 checkpointer 才能工作 | 两者必须同时配置 |
| 关键操作（封禁 IP）没有人工审核 | auto_action 直接执行了高危操作 | 对不可逆操作配置 interrupt_before |

## 附录：面试高频追问汇总

### Q: LangGraph 和 LangChain Agent 的区别是什么？

**答案**：LangChain 的 `initialize_agent` 是"内置 Agent"，黑盒化程度高，定制灵活性低。LangGraph 用**有向图**显式建模 Agent 流程，每个节点是显式函数、每条边是显式路由、状态是显式定义。好处：可观测、可干预、可测试。代价：需要更多代码。**简单任务用 LangChain Agent，复杂多步流程用 LangGraph。**

### Q: 为什么要用 Checkpointer 而不是直接用数据库？

**答案**：Checkpointer 专门为 LangGraph 的状态机语义设计（部分更新 merge、checkpoint 序列化、thread 隔离）。直接用数据库需要自己实现这些逻辑，开发量大。生产环境可以在 Checkpointer 底层接 PostgreSQL（`PostgresSaver`），实现持久化的同时享受 LangGraph 的状态管理能力。

### Q: loop_count 机制和 LLM 的自适应停止相比，优势在哪里？

**答案**：确定性 vs 不确定性的区别。loop_count 是**硬性约束**，3 次就 3 次，任何情况下都不会超过。LLM 自适应停止依赖模型的自我判断，同一输入可能不同次数，不稳定。生产环境（尤其是 SOC 场景）必须用确定性约束，不能让 LLM 自主决定跑多少轮。
