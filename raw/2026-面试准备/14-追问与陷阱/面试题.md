### 状态回退与置信度
```
告警研判中，如果后续节点认为前面节点的判断有问题，是否支持回退？怎么设计的？
追问：
- loop_count 到达上限后的兜底策略是什么？
```
**[10秒结论]** LangGraph 通过 Checkpointer 的状态快照支持回退，但回退需要外部触发；loop_count 到达上限后强制路由到 auto_close 节点，保证图有终结。

**[90秒拆解]**
- **回退设计**：Checkpointer 保留完整状态历史，外部系统（如前端）可以判断置信度后调用 `update_state` 修改状态并重新触发后续节点
- **实际场景**：enrich 节点发现 IP 情报评分比 triage 节点预估的严重 → 通过 `interrupt_before` 暂停，外部决策后 `invoke` 继续或重新触发 triage
- **loop_count 兜底**：`route_after_triage` 中检查 `loop_count >= 3` 时强制路由到 `auto_close`，不再进入 enrich

**[20秒收尾]** 回退是异常流程的兜底，不是正常设计，正常情况应通过 prompt 质量和置信度阈值保证首次研判准确。

### 告警风暴场景
```
当 SOC 平台同时涌入大量告警（比如万台主机同时被扫描），你的 Agent 是串行还是并行处理？有没有做批量节流或优先级队列？
```
**[10秒结论]** 并行处理 + 优先级队列：同一批告警并行处理不同告警（利用异步 I/O），但单个 Agent 实例有并发限制；用 Redis 优先级队列（P0 优先处理）做批量节流。

**[90秒拆解]**
- **并行策略**：不同告警之间完全并行（每条告警独立触发 Agent），同一告警内部的状态机节点串行执行
- **批量节流**：Redis 优先级队列，`BRPOPLPUSH` 原子操作将告警移入处理队列；超过 100 条告警/分钟时暂停拉取新告警，优先处理 P0
- **实现**：用 Redis Sorted Set 存储告警优先级（score=告警级别），每次取出 score 最小的 P0 告警处理

**[20秒收尾]** 告警风暴是最考验系统吞吐量的场景，异步 + 优先级队列 + 并发限制三招组合应对。

### Prompt版本管理
```
你们有多少个 prompt？每个节点的 prompt 是怎么管理、版本化的？改 prompt 后有没有做自动化回归测试？
```
**[10秒结论]** 用版本化字典管理 Prompt 模板（如 `PROMPTS["v1.1"]`），通过 `state["prompt_version"]` 动态选择；改 Prompt 后通过影子模式（shadow mode）对比新旧输出差异，再决定是否上线。

**[90秒拆解]**
- **版本化管理**：`PROMPTS = {"v1.0": "...", "v1.1": "..."}`，通过 state 字段切换，支持灰度回滚
- **自动化回归测试**：改动 Prompt 后，用 100 个已标注样本批量推理，对比新旧 Prompt 的输出差异；输出差异率 > 5% 时触发人工 review
- **Schema 版本同步**：Prompt 和对应的 Pydantic Output Schema 同步版本号，确保 LLM 输出始终可解析

**[20秒收尾]** Prompt 改错是 Agent 系统最常见的故障来源，自动化回归测试是生产级 Prompt 管理的必备机制。

### RAG-Agent耦合方式
```
RAG 检索结果是怎么传给 Agent 的？是作为 Tool 的 Context，还是拼到 State 里？检索时机是每轮都查还是按需触发？
```
**[10秒结论]** RAG 作为 Tool 接入 Agent（LangGraph `bind_tools`），按需触发而非每轮都查；Tool 的 description 作为 LLM 决定是否调用的依据。

**[90秒拆解]**
- **接入方式**：`lookup_rag_knowledge` 定义为 Tool，LLM 自主决定何时调用
- **触发时机**：不是每轮都查，而是 LLM 发现当前上下文不足以回答时（通过 prompt 提示"可用知识库"）才触发 Tool
- **State vs Context**：RAG 结果作为 Tool 返回值追加到 messages 中，而非直接修改 State，保证 State 最小化
- **缓存**：高频 query 的 RAG 结果缓存到 Redis，TTL=1h，减少重复检索开销

**[20秒收尾]** RAG 作为 Tool 接入比把检索结果直接塞 State 更合理，避免 State 膨胀和 token 浪费。

### BGEReranker部署细节
```
bge-reranker-v2-m3 是 GPU 推理还是 CPU？Qdrant 和 Reranker 之间的网络延迟是多少？有没有做缓存？
```
**[10秒结论]** bge-reranker-v2-m3 用 GPU 推理（RTX 4090×1，INT8 量化）；Qdrant 和 Reranker 部署在同一台机器（内网 0ms 延迟），热结果缓存到 Redis。

**[90秒拆解]**
- **GPU vs CPU**：GPU 推理延迟 ~10ms/batch-32，CPU 推理延迟 ~100ms/batch-32，差距 10x，GPU 是生产必选
- **部署架构**：Qdrant（6333）+ Reranker（CUDA）部署在同一台高配机器（32C/64G/RTX 4090 24G），避免网络开销
- **缓存**：高频 query 的 Reranker 结果（query hash → top-3 结果）缓存 Redis，TTL=3600s，命中率约 20%
- **INT8 量化**：显存从 24G → 12G，延迟降低 30%，精度损失 < 1%

**[20秒收尾]** Reranker 是整个 RAG 链路中延迟贡献最大的环节（比向量检索慢 10x），GPU 推理 + 缓存是核心优化手段。

### Chunking策略
```
SOP、历史案例、漏洞库各自的 chunk 策略一样吗？有没有做语义分句（避免一句话被切断）？
```
**[10秒结论]** SOP 按层级结构（保留步骤链）、历史案例按事件边界、漏洞库按 CVE 条目；语义分句用正则 `\n\n` 切分段落，避免单句被切断。

**[90秒拆解]**
- **SOP**：按 section 级别 chunk，保留完整步骤链；同时生成 doc-level summary chunk
- **历史案例**：以事件边界切分，每个案例独立成块，避免多事件混杂导致混淆
- **漏洞库**：按 CVE 条目粒度，每个 CVE 成独立 chunk，metadata 包含 CVE ID 和 CVSS 评分
- **语义分句**：先按 `\n\n` 切分段落，再按 `max_tokens=512` 合并；单句被切断用 overlap=64 缓解

**[20秒收尾]** 不同类型的知识需要不同的 chunk 策略，通用策略（纯固定窗口）会牺牲知识类型的特殊性。

### 知识库更新机制
```
告警处置完后会写回知识库吗？知识库的增量更新和重排序是怎么做的？有没有做版本 diff？
```
**[10秒结论]** 告警处置完成后，分析师的选择（升级/降级/确认）回流写入选中案例的 metadata 标签；增量更新通过 content hash 判断是否变化，变化则 upsert + 标记重排序；版本 diff 记录变更日志。

**[90秒拆解]**
- **写回机制**：分析师确认告警为 true_positive 时，自动将该告警上下文（告警描述、研判结果、处置动作）追加写入历史案例集（经审核后）
- **增量更新**：content hash 比对，未变化跳过写入；变化则 upsert 新向量 + 标记 `needs_rerank`
- **版本 diff**：记录变更内容（哪个 SOP 更新了、新增了哪个案例），用于审计和问题溯源
- **定时全量重建**：每月执行一次全量重建，解决向量空间漂移问题

**[20秒收尾]** 知识库是活系统，处置完的告警是宝贵的经验积累，经审核后写回知识库是持续提升 RAG 质量的核心手段。

### Embedding模型选型
```
用的哪个 Embedding 模型？为什么选它？有没有对比过 text-embedding-3-large 或国产模型？
```
**[10秒结论]** 选 BAAI/bge-large-zh-v1.5（1024维），中文效果好、支持本地部署、无 API 费用；对比过 text-embedding-3-large（通用强但 API 费用高）和 text2vec（精度弱于 bge）。

**[90秒拆解]**
- **选型对比**：
  - BGE-large-zh-v1.5：中文效果好，1024 维，MTEB 精度 ~65.5，本地部署无 API 成本
  - text-embedding-3-large：通用精度高，3072 维，OpenAI API 按 token 收费（贵）
  - text2vec-base-chinese：轻量但精度弱于 bge
- **本地部署优势**：数据不出域、无 API 延迟、无 token 计费，专有云环境首选
- **国产模型**：text2vec、m3e 精度通常弱于 bge，且生态（量化工具、评测基准）不如 bge 完善

**[20秒收尾]** Embedding 模型选型决定 RAG 的召回上限，BGE-large-zh-v1.5 是当前中文生产环境性价比最优选择。

### 向量检索调优
```
Qdrant 的 hnsw:ef / hnsw:m 参数怎么配的？召回率怎么测量？有没有做过 Bad Case 分析？
```
**[10秒结论]** 生产参数：hnsw:m=16、ef_construct=200、hnsw_ef=128；召回率通过 Recall@K 测量；Bad Case 分析发现 CVE 编号检索和跨文档关系丢失是主要瓶颈。

**[90秒拆解]**
- **参数设置**：m=16（内存与精度平衡）、ef_construct=200（构建精度）、hnsw_ef=128（搜索精度）
- **召回率测量**：Recall@20 > 0.85 为达标；每周用 100 个标注 query 评估
- **Bad Case 分析**：
  - CVE 编号检索失败（tokenizer 数字切分问题）→ 预处理保留 CVE 格式
  - 跨文档关系丢失（chunk 边界切断）→ overlap=64 + metadata group_key 关联
  - 语义相近但实体不同的案例混淆 → 事件级 chunk 隔离

**[20秒收尾]** Bad Case 分析是 RAG 优化的核心驱动，每发现一个 bad case 就修复一类问题，持续迭代才能逼近 90% 准确率目标。

### 模型Fallback链
```
cheap 模式调用失败会自动切到 fast 吗？还是直接报错？这个降级策略是怎么配置的？
```
**[10秒结论]** 默认开启降级链：cheap 失败 → 自动切 fast → fast 失败 → 切 powerful；降级策略通过配置中心统一管理，fallback chain 最多 3 级。

**[90秒拆解]**
- **配置方式**：`router.route({"cheap"})` 时，自动从注册表中找同 Tag 下的其他备选模型组成链
- **降级顺序**：cheap (gpt-4o-mini) → fast (qwen-plus) → powerful (gpt-4o)，成本递增
- **不降级的场景**：显式指定 `fallback_enabled=False`（如某些严格审计场景需明确记录调用了哪个模型）
- **降级监控**：每次降级记录日志和指标，降级率 > 5% 触发告警（说明主模型可能有问题）

**[20秒收尾]** Fallback 链保证服务可用性，但降级是有代价的（成本↑、延迟↑），所以降级率本身也是要监控的指标。

### Token成本监控
```
你们有没有统计过每月的 Token 消耗？cheap/fast/powerful 各占多少比例？有没有做成本预警？
```
**[10秒结论]** Token 消耗记录到 SQLite，按 mode（cheap/fast/powerful）分组统计月度和日度报表；每日预算告警；cheap 流量应占 > 70%。

**[90秒拆解]**
- **记录字段**：timestamp / model / mode / prompt_tokens / completion_tokens / cost_usd / latency_ms / cached
- **月度报表**：
  - cheap ~60-70% 的流量（分类、格式化、提取）
  - fast ~20-30% 的流量（对话、摘要）
  - powerful ~5-10% 的流量（复杂推理、代码生成）
- **成本预警**：每日消耗超过 $100（可配置）触发企业微信告警
- **分析重点**：cheap 占比是否足够高、降级率是否异常、缓存命中率是否下降

**[20秒收尾]** Token 成本管理的目标不是越省越好，而是在保证质量的前提下用合适的模型做合适的事，cheap 占比是架构合理性的指标。

### Ollama私有化部署
```
Ollama 跑在什么硬件上？多用户并发时怎么控制队列和超时？有没有遇到显存溢出？
```
**[10秒结论]** 部署在 32C/64G/RTX 4090 24G 的高配机器；并发通过 asyncio.Semaphore 控制；OOM 时自动降级到云端 API。

**[90秒拆解]**
- **硬件配置**：qwen2.5:7b（Q4 量化）→ 8GB VRAM，4 并发；qwen2.5:14b（Q4 量化）→ 16GB VRAM，2 并发
- **并发控制**：`Semaphore(max_concurrent=2)` 限制并发请求数，超出排入队列
- **OOM 处理**：
  ```python
  except (OllamaOOMError, OllamaTimeoutError):
      return await fallback_cloud.chat(messages)  # 自动降级到云端 API
  ```
- **显存溢出**：多用户并发时显存碎片导致 OOM，通过减少并发数 + 定期重启 Ollama 服务缓解

**[20秒收尾]** Ollama 的问题是显存碎片导致的 OOM，精打细算的并发控制是生产部署的关键。

### SchemaValidation
```
OpenAI / 通义千问 / Ollama 混用时，输出 schema 不一致怎么处理？有没有用 Pydantic 做输出校验？失败后怎么降级？
```
**[10秒结论]** Pydantic BaseModel 定义输出 Schema，失败后三步兜底：正则提取 JSON → 字段级正则提取 → 部分填充 + is_fallback=True 标记；失败率 > 5% 的模型触发告警。

**[90秒拆解]**
- **Pydantic 校验**：`model_validate_json()` 解析 LLM 输出，ValidationError 触发降级流程
- **三步降级**：① JSON 正则提取 `{...}` → ② 字段级正则 `(?:"field":")([^",\n}]+)` → ③ 部分填充默认值 + `is_fallback=True`
- **监控**：每个模型记录 validation 失败率，超过 5% 触发告警并可能触发模型切换
- **Pydantic field_validator**：在 Schema 层对枚举值做校验，拒绝不合规枚举值输出

**[20秒收尾]** 多模型混用时输出格式是最大的不稳定因素，Pydantic 校验层是生产系统的安全网。

### DSL定义
```
剧本 DSL 长什么样？JSON Schema？Python class？还是 YAML？请举例一个简单剧本的定义。
```
**[10秒结论]** YAML DSL 定义剧本结构（节点+边+变量），编译层将 YAML AST 转换为 Python Class，再由 LangGraph 执行。

**[90秒拆解]**
- **YAML 示例**（钓鱼邮件封禁）：
  ```yaml
  name: phishing_lockdown
  nodes:
    - id: extract_ioc
      type: action
      action: extract_ioc
    - id: human_approve
      type: approval
      timeout: 3600
      approver_role: soc_analyst
    - id: block_domain
      type: action
      action: block_email_sender
      condition: "{{ human_approve.approved == true }}"
  ```
- **编译层**：`yaml.safe_load()` → `PlaybookNode` dataclass → LangGraph `StateGraph` 节点注册
- **边推断**：线性剧本从节点顺序推断 edges；复杂剧本需显式定义 conditional_edges

**[20秒收尾]** DSL 是 SOAR 平台的核心抽象，YAML 让安全分析师能编写剧本，LangGraph 让剧本能执行。

### 执行上下文隔离
```
多个剧本并行执行时，怎么保证互相不干扰（比如两个剧本同时想封禁同一个 IP）？
```
**[10秒结论]** `ResourceLockManager` 实现资源级互斥锁（IP/域名/主机）；以 `resource_type:resource_id` 为键获取 `asyncio.Lock`；生产环境用 Redis 分布式锁保证多实例隔离。

**[90秒拆解]**
- **锁实现**：
  ```python
  key = f"ip:{ip_address}"
  async with lock_mgr.acquire("ip", ip, playbook_instance_id, timeout=30):
      # 封锁操作
  ```
- **分布式场景**：用 Redis `SET key instance_id NX EX ttl` 实现跨进程锁，超时自动释放防止死锁
- **超时处理**：`ResourceLockTimeout` 异常抛出，外部可选择重试或降级

**[20秒收尾]** 资源锁是防止多剧本并发冲突的必备机制，没有锁的 SOAR 系统在并发场景下会产生难以调试的竞态问题。

### 剧本执行超时
```
一个剧本最长能跑多久？超时后是挂起还是中止？中止后是否有告警通知？
```
**[10秒结论]** 默认剧本超时 7200s（2 小时）；超时后剧本状态更新为 `aborted`，后台协程触发通知 SOC Lead + 生成事故报告；中止后不自动恢复，需人工决定是否重跑。

**[90秒拆解]**
- **超时监控**：后台协程每 10s 检查一次剧本执行时间，超过 deadline 则触发中止
- **中止后的告警**：企业微信通知 SOC Lead：`剧本 {name} 执行超时，已自动中止`
- **事故报告**：自动生成执行日志摘要（各节点执行时间、失败节点、超时原因）
- **恢复流程**：人工 review 事故报告，决定是跳过超时节点继续还是重新执行

**[20秒收尾]** 剧本超时是 SOAR 系统中最需要人工介入的异常，中止 + 通知 + 报告三步缺一不可。

### 剧本开发周期
```
一个新剧本从需求提出到上线，平均要多久？有没有做自动化剧本测试？
```
**[10秒结论]** 简单剧本（5 步以内，无需复杂条件分支）1-2 天；复杂剧本（含多并行分支、审批节点、事务性执行）1-2 周；自动化测试覆盖 pytest + YAML Schema 校验。

**[90秒拆解]**
- **开发流程**：需求提出 → YAML 草稿设计 + 评审 → JSON Schema 校验 → pytest 单元测试（mock 外部 API）→ 集成测试（连真实 SIEM/EDR 模拟环境）→ 安全评审（审批节点配置审查）→ 灰度发布（影子模式运行 48h）→ 上线
- **自动化测试**：
  - pytest 对节点函数做单元测试（mock 所有外部 API）
  - YAML Schema 校验（确保 DSL 合法）
  - 死节点检测（确保图有终结）
- **质量指标**：回滚触发频率 < 2%，执行成功率 > 98%

**[20秒收尾]** 剧本开发的瓶颈往往在安全评审环节，提前让安全团队介入设计阶段可大幅缩短评审周期。

### SPL-DSL生成准确率
```
NL → SPL 的准确率大概多少？哪些类型的查询容易出错？有专门做过 Bad Case 分析吗？
```
**[10秒结论]** Few-shot 路线覆盖约 80% 常见查询（统计查询、搜索过滤、时间范围），准确率约 85%；复杂嵌套查询（join、transaction）和长窗口时间查询是主要错误来源。

**[90秒拆解]**
- **准确率分布**：
  - 简单搜索查询（`index=xxx | where ...`）：~92%
  - 统计聚合查询（`stats count by ... | where ...`）：~85%
  - 时间窗口查询：~80%（自然语言时间表达如"上周"易误解析）
  - 嵌套 join 查询：~65%（跨 index 关联最难）
- **Bad Case 分析**：每月 review 100 条生成结果，发现错误主要在"时间范围表达"和"OR/AND 逻辑混淆"
- **改进方法**：增加时间表达专项 few-shot 示例；复杂查询兜底到模板选择器（模板 + 参数填充）

**[20秒收尾]** NL to SPL 的瓶颈是复杂嵌套查询，简单查询已经很稳定，复杂查询用模板兜底是务实的工程选择。

### 查询结果校验
```
生成的 SPL 如果语法错误，怎么处理？有没有做语法校验和白名单过滤？
```
**[10秒结论]** AST 语法校验层拒绝语法错误和危险操作；白名单过滤 index/field/command；语法错误时返回用户友好提示而非原始错误。

**[90秒拆解]**
- **AST 校验**：ANTLR 生成 SPL AST，拒绝不合语法或包含危险命令（`script`、`delete`、`index`）的语句
- **白名单过滤**：
  - index 白名单：用户只能查询自己有权访问的 index
  - field 白名单：限制返回字段数量，防止 `SELECT *`
  - 时间窗口限制：最多 30 天
- **错误处理**：语法错误 → 返回"查询语法有误，请尝试简化" + 可选的建议修正；恶意操作 → 记录审计日志 + 返回"权限不足"

**[20秒收尾]** 查询安全是 NL to SPL 的最后一道防线，AST 校验 + 白名单是标准配置。

### 大列表内存优化
```
告警分析页面渲染上千条告警时，有没有遇到内存问题？怎么优化的？（虚拟滚动、分页、懒加载？）
```
**[10秒结论]** react-window 做虚拟滚动（只渲染可见区域的 20 个 DOM 节点）；分页（每页 50 条）；HEAP 使用稳定控制在 150MB 以内。

**[90秒拆解]**
- **react-window**：`FixedSizeList` / `VariableSizeList` 按行高渲染可见行，10000 行 → 20 个 DOM 节点
- **分页**：API 层支持 cursor pagination（基于 last_id 而非 OFFSET，避免深度分页 O(n) 问题）
- **内存控制**：`useMemo` 缓存每页数据；大对象及时释放（避免闭包引用导致内存泄漏）
- **监控**：Chrome DevTools Memory Profiler + React DevTools Profiler 定位 re-render 和内存泄漏

**[20秒收尾]** 大列表优化的优先级是：先虚拟滚动（收益最大），再分页（覆盖极端情况），最后才是 memo 类小优化。
