### 数据仓库分层
```
ODS/DWD/DWS/ADS 四层各自的职责是什么？为什么要分层？不分层行不行？
追问：
- ODS 层的数据和源表是什么关系？
- DWD 层做哪些清洗？
```
**[10秒结论]** ODS（原始数据）→ DWD（清洗明细）→ DWS（轻量汇总）→ ADS（应用数据）；分层实现数据复用、职责清晰、SQL 复杂度可控；不分层会让所有逻辑混在一起，难以维护。

**[90秒拆解]**
- **ODS 层**：Operational Data Store，原始数据层，1:1 或 N:1 映射源表，不做任何清洗，保留原始凭证便于回溯；partition 按加载时间划分
- **DWD 层**：Data Warehouse Detail，明细数据层，空值填充、异常值处理、字段类型统一、编码规范化、时间格式标准化
- **DWS 层**：Data Warehouse Summary，轻量汇总宽表，按业务主题+时间周期组织，支持下游报表直接查询
- **ADS 层**：Application Data Store，应用数据层，直接面向 BI 报表或业务系统
- **分层价值**：血缘清晰（出问题能定位是原始数据还是计算逻辑）、复用性高（DWD 可被多个 DWS 复用）、SQL 复杂度可控

**[20秒收尾]** 小团队两层（ODS + ADS）可以跑，但 50+ 表的复杂业务必须三层起步，否则 ADS SQL 会膨胀到无法维护。

### DWD 层清洗
```
DWD 层清洗的核心逻辑是什么？空值怎么处理？异常值怎么检测？
追问：
- 脱敏怎么做？手机号脱敏用什么策略？
- DWD 层 SQL 异常值检测的典型写法？
```
**[10秒结论]** DWD 清洗策略：业务含义上为 0 的字段填充 0（如未发生金额）、真实缺失保留 NULL（如未支付时间）；异常值通过 SQL 统计检测（金额 < 0、数量 > 1000、偏离均值 3σ）；隐私字段统一脱敏。

**[90秒拆解]**
- **空值策略**：
  - 应为 0 的字段：`fillna(0)`（未发生金额、数量）
  - 真实缺失：`保留 NULL`（未支付的支付时间）
- **脱敏规则**：
  - 手机号：`phone[:3] + '****' + phone[-4:]`（保留前三位和后四位）
  - 身份证：`前6位 + '**********' + 后4位`
- **异常值检测 SQL**：
  ```sql
  CASE
    WHEN order_amt < 0 THEN 'NEGATIVE_AMT'
    WHEN goods_cnt > 1000 THEN 'EXCESSIVE_QTY'
    WHEN ABS(order_amt - avg_amt) > 3 * std_amt THEN 'OUTLIER_PRICE'
  END
  ```
- **时间标准化**：统一为 UTC 时间戳（秒级），消除时区歧义

**[20秒收尾]** DWD 层是数仓质量的守门人，清洗做好了，下游 DWS/ADS 才能稳定。

### 缓慢变化维（SCD）
```
SCD Type 1/2/3 各自的特点是什么？拉链表怎么实现？
追问：
- 拉链查询 SQL 怎么写？
- 拉链表和快照表各自适合什么场景？
```
**[10秒结论]** Type 1 直接覆盖不保留历史；Type 2 增加新行（start_date/end_date），保留完整历史；Type 3 增加新列保留当前和历史值；拉链表是 Type 2 的工程实现。

**[90秒拆解]**
- **三种类型**：
  - Type 1：直接覆盖，不留历史。适用于邮编等不需要追溯的字段
  - Type 2：新增一行，start_date=变更日期，end_date='9999-12-31'，历史行 end_date 更新为变更日期
  - Type 3：同行列出 current 和 previous 值，最多保留两期
- **拉链查询**：`WHERE start_date <= target_date AND end_date > target_date`
- **快照 vs 拉链**：快照查询简单（`WHERE dt = target_date`）但存储成本高；拉链存储高效但查询需时间条件；经验法则：字段 < 20 个、查询频繁用快照；字段多、需历史趋势用拉链

**[20秒收尾]** SCD Type 2 拉链表是数仓面试高频题，理解 start_date/end_date 的语义就能写对查询。

### ETL 调度
```
APScheduler 怎么用？BlockingScheduler 和 AsyncIOScheduler 有什么区别？
追问：
- misfire_grace_time 是干什么的？
- APScheduler 进程重启后任务状态会丢吗？
```
**[10秒结论]** BlockingScheduler 用于独立批处理进程，AsyncIOScheduler 用于 FastAPI 异步服务内嵌调度；misfire_grace_time 容忍调度失位；JobStore 选数据库保证任务持久化。

**[90秒拆解]**
- **BlockingScheduler**：适用于独立 ETL 进程，调度器 start() 会阻塞主线程
- **AsyncIOScheduler**：适用于 FastAPI 异步服务内嵌，不阻塞事件循环
- **misfire_grace_time**：任务应触发但未触发（如进程休眠）时，容忍的时间窗口。如 3600s 容忍 1 小时内失位
- **任务状态持久化**：默认内存存储，进程重启后丢失；生产用 PostgreSQL/MySQL 作为 JobStore
- **常用触发器**：`CronTrigger(day='*', hour=2, minute=0)` = 每天凌晨 2 点执行

**[20秒收尾]** APScheduler 是 Python ETL 调度的首选，但内存存储是生产陷阱，必须配置持久化 JobStore。

### 多实例防重复
```
多实例部署时怎么防止同一任务被重复执行？分布式锁怎么实现？
追问：
- Redis 锁和数据库锁各自的特点？
- 锁的 TTL 怎么设置？
```
**[10秒结论]** Redis 分布式锁（`SET key val NX EX ttl`）防重复执行，TTL = 2× 任务预估执行时间；数据库 `SELECT ... FOR UPDATE` 是兜底方案。

**[90秒拆解]**
- **Redis 锁**：
  ```python
  acquired = redis.set(lock_key, instance_id, nx=True, ex=ttl)
  if not acquired: raise RuntimeError("任务已被其他实例持有")
  ```
- **TTL 设置**：2× 任务预估执行时间，最少 30 分钟；TTL 太短任务未完成锁已过期，TTL 太长进程崩溃后锁不释放
- **数据库锁**（兜底）：`SELECT ... FOR UPDATE` 获取悲观锁，进程崩溃后自动释放（数据库保证）
- **防重复语义**：分布式锁保证"同一时刻最多一个实例执行同一任务"，结合幂等设计（重复执行不产生副作用）

**[20秒收尾]** 容器多副本部署时，ETL 任务的幂等设计 + 分布式锁是多实例防重的标准组合。

### 增量抽取与 CDC
```
增量抽取的判断条件怎么选？时间戳 CDC 和 binlog CDC 各自的适用场景？
追问：
- CDC 是什么？
- 增量抽取时边界值用 `>` 还是 `>=`？
```
**[10秒结论]** 时间戳 CDC（`WHERE updated_at > '{last_run}'`）适合有 `updated_at` 字段的表，实现简单；binlog CDC（Debezium+Kafka）适合实时同步和捕获删除操作。

**[90秒拆解]**
- **时间戳 CDC**：依赖 `updated_at`/`create_time` 字段，`WHERE updated_at > '{last_run}' AND updated_at <= '{current_run}'`；优点是实现简单，缺点是无法捕获删除操作
- **binlog CDC**：读取 MySQL binlog/Redo Log，实时性强，能捕获插入/更新/删除；需要开启 binlog 和 Canal/Debezium；侵入性较高
- **边界值**：`>` 而非 `>=`，防止边界时间点数据被重复处理
- **判断条件优先级**：时间戳（有 updated_at）→ 自增 ID → 全量比对（不得已）

**[20秒收尾]** 日级批处理用时间戳 CDC 足够了；分钟级实时同步才需要 binlog CDC，Kafka + Debezium 是标准组合。

### 数据质量监控
```
数据质量监控有哪些核心指标？空值率、重复率怎么监控？
追问：
- 告警阈值怎么设计？
- 波动检测怎么实现？
```
**[10秒结论]** 核心指标：空值率 < 1%（常规）/ < 0.01%（核心）、重复率 < 0.1%、行数波动与近7日均值偏差 ±30%；质量检查 SQL 内嵌于 ETL 流程，结果写入 dwd_data_quality_log。

**[90秒拆解]**
- **空值率**：`COUNTIF(field IS NULL) / COUNT(*)`，常规表 < 1%，核心交易表 < 0.01%
- **重复率**：`COUNT(*) - COUNT(DISTINCT order_id)`，核心表必须为 0
- **波动检测**：
  ```sql
  CASE WHEN COUNT(*) < AVG_CNT * 0.7 OR COUNT(*) > AVG_CNT * 1.3
       THEN 'WARN' ELSE 'OK' END
  ```
- **告警阈值**：数据工程师和业务方共同确定，超阈值触发企业微信/Slack 告警

**[20秒收尾]** 数据质量监控要嵌入 ETL 流程而不是人工检查，等数据出问题了才发现代价太大。

### 数据血缘追踪
```
数据血缘是什么？血缘追踪有哪些应用场景？
追问：
- 血缘表怎么设计？
- 血缘在实际工作中有哪些价值？
```
**[10秒结论]** 数据血缘记录表/字段的上下游关系；核心应用：影响分析（上游变更评估下游影响）、数据溯源（报表异常溯源原始数据）、合规审计（GDPR 等法规要求）。

**[90秒拆解]**
- **血缘表设计**：
  ```sql
  source_table / source_column → target_table / target_column
  transform_rule / etl_job_name
  ```
- **应用场景**：
  - 影响分析：某上游字段要下线 → 快速列出所有受影响下游表和报表
  - 数据溯源：ADS 层报表数据异常 → 逆向追溯到 ODS 层原始记录
  - 合规审计：GDPR 要求数据处理透明度，血缘记录满足审计要求
- **自动注册**：通过 ETL 代码装饰器自动收集血缘元数据，而非人工维护

**[20秒收尾]** 血缘追踪是数据治理的基础能力，没有血缘的数据平台就是一本糊涂账。
