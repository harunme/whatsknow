## 1. 数据仓库分层

### 1.1 为什么需要分层？

不加分层的数据仓库就像把所有文件堆在一个抽屉里——无法复用、难以维护、查询效率低下。分层解决三个核心问题：

| 问题 | 分层如何解决 |
|------|-------------|
| 数据重复计算 | 公共汇总层（DWS）被多个应用共享，避免每个报表都从 ODS 重新算起 |
|  ETL 逻辑混乱 | 每层职责单一，ODS 只做贴源，DWD 做清洗，DWS 做聚合，ADS 做应用 |
| 问题追溯困难 | 数据出问题能快速定位到具体哪一层，而非在几百行 ETL 代码里大海捞针 |

### 1.2 四层架构详解

#### ODS 层（Operational Data Store，原始数据层）

**定位：** 源系统的"一模一样"拷贝，1:1 映射，保持数据原貌。

```sql
-- ODS 层表命名规范：ods_{源系统}_{表名}_{日期分表后缀}
CREATE TABLE ods_mysql_alert_event_20260410 (
    raw_data     STRING    COMMENT 'JSON 原始报文，保留所有字段',
    extract_time TIMESTAMP COMMENT '数据抽取时间',
    source_db    STRING    COMMENT '来源数据库实例'
) COMMENT 'SOC 告警事件原始数据层';
```

**为什么保留 JSON 原始报文？**
因为数据被污染或格式变化时，可以从 ODS 重新解析，无需回源。ODS 是数据的"后悔药"。

#### DWD 层（Data Warehouse Detail，明细数据层）

**定位：** 对 ODS 进行清洗、标准化、字段类型统一的明细数据层。

```sql
-- DWD 层清洗示例：从 raw_data JSON 中提取关键字段
CREATE TABLE dwd_alert_event_detail (
    alert_id        BIGINT    COMMENT '告警唯一标识',
    src_ip          STRING    COMMENT '源 IP（标准化为字符串）',
    dst_ip          STRING    COMMENT '目标 IP',
    alert_type      STRING    COMMENT '告警类型（统一小写）',
    severity        TINYINT   COMMENT '严重程度 1-5',
    created_at      TIMESTAMP COMMENT '事件发生时间（UTC 转本地时区）',
    raw_payload     STRING    COMMENT '原始 JSON（脱敏后）',
    is_valid        BOOLEAN   COMMENT '是否通过数据质量校验'
) COMMENT 'SOC 告警事件明细数据层';
```

**DWD 核心处理逻辑：**
```python
def clean_alert_event(raw_row: dict) -> dict:
    """
    DWD 层标准化清洗规则：
    1. 时间统一转 UTC → 本地时区
    2. IP 去除空格、前后缀
    3. 告警类型统一小写
    4. 空字符串 → NULL
    5. 隐私字段脱敏
    """
    cleaned = {
        "alert_id":    int(raw_row.get("id", 0)),
        "src_ip":      str(raw_row.get("src_ip", "")).strip() or None,
        "dst_ip":      str(raw_row.get("dst_ip", "")).strip() or None,
        "alert_type":  raw_row.get("alert_type", "").lower().strip(),
        "severity":    int(raw_row.get("severity", 3)),
        "created_at":  parse_iso_time(raw_row.get("event_time")),
        "is_valid":    validate_alert(raw_row),
    }
    return cleaned
```

#### DWS 层（Data Warehouse Summary，汇总数据层）

**定位：** 按业务主题（告警类型、IP、时段）预先聚合的轻度宽表，支撑下游快速查询。

```sql
-- DWS 层：按告警类型 + 天维度的汇总表
CREATE TABLE dws_alert_type_daily (
    stat_date        DATE       COMMENT '统计日期',
    alert_type       STRING     COMMENT '告警类型',
    alert_cnt        BIGINT     COMMENT '告警总数',
    critical_cnt     BIGINT     COMMENT '严重告警数量（severity >= 4）',
    uniq_src_ip_cnt  BIGINT     COMMENT '独立源 IP 数量',
    uniq_dst_ip_cnt  BIGINT     COMMENT '独立目标 IP 数量',
    avg_severity     DECIMAL(3,2) COMMENT '平均严重程度',
    etl_time         TIMESTAMP  COMMENT 'ETL 任务时间'
) COMMENT '告警事件按类型+天维度汇总';
```

**⚠️ 注意：DWS 层的分区字段（stat_date）必须放在 GROUP BY 中**，否则跨分区查询会得到错误结果。

#### ADS 层（Application Data Store，应用数据层）

**定位：** 直接面向 BI 报表、大屏、API 接口的数据层，按业务需求定制。

```sql
-- ADS 层：SOC 安全大屏核心指标
CREATE TABLE ads_security_dashboard (
    stat_date        DATE       COMMENT '统计日期',
    total_alerts     BIGINT     COMMENT '当日告警总数',
    critical_rate    DECIMAL(5,4) COMMENT '严重告警占比',
    top_attack_types STRING     COMMENT 'Top3 攻击类型（JSON 数组）',
    top_src_ips      STRING     COMMENT 'Top10 源攻击 IP（JSON 数组）',
    response_time_p95 INT       COMMENT '告警平均响应时间 P95（分钟）'
) COMMENT 'SOC 安全态势大屏指标';
```

### 1.3 分层数据流向

```
源系统（MySQL/日志/告警API）
  │
  │  [全量/增量抽取]
  ▼
ODS（原始层：1:1 映射，保留原始 JSON）
  │
  │  [DWD 清洗任务]
  ▼
DWD（明细层：字段提取、类型统一、脱敏、异常标记）
  │
  │  [DWS 聚合任务]
  ▼
DWS（汇总层：按主题+时间维度的轻量聚合）
  │
  │  [ADS 应用任务]
  ▼
ADS（应用层：BI 报表、API 数据、大屏指标）
```

## 2. 范式与反范式

### 2.1 为什么要懂范式？

数据库范式是关系型数据库设计的理论基础。面试中问到"你们数仓为什么这么建模"，需要能用范式理论解释设计决策。

### 2.2 三大范式速解

| 范式 | 要求 | 例子 |
|------|------|------|
| **1NF** | 字段不可再分 | 地址字段不能是"北京市朝阳区xx"，要拆成"城市+区县+街道" |
| **2NF** | 非主键字段完全依赖主键（消除部分依赖） | 订单表中"客户名称"依赖"客户ID"而非"订单ID"，违反2NF |
| **3NF** | 非主键字段之间不存在传递依赖 | "订单金额"不应由"客户等级折扣率"计算得到（通过客户等级表间接依赖） |

### 2.3 星型模型 vs 雪花模型

在数仓维度建模中，最常用的是**星型模型**和**雪花模型**。

**星型模型（Star Schema）**

```
        ┌──────────────┐
        │  fact_alert   │  ← 事实表（中心）
        └──────┬───────┘
    ┌─────────┼─────────┐
    ▼         ▼         ▼
┌────────┐ ┌────────┐ ┌────────┐
│dim_time│ │dim_ip  │ │dim_rule│  ← 维度表（星芒）
└────────┘ └────────┘ └────────┘
```

```sql
-- 事实表：存储可量化的业务事件（告警记录）
CREATE TABLE fact_alert_event (
    alert_id      BIGINT PRIMARY KEY,
    time_id       BIGINT COMMENT '外键 → dim_time',
    src_ip_id     BIGINT COMMENT '外键 → dim_ip',
    rule_id       BIGINT COMMENT '外键 → dim_detection_rule',
    severity      TINYINT,
    is_true_positive BOOLEAN
);

-- 维度表：描述事实的环境上下文
CREATE TABLE dim_ip (
    ip_id         BIGINT PRIMARY KEY,
    ip_address    STRING,
    country       STRING,
    city          STRING,
    is_internal   BOOLEAN  -- 是否内网 IP
);
```

**雪花模型（Snowflake Schema）**

雪花模型是星型模型的扩展——维度表进一步被**规范化拆分成子维度表**。

```
fact_alert ──→ dim_ip ──→ dim_country ──→ dim_region
                  │
                  └──→ dim_isp
```

```sql
-- 雪花模型：dim_ip 被进一步规范化
CREATE TABLE dim_country (
    country_id    BIGINT PRIMARY KEY,
    country_name  STRING,
    region_id     BIGINT  -- 外键 → dim_region
);
```

**星型 vs 雪花对比：**

| 维度 | 星型模型 | 雪花模型 |
|------|---------|---------|
| 查询性能 | 更快（JOIN 少） | 略慢（多表 JOIN） |
| 存储空间 | 略大（数据冗余） | 更小（规范化去冗余） |
| 维护成本 | 低（结构简单） | 高（多表需同步更新） |
| 适用场景 | 宽表查询、BI 报表 | 维度复杂、更新频繁的维度 |
| 面试答案 | 大多数场景首选星型 | 维度表大且层级深时用雪花 |

**⚠️ 注意：数仓是写少读多的场景，反范式（适当冗余）可以大幅提升查询性能。** 不要为了追求"完美范式"而引入不必要的 JOIN。

### 2.4 宽表设计

宽表 = 将多个相关维度预先 JOIN 到一张大表，避免实时计算。

```sql
-- 告警事件宽表：把维度信息直接冗余到事实表
CREATE TABLE dwd_alert_event_wide (
    alert_id         BIGINT,
    event_time       TIMESTAMP,
    -- 来自 dim_time 的冗余字段
    stat_hour        INT,
    stat_day         DATE,
    day_of_week      INT,
    -- 来自 dim_ip 的冗余字段
    src_ip           STRING,
    src_country      STRING,
    src_city         STRING,
    src_is_internal  BOOLEAN,
    -- 来自 dim_rule 的冗余字段
    rule_name        STRING,
    rule_category    STRING,
    -- 来自 fact_alert 的度量字段
    severity         TINYINT
) PARTITION BY RANGE (stat_day);
```

**宽表优点：** 查询极简，单表全搞定。
**宽表缺点：** 数据冗余、更新复杂、存储成本上升。

## 3. SQL 核心：窗口函数与 JOIN

### 3.1 窗口函数详解

窗口函数与 `GROUP BY` 的本质区别：`GROUP BY` 是"压缩行"，窗口函数是"保留行同时计算"。

```sql
-- 场景：计算每条告警与同类型前一条告警的时间间隔
SELECT
    alert_id,
    alert_type,
    created_at,
    LAG(created_at) OVER (
        PARTITION BY alert_type   -- 按告警类型分组
        ORDER BY created_at        -- 组内按时间排序
    ) AS prev_alert_time,
    -- 计算时间差（小时）
    TIMESTAMPDIFF(
        HOUR,
        LAG(created_at) OVER (PARTITION BY alert_type ORDER BY created_at),
        created_at
    ) AS hours_since_same_type
FROM dwd_alert_event_detail
WHERE is_valid = TRUE;
```

**输出示例：**

| alert_id | alert_type | created_at | prev_alert_time | hours_since_same_type |
|----------|-----------|------------|----------------|----------------------|
| 1001 | malware | 2026-04-10 08:00 | NULL | NULL |
| 1002 | malware | 2026-04-10 09:30 | 2026-04-10 08:00 | 1 |
| 1003 | ddos | 2026-04-10 10:00 | NULL | NULL |

### 3.2 ROW_NUMBER / RANK / DENSE_RANK 对比

这三个函数都用于排名，但处理"并列"时行为不同：

```sql
SELECT
    src_ip,
    alert_cnt,
    ROW_NUMBER()  OVER (ORDER BY alert_cnt DESC) as rn,   -- 并列也排名次，跳过下一名
    RANK()        OVER (ORDER BY alert_cnt DESC) as rk,   -- 并列排名，但跳过重复
    DENSE_RANK()  OVER (ORDER BY alert_cnt DESC) as drk   -- 并列排名，不跳过（常用）
FROM (
    SELECT src_ip, COUNT(*) as alert_cnt
    FROM dwd_alert_event_detail
    GROUP BY src_ip
) t;
```

**结果对比（假设有两个 IP 都是 100 次告警）：**

| src_ip | alert_cnt | ROW_NUMBER | RANK | DENSE_RANK |
|--------|-----------|-----------|------|-----------|
| 1.1.1.1 | 100 | 1 | 1 | 1 |
| 2.2.2.2 | 100 | 2 | 1 | 1 |
| 3.3.3.3 | 90 | 3 | 3 | 2 |

**面试加分点：** RANK() 在 TOP-N 查询中容易产生"跨行断层"，生产环境推荐 DENSE_RANK()。

### 3.3 SUM OVER / COUNT OVER（累计窗口）

```sql
-- 计算告警数的日累计量（Running Total）
SELECT
    stat_date,
    daily_alerts,
    SUM(daily_alerts) OVER (ORDER BY stat_date) as cumulative_alerts,
    -- 近 7 天移动平均
    AVG(daily_alerts) OVER (
        ORDER BY stat_date
        ROWS BETWEEN 6 PRECEDING AND CURRENT ROW
    ) as moving_avg_7d,
    -- 近 7 天告警占比
    daily_alerts / SUM(daily_alerts) OVER () as pct_of_total
FROM dws_alert_type_daily
WHERE stat_date >= '2026-04-01';
```

**ROWS BETWEEN 语法解释：**

| 子句 | 含义 |
|------|------|
| `ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW` | 从开头到当前行（默认值） |
| `ROWS BETWEEN 6 PRECEDING AND CURRENT ROW` | 包含当前行及之前 6 行（共 7 行） |
| `ROWS BETWEEN 3 PRECEDING AND 1 FOLLOWING` | 当前行 ±3 行范围 |

### 3.4 JOIN 类型详解与实战

```sql
-- 场景：给告警事件关联 IP 维度信息，并标注哪些告警没有匹配到 IP 维度
SELECT
    a.alert_id,
    a.src_ip,
    i.country,
    i.city,
    CASE
        WHEN i.ip_id IS NULL THEN 'IP_NOT_IN_DIM'
        ELSE 'MATCHED'
    END AS match_status
FROM dwd_alert_event_detail a
LEFT JOIN dim_ip i ON a.src_ip_id = i.ip_id   -- 告警表全保留，右表没有则 NULL
INNER JOIN dwd_detection_rule r ON a.rule_id = r.rule_id  -- 只保留有规则的告警
WHERE a.created_at >= '2026-04-10 00:00:00';
```

**JOIN 性能黄金法则：**
1. **小表驱动大表**：`A LEFT JOIN B` 时，B 表越小越好（布隆过滤器/广播JOIN）
2. **ON 条件有索引**：维度表的主键 JOIN 事实表的外键索引是最优路径
3. **避免笛卡尔积**：多表 JOIN 时确保每对表都有 ON 条件

### 3.5 UNION vs UNION ALL

```sql
-- UNION ALL：直接合并，不去重（效率高）
SELECT ip FROM ods_external_ip_log
UNION ALL
SELECT ip FROM ods_internal_ip_log;

-- UNION：去重合并（内部执行 DISTINCT，有额外排序开销）
SELECT ip FROM ods_external_ip_log
UNION
SELECT ip FROM ods_internal_ip_log;
```

**⚠️ 注意：**
- 数据量超过 1000 万行时，UNION 的去重开销会非常明显
- 如果明确知道两个结果集没有交集（如同一天的两个数据源），永远用 UNION ALL
- `SELECT 1 AS tag UNION ALL SELECT 2` 用于快速构造测试数据

## 4. ETL 调度系统

### 4.1 调度系统的核心职责

ETL 调度系统不只是"定时执行脚本"，它需要解决：

| 职责 | 说明 | 如果不做会怎样 |
|------|------|--------------|
| 定时触发 | 按 cron 表达式定时启动任务 | 只能手动运行，无法自动化 |
| 依赖管理 | 任务 A 完成后才能运行任务 B | 数据还没算出来就查，导致错误 |
| 失败重试 | 任务失败自动重试 N 次 | 单次网络抖动导致整条链路失败 |
| 告警通知 | 任务失败/超时通知负责人 | 问题发生后无人知晓，数据错误持续扩散 |
| 并发控制 | 同一任务不能同时运行两个实例 | 数据重复写入、分区冲突 |

### 4.2 APScheduler 调度器

APScheduler 是 Python 中最常用的轻量级调度库，适合单机 ETL 场景。

```python
from apscheduler.schedulers.blocking import BlockingScheduler
from apscheduler.events import EVENT_JOB_ERROR, EVENT_JOB_MISSED
import logging

logging.basicConfig(level=logging.INFO)
logger = logging.getLogger(__name__)

scheduler = BlockingScheduler(timezone='Asia/Shanghai')

def send_alert(message: str):
    """发送告警通知（可接入钉钉/飞书/企业微信）"""
    print(f"[ALERT] {message}")


def job_error_listener(event):
    """监听任务执行错误，触发告警"""
    if event.exception:
        logger.error(f"任务 {event.job_id} 失败: {event.exception}")
        send_alert(f"ETL 任务失败: {event.job_id} - {event.exception}")
    else:
        logger.info(f"任务 {event.job_id} 执行完成")


# 注册错误监听器
scheduler.add_listener(job_error_listener, EVENT_JOB_ERROR | EVENT_JOB_MISSED)


def daily_dim_etl():
    """DWD 维度表日级 ETL"""
    logger.info("开始执行: 维度表 ETL")
    # 实际 ETL 逻辑在这里
    pass


def daily_fact_etl():
    """DWD 事实表日级 ETL（依赖维度表完成）"""
    logger.info("开始执行: 事实表 ETL")
    pass


# 每日凌晨 2:00 执行维度表 ETL（较早运行，因为事实表依赖它）
scheduler.add_job(
    daily_dim_etl,
    'cron',
    day_of_week='*',
    hour=2,
    minute=0,
    misfire_grace_time=3600,   # 错过 1 小时内仍可执行（服务器停机恢复场景）
    id='dim_etl',              # 唯一 ID，用于依赖管理
    replace_existing=True      # 更新已存在的任务
)

# 每日凌晨 3:00 执行事实表 ETL
scheduler.add_job(
    daily_fact_etl,
    'cron',
    day_of_week='*',
    hour=3,
    minute=0,
    misfire_grace_time=3600,
    id='fact_etl',
    replace_existing=True
)

logger.info("ETL 调度器启动")
scheduler.start()
```

**misfire_grace_time 详解：** 如果服务器在 2:00 宕机，3:00 恢复时，`misfire_grace_time=3600` 意味着在 3:00 之前（2:00 + 1小时）触发都算"按时"，4:00 才恢复则跳过本次执行。

### 4.3 分布式锁：防止任务并发重复执行

单机调度没问题，但分布式环境下（多台服务器同时跑 ETL），**同一个任务可能被多个实例同时触发**，导致数据重复写入。

```python
import redis
import uuid
import time

class DistributedLock:
    """
    基于 Redis 的分布式锁实现。
    使用 SET NX EX 原子命令，保证只有一个实例能获取锁。
    """

    def __init__(self, redis_client: redis.Redis, lock_key: str, ttl_seconds: int = 3600):
        self.redis = redis_client
        self.lock_key = lock_key
        self.ttl = ttl_seconds
        self.instance_id = str(uuid.uuid4())  # 当前实例的唯一标识

    def acquire(self) -> bool:
        """
        尝试获取锁。
        NX = 不存在时才设置（保证互斥）
        EX = 设置过期时间（防止锁无法释放导致死锁）
        """
        acquired = self.redis.set(
            self.lock_key,
            self.instance_id,
            nx=True,   # 只有 key 不存在时才设置
            ex=self.ttl  # 过期时间（秒）
        )
        if acquired:
            print(f"[{self.instance_id}] 获取锁成功")
        else:
            print(f"[{self.instance_id}] 获取锁失败，已有其他实例持有")
        return bool(acquired)

    def release(self) -> bool:
        """
        释放锁。
        ⚠️ 重要：只能用 Lua 脚本保证原子性检查+删除，
        否则会出现检查和删除之间的竞态条件。
        """
        lua_script = """
        if redis.call("GET", KEYS[1]) == ARGV[1] then
            return redis.call("DEL", KEYS[1])
        else
            return 0
        end
        """
        result = self.redis.eval(lua_script, 1, self.lock_key, self.instance_id)
        return bool(result)


# 使用示例
def run_etl_with_lock(job_name: str):
    lock = DistributedLock(
        redis_client=redis.Redis(host='localhost', port=6379, db=0),
        lock_key=f"etl_lock:{job_name}",
        ttl_seconds=7200  # 2 小时超时
    )

    if not lock.acquire():
        raise RuntimeError(f"任务 {job_name} 正在被其他实例执行，跳过本次运行")

    try:
        # 执行实际 ETL 逻辑
        print(f"开始执行 ETL: {job_name}")
        time.sleep(5)
        print(f"ETL 完成: {job_name}")
    finally:
        lock.release()  # ⚠️ 必须在 finally 中释放，确保异常时也能释放
```

## 5. CDC (Change Data Capture)

### 5.1 为什么需要 CDC？

传统的 ETL 方式是**全量同步**（每天把整个表重新拉一遍），但随着数据量增长，问题暴露：

| 方式 | 数据量 100GB | 数据量 10TB | 数据量 1PB |
|------|-------------|------------|-----------|
| 全量同步 | 耗时数小时 | 不可接受 | 不可能 |
| CDC 增量同步 | 仅同步当天变更（约几十 MB）| 仅同步当天变更 | 仅同步当天变更 |

CDC 的本质：**只捕获"变化的数据"，而不是重新传输"所有数据"。**

### 5.2 常见 CDC 方案对比

| 方案 | 原理 | 延迟 | 实现复杂度 | 数据完整性 |
|------|------|------|-----------|-----------|
| **时间戳 CDC** | 轮询 `WHERE updated_at > last_run` | 分钟~小时级 | 低 | 可能漏数据（更新但时间未变）|
| **Binlog CDC** | 监听 MySQL binlog / Postgres WAL | 秒级甚至毫秒 | 高（需解析 binlog）| 高（记录所有变更）|
| **触发器 CDC** | 数据库触发器写增量表 | 准实时 | 中 | 依赖触发器可靠性 |
| **快照对比 CDC** | 定期全量快照，与上次对比 | 小时级 | 低 | 资源消耗大 |
| **日志挖掘 CDC** | 解析数据库日志文件 | 分钟级 | 高 | 高 |

### 5.3 Binlog CDC 工作原理

MySQL binlog 记录了所有数据修改操作（INSERT / UPDATE / DELETE），CDC 工具通过模拟 MySQL 主从复制协议，伪装成一个从节点来拉取 binlog：

```
MySQL Server
    │
    │  [开启 binlog]
    │  binlog_format = ROW（行级复制，完整记录变更前后数据）
    ▼
Debezium / Canal
    │  [解析 binlog 事件]
    │  将行级变更转换为结构化事件（包含操作类型+前后镜像）
    ▼
Kafka（消息队列，作为缓冲和分发层）
    │  [主题分区存储]
    │  可被多个消费者订阅（下游系统解耦）
    ▼
消费者（数据湖/数据仓库/Search索引）
```

## 6. 数据清洗

### 6.1 数据清洗的核心价值

> "Garbage in, garbage out." — 数据清洗是数据质量的守门人。再华丽的指标体系，也扛不住脏数据的冲击。

数据清洗的核心任务：
1. **修正错误**：拼写错误、格式不一致、乱码
2. **填补缺失**：可推导的填补，不可推导的标记
3. **消除重复**：去重但保留分析需要的记录
4. **识别异常**：发现超出业务合理范围的数据

### 6.2 空值处理策略

空值（NULL）处理是最常见的数据清洗问题。**错误的填补策略比不填补更有害。**

```sql
-- 场景：告警事件中的严重程度字段（severity）出现空值

-- ❌ 错误做法：将 NULL 填 0（掩盖问题，且 0 可能被用于计算导致错误均值）
UPDATE dwd_alert_event_detail SET severity = 0 WHERE severity IS NULL;

-- ✅ 正确做法：分类处理
SELECT
    alert_id,
    src_ip,
    -- 方案1：真实缺失（如设备未上报），保留 NULL
    severity,
    -- 方案2：使用 COALESCE 提供默认值（仅用于展示，不改写原数据）
    COALESCE(severity, 3) AS severity_display,  -- 未知时默认显示为"中"
    -- 方案3：用子查询填充（同一 IP 历史平均严重程度）
    COALESCE(
        severity,
        (SELECT AVG(severity) FROM dwd_alert_event_detail h
         WHERE h.src_ip = d.src_ip AND h.severity IS NOT NULL)
    ) AS severity_filled,
    -- 方案4：标记为"未知"（推荐，保留可追溯性）
    CASE WHEN severity IS NULL THEN 'UNKNOWN'
         ELSE CAST(severity AS STRING)
    END AS severity_tag
FROM dwd_alert_event_detail d;
```

**空值处理决策树：**

```
该字段为空的原因是什么？
├── 业务上应该有值但没采集到 → 用均值/众数/前向填充（按时间/ID）
├── 业务上本就没有该字段   → 保留 NULL 或标记为"NOT_APPLICABLE"
├── 上游数据污染/格式错误  → 标记为 ERROR，触发上游修复
└── 数据同步延迟          → 等待后重拉，标记为"PENDING"
```

### 6.3 异常值检测

异常值检测有两种核心方法：**基于规则**和**基于统计**。

**方法一：基于业务规则的硬编码检测**

```sql
SELECT
    alert_id,
    order_amt,
    goods_cnt,
    CASE
        WHEN order_amt < 0 THEN 'NEGATIVE_AMOUNT'
        WHEN order_amt > 1000000 THEN 'EXCESSIVE_AMOUNT'
        WHEN goods_cnt < 0 THEN 'NEGATIVE_QUANTITY'
        WHEN goods_cnt > 10000 THEN 'EXCESSIVE_QUANTITY'
        WHEN severity NOT BETWEEN 1 AND 5 THEN 'INVALID_SEVERITY'
        WHEN src_ip LIKE '%[^0-9.]%' THEN 'INVALID_IP_FORMAT'
        WHEN created_at > NOW() THEN 'FUTURE_TIMESTAMP'
        WHEN created_at < '2020-01-01' THEN 'ANCIENT_TIMESTAMP'
        ELSE 'NORMAL'
    END AS anomaly_type,
    -- 记录异常，用于后续分析和根因追溯
    CASE WHEN anomaly_type != 'NORMAL' THEN 1 ELSE 0 END AS is_anomaly
FROM dwd_alert_event_detail
WHERE stat_date = '2026-04-10';
```

**方法二：基于统计的异常检测（Z-Score / IQR）**

```sql
-- 计算告警数量 Z-Score，标记超过 ±3σ 的异常点
WITH stats AS (
    SELECT
        AVG(alert_cnt) as mean_cnt,
        STDDEV(alert_cnt) as std_cnt
    FROM dws_alert_type_daily
    WHERE stat_date BETWEEN DATE_SUB('2026-04-10', INTERVAL 30 DAY)
                        AND DATE_SUB('2026-04-10', INTERVAL 1 DAY)
),
alert_stats AS (
    SELECT
        alert_type,
        COUNT(*) as alert_cnt,
        (SELECT mean_cnt FROM stats) as mean_cnt,
        (SELECT std_cnt FROM stats) as std_cnt
    FROM dwd_alert_event_detail
    WHERE stat_date = '2026-04-10'
    GROUP BY alert_type
)
SELECT
    alert_type,
    alert_cnt,
    ROUND((alert_cnt - mean_cnt) / std_cnt, 2) as z_score,
    CASE
        WHEN ABS((alert_cnt - mean_cnt) / std_cnt) > 3 THEN 'EXTREME_OUTLIER'
        WHEN ABS((alert_cnt - mean_cnt) / std_cnt) > 2 THEN 'MODERATE_OUTLIER'
        ELSE 'NORMAL'
    END AS outlier_flag
FROM alert_stats;
```

**⚠️ 注意：异常值不一定要删除。** 告警数据中，异常值往往就是最需要关注的安全威胁（一次 DDoS 攻击可能产生正常 100 倍的流量），**标记异常但不轻易删除**，保留用于安全分析。

```python
def normalize_and_cast(df):

    import json
    def safe_parse_json(x):
        try:

        except (TypeError, json.JSONDecodeError):

    df['raw_json_parsed'] = df['raw_data'].apply(safe_parse_json)
```

### 6.5 去重策略

```sql
-- 场景：ODS 层有重复数据（网络抖动导致重复抽取）

-- 方案1：保留最早一条（Dedupe by First）
-- 适用于：业务上以第一条记录为准
DELETE FROM ods_alert_event t1
WHERE EXISTS (
    SELECT 1 FROM ods_alert_event t2
    WHERE t1.alert_id = t2.alert_id
      AND t1.extract_time > t2.extract_time   -- 有更早的记录
);

-- 方案2：保留最新一条（Update-in-place）
-- 适用于：业务数据可能被修正，需保留最新状态
DELETE FROM ods_alert_event t1
WHERE EXISTS (
    SELECT 1 FROM ods_alert_event t2
    WHERE t1.alert_id = t2.alert_id
      AND t1.extract_time < t2.extract_time    -- 有更新的记录
);

-- 方案3：保留全部，但 DWD 层用窗口函数去重

SELECT

    FIRST_VALUE(alert_type) OVER (PARTITION BY alert_id ORDER BY extract_time DESC) as alert_type,
```

## 7. SCD（Slowly Changing Dimensions，缓慢变化维度）

### 7.1 什么是 SCD？

维度表中的属性会随时间变化（如用户更换了手机号、IP 的地理位置变了），SCD 就是处理这些"变化"的策略。

**SOC 场景的例子：** 一个 IP 地址的归属地可能因为 ISP 分配策略变化而改变（从"北京"变成"上海"）。如何记录这种变化？

### 7.2 Type 1：直接覆盖（Overwrite）

**策略：** 新值覆盖旧值，不保留历史。

```sql
-- 原始数据
-- dim_ip: IP=1.2.3.4, country=中国, city=北京, start_date=2026-01-01

-- IP 归属地变更时，直接 UPDATE
UPDATE dim_ip
SET country = '中国', city = '上海', start_date = '2026-04-10'
WHERE ip_address = '1.2.3.4';
```

**适用场景：** 历史值没有分析价值（如联系方式、备注字段）。
**缺点：** 无法追溯历史，报表中过去的数据也会被"修正"。

### 7.3 Type 2：拉链表（Slowly Changing Dimension Type 2）

**策略：** 每次变化新增一行，用 `start_date` 和 `end_date` 标记有效时间范围。

```sql
-- 拉链表表结构
CREATE TABLE dim_ip_scd2 (
    ip_id         BIGINT,
    ip_address    STRING,
    country       STRING,
    city          STRING,
    is_internal   BOOLEAN,
    start_date    DATE,      -- 该版本生效日期
    end_date      DATE,      -- 该版本失效日期（9999-12-31 = 当前有效）
    is_current    BOOLEAN    -- 是否为当前版本（便利查询）
);

-- 初始数据（2026-01-01 生效）
INSERT INTO dim_ip_scd2 VALUES
(1, '1.2.3.4', '中国', '北京', false, '2026-01-01', '9999-12-31', true);

-- 2026-04-10，IP 归属地从北京变成上海
-- 步骤1：关闭旧版本（end_date = 变更日期 - 1）
UPDATE dim_ip_scd2
SET end_date = '2026-04-09', is_current = false
WHERE ip_address = '1.2.3.4' AND is_current = true;

-- 步骤2：插入新版本
INSERT INTO dim_ip_scd2 VALUES
(1, '1.2.3.4', '中国', '上海', false, '2026-04-10', '9999-12-31', true);
```

**查询某一天的历史快照：**

```sql
-- 查询 2026-03-01 时，IP 1.2.3.4 归属哪个城市？
SELECT ip_address, country, city
FROM dim_ip_scd2
WHERE ip_address = '1.2.3.4'
  AND start_date <= '2026-03-01'
  AND end_date   >  '2026-03-01';
-- 结果：city = '北京'（因为 2026-03-01 时，旧版本有效）

-- 查询当前有效的 IP 维度（最常用）
SELECT ip_address, country, city
FROM dim_ip_scd2
WHERE is_current = true;
```

### 7.4 拉链表的自动化实现

```python
def merge_scd2(current_df: pd.DataFrame, new_df: pd.DataFrame, primary_key: str = 'ip_id'):
    """
    SCD Type 2 拉链表合并逻辑：

    1. 找出 new_df 中有变化的行（与 current 版本的维度值不同）
    2. 变化行：关闭旧版本 + 插入新版本
    3. 无变化行：保持不变
    4. new_df 中新增的行（current 中没有）：直接插入
    """
    import pandas as pd

    today = pd.Timestamp.now().normalize()
    tomorrow = today + pd.Timedelta(days=1)
    far_future = pd.Timestamp('9999-12-31')

    # 1. 找出新增的行（current 中没有的）
    current_keys = set(current_df[primary_key])
    new_keys = set(new_df[primary_key])
    inserted_keys = new_keys - current_keys

    # 2. 找出需要更新的行（有变化）
    # 对比 current 的 is_current=true 行 和 new_df 的值
    current_active = current_df[current_df['is_current'] == True].copy()

    changed_rows = []
    if not current_active.empty:
        merged = new_df.merge(
            current_active[[primary_key, 'country', 'city']],
            on=primary_key,
            suffixes=('_new', '_old')
        )
        # 维度值有变化 → 需要 SCD2 更新
        changed = merged[
            (merged['country_new'] != merged['country_old']) |
            (merged['city_new'] != merged['city_old'])
        ]
        changed_rows = changed[primary_key].tolist()

    # 3. 关闭旧版本（变更行）
    current_df.loc[
        (current_df[primary_key].isin(changed_rows)) & (current_df['is_current'] == True),
        ['end_date', 'is_current']
    ] = [today, False]

    # 4. 生成新版本（变更行 + 新增行）
    new_versions = new_df[
        new_df[primary_key].isin(changed_rows + list(inserted_keys))
    ].copy()
    new_versions['start_date'] = today
    new_versions['end_date'] = far_future
    new_versions['is_current'] = True

    # 5. 合并：旧版本（变化行） + 旧版本（未变化行） + 新版本
    unchanged = current_df[~current_df[primary_key].isin(changed_rows)]
    result = pd.concat([unchanged, new_versions], ignore_index=True)

    return result
```

### 7.5 Type 3：新增列（Add New Attribute）

**策略：** 在同一行中保留当前值和历史值。

```sql

ALTER TABLE dim_ip ADD COLUMN city_change_date DATE;
```

**SCD Type 对比总结：**

| 类型 | 历史保留 | 表结构变化 | 查询复杂度 | 适用场景 |
|------|---------|-----------|-----------|---------|
| Type 1 | 不保留 | 无 | 最低 | 维度的临时备注、可覆盖字段 |
| Type 2 | 完整保留 | 增加 start/end/is_current | 中等 | 关键维度（IP归属地、用户等级、规则分类）|
| Type 3 | 保留上一步 | 增加历史列 | 较低 | 只关心"上一次变化"的场景 |

## 8. 数据质量保障

### 8.1 数据质量的四大维度

| 维度 | 定义 | SOC 场景例子 | 测量方法 |
|------|------|------------|---------|
| **完整性（Completeness）** | 必填字段是否有值 | severity 字段不能为空 | 空值率 |
| **一致性（Consistency）** | 数据跨系统是否一致 | ODS 和 MySQL 源数据行数是否一致 | 校验和对比 |
| **准确性（Accuracy）** | 数据是否反映真实情况 | 告警时间戳是否准确到秒 | 采样人工核查 |
| **及时性（Timeliness）** | 数据是否在预期时间内到达 | 告警事件 ETL 延迟不超过 15 分钟 | 端到端延迟监控 |

### 8.2 监控指标体系与阈值

```sql
-- 数据质量检查 SQL（每日 ETL 完成后执行）
WITH quality_checks AS (
    SELECT
        'dwd_alert_event_detail' as table_name,
        -- 1. 空值率检查
        ROUND(
            SUM(CASE WHEN severity IS NULL THEN 1 ELSE 0 END) * 100.0 / COUNT(*),
            4
        ) as null_rate_pct,
        -- 2. 重复率检查（alert_id 是主键）
        ROUND(
            SUM(CASE WHEN rn > 1 THEN 1 ELSE 0 END) * 100.0 / COUNT(*),
            4
        ) as duplicate_rate_pct,
        -- 3. 数据延迟检查（最新数据的时间戳）
        MAX(created_at) as latest_record_time,
        -- 4. 数据量环比（与昨天对比）
        COUNT(*) as today_cnt,
        LAG(COUNT(*)) OVER (ORDER BY stat_date) as yesterday_cnt
    FROM (
        SELECT *, ROW_NUMBER() OVER (PARTITION BY alert_id ORDER BY cdc_timestamp DESC) as rn
        FROM dwd_alert_event_detail
        WHERE stat_date = CURRENT_DATE
    ) deduped
    GROUP BY table_name
)
SELECT
    table_name,
    null_rate_pct,
    duplicate_rate_pct,
    latest_record_time,
    today_cnt,
    yesterday_cnt,
    -- 波动率 = |今天-昨天| / 昨天
    ROUND(ABS(today_cnt - yesterday_cnt) * 100.0 / NULLIF(yesterday_cnt, 0), 2) as波动率_pct,
    -- 告警判断
    CASE
        WHEN null_rate_pct > 1.0 THEN '⚠️ NULL_RATE_HIGH'
        WHEN duplicate_rate_pct > 0.01 THEN '⚠️ DUPLICATE_FOUND'
        WHEN latest_record_time < DATE_SUB(NOW(), INTERVAL 30 MINUTE)
             THEN '⚠️ DATA_DELAY'
        WHEN ABS(today_cnt - yesterday_cnt) * 100.0 / NULLIF(yesterday_cnt, 0) > 30
             THEN '⚠️ VOLUME_FLUCTUATION'
        ELSE '✅ PASS'
    END as quality_status
FROM quality_checks;
```

### 8.3 数据质量监控代码

```python
import pandas as pd
from dataclasses import dataclass
from typing import Optional
import logging

logger = logging.getLogger(__name__)

@dataclass
class DataQualityRule:
    """数据质量规则定义"""
    rule_name: str
    check_sql: str          # 执行质量检查的 SQL
    threshold: float        # 告警阈值
    severity: str           # CRITICAL / WARNING / INFO


@dataclass
class QualityResult:
    """质量检查结果"""
    rule_name: str
    actual_value: float
    threshold: float
    status: str  # PASS / FAIL
    message: str


class DataQualityMonitor:
    """
    数据质量监控器：
    1. 执行预定义的质量规则
    2. 记录检查结果
    3. 超阈值时触发告警
    """

    RULES = [
        DataQualityRule(
            rule_name="空值率检查",
            check_sql="""
                SELECT
                    ROUND(SUM(CASE WHEN severity IS NULL THEN 1 ELSE 0 END) * 100.0 / COUNT(*), 4)
                    as null_rate
                FROM dwd_alert_event_detail
                WHERE stat_date = '{target_date}'
            """,
            threshold=1.0,   # 空值率超过 1% 告警
            severity="WARNING"
        ),
        DataQualityRule(
            rule_name="数据量环比波动",
            check_sql="""
                SELECT
                    CASE
                        WHEN yesterday_cnt = 0 THEN 0
                        ELSE ABS(today_cnt - yesterday_cnt) * 100.0 / yesterday_cnt
                    END as fluctuation_pct
                FROM (
                    SELECT COUNT(*) as today_cnt
                    FROM dwd_alert_event_detail WHERE stat_date = '{target_date}'
                ) t1 CROSS JOIN (
                    SELECT COUNT(*) as yesterday_cnt
                    FROM dwd_alert_event_detail WHERE stat_date = DATE_SUB('{target_date}', INTERVAL 1 DAY)
                ) t2
            """,
            threshold=30.0,  # 波动超过 30% 告警
            severity="CRITICAL"
        ),
        DataQualityRule(
            rule_name="重复主键检查",
            check_sql="""
                SELECT
                    ROUND(SUM(CASE WHEN rn > 1 THEN 1 ELSE 0 END) * 100.0 / COUNT(*), 4)
                    as dup_rate
                FROM (
                    SELECT *, ROW_NUMBER() OVER (PARTITION BY alert_id ORDER BY cdc_timestamp DESC) as rn
                    FROM dwd_alert_event_detail
                    WHERE stat_date = '{target_date}'
                ) t
            """,
            threshold=0.01,   # 重复率必须 < 0.01%
            severity="CRITICAL"
        ),
        DataQualityRule(
            rule_name="数据新鲜度",
            check_sql="""
                SELECT
                    TIMESTAMPDIFF(MINUTE, MAX(created_at), NOW()) as delay_minutes
                FROM dwd_alert_event_detail
                WHERE stat_date = '{target_date}'
            """,
            threshold=30.0,  # 延迟超过 30 分钟告警
            severity="WARNING"
        )
    ]

    def __init__(self, db_connection):
        self.conn = db_connection

    def run_check(self, rule: DataQualityRule, target_date: str) -> QualityResult:
        sql = rule.check_sql.format(target_date=target_date)
        with self.conn.cursor() as cursor:
            cursor.execute(sql)
            result = cursor.fetchone()
            actual_value = float(result[0]) if result else None

        status = "FAIL" if actual_value and actual_value > rule.threshold else "PASS"
        message = f"{rule.rule_name}: 实际值={actual_value}, 阈值={rule.threshold}"

        return QualityResult(
            rule_name=rule.rule_name,
            actual_value=actual_value,
            threshold=rule.threshold,
            status=status,
            message=message
        )

    def run_all_checks(self, target_date: str) -> list[QualityResult]:
        results = []
        for rule in self.RULES:
            result = self.run_check(rule, target_date)
            results.append(result)
            if result.status == "FAIL":
                logger.warning(f"数据质量检查失败: {result.message}")
                self._send_alert(result)
            else:
                logger.info(f"数据质量检查通过: {result.message}")
        return results
```

## 9. 血缘追踪

### 9.1 为什么需要血缘追踪？

数据异常时，最关键的问题是：**这个数字是从哪里来的？** 血缘追踪让你从 ADS 报表的异常数字，一路追溯到 ODS 的原始记录。

**三个核心价值：**
1. **影响分析**：某个 ODS 表变更时，评估影响哪些下游 ETL 和报表
2. **数据溯源**：报表数字出错时，追溯到原始数据的生成路径
3. **合规审计**：满足 GDPR/网络安全法对数据处理留痕的要求

### 9.2 血缘的层级

```
┌─────────────────────────────────────────────────────┐
│                    ADS 应用层                        │  ← BI 大盘 / API
├─────────────────────────────────────────────────────┤
│                    DWS 汇总层                        │  ← 聚合指标
├─────────────────────────────────────────────────────┤
│                    DWD 明细层                        │  ← 字段级清洗
├─────────────────────────────────────────────────────┤
│                    ODS 原始层                        │  ← 源系统数据
└─────────────────────────────────────────────────────┘
```

```sql
-- 查询一个 ODS 表影响了哪些下游系统（向上追溯，影响分析）
SELECT DISTINCT
    target_db,
    target_table,
    target_column,
    etl_job_name
FROM etl_lineage
WHERE source_table = 'ods_alert_event'
ORDER BY target_db, target_table;

-- 查询哪些 ODS 表支撑了一个 ADS 报表（向下追溯，溯源分析）
WITH RECURSIVE downstream AS (
    -- 起点：ADS 层
    SELECT target_table as table_name, target_column as column_name, 1 as depth
    FROM etl_lineage
    WHERE target_table = 'ads_security_dashboard'

    UNION ALL

    -- 递归：找下一层
    SELECT l.target_table, l.target_column, d.depth + 1
    FROM etl_lineage l
    INNER JOIN downstream d ON l.source_table = d.table_name
    WHERE d.depth < 10  -- 防止无限递归
)
SELECT * FROM downstream;
```

### 9.5 字段级血缘（生产级别方案）

生产环境通常用 Apache Atlas 或 DataHub 等元数据管理工具自动采集血缘。以下是简化的自实现思路：

```python
class ColumnLineageExtractor:
    """
    从 SQL 脚本中解析字段级血缘。
    简化版：基于正则表达式匹配 "source_col → target_col" 注释或字段映射。
    """

    def parse_sql_lineage(self, sql: str) -> list[dict]:
        """
        从 SQL 语句中提取字段血缘关系。

        支持的映射模式：
        - col_a, col_b, col_c  FROM source_table          -- 简单映射
        - target_col AS (SELECT source_col FROM ...)       -- 子查询映射
        - INSERT INTO target SELECT source_col FROM ...    -- 标准 INSERT SELECT
        """
        import re

        lineages = []

        # 模式1: INSERT ... SELECT 形式（最常见）
        # INSERT INTO dwd_alert_event_detail (alert_id, src_ip)
        # SELECT id, ip FROM ods_alert_event;
        insert_pattern = r'INSERT\s+INTO\s+(\w+)\s*\(([^)]+)\)\s*SELECT\s+(.+?)\s*FROM\s+(\w+)'
        for match in re.finditer(insert_pattern, sql, re.IGNORECASE | re.DOTALL):
            target_table = match.group(1)
            target_cols = [c.strip() for c in match.group(2).split(',')]
            select_exprs = [e.strip() for e in match.group(3).split(',')]
            source_table = match.group(4)

            for target_col, select_expr in zip(target_cols, select_exprs):
                lineages.append({
                    "source_table": source_table,
                    "source_column": select_expr,
                    "target_table": target_table,
                    "target_column": target_col,
                    "transform_type": "DIRECT" if select_expr in [target_col, f"{source_table}.{target_col}"] else "TRANSFORM"
                })

        return lineages

    def build_lineage_graph(self, lineages: list[dict]) -> dict:
        """
        构建血缘图：{table_name: {columns: [...], parents: [...], children: [...]}}
        用于可视化展示血缘关系。
        """
        graph = {}
        for lg in lineages:
            for table in [lg['source_table'], lg['target_table']]:
                if table not in graph:
                    graph[table] = {'columns': set(), 'parents': set(), 'children': set()}

            graph[lg['source_table']]['columns'].add(lg['source_column'])
            graph[lg['target_table']]['columns'].add(lg['target_column'])
            graph[lg['source_table']]['children'].add(lg['target_table'])
            graph[lg['target_table']]['parents'].add(lg['source_table'])

        return graph


# 使用示例
extractor = ColumnLineageExtractor()
sql = """
INSERT INTO dwd_alert_event_detail (alert_id, src_ip, dst_ip, severity)
SELECT id, ip_src, ip_dst, severity FROM ods_alert_event;
"""
lineages = extractor.parse_sql_lineage(sql)
print(lineages)
# [{'source_table': 'ods_alert_event', 'source_column': 'id',
#   'target_table': 'dwd_alert_event_detail', 'target_column': 'alert_id',
#   'transform_type': 'TRANSFORM'}, ...]
```

```sql
CREATE TABLE IF NOT EXISTS ods_alert_event_cdc (

-- DWD 清洗逻辑（从 ODS 加载数据）
TRUNCATE TABLE dwd_alert_event_detail;

INSERT INTO dwd_alert_event_detail
SELECT
    alert_id,
    -- IP 标准化：去除空格、统一 NULL
    NULLIF(TRIM(src_ip), '') as src_ip,
    NULLIF(TRIM(dst_ip), '') as dst_ip,
    -- 告警类型统一小写 + 标准化映射
    CASE
        WHEN LOWER(alert_type) IN ('malware', 'virus', 'trojan') THEN 'malware'
        WHEN LOWER(alert_type) IN ('ddos', 'flood', 'amplification') THEN 'ddos'
        WHEN LOWER(alert_type) IN ('bruteforce', 'password_attempt') THEN 'bruteforce'
        WHEN LOWER(alert_type) IN ('intrusion', 'ids_alert', 'ips_alert') THEN 'intrusion'
        WHEN LOWER(alert_type) IN ('data_leak', 'exfiltration') THEN 'data_leak'
        ELSE 'other'
    END as alert_type,
    -- 严重程度范围检查（1-5）
    CASE
        WHEN severity BETWEEN 1 AND 5 THEN severity
        ELSE 3  -- 非法值默认为"中"
    END as severity,
    rule_id,
    rule_name,
    GREATEST(attack_cnt, 0) as attack_cnt,  -- 负数置 0
    src_country,
    dst_country,
    created_at,
    cdc_timestamp,
    SUBSTR(REPLACE(CAST(created_at AS STRING), '-', ''), 1, 8) as stat_date,  -- YYYYMMDD
    -- 异常标记：严重程度 >= 4 或时间戳异常
    (severity >= 4 OR created_at > NOW() OR created_at < '2020-01-01') as is_anomaly
FROM ods_alert_event_cdc
WHERE stat_date = '20260410'
  AND operation_type != 'd';  -- 跳过删除操作（软删除场景需另行处理）

INSERT INTO dws_alert_type_daily
SELECT
    stat_date,
    alert_type,
    COUNT(*) as alert_cnt,
    SUM(CASE WHEN severity >= 4 THEN 1 ELSE 0 END) as critical_cnt,
    COUNT(DISTINCT src_ip) as uniq_src_ip_cnt,
    COUNT(DISTINCT dst_ip) as uniq_dst_ip_cnt,
    MAX(severity) as max_severity,
    AVG(severity) as avg_severity,
    SUM(attack_cnt) as total_attack_cnt,
    CURRENT_TIMESTAMP as etl_time
FROM dwd_alert_event_detail
WHERE stat_date = '20260410'
GROUP BY stat_date, alert_type;
```

### 10.5 生产环境注意事项

| 场景 | 注意事项 |
|------|---------|
| **分区管理** | 必须按日期分区，每天 ETL 只处理当天分区，避免全表扫描 |
| **幂等性** | 所有 ETL 任务必须支持重复执行（Truncate + Insert 或 Delete + Insert），幂等是可靠 ETL 的前提 |
| **异常处理** | 任何一层失败都要回滚或标记，禁止脏数据流入下游 |
| **监控** | 每层 ETL 完成后记录行数，与历史均值对比，异常时告警 |
| **回溯** | 支持传入历史日期参数，重新计算历史数据 |

## 附录：面试高频问题速记

### Q: ODS / DWD / DWS / ADS 的核心区别？
> **ODS** 是源系统的"照片"，1:1 保留，不做修改；**DWD** 是"清洗后的照片"，字段标准化；**DWS** 是"相册分类"，按主题聚合；**ADS** 是"展示给用户看的一幅画"，直接服务业务。

### Q: 什么时候用拉链表而不是快照表？
> 字段会变化且**需要保留历史**时用拉链；字段基本不变或**只关心当前状态**时用快照表。维度表用拉链，事实表按业务需要（如每日快照）处理。

### Q: CDC 和时间戳轮询各有什么优劣？
> 时间戳轮询实现简单，但会漏掉"更新但时间戳未变"的数据，且对源表有查询压力。CDC 基于 binlog，变更不漏，但对数据库配置有要求（需开启 binlog），实现复杂度高。SOC 场景对数据完整性要求高，推荐 CDC。

### Q: 数据质量监控的核心指标？
> 空值率（< 1%）、重复率（主键表 = 0）、数据延迟（< 30 分钟）、数据量波动（环比 < ±30%）。四大维度：完整性、一致性、准确性、及时性。
