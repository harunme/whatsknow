## 细节

### [初] 数据仓库基础

**OLTP vs OLAP**

| 维度 | OLTP | OLAP |
|------|-------|------|
| 场景 | 日常业务操作 | 数据分析、报表 |
| 数据量 | 单条/少量 | 海量聚合 |
| 查询 | 简单、高并发 | 复杂、低并发 |
| 工具 | MySQL、PostgreSQL | ClickHouse、StarRocks |

**四层职责**
- ODS：原始数据层，1:1 映射源表，不做清洗，保留原始凭证
- DWD：明细数据层，空值填充、异常处理、字段类型统一、时间标准化
- DWS：汇总数据层，按业务主题+时间周期组织轻量宽表
- ADS：应用数据层，直接面向 BI 报表或业务系统

### [初] SQL 基础

**窗口函数**
```sql
SELECT
    alert_id,
    LAG(created_at) OVER (PARTITION BY src_ip ORDER BY created_at) as prev_alert_time,
    DATEDIFF(created_at, LAG(created_at) OVER (...)) as hours_since_last
FROM alert_events
```

**JOIN 优化**
- 小表驱动大表（LEFT JOIN 时右表尽量小）
- 确保 ON 条件有索引

**分页查询（Top-N）**
```sql
SELECT * FROM (
    SELECT *, ROW_NUMBER() OVER (ORDER BY created_at DESC) as rn
    FROM alert_events
) t WHERE rn <= 10
```

### [中] ETL 核心

**APScheduler**
```python
from apscheduler.schedulers.blocking import BlockingScheduler
scheduler = BlockingScheduler(timezone='Asia/Shanghai')
scheduler.add_job(
    daily_etl,
    'cron', day='*', hour=2, minute=0,
    misfire_grace_time=3600
)
```

**分布式锁（防重复执行）**
```python
acquired = redis.set(lock_key, instance_id, nx=True, ex=ttl)
if not acquired:
    raise RuntimeError("任务已被其他实例持有")
```

**增量抽取判断条件**
- 优先：时间戳字段 `WHERE updated_at > '{last_run}'`
- 其次：自增 ID
- 最后：全量比对

**CDC 方案对比**

| 方案 | 原理 | 适用场景 |
|------|------|---------|
| 时间戳 CDC | 轮询 updated_at | 日级批处理 |
| binlog CDC | 读取 binlog/Redo Log | 实时同步 |
| 触发器 CDC | 数据库触发器写增量表 | 历史遗留系统过渡 |

### [中] 数据清洗

**DWD 层清洗原则**
- 应为 0 的字段：`fillna(0)`
- 真实缺失：`保留 NULL`
- 隐私字段脱敏：`phone[:3] + '****' + phone[-4:]`

**异常值检测**
```sql
CASE
    WHEN order_amt < 0 THEN 'NEGATIVE_AMT'
    WHEN goods_cnt > 1000 THEN 'EXCESSIVE_QTY'
    WHEN ABS(order_amt - avg_amt) > 3 * std_amt THEN 'OUTLIER_PRICE'
END AS anomaly_flag
```

### [高] SCD 处理

**三种类型**
- Type 1：直接覆盖，不留历史
- Type 2：新增一行，start_date/end_date 最新=9999-12-31
- Type 3：同行保留 current 和 previous 值

**拉链表查询**
```sql
SELECT * FROM dwd_user_info_zipper
WHERE start_date <= '{target_date}'
  AND end_date > '{target_date}'
```

**快照 vs 拉链**：字段少+查询频繁用快照；字段多+需历史趋势用拉链

### [高] 数据质量

**监控指标阈值**
- 空值率：常规表 < 1%，核心表 < 0.01%
- 重复率：核心表必须为 0
- 波动检测：与近 7 日均值偏差 ±30%

**血缘追踪表**
```sql
source_table / source_column → target_table / target_column
transform_rule / etl_job_name
```

**血缘价值**：影响分析、数据溯源（报表异常→ODS 原始记录）、合规审计

### [高] 数仓建模

**宽表设计原则**
- 一张表覆盖一个主题域内所有常用维度
- 避免大宽表 JOIN，提前聚合
- PARTITION BY RANGE (日期分区)

**指标体系**
- 原子指标：不可拆分（订单数、支付金额）
- 派生指标：原子 + 时间周期（近7日订单数）
- 复合指标：多个原子计算（转化率 = 下单UV / 访问UV）
