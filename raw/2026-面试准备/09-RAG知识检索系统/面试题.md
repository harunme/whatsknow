## 答案

### A1: RAG 核心原理
**[10秒结论]** RAG（检索增强生成）通过外挂知识库解决 LLM 知识截止和幻觉问题；RAG 适合知识频繁更新、需溯源场景，Fine-tuning 适合固定模式、高频调用的任务。

**[90秒拆解]**
- **RAG 解决的问题**：① LLM 知识有截止日期，无法获取最新信息；② LLM 会产生幻觉（hallucination），生成看似合理但错误的内容；③ LLM 无法访问私有数据
- **RAG 流程**：用户问题 → Query Embedding → 向量数据库检索 Top-K → 将 K 个文档作为上下文 → LLM 生成 → 答案输出
- **RAG vs Fine-tuning**：
  - RAG：知识外挂，实时可更新，答案可溯源，但检索质量依赖 Embedding
  - Fine-tuning：知识内化到模型权重，推理快，但更新需重新训练（成本高）
- **选型原则**：先 RAG，效果不够再考虑 Fine-tuning；RAG 的维护成本远低于 Fine-tuning

**[20秒收尾]** SOC 平台的 SOP 和历史案例需要实时更新，RAG 是正确选择；如果是固定的分类模板，Fine-tuning 更合适。

---

### A2: Embedding 与向量数据库
**[10秒结论]** Embedding 将文本映射为固定维度向量，语义相近的文本向量距离更近；Qdrant（高性能专有云）、Milvus（超大规模）、Chroma（轻量原型）各有适用场景。

**[90秒拆解]**
- **主流中文模型**：BGE-large-zh-v1.5（1024维，本地部署）、GTE-large-zh（高精度）、text2vec-base-chinese（轻量）
- **向量数据库选型**：
  - Qdrant：Rust 实现，HNSW 参数精细可调，混合检索原生支持，P99 < 50ms，适合专有云生产
  - Milvus：分布式架构，支持10亿级向量，但运维复杂，适合超大规模
  - Chroma：Python 原生，轻量，适合本地开发/原型，内存模式即用
- **HNSW**：分层可导航小世界图，O(log N) 搜索复杂度；m 控制每层连接数，ef 控制搜索宽度
- **维度不是越高越好**：768→1024 有收益，再高收益递减；BGE-large（1024d）vs text-embedding-3-large（3072d），中文 SOC 场景精度差异 < 2%，但延迟和存储成本差异 > 3x

**[20秒收尾]** SOC 平台用 Qdrant + BGE-large-zh-v1.5，本地部署保证数据安全，P99 延迟 < 50ms。

### A3: Chunking 策略
**[10秒结论]** 固定窗口按 token 数硬切，语义分块按段落边界切保证语义完整；chunk_size=512 tokens 是 SOC 场景最佳平衡点，overlap=64 tokens 保留跨 chunk 上下文。

**[90秒拆解]**
- **固定窗口分块**：按 chunk_size 硬切，简单但可能切断句子
  ```python
  def fixed_window(text, chunk_size=512, overlap=64):
      start = 0
      while start < len(text):
          yield text[start:start+chunk_size]
          start += chunk_size - overlap
  ```
- **语义分块**：按段落切分，合并至阈值上限，保证语义完整性
- **Trade-off**：
  - < 128 tokens：上下文稀缺，关系断裂
  - > 1024 tokens：稀释核心信息，截断风险大
  - 512 tokens：SOC 场景最佳平衡（召回率+精确率）
- **SOP vs 历史案例**：SOP 用层级 chunk（保留步骤链）；历史案例用事件边界 chunk（每个事件独立成块）

**[20秒收尾]** chunking 是 RAG 的第一步，策略选错后续优化再牛也没用；先用语义分块 + 512 tokens，实测后再调。

### A4: 混合检索与 RRF
**[10秒结论]** 向量检索擅长语义相似，BM25 擅长专有名词（CVE、IP、进程名）精确匹配，两者融合覆盖更多查询类型；RRF 通过排名取倒数求和融合多路结果，k=60 是实验最优值。

**[90秒拆解]**
- **为什么混合**：向量检索对 "恶意软件" ↔ "病毒" 等语义相近词效果好；但 CVE 编号、IP 地址这类专有名词，向量相似度可能很低，必须靠 BM25 精确匹配
- **RRF 公式**：$RRF_{doc} = \sum_{r \in R} \frac{1}{k + rank_r(doc)}$
  - doc 在各检索路的排名 rank 越靠前，贡献越多
  - k=60 防止排名相同导致得分完全相同（平滑因子）
- **k 的影响**：k 过小 → 排名靠后文档被过度惩罚；k 过大 → 融合效果不明显；k=60 是 RRF 论文实验验证的经验最优值

**[20秒收尾]** Qdrant 原生支持 dense+sparse 混合检索，RRF 两阶段融合，端到端准确率提升 20-30 个百分点。

### A5: Reranker 重排序
**[10秒结论]** Reranker 用交叉编码器对 query+doc 做精细化相关性打分，将 ANN 检索的 top-20 重新排序输出 top-3；top_n=3 和 threshold=0.5 是 SOC 场景的经验参数。

**[90秒拆解]**
- **分工**：向量检索（ANN）快速召回 20 个候选 → Reranker 精细排序输出 3 个
- **BGE Reranker v2-m3**：交叉编码器，对每个 (query, doc) 配对独立计算相似度，比向量检索的双塔模型精度更高
- **参数设置**：
  - top_n=3：最终返回 3 条结果，太多让 LLM 上下文超载
  - threshold=0.5：低于 0.5 分直接丢弃，避免低相关 chunk 污染答案
- **延迟优化**：FP16 量化（延迟降低 40-60%）、批量推理（batch_size=32）、提前过滤低置信候选
- **何时不值得用**：召回候选已非常准确（如 < 10 个 doc）、对延迟极敏感（如 < 50ms）

**[20秒收尾]** 两阶段检索（ANN + Reranker）使准确率从 60% 提升到 83%，代价是增加约 30ms 延迟，是值得的 trade-off。

### A6: 向量检索调优
**[10秒结论]** m（每层连接数）和 ef_construct（构建索引时的搜索宽度）控制精度与内存的 trade-off；召回率通过 Recall@K 指标测量，生产应达到 Recall@20 > 0.85。

**[90秒拆解]**
- **HNSW 参数**：
  - `m`：16（通用）/32（高精度），越大内存越高，召回越高
  - `ef_construct`：200-512，构建时动态列表大小，越大精度越高，但构建时间增加 2-4x
  - `hnsw_ef`（运行时）：64-256，搜索时 ef，越大精度越高但延迟越大
  - `indexing_threshold`：20000，达到该点数后后台开始建索引
- **召回率测量**：
  - Recall@K = 相关文档在 top-K 中的比例
  - MRR = 第一个相关文档排名的倒数均值
  - 生产目标：Recall@20 > 0.85，MRR > 0.65
- **内存估算**：向量数 × 维度 × 4字节（float32）≈ 原始体积，HNSW 额外增加 20-30% 内存开销

**[20秒收尾]** Qdrant 的 HNSW 参数（m=16, ef_construct=200, hnsw_ef=128）是 SOC 场景的工程经验值，召回率和延迟达到最优平衡。

### A7: RAG 效果评估
**[10秒结论]** 四个维度：检索质量（Recall@K/MRR）、生成质量（RAGAS faithfulness/relevancy）、端到端准确率（分析师采纳率）、系统性能（P50/P99 延迟）；端到端准确率 78-83%。

**[90秒拆解]**
- **检索评估**：Recall@K 测量检索回来的 top-K 中有多少相关文档；MRR 测量第一个相关文档的平均排名
- **RAGAS**：自动评估工具，faithfulness（答案对上下文的忠实度）、answer_relevancy（答案与问题相关度）、context_precision（top contexts 精确率）
- **端到端测量**：分析师采纳率 = 采纳的回答数 / 总回答数；每周人工 review 100 条回答
- **Bad Case 类型**：CVE 编号无法召回（tokenizer 切分问题）、跨文档关系丢失（chunk 边界切断）、新文档无法召回（向量空间漂移）

**[20秒收尾]** SOC 场景 RAG 准确率 78-83% 意味着约 1/5 需要人工修正，是持续优化目标（90%）。

---

### A8: 知识库更新机制
**[10秒结论]** 增量更新通过 upsert 实现（新文档生成向量直接写入），alias 切换保证零停机（全量重建 → alias 原子切换）；定期全量重建解决向量空间漂移。

**[90秒拆解]**
- **增量更新**：单文档变更 → 生成新向量 → upsert 到在线 collection；增量写入触发后台 segment 合并
- **零停机全量重建**：创建新 collection → 全量数据重建索引 → alias 原子切换（Qdrant 支持 collections_alias）
- **向量空间漂移**：新增文档的向量分布可能与旧文档有偏差，定期全量重索引（每月一次）保证一致性
- **更新判断**：用 content hash 判断内容是否变化，未变化跳过写入；变更记录 version diff 用于审计

**[20秒收尾]** 告警 SOP 文档变更触发增量更新，Qdrant alias 切换保证用户无感知，版本 diff 记录用于问题溯源。
