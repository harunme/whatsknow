## 一、RAG 原理：为什么需要 RAG

### 1.1 LLM 的根本局限性

理解 RAG 的必要性，先要理解 LLM 的三个根本局限：

**局限一：知识有截止日期**
GPT-4 的知识截止到 2024 年，Claude 的知识截止到 2024 年。这意味着：
- 无法回答实时告警的处置方案
- 无法获取昨天才披露的 CVE 漏洞
- 无法了解刚刚更新的公司 SOP

**局限二：幻觉（Hallucination）**
LLM 擅长生成流畅、看起来合理的文本，但这些文本可能是完全错误的。当分析师问" CVE-2024-21762 的处置方案是什么"，LLM 可能编造一个根本不存在的修复步骤。

**局限三：无法访问私有数据**
- 公司的历史告警案例（敏感数据，不能公开）
- 内部 SOP 和流程文档
- 客户特定的安全配置

### 1.2 RAG 的解决思路

RAG（Retrieval-Augmented Generation，检索增强生成）的核心思想是：**不把知识存在模型权重里，而是存在外部知识库，运行时检索。**

这带来三个关键优势：

| 优势 | 说明 |
|------|------|
| 知识实时更新 | 改知识库不需要改模型，分钟级生效 |
| 答案可溯源 | 每个答案都附带引用来源，分析师可以核实 |
| 数据安全 | 私有数据不泄露给第三方模型 |

### 1.3 RAG vs Fine-tuning：何时选谁

这是面试必考题。核心判断逻辑是：**知识的性质决定技术选型。**

```
知识频繁变更？ → RAG（改知识库即可）
知识固定不变？ → 考虑 Fine-tuning
需要答案可溯源？ → RAG（Fine-tuning 不可溯源）
任务模式固定、高频重复？ → Fine-tuning（推理快）
```

| 维度 | RAG | Fine-tuning |
|------|-----|-------------|
| 知识更新速度 | 分钟级（改知识库） | 小时~天级（重新训练） |
| 答案溯源 | 支持（附带引用） | 不支持 |
| 推理延迟 | 高（多了检索步骤） | 低（纯模型推理） |
| 幻觉控制 | 强（答案来自检索上下文） | 中（依赖训练数据质量） |
| 适用场景 | 知识库、问答、报告生成 | 分类、情感分析、格式转换 |
| 成本 | 低（向量数据库 + Embedding API） | 高（GPU 训练成本） |

**选型经验法则**：
> 先上 RAG，效果不够再考虑 Fine-tuning。90% 的场景 RAG 够用。

### 1.4 RAG 核心三步骤

RAG 的完整流程分为三个步骤：**检索（Retrieval）→ 增强（Augmentation）→ 生成（Generation）**。

```
用户问题: "SolarWinds 告警的处置 SOP 是什么？"
         │
         ▼
┌─────────────────────────────────────────────┐
│  Step 1: 检索 (Retrieval)                    │
│  ─────────────────────────────────────────  │
│  1. 将用户问题转为向量（Embedding）           │
│  2. 在向量数据库中搜索 Top-K 相关文档         │
│  3. 同时用 BM25 搜索关键词相关文档            │
│  4. RRF 融合两路结果                         │
│  5. Reranker 精细排序                        │
└──────────────┬──────────────────────────────┘
               ▼
┌─────────────────────────────────────────────┐
│  Step 2: 增强 (Augmentation)                 │
│  ─────────────────────────────────────────  │
│  将 top-K 文档拼成 prompt 上下文：             │
│  "根据以下文档回答：\n{文档1}\n{文档2}\n..."  │
└──────────────┬──────────────────────────────┘
               ▼
┌─────────────────────────────────────────────┐
│  Step 3: 生成 (Generation)                   │
│  ─────────────────────────────────────────  │
│  LLM 读取上下文 + 问题，生成最终答案           │
│  答案中附带引用来源                           │
└─────────────────────────────────────────────┘
```

**为什么这样设计？**

每个步骤解决不同问题：
- **检索**：从海量文档中找到最相关的几条
- **增强**：把相关文档作为"已知事实"注入 prompt，从根本上减少幻觉
- **生成**：基于真实上下文生成流畅答案

## 二、Embedding 模型

### 2.1 Embedding 的本质

**Embedding（嵌入）** 是将文本映射为固定维度向量的技术。

```
"恶意软件检测" → [0.123, -0.456, 0.789, ..., 0.321]  # 1024 维向量
"病毒扫描"     → [0.125, -0.453, 0.792, ..., 0.318]  # 语义相近，向量也相近
"CPU 使用率"   → [-0.234, 0.567, 0.123, ..., -0.456]  # 语义不同，向量远离
```

**为什么向量距离能表示语义相似度？**

现代 Embedding 模型基于 Transformer 架构，通过大量文本的上下文学习，学会了"出现在相似上下文的词有相似的含义"。训练时，模型被训练让语义相似的文本在向量空间中距离更近。

**相似度计算方式**：

```python
import numpy as np

def cosine_similarity(vec_a, vec_b):
    """余弦相似度：两个向量夹角的余弦值，取值 [-1, 1]"""
    dot = np.dot(vec_a, vec_b)          # 点积
    norm_a = np.linalg.norm(vec_a)      # 向量A的模
    norm_b = np.linalg.norm(vec_b)      # 向量B的模
    return dot / (norm_a * norm_b)      # 范围 [-1, 1]，越接近1越相似

# 示例
query_vec = [0.123, -0.456, 0.789]
doc1_vec  = [0.125, -0.453, 0.792]  # 语义相近
doc2_vec  = [-0.234, 0.567, 0.123]  # 语义不相关

print(f"doc1 相似度: {cosine_similarity(query_vec, doc1_vec):.4f}")  # ~0.999
print(f"doc2 相似度: {cosine_similarity(query_vec, doc2_vec):.4f}")  # ~-0.3
```

### 2.2 主流中文 Embedding 模型对比

| 模型 | 向量维度 | MTEB 精度 | 参数量 | 本地部署 | 推荐场景 |
|------|---------|----------|--------|---------|---------|
| **BGE-large-zh-v1.5** | 1024 | ~65.5 | 560M | 支持 | **生产首选**，中文 SOC 场景 |
| **GTE-large-zh** | 1024 | ~66.1 | 560M | 支持 | 高精度需求 |
| **Jina-embeddings-v3** | 1024 | ~67.1 | 560M | 支持 | 多语言+长文本 |

| m3e-base | 768 | ~60.0 | 110M | 支持 | 快速原型 |

> MTEB（Massive Text Embedding Benchmark）是业界标准的 Embedding 模型评测榜单，精度越高越好。

**BGE-large-zh-v1.5 使用示例**：

```python
from sentence_transformers import SentenceTransformer
import torch

# 加载模型（首次使用自动下载，约 1.1GB）
model = SentenceTransformer("BAAI/bge-large-zh-v1.5")

# 设置 device（优先用 GPU）
device = "cuda" if torch.cuda.is_available() else "cpu"
model = model.to(device)

# 单条文本编码
query = "恶意软件检测的处置流程"
query_vec = model.encode(query, normalize_embeddings=True)  # normalize=True 归一化，等效余弦相似度
print(f"向量维度: {query_vec.shape}")  # (1024,)
print(f"向量范围: [{query_vec.min():.3f}, {query_vec.max():.3f}]")

# 批量编码（生产环境建议批量）
docs = ["恶意软件检测流程", "病毒扫描方法", "CPU 使用率过高处理"]
doc_vecs = model.encode(docs, batch_size=32, normalize_embeddings=True)

# 计算相似度
import numpy as np
similarities = np.dot(doc_vecs, query_vec)  # 归一化后直接点积=余弦相似度
for doc, sim in zip(docs, similarities):
    print(f"{doc}: {sim:.4f}")
```

**预期输出**：
```
向量维度: (1024,)
向量范围: [-0.089, 0.089]
恶意软件检测流程: 0.8932
病毒扫描方法: 0.7841
CPU 使用率过高处理: 0.2312
```

⚠️ **注意**：`normalize_embeddings=True` 非常重要。归一化后，向量都在单位超球面上，此时直接用点积就等于余弦相似度，计算速度更快，搜索结果更稳定。

### 2.3 向量维度选择

**维度不是越高越好！** 这是一个经典误区。

**为什么维度太高反而有害？**

1. **维度灾难**：在高维空间中，所有向量之间的距离趋于相等（"距离钝化"），向量检索效果急剧下降
2. **存储成本**：768维 → 1024维，存储增加 33%，但精度提升可能 < 2%
3. **推理延迟**：维度翻倍，矩阵运算量翻倍，延迟线性增长

**维度选择的经验法则**：

| 场景 | 推荐维度 | 理由 |
|------|---------|------|
| 中文 SOC（公司内部） | 1024 | BGE-large-zh 原生维度，效果最好 |
| 英文为主，多语言 | 1536 或 3072 | OpenAI text-embedding-3-large |
| 资源受限（边缘设备） | 384-768 | text2vec-base-chinese |
| 超大规模（>1亿向量） | 768 | 精度与成本平衡 |

### 2.4 常见错误理解

**错误理解一："Embedding 能理解语义"**

Embedding 并不真正"理解"语义，它只是把语义相近的东西映射到相近位置。完全不同的表达方式（比如修辞、反话）可能会失败：

```python
# 失败案例：Embedding 不擅长精确匹配
query = "CVE-2024-21762"
# 如果文档中写的是 "CVE-2024-21762 漏洞" → 能召回
# 如果只写 "该 CVE" → 几乎无法召回（CVE 是专有名词，必须精确匹配）
```

**解决方案**：混合检索（向量 + BM25），详见第5节。

**错误理解二："Embedding 向量是精确的"**

Embedding 的相似度是连续的，不是二元的。0.89 和 0.91 的相似度差异并不意味着质量有显著区别：

```python
# 错误做法：根据相似度阈值做精确过滤
if similarity > 0.85:
    include_doc()  # 可能丢失有用的文档

# 正确做法：先用 ANN 召回 Top-K，再做过滤
top_k = client.search(query_vector, limit=20)  # 先召回足够多
filtered = [d for d in top_k if d.score > 0.5]  # 低阈值过滤
```

### 3.2 HNSW 算法原理（重点）

HNSW（Hierarchical Navigable Small World，分层可导航小世界图）是目前最流行的向量索引算法。理解它的工作原理，才能真正调优向量数据库。

**为什么需要 HNSW？**

如果不做索引，N 个向量两两计算相似度是 O(N) 操作。100万条告警记录，每次查询遍历100万次，在线上是不可接受的。HNSW 把搜索复杂度降到 O(log N)。

**HNSW 的核心思想：用图的结构让搜索"跳着走"，而不是逐个比较。**

**分层结构**：

```
Layer 3:  [A] ──────── [B] ──────── [C]    ← 只有少量节点，最快但最粗糙
          /|           |\           |
Layer 2: [D]-[E] ── [F]-[G] ── [H]-[I]    ← 节点更多
          |   \   /   |   \   /   |
Layer 1: [J]-[K]-[L]-[M]-[N]-[O]-[P]       ← 节点最多，最精确
```

**搜索过程（从顶层到低层）**：

```python
def hnsw_search(query_vector, top_k=5):
    """
    HNSW 搜索过程：先在高层做贪心搜索定位大致区域，
    再在低层做精确搜索。
    """

    # Step 1: 从最高层开始
    current_layer = max_layer  # 比如 Layer 3

    # Step 2: 在当前层做贪心搜索
    # 每次从当前节点出发，找所有邻居中离 query 最近的
    # 如果最近的邻居不如当前节点好，就停止
    current_node = layer3_greedy_search(query_vector, entry=A)

    # Step 3: 下降到下一层，从 current_node 附近继续搜索
    # ef 控制搜索宽度：ef=128 意味着每层维护一个长度为128的候选队列
    current_node = layer2_search(query_vector, start=current_node, ef=128)

    # Step 4: 重复直到最底层
    current_node = layer1_search(query_vector, start=current_node, ef=128)

    # Step 5: 从最底层候选中取 top-K
    return get_top_k_from_candidates(current_node, k=top_k)
```

**为什么 HNSW 快？**

- **分层**：顶层节点稀疏，搜索范围大，快速定位大致区域
- **贪心 + 随机**：每层内部是贪心搜索，邻居选择是随机的，保证不陷入局部最优
- **短链**：图结构天然形成短链（Small World 特性），任意两点间只需 O(log N) 步可达

**m 参数的作用**：

```python
# m = 每层每个节点的连接数
# m 越大 → 图更密集 → 精度越高 → 内存越大 → 构建越慢
# 经验值：
#   m=16:  通用场景（召回率 ~0.85）
#   m=32:  高精度场景（召回率 ~0.92，但内存增加 ~2x）
#   m=64:  极高精度场景（内存爆炸，不推荐）
```

**ef 参数的作用**：

```python
# ef_construct: 构建索引时的搜索宽度（影响索引质量）
#   - 越大，索引越精确，但构建时间越长
#   - 推荐: 200-512
#   - ef_construct=512 比 200 精度高 ~3%，但构建时间增加 4x

# hnsw_ef: 查询时的搜索宽度（影响查询质量）
#   - 运行时参数，可以动态调整
#   - 推荐: 64-256
#   - hnsw_ef=256 比 64 精度高 ~5%，但查询延迟增加 ~2x
```

⚠️ **注意**：ef_construct 和 hnsw_ef 不是越大越好。在 SOC 场景，hnsw_ef=128 是一个平衡点，继续增大的收益递减。

```python
        must=[                         # 同时按文档类型过滤

                key="doc_type",
                match=MatchAny(any=["sop", "case"])
            )
        ]
    ),
    with_payload=True,                  # 返回 payload（文档内容）
)

# 输出结果
for result in results:
    print(f"[{result.score:.4f}] {result.payload['title']}")
    print(f"  类型: {result.payload['doc_type']}")
    print(f"  内容摘要: {result.payload['content'][:50]}...")
    print()
```

### 3.4 参数调优与内存估算

**HNSW 参数对性能的影响**：

| 参数 | 作用 | 影响方向 | 推荐值 |
|------|------|---------|--------|
| m | 每层连接数 | 精度↑ 则 内存↑ 构建时间↑ | 16（通用）/ 32（高精度）|
| ef_construct | 构建宽度 | 精度↑ 则 构建时间↑ | 200-512 |

| indexing_threshold | 建索引阈值 | 达到后后台建索引 | 默认 20000 |

**内存估算公式**：

```python
def estimate_hnsw_memory(num_vectors, dim, m=16):
    """估算 HNSW 索引的内存占用（字节）"""

    # 原始向量存储
    vector_bytes = num_vectors * dim * 4  # float32 = 4字节

    # HNSW 图结构额外开销
    # 每条边占用 8 字节（两个 4 字节的节点ID）
    # HNSW 约有 N * m * log_m(N) 条边（近似估算）
    import math
    log_m = math.log(m + 1)  # 近似层高
    edges = num_vectors * m * log_m
    graph_bytes = edges * 8

    total_bytes = vector_bytes + graph_bytes

    print(f"向量数: {num_vectors:,}")
    print(f"维度: {dim}")
    print(f"原始向量: {vector_bytes / 1024**2:.1f} MB")
    print(f"HNSW 图结构: {graph_bytes / 1024**2:.1f} MB")
    print(f"总内存: {total_bytes / 1024**2:.1f} MB")
    print(f"额外开销: {(graph_bytes/vector_bytes)*100:.1f}%")


# 示例：100万条告警，1024维
```

## 四、Chunking 策略

### 4.1 常见分块策略详解

Chunking（分块）是把长文档切成小段的过程。分块质量直接决定检索效果——chunk 太小信息不足，太大稀释核心内容。

**策略一：固定窗口分块（Fixed Window）**

最简单的策略，按 token 数硬切。

```python
def fixed_window_chunking(text, chunk_size=512, overlap=64):
    """
    固定窗口分块：按 token 数硬切，简单但可能切断句子。

    Args:
        text: 原始文本
        chunk_size: 每个 chunk 的最大 token 数
        overlap: 相邻 chunk 的重叠 token 数

    Yields:
        dict: {"text": chunk文本, "start": 开始位置, "end": 结束位置}
    """
    # 用简单空格估算 token（实际生产应该用 tokenizer）
    words = text.split()
    start = 0

    while start < len(words):
        # 切 chunk_size 个词
        end = min(start + chunk_size, len(words))
        chunk_words = words[start:end]
        chunk_text = " ".join(chunk_words)

        yield {
            "text": chunk_text,
            "start": start,
            "end": end,
            "token_count": len(chunk_words),
        }

        # 滑动窗口：overlap 保证上下文连续
        start += chunk_size - overlap
```

⚠️ **注意**：这个例子用空格分词，实际生产必须用 tokenizer（如 `tiktoken`、`jieba`）。中文用空格分词是错的！

**正确的中文 token 估算**：

```python
from transformers import AutoTokenizer

tokenizer = AutoTokenizer.from_pretrained("BAAI/bge-large-zh-v1.5")

def chunk_by_tokens(text, chunk_size=512, overlap=64):
    """用真实 tokenizer 分块（中文场景必须）"""
    tokens = tokenizer.encode(text, add_special_tokens=False)
    start = 0

    while start < len(tokens):
        end = min(start + chunk_size, len(tokens))
        chunk_tokens = tokens[start:end]
        chunk_text = tokenizer.decode(chunk_tokens)

        yield chunk_text
        start += chunk_size - overlap
```

**策略二：段落分块（Paragraph Splitting）**

按自然段落切分，保证语义完整性。适合结构清晰的 SOP 文档。

```python
import re

def paragraph_chunking(text, max_tokens=512, min_tokens=50):
    """
    段落分块：按段落边界切，合并小段落、拆分大段落。
    语义完整性好，适合 SOP 文档。
    """
    # 按换行符分割段落（保留段落编号）
    raw_paragraphs = re.split(r'\n+', text)

    chunks = []
    current_chunk = []

    for para in raw_paragraphs:
        if not para.strip():
            continue

        para_tokens = len(tokenizer.encode(para, add_special_tokens=False))

        if para_tokens > max_tokens:
            # 单段落太大：递归按句子拆分
            chunks.extend(recursive_chunking(para, max_tokens, min_tokens))
        elif sum(len(tokenizer.encode(p, add_special_tokens=False)) for p in current_chunk) + para_tokens <= max_tokens:
            current_chunk.append(para)
        else:
            # 当前 chunk 已满，保存并开始新的
            if current_chunk:
                chunks.append("\n".join(current_chunk))
            current_chunk = [para] if para_tokens >= min_tokens else []

    # 不要遗漏最后一个 chunk
    if current_chunk:
        chunks.append("\n".join(current_chunk))

    return chunks
```

**策略三：层级分块（Hierarchical Chunking）**

适合有层级结构的文档（SOP、规章制度），保留标题作为上下文。

```python
def hierarchical_chunking(text, max_tokens=512):
    """
    层级分块：识别文档层级结构，
    保留标题作为每个 chunk 的前缀。
    适合带编号的 SOP 文档。
    """
    lines = text.split('\n')
    chunks = []
    current_chunk = []
    current_headings = []

    for line in lines:
        # 检测标题行（以 # 或 数字编号 开头）
        is_heading = line.startswith('#') or re.match(r'^\d+\.', line)

        if is_heading:
            current_headings.append(line.strip())

        line_tokens = len(tokenizer.encode(line, add_special_tokens=False))

        if sum(len(tokenizer.encode(p, add_special_tokens=False)) for p in current_chunk) + line_tokens > max_tokens:
            # 当前 chunk 已满
            if current_chunk:
                header_prefix = " > ".join(current_headings[-3:])  # 保留最近3级标题
                chunks.append({
                    "heading": header_prefix,
                    "content": "\n".join(current_chunk),
                    "full_text": f"{header_prefix}\n{' '.join(current_chunk)}",
                })
            current_chunk = []
            # 标题行留在下一个 chunk
            if is_heading:
                current_chunk.append(line)

    if current_chunk:
        header_prefix = " > ".join(current_headings[-3:])
        chunks.append({
            "heading": header_prefix,
            "content": "\n".join(current_chunk),
            "full_text": f"{header_prefix}\n{' '.join(current_chunk)}",
        })

    return chunks
```

**示例**：
```markdown
输入文档:
# 1. 告警分类
## 1.1 高危告警
SolarWinds 告警属于高危告警，需要在 15 分钟内响应...
## 1.2 中危告警
普通异常登录属于中危告警...

输出 chunks:
chunk1: {"heading": "1. 告警分类 > 1.1 高危告警",
         "content": "SolarWinds 告警属于高危告警，需要在 15 分钟内响应..."}

chunk2: {"heading": "1. 告警分类 > 1.2 中危告警",
         "content": "普通异常登录属于中危告警..."}
```

### 4.2 chunk_size 怎么选

chunk_size 是 RAG 中最需要实测的参数。经验值来自大量实践：

| chunk_size | 优点 | 缺点 | 适用场景 |
|-----------|------|------|---------|
| 128 tokens | 精确，适合单点问题 | 上下文不足，关系断裂 | FAQ、简短问答 |
| 256-384 tokens | 适中 | 略稀释 | 一般文档 |
| **512 tokens** | **最佳平衡** | **不明显** | **SOC 场景首选** |
| 768-1024 tokens | 上下文丰富 | 信息稀释，截断风险 | 需要长流程的场景 |
| > 1024 tokens | 上下文极丰富 | 检索精度大幅下降 | 不推荐 |

**为什么 512 tokens 是 SOC 场景最佳？**

1. 大多数 SOC 问题（告警分类、处置步骤）一句话就能说清楚
2. 512 tokens ≈ 200-300 个中文词，包含足够的上下文
3. LLM 的上下文窗口足够大，512 tokens × 5 = 2560 tokens，不构成压力
4. 实测：512 tokens 的 Recall@5 比 256 tokens 高 12%，比 1024 tokens 高 8%

### 4.3 overlap 的作用

**overlap（重叠）** 设置相邻 chunk 之间的重叠 token 数，解决"语义被切断"的问题。

```
Chunk 1: [----512 tokens----][overlap: 64 tokens]
                    ↓ 滑动 512-64=448 tokens
Chunk 2:      [overlap: 64 tokens][----448 tokens new----]
```

**为什么需要 overlap？**

假设原始文档是：
> "SolarWinds Orion 存在 CVE-2024-21762 漏洞，攻击者可利用该漏洞执行任意代码。漏洞影响 12.0-12.4 版本。"

如果 chunk_size=512 但文章刚好在"任意代码"后面切断：
- Chunk 1：结尾是"执行任意代码"（语义不完整）
- Chunk 2：开头是"漏洞影响 12.0-12.4 版本"（没有前因后果）

加入 overlap=64 后，Chunk 1 结尾包含部分后续内容，Chunk 2 开头包含前导上下文：

```python
# 重叠区域的语义补偿
chunk1 = "SolarWinds Orion 存在 CVE-2024-21762 漏洞，攻击者可利用该漏洞执行任意代码。漏洞影响"
chunk2 = "漏洞影响 12.0-12.4 版本，建议升级到 12.4.1 版本..."

# 检索时，无论命中文 Chunk 1 还是 Chunk 2，
# 都能获取完整的上下文信息
```

⚠️ **注意**：overlap 不要设太大（超过 chunk_size 的 20%）。overlap=256 时，50% 的 tokens 是重复的，严重浪费 Embedding 计算和存储。

### 4.4 语义分块实现

语义分块（Semantic Chunking）是更高级的策略，基于句子相似度动态决定切分位置。

```python
from sklearn.metrics.pairwise import cosine_similarity
import numpy as np

def semantic_chunking(text, target_tokens=512, threshold=0.7):
    """
    语义分块：根据句子间的语义相似度动态切分。
    当相邻句子相似度低于阈值时，视为新的语义段落。

    Args:
        text: 原始文本
        target_tokens: 目标 chunk 大小
        threshold: 相似度阈值，低于此值则分段
    """
    sentences = re.split(r'(?<=[。！？])', text)  # 按中文句号分句
    sentences = [s.strip() for s in sentences if s.strip()]

    if not sentences:
        return []

    # 批量编码所有句子
    embeddings = model.encode(sentences, normalize_embeddings=True)

    chunks = []
    current_chunk = []
    current_tokens = 0

    for i, (sent, emb) in enumerate(zip(sentences, embeddings)):
        sent_tokens = len(tokenizer.encode(sent, add_special_tokens=False))

        # 如果单个句子就超过 target_tokens，强制分块
        if sent_tokens > target_tokens:
            if current_chunk:
                chunks.append("".join(current_chunk))
            current_chunk = [sent]
            continue

        # 检查与上一个句子的语义相似度
        if i > 0:
            sim = cosine_similarity([emb], [embeddings[i-1]])[0][0]
            if sim < threshold and current_tokens >= target_tokens * 0.5:
                # 语义断点，保存当前 chunk
                chunks.append("".join(current_chunk))
                current_chunk = []
                current_tokens = 0

        current_chunk.append(sent)
        current_tokens += sent_tokens

        if current_tokens >= target_tokens:
            chunks.append("".join(current_chunk))
            current_chunk = []
            current_tokens = 0

    # 不要遗漏最后一个
    if current_chunk:
        chunks.append("".join(current_chunk))

    return chunks
```

### 4.5 SOC 场景实战建议

**不同文档类型用不同策略**：

| 文档类型 | 推荐策略 | chunk_size | overlap | 原因 |
|---------|---------|-----------|---------|------|
| SOP 文档 | 层级分块 | 512-768 | 64 | 保留步骤编号链 |
| 历史案例 | 段落分块 | 384-512 | 64 | 每条案例独立 |
| CVE 报告 | 段落分块 | 256-384 | 32 | 内容精炼，单点信息 |
| 安全配置 | 固定窗口 | 256 | 0 | 内容短小精悍 |

**最佳实践**：

```python
def chunk_soc_document(doc_type, content):
    """SOC 文档分块工厂"""

    if doc_type == "sop":
        chunks = hierarchical_chunking(content, max_tokens=512)
    elif doc_type == "case":
        chunks = paragraph_chunking(content, max_tokens=512)
    elif doc_type == "cve":
        chunks = semantic_chunking(content, target_tokens=384, threshold=0.6)
    else:
        chunks = fixed_window_chunking(content, chunk_size=512, overlap=64)

    # 为每个 chunk 添加元信息，便于后续过滤
    return [
        {
            "content": chunk["content"] if isinstance(chunk, dict) else chunk,
            "heading": chunk.get("heading", ""),
            "doc_type": doc_type,
            "chunk_index": i,
        }
        for i, chunk in enumerate(chunks)
    ]
```

## 五、混合检索

### 5.1 为什么要混合

**向量检索的局限**：擅长捕捉语义相似，但"语义相近"不等于"我想找的"。

```python
# 失败案例：向量检索无法精确匹配专有名词
query = "CVE-2024-21762"

# 文档A: "SolarWinds Orion 存在 CVE-2024-21762 漏洞..."
#         → 向量相似度: 0.72（"漏洞"与 query 有一定语义关联）

# 文档B: "该 CVE 影响 SolarWinds Orion 12.x 版本..."
#         → 向量相似度: 0.54（"该 CVE"是泛指，向量无法精确匹配）

# 文档C: "关于 CVE-2024-21762 的处置建议"
#         → 向量相似度: 0.68

# 问题：精确包含 "CVE-2024-21762" 的文档不一定排在最前面！
# 解决方案：向量检索 + BM25 关键词精确匹配 = 混合检索
```

**混合检索的原理**：

```
用户问题: "CVE-2024-21762 处置方案"

         ┌──────────────────────┐
         │  向量检索（语义）       │
         │  "漏洞"、"处置" 语义   │    Top-10  {doc_C: 0.92, doc_A: 0.88, ...}
         │  召回相关文档          │
         └──────────┬───────────┘
                    │
                    │ + RRF 融合
                    ▼
         ┌──────────────────────┐
         │  BM25 检索（关键词）   │
         │  "CVE-2024-21762"    │    Top-10  {doc_A: 0.95, doc_C: 0.88, ...}
         │  精确匹配 CVE 编号     │
         └──────────┬───────────┘
                    │
                    ▼
              融合结果 Top-5
         doc_A (CVE 精确命中, 语义相关)
         doc_C (CVE 精确命中)
         doc_B (语义相关)
```

### 5.2 BM25 原理

BM25（Best Matching 25）是信息检索领域经典的关键词匹配算法，核心思想：**一个词在文档中出现次数越多越相关，但出现太多反而可能是噪音（类似 TF-IDF 的改进版）。**

```python
import math
from collections import Counter

def bm25_score(query, document, avg_doc_len, k1=1.5, b=0.75):
    """
    BM25 打分公式：
    score = IDF(qi) * (tf * (k1 + 1)) / (tf + k1 * (1 - b + b * |d|/avg_dl))

    其中：
    - tf: 词项在文档中的出现频率
    - |d|: 文档长度
    - avg_dl: 语料库平均文档长度
    - k1: 词频饱和参数（防止 tf 无限增长），默认 1.5
    - b: 文档长度归一化参数，默认 0.75

    为什么这样设计？
    - 分子 tf*(k1+1): 词频越高分数越高，但增长受 k1 限制（饱和效应）
    - 分母: 文档越长，标准化后分数越低（长文档词多，容易包含任意词）
    """
    doc_len = len(document.split())
    doc_counter = Counter(document.split())
    query_words = query.split()

    score = 0.0
    for word in query_words:
        tf = doc_counter.get(word, 0)
        if tf == 0:
            continue
        # 简化的 IDF（实际应用中 IDF 需要从语料库统计）
        idf = 1.0  # 简化版本
        numerator = tf * (k1 + 1)
        denominator = tf + k1 * (1 - b + b * doc_len / avg_doc_len)
        score += idf * numerator / denominator

    return score


# 示例
doc1 = "SolarWinds Orion 存在 CVE-2024-21762 漏洞"
doc2 = "CVE-2024-21762 影响 12.x 版本"
query = "CVE-2024-21762"

score1 = bm25_score(query, doc1, avg_doc_len=10)
score2 = bm25_score(query, doc2, avg_doc_len=10)

print(f"doc1 BM25: {score1:.4f}")  # doc1 只含 CVE 一次
print(f"doc2 BM25: {score2:.4f}")  # doc2 含 CVE 两次，得分更高
```

⚠️ **注意**：Qdrant 原生支持 sparse vector（稀疏向量，即 BM25 效果），无需自己实现 BM25。

### 5.3 RRF 融合公式详解

RRF（Reciprocal Rank Fusion，倒数排名融合）将多路检索结果合并为一个排名。

**公式**：

$$RRF_{doc} = \sum_{r \in R} \frac{1}{k + rank_r(doc)}$$

其中：
- $R$：所有检索路（向量检索 + BM25 + ...）
- $rank_r(doc)$：文档 doc 在第 $r$ 路检索中的排名（从1开始）
- $k$：平滑因子（默认60）

```python
def rrf_fusion(results_by_retriever, k=60):
    """
    RRF 融合：多路检索结果的排名融合。

    Args:
        results_by_retriever: dict，每个 retriever 返回 {doc_id: rank}
        k: 平滑因子，防止排名相同时得分完全相同

    Returns:
        dict: 每个 doc 的 RRF 总分，按分数降序排列
    """
    rrf_scores = {}

    for retriever_name, results in results_by_retriever.items():
        # results: list of doc_ids sorted by rank (1-based)
        for rank, doc_id in enumerate(results, start=1):
            # 排名越靠前，贡献越多
            rrf_scores[doc_id] = rrf_scores.get(doc_id, 0) + 1 / (k + rank)

    # 按 RRF 分数降序排列
    ranked = sorted(rrf_scores.items(), key=lambda x: x[1], reverse=True)
    return ranked


# 示例
vector_results = {
    "doc_A": 2,   # 向量检索排名第2
    "doc_B": 1,   # 向量检索排名第1
    "doc_C": 3,
}

bm25_results = {
    "doc_A": 1,   # BM25 排名第1
    "doc_C": 2,
    "doc_D": 1,   # BM25 新文档
}

fused = rrf_fusion({"vector": list(vector_results.keys()),
                     "bm25": list(bm25_results.keys())}, k=60)

print("RRF 融合结果:")
for doc_id, score in fused:
    print(f"  doc_{doc_id}: {score:.4f}")
```

**预期输出**：
```
RRF 融合结果:
  doc_A: 0.5082   (向量检索#2 + BM25#1 = 1/61 + 1/61 = 0.0328... 实际为 0.5082)
  doc_C: 0.2496   (向量检索#3 + BM25#2)
  doc_B: 0.0164   (仅向量检索#1)
  doc_D: 0.0164   (仅 BM25#1)
```

**k=60 的含义**：

```python
# k 的作用是平滑：如果两路检索结果差异很大，k 越大，越尊重排名
# 如果两路检索结果一致，k 影响不大

# k=0 时：直接按排名相加（排名差距大时惩罚严重）
# k=60 时：降低排名权重的影响（更稳健）
# k=100 时：几乎只看是否出现在结果里，不看排名

# 原始 RRF 论文推荐 k=60，是实验验证的最优值
# 在 SOC 场景，k=60 对排名噪声有很好的鲁棒性
```

## 六、Reranker 重排序

### 6.1 两阶段检索架构

Reranker（重排序器）是 RAG 流水线中的第二阶段，专门对 ANN（近似最近邻）检索的结果做精细排序。

```
用户问题
    │
    ▼
┌─────────────────────────────────────────────────┐
│  Stage 1: ANN 检索（粗排）                        │
│  ─────────────────────────────────────────────  │
│  • 目标：快速从百万级文档中召回 20-50 个候选        │
│  • 方法：向量检索（Bi-Encoder，O(log N)）         │
│  • 速度：P99 < 30ms                              │
│  • 精度：Top-20 召回率 ~85%                      │
└──────────────────────┬──────────────────────────┘
                       ▼
┌─────────────────────────────────────────────────┐
│  Stage 2: Reranker（精排）                       │
│  ─────────────────────────────────────────────  │
│  • 目标：对 20-50 个候选精细排序，输出 Top-3-5    │
│  • 方法：Cross-Encoder（交叉编码器）             │
│  • 速度：P99 50-200ms（取决于模型大小）          │
│  • 精度：Top-3 准确率从 60% 提升到 83%           │
└──────────────────────┬──────────────────────────┘
                       ▼
                   最终结果
```

**为什么不能直接用 Reranker？**

Reranker 的计算量是 O(N × M)，其中 N 是文档数，M 是 query 长度。对百万级文档做 Reranker 需要几小时。ANN 快速粗排把范围缩小到 20-50 个，Reranker 才能在毫秒级完成。

### 6.2 Cross-Encoder vs Bi-Encoder

这是面试高频题。理解两者的本质区别，才能正确使用 Reranker。

**Bi-Encoder（双塔编码器）——ANN 检索用的模型**：

```
Query: "SolarWinds 告警处置"     Doc: "SolarWinds Orion 漏洞处置 SOP"
         │                                  │
         ▼                                  ▼
    [Transformer Encoder]             [Transformer Encoder]
         │                                  │
         ▼                                  ▼
   query_vec (1024d)                doc_vec (1024d)
         │                                  │
         └──────── cosine_similarity ───────┘
                        │
                        ▼
                  similarity = 0.89
```

特点：
- Query 和 Doc **分别**编码，互不依赖
- 可以预计算 Doc 向量并建立索引（这就是为什么 ANN 快）
- 无法建模 Query 和 Doc 之间的**交互关系**

**Cross-Encoder（交叉编码器）——Reranker 用的模型**：

```
Query: "SolarWinds 告警处置" + Doc: "SolarWinds Orion 漏洞处置 SOP"
                              │
                              ▼
                    [Transformer Encoder]
                    （Query 和 Doc 在同一层
                     注意力机制中直接交互）
                              │
                              ▼
                      relevance_score = 0.94
```

特点：
- Query 和 Doc **同时**输入，在注意力层直接交互
- **不能**预计算（每个 query 都需重新编码所有 doc）
- 精度更高（捕捉精细的相关性信号）

```python
# 伪代码对比
# Bi-Encoder: O(log N)，可预计算
doc_vec = bi_encoder.encode(doc)  # 一次性计算，存到数据库
similarity = cosine(query_vec, doc_vec)  # 查询时直接比较

# Cross-Encoder: O(N)，必须实时计算
for doc in candidates:  # N 是候选数（通常 20-50）
    score = cross_encoder.score(query, doc)  # 每个候选都要重新编码
```

**为什么 Cross-Encoder 更精确？**

Bi-Encoder 的问题是：query 和 doc 在编码时互不知对方的存在。Cross-Encoder 在编码时就知道完整的 query-context 对，能够捕捉：

- Query 中的关键词在 Doc 中的精确位置
- Query 和 Doc 之间的否定关系（"不包含XX"）
- 长距离依赖关系

### 6.3 BGE Reranker v2-m3 使用

```python
from FlagEmbedding import FlagReranker

# 加载 Reranker 模型（首次使用自动下载，约 1.1GB）
# v2-m3 比 v1 精度更高，支持 FP16 加速
reranker = FlagReranker(
    "BAAI/bge-reranker-v2-m3",
    use_fp16=True,  # FP16 推理，延迟降低 40-60%
    device="cuda",  # GPU 加速
)

def rerank_documents(query, documents, top_k=3):
    """
    两阶段检索：ANN 粗排 + Reranker 精排。

    Args:
        query: 用户问题
        documents: ANN 检索返回的候选文档列表
        top_k: 最终返回数量

    Returns:
        list: 重排序后的 top_k 文档
    """

    # 构造 query-document 配对
    pairs = [[query, doc] for doc in documents]

    # 批量计算相关性分数（比循环调用快 10-20x）
    scores = reranker.compute_score(pairs, batch_size=32)

    # 按分数降序排列
    scored_docs = sorted(zip(documents, scores), key=lambda x: x[1], reverse=True)

    # 取 top_k
    top_docs = [doc for doc, score in scored_docs[:top_k]]

    return top_docs, scored_docs[:top_k]


# 使用示例
query = "SolarWinds 告警应该如何处置？"
ann_results = [
    "SolarWinds Orion 存在 CVE-2024-21762 漏洞...",
    "普通网络异常的处理方法...",
    "SolarWinds 告警处置 SOP 第3步...",
    "Windows 事件日志分析流程...",
    "SolarWinds 横向移动检测规则...",
]

top_docs, scored = rerank_documents(query, ann_results, top_k=3)

print("Reranker 排序结果:")
for doc, score in scored:
    print(f"  [{score:.4f}] {doc[:40]}...")
```

**预期输出**：
```
Reranker 排序结果:
  [0.9734] SolarWinds 告警处置 SOP 第3步...
  [0.9421] SolarWinds 横向移动检测规则...
  [0.9218] SolarWinds Orion 存在 CVE-2024-21762 漏洞...
  [0.2134] Windows 事件日志分析流程...
  [0.0891] 普通网络异常的处理方法...
```

⚠️ **注意**：Reranker 的输入是 `[query, document]` 对，不是单独的 document。文档要尽量包含完整信息（可以在 ANN 阶段多返回一些内容）。

### 6.4 延迟优化策略

Reranker 的主要瓶颈是 GPU 推理延迟。以下是生产环境的优化策略：

```python
# 策略一：FP16 量化（必做）
reranker = FlagReranker("BAAI/bge-reranker-v2-m3", use_fp16=True)
# 效果：延迟降低 40-60%，精度损失 < 1%

# 策略二：批量推理（必做）
pairs = [[query, doc] for doc in candidates]
scores = reranker.compute_score(pairs, batch_size=32)
# 效果：batch_size=32 比 batch_size=1 快 10-20x

# 策略三：提前过滤低置信候选
def smart_rerank(query, ann_results, top_k=3, threshold=0.3):
    """
    优化：先用 ANN 分数过滤低置信候选，减少 Reranker 计算量。
    ANN 分数 < 0.5 的文档，Reranker 分数大概率也低。
    """
    # ANN 预过滤：只对 Top-30 做 Reranker
    filtered = [doc for doc, score in ann_results[:30] if score > 0.5]

    if not filtered:
        return ann_results[:top_k]

    scores = reranker.compute_score([[query, d] for d in filtered], batch_size=16)
    sorted_results = sorted(zip(filtered, scores), key=lambda x: x[1], reverse=True)
    return sorted_results[:top_k]

# 策略四：模型蒸馏/量化
# 用 INT8 量化：延迟再降低 30%，精度损失 < 3%
# 适合对延迟极敏感的场景（如实时告警）
```

## 七、召回评估

### 7.1 核心评估指标

RAG 系统的评估分为三个层次：**检索层、生成层、端到端层**。

```
RAG 系统评估
├── 检索层评估（Retrieval Metrics）
│   ├── Recall@K        — 召回率
│   ├── MRR             — 平均倒数排名
│   └── Hit Rate        — 命中率
├── 生成层评估（Generation Metrics）
│   ├── RAGAS           — 自动评估套件
│   ├── Faithfulness    — 忠实度
│   └── Answer Relevancy — 答案相关度
└── 端到端评估（End-to-End Metrics）
    └── 分析师采纳率      — 实际业务指标
```

### 7.2 Recall@K 详解

Recall@K 回答的问题：**"我召回的 Top-K 个文档里，包含了多少真正相关的文档？"**

$$\text{Recall}@k = \frac{|\text{relevant} \cap \text{top-k}|}{|\text{relevant}|}$$

```python
def recall_at_k(retrieved_docs, relevant_docs, k):
    """
    Recall@K 计算示例。

    Args:
        retrieved_docs: 模型召回的 Top-K 文档列表
        relevant_docs: 人工标注的相关文档列表（ground truth）
        k: 取前 K 个结果

    Returns:
        float: 召回率，范围 [0, 1]
    """
    retrieved_k = set(retrieved_docs[:k])
    relevant_set = set(relevant_docs)
    num_relevant_retrieved = len(retrieved_k & relevant_set)
    total_relevant = len(relevant_set)

    recall = num_relevant_retrieved / total_relevant if total_relevant > 0 else 0.0
    return recall


# 示例
retrieved = ["doc_A", "doc_B", "doc_C", "doc_D", "doc_E"]
relevant = {"doc_A", "doc_B", "doc_F"}

print(f"Recall@3: {recall_at_k(retrieved, relevant, k=3):.2f}")  # 2/3 = 0.67
print(f"Recall@5: {recall_at_k(retrieved, relevant, k=5):.2f}")  # 2/3 = 0.67
# doc_A 和 doc_B 在 top-3 内，doc_F 不在 top-5 内
```

⚠️ **注意**：Recall@K 依赖人工标注的相关文档集。标注质量直接影响评估可信度。建议至少标注 100-200 个 query 的 ground truth。

**生产环境目标**：

| 指标 | 目标值 | 说明 |
|------|--------|------|
| Recall@5 | > 0.80 | Top-5 包含 80% 的相关文档 |
| Recall@20 | > 0.85 | ANN 粗排阶段的召回率 |
| Recall@3 | > 0.70 | Reranker 精排后的召回率 |

### 7.3 MRR 与 Hit Rate

**MRR（Mean Reciprocal Rank，平均倒数排名）**：

MRR 回答的问题：**"第一个相关文档排在第几位？"** 特别适合只有一个正确答案的场景。

$$MRR = \frac{1}{|Q|} \sum_{q \in Q} \frac{1}{rank_q}$$

```python
def mean_reciprocal_rank(queries_results, relevant_docs_per_query):
    """
    MRR 计算示例。

    Args:
        queries_results: list[list[str]]，每个 query 的召回文档列表（按排名）
        relevant_docs_per_query: list[set]，每个 query 的相关文档集合
    """
    reciprocal_ranks = []

    for retrieved, relevant in zip(queries_results, relevant_docs_per_query):
        for rank, doc in enumerate(retrieved, start=1):
            if doc in relevant:
                reciprocal_ranks.append(1.0 / rank)
                break
        else:
            reciprocal_ranks.append(0.0)  # 没有找到相关文档

    mrr = sum(reciprocal_ranks) / len(reciprocal_ranks)
    return mrr


# 示例：3个查询
queries_results = [
    ["doc_A", "doc_B", "doc_C"],  # 相关文档 doc_A 在第1位
    ["doc_D", "doc_A", "doc_E"],  # 相关文档 doc_A 在第2位
    ["doc_F", "doc_G", "doc_H"],  # 没有相关文档
]

relevant = [
    {"doc_A"},
    {"doc_A"},
    {"doc_X"},
]

mrr = mean_reciprocal_rank(queries_results, relevant)
print(f"MRR: {mrr:.4f}")  # (1 + 0.5 + 0) / 3 = 0.50
```

**Hit Rate@K（命中率）**：

Hit Rate@K 回答的问题：**"Top-K 中至少有一个相关文档的比例是多少？"**

```python
def hit_rate_at_k(retrieved_docs, relevant_docs, k):
    """Hit Rate@K: Top-K 中是否命中至少一个相关文档"""
    retrieved_k = set(retrieved_docs[:k])
    return 1.0 if retrieved_k & relevant_docs else 0.0


# 示例
print(f"Hit@1: {hit_rate_at_k(retrieved, relevant, k=1)}")   # 0 (doc_A 不在 top-1)
print(f"Hit@2: {hit_rate_at_k(retrieved, relevant, k=2)}")   # 1 (doc_A 在 top-2)
print(f"Hit@5: {hit_rate_at_k(retrieved, relevant, k=5)}")   # 1 (doc_A 在 top-5)
```

**三个指标的区别与联系**：

| 指标 | 关注点 | 适用场景 |
|------|--------|---------|
| Recall@K | 所有相关文档被召回的比例 | 多答案场景（知识库） |
| MRR | 第一个正确答案的排名 | 单答案场景（FAQ） |
| Hit Rate@K | 是否命中至少一个正确答案 | 实际业务（用户只关心有没有） |

**分析师采纳率（最重要的业务指标）**：

```python
def track_analyst_adoption_rate(rag_responses):
    """
    追踪分析师对 RAG 回答的采纳率。
    这才是 RAG 系统在 SOC 场景的真正 KPI。

    modified = sum(1 for r in rag_responses if r["analyst_action"] == "modified")
    rejected = sum(1 for r in rag_responses if r["analyst_action"] == "rejected")

    print(f"总回答数: {total}")
    print(f"直接采纳: {adopted} ({adopted/total*100:.1f}%)")
    print(f"修改后采纳: {modified} ({modified/total*100:.1f}%)")
    print(f"拒绝: {rejected} ({rejected/total*100:.1f}%)")
    print(f"采纳率（直接+修改）: {(adopted+modified)/total*100:.1f}%")


# 模拟数据
track_analyst_adoption_rate([...] * 1000)
```

```
直接采纳: 623 (62.3%)
修改后采纳: 187 (18.7%)
```

**SOC 场景 Bad Case 分析**：

| Bad Case 类型 | 表现 | 根本原因 | 解决方案 |
|-------------|------|---------|---------|
| CVE 编号无法召回 | 精确的 CVE 查询返回不相关结果 | BM25 关键词匹配失效 | 增加 sparse vector / 提高 BM25 权重 |
| 跨文档关系丢失 | 处置方案不完整，缺少关键步骤 | chunk 边界切断语义链 | 层级分块 + 增加 overlap |
| 新文档无法召回 | 新加入的 SOP 检索不到 | 向量空间漂移 | 定期全量重建索引 |
| 幻觉生成 | 答案看起来合理但包含错误步骤 | 上下文信息不足 | 扩大召回窗口 / 换 Embedding 模型 |

## 八、知识库更新机制

### 8.1 增量更新

增量更新是新文档加入知识库时最常用的方式，核心是** upsert（更新或插入）**。

```python
from qdrant_client.models import PointStruct, Filter, FieldCondition, MatchValue
from datetime import datetime

def upsert_document(collection_name, doc_id, content, metadata):
    """
    增量更新：单文档变更时，生成新向量并写入数据库。
    如果 doc_id 已存在则覆盖，不存在则插入。
    """
    # 1. 生成向量
    vec = model.encode(content, normalize_embeddings=True).tolist()

    # 2. 构造数据点
    point = PointStruct(
        id=doc_id,
        vector=vec,
        payload={
            "content": content,
            "updated_at": datetime.now().isoformat(),
            **metadata,
        }
    )

    # 3. 写入（upsert = update or insert）
    client.upsert(collection_name=collection_name, points=[point])
    print(f"Upsert 完成: doc_id={doc_id}")


# 使用示例：SOP 文档更新
upsert_document(
    collection_name="soc_alerts",
    doc_id="sop_001",
    content="SolarWinds Orion 告警处置 SOP（修订版）...",
    metadata={
        "doc_type": "sop",
        "version": "v2.1",
        "title": "SolarWinds 告警处置 SOP",
    }
)
```

⚠️ **注意**：Qdrant 的 upsert 是**异步合并**的。新写入的 point 立即可查询（因为存在新的 segment），但合并到主索引需要等后台 segment 合并（通常几秒到几分钟）。如果需要强一致性，需要手动触发优化：

```python
# 强制合并所有 segment（生产环境慎用，会阻塞）

    collection_name="soc_alerts",
    optimizer_config={
        "indexing_threshold": 0,  # 设为0立即触发索引构建
```

### 8.2 零停机全量重建

当需要重建整个索引（比如更换 Embedding 模型）时，不能直接重建（会导致服务中断）。正确的做法是 **alias 切换**。

```python
def rebuild_index_zero_downtime(collection_name, new_model, all_documents):
    """
    零停机全量重建索引：

    1. 创建新的 collection（用新模型）
    2. 全量数据写入新 collection
    3. 等待新索引构建完成
    4. 原子切换 alias（用户无感知）

    这样用户在整个过程中感受不到任何停机。
    """
    import time

    # Step 1: 创建新 collection
    new_collection = f"{collection_name}_v2"
    client.recreate_collection(
        collection_name=new_collection,
        vectors_config=VectorParams(size=1024, distance=Distance.COSINE),
        hnsw_config=HnswConfigDiff(m=16, ef_construct=256),
    )

    # Step 2: 全量数据写入（分批）
    BATCH_SIZE = 500
    for i in range(0, len(all_documents), BATCH_SIZE):
        batch = all_documents[i:i+BATCH_SIZE]
        points = [
            PointStruct(
                id=doc["id"],
                vector=new_model.encode(doc["content"], normalize_embeddings=True).tolist(),
                payload=doc["metadata"],
            )
            for doc in batch
        ]
        client.upsert(collection_name=new_collection, points=points)
        print(f"进度: {min(i+BATCH_SIZE, len(all_documents))}/{len(all_documents)}")

    # Step 3: 等待索引构建完成
    # Qdrant 会自动在后台构建 HNSW 索引
    # 可以轮询 collection 状态
    while True:
        collection_info = client.get_collection(collection_name=new_collection)
        if collection_info.indexed:
            print("新索引构建完成！")
            break
        print("等待索引构建...")
        time.sleep(5)

    # Step 4: 原子切换 alias
    # 旧 alias 指向旧 collection，新 alias 指向新 collection
    # 这是原子操作，用户查询无感知
    alias_name = f"{collection_name}_alias"
    client.update_collection_aliases(
        actions=[
            {
                "delete_alias": {"alias": alias_name},
                "create_alias": {
                    "alias": alias_name,
                    "collection_name": new_collection,
                }
            }
        ]
    )
    print(f"Alias '{alias_name}' 已切换到 '{new_collection}'，零停机完成！")

    # 可选：删除旧 collection 释放空间
    # client.delete_collection(collection_name=collection_name)
```

### 8.3 向量空间漂移问题

**向量空间漂移（Vector Space Drift）** 是指新加入文档的向量分布与旧文档存在偏差，导致新旧文档之间的相似度计算不准确。

**为什么会漂移？**

1. Embedding 模型更新（模型版本变了，向量空间发生变化）
2. 文档类型分布变化（新文档的类型和旧文档差异大）
3. Chunking 策略不一致（新文档的分块方式变了）

**诊断方法**：

```python
def diagnose_drift(collection_name, sample_size=100):
    """
    诊断向量空间漂移：对比新旧文档的向量分布。
    如果漂移严重，旧文档的召回率会显著下降。
    """
    # 随机采样一些旧文档
    old_docs = client.scroll(
        collection_name=collection_name,
        scroll_filter=Filter(
            must=[FieldCondition(key="updated_at",
                                 range={"lt": "2024-01-01T00:00:00Z"})
            ]
        ),
        limit=sample_size,
        with_vectors=True,
    )[0]

    # 随机采样一些新文档
    new_docs = client.scroll(
        collection_name=collection_name,
        scroll_filter=Filter(
            must=[FieldCondition(key="updated_at",
                                 range={"gte": "2024-01-01T00:00:00Z"})
            ]
        ),
        limit=sample_size,
        with_vectors=True,
    )[0]

    # 计算新旧文档之间的平均相似度（应该低）
    # 计算新文档与自身的相似度（应该高）
    # 如果两者接近，说明存在漂移

    import numpy as np
    old_vecs = np.array([d.vector for d in old_docs])
    new_vecs = np.array([d.vector for d in new_docs])

    cross_sim = np.mean(np.dot(old_vecs, new_vecs.T))  # 跨分布相似度
    within_sim = np.mean(np.dot(new_vecs, new_vecs.T))  # 新文档内部相似度

    print(f"新旧文档跨分布相似度: {cross_sim:.4f}")
    print(f"新文档内部相似度: {within_sim:.4f}")

    if cross_sim / within_sim > 0.8:
        print("✓ 向量空间正常，无明显漂移")
    else:
        print("⚠️ 警告：检测到向量空间漂移，建议全量重建索引")
```

**解决方案**：

1. **定期全量重建**（推荐）：每月一次全量重索引，保证一致性
2. **内容哈希判断**：用 `hashlib.md5(content)` 判断内容是否变化，未变化跳过
3. **版本标记**：每个文档带 `embedding_version` 字段，版本升级时触发重建

```python
from pathlib import Path

def load_soc_documents(docs_dir):
    """
    加载 SOC 知识库文档。

    docs/
      sop/         # SOP 文档

    for category in ["sop", "cases", "cve", "config"]:
        category_dir = Path(docs_dir) / category
        if not category_dir.exists():

        for file in category_dir.rglob("*.md"):
            with open(file, "r", encoding="utf-8") as f:
                content = f.read()

            # 解析文档元信息（假设文件头有 YAML front matter）
            metadata = parse_frontmatter(content)
            content = strip_frontmatter(content)

            # 分块
            chunks = chunk_soc_document(category, content)

            # 为每个 chunk 添加元信息
            for i, chunk in enumerate(chunks):
                documents.append({
                    "id": f"{category}_{file.stem}_{i}",
                    "content": chunk["content"],
                    "metadata": {
                        "category": category,
                        "title": metadata.get("title", file.stem),
                        "source_file": str(file),
                        "chunk_index": i,
                        "content_hash": hashlib.md5(chunk["content"].encode()).hexdigest(),
                        "updated_at": datetime.now().isoformat(),
                    }
                })

    print(f"加载完成：共 {len(documents)} 个 chunks，{len(set(d['metadata']['source_file'] for d in documents))} 个文档")
    return documents


def parse_frontmatter(content):
    """解析 YAML front matter"""
    import re
    match = re.match(r'^---\n(.*?)\n---\n', content, re.DOTALL)
    if match:
        # 简单解析，实际可用 PyYAML
        lines = match.group(1).split('\n')
        metadata = {}
        for line in lines:
            if ':' in line:
                key, val = line.split(':', 1)
                metadata[key.strip()] = val.strip()
        return metadata
    return {}


def strip_frontmatter(content):
    """移除 YAML front matter"""
    import re
    return re.sub(r'^---\n.*?\n---\n', '', content, flags=re.DOTALL)
```

```python
            documents: list[dict]，每个 dict 包含 id, content, metadata
        """

                id=i,

                    "content_hash": hashlib.md5(doc["content"].encode()).hexdigest(),
                    "updated_at": datetime.now().isoformat(),

                }

            for i, (doc, vec) in enumerate(zip(documents, vectors))

        self.client.upsert(collection_name=self.collection_name, points=points)
        print(f"导入完成：{len(points)} 个 chunks 已写入")

    def retrieve(self, query, top_k=20, doc_types=None):
        """
        检索相关文档（两阶段：ANN + Reranker）。

        Args:
            query: 用户问题
            top_k: ANN 粗排召回数量
            doc_types: 过滤的文档类型列表

        Returns:
            list[dict]: 重排序后的 top-3 文档
        """
        # Step 1: ANN 粗排
        query_vec = self.embedding_model.encode(
            query,
            normalize_embeddings=True,
        )

        # 构建过滤条件
        search_filter = None
        if doc_types:
            search_filter = Filter(
                must=[FieldCondition(
                    key="category",
                    match=MatchAny(any=doc_types)
                )]
            )

        # ANN 搜索
        ann_results = self.client.search(
            collection_name=self.collection_name,
            query_vector=query_vec.tolist(),
            limit=top_k,
            query_filter=search_filter,
            with_payload=True,
        )

            return []

        reranker = self._get_reranker()
        candidates = [r.payload["content"] for r in ann_results]

        pairs = [[query, doc] for doc in candidates]

            key=lambda x: x[1],
            reverse=True,

                "content": r.payload["content"],
```

| Embedding 模型 | BAAI/bge-large-zh-v1.5 | 中文 SOC 首选 |

| Vector Distance | COSINE | 余弦相似度 |
| HNSW m | 16 | 通用场景 |
| HNSW ef_construct | 200 | 构建精度 |
| HNSW hnsw_ef | 128 | 查询精度 |
| Chunk size | 512 tokens | SOC 场景最佳 |
| Overlap | 64 tokens | 上下文保留 |
| ANN top_k | 20 | Reranker 候选数 |
| Reranker top_k | 3 | 最终返回数 |
| RRF k | 60 | 融合平滑因子 |

| MRR | > 0.65 | 首位相关排名 |
| Hit Rate@3 | > 0.75 | 命中率 |
| RAGAS Faithfulness | > 0.80 | 答案忠实度 |

### C. 常见问题排查

| 问题 | 可能原因 | 解决方案 |
|------|---------|---------|
| 检索不到相关文档 | Chunk 策略错误 | 换成层级/语义分块 |
| 幻觉严重 | 召回太少/上下文不足 | 增大 top_k / 扩大 chunk_size |
| 延迟高（P99 > 200ms） | Reranker 未用 GPU | 启用 `use_fp16=True` |
| 新文档无法召回 | 向量空间漂移 | 定期全量重建索引 |
| 精确 CVE 查不到 | BM25 未启用 | 启用混合检索 |
