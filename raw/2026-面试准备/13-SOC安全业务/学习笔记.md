## 1. SOC 团队架构

### 1.1 为什么需要分层架构？

SOC（Security Operations Center）是企业安全的"大脑"，负责 7x24 小时监控、分析、响应安全事件。如果所有工作都交给一个人，会出现两个严重问题：

- **分析师疲劳**：一个人面对每天上万条告警，迟早会麻木、漏报
- **技能错配**：初级分析师做高难度 APT 分析，既浪费人才又效果差

因此，SOC 采用**分层架构**，让不同技能等级的人处理不同复杂度的告警，实现效率最大化。

### 1.2 分层架构详解

```
┌─────────────────────────────────────────────────────────────┐
│                     SOC 整体架构                            │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│   L1 分析师 ──→ L2 分析师 ──→ L3 分析师（安全专家）          │
│      ↓              ↓              ↓                        │
│  日常监控       深度调查        APT 分析                    │
│  告警分诊       事件定性        威胁狩猎                    │
│  基础响应       溯源分析        红蓝对抗                    │
│                                                             │
│   安全运营工程师（工具维护、规则编写、平台开发）              │
│   安全运营经理（SLA 管理、升级协调、汇报）                    │
│   事件响应团队（重大事件专项处置）                            │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

#### L1 分析师（Tier 1 / T1）

**职责**：SOC 的"前台接待"，每天面对大量告警的第一道关卡。

- 接收 SIEM 推送的告警，做初步判断（真/假/不确定）
- 常见任务：确认钓鱼邮件、封禁已知恶意 IP、创建工单
- SLA 要求：平均处理时间（MTT）< 15 分钟
- 需要的技能：熟悉常见攻击类型（暴力破解、恶意软件特征）、会操作 SIEM 界面

**典型场景**：收到一条告警"某员工主机触发了 malware alert"，L1 需要快速判断：
1. 该主机最近有没有用户投诉异常？
2. 告警的置信度是多少？
3. 是否需要升级给 L2？

#### L2 分析师（Tier 2 / T2）

**职责**：L1 的"后台支援"，处理 L1 无法判断、需要深度分析的告警。

- 深入调查事件，分析攻击链（Attack Chain）
- 关联多源日志（网络流量 + 主机日志 + 应用日志）
- 编写/优化检测规则，反馈给 L1
- SLA 要求：平均处理时间 < 2 小时

**典型场景**：L1 升级了一个"内网横向移动"告警，L2 需要：
1. 分析源主机的进程链（excel.exe → cmd.exe → powershell.exe 异常链）
2. 关联同一源 IP 对内网其他主机的连接记录
3. 判断是否为真实的横向移动攻击

#### L3 分析师 / 安全专家（Tier 3 / T3）

**职责**：SOC 的"专家会诊"，处理最复杂的安全事件。

- 高级威胁分析：APT 攻击识别、0-day 分析
- 威胁狩猎（Threat Hunting）：主动出击，发现规则盲区
- 红蓝对抗：模拟攻击测试 SOC 检测能力
- 检测规则设计：基于 ATT&CK 框架的全覆盖检测
- SLA：按需介入，不设硬性 SLA

**⚠️ 注意**：L3 不是接替 L1/L2 工作，而是做 L1/L2 做不了的事。很多公司 L3 只配置 2-5 人，重点是"专家咨询"而非日常告警处理。

### 1.3 配套角色

| 角色 | 主要工作 | 与 SOC 的关系 |
|------|---------|-------------|
| 安全运营工程师 | 维护 SIEM/EDR/NDR 工具、编写自动化脚本 | SOC 的"工具保障" |
| 安全运营经理 | 制定 SLA、管理告警升级、协调跨团队响应 | SOC 的"管理层" |
| 事件响应团队 | 重大事件（如勒索软件）专项处置 | SOC 的"突击队" |

### 1.4 MTTD 与 MTTR：衡量 SOC 效率的核心指标

这是面试高频问题，必须理解背后的含义。

- **MTTD（Mean Time to Detect，平均检测时间）**：攻击发生到被检测到的平均时间。MTTD 越短，说明检测能力越强。
- **MTTR（Mean Time to Respond，平均响应时间）**：检测到响应处置完成的平均时间。MTTR 越短，说明响应能力越强。

```
MTTD: 攻击发生 ───[攻击持续]───→ 被检测到 ──→ 响应完成
         ↑                              ↑
     时间起点                        MTTD                    MTTR
```

**业界基准**（参考 2024 Verizon DBIR）：
- 传统企业 MTTD：约 200 天（很多攻击潜伏数月才被发现）
- 领先水平：MTTD < 24 小时，MTTR < 30 天
- APT 攻击：平均潜伏 200+ 天才能被发现

**为什么 MTTD 这么重要？** 攻击在系统内停留的时间越长，造成的损失越大。IBM 研究表明，MTTD 每缩短一天，数据泄露成本可降低约 $100 万。

## 2. 告警生命周期

### 2.1 什么是告警生命周期？为什么要理解它？

告警生命周期是**一条告警从诞生到消亡的完整旅程**。理解这个生命周期，是设计 SOC 平台数据模型的基础——每一步都有对应的系统组件和数据存储需求。

### 2.2 六阶段详解

```
生成 ──→ 归类 ──→ 定级优先级 ──→ 调查 ──→ 响应 ──→ 关闭
 (1)     (2)        (3)          (4)      (5)      (6)
```

#### 阶段 1：生成（Generation）

**谁来生成？** SIEM（Security Information and Event Management）是告警的主要生产者。

SIEM 的工作原理（以 Elasticsearch + Winlogbeat 为例）：

```python
# 伪代码：SIEM 检测规则的核心逻辑
# 当检测到 "同一 IP 在 60 秒内登录失败超过 5 次" 时，触发告警

def evaluate_rule(logs: list[dict], rule: dict) -> list[Alert]:
    alerts = []
    for log in logs:
        # 按源 IP 分组，统计失败登录次数
        ip = log["source_ip"]
        if log["event"] == "login_failure":
            failure_count[ip] += 1

            # 超过阈值，触发告警
            if failure_count[ip] >= rule["threshold"]:
                alerts.append(Alert(
                    type="bruteforce",
                    source_ip=ip,
                    count=failure_count[ip],
                    severity=rule["severity"],
                    rule_id=rule["id"]
                ))
    return alerts
```

**⚠️ 注意**：告警生成依赖规则，规则质量决定告警质量。如果规则写得太宽（如"所有 HTTP 请求都触发告警"），就会产生大量无意义的告警。

#### 阶段 2：归类（Classification / Triage）

**归类做什么？** 把告警分配到正确的类别（攻击类型、ATT&CK 战术阶段），决定后续处理路径。

现代 SOC 平台用 AI 做自动归类：

```python
# 伪代码：LLM 辅助告警归类
# 输入：告警的原始日志上下文
# 输出：告警类型 + ATT&CK 战术 + 置信度

def classify_alert(alert: Alert, llm_client) -> ClassificationResult:
    prompt = f"""
    告警上下文：
    - 告警类型：{alert.type}
    - 源 IP：{alert.source_ip}
    - 目标资产：{alert.target_hostname}
    - 事件描述：{alert.description}
    - 相关进程：{alert.process_chain}

    请判断：
    1. 这个告警最可能的 ATT&CK 战术阶段是什么？（Initial Access / Execution / Persistence / ...）
    2. 置信度是多少？（0.0-1.0）
    3. 建议升级还是关闭？
    """

    response = llm_client.chat(prompt)
    return parse_classification(response)

# 示例输出：
# {
#   "tactic": "Lateral Movement",
#   "confidence": 0.78,
#   "recommendation": "escalate_to_L2"
# }
```

**为什么 AI 归类有价值？** 传统 SIEM 靠规则匹配，一个告警只有一个固定标签。AI 可以结合上下文理解"伪装成正常行为的攻击"，准确率更高。

#### 阶段 3：优先级（Prioritization）

**优先级怎么定？** 通常基于三个维度加权计算：

```
优先级得分 = 资产重要性权重 × 告警严重程度权重 × 威胁情报命中权重
```

```python
# 伪代码：告警优先级评分
def calculate_priority(alert: Alert, asset_db: dict, ti_db: dict) -> int:
    # 维度1：目标资产重要性（1-10 分）
    asset_score = asset_db.get(alert.target_asset_id, {}).get("criticality", 5)

    # 维度2：告警类型严重程度
    severity_scores = {
        "malware": 10,
        "lateral_movement": 9,
        "bruteforce": 7,
        "port_scan": 5,
        "unauthorized_access": 8,
    }
    severity_score = severity_scores.get(alert.type, 5)

    # 维度3：威胁情报命中（高置信度 IOC 命中 = 额外加分）
    ti_score = 0
    if ti_db.is_matched(alert.source_ip):  # IP 命中黑名单
        ti_score = 10
    elif ti_db.is_matched(alert.domain):    # 域名命中黑名单
        ti_score = 8

    # 综合评分（归一化到 P0-P3）
    total = asset_score * 0.3 + severity_score * 0.4 + ti_score * 0.3

    if total >= 8.5:
        return 0   # P0 紧急
    elif total >= 7.0:
        return 1   # P1 高
    elif total >= 5.0:
        return 2   # P2 中
    else:
        return 3   # P3 低
```

#### 阶段 4：调查（Investigation）

分析师对告警做深度分析，调阅关联上下文（历史告警、用户行为、资产信息）。

#### 阶段 5：响应（Response）

根据调查结论，执行处置动作：封锁 IP、隔离主机、通知用户、创建工单等。

#### 阶段 6：关闭（Closure）

告警处置完成，标记为"已关闭"，记录关闭原因（真告警/误报/降级/重复）。

## 3. 告警定级与 SLA

### 3.1 告警定级标准

| 级别 | 名称 | 含义 | 典型场景 | SLA 响应时间 |
|------|------|------|---------|------------|
| **P0** | 紧急（Critical） | 正在发生或已经造成重大损失的攻击 | 勒索软件运行中、核心数据库被拖库 | < 15 分钟 |
| **P1** | 高（High） | 明确的攻击行为，但尚未造成实质性破坏 | 横向移动进行中、钓鱼邮件内有木马 | < 1 小时 |
| **P2** | 中（Medium） | 可疑行为，需要关注但不是立即响应 | 暴力破解未成功、异常登录（非办公时间） | < 4 小时 |
| **P3** | 低（Low） | 信息性告警，基本无实际威胁 | 端口扫描（未突破边界）、低置信度告警 | < 24 小时 |

**⚠️ 注意**：定级标准不是一成不变的，同一种告警类型在不同资产上可能有不同级别。例如"异常进程启动"，发生在办公电脑上可能是 P2，发生在域控制器上就是 P1。

### 3.2 SLA 的意义

SLA（Service Level Agreement）是 SOC 服务的"承诺书"，规定了什么级别的事件应该多久响应。超时会怎样？

- **内部 SLA 超时**：触发升级机制（自动升级给 L2/L3）
- **客户 SLA 超时**：合同违约，可能面临罚款

```python
# 伪代码：SLA 超时自动升级
def check_sla_timeout(alert: Alert, config: SLAConfig) -> None:
    elapsed = time.now() - alert.created_at
    sla_deadline = config.get_sla(alert.priority)

    if elapsed > sla_deadline * 0.8 and not alert.escalated_once:
        # SLA 消耗 80%，自动升级
        escalate_alert(alert, target_tier=alert.tier + 1)
        alert.escalated_once = True
        notify_manager(f"告警 {alert.id} SLA 即将超时，已自动升级")

    if elapsed > sla_deadline:
        # SLA 超时
        alert.status = "overdue"
        notify_manager(f"告警 {alert.id} SLA 已超时！当前处理人：{alert.assignee}")
```

## 4. 威胁情报（IOC/STIX/TAXII）

### 4.1 为什么需要威胁情报？

光靠 SOC 平台自己的规则，检测能力是有限的。威胁情报的作用是**把外部的安全知识引入到本地检测中**。

举个例子：SOC 平台不知道某个 IP 是恶意 C2 服务器，但如果情报供应商告诉你，这个 IP 属于已知的僵尸网络，你就可以直接提升告警级别。

### 4.2 IOC 类型详解

IOC（Indicators of Compromise，妥协指标）是威胁情报的核心载体。

| IOC 类型 | 说明 | 示例 | 检测方法 |
|---------|------|------|---------|
| **IP 地址** | C2 服务器、僵尸网络节点 | `185.220.101.34` | DNS 日志、流量日志 |
| **域名** | 恶意域名、钓鱼 URL | `malware-c2.xyz` | DNS 查询日志 |
| **文件哈希** | 恶意软件哈希值 | `MD5: d41d8cd98f00b204e9800998ecf8427e` | EDR 文件事件 |
| **URL** | C2 通信路径、钓鱼链接 | `hxxp://evil.com/payload.exe` | Web 代理日志 |
| **CVEs** | 漏洞编号 | `CVE-2024-1701` | 漏洞扫描结果 |

**⚠️ 常见误区**：很多人以为 IOC 只是 IP 地址。实际上，文件哈希和 CVE 同样重要——一个 0-day 漏洞相关的 CVE 编号，可以帮助你在漏洞披露后迅速排查内部哪些资产受影响。

### 4.3 IOC 在告警中的实际应用

```python
# 伪代码：IOC 匹配引擎
# 告警触发时，自动查询 IOC 数据库，提升告警优先级

class TIEnricher:
    def __init__(self, ti_database: TIDatabase):
        self.ti_db = ti_database

    def enrich_alert(self, alert: Alert) -> Alert:
        """将 IOC 命中结果附加到告警上下文中"""
        enrichment = {
            "ip_hits": [],
            "domain_hits": [],
            "hash_hits": [],
            "cve_hits": [],
        }

        # 匹配 IP
        if hasattr(alert, "source_ip"):
            ip_result = self.ti_db.lookup_ip(alert.source_ip)
            if ip_result:
                enrichment["ip_hits"].append({
                    "ip": alert.source_ip,
                    "confidence": ip_result.confidence,
                    "threat_type": ip_result.threat_type,  # "C2", "Botnet", "Scanner"
                    "source": ip_result.feed_name,
                })

        # 匹配域名
        if hasattr(alert, "domain"):
            domain_result = self.ti_db.lookup_domain(alert.domain)
            if domain_result:
                enrichment["domain_hits"].append({
                    "domain": alert.domain,
                    "confidence": domain_result.confidence,
                    "threat_type": domain_result.threat_type,
                })

        alert.ti_enrichment = enrichment
        return alert

# 示例：告警 enrichment 结果

#             "source": "AlienVault OTX"
#         }
#     ],
#     "domain_hits": [],
```

### 4.4 STIX 和 TAXII：情报交换的标准语言

**STIX（Structured Threat Information Expression）**：描述威胁情报的结构化语言，本质是一个 JSON Schema。想象成"威胁情报的 JSON 格式标准"。

```json
// STIX 2.1 示例：一个恶意 IP 的完整描述
{
  "type": "indicator",
  "spec_version": "2.1",
  "id": "indicator--fc49e0d3-0b16-4d3b-b3f2-e7392b5a4f9e",
  "created": "2024-01-15T10:30:00Z",
  "modified": "2024-01-15T10:30:00Z",
  "name": "Known C2 Server - Emotet Botnet",
  "description": "IP associated with Emotet C2 infrastructure",
  "indicator_types": ["malware"],
  "pattern": "[ipv4-addr:value = '185.220.101.34']",
  "pattern_type": "stix",
  "valid_from": "2024-01-15T10:30:00Z",
  "valid_until": "2024-02-15T10:30:00Z",  // IOC 过期时间
  "confidence": 85,
  "labels": ["malware", "botnet"]
}
```

**TAXII（Trusted Automated eXchange of Intelligence Information）**：传输协议，负责在情报供应商和客户之间安全传输 STIX 数据。本质是一个**带认证的 HTTPS REST API**。

```python
# 伪代码：TAXII 客户端，从情报供应商拉取 IOC
import requests
from stix2 import Indicator

class TAXIIClient:
    def __init__(self, taxii_server: str, username: str, api_key: str):
        self.base_url = taxii_server
        self.auth = (username, api_key)

    def fetch_indicators(self, feed_name: str, since: str = None) -> list[Indicator]:
        """从 TAXII 服务器拉取最新 IOC"""
        headers = {"Accept": "application/taxii+json;version=2.1"}
        params = {}
        if since:
            params["added_after"] = since

        # 获取指定情报源的所有 bundle
        response = requests.get(
            f"{self.base_url}/api1/collections/{feed_name}/objects",
            headers=headers,
            auth=self.auth,
            params=params,
            verify=True  # HTTPS 证书校验
        )
        response.raise_for_status()
        return response.json()["objects"]

    def sync_to_local(self, feed_name: str) -> int:
        """同步情报到本地数据库"""
        indicators = self.fetch_indicators(feed_name, since=self.last_sync_time)
        count = 0
        for obj in indicators:
            if obj["type"] == "indicator":
                # 写入本地 IOC 数据库
                self.ioc_store.upsert(self._stix_to_ioc(obj))
                count += 1
        self.last_sync_time = datetime.utcnow().isoformat()
        return count
```

### 4.5 IOC 生命周期管理

**⚠️ 重要**：IOC 有保质期，过期不清理会导致两个问题：
1. **假阳性增加**：过期的 C2 IP 可能已被回收，现在分配给了正常用户
2. **性能浪费**：无效 IOC 占用匹配引擎的计算资源

```python
# 伪代码：IOC 自动过期清理
class IOCLifecycleManager:
    def cleanup_expired(self) -> int:
        """删除过期的 IOC，返回删除数量"""
        now = datetime.utcnow()

        # 软删除：标记为 inactive（保留审计记录）
        expired_iocs = self.session.query(IOC).filter(
            IOC.valid_until < now,
            IOC.status == "active"
        ).all()

        for ioc in expired_iocs:
            ioc.status = "inactive"
            ioc.deactivated_at = now
            # 写入变更日志（审计用）
            self.audit_log.record(
                action="ioc_deactivated",
                ioc_id=ioc.id,
                reason="ttl_expired",
                timestamp=now
            )

        self.session.commit()
        return len(expired_iocs)

    def get_ttl_by_confidence(self, confidence: int) -> int:
        """根据情报置信度设置不同的 TTL"""
        if confidence >= 90:
            return 90  # 高置信度 IOC，保留 90 天
        elif confidence >= 70:
            return 30  # 中置信度，保留 30 天
        else:
            return 7   # 低置信度，只保留 7 天
```

## 5. 告警降噪

### 5.1 告警降噪的本质问题

告警降噪是 SOC 平台最核心的工程难题。问题的根源在于：

**规则写得太精准 → 漏报率高（攻击绕过规则）**
**规则写得宽松 → 误报率高（大量假告警）**

降噪的目标是**在不增加漏报的前提下，显著降低误报率**，让分析师只处理真正需要关注的告警。

### 5.2 规则调优：数据驱动的方法

规则调优不是靠"感觉"收紧条件，而是靠数据说话：

```python
# 伪代码：基于历史数据评估规则质量
def evaluate_rule(rule_id: str, lookback_days: int = 30) -> RuleMetrics:
    alerts = query_alerts_by_rule(rule_id, last_n_days=lookback_days)
    total = len(alerts)

    # 统计处置结果分布
    dispositions = count_dispositions(alerts)
    # dispositions = {"true_positive": 12, "false_positive": 488, "escalated": 5}

    true_positive_rate = dispositions["true_positive"] / total
    false_positive_rate = dispositions["false_positive"] / total

    return RuleMetrics(
        rule_id=rule_id,
        total_triggers=total,
        true_positive_rate=true_positive_rate,
        false_positive_rate=false_positive_rate,
        avg_dwell_time=avg_dwell_time(alerts),  # 分析师花多少时间
        recommendation=determine_action(true_positive_rate, false_positive_rate)
    )

# 规则优化建议
def determine_action(tpr: float, fpr: float) -> str:
    if tpr > 0.5 and fpr < 0.1:
        return "KEEP"           # 好规则，继续用
    elif fpr > 0.7:
        return "TUNE"          # 误报太多，收紧条件
    elif tpr < 0.1:
        return "DISABLE"       # 几乎没有真告警，关闭
    else:
        return "MONITOR"       # 继续观察
```

**规则调优的具体操作**：
- **增加时间窗口限制**：原来"任何时间检测到暴力破解都告警" → 改为"非工作时间内检测到暴力破解"
- **增加资产过滤**：原来"所有主机" → 改为"生产服务器（排除测试机）"
- **增加 IP 白名单**：已知正常的扫描 IP（如内部安全扫描器）加入白名单

### 5.3 机器学习分类：超越规则的能力

规则能检测已知攻击模式，但无法检测"未知但异常"的行为。ML 模型通过学习历史告警的特征分布，可以识别规则覆盖不到的异常。

```python
# 伪代码：ML 告警分类模型
import numpy as np
from sklearn.ensemble import RandomForestClassifier
from sklearn.preprocessing import StandardScaler

class AlertClassifier:
    def __init__(self):
        self.model = RandomForestClassifier(n_estimators=100, random_state=42)
        self.scaler = StandardScaler()
        self.trained = False

    def extract_features(self, alert: Alert, asset_db: dict) -> np.ndarray:
        """
        将告警上下文转化为特征向量
        这是 ML 模型的关键：特征工程决定模型上限
        """
        features = []

        # 特征1：告警类型（one-hot 编码，假设有 20 种类型）
        features.extend(self._onehot_type(alert.type, known_types=20))

        # 特征2：目标资产重要性
        asset_info = asset_db.get(alert.target_asset_id, {})
        features.append(asset_info.get("criticality_score", 5) / 10.0)

        # 特征3：源 IP 是否在威胁情报黑名单
        features.append(1.0 if alert.ti_enrichment["ip_hits"] else 0.0)

        # 特征4：历史同类告警频率（7 天内）
        features.append(self._get_historical_freq(alert) / 100.0)

        # 特征5：告警时间（工作时间是正常行为的高峰）
        hour = alert.timestamp.hour
        features.append(1.0 if 9 <= hour <= 18 else 0.0)

        # 特征6：是否跨区域访问（如员工突然从海外登录）
        features.append(1.0 if alert.geo_location != alert.user_home_country else 0.0)

        return np.array(features)

    def train(self, labeled_alerts: list[Alert]):
        """用已标注的历史告警训练模型"""
        X = np.array([self.extract_features(a) for a in labeled_alerts])
        y = np.array([1 if a.disposition == "true_positive" else 0
                      for a in labeled_alerts])

        X_scaled = self.scaler.fit_transform(X)
        self.model.fit(X_scaled, y)
        self.trained = True

    def predict(self, alert: Alert, asset_db: dict) -> float:
        """预测告警为真阳性的概率"""
        if not self.trained:
            return 0.5  # 未训练时返回默认值

        X = self.scaler.transform([self.extract_features(alert, asset_db)])
        prob = self.model.predict_proba(X)[0][1]  # 返回真阳性概率
        return prob

    def should_suppress(self, alert: Alert, asset_db: dict,
                        confidence_threshold: float = 0.6) -> bool:
        """
        根据置信度决定是否推送告警
        ⚠️ 注意：置信度阈值需要持续调优
        阈值太高 → 漏报增加
        阈值太低 → 降噪效果不明显
        """
        prob = self.predict(alert, asset_db)
        return prob < confidence_threshold
```

### 5.4 Feedback 闭环：让数据持续优化模型

ML 模型最怕的是"训练数据过时"。Feedback 闭环确保模型随时间进化：

```
分析师处置告警（确认/误报/升级）
        ↓
处置结果写入训练数据集
        ↓
每月定期重新训练模型
        ↓
新模型部署到生产环境
        ↓
观测新模型在真实数据上的表现
        ↓
回到第一步（持续迭代）
```

**⚠️ 常见陷阱**：
1. **反馈延迟**：分析师处置结果没有及时回流，导致训练数据过期
2. **标签偏置**：分析师倾向于把复杂告警标注为"真告警"，简单告警标注为"误报"，导致模型学到了"复杂的才是真的"
3. **概念漂移**：攻击手法进化，2022 年的训练数据在 2026 年可能不适用

## 6. 横向移动检测

### 6.1 什么是横向移动？

横向移动（Lateral Movement）是攻击者在内网中**从一台主机跳到另一台主机**的过程。常见场景：

- 攻击者通过钓鱼邮件控制了员工电脑（第一台主机）
- 利用该电脑的凭证（如密码哈希），访问内网文件服务器
- 再利用文件服务器的凭证，访问域控制器
- 最终获取管理员权限

**为什么横向移动是 APT 的核心？** 初始入侵（钓鱼邮件）往往只能拿到普通员工权限，横向移动才是攻击者扩大战果的关键步骤。检测到横向移动，往往意味着这不是普通的恶意软件，而是有组织的攻击。

### 6.2 NDR 路线：网络流量检测

NDR（Network Detection and Response）通过分析网络流量检测横向移动，**不依赖主机日志**。

```python
# 伪代码：NDR 横向移动检测逻辑
# 检测特征：同一源 IP 在短时间内对多个内网主机的 445/3389 端口建立连接

def detect_lateral_movement_ndr(flow_logs: list[FlowLog]) -> list[LateralMovementAlert]:
    alerts = []

    # 按源 IP 分组
    source_ips = group_by_source_ip(flow_logs)

    for src_ip, flows in source_ips.items():
        # 筛选目标端口为 445（SMB）或 3389（RDP）的流量
        lateral_flows = [f for f in flows
                          if f.dest_port in [445, 3389]
                          and f.timestamp > time.now() - timedelta(minutes=30)]

        # 统计唯一目标主机数量
        unique_targets = set(f.dst_ip for f in lateral_flows)

        if len(unique_targets) >= LATERAL_MOVEMENT_THRESHOLD:
            alerts.append(LateralMovementAlert(
                source_ip=src_ip,
                target_ips=list(unique_targets),
                count=len(lateral_flows),
                ports=[f.dest_port for f in lateral_flows],
                time_window="30min",
                severity="high"
            ))

    return alerts

# 输出示例：
# LateralMovementAlert(
#     source_ip="10.0.1.45",       # 源主机 IP
#     target_ips=["10.0.2.10", "10.0.2.15", "10.0.2.20"],  # 被访问的 3 台主机
#     count=12,
#     ports=[445, 445, 445, 3389, ...],  # SMB 和 RDP 端口
#     time_window="30min",
#     severity="high"
# )
```

**NDR 的优势**：
- 不需要在每台主机安装 Agent，覆盖盲区（如 IoT 设备、打印机）
- 攻击者难以察觉自己在被监控（不修改主机行为）

**NDR 的劣势**：
- 加密流量无法检测内容（TLS 加密后只能看元数据）
- 误报率高（合法的共享文件访问也会产生 445 端口流量）

### 6.3 EDR 路线：主机日志检测

EDR（Endpoint Detection and Response）通过采集主机上的详细行为日志检测横向移动。

```python
# 伪代码：EDR 横向移动检测逻辑
# Windows 4688 事件：进程创建事件链分析

def detect_lateral_movement_edr(process_events: list[ProcessEvent]) -> list[Alert]:
    alerts = []

    # 按主机分组
    host_events = group_by_hostname(process_events)

    for hostname, events in host_events.items():
        # 提取 5 分钟内的进程链
        recent = [e for e in events if e.timestamp > time.now() - timedelta(minutes=5)]

        # 检测异常进程链（office/doc/pdf → cmd → powershell → download）
        suspicious_chains = find_process_chains(recent, chain_patterns=[
            ["excel.exe", "cmd.exe", "powershell.exe"],
            ["winword.exe", "rundll32.exe"],
            ["chrome.exe", "powershell.exe", "net.exe"],
        ])

        if suspicious_chains:
            alerts.append(Alert(
                type="suspicious_process_chain",
                hostname=hostname,
                chains=suspicious_chains,
                # 检查是否有远程执行特征
                remote_exec=check_remote_execution(events)
            ))

    # 检测同一账户短时间多主机登录
    # Windows 4624 事件：账户登录事件
    login_events = get_login_events(process_events)  # 过滤出 4624
    account_logins = group_by_account_name(login_events)

    for account, logins in account_logins.items():
        # 按时间窗口分组
        time_windows = sliding_window(logins, window_size=timedelta(minutes=10))
        for window in time_windows:
            unique_hosts = set(l.target_hostname for l in window)
            if len(unique_hosts) >= 3:
                alerts.append(Alert(
                    type="multi_host_login",
                    account=account,
                    hosts=list(unique_hosts),
                    count=len(window),
                    time_window="10min"
                ))

    return alerts
```

**⚠️ EDR 的常见检测挑战**：
1. **进程名伪造**：恶意软件可以改名成 `explorer.exe` 伪装，需要用哈希而非名称检测
2. **时间窗口**：攻击者故意把攻击时间拉长（低慢大攻击），单个时间窗口检测不到
3. **日志丢失**：如果 EDR Agent 被攻击者关闭，事件记录就中断了

### 6.4 NDR + EDR 联动：双路置信度叠加

单一数据源检测的置信度有限，**联动检测**可以显著提升准确性：

```
NDR 发现：源 IP 10.0.1.45 在 30 分钟内访问了 5 台内网主机（445 端口）
        ↓
EDR 确认：主机 10.0.1.45 上 excel.exe → cmd.exe → powershell.exe 进程链
        ↓
综合置信度：NDR(0.6) × EDR(0.8) = 0.48 → 提升告警级别
```

这就是现代 SOC 平台的 XDR（Extended Detection and Response）理念：**跨主机、跨数据源关联分析**。

## 7. APT 检测

### 7.1 APT 的本质特征

APT（Advanced Persistent Threat，高级持续性威胁）不是一种具体的攻击技术，而是一类**攻击策略的总称**。它的三个关键特征：

| 特征 | 含义 | 为什么难检测 |
|------|------|------------|
| **高级（Advanced）** | 使用 0-day 漏洞、定制的恶意工具 | 传统签名检测完全无效 |
| **持续（Persistent）** | 长期潜伏，持续获取情报 | 需要数月乃至数年的数据积累才能发现 |
| **威胁（Threat）** | 有组织的攻击者（国家级黑客/犯罪集团） | 攻击者有资源、有耐心、有明确目标 |

### 7.2 低慢大攻击：APT 的核心手法

"低慢大"是 APT 检测最重要的概念：

```
低：低频（每次行为都不超过检测阈值）
慢：慢速（攻击持续时间长，不引起注意）
大：大规模（目标是多个目标或整个网络）
```

**举例**：传统规则"同一 IP 1 分钟内登录失败 10 次 → 告警"。APT 的对策是"每天失败 9 次，连续 30 天"——每次都不触发规则，但累积效果等同暴力破解。

### 7.3 长周期行为基线

检测低慢大攻击，需要建立**长周期行为基线**，而不是只看单点事件：

```python
# 伪代码：长周期用户行为基线检测
# 核心思路：为每个用户建立"正常行为范围"，偏离则告警

class UserBehaviorBaseline:
    def __init__(self, lookback_days: int = 90):
        self.lookback_days = lookback_days
        self.baseline_window = timedelta(days=90)

    def build_baseline(self, user_id: str, historical_events: list[Event]) -> Baseline:
        """为用户建立 90 天行为基线"""
        # 计算各项指标的统计分布（均值 + 标准差）
        login_times = [e.timestamp.hour for e in historical_events
                       if e.type == "login"]
        login_locations = [e.geo_location for e in historical_events
                           if e.type == "login"]
        accessed_resources = [e.resource for e in historical_events
                               if e.type == "file_access"]

        return Baseline(
            user_id=user_id,
            # 登录时间分布：均值 + 3σ 范围
            login_hour_mean=np.mean(login_times),
            login_hour_std=np.std(login_times),
            # 常用登录地点
            frequent_locations=Counter(login_locations).most_common(5),
            # 常用资源访问
            frequent_resources=set(accessed_resources),
            # 数据量统计
            avg_daily_events=np.mean([
                count_daily_events(historical_events, day)
                for day in date_range(self.lookback_days)
            ])
        )

    def detect_anomaly(self, current_event: Event, baseline: Baseline) -> float:
        """
        检测当前事件是否偏离基线
        返回异常得分（0=完全正常，1=极度异常）
        """
        anomaly_score = 0.0

        # 检测点1：登录时间异常
        if hasattr(current_event, "timestamp"):
            hour = current_event.timestamp.hour
            z_score = abs(hour - baseline.login_hour_mean) / baseline.login_hour_std
            if z_score > 3:
                anomaly_score += 0.4  # 偏离 3σ，异常权重 0.4

        # 检测点2：登录地点异常（用户从未从这个地点登录过）
        if hasattr(current_event, "geo_location"):
            if current_event.geo_location not in baseline.frequent_locations:
                anomaly_score += 0.3

        # 检测点3：访问从未访问过的资源
        if hasattr(current_event, "resource"):
            if current_event.resource not in baseline.frequent_resources:
                anomaly_score += 0.3

        return min(anomaly_score, 1.0)  # 上限 1.0
```

### 7.4 威胁狩猎（Threat Hunting）

威胁狩猎是**主动出击**的检测方式——分析师基于假设，主动搜索 SIEM 中的异常，而不是等告警来了再处理。

**IOC 驱动狩猎**：情报部门发现了一个新的恶意域名，狩猎团队主动搜索"过去 30 天内有哪些主机解析了这个域名"

```python
# 伪代码：威胁狩猎查询
# 场景：情报发现恶意域名 evil-c2.xyz，主动排查内部感染情况

def threat_hunt_malicious_domain(domain: str, days_back: int = 30) -> HuntResult:
    # 查询 DNS 解析日志
    dns_logs = es.search(
        index="dns-logs-*",
        body={
            "query": {
                "bool": {
                    "must": [
                        {"match": {"query_name": domain}},
                        {"range": {"@timestamp": {
                            "gte": f"now-{days_back}d",
                            "lte": "now"
                        }}}
                    ]
                }
            }
        }
    )

    # 聚合：每个源 IP 解析了多少次
    ip_agg =聚合(dns_logs, group_by="source_ip")

    return HuntResult(
        domain=domain,
        affected_ips=[r.key for r in ip_agg.buckets],
        total_resolutions=ip_agg.total_count,
        severity="critical" if len(ip_agg.buckets) > 5 else "high"
    )
```

**⚠️ 威胁狩猎的常见假设**：
- "攻击者一定会尝试建立持久化" → 搜索新创建的启动项/计划任务
- "攻击者需要横向移动" → 搜索异常的 SMB/RDP 连接模式
- "攻击者会尝试提权" → 搜索管理员账户的非正常登录

## 8. 等保合规

### 8.1 什么是等保？

等保（网络安全等级保护）是中国的网络安全强制标准，模仿美国的 NIST CSF 框架。所有关键信息基础设施运营者都必须通过等保测评。

### 8.2 等保 2.0 五级划分

| 等级 | 名称 | 适用场景 | 测评周期 |
|------|------|---------|---------|
| **第一级** | 用户自主保护 | 普通信息系统 | 自主定级 |
| **第二级** | 系统审计 | 涉及重要业务信息的系统 | 每 2 年测评 |
| **第三级** | 安全标记 | 金融、政务、能源等关键行业 | **每年测评** |
| **第四级** | 结构化保护 | 关键基础设施核心系统 | 每半年测评 |
| **第五级** | 访问验证 | 国防、机密系统 | 持续监控 |

**大多数金融客户的 SOC 平台要求达到等保三级**，这是进入金融行业的门槛。

### 8.3 等保三级对 SOC 审计日志的核心要求

| 要求项 | 具体要求 | 落地方式 |
|--------|---------|---------|
| **保留时长** | 审计日志 ≥ 5 年，操作日志 ≥ 5 年 | 冷存储（WORM） |
| **完整性保护** | 日志写后不可篡改 | WORM 存储 + hash 链 |
| **备份** | 必须有备份，支持恢复 | 主备双写，定期演练 |
| **访问控制** | 日志访问需授权，重要操作审计 | RBAC + 日志审计 |
| **时间同步** | 所有日志时间戳必须同步（NTP） | 统一 NTP 服务器 |

### 8.4 防篡改机制：WORM + Hash 链

**为什么防篡改是核心要求？** 攻击者入侵后，往往会删除/修改安全日志来掩盖痕迹。如果日志可被篡改，就没有法律证据价值。

```python
# 伪代码：日志完整性保护（Hash 链）
# 每条日志写入后，立即计算 hash 并链接到前一条，形成不可篡改的链式结构

import hashlib
from datetime import datetime

class ImmutableLogWriter:
    def __init__(self, storage: WORMStorage):
        self.storage = storage
        self.last_hash = "GENESIS"  # 创世块 hash

    def write(self, log_entry: dict) -> str:
        """
        将日志写入 WORM 存储，并生成 hash 链
        返回本条日志的 hash
        """
        timestamp = datetime.utcnow().isoformat()

        # 构建带 hash 的日志记录
        payload = {
            "log_data": log_entry,
            "timestamp": timestamp,
            "prev_hash": self.last_hash,  # 指向前一条
            "sequence": self._get_next_sequence()
        }

        # 计算当前记录的 hash
        serialized = json.dumps(payload, sort_keys=True, default=str)
        current_hash = hashlib.sha256(serialized.encode()).hexdigest()

        payload["current_hash"] = current_hash

        # 写入 WORM 存储（只写一次，永不修改）
        self.storage.append(payload)

        # 更新 last_hash（内存状态，不是文件修改）
        self.last_hash = current_hash

        return current_hash

    def verify_integrity(self) -> bool:
        """
        读取历史日志，验证 hash 链完整性
        用于审计人员定期检查
        """
        logs = self.storage.read_all()

        for i in range(1, len(logs)):
            current = logs[i]
            prev = logs[i - 1]

            # 验证 prev_hash 指向正确
            if current["prev_hash"] != prev["current_hash"]:
                return False

            # 重新计算当前 hash 验证数据未被篡改
            reconstructed = {
                "log_data": current["log_data"],
                "timestamp": current["timestamp"],
                "prev_hash": current["prev_hash"],
                "sequence": current["sequence"]
            }
            expected = hashlib.sha256(
                json.dumps(reconstructed, sort_keys=True, default=str).encode()
            ).hexdigest()

            if expected != current["current_hash"]:
                return False

        return True
```

### 8.5 多租户隔离

SOC 平台服务多个客户时，必须严格隔离租户数据：

| 隔离层级 | 实现方式 | 注意事项 |
|---------|---------|---------|
| **网络层** | VPC/安全组隔离 | 租户间网络完全不通 |
| **数据库层** | 按 tenant_id 行级隔离 | 避免 JOIN 越界查询 |
| **应用层** | 每个请求强制校验 tenant_id | 防越权访问 |
| **BI 层** | Row-level security | 报表仅展示当前租户数据 |

**⚠️ 常见漏洞**：在 GraphQL 或 REST API 中，如果 `tenant_id` 是前端传入的参数而不是服务端从 Session 中获取，攻击者可以通过修改参数访问其他租户的数据（IDOR 漏洞）。

## 9. 应急响应（PDCERF）

### 9.1 PDCERF 六阶段模型

PDCERF 是由 Mandiant（后被 Google 收购）提出的应急响应方法论，是面试 SOC 相关岗位必掌握的内容。

```
准备(Preparation)
      ↓
识别(Identification)
      ↓
遏制(Containment)
      ↓
消除(Eradication)
      ↓
恢复(Recovery)
      ↓
总结(Lessons Learned)
```

### 9.2 各阶段详解

#### 阶段 1：准备（Preparation）

**核心任务**：建立应急响应的基础设施，确保事件发生时能快速启动。

- 制定应急响应预案（Playbook）
- 组建事件响应团队，明确分工
- 准备取证工具包（内存镜像、硬盘复制工具）
- 建立与外部机构的联系（网安部门、ISP、法律顾问）

**⚠️ 常见误区**：很多公司到事件发生了才手忙脚乱写预案。准备阶段的核心价值是**"平时多流汗，战时少流血"**。

#### 阶段 2：识别（Identification）

**核心任务**：确认是否真的发生了安全事件，判断事件范围和严重程度。

```python
# 伪代码：事件识别决策流程
def identify_incident(alert: Alert) -> IncidentReport:
    # 第一步：确认告警是否真实
    analyst_verdict = analyst_investigate(alert)

    if analyst_verdict == "false_positive":
        return IncidentReport(status="dismissed", reason="confirmed_false_positive")

    # 第二步：如果是真实事件，评估影响范围
    scope = assess_scope(alert)
    # 评估维度：
    # - 受影响的主机数量
    # - 受影响的数据类型（敏感数据/普通数据）
    # - 是否在核心业务系统
    # - 攻击者是否仍在活动

    # 第三步：确定事件级别
    severity = determine_severity(scope)

    return IncidentReport(
        confirmed=True,
        scope=scope,
        severity=severity,
        initial_indicators=collect_iocs(alert)
    )
```

#### 阶段 3：遏制（Containment）

**核心任务**：阻止攻击继续扩散，保护未受影响的资产。

遏制分为**短期遏制**（立即行动）和**长期遏制**（彻底方案）：

| 类型 | 动作 | 例子 |
|------|------|------|
| 短期遏制 | 快速隔离，攻击仍可继续观察 | 断网隔离感染主机、封锁攻击者 IP |
| 长期遏制 | 彻底修复，消除再次入侵路径 | 修复漏洞、撤销泄露凭证、重置密码 |

```python
# 伪代码：SOAR 剧本 - 勒索软件遏制
# 当 P0 告警确认是勒索软件时，自动化遏制流程

def ransomware_containment_playbook(incident: Incident):
    steps = []

    # 步骤1：立即隔离受影响主机
    steps.append(SOARStep(
        action="isolate_endpoint",
        target=incident.affected_hosts,
        auto_execute=True,  # 立即执行，不需要人工审批
        description="自动隔离勒索软件感染主机"
    ))

    # 步骤2：封锁勒索软件通信的 C2 域名/IP
    c2_iocs = threat_intel.query_ransomware_c2()
    steps.append(SOARStep(
        action="block_ioc",
        targets=c2_iocs,
        auto_execute=True,
        description="阻断勒索软件 C2 通信"
    ))

    # 步骤3：通知安全运营经理（P0 事件必须通知）
    steps.append(SOARStep(
        action="notify",
        target="soc_manager",
        message=f"P0 勒索软件事件确认，受影响主机：{incident.affected_hosts}",
        auto_execute=True
    ))

    # 步骤4：备份当前内存镜像（取证用）
    steps.append(SOARStep(
        action="capture_memory",
        target=incident.affected_hosts,
        auto_execute=False,  # 需要分析师手动操作
        description="采集受感染主机内存镜像"
    ))

    # 执行所有自动步骤
    for step in steps:
        if step.auto_execute:
            execute_step(step)
        else:
            assign_to_analyst(step, tier=2)

    return PlaybookExecution(
        steps_executed=len([s for s in steps if s.auto_execute]),
        steps_pending=len([s for s in steps if not s.auto_execute]),
        containment_status="in_progress"
    )
```

#### 阶段 4：消除（Eradication）

**核心任务**：彻底清除攻击者，消除所有入侵路径。

- 删除恶意文件/进程/注册表项
- 修复导致入侵的漏洞（打补丁、关闭服务）
- 重置所有被泄露的凭证（密码、API Key、证书）
- 检查是否有后门（Rootkit、持久化机制）

**⚠️ 注意**：消除不彻底是常见的复发原因。攻击者经常在清理后留下后门，等风声过了再回来。

#### 阶段 5：恢复（Recovery）

**核心任务**：让业务恢复正常运行，同时保持监控警惕。

- 从备份恢复系统和数据（确认备份未被感染）
- 分阶段恢复服务（先核心业务，后辅助业务）
- 加强监控（恢复后 72 小时内高频监控）
- 验证恢复正常（用户能正常登录、数据未丢失）

#### 阶段 6：总结（Lessons Learned）

**核心任务**：从事件中学习，防止同类事件再次发生。

```python
# 伪代码：事件复盘报告结构
def generate_lessons_learned(incident: Incident) -> IncidentReport:
    return {
        "incident_id": incident.id,
        "timeline": {
            "first_evidence": incident.first_seen,
            "detected_at": incident.detected_at,
            "contained_at": incident.contained_at,
            "eradicated_at": incident.eradicated_at,
            "recovered_at": incident.recovered_at,
        },
        "metrics": {
            "dwell_time": incident.detected_at - incident.first_evidence,  # 驻留时间
            "mtd": incident.detected_at - incident.first_evidence,        # 同 MTTD
            "mtr": incident.recovered_at - incident.detected_at,           # 响应时间
        },
        "root_cause": {
            "initial_access_vector": "钓鱼邮件 → 用户点击恶意链接",
            "how_attacker_persisted": "创建了隐藏的管理员账户",
            "why_not_detected_sooner": "EDR 规则未覆盖该恶意软件类型",
        },
        "action_items": [
            {"task": "为所有员工开展钓鱼邮件识别培训", "owner": "安全培训团队", "due": "2周内"},
            {"task": "添加该恶意软件类型的 EDR 检测规则", "owner": "SOC 工程师", "due": "1周内"},
            {"task": "审计所有管理员账户，禁用非必要账户", "owner": "AD 管理员", "due": "3天内"},
        ],
        "what_went_well": [
            "SOAR 剧本在 3 分钟内完成了主机隔离",
            "备份数据完整，恢复过程顺利"
        ],
        "what_could_improve": [
            "EDR 告警到人工确认的时间偏长（~45 分钟）",
            "部分日志保留周期不足 90 天，无法完整溯源"
        ]
    }
```

## 10. 实战：SOC 平台告警降噪优化案例

### 10.1 案例背景

某互联网公司 SOC 平台日均告警量 15,000 条，L1 分析师 8 人，人均每日处理 1,875 条告警，实际处理能力约 400 条/天，导致：
- 告警积压严重，P2/P3 告警平均延迟 24 小时才处理
- 分析师疲劳，P1 告警中也出现了误判
- SOC SLA 达标率仅 65%

### 10.2 问题诊断

通过数据分析发现三个主要问题：

```python
# 伪代码：告警数据分析 - 定位问题规则
def diagnose_alert_noise():
    # 问题1：头部规则贡献了 80% 的告警量
    top_rules = query("""
        SELECT rule_name, COUNT(*) as cnt, SUM(disposition='false_positive') as fp
        FROM alerts
        WHERE created_at > NOW() - INTERVAL 30 DAY
        GROUP BY rule_name
        ORDER BY cnt DESC
        LIMIT 20
    """)

    print("Top 告警规则分析：")
    for rule in top_rules:
        fp_rate = rule.fp / rule.cnt
        print(f"  {rule.rule_name}: {rule.cnt} 条，误报率 {fp_rate:.1%}")

    # 输出：
    #   rule_ssh_bruteforce: 4500 条，误报率 92%   ← 高误报规则 1
    #   rule_port_scan_detection: 3200 条，误报率 88%  ← 高误报规则 2
    #   rule_off_hours_login: 2100 条，误报率 76%  ← 高误报规则 3
    #   rule_malware_detected: 800 条，误报率 31%  ← 可接受
```

### 10.3 优化方案

#### 第一步：规则调优（减少 60% 告警）

```python
# 优化前的规则：SSH 暴力破解
# 触发条件：同一 IP 1 小时内 SSH 登录失败超过 10 次
original_rule = {
    "name": "ssh_bruteforce",
    "condition": "same_ip.failures >= 10",
    "time_window": "1h"
}
# 问题：误报率 92%（开发人员频繁输错密码、VPN 重连都触发）

# 优化后的规则：
optimized_rule = {
    "name": "ssh_bruteforce",
    "condition": "same_ip.failures >= 10 AND "
                 "same_ip.successes == 0 AND "  # 从未成功过（更可疑）
                 "target_user NOT IN ['admin', 'root']",  # 排除常见 admin 账户（安全扫描）
    "time_window": "1h",
    "exceptions": ["known_security_scanner_ips"]  # 安全扫描 IP 白名单
}
# 效果：告警量从 4500 条/天 → 800 条/天，误报率从 92% → 23%
```

#### 第二步：ML 置信度过滤（再减少 25%）

对剩余告警用 ML 模型打分，低置信度告警自动抑制：

```python
# ML 降噪效果验证
def evaluate_ml_filtering():
    # 测试集：过去 30 天告警
    test_alerts = load_labeled_alerts(days=30)
    # test_alerts: 3000 条，其中 450 条是真告警，2550 条是误报

    model = AlertClassifier()
    model.train(training_data)

    # 阈值 0.6：置信度 < 0.6 的不推送
    filtered = [a for a in test_alerts if model.predict_proba(a) >= 0.6]

    true_positives = [a for a in filtered if a.disposition == "true_positive"]
    false_positives = [a for a in filtered if a.disposition == "false_positive"]
    suppressed = [a for a in test_alerts if a not in filtered]

    print(f"过滤前总告警：{len(test_alerts)} 条")
    print(f"过滤后总告警：{len(filtered)} 条（减少 {(1-len(filtered)/len(test_alerts))*100:.0f}%）")
    print(f"  - 保留真告警：{len(true_positives)} 条（召回率 {len(true_positives)/450*100:.1f}%）")
    print(f"  - 保留误报：{len(false_positives)} 条")
    print(f"  - 抑制告警：{len(suppressed)} 条")
    print(f"  - 抑制的误报：{[a for a in suppressed if a.disposition=='false_positive']} 条")
    # 输出：
    # 过滤前总告警：3000 条
    # 过滤后总告警：950 条（减少 68%）
    #   - 保留真告警：380 条（召回率 84.4%）
    #   - 保留误报：570 条
    #   - 抑制告警：2050 条
    #   - 抑制的误报：1980 条（占被抑制的 96.6%）
```

#### 第三步：聚合优化（处理告警风暴）

当攻击者发动大规模扫描时，可能在几分钟内触发上千条告警。聚合将同一攻击的多条告警合并为一条事件：

```python
# 告警聚合策略
aggregation_rules = {
    # 规则1：同源 IP + 同目标端口 + 5 分钟内 → 合并
    "time_window_aggregation": {
        "group_by": ["source_ip", "dest_port"],
        "window": timedelta(minutes=5),
        "merge_alerts": True
    },
    # 规则2：同主机 + ATT&CK 同战术 → 合并
    "semantic_aggregation": {
        "group_by": ["target_hostname", "att&ck_tactic"],
        "window": timedelta(minutes=30),
        "merge_alerts": True
    }
}
# 效果：扫描告警从 3200 条/天 → 180 条事件（减少 94%）
```

### 10.4 最终效果

| 指标 | 优化前 | 优化后 | 变化 |
|------|-------|-------|------|
| 日均告警量 | 15,000 条 | 2,800 条 | **-81%** |
| L1 人均日处理 | 1,875 条 | 350 条 | **-81%** |
| P2/P3 处理延迟 | 24 小时 | 3 小时 | **-87.5%** |
| SLA 达标率 | 65% | 94% | **+29pp** |
| 真告警召回率 | ~90% | ~87% | -3pp（可接受） |
| 月均误报处理 | ~8,000 条 | ~600 条 | **-92.5%** |

**⚠️ 关键教训**：
1. **先调规则，再上 ML**：规则调优的效果通常比 ML 更稳定，优先处理高误报规则
2. **ML 阈值需要持续调优**：上线后每月评估，根据召回率调整阈值
3. **保留原始告警**：聚合展示层，不丢失底层数据，分析师随时可展开

## 附录：面试高频问题速查

| 问题 | 核心回答要点 |
|------|------------|
| SOC 为什么需要分层？ | 避免分析师疲劳 + 技能匹配 + SLA 分级 |
| MTTD 和 MTTR 是什么？ | 平均检测时间 + 平均响应时间，越短越好 |
| 告警降噪怎么做？ | 规则调优（数据驱动收紧条件）+ ML 置信度过滤 |
| 误报率怎么定义？ | 判定为无需处置的告警 / 总推送告警 |
| 横向移动怎么检测？ | NDR（网络流量）+ EDR（主机日志）双路线 |
| APT 的低慢大是什么？ | 低频 + 慢速 + 大规模，单点检测不了 |
| PDCERF 六阶段？ | 准备→识别→遏制→消除→恢复→总结 |
| 等保三级对日志的要求？ | 保留 ≥ 5 年 + 防篡改（WORM+hash 链）+ 备份 |
| IOC 过期怎么办？ | 设置 TTL，低置信度 IOC 短 TTL，到期自动失效 |
