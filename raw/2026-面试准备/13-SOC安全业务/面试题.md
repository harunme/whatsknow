# SOC 安全业务面试题

## 面试题

### Q1: 告警降噪
SOC 平台每天会产生多少告警？误报率大概多少？怎么通过规则调优减少噪音？有没有用过机器学习做告警分类？

**追问：**
- 误报率怎么定义？
- 规则调优的标准是什么？

### Q2: 威胁情报整合
你们的告警分析有没有对接外部威胁情报（TI）？用的哪家的？IOC（IP/域名/Hash）是怎么关联到告警的？

**追问：**
- STIX/TAXII 是什么？
- IOC 过期怎么清理？

### Q3: 告警聚合
海量告警怎么做聚合？同攻击链上的多个告警能不能合并成一个事件？

**追问：**
- 时间窗口聚合和语义聚合的区别？
- 告警合并会不会丢失关键信息？

### Q4: 攻击链建模
告警和攻击链（Kill Chain）的映射逻辑是什么？单个告警怎么判断它属于哪个攻击阶段？

**追问：**
- 横向移动检测是 SOC 的核心难题，怎么检测？
- NDR 和 EDR 路线各自的优缺点？

### Q5: 上下文丰富
告警触发后，分析师需要哪些上下文？这些数据是怎么关联展示的？

**追问：**
- 告警上下文包括哪些维度？
- 上下文太多会不会干扰分析？

### Q6: SIEM 日志采集
日志是怎么采集的？采集链路延迟是多少？怎么保证不丢日志？

**追问：**
- FileBeat 和 Syslog 的区别？
- Kafka 在日志采集中起什么作用？

### Q7: 应急响应流程
发生重大安全事件（比如勒索软件）时，SOC 的应急响应流程是什么？SOAR 剧本是怎么介入的？

**追问：**
- MTTD 和 MTTR 怎么缩短？
- 红蓝对抗有什么用？

### Q8: 合规审计
金融客户对审计日志有什么特殊要求？操作日志、告警处置记录保存多久？防篡改怎么做的？

**追问：**
- 等保是什么？等保三级有哪些要求？
- 多租户隔离怎么实现？

## 答案

### A1: 告警降噪
**[10秒结论]** 规则调优 + ML 分类双管齐下：规则太宽泛则收紧条件，ML 模型对告警做置信度打分，低置信告警不推送；误报率定义为"被分析师判定为无需处理的告警/总告警"。

**[90秒拆解]**
- **规则调优**：统计各规则的触发频率和分析师处理结果（真告警 vs 误报），对触发频率高但真告警率 < 5% 的规则收紧条件（如增加 IP 白名单、时间窗口限制）
- **ML 分类**：将告警上下文（告警类型、源 IP、目标资产、用户角色、历史行为基线）打包为特征向量，用分类模型预测是否为真告警；置信度 < 0.6 的不推送
- **误报率定义**：分析师判断"无需处置"的告警 / 总推送告警，不是"规则触发但实际无害"
- **反馈闭环**：分析师每次处置（确认/降级/升级）回流为训练数据，月度汇总评估各规则精确率和召回率

**[20秒收尾]** 告警降噪的目标是让分析师只处理真正需要关注的告警，而不是把所有告警都推给他们——减少疲劳比减少漏报同样重要。

### Q2: 威胁情报整合
**[10秒结论]** 对接 AlienVault OTX / Recorded Future 等 TI 供应商，STIX/TAXII 协议自动同步 IOC；IOC 以 IP/域名/Hash 为 Key 与告警上下文关联命中；过期 IOC（TTL 通常 30 天）自动清理。

**[90秒拆解]**
- **IOC 类型**：IP（C2 服务器、僵尸网络）、域名（钓鱼 URL、DGA 域名）、Hash（PE Hash）、URL
- **STIX/TAXII**：STIX 2.1 描述 IOC 结构（JSON 格式），TAXII 2.1 负责传输（PKI 认证的 HTTPS API）
- **关联流程**：告警触发 → 查询 TI Matcher（预计算面板）→ IOC 命中 → 自动提升告警优先级或触发专项剧本
- **IOC 过期清理**：IOC 有 `valid_until` 时间戳，过期后自动失效；低置信度 IOC 设置更短 TTL（如 7 天）
- **自建 TI 价值**：内部沙箱分析的恶意样本 Hash、蜜罐捕获的 C2 域名，属于第一方情报，置信度最高

**[20秒收尾]** TI 整合是 SOC 平台的增值层——当告警中的 IP 与高置信度 IOC 命中时，告警可信度大幅提升，可自动升级。

### Q3: 告警聚合
**[10秒结论]** 分层聚合：实时层（同主机同类型立即合并）→ 窗口层（5 分钟内同源 IP 合并）→ 异步层（跨主机跨战术 ML 聚类）；原始告警始终保留。

**[90秒拆解]**
- **时间窗口聚合**：同源 IP 在 5 分钟内触发的 NIDS 扫描告警合并为 1 个 Incident，避免告警风暴
- **攻击链聚合**：同主机告警按 ATT&CK 战术阶段构建图，单个告警升级为攻击链告警（如 ssh_bruteforce + account_create → credential_access）
- **语义聚类（ML）**：Sentence-BERT 嵌入告警文本，余弦相似度 > 0.85 自动归并，发现规则未覆盖的新型攻击模式
- **信息丢失担忧**：聚合发生在展示层，原始告警始终独立存储；分析师可随时"展开聚合"查看底层细节

**[20秒收尾]** 告警聚合不是合并数据，而是减少告警噪音；原始数据永远是第一位的，分析需求随时可以展开。

### Q4: 攻击链建模
**[10秒结论]** 告警类型到 ATT&CK 战术的静态映射表做初判，LLM 在映射候选集内做推理；横向移动检测用 NDR（网络流量异常）和 EDR（主机日志）双路线联动。

**[90秒拆解]**
- **映射逻辑**：告警类型（如 ssh_bruteforce）→ ATT&CK 战术候选集（如 {initial_access, credential_access}）→ LLM 结合上下文确定最终战术
- **横向移动检测**：
  - NDR 路线：同源 IP 对内网多主机的 445/3389 端口连接、同一源对多个目标的 SMB Session Setup 异常
  - EDR 路线：Windows 4688 Process Create 异常链（如 excel→cmd→powershell）、同一用户短时间多主机登录
- **双路线联动**：NDR 发现"内网横向路径图"，EDR 确认"源主机可疑进程"，两路信号关联置信度远高于单一数据源

**[20秒收尾]** 横向移动是 APT 的核心行为，检测横向移动能力是衡量 SOC 平台成熟度的重要标志。

---

### Q5: 上下文丰富
**[10秒结论]** 告警上下文包括：资产信息（重要性/所属业务）、用户身份（部门/角色/权限级别）、历史告警（7 天内同类告警）、威胁情报（TI IOC 命中）、ATT&CK 战术阶段；信息太多反而干扰，需分层展示。

**[90秒拆解]**
- **上下文维度**：
  - 资产：主机名、IP、操作系统、重要性等级（核心/普通/测试）、所属业务线
  - 用户：用户名、部门、职级、最近登录时间、账户有效期
  - 历史：同 IP/同用户/同主机 7 天内告警列表
  - TI：IP/域名/Hash 是否命中高置信度 IOC
  - ATT&CK：当前告警关联的战术阶段和已知攻击模式
- **分层展示**：分析师最关注的上下文（资产重要性、TI 命中、历史告警数）优先展示，其余通过"展开"按钮按需查看
- **避免干扰**：上下文噪音过多会干扰分析判断，需要 A/B 测试验证展示层设计的有效性

**[20秒收尾]** 好的告警详情页是分析师最常用的工作界面，上下文信息的优先级排序直接影响调查效率。

### Q6: SIEM 日志采集
**[10秒结论]** FileBeat 轻量 Agent 部署在目标主机，Kafka 做消息队列削峰，At-Least-Once 保证不丢日志；端到端采集延迟目标 < 5 秒。

**[90秒拆解]**
- **采集链路**：目标主机（FileBeat/WinBeat）→ Kafka（削峰填谷）→ Logstash/Consumer（解析归一化）→ Elasticsearch（存储索引）
- **FileBeat vs Syslog**：
  - FileBeat：断点续传、本地缓存+ACK、精确感知文件轮转
  - Syslog：UDP 推送，简单但可能丢包，适合网络设备
- **Kafka 的作用**：日志生产速率波动大（凌晨少，白天多），Kafka 作为缓冲区让下游消费者平稳消费，削峰填谷
- **At-Least-Once**：FileBeat 发送后等 ACK → Logstash 幂等写入 ES（`_id` = hash）→ Consumer 提交 OFFSET 后才删除本地缓存

**[20秒收尾]** 日志采集是 SIEM 数据质量的源头，Kafka + FileBeat 是互联网公司日志采集的标准架构。

### Q7: 应急响应流程
**[10秒结论]** 安全事件应急流程：检测→遏制→取证→恢复→事后复盘；SOAR 剧本自动化遏制动作（封锁 IP、隔离主机），人工做取证和恢复决策；MTTD 靠检测覆盖率，MTTR 靠 SOAR 自动化程度。

**[90秒拆解]**
- **应急流程**：确认事件（P0/P1 告警聚合后确认为安全事件）→ 遏制（SOAR 自动封锁/隔离）→ 取证（SIEM 溯源分析）→ 恢复（备份恢复、业务恢复）→ 复盘（事件报告、根因分析）
- **SOAR 介入点**：遏制阶段（封锁 IP、隔离主机、阻断会话）自动化；取证和恢复由分析师决策
- **MTTD 缩短**：提高检测覆盖率（ATT&CK Navigator 评估盲区）+ 降低采集延迟
- **MTTR 缩短**：SOAR 自动化遏制（从分钟级到秒级）+ 告警聚合减少分析师处理量
- **红蓝对抗**：每年 1-2 次攻防演练，验证 SOC 平台检测能力，发现 blind spots 并补规则

**[20秒收尾]** SOC 平台的价值最终体现在 MTTD 和 MTTR 上——检测越快、响应越快，攻击造成的损失越小。

### Q8: 合规审计
**[10秒结论]** 金融客户（深交所、银联）等保三级要求审计日志保留 ≥ 5 年、操作日志保留 ≥ 5 年、告警处置记录保留 ≥ 5 年；防篡改通过写后即不可改（WORM）存储 + hash 链完整性校验。

**[90秒拆解]**
- **等保三级要求**：审计日志保留 ≥ 5 年、必须防篡改、必须有备份、必须有访问控制
- **审计日志类型**：操作日志（谁在什么时间做了什么操作）、告警处置记录（分析师如何处理了告警）、剧本执行记录（SOAR 做了什么动作、结果如何）
- **防篡改**：WORM（Write Once Read Many）存储 + 每条日志 SHA256 哈希写入链式结构，读取时校验完整性
- **多租户隔离**：数据库层按 tenant_id 严格隔离（行级安全策略）、租户间数据不得互相访问告警数据、通过 BI 工具的 row-level security 实现逻辑隔离

**[20秒收尾]** 合规审计是 SOC 平台的门槛要求，等保三级认证是进入金融行业的入场券，数据完整性是审计日志的生命线。
