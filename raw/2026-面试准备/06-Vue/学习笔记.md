## 1. Vue 基础概念

### 1.1 什么是渐进式框架

Vue 的核心理念是**渐进式增强**（Progressive Enhancement）。它不像 Angular 或 Ember 那样要求你一开始就接受整套全家桶，而是从视图层（view）出发，你可以逐步引入更多能力：

| 渐进层级 | 引入的功能 | 适用场景 |
|----------|-----------|---------|
| 第一层 | 模板语法 + 响应式数据 | 简单的页面交互 |
| 第二层 | 组件系统 | 多人协作、模块化开发 |
| 第三层 | Vue Router | 单页应用（SPA） |
| 第四层 | 状态管理（Pinia） | 跨组件共享状态 |
| 第五层 | SSR / SSG（Nuxt） | SEO 优化、首屏加速 |

**为什么这样设计？** 渐进式的最大好处是**降低学习曲线**。你可以从最简单的 `<script>` 引入 CDN 版本开始写一个 todo 列表，然后逐步引入工程化工具（Vite）、组件系统、路由，最终形成完整应用。这种方式让团队可以根据项目规模选择合适的复杂度，而不是一刀切。

### 1.2 组件化思想

组件（Component）是 Vue 的核心抽象单位。一个组件本质上是一个**具有独立职责的 UI 单元**，它拥有自己的模板（template）、逻辑（script）和样式（style）。

```vue
<!-- CountButton.vue：一个最简单的组件 -->
<template>
  <button @click="count++">Clicked {{ count }} times</button>
</template>

<script>
export default {
  data() {
    return { count: 0 }
  }
}
</script>
```

**组件化的核心价值在于三点：**

1. **可复用性**（Reusability）：同一个组件可以在多个页面中使用，数据通过 props 传入，行为通过事件向外传递。
2. **可维护性**（Maintainability）：每个组件职责单一，修改一个组件不会意外影响其他组件。
3. **可测试性**（Testability）：组件的输入（props + events）和输出（DOM）都是可预测的。

在 SOC（安全运营中心）系统中，组件化的思想体现得尤为明显：告警卡片（AlertCard）、时间线（Timeline）、图表容器（ChartWrapper）都可以抽取为独立组件。

### 1.3 第一个 Vue 实例

```javascript
import { createApp } from 'vue'

const app = createApp({
  data() {
    return {
      message: 'Hello, SOC Alert System!',
      severity: 'high'
    }
  },
  computed: {
    labelClass() {
      return `severity-${this.severity}`
    }
  },
  methods: {
    handleAlert() {
      this.message = 'Alert acknowledged'
    }
  }
})

app.mount('#app')
```

```html
<!-- 对应的 HTML -->
<div id="app">
  <p :class="labelClass">{{ message }}</p>
  <button @click="handleAlert">Acknowledge</button>
</div>
```

**为什么要用 `createApp` 而不是 `new Vue`？** Vue 3 引入 `createApp` 是为了解决**应用实例隔离**问题。在 Vue 2 中，所有全局配置（`Vue.config`、`Vue.mixin`）都是全局共享的，多个 Vue 实例会相互影响。Vue 3 的 `createApp` 返回一个独立的 app 实例，每个实例有独立的全局配置，互不干扰。

## 2. 模板语法

### 2.1 插值（Interpolation）

模板中的 `{{ }}` 语法称为**插值表达式**（Mustache 语法）。Vue 在编译阶段将 `{{ message }}` 转换为类似以下的代码：

```javascript
// 编译前（模板）
<p>Alert {{ severity }}: {{ message }}</p>

// 编译后（render 函数）
h('p', null, [
  'Alert ',
  ctx.severity,          // 动态绑定
  ': ',
  ctx.message            // 动态绑定
])
```

**插值表达式中可以使用 JavaScript 表达式，但有以下限制：**

```html
<!-- 合法的表达式 -->
{{ count + 1 }}
{{ ok ? 'YES' : 'NO' }}
{{ message.split('').reverse().join('') }}
{{ user.name }}
{{ (a, b) => a + b }}  <!-- 箭头函数也可以 -->

<!-- 非法的语句（不是表达式） -->
{{ if (ok) { return message } }}   <!-- ❌ 不支持语句 -->
{{ a = 1 }}                          <!-- ⚠️ 赋值表达式在 Vue 2 中不可用，Vue 3 可用但不推荐 -->
```

⚠️ **注意：** 模板中不要写过于复杂的表达式。超过 3 行的逻辑应该提取到 computed 或 methods 中，否则模板难以阅读和维护。

### 2.2 指令（Directives）

Vue 指令是带有 `v-` 前缀的特殊属性。它们在模板编译时被解释为对应的 JavaScript 逻辑。

#### v-bind：动态绑定属性

```html
<!-- 普通写法 -->
<img v-bind:src="imageUrl" :alt="imageAlt" class="preview-img" />

<!-- 简写（最常用） -->
<img :src="imageUrl" :alt="imageAlt" :class="{ active: isSelected }" />
```

```javascript
data() {
  return {
    imageUrl: '/assets/alert-screenshot.png',
    imageAlt: '告警详情截图',
    isSelected: true
  }
}
```

**为什么需要 v-bind？** HTML 属性值默认是字符串字面量。`:src="imageUrl"` 告诉 Vue："不要把 `imageUrl` 当字符串用，而是从组件的 data/computed 中读取它的值作为属性值。"

#### v-if / v-else-if / v-else：条件渲染

```html
<div v-if="severity === 'critical'" class="alert alert-critical">
  紧急告警：{{ message }}
</div>
<div v-else-if="severity === 'warning'" class="alert alert-warning">
  警告：{{ message }}
</div>
<div v-else class="alert alert-info">
  信息：{{ message }}
</div>
```

**v-if 的工作机制：** v-if 在条件为 false 时，Vue 根本不会创建对应的 DOM 元素（通过控制 `vnode` 的生成）。如果条件频繁在 true/false 之间切换，v-if 会造成大量的 DOM 创建/销毁开销。

```javascript
// 编译后大概等价于：
if (severity === 'critical') {
  return h('div', { class: 'alert alert-critical' }, ['紧急告警：' + message])
} else if (...) {
  // ...
}
```

#### v-show：显示/隐藏（display 控制）

```html
<!-- v-show 只是切换 CSS 的 display 属性 -->
<div v-show="isLoggedIn">欢迎回来，{{ username }}</div>
```

| 对比项 | v-if | v-show |
|--------|------|--------|
| 初始渲染 | false 时不渲染 DOM | 始终渲染，只是 `display: none` |
| 切换成本 | 高（创建/销毁 DOM） | 低（仅切换 CSS） |
| 支持 template | 是 | 否 |
| 触发生命周期 | 是（完整的 create/destroy 周期） | 否（只是 CSS 切换） |

⚠️ **注意：** 不要在 v-if 和 v-show 之间死板选择，要根据**切换频率**和**初始条件**判断：
- 切换频率高（如 Tab 切换）→ v-show
- 初始条件大概率是 false，且展示后不再频繁切换 → v-if

#### v-for：列表渲染

```html
<!-- 基础用法 -->
<ul>
  <li v-for="alert in alertList" :key="alert.id">
    {{ alert.severity }} - {{ alert.message }}
  </li>
</ul>

<!-- 带索引 -->
<li v-for="(alert, index) in alertList" :key="alert.id">
  {{ index + 1 }}. {{ alert.message }}
</li>

<!-- 嵌套解构（Vue 3） -->
<li v-for="({ id, message }, i) in alertList" :key="id">
  {{ i + 1 }}. {{ message }}
</li>

<!-- 对象遍历 -->
<div v-for="(value, key, index) in alertDetail" :key="key">
  {{ index + 1 }}. {{ key }}: {{ value }}
</div>
```

```javascript
data() {
  return {
    alertList: [
      { id: 1, severity: 'critical', message: '异常登录行为' },
      { id: 2, severity: 'warning', message: '端口扫描' },
      { id: 3, severity: 'info', message: '配置变更' }
    ]
  }
}
```

⚠️ **注意：** `:key` 是 v-for 的最佳实践关键。Vue 的 Diff 算法依赖 key 来判断哪些节点可以复用。没有 key 时，Vue 会尝试就地复用节点（in-place patching），可能导致状态错乱（比如列表中的输入框内容混乱）。

**key 的最佳实践：**
```html
<!-- ✅ 优先用唯一 ID -->
<li v-for="alert in alertList" :key="alert.id">

<!-- ❌ 避免用数组索引作为 key -->
<li v-for="(alert, index) in alertList" :key="index">
<!-- 当列表顺序变化时，key 变化导致 DOM 重新渲染，性能反而更差 -->
```

#### v-on：事件处理

```html
<!-- 完整写法 -->
<button v-on:click="handleClick">确认</button>

<!-- 简写（最常用） -->
<button @click="handleClick">确认</button>

<!-- 带参数 -->
<button @click="handleAction('acknowledge', alertId)">确认</button>

<!-- 原生事件对象 $event -->
<button @click="handleClick($event)">点我</button>

<!-- 多个事件 -->
<button @click="one(), two()">两个都执行</button>
```

```javascript
methods: {
  handleClick(event) {
    console.log('点击了按钮', event.target)
    console.log('鼠标位置：', event.clientX, event.clientY)
  },
  handleAction(action, id) {
    // 调用 API
    this.$api.alert[action](id)
  }
}
```

**事件修饰符（Modifiers）：** Vue 提供了方便的事件修饰符，避免手动处理原生事件逻辑：

```html
<!-- .stop - 阻止冒泡（等效于 event.stopPropagation()） -->
<div @click="outer">
  <button @click.stop="inner">点我</button>
</div>

<!-- .prevent - 阻止默认行为（等效于 event.preventDefault()） -->
<form @submit.prevent="handleSubmit">
  <button type="submit">提交</button>
</form>

<!-- .capture - 捕获模式（从外到内触发） -->
<div @click.capture="handleCapture">捕获模式</div>

<!-- .self - 只有 e.target === e.currentTarget 时触发 -->
<div @click.self="handleSelf">只有点击自己才触发</div>

<!-- .once - 只触发一次 -->
<button @click.once="handleOnce">只执行一次</button>

<!-- .passive - 提升滚动性能（告诉浏览器不会 preventDefault） -->
<div @touchstart.passive="onScroll">滑动</div>

<!-- 组合：先阻止默认行为再阻止冒泡 -->
<form @submit.prevent.stop="handleSubmit">
```

#### v-model：双向绑定

`v-model` 是 Vue 中最常用也是最需要深入理解的指令。它在**表单输入**和**组件通信**中有不同的实现方式。

**在表单上的 v-model 本质上是什么？**

```html
<!-- v-model 的等价写法（以 input 为例） -->
<input v-model="searchText" />

<!-- 等价于： -->
<input :value="searchText" @input="searchText = $event.target.value" />
```

```javascript
// 对应的 data
data() {
  return { searchText: '' }
}
```

**不同表单元素的 v-model 行为：**

| 元素类型 | v-model 绑定属性 | 触发事件 |
|----------|----------------|---------|
| `<input text>` | value | input |
| `<input checkbox>` | checked | change |
| `<input radio>` | checked | change |
| `<textarea>` | value | input |
| `<select>` | value（单选）/selected（多选） | change |

```html
<!-- 复选框（单个） -->
<input type="checkbox" v-model="isConfirmed" />
<label>确认告警</label>

<!-- 复选框（多个） -->
<input type="checkbox" v-model="selectedSeverities" value="critical" />
<input type="checkbox" v-model="selectedSeverities" value="warning" />

<!-- 单选 -->
<input type="radio" v-model="priority" value="high" />
<input type="radio" v-model="priority" value="low" />

<!-- 下拉框 -->
<select v-model="selectedAlert">
  <option value="">请选择</option>
  <option value="critical">紧急</option>
  <option value="warning">警告</option>
</select>
```

⚠️ **v-model 的修饰符：**
```html
<!-- .lazy - 失焦时同步（而不是每次输入） -->
<input v-model.lazy="searchText" />

<!-- .trim - 自动去除首尾空格 -->
<input v-model.trim="username" />

<!-- .number - 自动转为数字 -->
<input v-model.number="count" type="text" />
```

## 3. 组件通信

### 3.1 props 和 $emit（父子通信）

**Props 向下：父组件向子组件传递数据**

```vue
<!-- 父组件 AlertList.vue -->
<template>
  <AlertCard
    v-for="alert in alertList"
    :key="alert.id"
    :alert="alert"
    @acknowledge="handleAck"
  />
</template>

<script>
import AlertCard from './AlertCard.vue'

export default {
  components: { AlertCard },
  data() {
    return {
      alertList: [
        { id: 1, severity: 'critical', message: '暴力破解攻击' },
        { id: 2, severity: 'warning', message: '异常访问' }
      ]
    }
  },
  methods: {
    handleAck(alertId) {
      console.log('确认告警 ID:', alertId)
      this.alertList = this.alertList.filter(a => a.id !== alertId)
    }
  }
}
</script>
```

```vue
<!-- 子组件 AlertCard.vue -->
<template>
  <div :class="['alert-card', `severity-${alert.severity}`]">
    <h3>{{ alert.message }}</h3>
    <button @click="onAck">确认</button>
  </div>
</template>

<script>
export default {
  // props 定义：数组语法（简单场景）
  // props: ['alert']

  // 对象语法（推荐，支持类型验证）
  props: {
    alert: {
      type: Object,       // 类型检查
      required: true,     // 是否必填
      default() {         // 默认值（对象/数组必须用函数返回）
        return {}
      },
      validator(value) {  // 自定义验证函数
        return ['critical', 'warning', 'info'].includes(value.severity)
      }
    }
  },
  emits: ['acknowledge'],  // Vue 3：声明发出的事件（更好的 IDE 支持和类型提示）
  methods: {
    onAck() {
      // 通知父组件：确认了这条告警
      this.$emit('acknowledge', this.alert.id)
    }
  }
}
</script>
```

⚠️ **Props 的单向数据流原则：** 父组件通过 props 传递数据给子组件，**子组件不能直接修改 props**。如果子组件需要"修改"这个值，正确的做法是：
1. 在子组件中用 props 的值初始化一个本地 data。
2. 或者子组件 emit 一个事件，让父组件来决定如何修改。

```javascript
// ❌ 错误做法
props: ['initialCount'],
mounted() {
  this.initialCount++  // 不应该直接修改 props
}

// ✅ 正确做法
props: ['initialCount'],
data() {
  return { count: this.initialCount }  // 用本地 data 承接
}

// ✅ 或者 emit 事件让父组件处理
methods: {
  increment() {
    this.$emit('update:count', this.count + 1)
  }
}
```

### 3.2 v-model 在组件上的工作原理

**在组件上使用 v-model：** 这是 Vue 组件通信中最重要的模式之一。

```vue
<!-- 父组件使用 v-model -->
<template>
  <SearchInput v-model="searchText" />
  <!-- 等价于 -->
  <SearchInput :modelValue="searchText" @update:modelValue="searchText = $event" />
</template>

<script>
export default {
  data() { return { searchText: '' } }
}
</script>
```

```vue
<!-- 子组件 SearchInput.vue -->
<template>
  <input
    :value="modelValue"
    @input="$emit('update:modelValue', $event.target.value)"
    placeholder="搜索告警..."
  />
</template>

<script>
export default {
  props: ['modelValue'],
  emits: ['update:modelValue']
}
</script>
```

**Vue 3.4+ 的 `defineModel`（推荐）：** 简化双向绑定，省去手动定义 props 和 emits：

```vue
<!-- Vue 3.4+ 子组件 -->
<script setup>
const model = defineModel()
// 使用起来就像使用普通变量一样
</script>
<template>
  <input v-model="model" />
</template>
```

### 3.3 $attrs 和 $listeners（跨代传值）

**$attrs：** 父组件传递的、**子组件没有通过 props 接收**的所有属性（包括 class、style、data-*、事件监听器）。

**$listeners（Vue 2 独有）：** 所有父组件传递的事件监听器。Vue 3 中已合并到 $attrs 中。

```vue
<!-- 父组件 -->
<template>
  <!-- 给子组件传递 id、class、data-source 属性，以及 click 和 focus 事件 -->
  <BaseButton
    id="confirm-btn"
    class="btn-primary"
    data-action="confirm"
    @click="handleClick"
    @focus="handleFocus"
  />
</template>
```

```vue
<!-- BaseButton 子组件（没有声明任何 props 和 emits） -->
<template>
  <!-- $attrs 中包含 id、data-source、class 和所有事件 -->
  <button v-bind="$attrs" class="base-btn">
    <slot />
  </button>
</template>

<!-- 编译后等价于 -->
<template>
  <button
    id="confirm-btn"
    class="base-btn btn-primary"  <!-- 合并了组件的 class -->
    data-action="confirm"
    @click="handleClick"
    @focus="handleFocus"
  >
    <slot />
  </button>
</template>
```

⚠️ **注意：** Vue 3 中，如果子组件在 `<script setup>` 中使用了 `defineProps` 或 `defineEmits`，$attrs 会**自动不再包含**已经在 props/emits 中声明的属性和事件。这是 Vue 3 对 attrs 的改进。

### 3.4 provide 和 inject（跨级通信）

当组件嵌套层级较深时（比如 5 层以上），用 props 一层层传递叫做 **prop drilling**，非常不优雅。provide/inject 允许**祖先组件一次性向所有后代注入**数据或方法。

```vue
<!-- 最顶层组件 App.vue -->
<script>
import { provide } from 'vue'
import AlertStore from './store/alert'

export default {
  setup() {
    // 注入响应式数据
    provide('alertCount', ref(0))

    // 注入只读引用（防止后代修改）
    provide('alertStore', readonly(AlertStore))

    // 注入方法
    provide('dismissAlert', (id) => {
      AlertStore.dismiss(id)
    })
  }
}
</script>
```

```vue
<!-- 中间层组件（不需要知道这些数据，用不着 props 接收） -->
<template>
  <div class="layout">
    <Sidebar />
    <MainContent />
  </div>
</template>
```

```vue
<!-- 深层后代组件 AlertBadge.vue -->
<script>
import { inject } from 'vue'

export default {
  setup() {
    const alertCount = inject('alertCount')
    const dismissAlert = inject('dismissAlert')

    return { alertCount, dismissAlert }
  }
}
</script>

<template>
  <span class="badge" @click="dismissAlert(alertCount.id)">
    {{ alertCount }}
  </span>
</script>
```

**provide/inject 的应用场景：**
- 主题系统（ThemeProvider / useTheme）
- 当前登录用户信息（避免每个组件都 props 传入）
- 全局配置（API 地址、WebSocket 连接）
- SOC 系统中的告警实时推送上下文

⚠️ **注意：** provide/inject **不是响应式的**（注入的普通值不会自动更新）。如果需要响应式，必须注入 `ref` 或 `reactive` 对象，后代使用时才能保持响应式。也可以使用 `toRef` 技巧。

### 3.5 兄弟组件通信

兄弟组件之间没有直接的父子关系，需要借助**事件总线**或**状态管理库**：

**方案一：父组件作为中间人（最简单）**

```vue
<template>
  <AlertFilter @change="handleFilterChange" />  <!-- 兄弟组件1 -->
  <AlertList :filter="currentFilter" />        <!-- 兄弟组件2 -->
</template>
```

**方案二：EventBus（Vue 2 常用，Vue 3 中不推荐）**

```javascript
// eventBus.js
import { emitter } from 'mitt'  // Vue 3 推荐用 mitt
export default new mitt()
```

```javascript
// AlertFilter 组件（发送）
import emitter from './eventBus'
emitter.emit('filter-change', newFilter)
```

```javascript
// AlertList 组件（接收）
import emitter from './eventBus'
onMounted(() => {
  emitter.on('filter-change', (filter) => {
    this.currentFilter = filter
  })
})
onUnmounted(() => {
  emitter.off('filter-change')
})
```

⚠️ **注意：** Vue 3 已移除 `$on`、`$off`、`$once` 实例方法，所以不再推荐用 `new Vue()` 作为事件总线，推荐使用独立的 event emitter 库如 `mitt` 或 `tiny-emitter`。

**方案三：Pinia（2023 年后主流方案）**

```javascript
// stores/filter.js
import { defineStore } from 'pinia'
export const useFilterStore = defineStore('filter', {
  state: () => ({ currentFilter: 'all' }),
  actions: {
    setFilter(filter) { this.currentFilter = filter }
  }
})
```

```javascript
// AlertFilter 组件
const filterStore = useFilterStore()
filterStore.setFilter('critical')

// AlertList 组件
const filterStore = useFilterStore()
// computed 中使用 filterStore.currentFilter 自动响应式
```

## 4. 生命周期

### 4.1 完整生命周期图解

```
Vue 2                              Vue 3
--------                           --------
beforeCreate  ← setup()            （setup 在 beforeCreate 之前）
created                            beforeCreate
beforeMount                        created
mounted                            beforeMount  ← onBeforeMount
                                   mounted      ← onMounted
beforeUpdate                       beforeUpdate ← onBeforeUpdate
updated                            updated      ← onUpdated
beforeDestroy                       beforeUnmount ← onBeforeUnmount
destroyed                          unmounted    ← onUnmounted
activated                          activated    ← onActivated
deactivated                        deactivated  ← onDeactivated
errorCaptured                      errorCaptured
                                   renderTracked
                                   renderTriggered
```

### 4.2 各阶段详解与代码示例

```javascript
// Vue 3 Options API 写法
export default {
  // ❌ data/methods 不可用（setup 刚执行，data 还未初始化）
  beforeCreate() {
    console.log('beforeCreate:', this.message)  // undefined
  },

  // ✅ data/methods 已就绪，DOM 不可用
  // 适合：初始化非 DOM 依赖的状态、发起初始 API 请求
  created() {
    console.log('created:', this.message)        // 'Hello'
    console.log('DOM 存在吗?', this.$el)        // undefined
    this.fetchAlertList()  // 最常见的 API 调用时机
  },

  // 模板已编译成 render 函数，但 DOM 还未挂载
  beforeMount() {
    console.log('即将挂载，$el 是:', this.$el)   // undefined
  },

  // ✅ DOM 已挂载，可以操作 $el 和 $refs
  // 适合：DOM 操作（图表初始化、F第三方控件挂载）、启动定时器
  mounted() {
    console.log('DOM 已挂载', this.$el.innerHTML)
    this.initChart()      // 图表库（如 ECharts）需要 DOM 就绪
    this.startPolling()   // 启动定时轮询
  },

  // 数据变化，DOM 更新前（可以在这里修改状态，但会引发额外更新）
  beforeUpdate() {
    console.log('数据变化了，DOM 即将更新')
  },

  // DOM 更新完成（避免在这里修改状态，否则可能引发无限循环）
  updated() {
    console.log('DOM 更新完成')
    // ✅ 适合：在数据变化后基于新的 DOM 状态做后续处理
  },

  // 组件卸载前（清理副作用）
  beforeUnmount() {
    this.stopPolling()    // 清理定时器
    this.ws?.disconnect() // 断开 WebSocket
  },

  // 组件完全卸载（可以在这里做最终清理）
  unmounted() {
    console.log('组件卸载完毕')
  }
}
```

### 4.3 keep-alive 与缓存

`<keep-alive>` 是 Vue 内置的抽象组件，它将动态组件**缓存到内存中**而不是销毁，从而保留组件状态。

```vue
<!-- App.vue -->
<template>
  <div id="app">
    <nav>
      <router-link to="/alerts">告警列表</router-link>
      <router-link to="/analysis">分析</router-link>
    </nav>

    <!-- keep-alive 包裹路由视图 -->
    <keep-alive :include="['AlertList', 'AnalysisDashboard']" :max="3">
      <router-view />
    </keep-alive>
  </div>
</template>
```

**为什么需要缓存？** 在 SOC 系统中，用户经常在"告警列表"和"告警详情"之间来回切换。如果每次切换都销毁和重新创建 AlertList 组件，不仅页面会闪烁，还会导致：
- 重新请求 API（浪费带宽，增加延迟）
- 丢失用户的筛选/排序状态（体验差）
- 重新初始化图表（消耗性能）

**被缓存的组件会触发额外两个生命周期钩子：**

```javascript
// AlertList.vue
export default {
  // 组件首次挂载时触发（相当于 mounted）
  activated() {
    console.log('AlertList 被激活了')
    // ✅ 可以在这里恢复暂停的定时器、重新连接 WebSocket
    this.startPolling()
  },

  // 组件被缓存停用时触发（相当于 unmounted）
  deactivated() {
    console.log('AlertList 被缓存了（未销毁）')
    // ✅ 适合在这里暂停轮询、断开连接，节省资源
    this.stopPolling()
  }
}
```

⚠️ **注意：** 使用 `<keep-alive>` 时，如果被缓存的组件内部有**定时器或 WebSocket 连接**，一定要在 `deactivated` 中清理，在 `activated` 中恢复。否则后台的定时器会持续消耗资源。

### 4.4 setup 中的生命周期钩子

Vue 3 的 Composition API 中，生命周期钩子通过 `on` 前缀的函数调用：

```javascript
import { onMounted, onUnmounted, onActivated, onDeactivated } from 'vue'

export default {
  setup() {
    const polling = ref(null)

    onMounted(() => {
      // setup 中的 mounted
      console.log('组件已挂载')
      polling.value = setInterval(() => fetchAlerts(), 5000)
    })

    onActivated(() => {
      // keep-alive 激活时
      console.log('从缓存激活，重新开始轮询')
      polling.value = setInterval(() => fetchAlerts(), 5000)
    })

    onDeactivated(() => {
      // keep-alive 缓存时
      console.log('缓存，停止轮询节省资源')
      clearInterval(polling.value)
    })

    onUnmounted(() => {
      // 组件真正销毁时
      clearInterval(polling.value)
    })

    // 其他 setup 中可用的生命周期
    // onBeforeCreate   → 直接在 setup 中写
    // onCreated        → 直接在 setup 中写（setup 等价于 created）
    // onBeforeMount    → onBeforeMount
    // onBeforeUpdate   → onBeforeUpdate
    // onBeforeUnmount  → onBeforeUnmount
    // onErrorCaptured  → onErrorCaptured
  }
}
```

## 5. Composition API

### 5.1 为什么引入 Composition API

Composition API 是 Vue 3 引入的最重要的新特性。要理解它，首先要理解 **Options API 的逻辑分散问题**。

**Options API 的问题：** 当一个功能涉及多个关注点（比如一个告警列表组件需要：数据请求、分页逻辑、搜索筛选、图表绑定、权限判断）时，这些逻辑会**分散在不同选项**中：

```javascript
// Options API：一个功能散落在 6 个选项中
export default {
  data() {
    return {
      alerts: [],
      searchText: '',
      page: 1,
      pageSize: 20,
      chartData: null
    }
  },
  computed: {
    filteredAlerts() { /* 搜索逻辑 */ },
    totalPages() { /* 分页逻辑 */ }
  },
  methods: {
    async fetchAlerts() { /* API 请求 */ },
    search() { /* 搜索 */ },
    changePage() { /* 分页 */ },
    initChart() { /* 图表 */ }
  },
  watch: {
    searchText: 'fetchAlerts'
  },
  mounted() {
    this.fetchAlerts()
    this.initChart()
  }
}
```

如果现在要提取"告警列表"逻辑为可复用单元（hook），Options API 几乎无法优雅地做到——mixins 会有命名冲突和数据来源不清晰的问题。

**Composition API 的解决方案：** 把**同一个功能相关的所有代码**放在一起：

```javascript
// useAlertList.js - 一个自定义组合式函数（hook）
import { ref, computed, onMounted, watch } from 'vue'

export function useAlertList() {
  // 1. 所有相关状态放在一起
  const alerts = ref([])
  const searchText = ref('')
  const page = ref(1)

  // 2. 所有相关逻辑放在一起
  const filteredAlerts = computed(() =>
    alerts.value.filter(a => a.message.includes(searchText.value))
  )

  const totalPages = computed(() => Math.ceil(filteredAlerts.value.length / 20))

  async function fetchAlerts() {
    const res = await fetch(`/api/alerts?page=${page.value}`)
    alerts.value = await res.json()
  }

  watch(searchText, fetchAlerts)
  onMounted(fetchAlerts)

  // 3. 返回所有需要暴露的东西
  return { alerts, searchText, page, filteredAlerts, totalPages, fetchAlerts }
}
```

```javascript
// 组件中使用：代码量大幅减少，逻辑清晰
import { useAlertList } from './composables/useAlertList'

export default {
  setup() {
    // 一行代码引入完整功能
    const { alerts, searchText, page, totalPages, fetchAlerts } = useAlertList()
    return { alerts, searchText, page, totalPages, fetchAlerts }
  }
}
```

### 5.2 setup 函数详解

`setup` 是 Vue 3 组件的入口函数，它在组件实例创建**之前**执行，且函数中的 `this` 是 `undefined`。

```javascript
export default {
  setup(props, context) {
    // props：父组件传入的 props（响应式，不可直接修改）
    console.log(props.alertId)

    // context：包含组件的上下文（非响应式）
    // context.attrs  → $attrs（未声明的 props）
    // context.slots  → $slots
    // context.emit   → $emit
    // context.expose → 暴露公共属性（通过模板 ref 访问）

    // setup 返回的对象，可在模板中使用
    return {
      localMessage: 'Hello from setup'
    }
  }
}
```

### 5.3 ref 和 reactive

| 对比项 | ref | reactive |
|--------|-----|---------|
| 适用类型 | 基本类型（string/number/boolean） | 对象/数组 |
| 访问方式 | `.value`（JS 中必须，模板中自动解包） | 直接访问 |
| 替换整个对象 | `obj.value = newObj` | `Object.assign(obj, newObj)` |
| 解构后保留响应式 | 否（需用 toRefs） | 否（需用 toRefs） |

```javascript
import { ref, reactive, toRefs, isRef, unref } from 'vue'

// ref：适合包装基本类型
const count = ref(0)
count.value++            // JS 中必须用 .value
console.log(count.value) // 1

// reactive：适合包装对象（深层响应式）
const state = reactive({
  username: 'admin',
  permissions: ['read', 'write'],
  settings: { theme: 'dark' }
})

state.username = 'analyst'  // 直接修改，不需要 .value
state.settings.theme = 'light'

// ⚠️ 注意：reactive 存在"丢失响应式"的问题
const state2 = reactive({ count: 0 })
const { count: count2 } = state2  // ❌ count2 不再是响应式的
// ✅ 正确做法：用 toRefs
const { count: count3 } = toRefs(state2)  // count3 是 ref，保持响应式

// 工具函数
console.log(isRef(count))      // true，ref 返回 true
console.log(unref(count))      // 0，获取 ref 的内部值（如果不是 ref 也直接返回值）
```

⚠️ **注意：** 不要对 reactive 对象进行解构，解构后变量会**丢失响应式**，因为 reactive 返回的是 Proxy，分解出来的普通变量不再被代理。

```html
<!-- 模板中不需要 .value，Vue 自动解包 -->
<template>
  <p>计数器：{{ count }}</p>  <!-- ✅ 自动解包 -->
  <p>用户名：{{ username }}</p> <!-- ✅ reactive 对象的属性也自动解包 -->
</template>
```

### 5.4 computed 和 watch

```javascript
import { ref, computed, watch, watchEffect } from 'vue'

// computed：计算属性（有缓存，依赖不变不重算）
const severity = ref('critical')
const alertCount = ref(42)

// 自动追踪依赖：只有 alertCount 变化时才重新计算
const alertLabel = computed(() => {
  return `${alertCount.value} 个${severity.value === 'critical' ? '紧急' : '其他'}告警`
})

// computed 两种写法（有 get/set）
const fullName = computed({
  get: () => `${firstName.value} ${lastName.value}`,
  set: (val) => { [firstName.value, lastName.value] = val.split(' ') }
})

// watch：显式监听指定数据（懒执行，初始不执行）
watch(alertCount, (newVal, oldVal) => {
  console.log(`告警数从 ${oldVal} 变为 ${newVal}`)
  if (newVal > 100) {
    this.triggerNotification()
  }
}, {
  immediate: true,      // 立即执行一次（初始值也会触发）
  deep: true,          // 深度监听（监听对象内部属性变化）
  flush: 'post'        // 'pre' | 'post' | 'sync'，控制回调时机
})

// watchEffect：自动收集依赖（立即执行）
watchEffect(() => {
  // 自动追踪 alertCount 和 severity 的读取
  console.log(`状态：${alertCount.value} 个 ${severity.value} 告警`)
  if (alertCount.value > 0) {
    this.hasUnread = true  // 自动追踪依赖
  }
})

// 监听多个数据源
watch([alertCount, severity], ([newCount, newSeverity], [oldCount, oldSeverity]) => {
  console.log(`${oldSeverity}:${oldCount} → ${newSeverity}:${newCount}`)
})
```

**computed vs methods 的核心区别：**

```javascript
// computed 有缓存
const first = computed(() => {
  console.log('computed 执行了')  // 只在依赖变化时打印
  return this.alertList[0]
})

// methods 无缓存（每次调用都重新执行）
const firstMethod = () => {
  console.log('method 执行了')  // 每次访问都打印
  return this.alertList[0]
}

// 在模板中使用时：
{{ first }}           // 第二次访问不打印（用了缓存）
{{ firstMethod() }}    // 每次都打印（重新执行）
```

⚠️ **注意：** 如果计算属性中涉及**副作用**（如 API 调用、DOM 操作、定时器），绝对不能用 computed，应该用 watch/watchEffect。

### 5.5 toRefs 的作用

`toRefs` 将一个 reactive 对象转换为普通对象，其中每个属性都是对原对象对应属性的 ref。这解决了 reactive 对象解构后丢失响应式的问题：

```javascript
const state = reactive({ count: 0, name: 'vue' })

// ❌ 普通解构：count 和 name 变成普通值，不再响应式
const { count, name } = state
count++  // 不会触发响应式更新（修改的不是 proxy）

// ✅ 用 toRefs：每个属性都变成 ref，保持响应式
const { count, name } = toRefs(state)
count.value++  // ✅ 响应式更新（count 是 ref，修改的是 .value）
// 或者直接修改原对象
state.count++   // ✅ 也响应式（修改的是 proxy）

// toRef：单独从一个 reactive 属性创建 ref
import { toRef } from 'vue'
const count = toRef(state, 'count')  // 创建指向 state.count 的 ref
```

### 5.6 Options API vs Composition API

| 对比维度 | Options API | Composition API |
|----------|------------|----------------|
| 代码组织 | 按选项类型组织（data/methods/computed 分开） | 按功能组织（相关逻辑放一起） |
| 逻辑复用 | mixins（有命名冲突和数据来源不清晰问题） | 自定义 hook（逻辑清晰，无冲突） |
| TypeScript 支持 | 需要 class-mixins 装饰器 | 原生支持，类型推断好 |
| 代码压缩 | 较难（data/methods 名字都要保留） | 较易（可自由命名） |
| 学习曲线 | 直观，适合入门 | 需要适应，更灵活 |
| Tree-shaking | 不支持 | 支持（未使用的 API 不打包） |

## 6. Vue Router

### 6.1 两种路由模式对比

| 对比项 | hash 模式 | history 模式 |
|--------|----------|-------------|
| URL 外观 | `/#/alerts/123` | `/alerts/123` |
| 页面刷新 | 不触发（`#` 后面的变化不发送请求） | 触发（作为路径发送到服务器） |
| 服务器配置 | 无需配置 | 需要配置 fallback（所有路径指向 index.html） |
| 微信/企微分享 | 可能被截断 | 完整 URL |
| 兼容性 | IE8+ | 需要 history API（IE10+） |
| SEO | 不友好（搜索引擎通常不索引 hash） | 友好（正常 URL） |

**hash 模式原理：** 监听 `hashchange` 事件，当 `#` 后面的部分变化时，浏览器不发送请求，Vue Router 捕获变化并渲染对应组件。

```javascript
// hash 路由的核心原理
window.addEventListener('hashchange', () => {
  const hash = window.location.hash.slice(1)  // 去掉 #
  this.currentRoute = hash || '/'
  renderComponent(this.currentRoute)
})
```

**history 模式原理：** 使用 HTML5 History API（pushState/replaceState）改变 URL，同时在浏览器历史记录中插入记录。监听 `popstate` 事件处理前进/后退。

```javascript
// history 路由的核心原理
history.pushState({ page: 1 }, '', '/alerts')
// 注意：这不会触发页面刷新，只是更新 URL

window.addEventListener('popstate', (e) => {
  // 浏览器前进/后退时触发
  this.currentRoute = window.location.pathname
  renderComponent(this.currentRoute)
})
```

⚠️ **注意：** history 模式下，直接访问 `/alerts/123` 时，浏览器会向服务器发送请求 `/alerts/123`。如果服务器没有配置重定向到 `index.html`，会返回 404。所以**所有 history 模式的 SPA 项目都需要服务器配置 fallback**。

```nginx
# Nginx 配置示例
location / {
  try_files $uri $uri/ /index.html;
}
```

### 6.2 路由懒加载

路由懒加载是 Vue 项目最重要的性能优化手段之一。它将每个路由对应一个独立的 JS chunk，只有访问该路由时才下载对应 JS。

```javascript
import { createRouter, createWebHistory } from 'vue-router'

const routes = [
  // ✅ 方式1：箭头函数（最常用）
  {
    path: '/dashboard',
    component: () => import('./views/Dashboard.vue')
  },

  // ✅ 方式2：带注释的 chunk 名称（方便调试）
  {
    path: '/alerts',
    component: () => import(/* webpackChunkName: "alerts" */ './views/AlertList.vue')
  },

  // ✅ 方式3：命名视图的懒加载
  {
    path: '/alerts/:id',
    components: {
      default: () => import('./views/AlertDetail.vue'),
      sidebar: () => import('./views/AlertSidebar.vue')
    }
  },

  // ⚠️ 非懒加载（首页不推荐，但子路由可以用）
  {
    path: '/',
    component: Home  // 同步导入，首页立即可用
  }
]
```

**懒加载的效果：** 未访问的路由不会打包到首屏 JS 中。假设有 10 个路由页面，同步打包时主包 500KB；懒加载后首屏只需 80KB，用户访问时才按需加载其余页面。

### 6.3 动态路由与导航守卫

**动态路由：**

```javascript
const router = createRouter({
  routes: [
    // 静态路由
    { path: '/', component: Home },
    // 动态路由（:id 是参数）
    { path: '/alerts/:id', component: AlertDetail },
    // 可选参数
    { path: '/users/:id?', component: UserProfile }
  ]
})

// 编程式导航
router.push('/alerts/123')           // → /alerts/123
router.push({ name: 'alert-detail', params: { id: 123 } })  // 命名路由
router.replace('/alerts/456')        // 替换当前记录（不留下后退痕迹）
```

**路由参数获取：**

```javascript
// 在组件中获取路由参数
const route = useRoute()  // Vue Router 4 的组合式 API
const alertId = route.params.id

// 或 Options API
this.$route.params.id
```

**导航守卫（Navigation Guards）：**

```javascript
// 路由配置文件中定义守卫
import { createRouter, createWebHistory } from 'vue-router'
import { useAuthStore } from './stores/auth'

const router = createRouter({
  history: createWebHistory(),
  routes: [...]
})

// ⭐ 全局前置守卫（最常用，适合权限验证）
router.beforeEach(async (to, from, next) => {
  // to：即将进入的路由
  // from：从哪个路由来
  // next：决定是否放行

  // 公共路由白名单
  const publicPages = ['/login', '/403']
  const requiresAuth = !publicPages.includes(to.path)

  if (requiresAuth) {
    const authStore = useAuthStore()
    if (!authStore.isAuthenticated) {
      // 未登录，跳转到登录页，记住要去的页面
      next({ path: '/login', query: { redirect: to.fullPath } })
    } else {
      next()  // 已登录，放行
    }
  } else {
    next()  // 公共页面，直接放行
  }
})

// 全局解析守卫（在组件解析后、导航确认前调用）
router.beforeResolve(async (to) => {
  // 适合在这里做数据预取（确保所有组件的 asyncData 加载完成）
  console.log('所有组件已解析，导航即将确认')
})

// 全局后置钩子（适合做页面埋点、修改页面标题）
router.afterEach((to, from) => {
  document.title = to.meta.title || 'SOC Platform'
```

## 7. 响应式原理

### 7.1 Vue 2 的 Object.defineProperty

Vue 2 在初始化组件时，会递归遍历 data 中的所有属性，通过 `Object.defineProperty` 将每个属性转换为 getter/setter。

```javascript
// Vue 2 响应式原理的核心代码（简化版）
function defineReactive(obj, key, value) {
  // 递归：为嵌套对象也添加响应式
  let childObj = observe(value)

  Object.defineProperty(obj, key, {
    enumerable: true,      // 可枚举（可被 for...in 遍历）
    configurable: true,    // 可配置（可删除、可修改描述符）

    // getter：读取属性时触发，收集依赖（谁在读取这个值）
    get() {
      if (Dep.target) {
        // Dep.target 是当前的 Watcher（订阅者）
        // 将当前 Watcher 加入这个属性的依赖队列
        dep.addSub(Dep.target)
      }
      return value
    },

    // setter：修改属性时触发，通知所有依赖更新
    set(newValue) {
      if (newValue === value) return  // 值没变，不触发更新
      value = newValue
      // 通知所有订阅了这个属性的 Watcher
      dep.notify()
    }
  })
}

// Watcher：当数据变化时，执行 render 函数更新视图
// Dep：管理所有 Watcher，notify 时通知它们全部更新
```

**Object.defineProperty 的局限性：**

```javascript
// ❌ 无法检测新增属性
const vm = new Vue({ data: { a: 1 } })
vm.b = 2        // 新增属性 b，不是响应式的
Vue.set(vm, 'b', 2)  // ✅ 正确做法

// ❌ 无法检测删除属性
delete vm.a     // 删除后不会触发更新
Vue.delete(vm, 'a')  // ✅ 正确做法

// ❌ 无法检测数组索引赋值（Vue 2.2+ 已修复）
vm.items[0] = { message: 'changed' }  // 不响应式
Vue.set(vm.items, 0, { message: 'changed' })  // ✅ 正确做法
vm.items.splice(0, 1, { message: 'changed' })  // ✅ 也正确

// ❌ 无法检测数组长度的直接修改
vm.items.length = 0  // 不触发更新
vm.items.splice(0)   // ✅ 正确做法
```

### 7.2 Vue 3 的 Proxy

Vue 3 使用 Proxy 代理整个对象，而不是遍历每个属性设置 getter/setter：

```javascript
// Vue 3 响应式原理（简化版）
function reactive(obj) {
  return new Proxy(obj, {
    // get：读取属性时触发
    get(target, key, receiver) {
      // track：收集依赖，建立 target[key] → effect 的映射
      track(target, key)

      // Reflect 保证 this 指向正确（处理继承链问题）
      const result = Reflect.get(target, key, receiver)

      // 如果是对象，递归返回代理（深层响应式）
      if (result !== null && typeof result === 'object') {
        return reactive(result)
      }
      return result
    },

    // set：写入属性时触发
    set(target, key, newValue, receiver) {
      const oldValue = target[key]
      const result = Reflect.set(target, key, newValue, receiver)

      // 只有值真的变化了才触发
      if (oldValue !== newValue) {
        // trigger：触发所有依赖更新
        trigger(target, key, newValue)
      }
      return result
    },

    // deleteProperty：删除属性时触发
    deleteProperty(target, key) {
      const hadKey = Object.prototype.hasOwnProperty.call(target, key)
      const result = Reflect.deleteProperty(target, key)
      if (result && hadKey) {
        trigger(target, key, undefined, 'DELETE')
      }
      return result
    },

    // has：in 操作符时触发
    has(target, key) {
      track(target, key)
      return Reflect.has(target, key)
    }
  })
}
```

**Proxy 的优势：**

```javascript
// ✅ 新增属性自动响应式
const state = reactive({ a: 1 })
state.b = 2  // ✅ 响应式（Proxy 的 set 自动拦截）

// ✅ 删除属性自动响应式
delete state.a  // ✅ 响应式（Proxy 的 deleteProperty 拦截）

// ✅ 支持数组索引检测（Vue 3.3+）
const list = reactive([])
list[0] = 'item'  // ✅ 响应式

// ✅ 支持 Map/Set/WeakMap/WeakSet
const map = reactive(new Map())
map.set('key', 'value')  // ✅ 响应式
```

⚠️ **Proxy 的陷阱：**
```javascript
// ⚠️ reactive 会对原对象做深拷贝？不，只是包装了一层 Proxy
const original = { a: 1 }
const state = reactive(original)
console.log(state === original)  // false（Proxy != 原对象）

// ⚠️ 直接替换整个对象会丢失响应式
state = reactive({ a: 2 })  // ❌ state 指向了新对象，原来的 Proxy 引用丢失

// ✅ 应该修改原对象内容
Object.assign(state, { a: 2 })  // ✅ 在原 Proxy 上修改
```

### 7.3 依赖收集与触发更新

Vue 响应式系统的核心是**"发布-订阅"模式**，但这里的"订阅者"是特殊的 Watcher，与 DOM 更新直接挂钩：

```
组件 render()
    ↓ 读取数据（触发 getter）
    ↓
data.name 被读取
    ↓
Dep.target（当前 Watcher）被收集到 name 的 dep.subs 中
    ↓
dep.subs = [Watcher-A, Watcher-B]
    ↓
用户修改 name
    ↓ 触发 setter
    ↓
dep.notify()
    ↓
Watcher-A.update() → 重新 render（更新视图）
Watcher-B.update() → 重新 compute（重新计算 computed）
```

**computed 的缓存机制也基于此：** computed 的 getter 函数在计算时同样会触发 getter 并收集依赖。当依赖未变化时，computed 的 getter 不会重新执行，直接返回缓存值。

## 8. Vue 3 新特性

### 8.1 Teleport（瞬间传送到任意 DOM 位置）

Teleport 是 Vue 3 最实用的新特性之一。它解决了**组件逻辑位置和 DOM 位置不一致**的问题——组件逻辑上属于某处，但希望渲染到 DOM 的另一个位置。

**典型场景：** 模态框（Modal）。Modal 在逻辑上是某个页面的子组件（比如 `<UserSettings>`），但如果页面有 `overflow: hidden` 或复杂的层叠上下文，Modal 的 z-index 和定位会出问题。

```vue
<!-- 问题：不使用 Teleport 时 -->
<UserSettings>
  <div class="settings-panel">
    <Modal v-if="showModal">
      <!-- Modal 的 z-index 受限于 .settings-panel 的层叠上下文 -->
      <!-- 如果 .settings-panel overflow: hidden，Modal 会被裁剪 -->
    </Modal>
  </div>
</UserSettings>
```

```vue
<!-- 解决方案：Teleport 将 Modal 传送到 body 下 -->
<template>
  <UserSettings>
    <div class="settings-panel">
      <h3>用户设置</h3>
      <button @click="showModal = true">打开模态框</button>

      <!-- to="body"：Teleport 会把内部内容渲染到 <body> 下 -->
      <Teleport to="body">
        <Modal v-if="showModal" @close="showModal = false">
          <template #header>
            <h2>确认操作</h2>
          </template>
          <p>确定要执行此操作吗？</p>
          <template #footer>
            <button @click="showModal = false">取消</button>
            <button @click="confirm">确认</button>
          </template>
        </Modal>
      </Teleport>
    </div>
  </UserSettings>
</template>
```

**Teleport 可以传送到任意 CSS 选择器：**
```html
<Teleport to="#modal-container">...</Teleport>
<Teleport to="body">...</Teleport>
<Teleport :to="destination">...</Teleport>  <!-- 动态 -->
```

⚠️ **注意：** Teleport 的 `to` 目标元素必须在 Teleport 组件挂载**之前**就存在于 DOM 中，否则Teleport 的内容会被延迟渲染（直到目标元素出现）。

### 8.2 Suspense（异步组件加载占位）

`<Suspense>` 允许你在等待嵌套的异步组件加载时展示一个后备（fallback）内容。

```vue
<!-- AlertDashboard.vue -->
<template>
  <Suspense>
    <!-- 异步组件（setup 中有 async/await） -->
    <template #default>
      <AlertList />    <!-- AlertList.vue 的 setup() 是 async 函数 -->
    </template>

    <!-- 加载中显示的内容 -->
    <template #fallback>
      <div class="loading-spinner">
        <span>加载告警数据中...</span>
      </div>
    </template>
  </Suspense>
</template>
```

```vue
<!-- AlertList.vue -->
<script setup>
// setup 可以是 async 函数（Composition API 的优势）
// 这样组件会被 Suspense 识别为异步依赖
const alertList = await fetchAlertList()

return { alertList }
</script>
```

**典型应用场景：** SOC 系统中，告警列表的数据来自多个独立 API（告警主体、地理位置、用户信息），可以用 Suspense 优雅地处理加载状态，而不需要手动管理 loading 状态。

### 8.3 Fragment（多根节点组件）

Vue 2 中每个组件的模板**只能有一个根节点**：

```html
<!-- Vue 2 ❌ 错误 -->
<template>
  <div>Title</div>
  <div>Content</div>
</template>

<!-- Vue 2 ✅ 正确（必须包一层） -->
<template>
  <div>
    <div>Title</div>
    <div>Content</div>
  </div>
</template>
```

```html
<!-- Vue 3 ✅ 支持多根节点 -->
<template>
  <div class="header">告警管理</div>
  <div class="main">
    <AlertList />
  </div>
  <div class="footer">SOC Platform v3.0</div>
</template>
```

⚠️ **注意：** 多根节点组件（Fragment）**没有 $parent.$children 的确定性引用**。如果组件内部有错误，Vue 3 的错误边界需要使用 `onErrorCaptured` 来处理。

### 8.4 CSS v-bind（在 style 中直接用响应式变量）

## 9. Pinia vs Vuex

### 9.1 Pinia 的优势

Pinia 是 Vue 官方推荐的新一代状态管理库（由 Vue 核心团队成员维护），Vuex 5 的提案最终演变成了 Pinia。

| 对比项 | Vuex | Pinia |
|--------|------|-------|
| API 复杂度 | 高（需要 mutation/action/state/getter 四种概念） | 低（只有 state/getters/actions） |
| TypeScript 支持 | 需要手动装饰器或 patch | 原生 TS 支持，类型推断完美 |
| Vue DevTools | 支持（但 store 有上限） | 支持（无限 store，插件更完善） |
| 体积 | ~26KB (gzip) | ~10KB (gzip) |
| mutation | 有（Vue 2 遗留概念） | 无（Pinia 只有一个 actions） |
| 模块热更新（HMR） | 需要手动处理 | 自动支持 |
| 组合式写法 | 用起来不自然 | 专为组合式 API 设计 |

### 9.2 Pinia 核心用法

```javascript
// stores/alert.js
import { defineStore } from 'pinia'
import { ref, computed } from 'vue'

export const useAlertStore = defineStore('alert', () => {
  // ===== 1. State（用 ref 定义） =====
  const alerts = ref([])
  const loading = ref(false)
  const filter = ref('all')

  // ===== 2. Getters（用 computed 定义） =====
  const filteredAlerts = computed(() => {
    if (filter.value === 'all') return alerts.value
    return alerts.value.filter(a => a.severity === filter.value)
  })

  const criticalCount = computed(() =>
    alerts.value.filter(a => a.severity === 'critical').length
  )

  // ===== 3. Actions（用普通函数定义） =====
  async function fetchAlerts() {
    loading.value = true
    try {
      const res = await fetch('/api/alerts')
      alerts.value = await res.json()
    } finally {
      loading.value = false
    }
  }

  function acknowledgeAlert(id) {
    const alert = alerts.value.find(a => a.id === id)
    if (alert) alert.status = 'acknowledged'
  }

  return { alerts, loading, filter, filteredAlerts, criticalCount, fetchAlerts, acknowledgeAlert }
})
```

```javascript
// 组件中使用
import { storeToRefs } from 'pinia'

export default {
  setup() {
    const alertStore = useAlertStore()

    // ✅ storeToRefs：保持响应式，同时允许解构
    const { alerts, loading, criticalCount } = storeToRefs(alertStore)

    // ✅ 方法直接解构（不需要 storeToRefs）
    const { fetchAlerts, acknowledgeAlert } = alertStore

    return { alerts, loading, criticalCount, fetchAlerts, acknowledgeAlert }
  }
}
```

⚠️ **注意：** Pinia 的 store 是**自动独立的**（不需要 modules 配置）。每个 defineStore 创建一个独立的 store，跨 store 引用时直接 import 即可，不需要 Vuex 那种嵌套 modules 的复杂结构。

## 10. 性能优化

### 10.1 渲染性能：v-if / v-show / v-memo

```html
<!-- v-if vs v-show 选择决策树 -->
频繁切换？
  是 → v-show（不销毁组件，切换 CSS display）
  否 → 首次渲染条件大概率是 false？
          是 → v-if（不渲染 DOM，节省内存）
          否 → 考虑预渲染或骨架屏

<!-- v-memo：Vue 3.2+ 记忆指定依赖，依赖不变则跳过渲染 -->
<div v-for="alert in alertList" :key="alert.id" v-memo="[alert.status, alert.assignee]">
  <!-- 只有 alert.status 或 alert.assignee 变化时才重新渲染 -->
  <!-- alert.message 变化不会触发重新渲染 -->
  <AlertCard :alert="alert" />
</div>
```

### 10.2 computed vs methods

computed 会**缓存结果**，只有依赖变化时才重新计算。methods 每次调用都重新执行。

```javascript
// 大量计算时性能差异显著
const largeList = ref([...Array(10000).keys()])

// computed：大量计算只在初始化和依赖变化时执行
const filteredList = computed(() =>
  largeList.value.filter(n => n % 2 === 0)
)

// methods：每次访问都重新过滤
const getFilteredList = () =>
  largeList.value.filter(n => n % 2 === 0)

// 模板中使用：
{{ filteredList }}         // computed，快速（用了缓存）
{{ getFilteredList() }}   // methods，每次都慢（重新过滤）
```

### 10.3 路由懒加载

```javascript
// 生产环境一定要用懒加载，否则首屏包体积过大
// 首屏加载时间（FMP）直接影响用户体验和 SEO
const routes = [
  { path: '/', component: () => import('./views/Home.vue') },
  { path: '/alerts', component: () => import('./views/AlertList.vue') },
  { path: '/analysis', component: () => import('./views/Analysis.vue') }
]
```

### 10.4 keep-alive 与状态缓存

对于 SOC 系统中的复杂表单和列表页面，keep-alive 是最重要的优化手段之一：

```vue
<!-- 对 AlertList 和 AlertDetail 做缓存（最多缓存 5 个） -->
<keep-alive :include="['AlertList', 'AlertDetail']" :max="5">
  <router-view />
</keep-alive>
```

**效果：** 用户从告警列表进入详情页，再返回列表页，列表页的状态（筛选条件、分页位置、已加载数据）全部保留，不重新请求 API，不重新渲染 DOM。

### 10.5 其他常用优化手段

**合理使用 key：** 列表渲染时，始终提供唯一 `:key`（避免用 index）。当列表数据顺序可能变化时，key 是 diff 算法的关键依据。

**组件异步加载：** 对于非首屏必须的组件，使用 `defineAsyncComponent` 按需加载：

```javascript
import { defineAsyncComponent } from 'vue'
import Loading from './Loading.vue'
import Error from './Error.vue'

// 图表库（很大，只在用户打开分析页面时加载）
const HeavyChart = defineAsyncComponent({
  loader: () => import('./HeavyChart.vue'),
  loadingComponent: Loading,  // 加载中显示
  errorComponent: Error,      // 出错时显示
  delay: 200,                  // 200ms 后才显示 loading（避免闪烁）
  timeout: 10000               // 超时 10s 显示 error
})
```

**长列表优化：** 对于上百条告警的列表，使用虚拟滚动（`vue-virtual-scroller`）只渲染可见区域。假设有 10000 条告警，实际 DOM 只渲染 20 条左右，滚动时动态替换内容。

```html
<!-- 虚拟滚动示例 -->
<RecycleScroller
  class="alert-scroller"
  :items="alertList"      <!-- 10000 条 -->
  :item-size="72"         <!-- 每条 72px 高 -->
  key-field="id"
>
  <template #default="{ item }">
    <AlertCard :alert="item" />
  </template>
</RecycleScroller>
```
