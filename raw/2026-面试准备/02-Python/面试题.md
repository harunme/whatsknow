### 装饰器
```
装饰器的原理是什么？怎么实现一个带参数的装饰器？functools.wraps 的作用是什么？
追问：
- 装饰器会影响原函数的 `__name__` 和 `__doc__` 吗？
- 类装饰器和函数装饰器有什么区别？
```
**[10秒结论]** 装饰器是一个接收函数并返回新函数的函数，通过闭包在原函数执行前后注入额外逻辑；带参数的装饰器需要三层嵌套；`functools.wraps` 用于保留原函数的元数据。

**[90秒拆解]**
- **原理**：装饰器本质是高阶函数，接收被装饰函数作为参数，返回包装后的新函数
- **带参数装饰器**（三层嵌套）：
  ```python
  def retry(max_attempts=3, delay=1.0):
      def decorator(func):
          @functools.wraps(func)  # 保留原函数元数据
          def wrapper(*args, **kwargs):
              for i in range(max_attempts):
                  try:
                      return func(*args, **kwargs)
                  except Exception:
                      if i < max_attempts - 1:
                          time.sleep(delay)
              raise
          return wrapper
      return decorator
  ```
- **`functools.wraps`**：将原函数的 `__name__`、`__doc__`、`__annotations__` 等元数据拷贝到 wrapper，避免装饰后函数名变成 `wrapper`

**[20秒收尾]** FastAPI 中装饰器常用于横切关注点（日志、认证、重试），比依赖注入更适合非业务逻辑。

### 生成器与迭代器
```
生成器和迭代器的区别？yield 和 return 的区别？什么场景下用生成器比列表推导式更好？
追问：
- `StopIteration` 异常什么时候会被触发？
- 什么是生成器表达式？
```

**[10秒结论]** 迭代器是实现了 `__iter__` 和 `__next__` 的对象，生成器是用 yield 关键字懒产生值的特殊迭代器，避免一次性加载全部数据到内存。

**[90秒拆解]**
- **yield vs return**：return 结束函数并返回值，yield 暂停函数并生成一个值，函数调用者可以从暂停点继续执行
- **生成器优势**（vs 列表推导式）：
  - 内存占用 O(1)（列表推导式一次性把所有元素加载到内存）
  - 适合处理大数据流或无限序列
  - 可以表示状态ful 的计算（如斐波那契数列）
- **StopIteration**：`for` 循环会自动捕获，外部手动调用 `next()` 时在无可返回值时抛出

**[20秒收尾]** 在 ETL 场景中，处理百万级日志行时，用生成器逐行读取比 `pd.read_csv` 全量加载内存效率高 10 倍以上。

### 协程
```
asyncio 的事件循环是怎么工作的？async/await 和同步代码有什么区别？什么场景适合用协程而不是多线程？
追问：
- `asyncio.gather` 和 `asyncio.create_task` 的区别是什么？
- 为什么 asyncio 不能让 CPU 密集型任务变快？
```

**[10秒结论]** asyncio 事件循环是一个单线程调度器，通过 I/O 多路复用监听协程的 await 调用并切换执行；协程适合 I/O 密集型任务，多线程适合 CPU 密集型任务。

**[90秒拆解]**
- **事件循环原理**：`while True` 循环从任务队列取就绪协程执行，遇到 `await` 时挂起当前协程，切换执行其他协程，实现单线程高并发处理 I/O
- **async/await vs 同步**：async/await 定义协程函数，不阻塞事件循环；同步代码会阻塞整个事件循环
- **适合协程的场景**：HTTP 请求、数据库查询、Redis 操作等 I/O 等待期间 CPU 闲置的情况。**不适合**：CPU 密集计算（加解密、图像处理），会阻塞事件循环
- **`gather` vs `create_task`**：gather 等待全部完成返回结果列表；create_task 立即返回 Task 对象，不阻塞，可精细控制取消和超时

**[20秒收尾]** FastAPI 异步路由中使用 `asyncpg`（异步 PostgreSQL）而非 `psycopg2`（同步），否则会阻塞事件循环，失去异步优势。

### 并发编程
```
多线程（threading）、多进程（multiprocessing）、异步（asyncio）各自的适用场景？GIL 对多线程的影响是什么？
追问：
- GIL 在哪些情况下会释放？
- Python 3.12 对 GIL 有什么变化？
```

**[10秒结论]** asyncio 适合高并发 I/O，threading 适合阻塞 I/O 但受 GIL 限制，multiprocessing 绕过 GIL 可用多核；GIL 限制同一时刻只有一个线程执行 Python 字节码。

**[90秒拆解]**
- **三者对比**：
  - asyncio：协程，单线程，高并发 I/O，不绑多核
  - threading：OS 线程，抢占式调度，受 GIL 限制（I/O 期间 GIL 释放），适合阻塞 I/O
  - multiprocessing：进程，绕过 GIL，可用多核，适合 CPU 密集型
- **GIL 影响**：GIL 限制同一时刻只有一个线程执行 Python 字节码，但 I/O 操作、c 扩展调用时会释放 GIL，所以 threading 对 I/O 密集型仍有效
- **Python 3.12 变化**：PEP 703 引入 free-threaded CPython，可选禁用 GIL，但生态兼容性仍在建设中

**[20秒收尾]** 在 SOC 告警处理场景中，告警数据的 ETL 计算（CPU 密集）用 multiprocessing，告警发送 HTTP 通知（I/O 密集）用 asyncio。

### 面向对象
```
类变量和实例变量的区别？`__init__` 和 `__new__` 的区别？Python 的多继承是怎么实现的（MRO）？
追问：
- 什么是钻石继承问题？Python 怎么解决？
- Mixin 是什么？有什么特点？
```

**[10秒结论]** 类变量所有实例共享，实例变量每个实例独立；`__new__` 创建实例返回对象引用，`__init__` 初始化实例属性；Python 用 C3 线性化算法解决多继承的 MRO 问题。

**[90秒拆解]**
- **类变量 vs 实例变量**：类变量在 class 体内定义，所有实例共享；实例变量在 `__init__` 中通过 `self.xxx` 定义，每个实例独立
- **`__new__` vs `__init__`**：`__new__` 创建并返回实例对象（控制对象创建过程，如实现单例），`__init__` 初始化实例属性
- **MRO**：多继承时 Python 用 C3 线性化算法决定方法解析顺序，可用 `ClassName.__mro__` 查看
- **Mixin 特点**：仅提供额外方法而不实例化的基类，用于多重继承混入功能，通过组合而非继承复用代码

**[20秒收尾]** 用 `@dataclass` 装饰器代替手写 `__init__`/`__repr__`/`__eq__`，代码更简洁。

### 内存管理与垃圾回收
```
引用计数 + 循环引用检测是怎么工作的？什么情况下会导致内存泄漏？Python 有哪些常用调优参数？
追问：
- `gc.collect()` 什么时候需要手动调用？
- 什么是分代回收？三代分别存放什么对象？
```

**[10秒结论]** Python 用引用计数主要管理内存，通过 GC 模块检测循环引用；内存泄漏通常由全局字典/列表无限增长、缓存无上限、未关闭文件句柄导致。

**[90秒拆解]**
- **引用计数**：每个对象维护引用计数器，为 0 时立即释放。循环引用（A→B→A）使计数器永远不为 0，需要 GC 定期检测
- **分代回收**：Python GC 将对象分为三代（年轻代、老年代、永久代），新创建的对象在年轻代，存活越久越可能长期存在，GC 对年轻代更频繁
- **常见内存泄漏**：
  - 全局列表无限 `append`
  - `@lru_cache(maxsize=None)` 无上限缓存
  - 未关闭数据库连接或文件句柄
  - 回调函数引用大对象

**[20秒收尾]** 用 `memory_profiler` 逐行分析内存占用，用 `weakref` 处理需要引用对象但不希望阻止 GC 的场景。

### Web 框架对比
```
Flask、Django、FastAPI 各自的特点是什么？FastAPI 为什么性能好？Pydantic 在里面起了什么作用？
追问：
- FastAPI 的依赖注入是怎么实现的？
- Django ORM 和 SQLAlchemy 各适合什么场景？
```

**[10秒结论]** Flask 轻量灵活，Django 全功能覆盖，FastAPI 异步高性能 + Pydantic 自动序列化验证；FastAPI 快在 ASGI 异步架构 + Pydantic v2 Rust 加速序列化。

**[90秒拆解]**
- **Flask**：微框架，只处理请求分发的核心逻辑，ORM/表单验证等需手动集成。适合小项目和 API 原型
- **Django**：全功能框架（MVC + ORM + Admin + Auth），学习曲线陡，适合内容管理系统
- **FastAPI**：API 优先，Python 3.6+ async/await 原生支持，自动生成 OpenAPI 文档，性能接近 Node.js
- **FastAPI 快的原因**：
  - Starlette（底层）+ Uvicorn（ASGI 服务器）异步架构
  - Pydantic v2 的 `pydantic-core` 用 Rust 写验证逻辑，比纯 Python 快数十倍
  - `model_dump_json()` 使用 orjson 序列化，比标准库快 2-3 倍

**[20秒收尾]** SOC 后端用 FastAPI + SQLAlchemy + Pydantic，异步 I/O 处理告警接入，性能比 Flask 高 10 倍以上。

### 类型提示
```
为什么要用类型提示（type hint）？typing 模块里的 List、Optional、Union、Callable 怎么用？运行时有效吗？
追问：
- `TypeVar` 和 `Generic` 是什么？
- Pydantic 的 BaseModel 和 dataclass 哪个更好用？
```

**[10秒结论]** 类型提示用于静态代码分析和 IDE 自动补全，不影响运行时行为；`List`、`Optional`、`Union`、`Callable` 等用于复杂类型签名。

**[90秒拆解]**
- **为什么用类型提示**：IDE 自动补全和报错提示、mypy/pyright 静态检查发现 Bug、代码即文档
- **常用类型**：
  - `List[int]`：`list[int]`（Python 3.9+）
  - `Optional[str]`：`str | None`
  - `Union[int, str]`：`int | str`
  - `Callable[[int, str], bool]`：接收 (int, str) 返回 bool 的函数
- **运行时有效性**：默认不生效（Python 不会检查），需用 `beartype` 等库或在 CI 中集成 mypy

**[20秒收尾]** Pydantic v2 的 BaseModel 利用类型提示自动做数据验证，是类型提示在生产中价值最大的地方。

### 函数参数与可变默认参数陷阱
```
Python 函数参数有哪些类型？什么是可变默认参数陷阱？怎么避免？
追问：
- `*args` 和 `**kwargs` 有什么区别？
- `is` 和 `==` 的区别？什么场景必须用 `is`？
- Python 的浅拷贝和深拷贝有什么区别？
```

**[10秒结论]** Python 有位置参数、默认参数、`*args`、`**kwargs` 五种参数；可变默认参数（如 `def f(items=[])`）在函数定义时创建一次，后续调用共享同一个对象，是 Python 最常见陷阱之一。

**[90秒拆解]**
- **五种参数类型**：`def f(a, b=1, *args, c, **kwargs)` —— a 位置必填，b 默认参数，*args 可变位置元组，c 关键字参数，**kwargs 可变关键字字典
- **可变默认参数陷阱**：
  ```python
  def add(item, items=[]):  # 危险！
      items.append(item)
      return items
  add(1)  # [1]
  add(2)  # [1, 2] ← 不是 [2]！
  ```
  **原因**：默认列表在函数定义时创建，存在于函数对象的 `__defaults__` 中，所有调用共享同一对象
  **正确写法**：`def add(item, items=None): items = items or []`
- **`is` vs `==`**：前者比较身份（内存地址），后者比较值。`None` 判断必须用 `is`（单例）
- **浅拷贝 vs 深拷贝**：浅拷贝 `list.copy()` / `[:]` 只拷贝一层，深拷贝 `copy.deepcopy()` 递归拷贝所有嵌套对象

**[20秒收尾]** 腾讯云 SOC 项目里，AI 任务队列的参数处理用过 `*args/**kwargs` 解包请求体，可变默认参数在装饰器和工具函数里最容易踩坑。

### 元类
```
元类（metaclass）是什么？怎么用 `__new__` 和 `__init__` 控制类的创建过程？ORM 怎么用元类实现字段注册？
追问：
- `type(name, bases, attrs)` 和 `class Meta:` 的区别？
- metaclass 和 class factory 函数的区别？
```
**[10秒结论]** 元类是"类的类"，控制类的创建过程；通过 `__new__` 修改类属性字典，`__init__` 初始化类；ORM 用元类自动将类属性注册为字段映射。

**[90秒拆解]**
- **原理**：普通对象的类是 `type`，元类的类是自定义的 metaclass。`class MyClass(metaclass=MyMeta)` 时，Python 调用 `MyMeta('MyClass', (), {...})` 创建类
- **`__new__` vs `__init__`**：`__new__` 在类创建时调用，返回类对象；`__init__` 在类创建后初始化。修改类属性用 `__new__`，注册到全局表用 `__init__`
- **ORM 实现示例**：
  ```python
  class Field:
      def __init__(self, column_type):
          self.column_type = column_type

  class ModelMeta(type):
      def __new__(mcs, name, bases, attrs):
          fields = {}
          for k, v in attrs.items():
              if isinstance(v, Field):
                  fields[k] = v
          attrs['_fields'] = fields
          return super().__new__(mcs, name, bases, attrs)

  class Model(metaclass=ModelMeta):
      pass

  class User(Model):
      name = Field('VARCHAR(255)')
      age = Field('INT')
  # User._fields → {'name': Field(...), 'age': Field(...)}
  ```
  无需手动声明字段，元类自动扫描并注册
- **`type()` 直接调用**：`type('User', (Model,), {'name': Field('VARCHAR')})` 等价于 `class User(Model): name = Field('VARCHAR')`

**[20秒收尾]** 元类是 Python 最强大的元编程工具之一，Django ORM（`class Meta`）、SQLAlchemy（Declarative Base）都用元类实现 ORM 字段映射。

### 包管理
```
pip / poetry / uv 各自的特点是什么？requirements.txt 和 pyproject.toml 的区别？实际项目怎么选？
追问：
- `pip install -r requirements.txt` 和 `poetry install` 的区别？
- 依赖版本冲突怎么解决？
```
**[10秒结论]** pip 是基础包管理器，poetry 提供依赖锁定和虚拟环境管理，uv 是新一代超高速工具；`requirements.txt` 只记录包，`pyproject.toml` 是 Python 项目标准配置，可包含依赖、构建配置、元数据。

**[90秒拆解]**
- **pip**：最基础，`pip install package` 全局或虚拟环境安装；`requirements.txt` 无锁定机制，`pip freeze` 导出已安装版本
- **poetry**：用 `pyproject.toml` 管理依赖和项目元数据，自动生成 `poetry.lock` 锁定精确版本；`pyproject.toml` 是 PEP 517/518 标准，现代 Python 项目推荐
  ```toml
  [tool.poetry.dependencies]
  python = "^3.11"
  fastapi = "^0.110.0"

  [tool.poetry.group.dev.dependencies]
  pytest = "^8.0.0"
  ```
- **uv**：Rust 编写，比 pip 快 10-100 倍；支持 `uv pip install` 替代 pip，也支持 `uv venv` / `uv sync` 完整项目管理；适合需要高速 CI 的团队
- **选型建议**：
  - 小项目/脚本：`pip + requirements.txt`
  - 团队协作/库开发：poetry（锁文件 + 依赖组管理）
  - 超大 monorepo / 追求 CI 速度：uv
- **版本冲突解决**：poetry 自动求解器最可靠；pip-compile 生成严格锁文件；uv 的依赖解析比 pip 快 10x

**[20秒收尾]** 腾讯云 SOC 项目用 Poetry 管理 Python 依赖，pyproject.toml + poetry.lock 保证团队成员环境一致，CI 也用 `poetry install`。
