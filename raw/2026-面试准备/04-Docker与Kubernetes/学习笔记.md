## 1. Docker 基础概念

### 1.1 三大核心概念

Docker 的核心围绕三个概念展开：**镜像（Image）**、**容器（Container）**、**仓库（Registry）**。理解它们之间的关系是掌握 Docker 的第一步。

```
镜像（Image）     →  类的静态定义（只读模板）
容器（Container）→  对象实例（基于镜像运行，可读写）
仓库（Registry）  →  Maven/NPM 仓库，存放镜像版本
```

**镜像** 是一个分层的只读文件系统，每一层记录了对父层的增量修改。例如一个 Node.js 镜像可能在 Ubuntu 基础层之上叠加了 Node 运行时层、应用代码层。这就是 Docker 轻量级的根本原因——层与层之间共享资源。

**容器** 是镜像的运行实例。容器与容器之间彼此隔离，拥有自己的文件系统、网络、进程空间。当你 `docker run` 一个镜像时，实际上是在镜像层之上加了一层**可写层（Writable Layer）**，所有文件变更都发生在这层，而不会修改下层的镜像本身。这保证了镜像的不可变性（Immutable）。

**仓库** 是镜像的版本化存储服务，最常见的是 Docker Hub。`nginx:latest` 就是从 Docker Hub 拉取的官方 nginx 镜像。

### 1.2 Docker vs 虚拟机

| 对比维度 | Docker | 虚拟机（VM） |
|---|---|---|
| 架构 | 共享宿主机内核，容器运行在用户空间 | 每个 VM 有独立内核 |
| 启动速度 | 秒级（~100ms） | 分钟级（~1min） |
| 资源占用 | 极小（MB 级） | 较大（GB 级） |
| 隔离性 | 进程级隔离 | 硬件级隔离（更安全） |
| 性能损耗 | ~2-3%（可忽略） | ~10-20% |

**为什么 Docker 更轻量？** 关键在于"共享内核"四个字。虚拟机启动时需要引导整个操作系统内核，而容器只是宿主机上的一个进程（通过 namespace 和 cgroup 实现隔离）。这让容器在 SOC 安全监控场景下也具备实际价值——安全 Agent 容器化部署后，可以秒级启动、秒级扩缩容，完美适配威胁响应场景。

```bash
# 在容器内看到的进程 ID，在宿主机上就是同一个进程
$ docker run --rm nginx ps aux
# 容器内 PID 1 的进程，对应宿主机上某个 PID
$ docker run --rm --pid=host nginx ps aux  # 用 --pid=host 可以看到宿主机进程
```

⚠️ **注意：** 共享内核也意味着"容器逃逸"漏洞的影响面比 VM 更大。生产环境运行特权容器（`--privileged`）前务必三思，SOC 场景中涉及敏感日志采集时更需谨慎。

## 2. Docker 常用命令

### 2.1 镜像操作

```bash
# 拉取镜像（默认从 Docker Hub）
docker pull nginx:1.25-alpine
# 等价于 docker pull docker.io/library/nginx:1.25-alpine

# 查看本地镜像
docker images
docker images nginx  # 过滤

# 删除镜像（必须先删除所有相关容器）
docker rmi nginx:1.25-alpine
docker rmi $(docker images -q)  # 删除所有镜像（危险！）

# 构建镜像（-t 指定名称标签，末尾的 . 表示构建上下文）
docker build -t my-app:1.0.0 .

# 推送镜像到仓库
docker tag my-app:1.0.0 registry.example.com/soc-agent:1.0.0
docker push registry.example.com/soc-agent:1.0.0
```

### 2.2 容器操作

```bash
# 启动一个交互式容器（-it 分配伪终端）
docker run -it --rm ubuntu:22.04 /bin/bash
# --rm：容器退出后自动删除（适合临时测试）

# 后台运行容器（-d）
docker run -d --name soc-agent -p 8080:8080 \
  -e LOG_LEVEL=info \
  -v /data/soc-logs:/app/logs \
  my-agent:latest

# 查看运行中的容器
docker ps
docker ps -a  # 包含已停止的

# 查看容器内进程
docker top soc-agent

# 查看实时资源占用（类似 top）
docker stats
docker stats soc-agent --no-stream  # 单次输出，不持续

# 查看容器日志
docker logs -f soc-agent          # 跟踪日志
docker logs --tail 100 soc-agent   # 只看最后100行
docker logs --since 2026-04-10 soc-agent  # 过滤时间

# 进入运行中的容器（推荐 exec，attach 易出问题）
docker exec -it soc-agent /bin/bash
docker exec soc-agent ls /app/logs  # 不进入容器，执行单条命令

# 复制文件（容器 ↔ 宿主机）
docker cp soc-agent:/app/logs/error.log ./error.log
docker cp ./config.yaml soc-agent:/app/config/

# 停止 / 启动 / 删除容器
docker stop soc-agent
docker start soc-agent
docker rm soc-agent
docker rm -f soc-agent  # 强制删除运行中的容器
```

### 2.3 `docker run` 参数详解

`docker run` 是最常用的命令，参数众多，以下逐一拆解：

```bash
docker run [OPTIONS] IMAGE [COMMAND] [ARG...]

# 核心参数分类：
# 1. 后台与名称
-d               # Detached，后台运行
--name my-app    # 给容器指定名称（不指定则随机生成）

# 2. 端口映射（宿主机端口:容器端口）
-p 8080:8080     # 将宿主机的 8080 映射到容器的 8080
-p 127.0.0.1:8080:8080  # 仅限本地访问（安全性更高）
-p 8080-8085:8080-8085  # 端口范围映射

# 3. 卷挂载（宿主机路径:容器路径）
-v /data/logs:/app/logs:ro       # 冒号后加 ro 表示只读
-v my-volume:/app/data           # 使用命名卷
-v $(pwd)/config:/app/config     # 当前目录挂载

# 4. 环境变量
-e APP_ENV=production
-e DB_HOST=db.internal -e DB_PORT=5432
--env-file .env                  # 从文件加载多个环境变量

# 5. 资源限制（生产环境必备！）
--memory=512m           # 内存上限
--cpus=1.5               # CPU 核心数上限
--memory-swap=1g         # 内存+Swap 总上限（设为与 --memory 相同可禁用 Swap）
--pids-limit=100         # 限制容器内最大进程数

# 6. 重启策略（生产环境推荐）
--restart=always         # 容器退出后自动重启
--restart=on-failure:3   # 仅在非零退出码时重启，最多3次
--restart=unless-stopped # 除非手动停止，否则始终重启

# 7. 安全相关
--user 1000:1000         # 以非 root 用户运行
--cap-drop=ALL           # 移除所有 Linux capabilities
--read-only              # 容器文件系统设为只读
```

⚠️ **注意：** `-v` 参数如果宿主机路径不存在，Docker 会自动创建它，但目录权限可能不是你期望的（通常创建为 root 所有）。建议始终确保宿主机目录存在且权限正确。

## 3. Dockerfile 详解

### 3.1 核心指令

Dockerfile 是描述镜像构建步骤的脚本文件，每一条指令都会在镜像中产生一个层（Layer）。

```dockerfile
# ===== 基础镜像选择 =====
# 建议优先使用官方轻量镜像，减少攻击面
FROM node:20-alpine          # 生产推荐用 alpine，体积小（约 150MB）
# FROM node:20               # 普通版本（约 1.1GB），调试用

# ===== 文件复制 =====
# COPY 从构建上下文复制文件到容器
COPY package*.json /app/
COPY ./src /app/src
COPY --chown=node:node ./config /app/config  # 指定所有者

# ADD 的特殊能力（多数场景用 COPY 即可）：
# - 支持从 URL 下载文件
# - 支持自动解压 tar 文件
ADD https://example.com/app.tar.gz /app/
# 日常使用 COPY 就够了，除非你需要 URL 或自动解压功能

# ===== 运行命令 =====
# RUN 在构建时执行命令，每条 RUN 产生一个镜像层
RUN apt-get update && apt-get install -y --no-install-recommends \
    postgresql-client \
    curl \
    && rm -rf /var/lib/apt/lists/*  # 清理缓存，减小镜像体积！

# ===== 端口声明 =====
# 声明容器运行时监听的端口（仅文档作用，不实际绑定）
EXPOSE 8080
EXPOSE 8080 9090

# ===== 环境变量 =====
ENV APP_ENV=production
ENV LOG_LEVEL=info

# ===== 工作目录 =====
WORKDIR /app

# ===== 启动命令 =====
# CMD：可以被 docker run 覆盖的命令参数
CMD ["node", "server.js"]
# 上例写法是 exec 形式（推荐），会直接用 exec 系统调用
# shell 形式：CMD node server.js（会通过 /bin/sh -c 执行，不推荐）

# ENTRYPOINT：不可被轻易覆盖的入口点
# 与 CMD 配合使用，CMD 提供默认参数
ENTRYPOINT ["python", "agent.py"]
CMD ["--help"]  # 运行时 docker run agent 会执行 python agent.py --help
```

### 3.2 CMD vs ENTRYPOINT：核心区别

这是面试高频问题，理解其区别的关键是理解"谁占主导"。

```dockerfile
# 场景1：CMD 可被 docker run 覆盖
FROM ubuntu
CMD ["echo", "default message"]
# docker run image           → 输出 "default message"
# docker run image echo hi   → 输出 "hi"（覆盖了 CMD）

# 场景2：ENTRYPOINT + CMD 组合
FROM ubuntu
ENTRYPOINT ["echo"]
CMD ["default message"]
# docker run image           → 输出 "default message"
# docker run image hello      → 输出 "hello"（ENTRYPOINT 固定，CMD 被替换）
```

**实际应用场景：**
- 一个 Python 安全扫描 Agent 的 Dockerfile：
```dockerfile
FROM python:3.11-alpine
WORKDIR /app
COPY requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt
COPY . .
# ENTRYPOINT 固定为 python agent.py，CMD 提供默认扫描参数
ENTRYPOINT ["python", "agent.py"]
CMD ["--mode=scan", "--targets=/data/hosts.txt"]
# 用户可以 docker run my-agent --mode=quick 自定义参数
```

### 3.3 编写 Dockerfile 的最佳实践

```dockerfile
# 反面教材：一条 RUN 走天下（产生的层很大，且缓存无效时需要重建）
FROM ubuntu
RUN apt-get update
RUN apt-get install -y nginx
RUN apt-get install -y redis-server
RUN mkdir /app
COPY . /app
CMD ["nginx"]

# 最佳实践：合并 RUN 层 + 清理缓存 + 利用构建缓存
FROM ubuntu:22.04
# 将依赖安装和代码复制分离，利用 Docker 的层缓存
COPY requirements.txt /tmp/
RUN apt-get update && apt-get install -y --no-install-recommends \
        nginx redis-server curl \
    && rm -rf /var/lib/apt/lists/* /tmp/*  # 清理构建缓存
WORKDIR /app
COPY . .
RUN chmod +x ./entrypoint.sh
USER node  # 创建非 root 用户运行（安全加固）
ENTRYPOINT ["./entrypoint.sh"]
```

⚠️ **注意：** Docker 构建时会缓存每一条指令的结果（层缓存）。修改 Dockerfile 中间某一行后，其后的所有层都会失效。最佳实践是**把变化频繁的操作（COPY 源代码）放在 Dockerfile 末尾**，把变化少的操作（系统依赖安装）放在前面。

## 4. 多阶段构建

### 4.1 为什么需要多阶段构建？

一个真实的编译型语言项目（如 Go、C++、Rust）或前端构建场景中，最终运行时**并不需要**编译器、构建工具、测试框架。整个构建产物可能只有 50MB，但包含编译器后体积膨胀到 3GB。多阶段构建（Multi-stage Build）正是为了解决这个问题。

```dockerfile
# ===== 第一阶段：构建（Builder Stage） =====
FROM golang:1.22-alpine AS builder
WORKDIR /app
COPY . .
# 编译 Go 程序，会产生 ~50MB 的二进制文件
RUN CGO_ENABLED=0 GOOS=linux go build -o soc-agent ./cmd/agent

# ===== 第二阶段：运行（Runtime Stage） =====
# 从零开始，使用极简的基础镜像
FROM scratch
# 仅复制构建阶段的产物——编译器完全不会进入最终镜像！
COPY --from=builder /app/soc-agent /soc-agent
# 如果需要 CA 证书（Go 程序访问 HTTPS 时需要）
COPY --from=builder /etc/ssl/certs/ca-certificates.crt /etc/ssl/certs/
ENTRYPOINT ["/soc-agent"]
```

最终镜像体积：从含 Go 工具链的 ~800MB 缩减到 ~50MB，**减小了 16 倍**。

### 4.2 .dockerignore 文件

`.dockerignore` 的作用等同于 `.gitignore`——在 `docker build` 时告诉 Docker 忽略哪些文件不发送给构建上下文（Context）。如果没有这个文件，`COPY . /app` 会把整个项目目录（包括 `.git`、依赖缓存、node_modules）一股脑发送到 Docker daemon，大幅拖慢构建速度甚至造成安全问题。

```bash
# .dockerignore 示例
.git
.gitignore
*.md
node_modules
dist
.env
.env.*
vendor
tests
*.log
.DS_Store
```

⚠️ **注意：** `.dockerignore` 忽略 `node_modules` 后，`COPY . /app` 会把宿主机的 node_modules 也排除在外。如果你的应用依赖 node_modules（未使用 npm install），则需要在 Dockerfile 中执行 `RUN npm install`。

### 4.3 多阶段构建的常见模式

```dockerfile
# 模式1：前后端分离构建
# 第一阶段：构建前端
FROM node:20-alpine AS frontend-builder
WORKDIR /app
COPY package*.json ./
RUN npm ci
COPY . .
RUN npm run build

# 第二阶段：构建后端
FROM python:3.11-slim AS backend-builder
WORKDIR /app
COPY requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt
COPY . .

# 第三阶段：最终镜像（同时包含前端产物和后端代码）
FROM python:3.11-slim
COPY --from=frontend-builder /app/dist /static
COPY --from=backend-builder /app /app
CMD ["gunicorn", "--bind", "0.0.0.0:8000", "app:app"]
```

## 5. Docker 网络

### 5.1 四种网络模式

Docker 安装后会默认创建一个 `bridge` 网络，并自动为容器分配 IP。理解四种网络模式是解决容器间通信问题的基础。

```bash
# 查看 Docker 网络
docker network ls
# NETWORK ID     NAME       DRIVER    SCOPE
# abc123...      bridge     bridge    local
# def456...      host       host      local
# ghi789...      none       null      local
```

| 模式 | 驱动 | 说明 | 适用场景 |
|---|---|---|---|
| bridge | bridge | 默认模式，容器连接到 Docker 虚拟网桥 | 单主机容器通信 |
| host | host | 容器直接使用宿主机网络命名空间 | 追求极致网络性能 |
| none | null | 容器完全无网络 | 离线任务、安全隔离 |
| overlay | overlay | 跨主机容器通信（Docker Swarm） | 多节点集群 |
| macvlan | macvlan | 容器拥有独立 MAC 地址 | 需直连物理网络 |

### 5.2 默认 Bridge vs 自定义 Bridge

```bash
# 默认 bridge（Docker 自动创建）
# 容器间通过 IP 地址通信（不推荐，IP 会变）
docker run -d --name soc-db redis:alpine
docker run -it --rm --name soc-agent \
  --link soc-db:db \  # deprecated，但仍可用
  alpine sh
# 在 soc-agent 容器内：ping db（通过 --link 解析到 soc-db 的 IP）

# 自定义 bridge（推荐）：支持自动 DNS 解析
docker network create --driver bridge my-network
docker run -d --name soc-db --network my-network redis:alpine
docker run -it --rm --name soc-agent --network my-network alpine sh
# 容器内直接 ping soc-db（无需 --link！自动 DNS 解析）
```

**自定义 bridge 的优势：**
- 容器间通过容器名而非 IP 进行 DNS 解析（类似 K8s Service）
- 网络隔离，指定了 `--network` 的容器才能互相通信
- 同一自定义 bridge 下的容器共享相同的 DNS 服务器

### 5.3 容器间通信原理

```bash
# 查看容器的网络详情（查看 IP、网关、DNS 等）
docker inspect soc-db --format '{{json.NetworkSettings.Networks}}' | python -m json.tool

# 典型输出：
# {
#   "my-network": {
#       "IPAddress": "172.18.0.2/16",
#       "Gateway": "172.18.0.1",
#       "DNS": ["172.18.0.1"],
#       "MacAddress": "02:42:ac:12:00:02"
#   }
# }

# 查看 bridge 网络详情
docker network inspect bridge
```

容器加入同一个 bridge 网络后，Docker daemon 充当 DNS 服务器（监听 172.17.0.1:53），所有容器通过容器名进行域名解析。如果找不到匹配的容器名，才会转发到外部 DNS（如 8.8.8.8）。

⚠️ **注意：** `host` 模式会跳过 Docker 的虚拟网络栈，容器直接使用宿主机网络。这意味着端口冲突风险增大（宿主机已有服务占用的端口无法再用），同时容器内进程对宿主机网络完全可见，安全风险上升。SOC 安全 Agent 如果需要抓包分析流量，建议使用 `host` 模式（方便直接访问网卡），但要做好网络隔离措施。

## 6. 数据持久化

### 6.1 三种挂载类型

Docker 提供了三种将宿主机存储暴露给容器的方式，理解它们的生命周期差异至关重要。

```bash
# 1. Volume（命名卷）：Docker 完全管理，最安全可靠
docker volume create soc-data
docker run -d --name soc-agent -v soc-data:/app/data my-agent
# 存储位置（macOS）：~/Library/Containers/com.docker.docker/Data/vms/...
# 存储位置（Linux）：/var/lib/docker/volumes/

# 2. Bind Mount（绑定挂载）：直接映射宿主机目录
docker run -d --name soc-agent \
  -v /Users/binwang/soc/logs:/app/logs \
  -v $(pwd)/config:/app/config:ro \  # 只读保护
  my-agent
# 宿主机路径完全可控，但权限取决于宿主机目录

# 3. tmpfs Mount：存储在内存中，容器停止即消失
docker run -d --name soc-agent \
  --tmpfs /app/cache:rw,noexec,size=256m \
  my-agent
# 适用于临时敏感数据（如处理中的密钥），完全不上磁盘
```

### 6.2 三种挂载的生命周期对比

| 维度 | Volume | Bind Mount | tmpfs Mount |
|---|---|---|---|
| 数据持久性 | 持久（容器删除后仍保留） | 持久（宿主机目录） | 非持久（内存，容器停即删） |
| 存储位置 | Docker 管理 | 宿主机任意位置 | 内存 |
| 性能 | 良好 | 最优（直接磁盘） | 最快（内存） |
| 备份方式 | `docker run --rm -v my-vol:/src alpine tar ...` | 常规文件备份 | 不适用 |
| 权限控制 | Docker 内部管理 | 依赖宿主机 | 容器内完全控制 |
| 典型用途 | 数据库数据、日志聚合 | 源代码热更新、开发调试 | 临时缓存、密钥处理 |

```bash
# 管理 Volume 的常用命令
docker volume ls
docker volume inspect soc-data
docker volume rm soc-data        # 删除前需确保无容器使用
docker volume prune             # 清理所有未使用的卷
```

⚠️ **注意：** Bind Mount 的一个常见坑是**宿主机目录不存在时 Docker 会创建它并赋予 root 权限**，导致应用因权限不足无法写入。建议在 bind mount 宿主机目录前确保该目录存在且权限正确（`chmod 755` 或由 root 创建）。

### 6.3 生产环境数据管理建议

在 SOC 系统中，日志采集 Agent 产生的告警数据必须持久化，不能因容器重启丢失。以下是推荐的存储策略：

```bash
# 推荐：命名卷存储告警数据，Bind Mount 存储配置文件
docker run -d \
  --name soc-logger \
  --restart=always \
  -v soc-alerts:/app/alerts \                   # 命名卷：告警持久化
  -v /data/soc/config/logger.yaml:/app/config/logger.yaml:ro \
  -v /data/soc/certs:/app/certs:ro \
  --memory=512m --memory-swap=512m \            # 限制内存，禁止 Swap 保护敏感数据
  --read-only \                                 # 文件系统只读
  --tmpfs /app/tmp:rw,noexec,nosuid,size=64m \ # 临时文件用内存
  soc-logger:latest
```

## 7. Kubernetes 基础

### 7.1 核心资源对象

Kubernetes（K8s）是一个容器编排平台，其核心是**声明式 API**：你声明期望状态，K8s 负责达成并维持该状态。理解各资源对象之间的关系至关重要。

```
                    ┌─────────────────────────┐
                    │     Cluster             │
                    │                         │
  ┌─────────────┐   │  ┌─────────────────┐   │
  │   ConfigMap │   │  │   Namespace     │   │
  │  (配置)      │   │  │   (命名空间)    │   │
  └─────────────┘   │  └────────┬────────┘   │
                     │           │             │
                     │  ┌────────▼────────┐   │
                     │  │   Deployment    │   │
                     │  │  (应用定义)      │   │
                     │  └────────┬────────┘   │
                     │           │             │
                     │  ┌────────▼────────┐   │
                     │  │      ReplicaSet  │   │
                     │  └────────┬────────┘   │
                     │           │             │
                     │  ┌────────▼────────┐   │
                     │  │      Pod        │   │
                     │  │  (最小调度单位)   │   │
                     │  └─────────────────┘   │
                     │                         │
                     │  ┌─────────────────┐   │
                     │  │     Service     │   │
                     │  │  (服务发现/负载)  │   │
                     │  └─────────────────┘   │
                     │                         │
                     │  ┌─────────────────┐   │
                     │  │    Ingress      │   │
                     │  │  (HTTP 路由)    │   │
                     │  └─────────────────┘   │
                     └─────────────────────────┘
```

**Pod** 是 K8s 的最小调度单位。一个 Pod 里可以运行一个或多个容器（通常是一个），这些容器共享同一个网络命名空间（Pod 内的容器可以用 localhost 互相访问）和存储卷。Pod 是临时性的——一旦 Pod 被删除，其 IP 和存储都会丢失。

**Deployment** 管理 Pod 的期望副本数，并负责应用的更新和回滚。Deployment 会自动创建和管理 ReplicaSet，ReplSet 确保指定数量的 Pod 实例始终在运行。

**Service** 为一组 Pod 提供稳定的访问入口（ClusterIP），解决 Pod IP 不固定的问题。Service 通过标签选择器（Label Selector）找到后端 Pod，自动做负载均衡。

**Ingress** 管理 HTTP/HTTPS 路由规则，将外部请求路由到集群内的 Service。

**ConfigMap** 存储非敏感的配置数据（如 JSON、YAML、环境变量），Pod 可以在启动时挂载这些配置。**⚠️ 注意：敏感数据（如密码、API Key）请使用 Secret**，Secret 有 Base64 编码和更严格的访问控制（RBAC）。

**PV（PersistentVolume）** 是集群级别的存储资源，由管理员预分配或动态provisioning。**PVC（PersistentVolumeClaim）** 是 Pod 对存储的请求，Pod 通过 PVC 挂载到具体路径。

### 7.2 常用资源 YAML 示例

```yaml
# Deployment 示例：SOC 安全 Agent

metadata:
  name: soc-agent
  namespace: soc-prod

    app: soc-agent
    tier: security

  template:
```

## 9. K8s 探针机制

### 9.1 三种探针详解

健康检查是 K8s 生产级运维的核心。K8s 提供了三种探针，每种有不同用途：

```yaml
spec:
  containers:
  - name: soc-agent
    image: registry.example.com/soc-agent:1.0.0
    # 启动探针（startupProbe）：容器启动完成后才开启其他探针
    # 用途：给慢启动的应用（如需要加载大模型的安全 Agent）充足启动时间
    startupProbe:
      httpGet:
        path: /health/startup
        port: 8080
      initialDelaySeconds: 10    # 启动后等待 10 秒再开始检测
      periodSeconds: 5            # 每 5 秒检测一次
      failureThreshold: 30       # 连续失败 30 次（约 150 秒）则判定启动失败
      # 如果 startupProbe 失败，K8s 会 kill 容器并重启

    # 存活探针（livenessProbe）：判断容器是否存活，需重启时重启
    # 用途：检测应用内部死锁、OOM、hang 住的情况
    livenessProbe:
      httpGet:
        path: /health/live
        port: 8080
      initialDelaySeconds: 15
      periodSeconds: 10
      failureThreshold: 3        # 连续 3 次失败则重启容器
      successThreshold: 1        # 成功阈值（liveness 必须为 1）

    # 就绪探针（readinessProbe）：判断容器是否准备好接收流量
    # 用途：优雅上线——模型未加载完、大量请求堆积时暂不加入 Service 负载均衡
    readinessProbe:
      httpGet:
        path: /health/ready
        port: 8080
      initialDelaySeconds: 5
      periodSeconds: 5
      failureThreshold: 3
      successThreshold: 1
      # 注意：readinessProbe 失败不会重启容器，只会将 Pod 从 Service 端点中移除
```

⚠️ **注意：startupProbe 与 livenessProbe 不要同时设置相同的 initialDelaySeconds。** startupProbe 的设计就是为了解决"initialDelaySeconds"在 livenessProbe 中的两难处境：如果设置太长，会延迟检测真正的故障；如果设置太短，容器还没启动完成就被判定为失败。

### 9.2 三种检测方式

```yaml
# HTTP GET：应用有 HTTP 服务器时最常用
livenessProbe:
  httpGet:
    path: /healthz
    port: 8080
    httpHeaders:          # 可添加自定义 Header
    - name: X-Custom-Header
      value: "soc-agent"

# TCP Socket：适用于非 HTTP 协议（Redis、MySQL、gRPC）
readinessProbe:
  tcpSocket:
    port: 6379

# Exec：在容器内执行命令，检查命令退出码
livenessProbe:
  exec:
    command:
    - sh
    - -c
    - "redis-cli ping | grep PONG"
```

⚠️ **注意：探针的 failureThreshold 和 periodSeconds 共同决定了最大检测周期。** 例如 `periodSeconds=10, failureThreshold=3` 意味着：如果容器出现问题，最多需要 30 秒才能触发重启。在 SOC 告警系统中，如果对实时性要求极高，可以将 periodSeconds 设为 2-3 秒，但要注意这会增加 API server 的压力。

## 10. 工作负载：扩缩容与更新策略

### 10.1 Deployment 扩缩容

```bash
# 手动扩缩容（临时调整）
kubectl scale deployment soc-agent --replicas=10 -n soc-prod

# 滚动更新（RollingUpdate 策略）
# 这是 Deployment 的默认更新策略
kubectl set image deployment/soc-agent agent=registry.example.com/soc-agent:1.1.0 -n soc-prod
# 等价于修改镜像标签后 kubectl apply

# 查看滚动更新状态
kubectl rollout status deployment/soc-agent -n soc-prod
# 输出示例：
# Waiting for deployment "soc-agent" rollout to finish: 1 out of 3 new replicas have been updated...
# Waiting for deployment "soc-agent" rollout to finish: 2 out of 3 new replicas have been updated...
# deployment "soc-agent" successfully rolled out

# 回滚到上一版本
kubectl rollout undo deployment/soc-agent -n soc-prod

# 回滚到指定版本
kubectl rollout history deployment/soc-agent -n soc-prod
# revision 1  -> 1.0.0
# revision 2  -> 1.1.0（有问题）
kubectl rollout undo deployment/soc-agent --to-revision=1 -n soc-prod

# 暂停/恢复滚动更新（适合分阶段更新）
kubectl rollout pause deployment/soc-agent -n soc-prod
kubectl apply -f soc-agent-v2.yaml  # 多次修改
kubectl rollout resume deployment/soc-agent -n soc-prod
```

### 10.2 滚动更新的底层机制

理解滚动更新的关键参数，可以避免更新过程中服务中断：

```yaml
spec:
  strategy:
    type: RollingUpdate
    rollingUpdate:
      maxSurge: 1        # 最多同时运行 4 个 Pod（3+1）
      maxUnavailable: 0  # 滚动期间不允许有 Pod 不可用
      # 配置效果：先新增 1 个新 Pod，健康检查通过后再终止 1 个旧 Pod
      # 最终状态：4 个 Pod 短暂并存，然后稳定在 3 个
```

**如何选择 maxSurge 和 maxUnavailable：**

| 场景 | maxSurge | maxUnavailable | 效果 |
|---|---|---|---|
| 高可用优先（生产推荐） | 1 | 0 | 资源多 33%，但始终全量服务 |
| 资源优先 | 0 | 1 | 资源最小，但短暂降级 |
| 快速迭代 | 25% | 25% | 百分比写法，副本多时更灵活 |

⚠️ **注意：** 如果你的 SOC Agent 容器需要加载大型 ML 模型（启动耗时 30-60 秒），务必确保 `startupProbe` 的 `failureThreshold × periodSeconds` > 模型加载时间，否则滚动更新时新 Pod 会反复被 K8s kill，导致更新永远无法完成。

### 10.3 自动扩缩容（HPA）

```bash
# 部署 HorizontalPodAutoscaler
kubectl autoscale deployment soc-agent \
  --cpu-percent=70 \
  --min=2 --max=10 \
  -n soc-prod

# 查看 HPA 状态

```

### 10.4 Pod 调度与资源管理

```yaml
# 节点亲和性（Node Affinity）：控制 Pod 调度到哪些节点
spec:
  affinity:
    nodeAffinity:
      # 硬性要求：必须调度到有 SSD 的节点
      requiredDuringSchedulingIgnoredDuringExecution:
        nodeSelectorTerms:
        - matchExpressions:
          - key: disk-type
            operator: In
            values:
            - ssd
      # 软性偏好：优先调度到高内存节点
      preferredDuringSchedulingIgnoredDuringExecution:
      - weight: 1
        preference:
          matchExpressions:
          - key: memory-size
            operator: Gt
            values:
            - "16"

# Pod 反亲和性（Anti-Affinity）：将多个副本调度到不同节点（高可用）
    affinity:
      podAntiAffinity:
        requiredDuringSchedulingIgnoredDuringExecution:
        - labelSelector:
          - matchExpressions:
            - key: app
              operator: In
              values:
              - soc-agent
          topologyKey: kubernetes.io/hostname
          # topologyKey: "kubernetes.io/hostname" 表示不同节点
          # 即：绝不把两个 soc-agent Pod 调度到同一节点
```

## 附录：面试高频问题速记

| 问题 | 核心答案 |
|---|---|
| Docker 和 VM 的本质区别？ | Docker 共享宿主机内核（进程级隔离），VM 有独立内核（硬件级隔离） |
| 容器为何轻量？ | 分层镜像 + 共享宿主机内核，无需启动完整 OS |
| CMD 和 ENTRYPOINT 的区别？ | CMD 可被 `docker run` 参数覆盖，ENTRYPOINT 固定为执行入口 |
| 多阶段构建的目的是什么？ | 分离构建环境和运行环境，减小最终镜像体积 |
| Docker 三种网络模式的区别？ | bridge（默认，隔离）、host（共享宿主机网络）、none（无网络） |
| Volume vs Bind Mount？ | Volume 由 Docker 管理（安全可移植），Bind Mount 映射宿主机目录（灵活可控） |
| K8s 中 Pod 和 Container 的关系？ | Pod 是最小调度单位，一个 Pod 可包含多个共享网络/存储的容器 |
| livenessProbe vs readinessProbe？ | livenessProbe 失败 → 重启容器；readinessProbe 失败 → 从 Service 摘除流量 |
| 滚动更新 maxSurge vs maxUnavailable？ | maxSurge 控制最多多出几个 Pod，maxUnavailable 控制最多少几个 Pod |
| 什么是声明式 API？ | 你声明期望状态（YAML），K8s 负责持续调谐到该状态，无需告知具体操作步骤 |
