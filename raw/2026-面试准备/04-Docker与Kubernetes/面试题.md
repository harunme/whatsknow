### 核心概念

```
容器和虚拟机的区别是什么？Dockerfile 里的 FROM、RUN、COPY、EXPOSE、CMD、ENTRYPOINT 各自的作用和区别？
追问：
- 容器内 root 和宿主机 root 是一样的吗？
- `docker run` 和 `docker start` 有什么区别？
```
**[10秒结论]** 容器共享宿主机内核，轻量（MB 级）启动快（毫秒级）；VM 有独立内核，重（GB 级）启动慢（分钟级）；Dockerfile 中 CMD 可被覆盖，ENTRYPOINT 不可变。

**[90秒拆解]**
- **容器 vs VM**：容器基于 Linux Namespace（隔离进程）和 Cgroup（限制资源），共享宿主机内核；VM 通过 Hypervisor 虚拟化硬件，每台 VM 有独立 OS
- **Dockerfile 指令**：
  - `FROM`：指定基础镜像
  - `RUN`：构建时执行命令
  - `COPY`：复制文件到镜像（优先用）
  - `ADD`：复制文件，可自动解压 tar 文件和从 URL 下载（COPY 更明确）
  - `EXPOSE`：声明端口（文档作用，不实际绑定）
  - `CMD`：容器启动默认命令，可被 `docker run` 参数覆盖
  - `ENTRYPOINT`：固定入口，`docker run` 参数追加到其后面

**[20秒收尾]** `CMD` vs `ENTRYPOINT`：CMD 定义默认参数，ENTRYPOINT 定义不可变的入口。推荐写法是两者配合：`ENTRYPOINT ["python", "app.py"]` + `CMD ["--port", "8080"]`。

### 容器隔离原理
```
Docker 的 Namespace 和 Cgroup 是什么？它们分别负责什么？容器和宿主机共享内核会带来哪些安全风险？
追问：
- 6 种 Namespace 分别隔离什么资源？
- Cgroup 的 cpu、memory、blkio 控制器各自的作用？
- 容器逃逸是怎么发生的？
```
**[10秒结论]** Namespace 负责隔离（进程、网络、挂载等资源），Cgroup 负责限制（CPU、内存、I/O）；两者共同实现容器隔离，但共享内核导致容器不是完全隔离的安全边界。

**[90秒拆解]**
- **Namespace（隔离）**：Linux 内核提供 6 种命名空间，容器内进程以为自己独享整套资源
  - `PID Namespace`：容器内进程 ID 从 1 开始，宿主机看到的是另一个 PID
  - `Network Namespace`：独立网卡、IP 地址、路由表、iptables 规则
  - `Mount Namespace`：独立文件系统视图，`/proc` 等特殊挂载隔离
  - `UTS Namespace`：独立主机名和域名
  - `IPC Namespace`：独立信号量、共享内存、消息队列
  - `User Namespace`：用户/组 ID 映射，容器内 root 映射为宿主机普通用户
- **Cgroup（限制）**：
  - `cpu`：控制 CPU 时间片分配（如 `--cpus=0.5` 限制 0.5 核）
  - `memory`：限制内存使用量 + swap，超限触发 OOM Killer
  - `blkio`：限制块设备 I/O 带宽（`--device-read-bps`）
  - `pids`：限制最大进程数（防止 fork 炸弹）
- **共享内核风险**：
  - 内核漏洞可导致容器逃逸（Dirty COW、CVE-2022-0847）
  - 容器内 root 通过 User Namespace 映射为普通用户，但内核 syscall 仍以真实权限执行
  - 解决方案：Seccomp 过滤危险 syscall、AppArmor/Selinux 强制访问控制、非 root 运行、最小权限原则

**[20秒收尾]** 理解 Namespace 和 Cgroup 才能理解为什么容器比 VM 轻量（共享内核）但也比 VM 隔离性弱——容器逃逸是真实威胁，生产环境必须用最小权限 + 内核安全加固。

### 镜像优化

```
怎么减小 Docker 镜像体积？多阶段构建（multi-stage build）是什么？.dockerignore 怎么写？
追问：
- `COPY` 和 `ADD` 的区别？
- 为什么合并 RUN 指令能减小镜像体积？
```
**[10秒结论]** 多阶段构建分离构建和运行环境，只复制最终产物到精简镜像；合并 RUN 减少层数；.dockerignore 排除不需要文件。

**[90秒拆解]**
- **多阶段构建**：
  ```dockerfile
  # Stage 1: 构建
  FROM golang:1.22 AS builder
  WORKDIR /app
  RUN go build -o myapp

  # Stage 2: 运行（极简镜像）
  FROM alpine:3.19
  COPY --from=builder /app/myapp .
  CMD ["./myapp"]
  ```
  最终镜像只有 alpine + 二进制文件，不含源码和构建工具
- **COPY vs ADD**：COPY 是纯文件复制，推荐使用；ADD 额外支持自动解压 tar 和 URL 下载（容易产生意外行为）
- **减小体积原则**：轻量基础镜像（alpine/distroless）、合并 RUN 减少层数、每条 RUN 后清理缓存（`apt-get clean && rm -rf /var/lib/apt/lists/*`）

**[20秒收尾]** 前端 React 应用用 `node:20-alpine` 构建，`nginx:alpine` 运行，最终镜像控制在 30MB 以内。

### 网络

```
bridge、host、none、overlay 四种网络模式的区别？容器间是怎么通信的？Docker Compose 里的 networks 怎么配置？
追问：
- 容器如何访问外网？
- 自定义 bridge 网络和默认 bridge 有什么区别？
```
**[10秒结论]** bridge（默认，网桥隔离）、host（共享宿主机网络栈）、none（禁用网络）、overlay（跨主机）；自定义 bridge 有内置 DNS，容器间可用名字直接访问。

**[90秒拆解]**
- **四种模式**：
  - bridge（默认）：连 docker0 网桥，NAT 通信，需 `-p` 映射端口
  - host：直接用宿主机网络栈，性能好但端口冲突，不隔离
  - none：只有 loopback，无网络
  - overlay：Docker Swarm 跨主机容器通信，VXLAN 隧道
- **容器间通信**：同 bridge 网络下，Docker 内置 DNS 解析容器名到 IP，直接用服务名作为 hostname
- **Docker Compose networks**：`networks` 定义隔离网络，`frontend` 和 `backend` 网络隔离，web 只能访问 db（通过 service 名）

**[20秒收尾]** 生产环境中不同服务用不同 bridge 网络隔离，数据库不暴露到外网，只允许应用容器访问。

### 数据持久化

```
Volume、Bind Mount、tmpfs Mount 三种挂载方式的区别？数据容器（data container）现在还推荐用吗？
追问：
- 怎么把宿主机文件复制到容器？
- 容器删除后 Volume 数据还在吗？
```
**[10秒结论]** Volume（Docker 管理）、Bind Mount（宿主机路径）、tmpfs（内存，容器消失数据消失）；数据容器模式已被 Volume 替代。

**[90秒拆解]**
- **三种挂载方式**：
  - Volume：存储在 `/var/lib/docker/volumes`，Docker 管理，独立于容器生命周期
  - Bind Mount：映射宿主机任意路径，依赖宿主机文件存在
  - tmpfs：存储在内存，容器停止即消失，适合存密码/token 等敏感数据
- **数据容器已不推荐**：用 `docker run --volumes-from data_container` 的方式已被 docker volume 替代
- **容器删除后 Volume 是否保留**：`docker rm` 不删除 Volume，需单独 `docker volume rm` 删除；命名 Volume 可通过 `docker volume create` 预创建

**[20秒收尾]** MySQL/PostgreSQL 等数据库必须用 Volume 持久化数据，临时文件用 tmpfs，性能和安全性都更好。

### Docker Compose

```
docker-compose.yml 里 version、services、volumes、networks、depends_on 怎么用？profiles 是什么？重启策略有哪些？
追问：
- `condition: service_healthy` 是什么？
- docker-compose 怎么实现滚动更新？
```
**[10秒结论]** docker-compose.yml 定义多容器服务编排；`depends_on` 控制启动顺序，`condition: service_healthy` 真正等待下游服务就绪；profiles 控制服务按需启动。

**[90秒拆解]**
- **核心字段**：
  - `services`：定义各容器配置
  - `volumes`/`networks`：共享资源定义
  - `depends_on`：`condition: service_healthy` 等待健康检查通过后再启动依赖方
  - `profiles`：用 `--profile` 参数控制服务是否启动（如开发工具不在线上启动）
- **重启策略**：`no`（永不重启）、`always`（总是）、`on-failure[:n]`（失败时重启 n 次）、`unless-stopped`（手动停止后不重启）
- **滚动更新**：Docker Compose 不内置滚动更新，需用 `docker-compose up -d --no-deps --build -p 2 service && docker-compose up -d --no-deps --scale service=2` 手动实现

**[20秒收尾]** `depends_on: condition: service_healthy` 是生产级配置关键实践，确保下游服务真正就绪后才启动依赖方。

### 安全

```
容器内 root 和宿主机 root 有什么区别？--user 怎么限制权限？Seccomp、AppArmor、capabilities 是什么？镜像签名怎么做的？
追问：
- `--privileged` 模式有什么风险？
- 只读文件系统（--read-only）怎么用？
```
**[10秒结论]** 容器内 root 是宿主机 user namespace 映射后的虚拟 UID，不是真正的 root；用 `--user` 限制权限；Seccomp/AppArmor/capabilities 是 Linux 内核安全加固三件套。

**[90秒拆解]**
- **容器 root ≠ 宿主机 root**：容器内的 root 在宿主机上是普通用户（如 UID 100000），capabilities 受限（不能 mount）；`--privileged` 关闭所有安全限制，生产禁止使用
- **限制权限**：
  ```bash
  docker run --user 1000:1000 nginx  # 指定运行用户
  docker run --read-only --tmpfs /tmp nginx  # 只读文件系统 + 临时文件用内存
  ```
- **三大安全内核特性**：
  - Seccomp：过滤系统调用，Docker 默认禁用 ~44 个危险 syscall
  - AppArmor：程序轮廓限制，限制能访问的文件/网络资源
  - Capabilities：将 root 权限拆分为多个 Capability，按需授予（如 `NET_BIND_SERVICE`）

**[20秒收尾]** 生产最佳实践：非 root 用户运行、最小权限、只读根文件系统 + tmpfs、禁用 privileged 模式。

### K8s vs Docker Swarm

```
为什么需要 Kubernetes？Pod、Deployment、Service、Ingress、ConfigMap、PV/PVC 各自的概念？Docker Compose 能替代 K8s 吗？
追问：
- Pod 内多个容器共享什么命名空间？
- 什么是 Init Container？
```
**[10秒结论]** K8s 是生产级容器编排平台，提供自愈（Pod 自动重启）、扩缩容（自动 HA）、服务发现（DNS）；Docker Compose 只适合开发测试。

**[90秒拆解]**
- **核心对象**：
  - Pod：最小调度单元，可包含多个共享 network/pid/ipc 命名空间的容器
  - Deployment：管理 Pod 副本数，提供滚动更新和回滚
  - Service：固定访问入口（ClusterIP/NodePort/LoadBalancer）
  - Ingress：HTTP/S 入口路由（nginx/Traefik）
  - ConfigMap：非敏感配置；Secret：敏感数据
  - PV/PVC：持久化存储抽象
- **Compose vs K8s**：Compose 无自愈能力，K8s 自动重启失败的 Pod；Compose 无自动扩缩容，K8s 有 HPA；生产环境必须用 K8s
- **Init Container**：Pod 内最先执行的容器，用于等待依赖服务就绪或执行初始化任务

**[20秒收尾]** K8s 1.24 后不再内置 Dockershim，通过 containerd 或 cri-dockerd 对接 Docker。实际运行时环境用 containerd 性能更好。

### 排错与生产

```
容器起不来、网络不通、存储挂不上怎么排查？日志怎么收集？健康检查（HEALTHCHECK）怎么写？
追问：
- `docker logs` 和容器内应用日志有什么区别？
- 怎么查看容器的实时资源占用？
```
**[10秒结论]** 容器问题用 `docker logs` 查日志 + `docker exec` 交互调试；网络用 `docker network inspect`；日志用 ELK/Loki 收集；HEALTHCHECK 写在内层脚本内。

**[90秒拆解]**
- **容器起不来**：`docker ps -a` 看状态 → `docker logs` 查日志 → `docker inspect` 查详细信息 → `docker run -it --rm image /bin/sh` 交互调试
  - 常见原因：端口冲突、健康检查失败、权限不足（挂载卷）
- **网络不通**：`docker exec container ping target` 测试连通性 → `docker exec nslookup` 查 DNS → `docker network inspect` 查网络配置
- **存储挂不上**：`docker volume inspect` 查 Volume 状态 → 检查 mountpath 是否存在
- **日志收集**：生产用 Filebeat/Fluentd 收集 `docker logs` 输出到 Elasticsearch；或配置 json-file 日志驱动直接写 ES
- **HEALTHCHECK**：
  ```dockerfile
  HEALTHCHECK --interval=30s --timeout=3s --retries=3 \
    CMD curl -f http://localhost:8080/health || exit 1
  ```

**[20秒收尾]** K8s 排错：`kubectl get pods` → `kubectl describe pod` → `kubectl logs --previous`（查看崩溃前日志）。

### [初] Docker 基本概念与命令

**核心概念**：
- 镜像（Image）：只读模板，`python:3.11-slim` 是官方镜像
- 容器（Container）：镜像的运行实例，可写可删
- Dockerfile：构建镜像的脚本

**基础命令速查**：
```bash
# 镜像操作
docker pull python:3.11-slim
docker build -t myapp:1.0 .
docker images
docker rmi myapp:1.0

# 容器操作
docker run -d -p 8080:8000 --name myapp myapp:1.0
# -d 后台运行，-p 端口映射，--name 命名
docker ps / docker ps -a  # 运行中 / 所有容器
docker stop/start/restart myapp
docker rm myapp  # 删除容器
docker logs -f --tail 100 myapp  # 查看日志
docker exec -it myapp /bin/bash  # 进入容器
docker stats  # 实时资源占用
```

### [中] Dockerfile 核心指令与镜像优化

```dockerfile
# 构建阶段
FROM node:20-alpine AS builder

# 运行阶段

COPY --from=builder /app/dist /usr/share/nginx/html
COPY nginx.conf /etc/nginx/conf.d/default.conf
EXPOSE 80
CMD ["nginx", "-g", "daemon off;"]
```

```
node_modules/
.git/
.env

tests/
__pycache__/
```

### [中] Docker 网络与存储

**网络模式**：bridge（默认）/ host / none / overlay

**三种挂载**：
```bash
# Volume（推荐）
docker run -v my-data:/var/lib/postgres/data postgres:15

# Bind Mount（开发热更新）
docker run -v $(pwd)/src:/app/src nginx

# tmpfs（敏感数据）
docker run --tmpfs /run/secrets nginx
```

### [高] Kubernetes 核心对象

**核心 YAML**：
```yaml
# Deployment
apiVersion: apps/v1
kind: Deployment
metadata:
  name: my-app
spec:
  replicas: 3
  selector:
    matchLabels:
      app: my-app
  template:
    metadata:
      labels:
        app: my-app
    spec:
      containers:
        - name: app
          image: myapp:v1
          ports:
            - containerPort: 8080
          readinessProbe:
            httpGet:
              path: /health
              port: 8080
          livenessProbe:
            tcpSocket:
              port: 8080

---
# Service
apiVersion: v1
kind: Service
metadata:
  name: my-app-svc
spec:
  selector:
    app: my-app
  ports:
    - port: 80
      targetPort: 8080
  type: ClusterIP  # ClusterIP / NodePort / LoadBalancer
```

**探针区别**：
- `readinessProbe`：就绪探针，流量才进入 Pod
- `livenessProbe`：存活探针，失败重启 Pod
- `startupProbe`：启动探针，容器启动完成前禁用其他探针

### [高] CI/CD 与灰度发布

```groovy
stage('Build & Push') {
    sh 'docker build -t $IMAGE_NAME:$TAG .'
    sh 'docker push $IMAGE_NAME:$TAG'

stage('Deploy to K8s') {
```

**回滚**：
```bash
kubectl rollout undo deployment/my-app
kubectl rollout history deployment/my-app
kubectl rollout undo deployment/my-app --to-revision=3
```
