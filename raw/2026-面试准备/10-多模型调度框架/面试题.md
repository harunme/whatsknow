## 答案

### A1: 模型抽象层设计
**[10秒结论]** 统一抽象层通过 `LLMResponse` 数据类抹平所有 API 差异，后续业务代码完全不感知底层模型；Ollama 的 usage=0 通过本地 tokenizer 估算。

**[90秒拆解]**
- **API 差异**：OpenAI 返回 `choices[0].message.content`，Claude 返回 `content[0].text`，Ollama 返回 `message.content`，三者路径完全不同
- **统一接口**：
  ```python
  class LLMResponse:
      content: str
      model: str
      usage: dict  # {prompt_tokens, completion_tokens, total_tokens}
      finish_reason: str
  ```
- **Ollama usage 问题**：Ollama REST API 不返回 token 统计，通过本地 `tiktoken` 或字符数估算（中文约 1.5 字/token）
- **设计原则**：一个抽象基类 `BaseLLMClient`，子类实现 `chat()` 方法；切换模型只需改一行配置

**[20秒收尾]** 统一抽象层是架构基础，有了它才能做 Tag 路由、Fallback 降级、成本监控等上层功能。

---

### A2: Tag 路由机制
**[10秒结论]** Tag 是集合标签，路由层用集合运算匹配注册模型；多 Tag 时要求候选模型的 tags 集合 superset of required_tags；priority 控制同 Tag 下的模型选择优先级。

**[90秒拆解]**
- **Tag 定义**：
  - cheap：gpt-4o-mini / qwen-turbo，低成本 + 低温，格式化输出
  - fast：qwen-plus，适合对话摘要
  - powerful：gpt-4o / llama3，复杂推理
- **匹配逻辑**：
  ```python
  candidates = [cfg for cfg in self._all_configs()
               if required_tags.issubset(cfg.tags)]
  return max(candidates, key=lambda x: x.priority).client
  ```
- **priority**：同 Tag 下多模型时，priority 高的优先（可配置成本优先或质量优先）
- **无匹配处理**：抛出 `NoModelMatchError`，业务侧降级到默认模型

**[20秒收尾]** Tag 路由将业务意图和技术实现解耦，新模型上线只需注册 Tag，无需改业务代码。

---

### A3: Fallback 降级链
**[10秒结论]** Fallback 链按成本从低到高排列（cheap→fast→powerful），每级最多重试 2 次用指数退避；链过长会增加总超时时间，应控制在 3 级以内。

**[90秒拆解]**
- **降级顺序**：必须从成本低到成本高排列——cheap 失败切 fast 成本最多增 3-5 倍；若 expensive 作为主模型，降级失败就没有更高级别可用
- **指数退避**：`delay = base * 2^attempt`，避免瞬时故障时无效重试
- **链长度控制**：最多 3 级（主+2备），超过则增加超时风险且收益递减
- **记录元数据**：在 `raw_response._tried_models` 记录所有尝试过的模型，用于事后成本分析

**[20秒收尾]** 降级链设计的第一原则是成本递增，第二原则是最多重试 2 次（避免雪崩式重试）。

---

### A4: Schema Validation
**[10秒结论]** Pydantic BaseModel 定义输出 Schema，`model.with_structured_output()` 强制 LLM 输出符合格式；多模型格式差异通过正则提取 + JSON 容错解析 + 字段级降级三步兜底。

**[90秒拆解]**
- **Pydantic 定义**：
  ```python
  class TriageOutput(BaseModel):
      severity: str
      attack_stage: str
      mitre_tactics: list[str]
      confidence: float
  ```
- **多模型兜底策略**：① 正则提取 `{...}` JSON 块；② JSON 容错解析；③ 字段级正则提取可用内容；④ 部分填充 + `is_fallback=True` 标记
- **失败率监控**：某模型 validation 失败率 > 5% 说明输出不稳定，应切换模型或优化 prompt
- **Schema Validation 价值**：在模型输出不稳定时保证系统不崩溃，是生产级多模型调度的必备保障

**[20秒收尾]** Validation 失败率是关键监控指标，每月分析失败 case 优化 prompt，形成闭环。

---

### A5: Token 成本优化
**[10秒结论]** Redis 语义缓存通过 prompt hash 作为 key，命中即返回成本为 0；Prompt 压缩保留 system + 最近 N 条对话，按 token 预算截断历史。

**[90秒拆解]**
- **语义缓存**：
  ```python
  key = sha256(json.dumps({"messages": m, "temperature": t}, sort_keys=True))
  cached = await redis.get(f"llm:cache:{key}")
  ```
- **命中率**：客服场景 30-50%，告警研判场景（同类告警重复）约 20-30%
- **Prompt 压缩**：
  - 保留 system prompt（固定开销）
  - 保留最近 N 条对话（按 token 预算截断）
  - 估算：中文按字符数，英文按单词数 × 1.3
- **缓存防刷**：对同一 IP 的请求频率限流，避免用大量唯一 prompt 刷满 Redis

**[20秒收尾]** 三板斧：缓存优先（命中即返回）、分层 Tag（简单任务强制 cheap）、月度 Review（识别异常消耗）。

---

### A6: Ollama 私有化部署
**[10秒结论]** 7B 模型需 8GB 显存（RTX 4090×1），13B 需 16GB；并发通过信号量控制；OOM 时自动降级到云端 API。

**[90秒拆解]**
- **硬件配置**：7B Q4 → 8GB VRAM（4并发）；13B Q4 → 16GB VRAM（2并发）；70B Q4 → 48GB+ VRAM（1并发）
- **并发控制**：信号量 `asyncio.Semaphore(max_concurrent)` 限制并发数，超时用 `wait_for` 控制
- **OOM 降级**：
  ```python
  except (OllamaOOMError, OllamaTimeoutError) as e:
      return await fallback_cloud.chat(messages)  # 降级到云端 API
  ```
- **Ollama vs 云端混用**：主力流量走 Ollama（成本低、数据不出域），OOM/超时/峰值时切换云端 API

**[20秒收尾]** 专有云必须上 Ollama（数据安全），云端 API 做兜底（保证可用性），混合架构是最佳实践。

---

### A7: 多模型输出一致性
**[10秒结论]** Few-shot Prompt 提供相同示例减少随机性；输出结构强制约束（Pydantic Schema）；一致性评分用 powerful 模型对 cheap/fast 输出做自动评估。

**[90秒拆解]**
- **Few-shot 策略**：每个模式提供相同的 few-shot 示例，减小模型间的回答风格差异
- **结构强制约束**：Pydantic BaseModel 限制必须输出的字段，减少格式不一致
- **一致性评分**：用 gpt-4o 对 cheap/fast 的输出做结构化评估，输出格式一致性和内容相似度，超阈值则重新生成
- **温度控制**：cheap 模式用 temperature=0.1（极低随机性），fast 用 0.5-0.7

**[20秒收尾]** 多模型一致性的核心是减少随机性来源：Few-shot + 低温 + 结构化约束，三者缺一不可。

---

### A8: 成本监控
**[10秒结论]** Token 消耗记录到 SQLite/PG，通过月度统计按 mode 分组汇总；成本预警阈值每日检查，超过则告警；每月 Review 识别 cheap/fast/powerful 的使用占比和异常。

**[90秒拆解]**
- **记录字段**：timestamp / model / mode / prompt_tokens / completion_tokens / cost_usd / latency_ms / cached
- **月度统计**：
  ```sql
  SELECT mode, SUM(prompt_tokens), SUM(cost_usd), COUNT(*)
  FROM token_usage WHERE timestamp LIKE '2026-03%'
  GROUP BY mode
  ```
- **成本预警**：每日累计消耗超过 $100（可配置）触发告警，通过企业微信/Slack 通知
- **月度 Review 指标**：cheap/fast/powerful 各占 Token 消耗比例、每千次请求成本、缓存命中率

**[20秒收尾]** 成本监控的三个关键数字：cheap 应占 > 70% 流量、缓存命中率 > 30%、月成本增幅 < 10%。
