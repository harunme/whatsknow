# Agent Harness

AI Agent 的运行时工程框架，为 Agent 提供方向盘+护栏+仪表盘，使智能能可靠地转化为有用的工作。

## 一句话定义

**Harness = 马具 = 让马的力量可靠地转化为有用工作的那套东西。**

AI Agent 是发动机，Harness 是整辆车。引擎再强，没有方向盘和刹车，哪儿也去不了。

## 7 大核心模块

| # | 模块 | 作用 | 典型实现 |
|---|---|---|---|
| 1 | **沙箱隔离** | 安全执行代码/工具调用，出了事不影响外面 | Docker 容器，用完销毁 |
| 2 | **预算控制** | 防止死循环烧钱，每任务/每天硬上限 | Token 实时监控 + 超时停止 |
| 3 | **人类介入** | 关键操作审批，Agent 拿不准时暂停喊人 | 审批流 |
| 4 | **可观测性** | 记录思考轨迹、工具调用、Token 消耗，出错可回溯 | 结构化日志 |
| 5 | **断点续跑** | 长任务保存状态，崩溃后自动恢复接着跑 | JSON 格式进度文件 |
| 6 | **错误处理** | 自动识别错误类型，该重试的重试，不行就终止 | 重试策略 + 熔断器 |
| 7 | **工具框架** | 统一接入 API/数据库/浏览器/代码解释器 | 参数校验 + 超时控制 |

## 5 条设计原则

1. **默认不信任 Agent** — 所有操作在沙箱里，写操作要审批，任何操作有配额
2. **完全可观测** — 必须能回答：为什么这么选？Token 花在哪？哪里错了？
3. **预算铁律** — Agent 总能给你花完钱，每个任务有上限
4. **人永远在回路** — 不要相信"完全自主 Agent"，至少现在不行
5. **为失败而设计** — Agent 一定会失败，任务一定会中断，必须能保存状态、断点续跑

## Harness vs Agent 框架

| 维度 | Agent 框架 | Harness |
|---|---|---|
| **解决的问题** | "智能"问题 | "工程"问题 |
| **提供内容** | 组件、Prompt 模板、工具抽象、记忆接口 | 沙箱、监控、预算、人类介入口 |
| **代表产品** | LangChain、CrewAI | Claude Code 内置 Harness |
| **关系** | 画好蓝图 | 搭好工厂车间 |

> 你用 LangChain 写了 Agent = 画好蓝图。要在生产环境给用户用，还需要 Harness。

## 实证数据

LangChain 的编程 Agent：
- 基准测试：52.8%，排名前 30 开外
- 只改 Harness，不换模型：66.5%，冲进前 5
- **同一模型，不同 Harness，结果天差地别**

## 反直觉发现

Vercel 的 v0 编程 Agent：
- 工具从 15 个减到 2 个
- 准确率从 80% 飙升到 100%
- **工具太多反而让 Agent 困惑，Harness 的工具层需要精心约束选择范围**

## Claude Code 的 Harness

Claude Code 是 Harness 的典型产品实现，其架构在《御舆》书中有 42 万字的深度剖析。Claude Code 的：
- 权限管线（四阶段）= 人类介入 + 安全边界
- 记忆系统（MEMORY.md）= 断点续跑
- 上下文管理（渐进压缩）= 预算控制
- 钩子系统 = 可观测性 + 扩展点
